# 코드 에이전트 오케스트레이션

## 문서 목적과 범위

2026년 현재, 단일 AI 어시스턴트와 동기적 루프로 협업하던 방식에서 여러 에이전트가 각자의 컨텍스트 윈도우와 파일 범위를 가지고 비동기적으로 동작하는 오케스트레이터 모델로의 전환이 진행 중이다. 서브에이전트(Subagents), 에이전트 팀(Agent Teams), 계층적 위임 등 세 가지 핵심 패턴이 멀티 에이전트 코딩의 기본 구조를 이루며, 각각 병렬성·전문화·격리·복합 학습이라는 효과를 제공한다. 공유 태스크 리스트와 피어 투 피어 메시징은 에이전트 팀의 핵심 조율 메커니즘이며, 의존성 자동 해제와 병목 방지를 가능하게 한다.

2026년 기준 도구 생태계는 인프로세스 서브에이전트(Tier 1), 로컬 오케스트레이터(Tier 2), 클라우드 비동기 에이전트(Tier 3)의 세 계층으로 구분되며, 대부분의 개발자는 세 계층 모두를 용도에 맞게 혼합 사용한다. 병목은 코드 생성이 아닌 검증(Verification)으로 이동했으며, 플랜 승인·훅(Hooks)·`AGENTS.md`를 통한 품질 게이트와 인간 리뷰가 멀티 에이전트 체계의 신뢰성을 결정하는 핵심 요소이다.

본 문서는 Steve Yegge가 정리한 AI 도구 활용 8단계 프레임워크 가운데 레벨 5부터 레벨 8까지를 다룬다.

## 단일 지휘에서 오케스트레이터로의 전환

6개월 전까지 대부분의 개발자는 단일 AI 어시스턴트와 동기적으로 작업했으며, 단일 컨텍스트 윈도우가 작업의 상한선이었다. 현재 가장 생산적인 개발자들은 각자의 컨텍스트 윈도우와 파일 범위, 책임 영역을 가진 여러 에이전트를 비동기적으로 조율하는 방식으로 전환하였다.

컨덕터(Conductor) 모델은 단일 에이전트를 실시간 동기 방식으로 안내하는 모델이며, Claude Code CLI와 Cursor 에디터 내 에이전트 모드가 대표 도구이다. 오케스트레이터(Orchestrator) 모델은 각자의 컨텍스트 윈도우를 가진 여러 에이전트를 비동기로 조율하는 모델이며, Agent Teams, Conductor, Codex, Copilot Coding Agent가 대표 도구이다. 오케스트레이터로서는 명확한 스펙 작성, 작업 분해, 결과물 검증이라는 새로운 역량이 필요하다.

## AI 지원 코딩 성숙도 8단계

다음 단계 구분은 Steve Yegge가 정리한 AI 도구 활용 8단계 프레임워크에 따른 것이다.

### IDE 시대

- **L1 — AI 없음:** 전통적인 개발 워크플로우이다.
- **L2 — IDE 내 에이전트, 권한 수동 승인:** 모든 파일 변경을 직접 허가하며 완전 수동 제어이다.
- **L3 — YOLO 모드:** 에이전트가 IDE에서 자유롭게 실행하며 신뢰가 상승한다.

### 에이전트 우선

- **L4 — diff가 사라지고 대화가 주도:** 매번 diff를 검토하지 않고 에이전트 행동을 관찰하며 방향 제시에 집중한다.
- **L5 — 에이전트 우선, IDE는 나중:** 에이전트 대화가 주 작업 공간이고 IDE는 코드 확인용이다.

### 오케스트레이션

- **L6 — 에이전트 멀티플렉싱:** 기다리기 지루하여 에이전트를 하나씩 더 실행하고, 여러 스트림 사이를 오가다 멈출 수 없는 상태이다.
- **L7 — 에이전트 10개 이상, 수동 관리:** 「이런, 엉망이 됐다.」와 같이 잘못된 컨텍스트가 잘못된 에이전트에게 전달되며 「Claude Code가 Claude Code를 실행하면 어떨까?」라는 질문이 시작된다.
- **L8 — 나만의 오케스트레이터 구축:** 에이전트 생성·라우팅·관리를 직접 코드로 작성하는 코디네이션 레이어를 스스로 구현한다.

대부분의 개발자는 레벨 3~4에 머물러 있다. 오케스트레이션 계층은 레벨 6부터 시작하며, 레벨 5까지 올라온 역량과는 근본적으로 다른 스킬셋을 요구한다.

## 단일 에이전트의 한계와 멀티 에이전트를 쓰는 이유

### 단일 에이전트의 한계

**컨텍스트 과부하:** 단일 에이전트는 보유 가능한 정보량이 제한되며, 대규모 코드베이스는 단일 컨텍스트 윈도우를 압도한다.

**전문화 부재:** 데이터 레이어, API, UI, 테스트를 모두 처리하는 에이전트는 제너럴리스트에 불과하며, 데이터 레이어만 전담하는 에이전트가 훨씬 우수한 DB 코드를 작성한다.

**조율 부재:** 헬퍼 에이전트를 추가해도 서로 소통·태스크 공유·의존성 해결이 불가능하며, 조율 프리미티브 없이 에이전트를 늘릴수록 관리 난이도가 높아진다.

서브에이전트는 앞의 두 문제를, 에이전트 팀은 세 가지 문제 모두를 해결한다.

### 멀티 에이전트를 쓰는 이유

**병렬성(3배 처리량):** 프론트엔드·백엔드·테스트 에이전트가 동시에 작업한다.

**전문화(집중된 컨텍스트):** 각 에이전트는 자신이 담당하는 파일만 인식하며, `db.js`만 아는 에이전트가 전체 코드베이스를 다루는 에이전트보다 더 나은 DB 코드를 작성한다.

**격리(안전한 실행):** Git 워크트리가 각 에이전트에게 독립적인 작업 디렉터리를 제공하며 병합 충돌이 없다.

**복합 학습:** `AGENTS.md` 파일이 세션 간 패턴과 주의사항을 축적하여 매 세션이 다음 세션을 개선한다.

집중된 에이전트 세 개가 세 배 오래 작업하는 제너럴리스트 에이전트 하나보다 일관되게 높은 성과를 낸다.

## 핵심 패턴

### 패턴 1: 서브에이전트 — 집중된 위임

Task 도구를 사용해 부모 오케스트레이터에서 전문화된 자식 에이전트를 생성한다. 가장 단순한 멀티 에이전트 패턴으로 먼저 시도해 볼 수 있다.

**구체적 예시:** Claude Code에 「Express와 SQLite로 북마크 매니저 Link Shelf를 구축하라」는 프롬프트를 주면 부모 오케스트레이터가 세 서브에이전트 브리프로 분해한다. 데이터 레이어 서브에이전트는 `db.js`를 구축한 뒤 `DATA.md` 보고서를 작성한다. 비즈니스 로직 서브에이전트는 `validation.js`를 구축한 뒤 `LOGIC.md` 보고서를 작성한다. API 라우트 서브에이전트는 `DATA.md`와 `LOGIC.md`를 읽은 뒤 `server.js`를 구축한다. 처음 두 서브에이전트는 독립적으로 병렬 실행되고, 세 번째는 두 보고서가 완료된 뒤 시작하며 부모가 의존성 그래프를 수동 관리한다.

**서브에이전트의 한계:** 부모가 의존성 그래프를 수동 관리해야 하며, 에이전트 간 피어 메시징·공유 태스크 리스트가 없다. 파일 범위 관리가 느슨하면 충돌이 발생할 수 있다. 총 약 22만 토큰으로 비용 중립적 수준이다.

### 계층적 서브에이전트(팀의 팀)

오케스트레이터가 여섯 개 서브에이전트를 직접 생성하는 대신 피처 리드 두 명을 생성하고, 각 피처 리드가 자체적으로 전문가 두세 명을 생성하는 구조이다. 부모 오케스트레이터는 두 에이전트만 관리하여 컨텍스트를 깔끔하게 유지한다. 피처 리드 A는 「검색 기능 구축」이라는 브리프를 받아 자체적으로 분해한다. 실제 엔지니어링 조직의 운영 방식과 동일한 원리로, VP가 개별 엔지니어에게 직접 태스크를 배정하지 않고 테크 리드를 통해 전달하는 방식과 같다.

### 패턴 2: 에이전트 팀 — 진정한 병렬 실행

에이전트 팀은 Claude Code의 실험적 기능이며, `export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 명령으로 활성화한다.

**아키텍처 세 계층:** 팀 리드(Team Lead)는 작업 분해, 태스크 리스트 생성, 결과 종합을 담당한다. 공유 태스크 리스트는 상태(`pending`, `in_progress`, `completed`, `blocked`)·의존성 추적·파일 잠금을 제공한다. 팀원(Teammates)은 각자 독립적인 Claude Code 인스턴스이며 자체 컨텍스트 윈도우를 보유하고 tmux 분할 패널에서 실행한다.

팀원은 공유 리스트에서 태스크를 자율 선택하며 피어 투 피어 메시징으로 직접 소통한다(리드를 거치지 않음). 한 팀원이 태스크 완료 후 완료 표시를 하면 이에 의존하던 블로킹된 태스크가 자동으로 잠금 해제된다. `Ctrl+T`로 태스크 리스트의 시각적 오버레이를 토글할 수 있다.

**에이전트 팀 핵심 메커니즘:** 공유 태스크 리스트에서 백엔드 팀원이 검색 API를 완료로 표시하면 블로킹된 테스트 작성 태스크가 자동으로 `pending` 상태로 전환되며, 파일 잠금으로 동시 편집을 방지한다. 피어 메시징으로 백엔드 에이전트가 프론트엔드 에이전트에게 API 계약을 직접 전달할 수 있다(예: `GET /search?q= returns [{id,title,url}]`). 이는 리드가 조율 병목이 되는 것을 방지한다. 팀원이 유휴 상태가 되면 리드에게 자동 알림이 간다.

**에이전트 팀 핵심 권장 사항:** 팀원 세다섯 명이 최적점이며 토큰 비용은 팀 규모에 비례해 선형 증가한다. 플랜 승인 절차에서는 팀원이 구현 전 플랜을 작성하면 리드가 검토·승인 또는 거부하며, 코드가 존재하기 전에 아키텍처 문제를 발견하는 것이 훨씬 저렴하다. 집중된 팀원 세 명이 산만한 팀원 다섯 명보다 일관되게 우수한 성과를 낸다.

**에이전트 팀 신뢰성 팁:** 루프 가드레일과 반성 단계로 모든 팀원에게 `MAX_ITERATIONS=8` 상한선을 설정한다. 각 재시도 전 「무엇이 실패했나? 어떤 구체적 변경이 수정을 가져올까? 같은 접근 방식을 반복하고 있나?」라는 반성 프롬프트를 강제하면 교착 에이전트 발생이 대폭 감소한다. 전담 `@reviewer` 팀원으로 Claude Opus 4.6(읽기 전용)을 reviewer로 지정하고 lint·테스트·보안 스캔 도구만 사용하며 매 `TaskCompleted` 이벤트에 자동 트리거한다. 빌더 세네 명당 reviewer 한 명 비율을 두며 리드는 항상 검토 완료된 코드만 수신한다.

### 패턴 3: 규모에서의 오케스트레이션

다섯 개, 열 개, 스무 개 이상의 에이전트를 여러 레포와 피처에 걸쳐 관리할 때는 전용 오케스트레이션 도구가 필요하다.

**Tier 1 — 인프로세스 서브에이전트·팀:** Claude Code 서브에이전트·에이전트 팀이 여기에 해당한다. 단일 터미널 세션이며 추가 도구가 불필요하므로 여기서 시작한다.

**Tier 2 — 로컬 오케스트레이터:** 독립된 워크트리에서 여러 에이전트를 실행하고 대시보드·diff 리뷰·병합 제어를 유지한다. 알려진 코드베이스에서 3~10개 에이전트에 최적이다. Conductor, Vibe Kanban, Gastown, OpenClaw + Antfarm, Claude Squad, Antigravity, Cursor Background Agents가 해당한다.

**Tier 3 — 클라우드 비동기 에이전트:** 태스크 배정 후 노트북을 닫고 PR을 기다리는 방식이며 클라우드 VM에서 실행된다. Claude Code Web, GitHub Copilot Coding Agent, Jules by Google, Codex Web by OpenAI가 해당한다.

2026년 대부분의 개발자는 세 계층 모두를 사용한다. Tier 1은 인터랙티브 작업, Tier 2는 병렬 스프린트, Tier 3은 야간 백로그 처리에 쓴다.

## 도구 개요

### Conductor by Melty Labs

Mac에서 멀티 에이전트 오케스트레이션을 시작하는 가장 빠른 방법이다. 각각 독립된 git 워크트리에서 여러 Claude Code·Codex 에이전트를 병렬 실행한다. 시각적 대시보드와 diff 우선 리뷰 UI를 제공하며 API 비용만 부담한다(무료). macOS 전용(Apple Silicon·Intel)이다. 동일 레포에서 3~8개 피처를 병렬 작업하며 시각적 감독이 필요한 경우에 최적이다.

### Claude Code Web

`claude.ai/code`에서 접근한다. 완전 브라우저 기반이며 터미널이 불필요하다. GitHub 레포 연결 후 태스크 설명을 입력하면 Anthropic이 관리하는 클라우드 VM에서 실행된다. 중간 스티어링, 자동 PR 생성, iOS 앱 접근을 지원한다. 팀(터미널)은 에이전트와 함께 작업하는 방식이고 웹(브라우저)은 위임하고 자리를 비우는 방식이다.

### GitHub Copilot Coding Agent

IDE의 Copilot 에이전트 모드(동기·인터랙티브)와 구별된다. GitHub의 Copilot Coding Agent는 완전 비동기이다. 모든 GitHub 이슈를 `@copilot`에 배정하거나 Agents 패널에서 시작하면 GitHub Actions 환경에서 드래프트 PR이 생성된다. 태그 전에 자체 리뷰 루프를 실행한다. Claude Code·Codex 등 서드파티 에이전트도 동일 패널에서 접근할 수 있다. Slack, Jira, Linear, Azure Boards에서 트리거할 수 있다.

### Jules by Google

Google의 비동기 클라우드 코딩 에이전트이며 Gemini 기반이다. GitHub 레포 연결 → 태스크 설명 → Jules가 플랜 생성 → 승인 후 클라우드 VM에서 실행 → 전체 추론 과정과 터미널 로그가 포함된 PR이 반환된다. 오디오 체인지로그, 중간 태스크 중단, GitHub 이슈를 직접 파이핑하는 Jules Tools CLI를 제공한다. 레포의 `AGENTS.md`를 추가 설정 없이 자동으로 읽는다.

### OpenAI Codex Web

각 태스크가 GitHub 레포가 사전 로드된 독립된 샌드박스 컨테이너에서 실행된다. Web, CLI(오픈소스), macOS 앱, IDE 익스텐션, GitHub 통합을 ChatGPT 계정으로 연결하는 서피스 에코시스템을 보유한다. 검증 가능한 증거(Verifiable Evidence) 기능으로 모든 태스크에 대해 터미널 로그와 테스트 출력의 인용을 반환하여 실행 내역을 감사할 수 있다.

### Cursor Cloud Agents + Glass

웹, 데스크톱 앱, Slack, Linear, GitHub, API, PWA(`cursor.com/agents` 설치)에서 에이전트를 시작할 수 있다. Glass는 Cursor의 새 인터페이스로 에이전트 관리를 기본 화면으로 하고 기존 에디터는 필요 시 접근하는 보조 도구로 전환한다. 전체 생태계 전반에서 컨트롤 플레인이 주요 경험이 되고 에디터가 그 아래 하나의 악기가 되는 패턴을 반영한다.

### Vibe Kanban

「둠스크롤링 갭」(에이전트 작업 중 2~5분의 공백)을 해결한다. 태스크 카드를 생성한 뒤 `In Progress`로 드래그하면 각자의 워크트리·브랜치가 생성된다. 보드 내에서 diff 리뷰 및 실행 중 에이전트에 피드백을 전송할 수 있다. Claude Code, Codex, Gemini CLI, Amp, Cursor Agent CLI 등을 지원한다. 크로스 플랫폼(Mac, Windows, Linux)이며 무료이고 BYOK이다.

## 규모 확장 운영

### 멀티 모델 라우팅

모든 태스크에 가장 비싼 모델이 필요하지 않다. `MODEL_ROUTING.md` 파일을 생성하여 역할별로 라우팅한다. 플랜·아키텍처에는 저렴한 Gemini·Claude·OpenAI 모델을, 구현에는 Sonnet, Opus 또는 Codex를, 리뷰에는 전용 보안 모델을 둔다.

### 워크트리 라이프사이클 스크립트

세 가지 셸 앨리어스로 반복 작업을 자동화한다. `agent-spin <feature>`는 워크트리·브랜치 생성과 에이전트 시작을, `agent-merge <feature>`는 리베이스·리뷰·PR 생성을, `agent-clean`은 완료된 워크트리 제거를 수행한다. 약 12줄의 bash이며 Conductor가 이를 시각적으로 처리한다.

### 인간이 작성한 AGENTS.md만 허용

ETH Zurich Gloaguen et al.의 연구에서 LLM이 생성한 `AGENTS.md` 파일은 이점이 없으며 평균 약 3% 성공률 저하와 추론 비용 20% 이상 증가를 유발함을 확인하였다. 개발자가 작성한 컨텍스트 파일은 약 4% 성능 향상을 제공한다. 에이전트가 `AGENTS.md`에 직접 쓰는 것을 절대 허용하지 말 것이며 리드가 모든 줄을 승인해야 한다. `STYLE`, `GOTCHAS`, `ARCH_DECISIONS`, `TEST_STRATEGY` 등 명확한 섹션으로 간결하게 유지한다.

## 품질 게이트와 검증 병목

### 세 가지 품질 게이트

**플랜 승인:** 팀원이 코딩 시작 전 플랜을 작성하면 리드가 검토·승인 또는 거부한 뒤 구현한다. 나쁜 플랜을 수정하는 비용이 나쁜 코드를 수정하는 비용보다 훨씬 저렴하다.

**훅(Hooks):** 라이프사이클 이벤트의 자동화된 체크이다. `TeammateIdle` 훅은 에이전트가 작업을 중단하기 전 모든 테스트 통과 여부를 확인한다. `TaskCompleted` 훅은 완료 표시 전 lint·테스트를 실행한다. 훅이 실패하면 에이전트는 통과할 때까지 계속 작업한다.

**AGENTS.md를 통한 복합 학습:** 발견된 패턴·주의사항·스타일 선호를 포착하며 모든 에이전트가 세션 시작 시 읽고 매 세션마다 추가된다.

### 병목은 검증으로 이동

에이전트는 인상적인 출력을 놀라운 속도로 생성할 수 있으나 그 출력이 정확한지 확신하는 것이 어려운 부분이다. 변경 전 통과하는 테스트가 변경으로 인한 회귀를 잡아낼 것을 보장하지 않는다. 에이전트는 기술적으로 유효하지만 중요한 케이스를 놓치는 테스트를 작성할 수 있다. 컨텍스트 윈도우 한계로 인해 대규모 코드베이스에서 작업하는 에이전트는 현재 뷰 밖의 중요한 제약을 놓칠 수 있다. 단일 개발자에게는 성가신 엣지 케이스인 불안정한 환경이, 40개 에이전트가 동시에 같은 불안정한 테스트를 마주칠 때는 시스템적 블로커가 된다. 검증 인프라가 생성 역량을 따라잡을 때까지 인간 리뷰는 선택적 부가 비용이 아닌 안전 시스템이다.

## Ralph Loop와 자기 개선 에이전트

### Ralph Loop 패턴

Geoffrey Huntley와 Ryan Carson이 대중화하였다. 「잠자는 동안 배송」의 배후 패턴이다. Carson의 독립 도구 ralph(`snarktank/ralph`)가 핵심 루프를 구현하며 Antfarm 프로젝트는 OpenClaw 위에 멀티 에이전트 오케스트레이션을 추가한다.

**5단계 사이클:** Pick(`tasks.json`에서 다음 태스크 선택) → Implement(변경 수행) → Validate(테스트·타입·lint 실행) → Commit(체크 통과 시 커밋 및 태스크 상태 업데이트) → Reset(에이전트 컨텍스트 초기화 후 다음 태스크로).

**핵심 인사이트:** 각 반복마다 리셋함으로써 혼동 누적을 방지한다. 작고 범위가 명확한 태스크는 하나의 거대한 프롬프트보다 환각이 적고 더 깔끔한 코드를 생성한다.

**안전 장치:** 자동 재시도를 위해 오류를 피드백으로 전달하되 3회 이상 교착 상태이면 종료 및 재배정한다. 항상 피처 브랜치에서 작업한다. 반복·시간·토큰 상한선을 설정한다. 에이전트가 PR을 생성하며 병합 전 인간 리뷰를 둔다.

**컨텍스트 리셋 간 네 가지 메모리 채널:** git 커밋 히스토리, 진행 로그, 태스크 상태 파일(`tasks.json`), 장기 의미 기억으로서의 `AGENTS.md`를 유지한다.

### 에이전트를 시간이 지날수록 더 스마트하게

**`REFLECTION.md`를 통한 자기 반성:** 매 태스크 후 「무엇이 놀라웠나, `AGENTS.md`에 추가할 패턴 하나, 프롬프트 개선 하나」를 작성하도록 강제한다. 리드가 검토하고 승인된 학습 내용을 병합한다.

**토큰 예산과 종료 기준:** 에이전트별 상한선을 설정한다(예: 프론트엔드 18만 토큰, 백엔드 28만 토큰). 예산의 85%에서 자동 일시정지 및 리드 알림을 둔다. 동일 오류에서 3회 이상 교착이면 종료하고 새 에이전트에 재배정한다.

### Beads와 영구 메모리

Gastown의 「beads」 패턴은 전체 출처를 가진 모든 결정과 결과의 불변·git 기반 기록이다. 에이전트는 태스크 그래프와 SQL 주소 가능한 데이터 플레인을 통해 과거 beads를 조회한다. 단순 벡터 기반 RAG가 아닌 구조화된 쿼리 가능한 제도적 기억이다.

## 운영 규율과 팩토리 모델

### 이 모든 것을 작동시키는 규율

인간의 병목은 버그가 아니라 피처였다. 인간 속도로 코드를 작성할 때는 일찍 고통을 느낀다. 테스트 실패, 코드 리뷰 지적, 중복 발견 등 고통이 즉각적이어서 진행하면서 수정한다. 오케스트레이션된 에이전트 군단에서는 자연스러운 병목이 없다. 작은 실수(코드 냄새, 중복, 불필요한 추상화)가 따라잡을 수 없는 속도로 복합 증폭된다. 루프에서 빠져나온 상태이므로 아키텍처가 새 기능을 허용하지 않게 될 때까지 고통을 느끼지 못한다. 모든 품질 게이트(플랜 승인, 훅, 토큰 예산, 인간 리뷰)가 존재하는 이유는 없으면 에이전트 코딩으로 스스로를 막다른 골목에 몰아넣게 되기 때문이다.

### 태스크를 위임하되 판단은 보유할 것

**에이전트에게 맡겨야 하는 것:** 명확한 합격·불합격 기준이 있는 범위가 명확한 태스크, 보일러플레이트, 마이그레이션, 테스트 스캐폴딩, 직접 시도할 시간이 없는 접근법 탐색이다.

**직접 보유해야 하는 것:** 아키텍처와 API 설계이다(에이전트는 훈련 데이터에서 많은 나쁜 아키텍처를 학습해 엔터프라이즈 패턴을 스타트업에 그대로 적용할 수 있다). 만들지 않을 것을 결정하는 일이다(No라고 말하는 것은 에이전트에게 없는 기능이다). 전체 시스템 컨텍스트로 에이전트 출력을 리뷰한다. 자신의 시스템에 대한 이해를 잃으면 수정·확장·오작동 감지 능력을 잃는다.

### 스펙이 레버리지

50개 에이전트를 병렬로 오케스트레이션할 때 모호한 사고는 단순히 속도를 늦추는 것이 아니라 수십 개의 병렬 실행 전체에 증폭된다. 평범한 출력과 탁월한 출력의 차이는 거의 전적으로 스펙의 품질에 달려 있다. 모호한 스펙은 전체 플릿에 오류를 증폭하고, 명확한 아키텍처·통합 경계·엣지 케이스·불변 조건이 있는 정밀한 스펙은 전체에 걸쳐 정밀한 구현으로 증폭된다. 코드를 타이핑하는 기계적 작업은 자동화되는 중이며 시스템을 이해하는 인지적 작업은 자율적 작업자 전체 플릿에 걸쳐 증폭된다.

### 팩토리 모델

더 이상 단순히 코드를 작성하는 것이 아니라 소프트웨어를 만드는 팩토리를 구축하는 단계이다.

**6단계 생산 라인:** Plan(수용 기준이 있는 스펙 작성) → Spawn(팀 생성 및 에이전트 배정) → Monitor(5~10분마다 진행 상황 확인·블로커 해결, 호버링 금지) → Verify(테스트 실행·코드 리뷰, 검증이 병목) → Integrate(브랜치 병합·충돌 해결) → Retro(새 패턴으로 `AGENTS.md` 업데이트, 복합 학습).

**실용적 팁:** WIP 한도를 설정하여 의미 있게 리뷰할 수 있는 것보다 많은 에이전트 실행을 금지한다. 3~5개가 최적점이다. 종료 기준을 정의하여 동일 오류에서 3회 이상 교착 시 중단 및 재배정한다. 비동기 체크인으로 5~10분마다 진행 상황을 확인하며 에이전트가 자율적으로 작업하게 한다. 파일 하나에 오너 하나로 두 에이전트가 동일 파일을 편집하지 않도록 하며 충돌은 속도를 죽인다.

### 오늘 시작할 다섯 가지 패턴

1. **서브에이전트로 분해:** Task 도구로 특정 브리프와 파일 소유권이 있는 집중된 자식 에이전트를 생성한다. 설정이 불필요하며 오늘 시작할 수 있다.
2. **에이전트 팀으로 병렬성:** `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`을 활성화하고 리드와 팀원 세 명을 생성하며 공유 태스크 리스트로 조율한다.
3. **Git 워크트리로 격리:** 각 에이전트에 자체 워크트리를 제공하면 병합 충돌이 없으며 Conductor가 자동 처리한다.
4. **신뢰를 위한 품질 게이트:** 위험한 변경에 플랜 승인을 요구하고 태스크 완료 시 테스트를 실행하는 훅을 추가하며 검증 없이 에이전트 출력을 신뢰하지 않는다.
5. **`AGENTS.md`로 복합 학습:** 패턴·주의사항·스타일 선호를 문서화하고 매 세션이 읽고 업데이트하여 지식이 복합 증폭된다.

## 다중 에이전트 오케스트레이션이 잘 맞지 않았던 경험

최근 AI로 「제로 휴먼 컴퍼니」를 운영할 수 있게 해주는 Paperclip 프로젝트가 이슈가 되었다. 에이전트들을 회사 조직도처럼 구성하여 CEO, 팀장, 실무자 역할을 부여하고 회사의 미션과 세부 프로젝트들을 정의하면 자율적으로 일하기 시작한다. 한 달 만에 GitHub 스타 4만 개를 돌파했다.

그에 앞서 비슷한 접근으로 에이전트들을 「도시」로 은유한 Gastown도 있으며, Anthropic의 Claude Code Agent Teams, Cursor의 클라우드 에이전트까지 업계 전반에서 「에이전트 여러 개를 조직처럼 굴리자」는 흐름이 뚜렷하다.

필자는 이 흐름에 따라 Gastown과 Paperclip을 사용해 보았다. API 토큰 비용으로 계산하면 약 5,000달러 정도 지출하였다. 결론적으로 효과가 있는 영역은 일부 있었으나 한계가 훨씬 뚜렷하였다.

토큰 비용은 단일 에이전트 대비 5~10배 소비되는데 생산성이 비례하지 않았으며, 에이전트들이 맥락을 잃거나 인수인계가 중간에 끊기거나 틀린 결과를 스스로 「완료」로 처리하는 문제가 반복되었다. 이에 따라 필자는 겪은 문제와 인사이트를 다음 네 주제로 정리하였다. 멀티 에이전트 오케스트레이션의 세 가지 구조적 병목, 「도시」나 「회사」 메타포가 잘 먹히지 않는 이유, 실전에서 강건한 구조, 업무에서 에이전트 위임 가능 영역을 판별하는 다섯 가지 기준과 스코어링이다.

에이전트 시대에 「무엇을 맡기고 무엇을 맡기지 말아야 하는가」를 고민하는 이들에게 도움이 되기를 바란다.

## Paperclip 제품 개요

Paperclip은 AI 에이전트 팀이 자율적으로 회사를 운영하도록 설계된 오픈소스 오케스트레이션 도구이다. 여러 AI 에이전트를 조직도, 예산, 목표, 거버넌스 체계로 통합 관리한다. OpenClaw, Claude Code, Codex 등 다양한 에이전트를 직원으로 운용한다.

업무 목표 중심의 운영 관리 기능을 제공한다. 각 작업이 회사의 미션과 연결되어 목표 정렬(goal alignment)을 유지한다. 에이전트가 무엇을, 왜 수행하는지 명확히 인식하도록 설계하였다. Heartbeat 시스템으로 주기적 작업 수행 및 상태 점검을 하며 에이전트별 월별 예산 한도 설정을 지원한다.

다중 회사(Multi-Company) 지원으로 하나의 배포 인스턴스로 여러 회사를 독립적으로 운영할 수 있으며 각 회사별 데이터와 감사 로그가 완전히 분리된다. 조직 관리와 감사 추적을 위한 티켓 기반 시스템을 제공한다. 모든 대화와 결정 과정을 불변 감사 로그로 기록한다. 승인, 중단, 해고 등 거버넌스 제어 기능을 관리자 권한으로 수행한다.

### 핵심 기술적 특징

- **Atomic execution:** 중복 작업 및 예산 초과 방지
- **Persistent agent state:** 재시작 시 문맥 손실 최소화
- **Runtime skill injection:** 재학습 없이 워크플로우 확장
- **Governance with rollback:** 안전한 구성 변경 관리
- **Goal-aware execution:** 작업의 목적 일관성 유지

### Paperclip이 해결하는 것

다수의 AI 탭과 세션을 관리하기 어려운 환경에서 중앙 집중형 오케스트레이션을 제공한다. 에이전트 간 문맥 공유, 비용 추적, 반복 작업 자동화를 통합 관리한다. 기존의 폴더 기반 설정이나 수동 조정 대신 조직 단위의 자동 운영 체계를 구현한다.

### 기술 스택 및 실행 환경

Node.js 20+, pnpm 9.15+ 기반이다. 내장 PostgreSQL 데이터베이스를 자동 생성한다. Docker Compose를 통한 원클릭 셋업을 지원한다.

### 향후 로드맵

ClipMart를 통한 전체 회사 템플릿 다운로드 및 실행 기능, 클라우드 에이전트(Codex, Cursor 등) 통합, 플러그인 시스템 및 문서 개선 계획이 있다. MIT 라이선스 기반 오픈소스이다.

## Paperclip과 Gastown에서 영감을 받은 영구기관(perpetual engine) 컨셉의 오픈소스 AI 에이전트 프레임워크

Paperclip과 Gastown에서 영감을 받아, 토큰만 있으면 AI가 스스로 일을 이어가며 제품을 키워나가는 「영구기관(perpetual engine)」 컨셉의 오픈소스 에이전트 프레임워크를 만들었다.

핵심 아이디어는 AI를 단발성 호출이 아니라 지속적으로 작동하는 시스템으로 바라보자는 것이다. 이 프레임워크는 작업을 스스로 생성하고 우선순위를 조정하며 결과를 평가하고 다음 액션을 이어 간다. 즉 사람이 매번 개입하지 않아도 「일이 계속 굴러가는 구조」를 만드는 데 초점을 맞추었다.

이 프로젝트에서 중요하게 본 부분은 「가시성」이다. AI가 만든 것을 단순 텍스트 로그로 남기는 것이 아니라 생성한 UI 시안이나 결과물을 시각적으로 확인할 수 있고 작성한 문서나 리서치 결과도 구조적으로 볼 수 있도록 하였다. 따라서 단순히 「돌아간다」가 아니라 AI가 지금 어떤 방향으로 일하고 있고 무엇을 만들고 있는지를 한눈에 파악할 수 있다.

특히 스타트업 맥락에서 리서치 → 구현 → 테스트 → 개선 사이클을 자동화하고 제한된 리소스(토큰) 안에서 최대 성과를 내도록 설계하였다. 아직 완성형이라기보다 하나의 실험에 가깝다. 「AI가 실제로 제품을 계속 성장시킬 수 있는가?」에 대한 탐구로 볼 수 있다.
