# Claude Code Skills

## 배경과 범위

Claude Code에서 Skills는 가장 많이 사용되는 확장 포인트 중 하나로, Anthropic 내부에서 수백 개의 스킬을 실제로 운용하며 축적한 실전 노하우를 공개하였다. 스킬은 단순 마크다운 파일이 아니라 스크립트·에셋·데이터 등을 포함하는 폴더 구조이며, 에이전트가 폴더를 탐색하고 활용할 수 있는 형태이다. 라이브러리 레퍼런스, 제품 검증, 데이터 분석, 코드 스캐폴딩, CI/CD 등 아홉 가지 스킬 카테고리로 분류되며, 좋은 스킬은 하나의 카테고리에 깔끔히 맞아야 한다. 스킬 작성 시 Gotchas 섹션, 파일 시스템 활용, 점진적 공개(Progressive Disclosure), 데이터 저장 등의 실전 팁이 핵심이다. 조직 확장 시에는 내부 플러그인 마켓플레이스를 통해 스킬을 배포하고, 사용량 측정 훅으로 효과를 추적하는 구조가 권장된다.

## Skills의 정의

Skills에 대한 흔한 오해는 「그저 마크다운 파일이다」라는 것이지만, 실제 형태는 스크립트·에셋·데이터 등을 포함하는 폴더 구조이다. 에이전트는 이 폴더를 탐색하고, 내용을 발견하고, 조작할 수 있다.

## Claude Code와 Skills의 결합

Claude Code에서 Skills는 다양한 설정 옵션을 제공하며, 동적 훅(dynamic hooks) 등록도 가능하다. 가장 흥미로운 스킬들은 이러한 설정 옵션과 폴더 구조를 창의적으로 활용한다.

## 스킬 분류의 원칙

내부에서 사용 중인 스킬을 모두 분류한 결과, 반복적으로 나타나는 몇 가지 카테고리로 클러스터링된다. 좋은 스킬은 하나의 카테고리에 깔끔히 맞고, 혼란스러운 스킬은 여러 카테고리에 걸쳐 있다.

## 아홉 가지 카테고리

### 1. Library & API Reference

이 카테고리의 스킬은 라이브러리, CLI, SDK를 올바르게 사용하는 방법을 설명한다. 대상은 내부 라이브러리뿐 아니라 Claude Code가 자주 실수하는 일반 라이브러리를 포함한다. 레퍼런스 코드 스니펫 폴더와 주의사항(gotchas) 목록을 포함하는 경우가 많다.

예시: `billing-lib`(내부 결제 라이브러리의 엣지 케이스), `internal-platform-cli`(내부 CLI 래퍼의 모든 서브커맨드와 사용 예시), `frontend-design`(디자인 시스템 적용 개선).

### 2. Product Verification

코드가 정상 동작하는지 테스트하고 검증하는 방법을 기술하는 스킬이다. Playwright, tmux 등 외부 도구와 결합하여 사용하는 경우가 많다. Claude의 출력 정확성 보장에 매우 유용하며, 엔지니어가 일주일을 투자해서라도 검증 스킬을 우수하게 만들 가치가 있다. Claude가 출력을 영상으로 녹화하도록 하거나, 각 단계에서 상태에 대한 프로그래매틱 어설션을 강제하는 기법이 권장된다.

예시: `signup-flow-driver`(가입→이메일 인증→온보딩을 헤드리스 브라우저로 수행), `checkout-verifier`(Stripe 테스트 카드로 결제 UI 구동 후 인보이스 상태 검증), `tmux-cli-driver`(TTY가 필요한 인터랙티브 CLI 테스트용).

### 3. Data Fetching & Analysis

데이터 및 모니터링 스택에 연결하는 스킬이다. 크레덴셜이 포함된 데이터 가져오기 라이브러리, 특정 대시보드 ID, 일반적 워크플로우 안내 등을 포함할 수 있다.

예시: `funnel-query`(가입→활성화→결제 퍼널에 필요한 이벤트와 canonical user_id가 있는 테이블), `cohort-compare`(두 코호트의 리텐션/전환율 비교 및 통계적 유의성 플래깅), `grafana`(데이터소스 UID, 클러스터명, 문제→대시보드 룩업 테이블).

### 4. Business Process & Team Automation

반복적인 워크플로우를 하나의 명령으로 자동화하는 스킬이다. 비교적 단순한 지시문일 수 있으나 다른 스킬이나 MCP에 대한 복잡한 의존성을 가질 수 있다. 이전 실행 결과를 로그 파일에 저장하면 모델이 일관성을 유지하고 이전 실행을 반영하는 데 도움이 된다.

예시: `standup-post`(티켓 트래커, GitHub 활동, Slack을 종합한 포맷 스탠드업), `create-ticket`(스키마 강제 및 생성 후 워크플로우), `weekly-recap`(머지된 PR+닫힌 티켓+배포를 요약 포스트로 작성).

### 5. Code Scaffolding & Templates

코드베이스의 특정 기능에 대한 프레임워크 보일러플레이트를 생성하는 스킬이다. 조합 가능한 스크립트와 결합할 수 있으며, 코드만으로는 다 커버할 수 없는 자연어 요구사항이 있을 때 특히 유용하다.

예시: `new-framework-workflow`(어노테이션 포함 새 서비스/워크플로우/핸들러 스캐폴딩), `new-migration`(마이그레이션 파일 템플릿과 주의사항), `create-app`(인증, 로깅, 배포 설정이 미리 연결된 새 내부 앱).

### 6. Code Quality & Review

조직 내 코드 품질을 강제하고 코드 리뷰를 돕는 스킬이다. 최대한의 견고성을 위해 결정론적 스크립트나 도구를 포함할 수 있으며, 훅이나 GitHub Action의 일부로 자동 실행할 수도 있다.

예시: `adversarial-review`(새로운 시각의 서브에이전트가 비판→수정→반복하여 지적이 니트픽 수준으로 떨어질 때까지 진행), `code-style`(Claude가 기본적으로 잘 못하는 코드 스타일 강제), `testing-practices`(테스트 작성 방법과 테스트 대상 안내).

### 7. CI/CD & Deployment

코드베이스 내에서 코드를 가져오고, 푸시하고, 배포하는 스킬이다. 다른 스킬을 참조하여 데이터를 수집할 수 있다.

예시: `babysit-pr`(PR 모니터링→flaky CI 재시도→머지 충돌 해결→자동 머지 활성화), `deploy-service`(빌드→스모크 테스트→점진적 트래픽 롤아웃→에러율 비교→회귀 시 자동 롤백), `cherry-pick-prod`(격리된 worktree→cherry-pick→충돌 해결→템플릿 PR 생성).

### 8. Runbooks

증상(Slack 스레드, 알림, 에러 시그니처 등)을 입력받아 멀티 도구 조사를 수행하고 구조화된 리포트를 생성하는 스킬이다.

예시: `service-debugging`(증상→도구→쿼리 패턴 매핑), `oncall-runner`(알림 가져오기→일반적 원인 체크→결과 포맷팅), `log-correlator`(요청 ID로 관련 시스템 전체 로그 수집).

### 9. Infrastructure Operations

일상적 유지보수와 운영 절차를 수행하며, 파괴적 작업에 대한 가드레일을 포함하는 스킬이다. 엔지니어가 중요 운영에서 베스트 프랙티스를 따르기 쉽게 해준다.

예시: `resource-orphans`(고아 Pod/볼륨 탐색→Slack 알림→대기 기간→사용자 확인→계단식 정리), `dependency-management`(조직의 의존성 승인 워크플로우), `cost-investigation`(스토리지/이그레스 비용 급증 원인 조사용 버킷 및 쿼리 패턴).

## 작성 원칙과 실무 팁

### 당연한 내용은 쓰지 않는다

Claude Code는 코드베이스에 대해 이미 많이 알고 있고, 코딩에 대한 기본 의견도 가지고 있다. 지식 중심 스킬을 만든다면 Claude의 일반적 사고방식에서 벗어나게 하는 정보에 집중해야 한다. `frontend-design` 스킬이 좋은 예시로, Anthropic 엔지니어가 고객과 반복하며 Claude의 디자인 감각을 개선하기 위해 만들었으며, Inter 폰트와 보라색 그라디언트 같은 전형적 패턴을 피하도록 구성한다.

### Gotchas 섹션을 둔다

모든 스킬에서 가장 높은 신호 가치를 가지는 콘텐츠가 Gotchas 섹션이다. Claude가 스킬 사용 시 흔히 만나는 실패 지점에서 축적해야 하며, 시간이 지나면서 이러한 gotchas를 계속 업데이트하는 것이 이상적이다.

### 파일 시스템과 점진적 공개를 쓴다

스킬은 폴더이므로 전체 파일 시스템을 컨텍스트 엔지니어링과 점진적 공개의 수단으로 활용해야 한다. Claude에게 스킬에 어떤 파일이 있는지 알려주면 적절한 시점에 읽는다. 가장 단순한 형태는 상세 함수 시그니처와 사용 예시를 `references/api.md` 같은 별도 마크다운으로 분리하는 것이다. 최종 출력이 마크다운인 경우 `assets/` 폴더에 템플릿 파일을 둘 수 있다. 레퍼런스, 스크립트, 예시 등의 폴더가 Claude의 작업 효율을 높인다.

### 과도한 제약을 두지 않는다

Claude는 지시사항을 따르려 하지만, 스킬은 재사용성이 높으므로 지나치게 구체적인 지시는 주의해야 한다. 필요한 정보는 주되, 상황에 유연하게 적응할 수 있는 여지를 두어야 한다.

### 설정(Setup) 단계를 설계한다

일부 스킬은 사용자로부터 컨텍스트를 수집하는 설정 단계가 필요하다. 예로 스탠드업을 Slack에 올리는 스킬이라면 어느 채널에 올릴지 물어야 한다. 좋은 패턴으로는 설정 정보를 스킬 디렉토리의 `config.json` 파일에 저장하고, 설정이 안 되어 있으면 에이전트가 사용자에게 묻는 방식이다. 구조화된 다중 선택 질문을 제시하려면 AskUserQuestion 도구 사용을 지시할 수 있다.

### description 필드는 모델을 위한 트리거 설명이다

Claude Code가 세션을 시작하면 모든 가용 스킬의 description 목록을 빌드한다. Claude는 이 목록을 스캔하여 이 요청에 맞는 스킬이 있는지 판단한다. 따라서 description 필드는 요약이 아니라 이 스킬을 언제 트리거해야 하는지의 설명이다.

### 메모리 및 데이터 저장

스킬에 데이터를 저장하는 형태의 메모리를 포함할 수 있다. 단순한 텍스트 로그 파일이나 JSON 파일부터 SQLite 데이터베이스까지 다양한 형태가 가능하다. 예로 `standup-post` 스킬이 `standups.log`에 모든 작성 이력을 저장하면, 다음 실행 시 Claude가 자신의 이력을 읽고 어제 이후 변경사항을 파악할 수 있다. 스킬 디렉토리에 저장한 데이터는 스킬 업그레이드 시 삭제될 수 있으므로 `${CLAUDE_PLUGIN_DATA}`라는 안정적 폴더에 저장해야 한다.

### 스크립트와 라이브러리 제공

Claude에게 줄 수 있는 가장 강력한 도구 중 하나가 코드 자체이다. 스크립트와 라이브러리를 제공하면 Claude가 보일러플레이트 재구성 대신 구성(composition)에 집중할 수 있다. 예로 데이터 과학 스킬에 이벤트 소스에서 데이터를 가져오는 헬퍼 함수 라이브러리를 포함하면, Claude가 이 기능들을 조합하여 즉석에서 스크립트를 생성하고 「화요일에 무슨 일이 있었나?」 같은 복잡한 분석에 활용한다.

### On Demand Hooks

스킬이 호출될 때만 활성화되고 세션 동안만 지속되는 훅을 포함할 수 있다. 항상 실행하기에는 부담스럽지만 특정 상황에서 매우 유용한 강한 의견의 훅에 적합하다. 예시는 다음과 같다.

- `/careful` — `rm -rf`, `DROP TABLE`, `force-push`, `kubectl delete`를 PreToolUse 매처로 차단하며, 프로덕션 작업 시에만 활성화한다.
- `/freeze` — 특정 디렉토리 외의 모든 Edit/Write를 차단하며, 디버깅 시 의도치 않은 수정 방지에 유용하다.

## 배포, 조합, 측정

### 공유 방법

스킬의 큰 장점 중 하나는 팀 전체와 공유할 수 있다는 점이다. 두 가지 방식이 존재한다. 첫째, 스킬을 `./.claude/skills` 아래에 리포지토리에 체크인한다. 둘째, 플러그인으로 만들어 Claude Code Plugin 마켓플레이스에 업로드하고 사용자가 설치하게 한다.

### 마켓플레이스와 거버넌스

소규모 팀이 소수 리포에서 작업할 때는 리포에 체크인하는 방식이 적합하다. 체크인된 스킬은 모델의 컨텍스트에 조금씩 추가되므로, 규모가 커지면 내부 플러그인 마켓플레이스가 유리하다. 마켓플레이스에 들어갈 스킬을 결정하는 중앙 팀은 없으며, 가장 유용한 스킬이 자연스럽게 발견되는 방식이다. 시도해볼 스킬이 있으면 GitHub의 sandbox 폴더에 업로드하고 Slack 등에서 안내한다. 충분한 견인력(traction)을 얻으면 스킬 소유자가 마켓플레이스로 이동하는 PR을 올린다. 나쁘거나 중복된 스킬이 쉽게 만들어질 수 있으므로 릴리스 전 큐레이션 메커니즘이 중요하다.

### 스킬 조합(Composing Skills)

스킬 간 의존성이 필요할 수 있다 (예: 파일 업로드 스킬과 CSV 생성 및 업로드 스킬의 조합). 마켓플레이스나 스킬에 의존성 관리가 네이티브로 내장되어 있지는 않으나, 다른 스킬을 이름으로 참조하면 설치되어 있는 경우 모델이 호출한다.

### 스킬 측정

스킬의 성과를 파악하기 위해 PreToolUse 훅으로 회사 내 스킬 사용량을 로깅할 수 있다. 인기 있는 스킬이나 기대 대비 트리거가 부족한 스킬을 찾는 데 활용한다.

## 맺음말

Skills는 에이전트를 위한 매우 강력하고 유연한 도구이지만, 아직 초기 단계이며 모두가 최선의 활용법을 찾아가는 과정에 있다. 이 글은 확정적 가이드가 아니라 실전에서 효과가 있었던 팁 모음이다. 대부분의 스킬은 몇 줄과 하나의 gotcha에서 시작했으며, Claude가 새로운 엣지 케이스를 만날 때마다 사람들이 계속 추가하면서 개선된다.
