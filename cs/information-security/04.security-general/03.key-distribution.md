# 03. 키 분배 프로토콜 (Key Distribution Protocols)

## §0. 들어가며

> **키 분배 (Key Distribution)**: 통신 양방향이 안전한 통신에 사용할 비밀 키를 교환하거나 합의하는 과정.[^nist-57]

5 단원 (암호 알고리즘) 의 모든 알고리즘은 "**양방향이 같은 키를 공유한다**" 또는 "**한쪽이 다른 쪽의 공개키를 안다**" 는 가정 위에서 동작한다. 이 가정을 어떻게 충족하는가가 키 분배의 문제이며, 키 분배가 실패하면 그 위에서 동작하는 모든 암호화·인증이 무력화된다.

본 단원은 4과목 출제기준 1.3 의 두 영역을 다룬다:
- **대칭 키 기반 분배 방식의 원리 및 운영** — §2 (KDC·Needham-Schroeder·Otway-Rees·Kerberos)
- **공개 키 기반 분배 방식의 원리** — §3 (Diffie-Hellman·ECDH·RSA-KEM·STS)

## §1. 키 관리 개요

### §1-1. 키 관리 생명주기

> **키 관리 생명주기 (Key Lifecycle)**: 키의 생성부터 폐기까지의 전 과정을 단계별로 정의한 모델. NIST SP 800-57 Part 1 Rev. 5 가 표준화한다.[^nist-57]

키 관리는 단순히 "키를 만드는" 일이 아니라 키의 일생 전체를 다룬다. 한 단계에서 실패하면 — 예컨대 폐기 단계에서 키가 디스크에 남아있다면 — 시스템 전체의 보안이 그 단일 실패 지점에서 무너진다.

NIST SP 800-57 의 키 생명주기 단계:

| 단계 | 영문 | 설명 |
|---|---|---|
| 생성 | Generation | 안전한 난수원으로 키 생성 (NIST SP 800-90A·B·C)[^nist-90a] |
| 등록·배포 | Distribution | 정당한 주체에게 키를 전달 (본 단원의 핵심 주제) |
| 저장 | Storage | 무단 접근으로부터 키를 보호 (HSM·키링·암호화 저장) |
| 사용 | Usage | 정의된 용도로만 키 사용 (암호화·서명·MAC 분리) |
| 갱신 | Update | 키 노출 위험 완화를 위한 주기적 교체 |
| 백업·복구 | Archive·Recovery | 손실 대비 안전한 백업과 복구 절차 |
| 폐기 | Destruction | 사용 종료 후 안전한 삭제 (zeroization, 매체 파기) |

### §1-2. 키 분배의 두 패러다임

키를 양방향이 공유하는 방식은 본질적으로 두 가지로 나뉜다:

| 패러다임 | 영문 | 핵심 메커니즘 | 대표 사례 |
|---|---|---|---|
| **키 전송** | Key Transport | 한쪽이 키를 생성하고 다른 쪽에 안전하게 전달 | RSA-KEM, RSA-OAEP, KDC 기반 분배 |
| **키 합의** | Key Agreement | 양쪽이 정보를 교환하여 공통 비밀을 함께 도출 (어느 한쪽이 단독으로 키를 결정하지 않음) | Diffie-Hellman, ECDH, MQV |

**왜 이 구분이 중요한가?** — 키 전송 방식은 한쪽(보통 송신자) 의 난수 품질에 키 전체의 보안이 좌우된다. 키 합의 방식은 양쪽 정보가 결합되어 키가 만들어지므로, 한쪽 단말의 난수가 약해도 다른 쪽이 보강 가능하다. 또한 키 합의는 **Forward Secrecy** (§3-4) 를 자연스럽게 제공할 수 있다.

[^nist-90a]: NIST SP 800-90A Rev. 1, *Recommendation for Random Number Generation Using Deterministic Random Bit Generators*, 2015. https://csrc.nist.gov/publications/detail/sp/800-90a/rev-1/final

## §2. 대칭키 기반 키 분배

### §2-1. KDC — Key Distribution Center

> **KDC (Key Distribution Center)**: 모든 사용자와 사전에 공유한 장기 비밀키를 보유하고, 두 사용자가 통신할 때 단기 세션 키를 발급해 주는 신뢰할 수 있는 제3자(TTP).

대칭키만 사용하는 환경에서 N 명의 사용자가 모두 서로 통신해야 한다면 N(N-1)/2 개의 비밀키가 필요하다. 1,000 명이라면 약 50 만 개. 이 폭발적 증가를 막는 것이 KDC 의 발상이다 — KDC 와 각 사용자 간 1 개씩만 비밀키를 두고(=N 개), 사용자 간 통신은 KDC 가 발급한 세션 키로 한다.

**KDC 기반 분배의 일반 흐름**:
```
1. Alice → KDC : "나는 Bob 과 통신하고 싶음"
2. KDC → Alice : { 세션 키 K_S, Bob 용 티켓 [K_S]_K_B } 를 K_A 로 암호화
3. Alice → Bob  : Bob 용 티켓 [K_S]_K_B 전달
4. Alice ↔ Bob : K_S 로 암호화된 통신
```

KDC 의 한계:
- **단일 장애점 (SPOF)** — KDC 가 침해되면 모든 사용자 키 노출
- **온라인 의존** — 매 통신 시작마다 KDC 와 통신 필요
- **확장성** — 한 KDC 가 처리 가능한 사용자 수 제한

> *cross-ref*: KDC 의 구체 구현으로 Kerberos 가 있으며 [01 단원 §5-2](./01.authentication.md) 에서 본격 다루고, 본 단원 §2-4 에서 키 분배 측면으로 재조명한다.

### §2-2. Needham-Schroeder 대칭키 프로토콜

> **Needham-Schroeder Symmetric Key Protocol**: KDC 기반 키 분배 프로토콜의 효시. Needham & Schroeder 1978 (CACM) 가 제안.[^ns]

Needham-Schroeder 의 발상은 KDC 기반 분배에 **재전송 공격 방지(논스)** 를 결합한 것이다. 그러나 1981 년 Denning & Sacco 가 이 프로토콜의 결함을 발견했고, 이후의 모든 키 분배 프로토콜이 이 결함을 어떻게 해결하느냐에 영향을 받았다.

**프로토콜 흐름** (A = Alice, B = Bob, S = KDC, K_AS·K_BS = 사전 공유 장기 키):
```
1. A → S : A, B, N_A
2. S → A : { N_A, B, K_AB, { K_AB, A }_K_BS }_K_AS
3. A → B : { K_AB, A }_K_BS
4. B → A : { N_B }_K_AB
5. A → B : { N_B − 1 }_K_AB
```

- N_A, N_B 는 각자 생성한 논스(nonce)
- K_AB 는 KDC 가 발급한 세션 키
- 4·5 단계는 양방향이 K_AB 를 보유함을 상호 확인

**Denning-Sacco 공격 (재전송 결함)**: 단계 3 의 메시지 `{ K_AB, A }_K_BS` 는 시간 정보(타임스탬프) 가 없다. 공격자가 과거 세션의 K_AB 를 어떤 경로로든 노출시키는 데 성공하면, 그 K_AB 와 단계 3 메시지를 재전송하여 Bob 에게 "지금 Alice 와 통신 시작" 인 것처럼 위장할 수 있다(Bob 입장에서는 메시지가 정당한 K_BS 로 암호화되었으므로 거부할 근거가 없다).

**해결책**: 후속 프로토콜(Otway-Rees, Kerberos) 은 **타임스탬프** 또는 **양방향 논스**를 추가해 이 결함을 보완했다.

> *시험 포인트*: Needham-Schroeder 의 **재전송 공격 결함**과 **Denning-Sacco** 라는 이름은 시험 단골. 후속 프로토콜이 도입한 보완책(타임스탬프) 도 함께 출제된다.

### §2-3. Otway-Rees 프로토콜

> **Otway-Rees Protocol**: Needham-Schroeder 의 재전송 결함을 보완하기 위해 Otway & Rees 1987 가 제안한 KDC 기반 키 분배 프로토콜.[^or]

Otway-Rees 의 핵심 개선은 **세션 식별자(session identifier) M 을 도입**하여 같은 세션의 모든 메시지가 동일 M 을 갖도록 하고, KDC 가 양쪽의 논스를 함께 검증하는 것이다.

**프로토콜 흐름** (M = 세션 ID):
```
1. A → B : M, A, B, { N_A, M, A, B }_K_AS
2. B → S : M, A, B, { N_A, M, A, B }_K_AS, { N_B, M, A, B }_K_BS
3. S → B : M, { N_A, K_AB }_K_AS, { N_B, K_AB }_K_BS
4. B → A : M, { N_A, K_AB }_K_AS
```

- 세션 ID M 이 모든 메시지에 포함되어 KDC 가 같은 세션의 메시지인지 검증 가능
- KDC 는 M, A, B 가 메시지 외부와 내부에서 일치하는지 확인 후에만 키 발급
- A·B 각자가 자신의 논스 N_A·N_B 를 K_AB 와 함께 받아 KDC 가 같은 세션에 응답한 것임을 확인

> *시험 포인트*: Otway-Rees 는 Needham-Schroeder 의 재전송 결함 해결을 위해 **세션 ID** 와 **양방향 논스 검증**을 도입한 프로토콜이다.

### §2-4. Kerberos 키 분배 측면

[01 단원 §5-2](./01.authentication.md) 에서 다룬 Kerberos v5 (RFC 4120) 를 키 분배 프로토콜로 재조명하면 다음과 같은 발전이 보인다:

| 프로토콜 | 핵심 개선 |
|---|---|
| Needham-Schroeder (1978) | KDC 기반 + 논스 |
| Otway-Rees (1987) | 세션 ID + 양방향 논스 검증 |
| **Kerberos v5 (2005, RFC 4120)** | **타임스탬프** + **TGT/ST 2단 티켓** + **상호 인증** |

Kerberos 의 타임스탬프 기반 신선도 검증(±5분 클럭 스큐) 이 Needham-Schroeder 의 재전송 결함을 근본적으로 해결한다. 다만 이 보안 속성을 위해 **시각 동기 NTP 의존** 이라는 새로운 운영 부담이 도입된다(NTP 자체가 공격 표적이 됨).

> *시험 포인트*: Kerberos 의 키 분배 관점에서의 핵심은 ① TGT(Ticket Granting Ticket) 와 ST(Service Ticket) 의 2단 티켓 구조, ② AS 가 발급한 K_C-TGS 와 TGS 가 발급한 K_C-S 의 단기 세션 키 분리, ③ 타임스탬프 기반 재전송 방지.

[^ns]: R. M. Needham and M. D. Schroeder, "Using Encryption for Authentication in Large Networks of Computers," *Communications of the ACM*, vol. 21, no. 12, pp. 993–999, 1978. https://doi.org/10.1145/359657.359659
[^or]: D. Otway and O. Rees, "Efficient and Timely Mutual Authentication," *ACM SIGOPS Operating Systems Review*, vol. 21, no. 1, pp. 8–10, 1987. https://doi.org/10.1145/24592.24594

## §3. 공개키 기반 키 분배

### §3-1. 공개키 분배의 발상

대칭키 분배는 KDC 같은 신뢰 제3자에 의존한다. 공개키 암호의 등장(Diffie & Hellman 1976) 이후로는 두 가지 새로운 가능성이 열렸다:

1. **키 전송 (Key Transport)** — 송신자가 세션 키를 생성하여 수신자의 공개키로 암호화해 전송 (RSA-KEM, RSA-OAEP)
2. **키 합의 (Key Agreement)** — 양쪽이 공개 매개변수를 교환하여 공통 비밀을 함께 도출 (Diffie-Hellman)

이 두 방식은 KDC 와 달리 사전 공유 비밀이 필요 없다는 점이 결정적 우월성이다. 다만 **MITM 방어를 위한 신뢰 인프라(PKI 또는 사전 인증)** 가 별도 필요하다(§3-5).

### §3-2. Diffie-Hellman 키 합의 — RFC 2631

> **Diffie-Hellman (DH) Key Agreement**: 양쪽이 공개 매개변수를 교환하여 공통 비밀을 도출하는 키 합의 알고리즘. Diffie & Hellman 1976 (IEEE Trans. Inf. Theory) 가 제안한 공개키 암호의 효시.[^dh][^rfc-2631]

DH 의 수학적 토대는 **이산로그 문제의 어려움** — 큰 소수 p, 생성자 g 가 공개되어 있을 때 g^a mod p 를 알아도 a 를 계산하기 어렵다 — 에 있다.

**프로토콜 흐름**:
```
공개 매개변수 (사전 합의): p (큰 소수), g (Z_p* 의 생성자)

Alice                                    Bob
─────                                    ───
a ← random (개인 키)                     b ← random (개인 키)
A = g^a mod p (공개 키)        ──A──>
                                <──B──   B = g^b mod p (공개 키)
K = B^a mod p                            K = A^b mod p
   = g^(ab) mod p                           = g^(ab) mod p

→ 양쪽이 동일한 공유 비밀 K = g^(ab) mod p 를 도출
```

**보안 가정**: **CDH (Computational Diffie-Hellman) 가정** — g, g^a, g^b 가 주어졌을 때 g^(ab) 를 계산하는 것이 어렵다. 더 강한 **DDH (Decisional Diffie-Hellman) 가정** 도 키 도출 함수와 결합 시 사용된다.

**권장 매개변수 (NIST SP 800-56A Rev. 3)**:[^nist-56a]
- 안전 소수 p 의 길이: 2048 bit 이상 (현재 권장)
- p, g 는 RFC 3526 의 표준 그룹 또는 FIPS 186-5 부록의 검증된 매개변수 사용

> *시험 포인트*: DH 의 보안 = **이산로그 문제의 어려움**. 공유 비밀 K = **g^(ab) mod p**. 양쪽이 자신의 개인키 a·b 를 노출하지 않고도 공통 키 도출.

### §3-3. ECDH — 타원곡선 디피-헬만

> **ECDH (Elliptic Curve Diffie-Hellman)**: DH 의 곱셈 군(Z_p*) 을 타원곡선 군(E(F_p)) 으로 대체한 변형. NIST SP 800-56A Rev. 3 가 표준화한다.[^nist-56a]

ECDH 의 수학적 토대는 **타원곡선 이산로그 문제(ECDLP)**다 — 같은 보안 강도에서 키 길이가 훨씬 짧다.

| 보안 수준 | RSA / DH 키 길이 | ECC 키 길이 |
|---|---|---|
| 80 bit | 1024 bit | 160 bit |
| 112 bit | 2048 bit | 224 bit |
| 128 bit | 3072 bit | 256 bit |
| 192 bit | 7680 bit | 384 bit |
| 256 bit | 15360 bit | 512 bit |

(NIST SP 800-57 Part 1 Rev. 5 표 2 기준)[^nist-57]

**권장 곡선**:
- **NIST 곡선** — P-256 (secp256r1), P-384, P-521 (FIPS 186-5)[^fips-186]
- **Curve25519** — 성능·구현 안전성 우수, RFC 7748 에서 X25519 키 합의로 표준화[^rfc-7748]
- **Curve448** — 더 높은 보안 강도, RFC 7748 에서 X448 로 표준화

**프로토콜 흐름**:
```
공개 매개변수: 타원곡선 E, 베이스 포인트 G, 곡선 위수 n

Alice                                    Bob
─────                                    ───
a ← random (1 ≤ a < n)                  b ← random (1 ≤ b < n)
A = a·G (점 곱셈)              ──A──>
                                <──B──   B = b·G
K = a·B = (a·b)·G                       K = b·A = (a·b)·G

→ 공유 비밀 K 의 x 좌표를 KDF 입력으로 사용
```

> *시험 포인트*: ECDH 는 동일 보안 강도에서 DH 보다 훨씬 짧은 키를 사용. 256 bit ECC = 3072 bit RSA/DH ≈ 128 bit 대칭 키 보안.

### §3-4. Forward Secrecy 와 DHE/ECDHE

> **Forward Secrecy (FS / PFS, 순방향 비밀성)**: 장기 비밀키가 사후에 노출되어도 이전에 수행한 통신의 비밀성이 보장되는 보안 속성.

DH·ECDH 의 큰 매력 중 하나가 Forward Secrecy 를 자연스럽게 제공한다는 점이다. 매 세션마다 새로운 임시(ephemeral) 키쌍을 생성하면 — DH**E** (DH Ephemeral) · ECD**H**E (ECDH Ephemeral) — 세션 키 K = g^(ab) 는 그 세션 종료 후 폐기되며, 장기 키와 무관하게 안전하다.

| 키 합의 모드 | 키 수명 | Forward Secrecy |
|---|---|---|
| DH·ECDH (정적) | 장기 (인증서에 고정) | ❌ — 장기 키 노출 시 모든 과거 세션 노출 |
| DH**E**·ECD**H**E (임시) | 세션마다 새로 | ✅ — 임시 키는 세션 종료 후 폐기 |

**TLS 1.3 의 결정**: TLS 1.3 (RFC 8446) 은 **모든 키 합의를 (EC)DHE 로만 허용**한다 — RSA 키 전송은 폐기되었다. Forward Secrecy 가 사실상 의무화된 것이다.

> *cross-ref*: TLS cipher suite 에서 ECDHE 의 의미는 [2과목 07 단원](../02.network-security/07.security-protocols.md) 에서 다뤘다. 본 단원은 키 합의 자체의 보안 속성에 집중한다.

### §3-5. MITM 방어 — 인증된 DH 와 STS

DH·ECDH 의 치명적 약점은 **MITM 공격**이다. Alice 와 Bob 사이에 공격자 M 이 끼어들어 각각과 별도 DH 키 합의를 수행하면, M 은 Alice-M·M-Bob 두 세션 키를 모두 알면서 양쪽에는 정상 통신처럼 보이게 할 수 있다.

**해결책 1 — 인증된 DH (Authenticated DH)**: DH 매개변수 교환 시 서명·MAC·PKI 인증서로 출처를 인증. TLS·SSH·IKEv2 가 모두 이 방식.

**해결책 2 — Station-to-Station (STS) 프로토콜**: Diffie, van Oorschot, Wiener 1992 가 제안한 인증된 DH 의 표준적 형태.

**STS 프로토콜 흐름** (Cert_A·Cert_B = 사전 발급된 X.509 인증서, Sig_A·Sig_B = 각자의 개인키 서명):
```
1. A → B : g^a
2. B → A : g^b, Cert_B, Sig_B(g^b, g^a)
3. A → B : Cert_A, Sig_A(g^a, g^b)
   (양쪽이 K = g^(ab) 도출 후 MAC 으로 핸드셰이크 검증)
```

서명에 `g^a` 와 `g^b` 가 모두 포함되어 있어, MITM 이 자신의 g^m 을 끼워넣어도 정당한 서명을 만들 수 없다(자기 인증서의 개인키를 갖지 않은 g^a·g^b 에 대한 서명을 위조 불가).

> *시험 포인트*: DH 의 MITM 공격은 **인증되지 않은 DH 의 본질적 약점**. 해결책은 ① 인증서 기반 서명(STS), ② 사전 공유 비밀 기반 MAC, ③ TLS·SSH·IKEv2 같은 통합 프로토콜에 의한 보호.

### §3-6. RSA 기반 키 캡슐화 — RSA-KEM·RSA-OAEP

> **RSA-KEM (Key Encapsulation Mechanism)**: 송신자가 임의 정수 z 를 RSA 로 암호화해 전송하고, 수신자가 z 를 복호화한 후 KDF(z) 로 세션 키를 도출하는 키 전송 방식. NIST SP 800-56B Rev. 2 가 표준화한다.[^nist-56b]

> **RSA-OAEP (Optimal Asymmetric Encryption Padding)**: RSA 암호문에 OAEP 패딩을 적용해 안전성을 강화한 방식. RFC 8017 (PKCS#1 v2.2) 가 표준화한다.[^rfc-8017]

RSA-KEM 의 흐름:
```
1. 송신자 Alice :
   - z ← random (RSA 모듈러스 n 보다 작은 임의 정수)
   - c = z^e mod n  (수신자 Bob 의 공개키 (n, e) 로 암호화)
   - K = KDF(z)
   - c 를 Bob 에게 전송
2. 수신자 Bob :
   - z = c^d mod n  (자신의 개인키 d 로 복호화)
   - K = KDF(z)
3. 양쪽이 동일한 K 보유
```

RSA 키 전송과 DH·ECDH 키 합의의 비교:

| 측면 | RSA-KEM | DH·ECDH |
|---|---|---|
| 키 생성 주체 | 송신자 단독 | 양쪽 공동 |
| Forward Secrecy | ❌ (수신자 장기 키 노출 시 과거 세션 모두 노출) | ✅ (DHE·ECDHE 로 매 세션 임시 키) |
| 계산 비용 | RSA 단방향 (송신자만 암호화) | 양쪽 모두 지수승 |
| TLS 1.3 사용 | ❌ 폐기 | ✅ 의무 |

> *시험 포인트*: RSA 키 전송은 **Forward Secrecy 미제공** 이 본질적 약점. TLS 1.3 이 RSA 키 전송을 폐기한 핵심 이유.

## §4. 키 도출 함수 (KDF)

> **KDF (Key Derivation Function)**: 키 합의·전송으로 얻은 원시 비밀(g^(ab) 등) 을 암호 알고리즘에 사용 가능한 실제 키로 변환하는 함수. NIST SP 800-56C Rev. 2 가 키 합의 후의 KDF 를 표준화한다.[^nist-56c]

KDF 가 필요한 이유는 — 원시 DH 출력값(g^(ab) mod p) 의 비트가 균일 분포가 아니기 때문이다. 그대로 AES 키로 사용하면 일부 키 공간으로 편향된다. KDF 는 다음 3 가지 기능을 동시에 수행한다:

1. **랜덤성 추출** — 비균일 비밀에서 균일 랜덤 키 추출
2. **키 분리** — 동일 비밀에서 암호화 키·MAC 키·IV 등을 분리 도출
3. **키 신선도** — 컨텍스트(세션 ID·타임스탬프) 를 입력에 포함해 같은 비밀에서 다른 키 생성

**대표 KDF**:
- **HKDF** (RFC 5869) — HMAC 기반, TLS 1.3·Signal Protocol 에서 사용[^rfc-5869]
- **NIST SP 800-108** — KMAC·HMAC·CMAC 기반 KDF
- **NIST SP 800-132** — 패스워드 기반 KDF (PBKDF2)

> *cross-ref*: HKDF 의 내부 HMAC 은 [06 단원 §5](./06.hash-and-mac.md), PBKDF2·Argon2 는 [06 단원 §7](./06.hash-and-mac.md) 에서 다룬다.

[^nist-56c]: NIST SP 800-56C Rev. 2, *Recommendation for Key-Derivation Methods in Key-Establishment Schemes*, 2020. https://csrc.nist.gov/publications/detail/sp/800-56c/rev-2/final
[^rfc-5869]: RFC 5869, *HMAC-based Extract-and-Expand Key Derivation Function (HKDF)*, Krawczyk & Eronen, 2010. https://www.rfc-editor.org/rfc/rfc5869

## §5. 키 보호 인프라

### §5-1. HSM — 하드웨어 보안 모듈

> **HSM (Hardware Security Module)**: 암호 키의 생성·저장·연산을 수행하는 전용 하드웨어. 키가 평문으로 외부에 노출되지 않도록 물리적·논리적 보호를 제공.

HSM 의 핵심 보안 속성:
- **키 무반출** — 키는 HSM 내부에서만 생성·사용되며 평문으로 외부 전달 불가
- **물리적 변조 방지** — 외부 침입 탐지 시 자가 파괴(zeroization)
- **FIPS 140 검증** — NIST FIPS 140-2 / 140-3 인증을 받은 HSM 만 정부·금융 환경 사용 허용[^fips-140]

활용처: CA 의 루트 키 보호, 금융 결제 시스템의 마스터 키, 클라우드 KMS 의 신뢰 루트.

### §5-2. 키 백업·키 에스크로

**키 백업 (Key Backup)**: 키 손실에 대비해 안전한 사본을 보관. HSM 의 보호된 환경에서 m-of-n 분할 백업 권장.

**키 에스크로 (Key Escrow)**: 정부·법 집행 기관이 필요 시 키를 복구할 수 있도록 제3자에게 키 복사본을 위탁. 1990 년대 미국의 Clipper Chip 이 대표 사례이며 시민 자유·기술 신뢰성 측면에서 큰 논란을 빚었다.

> *시험 포인트*: 키 백업과 키 에스크로의 차이 — 백업은 **소유자 자신**의 키 손실 대비, 에스크로는 **제3자(정부)** 가 키 복구 권한을 보유.

[^fips-140]: NIST FIPS 140-3, *Security Requirements for Cryptographic Modules*, 2019. https://csrc.nist.gov/publications/detail/fips/140/3/final

## §6. 단원 요약

1. **키 관리 생명주기 7 단계** — 생성·등록·저장·사용·갱신·백업·폐기.
2. **두 패러다임** — 키 전송(한쪽 생성·전달) vs 키 합의(양쪽 공동 도출).
3. **Needham-Schroeder** — KDC 기반 분배의 효시. **Denning-Sacco 재전송 결함** 보유.
4. **Otway-Rees** — 세션 ID + 양방향 논스로 N-S 결함 보완.
5. **Kerberos v5** — 타임스탬프 + TGT/ST 2단 티켓. NTP 의존이 새 운영 부담.
6. **DH 키 합의** — 공유 비밀 K = **g^(ab) mod p**. 보안 = 이산로그 문제 어려움.
7. **ECDH** — 타원곡선 변형. 동일 보안 강도에서 키가 짧음 (256 bit ECC ≈ 3072 bit RSA).
8. **Forward Secrecy** — DH**E**·ECD**H**E (임시 키) 만 제공. 정적 DH·RSA 키 전송은 미제공.
9. **MITM 방어** — 인증된 DH (STS·TLS·SSH·IKEv2). 서명에 양쪽 공개값 포함.
10. **RSA-KEM·RSA-OAEP** — 키 전송 방식. Forward Secrecy 미제공으로 TLS 1.3 폐기.
11. **KDF** — 원시 비밀에서 균일 랜덤 키 추출. HKDF (RFC 5869) 가 대표.
12. **HSM** — 키 무반출 하드웨어. FIPS 140-3 인증 필수 (정부·금융).

## §7. 시험 함정 정리

| 함정 | 정답 |
|---|---|
| 대칭키 분배가 필요한 키 수? | N(N-1)/2 (사용자 N 명 모두 통신 시). KDC 도입 시 N. |
| Needham-Schroeder 의 결함? | **Denning-Sacco 재전송 공격** — 단계 3 메시지에 신선도 정보 부재. |
| Otway-Rees 의 N-S 보완책? | **세션 ID(M)** + KDC 의 **양방향 논스 검증**. |
| Kerberos 의 N-S 보완책? | **타임스탬프** + TGT/ST 2단 티켓. NTP 의존 도입. |
| DH 의 공유 비밀 수식? | K = **g^(ab) mod p**. |
| DH 의 보안 가정? | **CDH (Computational Diffie-Hellman)** — 이산로그 문제 어려움. |
| ECDH 의 256 bit 키 = 몇 bit RSA? | **3072 bit RSA** (NIST SP 800-57 Part 1 Rev. 5). |
| Forward Secrecy 를 제공하는 키 합의? | DH**E** · ECD**H**E (임시·ephemeral 키 사용). 정적 DH·RSA 키 전송 ❌. |
| TLS 1.3 가 폐기한 키 교환? | **RSA 키 전송**. Forward Secrecy 미제공이 주된 이유. |
| DH 의 MITM 방어? | **인증된 DH** (STS, TLS, SSH, IKEv2) — 서명·MAC 으로 출처 인증. |
| KDF 가 필요한 이유? | DH 출력의 **비균일 분포** → 균일 랜덤 키 추출. + 키 분리. |
| 키 백업 vs 키 에스크로? | 백업 = **소유자 자신** 의 손실 대비, 에스크로 = **제3자** 가 복구 권한 보유. |
| HSM 의 핵심 보안 속성? | **키 무반출** (평문 외부 노출 금지) + 물리적 변조 방지 + FIPS 140 인증. |
