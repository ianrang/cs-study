# 02. 접근통제 (Access Control)

## §0. 들어가며

> **접근통제 (Access Control)**: 주체(subject) 가 객체(object) 에 수행하는 행위를 정책에 따라 허용·거부하는 보안 메커니즘.[^nist-53]

[01 단원](./01.authentication.md) 의 인증이 "이 주체가 누구인가" 에 답한다면, 본 단원의 접근통제는 "이 주체가 무엇을 할 수 있는가" 에 답한다. 두 단계는 보안 시스템의 직렬 단계로 — 인증을 통과한 주체에 대해서만 접근통제가 의미를 가진다.

본 단원은 4과목 출제기준 1.2 의 세 영역을 다룬다:
- **접근통제 정책의 이해 및 구성 요소** — §1
- **접근통제 정책의 특징 및 적용 범위 (DAC·MAC·RBAC·ABAC)** — §2·§3
- **접근통제 기법과 각 모델의 특징** (Bell-LaPadula·Biba·Clark-Wilson·Brewer-Nash) — §4·§5·§6

접근통제는 시험에서 두 가지 형태로 출제된다 — ① 정책 분류(DAC·MAC·RBAC) 의 차이를 묻거나, ② 보안 모델(BLP·Biba) 의 규칙(NRU·NWD·NRD·NWU) 을 묻는다. 이 두 영역의 정확한 구분이 본 단원의 학습 목표다.

[^nist-53]: NIST SP 800-53 Rev. 5, *Security and Privacy Controls for Information Systems and Organizations*, 2020 (Updated 2023). https://csrc.nist.gov/publications/detail/sp/800-53/rev-5/final

## §1. 접근통제 개요

### §1-1. 정의와 보안 목표

> 접근통제는 정보 보호의 세 핵심 목표 — **기밀성(Confidentiality)·무결성(Integrity)·가용성(Availability)** 의 CIA Triad — 중 주로 기밀성과 무결성을 강제하는 메커니즘이다.[^nist-53]

접근통제가 CIA 와 어떻게 연결되는지 분명히 해 둘 필요가 있다. 기밀성은 "허가된 주체만 객체를 읽을 수 있다" 는 요구이며 BLP 모델이 이를 강제한다(§4-1). 무결성은 "허가된 주체만 객체를 변경할 수 있다" 는 요구이며 Biba·Clark-Wilson 모델이 이를 강제한다(§4-2·§4-3). 가용성은 접근통제의 직접 목표는 아니지만 — DoS 공격 방어를 통해 간접적으로 보호된다.

### §1-2. 구성 요소

| 구성 요소 | 영문 | 설명 |
|---|---|---|
| 주체 | Subject | 객체에 접근하려는 능동적 개체 (사용자·프로세스·디바이스) |
| 객체 | Object | 보호 대상이 되는 수동적 자원 (파일·디렉터리·DB 레코드·네트워크 자원) |
| 접근 권한 | Access Right | 주체가 객체에 수행 가능한 행위 (읽기·쓰기·실행·삭제·변경) |
| 참조 모니터 | Reference Monitor | 모든 접근 요청을 가로채 정책에 따라 허용·거부 결정을 내리는 추상 머신 |
| TCB | Trusted Computing Base | 참조 모니터를 구현한 하드웨어·펌웨어·소프트웨어의 집합. 보안 정책 강제의 핵심 |

**참조 모니터의 3대 요건** (Anderson 1972 가 제시하고 TCSEC[^tcsec] 이 채택):
1. **격리성(Isolation)** — 변조될 수 없도록 보호되어야 한다
2. **완전성(Completeness)** — 모든 접근을 가로채야 하며 우회할 수 없어야 한다
3. **검증 가능성(Verifiability)** — 작고 단순해 분석·검증이 가능해야 한다

> *시험 포인트*: 참조 모니터의 3요건을 묻는 문제가 빈출. "격리성·완전성·검증 가능성" 또는 영문(tamperproof·complete·verifiable·non-bypassable·always invoked) 으로 출제된다.

### §1-3. 접근통제의 일반 원리

접근통제는 다음 4단계를 직렬로 수행한다:

1. **식별 (Identification)** — 주체가 자신을 누구인지 주장 (사용자명·UID·디바이스 ID)
2. **인증 (Authentication)** — 주장된 신원이 사실임을 검증 (01 단원 참조)
3. **인가 (Authorization)** — 인증된 주체가 요청한 객체·행위에 권한이 있는지 정책에 따라 결정
4. **책임 추적성 (Accountability)** — 모든 접근을 로그로 남겨 사후 추적·감사 가능하게 함

이 4단계 중 본 단원의 핵심은 **인가** 다. 인가는 "어떤 정책으로 결정할 것인가"(§2 — DAC·MAC·RBAC·ABAC) 와 "어떤 자료구조로 권한을 표현할 것인가"(§3 — ACL·Capability) 의 두 차원을 갖는다.

## §2. 접근통제 정책

### §2-1. DAC — 임의적 접근통제

> **DAC (Discretionary Access Control, 임의적 접근통제)**: 객체의 소유자가 다른 주체에게 자신의 객체에 대한 접근 권한을 임의로 부여할 수 있는 정책.[^nist-53]

DAC 의 본질은 **"임의적(discretionary)"** 이라는 단어에 있다 — 객체의 소유자가 자기 권한을 다른 주체에게 위임하는 결정을 자유롭게 내릴 수 있다. UNIX/Linux 의 파일 권한(`chmod`, `chown`), Windows NTFS 의 ACL 이 대표 사례다. 사용자가 자신의 파일에 대해 `chmod 644` 또는 `chmod 777` 을 자유롭게 결정할 수 있는 것은 DAC 의 정수다.

DAC 의 장점:
- 유연성 — 사용자 자율로 권한 관리
- 구현 단순 — UNIX 의 owner/group/other 모델

DAC 의 한계:
- **트로이 목마 취약성** — 악성 코드가 사용자 권한으로 실행되면 사용자 자신의 권한 한도 내에서 모든 것을 할 수 있다(파일 권한을 임의 변경, 다른 사용자에게 공유)
- **정보 흐름 제어 불가** — 한 번 다른 주체에게 권한을 부여하면 그 주체가 다시 제3자에게 재위임하는 것을 막을 수 없다(권한 전이 문제)

> *시험 포인트*: DAC 의 가장 큰 약점은 **트로이 목마**와 **정보 흐름 제어 불능**이다. MAC(§2-2) 가 등장한 이유가 바로 이 두 문제 해결이다.

### §2-2. MAC — 강제적 접근통제

> **MAC (Mandatory Access Control, 강제적 접근통제)**: 시스템(보안 관리자) 이 객체와 주체에 보안 등급을 부여하고, 등급 간 정해진 규칙에 따라 접근을 강제하는 정책. 객체 소유자라도 정책을 변경할 수 없다.[^nist-53]

MAC 의 본질은 **"강제적(mandatory)"** 이라는 단어에 있다 — 정책은 시스템 차원에서 결정되며 개별 사용자가 임의로 변경할 수 없다. 군·정부의 기밀 분류 체계가 대표 사례이며(Top Secret > Secret > Confidential > Unclassified), 이를 컴퓨터 시스템에 적용한 것이 BLP 모델(§4-1) 과 Biba 모델(§4-2) 이다.

MAC 의 핵심 원리:
- **객체와 주체에 보안 등급(레이블) 부여** — 예: 문서 = Top Secret, 사용자 = Secret
- **등급 간 규칙 강제** — 예: BLP 의 "Secret 사용자는 Top Secret 문서를 읽을 수 없다"
- **소유자가 변경 불가** — 사용자가 자신의 등급이나 객체의 등급을 임의로 변경할 수 없음
- **다중 레벨 보안 (Multi-Level Security, MLS)** — 서로 다른 등급의 정보가 동일 시스템에서 안전하게 공존

MAC 의 장점:
- **트로이 목마 방어** — 악성 코드가 사용자 권한으로 실행되어도 등급 규칙을 위반할 수 없음
- **정보 흐름 제어** — 등급 규칙이 정보 누설(downgrade) 방지

MAC 의 단점:
- **유연성 부족** — 모든 객체·주체에 등급 부여 필요, 운영 부담 큼
- **상업 환경 부적합** — 본래 군용 모델, 일반 기업 환경의 동적 권한 관리에는 비효율

대표 구현: SELinux (Linux), AppArmor (Linux), Trusted Solaris

> *시험 포인트*: DAC vs MAC 비교는 시험 단골. 핵심 차이는 ① 권한 결정 주체(소유자 vs 시스템), ② 변경 가능성(가능 vs 불가), ③ 트로이 목마 저항(약 vs 강).

### §2-3. RBAC — 역할 기반 접근통제

> **RBAC (Role-Based Access Control, 역할 기반 접근통제)**: 권한을 직접 사용자에게 부여하지 않고 **역할(role)** 에 부여한 후, 사용자에게 역할을 할당하는 정책. ANSI INCITS 359-2012 가 표준이다.[^ansi-rbac]

RBAC 의 발상은 — **사용자 ↔ 권한의 직접 매핑이 N×M 복잡도를 갖는 반면, 사용자 ↔ 역할 ↔ 권한의 간접 매핑은 N+M+R 복잡도로 줄어든다**는 데 있다. 100 명의 사용자가 50 종의 권한을 갖는 시스템은 직접 매핑 시 최대 5,000 개의 매핑을 관리해야 하지만, 10 개의 역할을 거치면 100+10+50×10 ≈ 600 개로 줄어든다.

RBAC 의 핵심 모델:
- **사용자 (User)** — 시스템 사용 주체
- **역할 (Role)** — 직무·책임에 따른 권한 묶음 (예: 관리자·운영자·감사자)
- **권한 (Permission)** — 객체에 대한 행위 (예: 파일 X 에 대한 읽기)
- **세션 (Session)** — 사용자가 활성화한 역할의 부분집합

RBAC 의 장점:
- **확장성** — 역할 정의·할당이 권한 직접 부여보다 간결
- **직무 분리(SoD) 자연스러움** — 동일 사용자에게 상충하는 역할(예: 결재자 + 결재 검증자) 부여 금지를 정책으로 강제 가능
- **상업 환경 적합** — 대규모 조직의 권한 관리에 표준화 가능

RBAC 표준은 §6 에서 본격 다룬다.

> *시험 포인트*: RBAC 의 핵심은 "권한 ↔ 역할" 매핑이지 "권한 ↔ 사용자" 매핑이 아니다. 이 차이가 RBAC 와 DAC 를 구분하는 핵심.

### §2-4. ABAC — 속성 기반 접근통제

> **ABAC (Attribute-Based Access Control, 속성 기반 접근통제)**: 주체·객체·환경의 속성(attribute) 에 대한 정책 표현식으로 접근을 결정하는 정책. NIST SP 800-162 가 표준화한다.[^nist-162]

ABAC 의 발상은 — RBAC 의 "역할" 만으로는 표현할 수 없는 동적·맥락적 정책(예: "근무 시간 중에만 + 회사 IP 에서만 + 정규직 사원만 접근 가능") 을 속성의 조합으로 표현한다는 것이다. ABAC 의 정책은 다음 4 가지 속성 카테고리의 조합으로 작성된다:

1. **주체 속성 (Subject Attributes)** — 부서·직급·정규직 여부·보안 등급
2. **객체 속성 (Object Attributes)** — 분류·소유자·생성일·민감도
3. **행위 속성 (Action Attributes)** — 읽기·쓰기·실행
4. **환경 속성 (Environment Attributes)** — 시간·위치·디바이스·네트워크

정책 예시: "주체.부서 = '인사팀' AND 객체.분류 = '인사기록' AND 환경.시간 ∈ [09:00, 18:00] AND 환경.위치 = '회사내부' → 허용"

ABAC 표준 구현: **XACML (eXtensible Access Control Markup Language)** — OASIS 표준, XML 기반 정책 언어.

| 정책 | 권한 결정 기준 | 대표 구현 |
|---|---|---|
| DAC | 객체 소유자의 임의 결정 | UNIX 파일 권한, NTFS ACL |
| MAC | 시스템 강제 보안 등급 | SELinux, BLP·Biba 모델 |
| RBAC | 사용자가 가진 역할 | 기업 IAM, AD 그룹 정책 |
| ABAC | 다차원 속성 조합 정책 | XACML, AWS IAM Policy |

> *시험 포인트*: ABAC 는 RBAC 의 "역할" 보다 더 세밀한 동적 정책을 표현 가능. 환경 속성(시간·위치) 을 정책에 포함할 수 있다는 점이 RBAC 와 가장 큰 차이.

[^nist-162]: NIST SP 800-162, *Guide to Attribute Based Access Control (ABAC) Definition and Considerations*, 2014 (Updated 2019). https://csrc.nist.gov/publications/detail/sp/800-162/final

## §3. 접근통제 메커니즘

### §3-1. 접근통제 행렬

> **접근통제 행렬 (Access Control Matrix)**: 행이 주체, 열이 객체이고 각 셀에 해당 주체-객체 쌍의 접근 권한이 기록된 자료구조. Lampson 1971 의 보호 모델에 기반한다.

접근통제 행렬은 개념적으로는 명료하지만 실제 시스템에서 그대로 구현되지 않는다 — 주체 N 명, 객체 M 개의 시스템에서 N×M 의 행렬은 대부분 빈 셀이며 메모리 비효율이 크다. 따라서 실제 구현은 행렬의 **열 방향 분해(ACL)** 또는 **행 방향 분해(Capability List)** 중 하나를 사용한다.

| | File1 | File2 | Printer1 |
|---|---|---|---|
| Alice | rw | r | x |
| Bob | r | rw | x |
| Carol | — | r | — |

### §3-2. ACL — Access Control List

> **ACL (Access Control List)**: 객체별로 "이 객체에 누가 어떤 권한을 가지는가" 를 나열한 목록. 접근통제 행렬의 **열 방향 분해**.

ACL 은 객체 입장에서 본 권한 목록이다. 위 행렬을 ACL 로 분해하면:
- File1 의 ACL: { (Alice, rw), (Bob, r) }
- File2 의 ACL: { (Alice, r), (Bob, rw), (Carol, r) }
- Printer1 의 ACL: { (Alice, x), (Bob, x) }

UNIX 파일 권한 (`-rw-r--r--`), Windows NTFS, 라우터 ACL, AWS S3 버킷 정책 등이 모두 ACL 기반이다.

ACL 의 장점:
- **객체별 권한 검토 용이** — 한 파일에 누가 접근 가능한지 바로 확인 가능
- **객체 단위 접근 제어 효율적** — 파일 시스템 같은 객체 중심 시스템에 자연스러움

ACL 의 단점:
- **주체별 권한 검토 어려움** — 특정 사용자가 어떤 객체에 접근 가능한지 알려면 모든 객체의 ACL 을 검색해야 함
- **사용자 삭제 시 부담** — 모든 객체의 ACL 에서 해당 사용자를 제거해야 함

### §3-3. Capability List

> **Capability List (능력 목록)**: 주체별로 "이 주체가 어떤 객체에 어떤 권한을 가지는가" 를 나열한 목록. 접근통제 행렬의 **행 방향 분해**.

Capability 는 주체 입장에서 본 권한 목록이다. 위 행렬을 Capability 로 분해하면:
- Alice 의 Capability: { (File1, rw), (File2, r), (Printer1, x) }
- Bob 의 Capability: { (File1, r), (File2, rw), (Printer1, x) }
- Carol 의 Capability: { (File2, r) }

Kerberos 의 티켓(01 단원 §5-2), Capability-based 운영체제(EROS·seL4), 분산 시스템의 토큰 기반 권한 관리가 Capability 모델이다.

Capability 의 장점:
- **주체별 권한 검토 용이** — 한 사용자가 어떤 권한을 가졌는지 바로 확인 가능
- **분산 시스템 친화적** — Capability 를 토큰화하여 네트워크로 전달 가능 (OAuth 2.0 의 Bearer Token 도 Capability 적 발상)

Capability 의 단점:
- **객체별 권한 검토 어려움** — 한 객체에 누가 접근 가능한지 알려면 모든 주체의 Capability 를 검색해야 함
- **권한 회수 어려움** — 한 번 발급한 Capability 를 모두 회수하려면 추적이 필요

### §3-4. ACL vs Capability 비교

| 측면 | ACL | Capability |
|---|---|---|
| 행렬 분해 방향 | 열 (객체별) | 행 (주체별) |
| "누가 X 에 접근 가능?" | 빠름 | 느림 |
| "주체 Y 의 모든 권한?" | 느림 | 빠름 |
| 권한 회수 | 쉬움 (객체 ACL 수정) | 어려움 (Capability 추적) |
| 권한 위임 | 어려움 (관리자 개입) | 쉬움 (Capability 전달) |
| 대표 구현 | UNIX·NTFS·라우터·S3 | Kerberos·OAuth Bearer·seL4 |

> *시험 포인트*: 접근통제 행렬을 객체 중심으로 보면 ACL, 주체 중심으로 보면 Capability. 이 둘은 같은 정보의 다른 표현이며 트레이드오프 관계.

## §4. 접근통제 모델

접근통제 모델은 §2 의 정책(DAC·MAC) 을 수학적으로 정형화한 것이다. 본 절의 4개 모델 중 BLP 와 Biba 는 단순 규칙 기반이고, Clark-Wilson 과 Brewer-Nash 는 응용 무결성·이해충돌 회피를 위한 더 풍부한 모델이다.

### §4-1. Bell-LaPadula Model — 기밀성 모델

> **Bell-LaPadula (BLP) Model**: 군·정부의 다중 레벨 보안(MLS) 환경에서 **기밀성(confidentiality)** 을 강제하는 모델. Bell & LaPadula 1973·1976 가 MITRE 에서 개발.[^blp]

BLP 의 두 핵심 규칙:
- **단순 보안 속성 (Simple Security Property)** — *No Read Up (NRU)* — 주체는 자신의 등급보다 **높은** 등급의 객체를 **읽을 수 없다**
- *** 속성 (Star Property, 별 속성)** — *No Write Down (NWD)* — 주체는 자신의 등급보다 **낮은** 등급의 객체에 **쓸 수 없다**

| 규칙 | 영문 | 의미 |
|---|---|---|
| 단순 보안 속성 | Simple Security Property (NRU) | 자기보다 높은 등급 객체 읽기 금지 |
| * 속성 | * Property (NWD) | 자기보다 낮은 등급 객체 쓰기 금지 |
| 임의 보안 속성 | Discretionary Security Property | 추가 DAC 정책으로 보완 가능 |

**왜 NWD 인가?** — Top Secret 사용자가 Secret 파일에 쓰기를 허용하면, 사용자가 의도적·무의식적으로 Top Secret 정보를 Secret 파일로 복사할 수 있다(downgrade). 이를 막는 것이 NWD 의 본질이다 — "정보는 위에서 아래로 흐를 수 없다" 라는 기밀성 원칙의 직접 구현.

**예시 (Top Secret > Secret > Confidential > Unclassified)**:
- Secret 사용자: Confidential·Unclassified 파일 읽기 OK, Top Secret 파일 읽기 ✗
- Secret 사용자: Top Secret 파일 쓰기 OK (★), Confidential·Unclassified 파일 쓰기 ✗
- (★ "Write Up" 은 NWD 와 모순되지 않음 — 자기보다 높은 등급에는 쓸 수 있다는 뜻 — 다만 실제 구현에서는 무결성 우려로 제한하는 경우가 많음)

> *시험 포인트*: BLP = 기밀성 + NRU + NWD. 정보 흐름은 "낮은 → 높은" 방향만 허용된다. "Write Up, Read Down" 으로 외워도 같은 의미.

### §4-2. Biba Model — 무결성 모델

> **Biba Integrity Model**: BLP 와 정반대 방향의 규칙으로 **무결성(integrity)** 을 강제하는 모델. Biba 1977 가 MITRE 에서 개발.[^biba]

Biba 의 두 핵심 규칙:
- **단순 무결성 속성 (Simple Integrity Property)** — *No Read Down (NRD)* — 주체는 자신의 등급보다 **낮은** 등급의 객체를 **읽을 수 없다**
- *** 무결성 속성 (Star Integrity Property)** — *No Write Up (NWU)* — 주체는 자신의 등급보다 **높은** 등급의 객체에 **쓸 수 없다**

| 규칙 | 영문 | 의미 |
|---|---|---|
| 단순 무결성 속성 | Simple Integrity Property (NRD) | 자기보다 낮은 등급 객체 읽기 금지 |
| * 무결성 속성 | * Integrity Property (NWU) | 자기보다 높은 등급 객체 쓰기 금지 |
| 호출 속성 | Invocation Property | 자기보다 높은 등급 주체 호출 금지 |

**왜 NRD 인가?** — 신뢰성 높은 (등급 높은) 주체가 신뢰성 낮은 (등급 낮은) 객체에서 정보를 가져오면, 신뢰 시스템에 오염된 정보가 유입된다. NRD 는 "낮은 신뢰성 정보가 높은 신뢰성 시스템에 침투할 수 없다" 를 강제한다.

**왜 NWU 인가?** — 등급 낮은 주체가 등급 높은 객체에 쓰면 신뢰 시스템에 오염을 주입할 수 있다. NWU 는 이를 막는다.

| 모델 | 보호 대상 | 읽기 규칙 | 쓰기 규칙 | 정보 흐름 방향 |
|---|---|---|---|---|
| BLP | 기밀성 | NRU (위로 못 읽음) | NWD (아래로 못 씀) | 낮은 → 높은 |
| Biba | 무결성 | NRD (아래로 못 읽음) | NWU (위로 못 씀) | 높은 → 낮은 |

> *시험 포인트*: Biba 는 BLP 의 반대. 시험에서 가장 자주 헷갈리는 것이 NRU/NWD/NRD/NWU 의 매핑. **BLP 는 위로 흐르고(read down·write up), Biba 는 아래로 흐른다(read up·write down)** 는 식으로 외울 것 — 그러나 "BLP 에서는 NRU(no read up)" 라는 표현이 시험에서 직접 출제되는 형태이므로 약어를 통째로 외우는 편이 안전하다.

### §4-3. Clark-Wilson Model — 상업적 무결성

> **Clark-Wilson Model**: 상업·금융 환경의 무결성을 위한 모델. Clark & Wilson 1987 (IEEE S&P) 가 회계 시스템의 부정 방지 원리에서 도출.[^cw]

Clark-Wilson 은 BLP·Biba 의 단순 등급 모델을 넘어 **잘 정의된 트랜잭션(WFT)** 과 **직무 분리(SoD)** 를 통해 무결성을 강제한다. 회계 분야의 "이중 입력 부기(double-entry bookkeeping)" 와 "직무 분리" 원리의 컴퓨터화로 볼 수 있다.

**핵심 구성 요소**:
- **CDI (Constrained Data Item)** — 무결성 제약을 받는 데이터 (예: 회계 장부)
- **UDI (Unconstrained Data Item)** — 외부 입력 등 무결성 검증 전 데이터
- **TP (Transformation Procedure)** — CDI 를 변경하는 유일한 합법 절차 (잘 정의된 트랜잭션)
- **IVP (Integrity Verification Procedure)** — CDI 의 무결성 상태를 검증하는 절차

**핵심 규칙**:
1. CDI 는 오직 TP 를 통해서만 변경된다 (직접 접근 금지)
2. TP 는 CDI 를 무결한 상태에서 무결한 상태로 변환한다 (보존)
3. UDI 는 TP 를 거쳐 검증·변환된 후에만 CDI 가 될 수 있다
4. **직무 분리** — TP 를 작성하는 자, 실행하는 자, 검증하는 자는 분리되어야 한다

> *시험 포인트*: Clark-Wilson 의 핵심 키워드는 ① CDI/UDI/TP/IVP, ② 잘 정의된 트랜잭션 (Well-Formed Transaction), ③ 직무 분리 (Separation of Duty). 상업적 무결성·회계 부정 방지가 출제 컨텍스트.

### §4-4. Brewer-Nash Model — Chinese Wall

> **Brewer-Nash Model (Chinese Wall)**: 컨설팅·금융 등에서 **이해충돌(conflict of interest) 회피** 를 위한 동적 접근통제 모델. Brewer & Nash 1989 (IEEE S&P) 가 영국 금융 규제(Chinese Wall 정책) 에서 도출.[^bn]

Brewer-Nash 의 발상은 — 컨설턴트가 한 회사의 정보를 본 후에는 그 경쟁사의 정보에 접근하면 이해충돌이 발생한다는 직관에서 출발한다. 이를 **"일단 한 회사를 본 후에는 그 경쟁사 정보를 자동으로 차단"** 하는 동적 정책으로 구현한다.

**모델 구조**:
- **객체 (Object)** — 개별 회사 정보 항목
- **회사 데이터셋 (Company Dataset, CD)** — 한 회사의 모든 객체
- **이해충돌 클래스 (Conflict of Interest Class, COI)** — 경쟁 관계의 회사 데이터셋들

**규칙**:
1. **단순 보안 규칙 (Simple Security Rule)** — 주체는 어떤 객체든 처음에는 접근 가능하나, 일단 한 CD 의 객체에 접근하면 같은 COI 클래스의 다른 CD 객체에는 접근 불가
2. *** 속성 규칙 (* Property Rule)** — 주체가 객체에 쓰려면, 그 객체와 같은 CD 의 객체만 읽었어야 함 (다른 CD 의 정보가 흘러들어갈 위험 차단)

**예시**: 컨설턴트가 "은행 A" 의 자료에 접근하면, "은행 B"·"은행 C" 의 자료(동일 COI 클래스 = 금융권) 에는 접근 차단된다. 그러나 "정유사 X" 의 자료(다른 COI 클래스) 에는 접근 가능하다.

> *시험 포인트*: Brewer-Nash 의 핵심은 **이해충돌 회피**, **COI 클래스**, **동적 접근통제**(접근 이력에 따라 권한이 변함). 컨설팅·금융 분야의 이중 고용 방지가 출제 컨텍스트.

### §4-5. Lattice Model·Information Flow Model

> **Lattice Model**: 부분 순서(partial order) 와 최소상한·최대하한이 정의된 격자(lattice) 구조에서 보안 등급을 표현하는 모델. Denning 1976 의 정보 흐름 모델에 기초.

BLP 와 Biba 가 단순 선형 등급(Top Secret > Secret > Confidential) 을 가정한다면, Lattice 는 **다차원 등급**을 자연스럽게 표현한다. 예: (Secret, {Project A, Project B}) > (Secret, {Project A}) — 등급은 같지만 카테고리가 더 많은 쪽이 상위.

**Information Flow Model**: 보안 등급 사이의 정보 흐름이 격자 구조의 부분 순서를 따라야 한다는 일반화된 모델. BLP 의 NWD 와 Biba 의 NWU 를 통합·일반화한다.

> *시험 포인트*: Lattice 와 Information Flow 는 BLP·Biba 의 일반화 모델로 시험에서 깊이 출제되지는 않지만, **다차원 보안 등급** 과 **격자 구조** 는 키워드로 등장할 수 있다.

## §5. 보안 등급과 평가 기준

### §5-1. TCSEC — Orange Book

> **TCSEC (Trusted Computer System Evaluation Criteria)**: 미국 국방부가 1983·1985 년 발표한 컴퓨터 보안 평가 기준. 책 표지 색상 때문에 "Orange Book" 으로 불린다.[^tcsec]

TCSEC 은 컴퓨터 시스템의 보안 수준을 4개 등급(D·C·B·A) 7개 단계로 분류했다:

| 등급 | 단계 | 명칭 | 핵심 요구사항 |
|---|---|---|---|
| D | D | Minimal Protection | 최소 보호 (실패 등급) |
| C | C1 | Discretionary Security Protection | DAC 기본 |
| | C2 | Controlled Access Protection | DAC + 감사·식별 |
| B | B1 | Labeled Security Protection | MAC + 보안 레이블 |
| | B2 | Structured Protection | MAC + 형식 보안 정책 |
| | B3 | Security Domains | MAC + 참조 모니터 강제 |
| A | A1 | Verified Design | 형식 검증된 설계 |

TCSEC 은 1990 년대에 ITSEC(유럽) 과 통합되어 현재의 **CC (Common Criteria, ISO/IEC 15408)** 로 대체되었다.[^cc]

### §5-2. Common Criteria — ISO/IEC 15408

> **Common Criteria for Information Technology Security Evaluation (CC)**: ISO/IEC 15408 표준. TCSEC·ITSEC 통합 후속 표준으로 1999 년 ISO 표준화.[^cc]

CC 는 **EAL (Evaluation Assurance Level) 1~7** 의 7단계 평가 등급을 정의한다:

| EAL | 명칭 |
|---|---|
| EAL1 | Functionally Tested |
| EAL2 | Structurally Tested |
| EAL3 | Methodically Tested and Checked |
| EAL4 | Methodically Designed, Tested, and Reviewed |
| EAL5 | Semiformally Designed and Tested |
| EAL6 | Semiformally Verified Design and Tested |
| EAL7 | Formally Verified Design and Tested |

대부분의 상용 OS·DBMS 는 EAL4 수준에서 인증된다. EAL7 은 형식 검증을 요구해 매우 제한적인 시스템(미사일 제어 등) 에만 적용된다.

### §5-3. 보안 등급 (군·정부)

미국 정부의 기밀 분류 체계는 BLP 모델의 직접 모티브가 되었다:

| 등급 | 영문 | 노출 시 영향 |
|---|---|---|
| Top Secret | TS | 국가 안보에 매우 심각한 손해 |
| Secret | S | 국가 안보에 심각한 손해 |
| Confidential | C | 국가 안보에 손해 |
| Unclassified | U | 비밀 분류 없음 |

> *시험 포인트*: TCSEC 의 B1 단계가 MAC 의 시작 (보안 레이블), C2 가 DAC 의 보강 (감사·식별). CC 의 EAL4 가 상용 OS 표준, EAL7 이 형식 검증 최고 단계. CC 는 ISO/IEC 15408 표준이라는 점도 시험 출제 가능.

[^cc]: ISO/IEC 15408-1:2022, *Information security, cybersecurity and privacy protection — Evaluation criteria for IT security — Part 1: Introduction and general model*. https://www.iso.org/standard/72891.html

## §6. NIST RBAC 표준 — ANSI INCITS 359-2012

> **NIST RBAC 표준**: NIST 가 1992 년 제안한 RBAC 모델이 2004 년 INCITS 359-2004 로 표준화되고 2012 년 개정된 표준. 4개 컴포넌트(Core·Hierarchical·Static SoD·Dynamic SoD) 로 구성된다.[^ansi-rbac]

### §6-1. Core RBAC

Core RBAC 은 RBAC 의 최소 필수 요건이다:
- **사용자(User) ↔ 역할(Role) 할당** — 사용자가 가질 수 있는 역할 정의
- **역할(Role) ↔ 권한(Permission) 할당** — 역할이 가지는 권한 정의
- **사용자 세션(Session)** — 사용자가 활성화한 역할의 부분집합. 세션 단위로 권한 행사

### §6-2. Hierarchical RBAC — 역할 계층

역할 사이에 **상속 관계** 를 도입한다. 상위 역할은 하위 역할의 모든 권한을 자동으로 보유.

예시: "관리자" > "팀장" > "팀원"
- 팀원 권한 = { 보고서 읽기 }
- 팀장 권한 = 팀원 권한 + { 보고서 작성, 결재 }
- 관리자 권한 = 팀장 권한 + { 권한 부여, 사용자 생성 }

### §6-3. Constrained RBAC — 직무 분리 (SoD)

**Static SoD (SSD)** — 사용자가 동시에 가질 수 없는 역할 쌍 정의 (영구적 분리)
- 예: "결재자" 와 "결재 검증자" 를 동일 사용자에게 할당 금지

**Dynamic SoD (DSD)** — 동일 세션에서 동시 활성화할 수 없는 역할 쌍 정의 (일시적 분리)
- 예: 사용자가 두 역할을 모두 가질 수 있으나 한 세션에서는 둘 중 하나만 활성화

> *시험 포인트*: NIST RBAC 4 컴포넌트 = Core + Hierarchical + Static SoD + Dynamic SoD. 직무 분리(SoD) 는 RBAC 의 핵심 개념이며 SSD vs DSD 차이가 시험에 출제될 수 있다.

## §7. 접근통제 운영 원칙

### §7-1. 책임 추적성 (Accountability)

> 시스템 내의 모든 행위가 누구에 의해 수행되었는지 사후에 추적 가능해야 한다는 원칙. ISO/IEC 27001:2022 의 핵심 통제 항목 중 하나.[^iso-27001]

책임 추적성을 위해 필수적인 요소:
- **고유 식별자** — 사용자마다 유일한 ID 부여, 공용 계정 금지
- **로깅** — 모든 접근·변경·삭제 행위 기록
- **로그 무결성** — 사후 조작이 불가능하도록 보호 (WORM 저장, 디지털서명)
- **시각 동기** — 로그 시각이 정확해야 사건 재구성 가능 (NTP 기반)

### §7-2. 최소 권한 (Principle of Least Privilege)

> 주체에게 업무 수행에 필요한 최소한의 권한만 부여한다는 원칙. Saltzer & Schroeder 1975 의 8 대 보안 설계 원칙 중 하나.

최소 권한의 구현:
- **권한 자동 회수** — 사용자 직무 변경·퇴직 시 권한 즉시 변경
- **임시 권한 (just-in-time access)** — 필요한 시점·시간에만 권한 부여
- **권한 분할** — 큰 권한을 여러 작은 권한으로 분할

### §7-3. 직무 분리 (Separation of Duties, SoD)

> 한 사용자가 단독으로 부정·실수가 발생할 수 있는 모든 단계를 수행하지 못하도록, 한 작업을 여러 단계로 분할하고 각 단계를 다른 사용자에게 할당하는 원칙.

직무 분리의 분류:
- **정적 직무 분리 (SSD)** — 영구적 분리 (한 사용자가 두 역할을 모두 가질 수 없음)
- **동적 직무 분리 (DSD)** — 일시적 분리 (한 세션에서 두 역할을 동시 활성화 불가)

대표 사례:
- 결재 요청자 ≠ 결재 승인자
- 코드 작성자 ≠ 코드 리뷰어
- 회계 입력자 ≠ 회계 검증자

[^iso-27001]: ISO/IEC 27001:2022, *Information security, cybersecurity and privacy protection — Information security management systems — Requirements*. https://www.iso.org/standard/27001

## §8. 단원 요약

1. **접근통제 4단계** — 식별 → 인증 → 인가 → 책임 추적성. 본 단원의 핵심은 **인가**.
2. **참조 모니터 3요건** — 격리성·완전성·검증 가능성 (tamperproof·complete·verifiable).
3. **4 정책의 결정 주체** — DAC: 객체 소유자 / MAC: 시스템 / RBAC: 역할 / ABAC: 다차원 속성.
4. **DAC 의 약점** — 트로이 목마 + 정보 흐름 제어 불가. MAC 이 등장한 이유.
5. **ACL vs Capability** — ACL 은 객체 중심(열 분해), Capability 는 주체 중심(행 분해).
6. **BLP** = 기밀성 + NRU(No Read Up) + NWD(No Write Down). 정보 흐름: 낮은 → 높은.
7. **Biba** = 무결성 + NRD(No Read Down) + NWU(No Write Up). 정보 흐름: 높은 → 낮은.
8. **Clark-Wilson** = 상업적 무결성 + CDI/UDI/TP/IVP + 잘 정의된 트랜잭션 + 직무 분리.
9. **Brewer-Nash (Chinese Wall)** = 이해충돌 회피 + COI 클래스 + 동적 접근통제.
10. **TCSEC**: D·C1·C2·B1·B2·B3·A1 의 7 단계. **CC (ISO/IEC 15408)**: EAL1~7. EAL4 가 상용 OS 표준.
11. **NIST RBAC** = Core + Hierarchical + Static SoD + Dynamic SoD.
12. **운영 3 원칙** — 책임 추적성·최소 권한·직무 분리.

## §9. 시험 함정 정리

| 함정 | 정답 |
|---|---|
| 참조 모니터의 3요건? | **격리성·완전성·검증 가능성** (tamperproof / complete / verifiable). |
| DAC 의 가장 큰 약점? | **트로이 목마**(사용자 권한 내에서 임의 행위) 와 **정보 흐름 제어 불가**(권한 전이 막을 수 없음). |
| BLP 의 두 규칙? | **NRU (No Read Up)** + **NWD (No Write Down)**. 기밀성. |
| Biba 의 두 규칙? | **NRD (No Read Down)** + **NWU (No Write Up)**. 무결성. |
| Clark-Wilson 의 핵심 4 요소? | **CDI / UDI / TP / IVP**. + 잘 정의된 트랜잭션 + 직무 분리. |
| Brewer-Nash 의 적용 분야? | **컨설팅·금융** 등 이해충돌이 발생할 수 있는 환경. COI 클래스 기반 동적 통제. |
| ACL 과 Capability 의 분해 방향? | ACL = **열(객체 중심)** / Capability = **행(주체 중심)**. |
| RBAC 의 권한 매핑 구조? | **사용자 ↔ 역할 ↔ 권한**. 역할이 중간 매개체. |
| ABAC 가 RBAC 보다 우월한 점? | **환경 속성**(시간·위치·디바이스) 을 정책에 포함 가능. 동적·맥락적 정책. |
| TCSEC 의 MAC 시작 등급? | **B1** (Labeled Security Protection). C2 까지는 DAC 기반. |
| CC 의 표준 번호와 EAL 범위? | **ISO/IEC 15408**, **EAL1 ~ EAL7**. EAL4 가 상용 OS 일반 인증. |
| NIST RBAC 의 4 컴포넌트? | Core / Hierarchical / **Static SoD** / **Dynamic SoD**. |
| Static SoD 와 Dynamic SoD 차이? | SSD = **영구적**(역할 자체 부여 금지), DSD = **세션 단위**(동시 활성화 금지). |
| MAC 모델의 대표 OS 구현? | **SELinux** (Linux), AppArmor, Trusted Solaris. |
