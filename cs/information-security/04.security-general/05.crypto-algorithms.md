# 05. 암호 알고리즘 (Cryptographic Algorithms)

## §0. 들어가며

> **암호학 (Cryptography)**: 비밀 정보를 보호하기 위해 평문을 암호문으로 변환하고 다시 복원하는 기술의 학문 분야.

본 단원은 4과목 출제기준 2.1 의 모든 세세항목을 다룬다:
- 암호 관련 용어·시스템 구성 — §1
- 암호 공격의 유형별 특징 — §3
- 대칭키 암호시스템 (종류·구조·운영 모드·공격) — §4·§5·§6
- 공개키 암호시스템 (특징·구조) — §7·§8
- 인수분해 기반·이산로그 기반 — §7·§8

암호학은 4과목에서 가장 분량이 크고 출제 빈도가 높은 영역이다. 본 단원의 학습 목표는 알고리즘의 수학적 깊이가 아니라 **각 알고리즘의 분류·구조·키 길이·라운드 수·운영 모드의 특징**을 정확히 구분하는 것이다.

## §1. 암호학 기초

### §1-1. 용어 정의

| 용어 | 영문 | 정의 |
|---|---|---|
| 평문 | Plaintext | 암호화되기 전의 원본 메시지 |
| 암호문 | Ciphertext | 암호화된 결과 |
| 키 | Key | 암호화·복호화에 사용되는 비밀 (또는 공개) 값 |
| 암호화 | Encryption | 평문 + 키 → 암호문 변환 |
| 복호화 | Decryption | 암호문 + 키 → 평문 복원 |
| 알고리즘 | Algorithm | 암호화·복호화 절차 (예: AES) |
| 프로토콜 | Protocol | 알고리즘을 활용한 통신 절차 (예: TLS) |
| 암호 분석 | Cryptanalysis | 암호 시스템의 약점을 찾는 연구 |

**암호 시스템 5 구성요소** (Shannon 1949 의 모델):
1. 평문 공간 (P) — 가능한 모든 평문의 집합
2. 암호문 공간 (C) — 가능한 모든 암호문의 집합
3. 키 공간 (K) — 가능한 모든 키의 집합
4. 암호화 함수 (E: P × K → C)
5. 복호화 함수 (D: C × K → P), D(E(p, k), k) = p

### §1-2. Kerckhoffs 원칙

> **Kerckhoffs 원칙 (Kerckhoffs's Principle)**: 1883 년 Auguste Kerckhoffs 가 제시한 6 원칙 중 가장 유명한 것 — "암호 시스템의 보안은 키의 비밀성에만 의존해야 하며, 알고리즘은 공개되어도 안전해야 한다".

이 원칙은 현대 암호학의 기본 가정이다. 알고리즘을 비밀로 두는 것 (security through obscurity) 은 다음 이유로 실패한다:
- **분석 부재** — 공개되지 않은 알고리즘은 학계의 검증을 받지 못해 약점이 묻힘
- **유출 시 재설계 비용** — 알고리즘이 한 번 노출되면 시스템 전체 교체 필요. 키만 비밀이면 키 교체로 해결
- **상호 운용성** — 표준 알고리즘이 있어야 다양한 시스템이 통신 가능

Kerckhoffs 원칙의 형식화는 Shannon 1949 의 "**적은 적이 시스템을 안다고 가정해야 한다**(the enemy knows the system)" 로 이어진다.

> *시험 포인트*: Kerckhoffs 원칙 = "알고리즘 공개, 키만 비밀". Shannon 의 형식화는 "적이 시스템을 안다고 가정". 현대 표준(AES·RSA·SHA) 이 모두 공개된 이유.

### §1-3. 암호 시스템 분류

암호 시스템은 다음 세 차원으로 분류된다:

**키 형태에 따른 분류**:
| 분류 | 영문 | 키 구조 | 대표 알고리즘 |
|---|---|---|---|
| 대칭키 | Symmetric | 양쪽이 동일한 키 공유 | AES·DES·SEED·RC4·ChaCha20 |
| 비대칭키 (공개키) | Asymmetric (Public Key) | 키쌍 (개인키·공개키) | RSA·ElGamal·ECC·DH |

**처리 단위에 따른 분류** (대칭키 한정):
| 분류 | 영문 | 처리 단위 | 대표 알고리즘 |
|---|---|---|---|
| 블록 암호 | Block Cipher | 고정 길이 블록 (64·128 bit) | DES·AES·SEED·ARIA |
| 스트림 암호 | Stream Cipher | 비트·바이트 단위 | RC4·ChaCha20·A5/1 |

**시대에 따른 분류**:
| 분류 | 영문 | 특징 | 대표 |
|---|---|---|---|
| 고전 암호 | Classical | 종이·기계 시대 | 시저·비제네르·Enigma |
| 현대 암호 | Modern | 컴퓨터 시대 | DES·AES·RSA |

**대칭키 vs 비대칭키 비교**:

| 측면 | 대칭키 | 비대칭키 |
|---|---|---|
| 키 관리 | N(N-1)/2 (KDC 사용 시 N) | N (각자 키쌍) |
| 속도 | 빠름 (HW 가속 시 GB/s) | 느림 (RSA 서명 ~수 ms) |
| 사용 용도 | 대량 데이터 암호화 | 키 교환·디지털서명·인증서 |
| 보안 토대 | 알고리즘 설계 | 수학적 난해성 (인수분해·이산로그) |
| 키 길이 (128 bit 보안) | 128 bit | RSA 3072 / ECC 256 |

> *시험 포인트*: 대칭키와 비대칭키의 본질적 차이는 ① 키 구조, ② 속도, ③ 키 길이. 실무에서는 둘을 결합 — **하이브리드 암호**(공개키로 세션 키 교환, 대칭키로 본 데이터 암호화) 가 표준.

## §2. 고전 암호

### §2-1. 시저 암호

> **시저 암호 (Caesar Cipher)**: 알파벳을 일정한 자릿수만큼 이동시키는 단순 치환 암호. 키 = 이동량 (1~25).

`HELLO` + 키 3 → `KHOOR`

키 공간이 26 (그중 25 가 유의미) 으로 매우 작아 무차별 대입에 즉시 무너진다. 또한 영어 문자 빈도(E·T·A·O 빈출) 분석으로 즉시 키를 추정할 수 있다.

### §2-2. 비제네르 암호

> **비제네르 암호 (Vigenère Cipher)**: 키워드를 반복 사용해 시저 암호의 다중 알파벳 변형을 적용하는 다중 치환 암호.

키워드 `KEY`, 평문 `HELLO` → 키 = `KEYKE`, 각 글자에 다른 시프트 적용.

비제네르는 19 세기까지 "해독 불가능한 암호" 로 여겨졌으나, 1863 년 Kasiski Test (반복되는 키 길이 추정) 와 빈도 분석 결합으로 해독 가능함이 밝혀졌다.

### §2-3. 일회용 패드 — Vernam Cipher

> **일회용 패드 (One-Time Pad, OTP)**: 메시지와 같은 길이의 진정한 랜덤 키를 한 번만 사용하여 XOR 하는 암호. Vernam 1917 가 제안.

OTP 의 보안성은 **정보론적 보안 (information-theoretic security)** 이라는 가장 강한 형태로 — Shannon 1949 가 증명한 바와 같이, 키가 진정 랜덤이고 메시지와 같은 길이며 한 번만 사용된다면 **무한한 계산 능력으로도 해독 불가능**하다.

OTP 의 비현실성:
- **키 길이 = 메시지 길이** — 키 분배 부담이 메시지 자체 분배와 같음
- **재사용 절대 금지** — 두 번째 사용 시 두 메시지의 XOR 노출
- **진정 랜덤 필요** — 의사 랜덤(PRNG) 으로는 정보론적 보안 불가

OTP 는 매우 제한된 환경(외교 핫라인 등) 에서만 실용적이며, 일반 환경에는 부적합하다.

> *시험 포인트*: 일회용 패드는 **정보론적으로 안전**한 유일한 실용 암호 (이론적). 현대 모든 실용 암호는 **계산론적으로만 안전** (충분한 계산 자원이 있으면 해독 가능하지만 현실적으로 불가능).

## §3. 암호 공격

### §3-1. 공격자 모델 — 입력 가용성에 따른 분류

| 공격 모델 | 영문 | 공격자가 가진 것 |
|---|---|---|
| 암호문 단독 | Ciphertext-Only Attack (COA) | 암호문만 |
| 기지 평문 | Known-Plaintext Attack (KPA) | 일부 (평문, 암호문) 쌍 |
| 선택 평문 | Chosen-Plaintext Attack (CPA) | 임의의 평문에 대한 암호문 받을 수 있음 |
| 선택 암호문 | Chosen-Ciphertext Attack (CCA) | 임의의 암호문에 대한 복호문 받을 수 있음 |
| 적응적 선택 암호문 | Adaptive CCA (CCA2) | CCA + 이전 응답을 보고 다음 질의 결정 가능 |

이 위계는 점점 강한 공격자 가정을 의미하며, 현대 암호 시스템은 일반적으로 가장 강한 **CCA2** 안전성을 목표로 한다. RSA-OAEP, AES-GCM 모두 CCA2 안전성이 증명된 표준이다.

> *시험 포인트*: 공격 모델의 위계 — COA < KPA < CPA < CCA < CCA2. 공격자에게 더 많은 입력을 허용할수록 강한 공격. 강한 공격에 안전한 시스템은 약한 공격에도 안전.

### §3-2. 차분 공격 (Differential Cryptanalysis)

> **차분 공격**: Biham & Shamir 1990 가 공개한 블록 암호 공격 기법. 평문 쌍의 차분(XOR) 이 암호문 쌍의 차분에 어떻게 전파되는지 통계적으로 분석.

차분 공격은 DES 의 S-box 설계가 차분 공격에 강하도록 미리 조정되어 있었다는 사실(IBM·NSA 가 1970 년대에 비공개로 알고 있었음) 로 유명하다. AES 의 S-box 도 차분 공격을 명시적으로 고려해 설계되었다.

### §3-3. 선형 공격 (Linear Cryptanalysis)

> **선형 공격**: Matsui 1993 가 제안한 블록 암호 공격 기법. 평문·암호문·키의 비트 사이의 선형 근사를 찾아 통계적으로 분석.

차분 공격이 DES 에 사용된 평문-암호문 쌍 수가 약 2^47 이라면, 선형 공격은 약 2^43 으로 더 적다.

### §3-4. 부채널 공격 (Side-Channel Attack)

> **부채널 공격**: 알고리즘의 수학적 약점이 아니라 구현의 물리적 부산물(전력·시간·전자기·소리) 에서 비밀 정보를 추출하는 공격.

| 부채널 | 설명 | 대표 사례 |
|---|---|---|
| 타이밍 공격 | 연산 시간 측정으로 키 추정 | Kocher 1996 — RSA 개인키 복원 |
| 전력 분석 | 전력 소모 패턴 분석 | DPA (Differential Power Analysis) |
| 전자기 분석 | 회로 방출 전자기파 분석 | TEMPEST (군용 차폐 표준) |
| 음향 분석 | CPU·키보드 소음 | Genkin 2014 — RSA 키 음향 추출 |
| 캐시 공격 | CPU 캐시 hit/miss 패턴 | Spectre·Meltdown (2018) |

**대응책**:
- **상수 시간 구현 (constant-time)** — 입력에 무관하게 같은 시간 소요. EdDSA 는 이를 명시적 설계 목표로 채택
- **블라인딩** — 무작위 값으로 입력을 변형해 부채널 누설 차단
- **물리적 차폐** — TEMPEST 표준

### §3-5. 무차별 대입과 생일 공격

> **무차별 대입 공격 (Brute Force)**: 키 공간 전체를 시도하는 공격. 키 길이 n bit 면 평균 2^(n-1) 회 시도.

| 키 길이 | 평균 시도 횟수 | 현재 시점에서의 의미 |
|---|---|---|
| 56 bit (DES) | 2^55 | 수 시간 (전용 HW) — 사실상 깨짐 |
| 80 bit | 2^79 | 수년 — Legacy 한계 |
| 128 bit (AES-128) | 2^127 | 사실상 불가능 |
| 256 bit (AES-256) | 2^255 | 사실상 불가능 (양자 컴퓨터에도 안전) |

**생일 공격 (Birthday Attack)**: 충돌 찾기 공격. n bit 해시에서 약 2^(n/2) 회 시도로 충돌 발견. 따라서 충돌 저항성 보안 = 해시 길이의 절반.

> *시험 포인트*: 무차별 대입은 키 길이가 결정. 생일 공격은 해시 길이의 절반(SHA-256 → 128 bit 충돌 저항). DES 의 56 bit 키는 깨짐 — 1998 년 EFF Deep Crack 머신이 공개적으로 깨뜨림.

## §4. 대칭키 블록 암호

### §4-1. DES — Data Encryption Standard

> **DES**: IBM 의 Lucifer 알고리즘을 변형해 NBS (현 NIST) 가 1977 년 표준화한 블록 암호. FIPS 46 의 후속인 FIPS 46-3 (1999) 까지 유지되었으나 2005 년 **표준 철회(Withdrawn)** 되었다.[^fips-46]

| 항목 | 값 |
|---|---|
| 블록 길이 | 64 bit |
| 키 길이 | 56 bit (+ 패리티 8 bit = 64 bit) |
| 라운드 수 | 16 |
| 구조 | Feistel |

**Feistel 구조**: 블록을 좌·우 절반으로 나누고, 매 라운드마다 한 쪽에 라운드 함수를 적용한 결과를 다른 쪽과 XOR. 16 라운드 후 좌·우 결합. **암호화·복호화가 같은 회로** 로 가능 (라운드 키 순서만 반대).

DES 의 폐기 사유:
- **56 bit 키 짧음** — 1998 년 EFF "Deep Crack" 이 22 시간 만에 키 복원
- **64 bit 블록 짧음** — 생일 경계로 32 GB 데이터에서 충돌 발생

### §4-2. 3DES — Triple DES

> **3DES (Triple DES, TDEA)**: DES 를 세 번 반복하여 키 길이를 늘린 변형. NIST SP 800-67 Rev. 2 가 표준화하나 **2023 년 12 월 31 일부로 사용 폐기 (deprecated for new applications)** 되었다.[^nist-67]

3DES 의 동작: `C = E_K3(D_K2(E_K1(P)))` — 첫 번째 키로 암호화, 두 번째 키로 복호화, 세 번째 키로 암호화.

| 항목 | 값 |
|---|---|
| 블록 길이 | 64 bit (DES 와 동일) |
| 키 길이 | 168 bit (실효 보안 ~112 bit, meet-in-the-middle 공격 고려) |
| 라운드 수 | 48 (DES 16 × 3) |

**왜 EDE 구조인가?** — `E_K3(D_K2(E_K1(P)))` 에서 K1=K2=K3 면 단순 DES 가 된다. 즉 **DES 호환성** 을 위해 EDE 구조가 채택되었다.

### §4-3. AES — Advanced Encryption Standard

> **AES**: NIST 가 2001 년 FIPS 197 로 표준화한 블록 암호. Rijndael (Daemen & Rijmen 1998) 이 5 년간의 공개 경쟁(AES Process) 끝에 선정되었다.[^fips-197]

| 항목 | 값 |
|---|---|
| 블록 길이 | 128 bit (16 byte) |
| 키 길이 | **128·192·256 bit** (3 변형) |
| 라운드 수 | **10·12·14** (각 키 길이에 대응) |
| 구조 | SPN (Substitution-Permutation Network) |

**SPN 구조의 매 라운드 4 단계**:
1. **SubBytes** — 8×8 S-box 로 바이트 치환 (비선형성)
2. **ShiftRows** — 행별로 다른 양만큼 시프트 (확산)
3. **MixColumns** — 열별로 GF(2^8) 행렬곱 (확산)
4. **AddRoundKey** — 라운드 키와 XOR

마지막 라운드는 MixColumns 생략. 10/12/14 라운드 = 키 길이 × 4 + 6 같은 직관적 관계.

**Feistel vs SPN**:
| 측면 | Feistel (DES·SEED·LEA) | SPN (AES·ARIA) |
|---|---|---|
| 암복호화 회로 | 같음 (라운드 키만 반대) | 다름 (역연산 필요) |
| 라운드 함수 비가역성 | 가능 | 불가능 (전체 가역 필요) |
| 병렬화 | 어려움 | 자연스러움 |

> *시험 포인트*: AES 의 3 가지 키 길이와 라운드 수 — **128/10, 192/12, 256/14**. SPN 구조. 블록 길이는 키 길이와 무관하게 항상 128 bit.

### §4-4. SEED — KISA 국산 블록 암호

> **SEED**: KISA 가 1998 년 개발한 국산 블록 암호. TTAS.KO-12.0004 가 한국 표준이며 RFC 4269 가 IETF 정보 표준.[^seed][^rfc-4269]

| 항목 | 값 |
|---|---|
| 블록 길이 | **128 bit** |
| 키 길이 | **128 bit** |
| 라운드 수 | **16** |
| 구조 | **Feistel** |

SEED 는 KCMVP 검증 모듈에 포함되며, 한국 정부·금융 시스템에서 광범위 사용된다. 다만 내부적으로 ARIA·LEA 등 더 새로운 국산 알고리즘이 등장하면서 SEED 의 신규 도입은 줄어드는 추세.

### §4-5. ARIA — Academy, Research Institute, Agency

> **ARIA**: 학계(Academy)·연구소(Research Institute)·정부(Agency) 의 협력으로 개발된 국산 블록 암호. KS X 1213-1 한국 표준 + RFC 5794 IETF 정보 표준.[^aria][^rfc-5794]

| 항목 | 값 |
|---|---|
| 블록 길이 | **128 bit** |
| 키 길이 | **128·192·256 bit** (AES 와 동일 변형) |
| 라운드 수 | **12·14·16** |
| 구조 | **SPN** (Involutional SPN — 암복호화에 같은 S-box 사용 가능) |

ARIA 가 AES 와 비슷한 매개변수를 가지면서도 별도 표준화된 이유는 ① 국산 암호 자립성, ② Involutional SPN 구조의 구현 효율성 차이.

### §4-6. LEA — Lightweight Encryption Algorithm

> **LEA**: 경량 환경 (IoT·모바일) 을 겨냥한 국산 블록 암호. KS X 3246 한국 표준.[^lea]

| 항목 | 값 |
|---|---|
| 블록 길이 | **128 bit** |
| 키 길이 | **128·192·256 bit** |
| 라운드 수 | **24·28·32** |
| 구조 | **ARX (Addition·Rotation·XOR)** — Feistel 의 경량 변형 |

ARX 구조의 장점: S-box 없이 단순 산술·논리 연산만 사용 → 32 bit MCU 에서도 빠른 구현 가능.

### §4-7. HIGHT — High security and lightweight

> **HIGHT**: 8-bit 마이크로프로세서 환경 (RFID·USN·스마트카드) 을 위한 초경량 블록 암호. TTAS.KO-12.0040 + KS X 1213-2 + ISO/IEC 18033-3.[^hight]

| 항목 | 값 |
|---|---|
| 블록 길이 | **64 bit** |
| 키 길이 | **128 bit** |
| 라운드 수 | **32** |
| 구조 | **Generalized Feistel (8 분할)** |

HIGHT 의 64 bit 블록은 일반 환경에서는 짧지만 RFID 같은 제약 환경에서는 트레이드오프 — 제약 환경에서 RAM·전력이 가장 큰 제약이므로 블록 길이를 양보하고 라운드 수로 보강.

### §4-8. 국산·국제 블록 암호 종합 비교

## §5. 블록 암호 운영 모드

> **운영 모드 (Mode of Operation)**: 블록 암호를 사용해 임의 길이 메시지를 처리하는 방식. NIST SP 800-38A·B·C·D·E·F·G 가 시리즈 표준이다.[^nist-38a]

블록 암호 자체는 한 블록 (예: AES 의 128 bit) 만 처리한다. 실제 메시지는 거의 항상 한 블록보다 길거나 짧으므로, **여러 블록을 어떻게 연결할 것인가**·**한 블록보다 짧은 끝부분을 어떻게 처리할 것인가** 의 문제가 운영 모드의 본질이다.

### §5-1. ECB — Electronic Codebook

> **ECB (Electronic Codebook)**: 각 블록을 독립적으로 암호화하는 가장 단순한 모드.

```
C_i = E_K(P_i)
```

**치명적 약점**: 같은 평문 블록은 항상 같은 암호문 블록 → 평문 패턴이 암호문에 그대로 노출.

유명한 시각적 예시: 펭귄 이미지를 ECB 로 암호화하면 윤곽이 그대로 보인다. 이로 인해 ECB 는 **사실상 사용 금지** — NIST SP 800-38A 도 일반 암호화에 권장하지 않는다.

### §5-2. CBC — Cipher Block Chaining

> **CBC (Cipher Block Chaining)**: 직전 암호문 블록과 현재 평문 블록을 XOR 후 암호화하는 모드.

```
C_0 = IV
C_i = E_K(P_i XOR C_(i-1))
```

**IV (Initialization Vector)** 가 필수이며 **무작위·예측 불가** 해야 한다 (예측 가능한 IV 사용 시 BEAST 공격, TLS 1.0 의 약점).

CBC 의 약점:
- **순차 처리** — 병렬화 불가 (이전 블록 암호화 완료 후 다음 블록 시작)
- **패딩 오라클 공격** — POODLE (2014), Lucky 13 (2013) 의 표적

### §5-3. CFB — Cipher Feedback

> **CFB (Cipher Feedback)**: 블록 암호를 스트림 암호처럼 사용하는 모드. 직전 암호문을 키로 가공해 평문과 XOR.

```
C_0 = IV
C_i = P_i XOR E_K(C_(i-1))
```

CFB 는 평문이 블록 길이보다 짧을 때(예: 1 byte 단위 통신) 유용하다.

### §5-4. OFB — Output Feedback

> **OFB (Output Feedback)**: CFB 와 비슷하지만 피드백 입력으로 직전 **출력** (블록 암호의 출력) 을 사용.

```
O_0 = IV
O_i = E_K(O_(i-1))
C_i = P_i XOR O_i
```

OFB 의 키 스트림은 평문과 무관하므로 사전 계산 가능 → 실시간 통신에 유리.

### §5-5. CTR — Counter

> **CTR (Counter)**: 카운터 값을 암호화한 결과를 키 스트림으로 사용. NIST SP 800-38A 의 권장 모드.

```
C_i = P_i XOR E_K(Nonce || i)
```

CTR 의 큰 장점:
- **완전 병렬화** — 각 블록 독립 처리
- **임의 위치 접근** — i 번째 블록만 복호화 가능
- **블록 정렬 불필요** — 평문 길이 임의

CTR 은 GCM·CCM 인증 암호의 토대이며, AES-CTR·AES-GCM 이 현대 표준에서 가장 많이 사용된다.

### §5-6. 인증 암호 — GCM

> **GCM (Galois/Counter Mode)**: CTR 모드의 암호화 + GMAC 의 인증을 결합한 인증 암호 (Authenticated Encryption). NIST SP 800-38D 가 표준화한다.[^nist-38d]

GCM 은 다음을 한 번에 제공한다:
- **기밀성** — CTR 암호화
- **무결성·인증** — GMAC (Galois MAC) 으로 데이터 + 부가 데이터 (AAD) 인증

GCM 의 출력 = (암호문, 인증 태그). 인증 태그가 검증 실패하면 평문 거부.

| 모드 | 인증 |
|---|---|
| CTR | ❌ |
| GCM = CTR + GMAC | ✅ |
| CCM = CTR + CBC-MAC | ✅ |

**TLS 1.3** 은 모든 암호 알고리즘을 인증 암호로만 허용 — AES-GCM, AES-CCM, ChaCha20-Poly1305 만 cipher suite 로 정의된다.

### §5-7. 인증 암호 — CCM

> **CCM (Counter with CBC-MAC)**: CTR 암호화 + CBC-MAC 인증. NIST SP 800-38C 가 표준화. WPA2, IEEE 802.15.4 (ZigBee) 에서 사용.[^nist-38c]

CCM 은 GCM 보다 단순하지만 직렬 처리(CBC-MAC) 부분이 있어 GCM 보다 느리다. GCM 이 일반적으로 선호되지만, 임베디드 환경에서는 CCM 이 단순성으로 채택되기도 한다.

### §5-8. XTS — 디스크 암호

> **XTS (XEX-based Tweaked-codebook with ciphertext Stealing)**: 디스크 섹터 단위 암호화에 특화된 모드. NIST SP 800-38E 가 표준화. BitLocker, dm-crypt LUKS, FileVault 에서 사용.[^nist-38e]

디스크 암호의 특수 요구:
- 섹터 단위 임의 위치 접근 (CTR 처럼) — 필요
- 인증 태그 추가 공간 없음 (섹터 크기 고정) — GCM 같은 인증 암호 사용 불가
- 같은 섹터의 다른 시점 암호화는 다른 결과여야 함 — Tweak 사용

XTS 는 인증을 제공하지 않으므로 일반 데이터 암호화에는 부적합하다 — **디스크 전용**.

### §5-9. KW — Key Wrapping

> **KW (Key Wrap)**: 한 키로 다른 키를 안전하게 암호화하는 모드. NIST SP 800-38F 가 표준화.[^nist-38f]

키 자체를 평문 데이터 처럼 일반 모드(CBC·CTR) 로 암호화하면 IV·MAC 공간 등이 추가로 필요해 효율적이지 않다. KW 는 키 길이를 그대로 유지하면서 인증을 함께 제공하는 특화 모드.

활용처: HSM 에서 마스터 키로 다른 키를 암호화해 외부 백업, KMS 의 envelope encryption.

### §5-10. 운영 모드 종합 비교

| 모드 | 표준 | 병렬화 | IV 필요 | 인증 | 주요 용도 |
|---|---|---|---|---|---|
| ECB | SP 800-38A | ✅ | ❌ | ❌ | (사용 금지) |
| CBC | SP 800-38A | 복호만 | ✅ | ❌ | TLS 1.0/1.1 (폐기), legacy |
| CFB | SP 800-38A | 복호만 | ✅ | ❌ | 스트림 모방 |
| OFB | SP 800-38A | ✅ (사전 계산) | ✅ | ❌ | 사전 계산 키 스트림 |
| CTR | SP 800-38A | ✅ | ✅ (Nonce) | ❌ | 일반 암호화 (인증 없음) |
| **GCM** | SP 800-38D | ✅ | ✅ | ✅ | **TLS 1.3 권장**, 일반 인증 암호 |
| CCM | SP 800-38C | 부분 | ✅ | ✅ | WPA2, IoT |
| XTS | SP 800-38E | ✅ | (Tweak) | ❌ | 디스크 암호 |
| KW | SP 800-38F | — | (불필요) | ✅ | 키 래핑 |

> *시험 포인트*: ECB 는 사용 금지(평문 패턴 노출), CBC 는 패딩 오라클 약점, CTR 는 인증 없음, GCM·CCM 이 인증 암호. TLS 1.3 은 GCM·CCM·ChaCha20-Poly1305 만 허용.

## §6. 스트림 암호

### §6-1. RC4 — Rivest Cipher 4

> **RC4**: Ron Rivest 가 1987 년 RSA Security 를 위해 설계한 스트림 암호. 1990 년대~2010 년대 SSL/TLS·WEP 에서 광범위 사용되었으나 다중 약점으로 **2015 년 RFC 7465 가 TLS 에서의 사용을 금지**했다.[^rfc-7465]

RC4 의 약점:
- **초기 키 스트림 편향** — 처음 256 byte 가 비균일 (FMS 공격, WEP 키 복원)
- **RC4 NOMORE 공격 (2015)** — TLS 의 RC4 cipher 에서 쿠키 복원 가능성 시연

### §6-2. ChaCha20 — RFC 8439

> **ChaCha20**: Bernstein 2008 가 설계한 스트림 암호. RFC 8439 가 ChaCha20-Poly1305 인증 암호 형태로 표준화.[^rfc-8439]

| 항목 | 값 |
|---|---|
| 키 길이 | **256 bit** |
| Nonce 길이 | 96 bit |
| 블록 카운터 | 32 bit |
| 출력 | 512 bit 키 스트림 블록 |

ChaCha20 의 장점:
- **소프트웨어 고성능** — AES-NI HW 가속이 없는 환경 (모바일·IoT) 에서 AES-GCM 보다 빠름
- **부채널 저항** — 상수 시간 구현 자연스러움 (Curve25519 와 같은 설계 철학)
- **TLS 1.3 표준 cipher suite** — `TLS_CHACHA20_POLY1305_SHA256`

ChaCha20-Poly1305 = ChaCha20 (스트림 암호) + Poly1305 (MAC). GCM 의 ChaCha20 버전.

> *시험 포인트*: RC4 는 사용 금지 (RFC 7465). ChaCha20 은 RFC 8439, TLS 1.3 cipher. AES 가 HW 가속 없는 환경에서는 ChaCha20 이 더 빠를 수 있다.

[^rfc-7465]: RFC 7465, *Prohibiting RC4 Cipher Suites*, Popov, 2015. https://www.rfc-editor.org/rfc/rfc7465
[^rfc-8439]: RFC 8439, *ChaCha20 and Poly1305 for IETF Protocols*, Nir & Langley, 2018. https://www.rfc-editor.org/rfc/rfc8439

## §7. 공개키 암호 — 인수분해 기반

### §7-1. RSA 알고리즘

> **RSA**: Rivest·Shamir·Adleman 1978 가 발표한 최초의 실용적 공개키 암호. **소인수 분해 문제의 어려움** (FACT 가정) 에 보안을 의존.[^rsa-1978]

**키 생성**:
```
1. 큰 소수 p, q 선택 (각 1024 bit 이상)
2. n = p · q  (모듈러스)
3. φ(n) = (p−1)(q−1)  (오일러 토션트)
4. 1 < e < φ(n) 이고 gcd(e, φ(n)) = 1 인 e 선택  (보통 65537)
5. d = e^(−1) mod φ(n)  (확장 유클리드)
6. 공개키 = (n, e), 개인키 = (n, d)
```

**암호화·복호화**:
```
암호화: c = m^e mod n
복호화: m = c^d mod n
```

**RSA 의 정확성**: m^(ed) = m^(1 mod φ(n)) = m mod n  (페르마-오일러 정리)

**보안 가정**: 큰 합성수 n = pq 의 인수분해는 어렵다(FACT 가정). 양자 컴퓨터의 Shor 알고리즘이 RSA 를 다항 시간에 깨므로 양자 후 암호 (PQC) 로의 전환이 진행 중.

### §7-2. RSA-OAEP — Optimal Asymmetric Encryption Padding

> **RSA-OAEP**: 단순 RSA 암호의 안전성 약점을 보완하기 위한 패딩 방식. RFC 8017 (PKCS#1 v2.2) 가 표준.[^rfc-8017]

단순 RSA 의 약점:
- **결정론적** — 같은 평문 → 같은 암호문 (사전 공격 가능)
- **수학적 구조 노출** — 단순 RSA 는 IND-CCA 안전하지 않음

OAEP 패딩은 다음을 추가:
1. 무작위 시드 (randomness) 로 결정론성 제거
2. 마스크 생성 함수 (MGF, 보통 SHA-256 기반) 로 평문과 시드 혼합
3. CCA2 안전성 증명 (Bellare-Rogaway 1994, Fujisaki-Okamoto 2001 의 random oracle 모델)

### §7-3. Rabin 암호 — Squaring-based

> **Rabin 암호**: Rabin 1979 가 발표. RSA 와 같은 인수분해 기반이지만 **인수분해와 동등하게 어려움이 증명**된 암호 (RSA 는 인수분해보다 쉬울 수도 있음).

**키 생성·암호화·복호화**:
```
키: n = pq (p, q 는 4k+3 형태 소수)
암호화: c = m^2 mod n
복호화: 4 개 제곱근 중 m 추정 (잉여 정보 필요)
```

Rabin 의 한 블록 암호화당 4 개의 가능한 평문 → 추가 식별 정보 필요. 이로 인해 실용 시스템에서는 거의 사용되지 않으며 학술적 의의가 크다.

[^rfc-8017]: RFC 8017, *PKCS #1: RSA Cryptography Specifications Version 2.2*, Moriarty et al., 2016. https://www.rfc-editor.org/rfc/rfc8017

## §8. 공개키 암호 — 이산로그 기반

### §8-1. ElGamal 암호

> **ElGamal 암호**: ElGamal 1985 가 발표한 공개키 암호. **이산로그 문제의 어려움** (DL 가정) 에 보안 의존. DSA 의 토대.[^elgamal-1985]

**키 생성**:
```
공개 매개변수: 큰 소수 p, 생성자 g
개인 키: x  (1 < x < p−1)
공개 키: y = g^x mod p
```

**암호화·복호화**:
```
암호화 (Bob → Alice):
  k ← random
  c_1 = g^k mod p
  c_2 = m · y^k mod p
  c = (c_1, c_2)

복호화 (Alice):
  s = c_1^x mod p  = g^(kx) mod p
  m = c_2 / s mod p
```

ElGamal 의 단점: 암호문이 평문의 **2 배 길이**(c_1, c_2 두 값). 그러나 보안 강점은 RSA 와 동등하며, 양자 후에는 RSA 와 함께 깨진다(이산로그도 Shor 로 깨짐).

### §8-2. ECC — 타원곡선 암호

> **ECC (Elliptic Curve Cryptography)**: 타원곡선 위의 점 군 연산을 활용한 공개키 암호 체계. Miller·Koblitz 1985 가 동시에 제안.

**타원곡선의 일반 형태**:
```
y² = x³ + ax + b  (mod p)
```

타원곡선 점들 사이에 정의된 점 덧셈 연산 (chord-and-tangent rule) 이 군 구조를 이루며, 이 군에서의 이산로그 문제 (ECDLP) 가 ECC 의 보안 토대다.

**ECC vs RSA·DH 비교** (NIST SP 800-57 Part 1 Rev. 5):

| 보안 강도 (대칭키 기준) | RSA / DH | ECC |
|---|---|---|
| 80 bit | 1024 | 160 |
| 112 bit | 2048 | 224 |
| **128 bit** | **3072** | **256** |
| 192 bit | 7680 | 384 |
| 256 bit | 15360 | 512 |

같은 보안 강도에서 ECC 키가 약 1/10 ~ 1/30 짧아 — IoT·모바일에서 ECC 가 표준이 되었다.

### §8-3. 권장 타원곡선

| 곡선 | 표준 | 보안 강도 | 용도 |
|---|---|---|---|
| **NIST P-256** (secp256r1) | FIPS 186-5 | 128 bit | TLS, 일반 |
| **NIST P-384** | FIPS 186-5 | 192 bit | 정부 고보안 |
| **NIST P-521** | FIPS 186-5 | 256 bit | 최고 보안 |
| **secp256k1** | SEC 2 | 128 bit | Bitcoin, Ethereum |
| **Curve25519** | RFC 7748 | 128 bit | TLS 1.3, Signal, SSH |
| **Curve448** | RFC 7748 | 224 bit | 고보안 키 합의 |

**Curve25519 와 Ed25519 의 관계**: Curve25519 는 키 합의 (X25519) 에 사용되는 곡선의 Montgomery 형태, Ed25519 는 같은 곡선의 Edwards 형태로 디지털서명 (EdDSA) 에 사용. 같은 수학적 구조의 두 표현.

> *시험 포인트*: ECC 의 핵심 장점은 **짧은 키**. 128 bit 보안 = ECC 256 bit = RSA 3072 bit. 권장 곡선 (P-256, secp256k1, Curve25519) 의 응용 분야 차이.

### §8-4. 공개키 암호 종합 비교

| 알고리즘 | 보안 토대 | 키 길이 (128 bit 보안) | 양자 후 안전 |
|---|---|---|---|
| RSA | 인수분해 (FACT) | 3072 | ❌ (Shor) |
| Rabin | 인수분해 (FACT, 동등 증명) | 3072 | ❌ |
| ElGamal | 이산로그 (DL) | 3072 | ❌ (Shor) |
| ECC (ECDH·ECDSA) | 타원곡선 이산로그 (ECDLP) | 256 | ❌ (Shor) |
| 격자 기반 (Kyber·Dilithium) | LWE·SIS 문제 | (NIST PQC 표준화 중) | ✅ |

## §9. 단원 요약

1. **Kerckhoffs 원칙** — 알고리즘 공개, 키만 비밀.
2. **대칭키 vs 비대칭키** — 대칭은 N(N-1)/2 키, 빠름. 비대칭은 N 키쌍, 느림. 하이브리드가 표준.
3. **공격자 모델 위계** — COA < KPA < CPA < CCA < CCA2.
4. **부채널 공격** — 타이밍·전력·전자기·음향·캐시. 상수 시간 구현이 대응.
5. **DES** — 64 bit 블록, 56 bit 키, 16 라운드, Feistel. **2005 폐기**.
6. **3DES** — 168 bit 키, 64 bit 블록, **2023 폐기**.
7. **AES** — 128 bit 블록, **128/192/256 bit 키 = 10/12/14 라운드**, SPN.
8. **국산 암호** — SEED (128/128/16/Feistel), ARIA (128/128~256/12~16/SPN), LEA (128/128~256/24~32/ARX), HIGHT (64/128/32/Feistel).
9. **운영 모드** — ECB ❌ / CBC (패딩 오라클) / CTR (인증 없음) / **GCM·CCM** ✅ 인증 암호 / XTS (디스크) / KW (키 래핑).
10. **TLS 1.3 cipher** — AES-GCM, AES-CCM, ChaCha20-Poly1305 (모두 인증 암호).
11. **RC4 폐기** — RFC 7465 (2015). **ChaCha20-Poly1305** — RFC 8439, TLS 1.3.
12. **RSA** — 인수분해 기반. RSA-OAEP 가 CCA2 안전한 표준 패딩.
13. **ECC** — 타원곡선 이산로그. **256 bit ECC = 3072 bit RSA** (128 bit 보안).
14. **Curve25519 / Ed25519** — 같은 곡선의 Montgomery / Edwards 형태. X25519 키 합의, Ed25519 서명.

## §10. 시험 함정 정리

| 함정 | 정답 |
|---|---|
| Kerckhoffs 원칙? | **알고리즘 공개, 키만 비밀**. Shannon 의 "적이 시스템을 안다" 와 동등. |
| AES 의 키 길이와 라운드 수? | **128/10, 192/12, 256/14**. 블록은 항상 128 bit. |
| AES 의 구조? | **SPN** (Substitution-Permutation Network). DES 의 Feistel 과 다름. |
| DES 의 키 길이와 라운드 수? | **56 bit (+ 패리티 8 bit) / 16 라운드 / 64 bit 블록 / Feistel**. |
| SEED 의 매개변수? | **128 bit 블록, 128 bit 키, 16 라운드, Feistel**. |
| ARIA 의 매개변수? | **128 bit 블록, 128/192/256 bit 키, 12/14/16 라운드, SPN**. |
| LEA 의 매개변수? | **128 bit 블록, 128/192/256 bit 키, 24/28/32 라운드, ARX**. |
| HIGHT 의 매개변수? | **64 bit 블록, 128 bit 키, 32 라운드, Generalized Feistel**. |
| Feistel 의 장점? | 암복호화 회로 동일 (라운드 키 순서만 반대). |
| ECB 모드의 약점? | **같은 평문 = 같은 암호문** → 평문 패턴 노출. 사용 금지. |
| CBC 모드의 약점? | **패딩 오라클 공격** (POODLE, Lucky 13). 순차 처리 (병렬화 불가). |
| CTR 모드의 약점? | **인증 없음**. 같은 nonce 재사용 시 키 스트림 노출. |
| 인증 암호 (Authenticated Encryption)? | **GCM, CCM, ChaCha20-Poly1305**. 기밀성 + 무결성·인증 동시 제공. |
| TLS 1.3 cipher suite? | AES-GCM, AES-CCM, **ChaCha20-Poly1305**. 모두 인증 암호. |
| 양자 후 RSA·ECC 안전성? | **❌ Shor 알고리즘** 으로 다항 시간에 깨짐. PQC 전환 중. |
| 128 bit 보안 강도의 키 길이? | **AES 128 / RSA 3072 / ECC 256**. NIST SP 800-57. |
| 공격자 모델 위계? | COA < KPA < **CPA < CCA < CCA2**. 강한 모델에 안전 → 약한 모델에도 안전. |
| 차분 공격 vs 선형 공격? | 차분 = **평문 차분이 암호문 차분에 미치는 영향** (Biham-Shamir). 선형 = **선형 근사** (Matsui). |
| 생일 공격의 보안 한계? | n bit 해시 → **n/2 bit 충돌 저항성**. SHA-256 → 128 bit. |
| RSA-OAEP 가 단순 RSA 보다 우월한 점? | **확률적**(무작위 시드) + **CCA2 안전성 증명**. |
