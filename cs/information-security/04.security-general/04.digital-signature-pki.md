# 04. 디지털서명과 PKI (Digital Signatures and PKI)

## §0. 들어가며

> **디지털서명 (Digital Signature)**: 송신자가 자신의 개인키로 메시지를 처리한 결과로, 수신자가 송신자의 공개키로 검증할 수 있는 데이터. 메시지 무결성·출처 인증·**부인 방지(non-repudiation)** 를 제공한다.[^fips-186]

> **PKI (Public Key Infrastructure)**: 공개키와 그 소유자의 신원을 결합한 디지털 인증서를 발급·관리·검증하는 신뢰 인프라. RFC 5280 이 인터넷 PKI 의 X.509 v3 인증서·CRL 프로파일을 정의한다.[^rfc-5280]

## §1. 디지털서명 개요

### §1-1. 보안 요구 조건

디지털서명은 종이 문서의 친필 서명과 인감 도장을 디지털 환경에 옮긴 것으로 볼 수 있지만, 보장하는 보안 속성은 더 강하다. 디지털서명이 만족해야 하는 5 가지 요구조건은 다음과 같다:

| 요구조건 | 영문 | 의미 |
|---|---|---|
| 위조 불가 | Unforgeable | 서명자의 개인키 없이는 정당한 서명을 만들 수 없음 |
| 사용자 인증 | Authentic | 서명을 통해 서명자의 신원 검증 가능 |
| 부인 방지 | Non-repudiation | 서명자가 사후에 자신의 서명을 부인할 수 없음 |
| 변경 불가 | Unalterable | 서명 후 메시지가 변경되면 검증이 실패함 |
| 재사용 불가 | Non-reusable | 한 메시지의 서명을 다른 메시지에 재사용 불가 |

이 5 가지 요구조건이 모두 만족되어야 디지털서명이 종이 서명을 대체할 법적·실무적 효력을 갖는다.

### §1-2. 디지털서명의 일반 구조

디지털서명은 다음 두 단계로 구성된다:

**서명 생성 (Signing)**:
```
1. 메시지 M 의 해시 h = H(M) 계산
2. 송신자의 개인키 sk 로 서명 σ = Sign(sk, h)
3. (M, σ) 전송
```

**서명 검증 (Verification)**:
```
1. 수신한 메시지 M' 의 해시 h' = H(M') 계산
2. 송신자의 공개키 pk 로 검증 Verify(pk, h', σ) → True/False
3. True → M' = M 이면서 송신자가 서명자임이 검증
```

**왜 메시지 자체가 아닌 해시를 서명하는가?** — RSA 와 같은 공개키 알고리즘은 모듈러스 크기로 입력이 제한되며 큰 메시지를 직접 처리할 수 없다. 또한 같은 알고리즘으로 다양한 길이의 메시지를 처리하려면 고정 길이로의 변환이 필요하다. 해시함수가 이 두 문제를 동시에 해결한다(임의 길이 → 고정 길이, 일방향성 + 충돌 저항성).

### §1-3. 디지털서명 vs MAC vs 해시

| 메커니즘 | 키 구조 | 무결성 | 출처 인증 | 부인 방지 | 검증자 |
|---|---|---|---|---|---|
| 해시 (SHA-256) | 키 없음 | O | X | X | 누구나 |
| MAC (HMAC·CMAC) | 대칭키 (양쪽 공유) | O | O (제한) | **X** | 키 보유자만 |
| 디지털서명 (RSA·ECDSA) | 비대칭 키쌍 | O | O | **O** | 누구나 (공개키) |

[01 단원 §6-2](./01.authentication.md) 에서 다룬 비교의 본질을 다시 정리하면 — **부인 방지는 검증자가 만들 수 없는 증거가 있어야 성립**하며, 이는 비대칭 키쌍의 개인키만이 만들 수 있는 디지털서명에서만 성립한다. MAC 은 검증자(수신자) 도 같은 키를 가지고 있어 자신이 만든 것인지 송신자가 만든 것인지 제3자가 판별할 수 없다.

> *시험 포인트*: 부인 방지는 **디지털서명만 제공**. MAC 은 출처 인증까지만(공유 키 보유자 둘 중 하나가 보냈음). 해시는 무결성만.

## §2. 디지털서명 알고리즘

FIPS 186-5 (2023) 가 미국 정부 표준으로 인정하는 디지털서명 알고리즘은 RSA·DSA·ECDSA·EdDSA 4 가지다. 본 절은 이 4 가지를 다룬다.[^fips-186]

### §2-1. RSA 서명 — PKCS#1 v1.5 와 RSA-PSS

> **RSA 디지털서명**: RSA 공개키 암호의 역방향 사용 — 개인키로 서명을 만들고 공개키로 검증. RFC 8017 (PKCS#1 v2.2) 가 두 패딩 방식 (v1.5·PSS) 을 정의한다.[^rfc-8017]

**RSASSA-PKCS1-v1_5** (RFC 8017 §8.2):
```
서명: σ = (PKCS1-Pad(H(M)))^d mod n
검증: σ^e mod n =?= PKCS1-Pad(H(M))
```
- 결정적(deterministic) — 같은 메시지는 항상 같은 서명
- 단순하지만 — 적응적 선택 평문 공격에 대한 안전성 증명이 약함

**RSASSA-PSS** (RFC 8017 §8.1, Probabilistic Signature Scheme):
- 매 서명마다 무작위 솔트 추가 → 같은 메시지도 매번 다른 서명
- **확률적 서명** — 적응적 선택 평문 공격에 안전함이 증명된 표준
- FIPS 186-5 가 새 응용에 권장하는 RSA 서명 방식

> *시험 포인트*: RSA-PSS 가 PKCS#1 v1.5 보다 안전. FIPS 186-5 는 새 시스템에 PSS 권장. v1.5 는 호환성 목적으로만 유지.

### §2-2. DSA — Digital Signature Algorithm

> **DSA**: NIST 가 1991 년 제안한 디지털서명 표준. ElGamal 서명을 기반으로 하며 **이산로그 문제** 의 어려움에 보안을 의존한다. FIPS 186-5 (2023) 에서는 **신규 사용 금지** 로 표시되었다.[^fips-186]

DSA 의 서명 (간략화):
```
공개 매개변수: p (소수), q (p−1 의 소인수), g (위수 q 의 생성자)
개인 키: x (1 < x < q)
공개 키: y = g^x mod p

서명 (메시지 해시 h):
1. k ← random (1 < k < q)  [매 서명마다 다른 임시 키]
2. r = (g^k mod p) mod q
3. s = k^(−1) (h + x·r) mod q
4. σ = (r, s)

검증:
w = s^(−1) mod q
u_1 = h·w mod q
u_2 = r·w mod q
v = ((g^u_1 · y^u_2) mod p) mod q
v =?= r
```

**중대한 약점**: DSA 는 임시 키 k 의 무작위성에 보안이 절대적으로 의존한다. k 가 두 서명에서 재사용되거나 예측 가능하면 — Sony PS3 의 ECDSA 사례(2010) 처럼 — 개인키가 즉시 유출된다.

> *시험 포인트*: DSA 는 FIPS 186-5 에서 **신규 사용 금지**. 기존 시스템 검증용으로만 유지. 약점은 **임시 키 k 재사용**.

### §2-3. ECDSA — Elliptic Curve DSA

> **ECDSA**: DSA 의 곱셈 군을 타원곡선 군으로 대체한 변형. 동일 보안 강도에서 키와 서명이 짧다. FIPS 186-5 가 표준화한다.[^fips-186]

ECDSA 의 서명 구조는 DSA 와 거의 동일하지만 타원곡선 점 곱셈을 사용한다. 권장 곡선:
- **NIST P-256·P-384·P-521** (FIPS 186-5)
- **secp256k1** — Bitcoin·Ethereum 에서 사용 (FIPS 186-5 외)

**서명 길이 비교** (128 bit 보안 기준):
- RSA-3072 서명: 약 384 byte
- ECDSA P-256 서명: 약 64 byte (r·s 각 32 byte)

**ECDSA 도 임시 키 k 재사용 시 즉시 개인키 유출** — Sony PS3 사례가 ECDSA 의 약점을 노출시킨 대표 사례. RFC 6979 가 결정론적 k 도출(deterministic k from hash) 을 제안해 이 문제를 일부 완화했다.[^rfc-6979]

### §2-4. EdDSA — Edwards-curve DSA

> **EdDSA**: 비틀린 Edwards 곡선(twisted Edwards curve) 을 사용하는 디지털서명 알고리즘. RFC 8032 가 Ed25519·Ed448 두 변형을 표준화한다.[^rfc-8032]

EdDSA 의 핵심 개선:
- **결정론적** (deterministic) — 임시 값을 메시지 해시에서 도출하므로 k 재사용 위험 본질적 부재
- **부채널 저항** — 일정 시간 연산 가능하도록 곡선 설계
- **빠른 성능** — Curve25519 가 P-256 보다 일반적으로 빠름

| 변형 | 곡선 | 보안 강도 | 키·서명 길이 |
|---|---|---|---|
| Ed25519 | Edwards25519 | 128 bit | 키 32 byte, 서명 64 byte |
| Ed448 | Edwards448 | 224 bit | 키 57 byte, 서명 114 byte |

**FIPS 186-5 에서의 위치**: 2023 년 FIPS 186-5 개정에서 EdDSA 가 처음으로 미국 정부 승인 알고리즘으로 추가되었다. 그 이전에는 RFC 8032 가 표준이지만 정부 인증 알고리즘은 아니었다.

| 알고리즘 | 보안 토대 | 키 길이 (128 bit 보안) | FIPS 186-5 |
|---|---|---|---|
| RSA-PSS / PKCS#1 v1.5 | 인수분해 | 3072 bit | 승인 |
| DSA | 이산로그 | 3072 bit | **신규 금지** |
| ECDSA | 타원곡선 이산로그 | 256 bit | 승인 |
| EdDSA (Ed25519) | 타원곡선 이산로그 | 256 bit | 승인 (2023~) |

> *시험 포인트*: 4 가지 디지털서명의 보안 토대 — RSA = 인수분해, DSA·ECDSA·EdDSA = 이산로그(EC 변형 포함). DSA 는 FIPS 186-5 에서 신규 금지.

[^rfc-8017]: RFC 8017, *PKCS #1: RSA Cryptography Specifications Version 2.2*, Moriarty et al., 2016. https://www.rfc-editor.org/rfc/rfc8017
[^rfc-6979]: RFC 6979, *Deterministic Usage of the Digital Signature Algorithm (DSA) and Elliptic Curve Digital Signature Algorithm (ECDSA)*, Pornin, 2013. https://www.rfc-editor.org/rfc/rfc6979
[^rfc-8032]: RFC 8032, *Edwards-Curve Digital Signature Algorithm (EdDSA)*, Josefsson & Liusvaara, 2017. https://www.rfc-editor.org/rfc/rfc8032

## §3. X.509 인증서 구조

### §3-1. X.509 v3 인증서 필드

> **X.509 인증서**: 공개키와 그 소유자의 신원을 결합하고 CA(Certification Authority) 의 디지털서명으로 신뢰성을 보증한 데이터 구조. ITU-T X.509 가 원본이며 인터넷 환경의 v3 프로파일은 RFC 5280 이 정의한다.[^rfc-5280]

X.509 v3 인증서의 필수 필드 (RFC 5280 §4.1):

| 필드 | 영문 | 설명 |
|---|---|---|
| 버전 | Version | v1·v2·v3. 현재는 v3 가 표준 |
| 시리얼 번호 | Serial Number | CA 가 발급한 인증서의 유일 식별자 |
| 서명 알고리즘 | Signature Algorithm | CA 가 인증서에 서명할 때 사용한 알고리즘 (예: sha256WithRSAEncryption) |
| 발급자 | Issuer | 인증서를 발급한 CA 의 X.500 DN |
| 유효 기간 | Validity | notBefore · notAfter (시작·종료 시각) |
| 주체 | Subject | 인증서가 보증하는 개체의 X.500 DN |
| 주체 공개키 정보 | SubjectPublicKeyInfo | 알고리즘 + 공개키 값 |
| 발급자 유일 ID | Issuer Unique ID | 선택적 (v2 이상) |
| 주체 유일 ID | Subject Unique ID | 선택적 (v2 이상) |
| 확장 | Extensions | v3 에서 추가된 확장 필드 (§3-2) |
| CA 서명 | Signature | 위 필드 전체에 대한 CA 의 디지털서명 |

### §3-2. X.509 v3 확장

X.509 v3 의 핵심 개선이 **확장(extensions)** 필드다. CA 와 응용이 필요에 따라 추가 정보를 표준화된 OID 로 인증서에 담을 수 있다.

| 확장 | OID | 설명 |
|---|---|---|
| **Subject Alternative Name (SAN)** | 2.5.29.17 | 주체의 추가 식별자 (DNS·IP·이메일·URI). 현대 TLS 인증서는 CN 대신 SAN 으로 도메인 검증 |
| **Key Usage** | 2.5.29.15 | 공개키의 허용 용도 (digitalSignature·keyEncipherment·keyCertSign·cRLSign 등) |
| **Extended Key Usage (EKU)** | 2.5.29.37 | 추가 용도 (serverAuth·clientAuth·codeSigning·emailProtection·timeStamping) |
| **Basic Constraints** | 2.5.29.19 | CA 여부 + 경로 길이 제약 (cA: TRUE/FALSE, pathLenConstraint) |
| **CRL Distribution Points** | 2.5.29.31 | CRL 을 가져올 URL |
| **Authority Information Access (AIA)** | 1.3.6.1.5.5.7.1.1 | OCSP 응답자 URL + CA 인증서 URL |
| **Certificate Policies** | 2.5.29.32 | 발급 정책 OID (DV·OV·EV 구분) |
| **Subject Key Identifier** | 2.5.29.14 | 주체 공개키의 식별자 |
| **Authority Key Identifier** | 2.5.29.35 | 발급 CA 의 키 식별자 (체인 검증 보조) |

**중요한 운영 사실**:
- **CN 매칭 폐기**: 현대 브라우저(Chrome 58+, 2017) 는 도메인 검증에 CN 을 사용하지 않으며 **SAN 만** 검증한다.
- **Basic Constraints cA=TRUE**: CA 가 아닌 인증서로 다른 인증서를 서명할 수 없도록 제약. 이 확장이 누락되면 어떤 인증서든 CA 로 동작 가능했던 과거 취약점(2002 년 IE 버그) 의 교훈.

> *시험 포인트*: X.509 v3 의 SAN·KeyUsage·EKU·BasicConstraints 4 개 확장이 가장 빈출. SAN 이 CN 을 대체했다는 사실, BasicConstraints 의 cA 플래그가 CA 권한을 결정한다는 사실 중요.

### §3-3. 인증서 종류 — DV·OV·EV

CA/Browser Forum 의 Baseline Requirements 가 인증서를 검증 수준에 따라 3 종류로 분류한다:

| 종류 | 영문 | 검증 수준 | 표시 |
|---|---|---|---|
| DV | Domain Validated | 도메인 소유 검증만 (이메일·DNS·HTTP) | 자물쇠 아이콘 |
| OV | Organization Validated | 도메인 + 조직 실재 검증 | 자물쇠 + 조직명 (인증서 보기 시) |
| EV | Extended Validation | 도메인 + 조직 + 법적 실재 + 수동 검증 | 과거 녹색 주소창 (현재 폐지) |

**Let's Encrypt** (2015~) 는 DV 인증서를 자동·무료 발급으로 보급해 HTTPS 채택률을 비약적으로 높였다.

## §4. PKI 구성요소와 운영

### §4-1. PKI 구성요소

> **PKI**: 인증서의 발급·관리·검증을 담당하는 신뢰 인프라. 다음 구성요소로 이루어진다.

| 요소 | 영문 | 역할 |
|---|---|---|
| CA | Certification Authority | 인증서 발급 + 폐기 + CRL/OCSP 운영 |
| RA | Registration Authority | 인증서 신청자의 신원 확인 (CA 의 위임) |
| VA | Validation Authority | 인증서 유효성 검증 서비스 (OCSP·SCVP 응답자) |
| 저장소 | Repository | 발급된 인증서·CRL 의 공개 저장소 (LDAP·HTTP) |
| 구독자 | Subscriber | 인증서를 발급받는 최종 개체 (사용자·서버·디바이스) |
| 의존자 | Relying Party | 인증서를 검증하고 사용하는 측 (예: 브라우저) |

### §4-2. PKI 신뢰 구조

PKI 는 CA 사이의 신뢰 관계를 어떻게 구성하느냐에 따라 3 가지 구조로 나뉜다.

#### §4-2-1. 계층 구조 (Hierarchical)

```
                Root CA
              /    |    \
        Sub-CA  Sub-CA  Sub-CA
         /  \    |       |
      End  End  End     End
```

- **루트 CA** 가 최상위 신뢰 앵커. 자기 자신을 서명(self-signed root certificate)
- 루트가 중간 CA(Sub-CA) 를 인증, 중간 CA 가 최종 인증서 발급
- 인증 경로는 하향식 단일 경로

**장점**:
- 단순한 신뢰 모델 — 루트만 신뢰하면 모든 하위 인증서 검증 가능
- 확장성 양호

**단점**:
- 루트 CA 침해 시 전체 PKI 붕괴 → 루트는 오프라인 보관 + HSM
- 서로 다른 도메인 간 신뢰 어려움 (별도 PKI 끼리는 무관)

대표 사례: 인터넷 TLS PKI (브라우저 루트 저장소), 한국 NPKI (KISA 가 최상위 루트 운영)

#### §4-2-2. 네트워크 구조 (Mesh / Cross-certification)

```
    CA-A ←→ CA-B
     ↕      ↕
    CA-C ←→ CA-D
```

- 동등한 CA 들이 서로를 상호 인증(cross-certify)
- 신뢰 경로가 다중 가능

**장점**:
- 단일 장애점 부재 — 한 CA 침해가 전체에 미치는 영향 제한적
- 도메인 간 신뢰 자연스러움

**단점**:
- 인증 경로 검색 복잡 — 다중 경로 중 유효한 것 탐색 필요
- 상호 인증 관리 부담

#### §4-2-3. 복합형 구조 — Bridge CA

```
      [Domain A]                [Domain B]
      Root CA-A                  Root CA-B
         |                          |
       Sub-CA  ←── Bridge CA ──→  Sub-CA
         |                          |
        End                        End
```

- 서로 다른 계층적 PKI 도메인을 **브리지 CA** 가 연결
- 브리지 CA 는 양쪽 도메인의 루트 CA 와 상호 인증

**장점**:
- 계층 구조의 단순성 + 네트워크 구조의 도메인 간 연결성
- 미국 연방 PKI Bridge (FBCA) 가 대표 — 정부·기업·교육 PKI 간 신뢰 연결

| 구조 | 신뢰 앵커 | 경로 검색 | 도메인 간 신뢰 |
|---|---|---|---|
| 계층 | 단일 루트 | 단순 | 어려움 |
| 네트워크 | 다수 동등 CA | 복잡 | 자연스러움 |
| 복합형 (브리지) | 도메인별 루트 + 브리지 | 중간 | 브리지로 연결 |

> *시험 포인트*: 3 구조의 핵심 차이는 ① 신뢰 앵커 수 (단일/다수/도메인별), ② 인증 경로 단순성, ③ 도메인 간 신뢰 방식. 시험에서 "PKI 구성방식" 문제는 이 3 구조의 특징을 묻는 형태로 빈출.

### §4-3. 인증서 폐기 — CRL

> **CRL (Certificate Revocation List)**: CA 가 폐기한 인증서의 시리얼 번호 목록. CA 가 디지털서명하여 발급. RFC 5280 §5 가 정의.[^rfc-5280]

인증서가 만료 전에 폐기되어야 하는 경우:
- 개인키 노출
- 인증서 정보 변경 (예: 직원 퇴직)
- CA 자체의 침해
- 발급 정책 위반

**CRL 의 두 종류**:

| 종류 | 영문 | 내용 |
|---|---|---|
| Full CRL | Full CRL | 모든 폐기된 인증서 전체 목록 |
| Delta CRL | Delta CRL | 직전 Base CRL 이후 변경 분만 |

Delta CRL 은 Full CRL 의 갱신 부담을 줄이기 위해 도입되었지만, 실무에서는 Full CRL 만 사용하는 경우가 많다.

**CRL 의 한계**:
- **신선도 지연** — 발급 주기 사이에는 갱신 없음 (보통 24 시간 ~ 1 주)
- **크기 폭증** — 폐기 누적으로 CRL 자체가 수 MB 까지 증가
- **다운로드 부담** — 검증할 때마다 전체 CRL 다운로드 (Delta 가 일부 완화)

### §4-4. 인증서 폐기 — OCSP

> **OCSP (Online Certificate Status Protocol)**: 인증서 단위로 폐기 상태를 실시간 질의하는 프로토콜. RFC 6960 이 정의.[^rfc-6960]

OCSP 는 CRL 의 한계를 극복하기 위해 도입되었다 — 검증자가 특정 인증서의 폐기 여부만 물어보고, OCSP 응답자(Validation Authority) 가 디지털서명한 응답으로 회신한다.

**OCSP 응답 상태 (RFC 6960 §2.2)**:
- `good` — 폐기되지 않음
- `revoked` — 폐기됨 (폐기 시각·이유 포함)
- `unknown` — 응답자가 해당 인증서를 모름

**OCSP 의 한계**:
- **개인 정보 누설** — 사용자가 어떤 사이트를 방문하는지 OCSP 응답자가 알게 됨
- **응답자 부하** — 대규모 트래픽이 응답자에 집중

### §4-5. OCSP Stapling — RFC 6066

> **OCSP Stapling**: 서버가 자신의 인증서에 대한 OCSP 응답을 미리 받아두고 TLS 핸드셰이크에 포함시켜 전달하는 방식. RFC 6066 §8 (Certificate Status Request 확장) 이 정의.[^rfc-6066]

Stapling 의 효과:
- 클라이언트가 OCSP 응답자에 직접 접속할 필요 없음 → 개인 정보 누설 차단 + 응답자 부하 분산
- TLS 핸드셰이크 지연 감소
- OCSP 응답자 장애 시에도 인증서 검증 가능

**Must-Staple** (RFC 7633): 인증서 확장으로 "이 인증서는 반드시 Stapled OCSP 응답과 함께 제시되어야 함" 을 표시. 클라이언트가 Stapling 부재 시 연결 거부.[^rfc-7633]

### §4-6. SCVP — Server-based Certificate Validation Protocol

> **SCVP**: 클라이언트가 인증서 검증 작업 자체를 신뢰할 수 있는 서버에 위임하는 프로토콜. RFC 5055 가 정의.[^rfc-5055]

SCVP 가 도입된 이유는 — 임베디드 디바이스나 모바일 환경에서 복잡한 인증서 체인 검증 (다중 CA·CRL·OCSP 통합) 을 직접 수행하기 어렵기 때문이다. SCVP 서버가 검증 작업을 대행하고 결과를 서명된 응답으로 회신한다.

| 프로토콜 | 표준 | 질의 단위 | 신뢰 모델 |
|---|---|---|---|
| CRL | RFC 5280 §5 | CA 단위 (전체 목록) | 분산 — 클라이언트가 직접 검증 |
| OCSP | RFC 6960 | 인증서 단위 (실시간) | 분산 — 클라이언트가 직접 검증 |
| OCSP Stapling | RFC 6066 §8 | 인증서 단위 (서버 첨부) | 분산 + 서버 보조 |
| SCVP | RFC 5055 | 검증 작업 전체 (위임) | 중앙화 — SCVP 서버 신뢰 |

> *시험 포인트*: CRL vs OCSP 의 핵심 차이 — CRL 은 **전체 목록 다운로드**, OCSP 는 **실시간 단건 질의**. OCSP Stapling 이 OCSP 의 개인 정보·부하 문제 해결.

[^rfc-6960]: RFC 6960, *X.509 Internet Public Key Infrastructure Online Certificate Status Protocol - OCSP*, Santesson et al., 2013. https://www.rfc-editor.org/rfc/rfc6960
[^rfc-6066]: RFC 6066, *Transport Layer Security (TLS) Extensions: Extension Definitions*, Eastlake, 2011. https://www.rfc-editor.org/rfc/rfc6066
[^rfc-7633]: RFC 7633, *X.509v3 Transport Layer Security (TLS) Feature Extension*, Hallam-Baker, 2015. https://www.rfc-editor.org/rfc/rfc7633
[^rfc-5055]: RFC 5055, *Server-Based Certificate Validation Protocol (SCVP)*, Freeman et al., 2007. https://www.rfc-editor.org/rfc/rfc5055

## §5. 응용 디지털서명

### §5-1. 은닉 서명 — Blind Signature

> **은닉 서명 (Blind Signature)**: 서명자가 메시지의 내용을 보지 못한 채로 서명하는 기법. Chaum 1982 (Crypto '82) 가 전자화폐를 위해 제안.[^chaum-blind]

은닉 서명의 발상은 — 서명자가 누구의 서명인지는 보증하면서도, 서명한 메시지가 무엇인지는 모르게 하여 서명자가 사후에 메시지와 서명자를 연결할 수 없게 한다는 것이다. 이는 전자화폐의 익명성, 전자 투표의 비밀성에 핵심적이다.

**RSA 기반 은닉 서명** (Chaum 의 원본 구성):
```
1. 사용자 Alice 는 메시지 M, 무작위 블라인딩 인자 r 선택
2. Alice → Signer: M' = M · r^e mod n  (블라인드된 메시지)
3. Signer → Alice: σ' = (M')^d mod n = (M · r^e)^d mod n
                       = M^d · r mod n
4. Alice: σ = σ' · r^(-1) mod n = M^d mod n  (블라인딩 제거)
5. σ 는 Signer 의 정당한 서명이지만 Signer 는 M 을 보지 못함
```

**응용**:
- **전자화폐** — 발행 은행이 사용자의 화폐 시리얼 번호를 모르도록 (Chaum 의 ecash, 1990s)
- **전자 투표** — 투표소가 누가 무엇을 투표했는지 모르도록
- **익명 자격 증명** — 발급자가 자격 사용처를 추적할 수 없도록

### §5-2. 이중 서명 — SET 의 Dual Signature

> **이중 서명 (Dual Signature)**: SET (Secure Electronic Transaction) 프로토콜에서 사용된 서명 기법. 한 사용자가 두 메시지를 동시에 서명하면서, 각 수신자에게는 자신과 관련된 메시지만 보이도록 한다. SET Specification Book 1 (1997) 이 정의.[^set]

SET 의 시나리오: 카드 소지자(Alice) 가 ① 상점(Merchant) 에 주문 정보(OI: Order Information) 를, ② 지급 은행(Payment Gateway) 에 결제 정보(PI: Payment Information) 를 동시에 보낸다. 상점은 카드 번호를 보아서는 안 되고, 은행은 주문 내역을 보아서는 안 된다.

**이중 서명 생성**:
```
1. PIMD = H(PI)         [결제 정보 메시지 다이제스트]
2. OIMD = H(OI)         [주문 정보 메시지 다이제스트]
3. POMD = H(PIMD || OIMD)  [통합 다이제스트]
4. DS = Sign(sk_Alice, POMD)  [Alice 의 개인키로 통합 다이제스트 서명]

상점에게: { OI, PIMD, DS, Alice 인증서 }
은행에게: { PI, OIMD, DS, Alice 인증서 }
```

**검증 방식** (상점 측):
1. 상점은 자신이 받은 OI 의 해시 = OIMD' 계산
2. PIMD 는 받은 그대로 사용
3. POMD' = H(PIMD || OIMD') 계산
4. Verify(pk_Alice, POMD', DS) → True 면 정당

상점은 PI 자체를 모르지만 PIMD 를 통해 결제 정보가 변조되지 않았음을 확인 가능. 은행도 마찬가지로 OI 자체를 모르지만 OIMD 로 주문 정보 무결성 검증.

> *시험 포인트*: 이중 서명은 **두 메시지를 한 서명으로** 보호하면서, 각 수신자에게 불필요한 정보를 숨기는 기법. SET 의 핵심 메커니즘이며 시험 단골.

> *cross-ref*: SET 프로토콜 전체는 [3과목 08 단원](../03.application-security/08.electronic-commerce-security.md) 에서 다룬다.

### §5-3. 그 외 응용 서명

| 서명 종류 | 설명 | 응용 |
|---|---|---|
| **그룹 서명** | 그룹 멤버가 자신의 신원은 숨긴 채 그룹의 정당한 멤버임을 입증하는 서명. Chaum & van Heyst 1991 | 익명 회의록, 익명 신고 |
| **다중 서명** | 여러 서명자가 같은 메시지에 모두 서명. 모든 서명자의 동의 필요 | 다중 결재, 멀티시그 지갑 |
| **위임 서명** | 원 서명자가 대리인에게 서명 권한을 위임. 위임 정보가 서명에 포함 | 회사 결재 위임, 자율 시스템 |
| **임계 서명** | n 명 중 t 명 이상이 협력해야 서명 가능 (t-of-n) | 분산 키 관리, 블록체인 다중 서명 |
| **비밀 서명** | 서명자만 검증 가능한 서명 (zero-knowledge proof 기반) | 학술 연구 단계 |

[^chaum-blind]: D. Chaum, "Blind Signatures for Untraceable Payments," *Advances in Cryptology — Proceedings of Crypto '82*, pp. 199–203, 1983. https://link.springer.com/chapter/10.1007/978-1-4757-0602-4_18

## §6. 한국 PKI 운영

### §6-1. 공인인증서 폐지와 인정 전자서명

> [전자서명법] 은 2020 년 6 월 9 일 전부개정되어 2020 년 12 월 10 일 시행됨에 따라, **공인인증서 제도가 폐지**되고 **전자서명 인정사업자** 제도로 전환되었다.[^esign-law]

법률 개정의 핵심 변화:
- **공인인증서 우월적 효력 폐지** — 모든 전자서명이 동등한 법적 효력. 공인·사설 구분 없음
- **국가가 인증서 발급을 독점 관리하지 않음** — 다양한 사설 인증 사업자(카카오·네이버·PASS·금융결제원 등) 가 시장 진입
- **인정사업자 제도** — 일정 기준을 충족한 전자서명 인증사업자를 KISA 가 인정 (강제 아닌 선택)

폐지된 "공인인증서" 라는 용어는 법률상 사용되지 않으며, 개정 후 발급되는 인증서는 **공동인증서** (구 공인인증서가 명칭 변경) 또는 각 사설 사업자의 인증서다.

### §6-2. NPKI — National PKI

> **NPKI (National PKI)**: 한국의 민간 공개키 기반구조. 최상위 루트 CA 는 KISA 가 운영하며, 그 아래 5 개 공인인증기관(현재는 인정사업자) 이 인증서를 발급한다.

NPKI 의 5 개 공인인증기관 (전자서명법 개정 전 명칭, 현재는 인정사업자):
- 한국정보인증 (KICA) — KIICA
- 한국전자인증 (Korea Electronic Certification Authority, CrossCert)
- 코스콤 (구 한국증권전산)
- 금융결제원 (Yessign)
- 한국무역정보통신 (TradeSign)

### §6-3. GPKI — Government PKI

> **GPKI (Government PKI)**: 한국 정부 기관 간 전자 행정을 위한 PKI. 행정안전부가 정책을 총괄하며 별도 루트 CA 를 운영.

GPKI 의 운영 영역:
- 정부 기관 간 전자 결재
- 공무원 신원 인증
- 정부 시스템 간 SSL/TLS

NPKI 와 GPKI 는 **분리된 PKI** 이며, 일반 시민이 GPKI 인증서를 직접 사용하지는 않는다.

### §6-4. KCMVP 와 암호 모듈 검증

> **KCMVP (Korea Cryptographic Module Validation Program)**: KISA 가 운영하는 한국 암호 모듈 검증 제도. 정부·금융·공공 기관에 도입되는 암호 모듈은 KCMVP 인증을 받아야 함.[^kcmvp]

KCMVP 는 미국 NIST 의 FIPS 140-2 / 140-3 와 유사한 검증 체계로, 국산 암호 알고리즘 (SEED·ARIA·LEA·HIGHT·LSH) 의 구현 정확성과 보안을 검증한다.

> *시험 포인트*: 한국 PKI 의 핵심은 ① 2020.12.10 공인인증서 폐지, ② NPKI 와 GPKI 분리, ③ KCMVP 인증 체계. 전자서명법은 2020.6.9 전부개정·2020.12.10 시행이라는 두 시점을 정확히 기억할 것.

[^esign-law]: 대한민국 [전자서명법], 법률 제17354호, 2020.6.9 전부개정, 2020.12.10 시행. 국가법령정보센터. https://www.law.go.kr/법령/전자서명법
[^kcmvp]: KISA 암호 모듈 검증 제도 (KCMVP). https://seed.kisa.or.kr/kcmvp/

## §7. 단원 요약

1. **디지털서명 5 요건** — 위조 불가·인증·부인 방지·변경 불가·재사용 불가.
2. **서명 = 해시 후 개인키 처리, 검증 = 해시 + 공개키 검증**. 메시지 자체가 아닌 해시 서명.
3. **부인 방지는 디지털서명만 제공** — MAC 은 출처 인증까지, 해시는 무결성만.
4. **FIPS 186-5 (2023) 4 알고리즘** — RSA·DSA·ECDSA·EdDSA. **DSA 는 신규 사용 금지**.
5. **RSA-PSS** > PKCS#1 v1.5 — 확률적 서명, 안전성 증명. FIPS 186-5 권장.
6. **DSA·ECDSA 약점** — 임시 키 k 재사용 시 개인키 즉시 유출. RFC 6979 결정론적 k 도출.
7. **EdDSA (Ed25519/Ed448)** — 결정론적·부채널 저항·고성능. FIPS 186-5 (2023) 부터 정부 승인.
8. **X.509 v3 핵심 확장** — SAN (CN 대체), KeyUsage, EKU, BasicConstraints (cA 플래그).
9. **PKI 3 구조** — 계층 (단일 루트), 네트워크 (상호 인증), 복합형 (브리지 CA).
10. **CRL vs OCSP** — CRL 은 전체 목록 다운로드, OCSP 는 실시간 단건 질의. **OCSP Stapling** 이 OCSP 의 개인 정보·부하 문제 해결.
11. **은닉 서명** (Chaum 1982) — 서명자가 메시지 내용을 모르도록. 전자화폐·전자 투표.
12. **이중 서명** (SET 1997) — 두 메시지에 한 서명, 각 수신자는 자신 관련 메시지만 봄.
13. **한국 PKI** — 2020.12.10 공인인증서 폐지 → 인정사업자 제도. NPKI / GPKI 분리. KCMVP 검증.

## §8. 시험 함정 정리

| 함정 | 정답 |
|---|---|
| 디지털서명의 5 요건? | **위조 불가·인증·부인 방지·변경 불가·재사용 불가**. |
| 서명 시 메시지 자체를 서명하는가? | **아니오** — 메시지의 **해시** 를 서명. RSA 입력 길이 제한 + 효율성. |
| MAC 의 부인 방지 제공? | **아니오** — 대칭키 공유로 검증자도 만들 수 있음. |
| FIPS 186-5 의 4 알고리즘? | **RSA·DSA·ECDSA·EdDSA**. DSA 는 **신규 사용 금지**. |
| RSA-PSS 가 PKCS#1 v1.5 보다 우월한 점? | **확률적 서명** — 같은 메시지도 매번 다른 서명. 안전성 증명 강함. |
| DSA·ECDSA 의 결정적 약점? | **임시 키 k 재사용** 시 개인키 즉시 유출. 해결책 RFC 6979 결정론적 k. |
| EdDSA 의 두 변형? | **Ed25519** (Edwards25519, 128 bit) · **Ed448** (Edwards448, 224 bit). |
| X.509 v3 의 SAN 의미? | **Subject Alternative Name** — CN 을 대체. 현대 브라우저는 SAN 만 검증. |
| Basic Constraints cA=TRUE 의미? | 이 인증서는 **CA 인증서**로, 다른 인증서 서명 가능. cA=FALSE 면 최종 인증서. |
| PKI 3 구조 비교? | 계층 (단일 루트·단순), 네트워크 (상호 인증·복잡), 복합형 (브리지·도메인 간). |
| CRL vs OCSP 차이? | CRL = **전체 목록 다운로드** (지연), OCSP = **실시간 단건 질의** (개인 정보 누설). |
| OCSP Stapling 의 효과? | 서버가 OCSP 응답 첨부 → 클라이언트의 OCSP 응답자 직접 접속 불필요. |
| 은닉 서명의 핵심? | 서명자가 **메시지 내용을 보지 못한 채** 서명. Chaum 1982. 전자화폐 응용. |
| 이중 서명의 핵심? | **두 메시지** 에 **한 서명** + 각 수신자는 자신 관련 메시지만 봄. SET 의 핵심. |
| 전자서명법 시행일? | **2020.6.9 전부개정·2020.12.10 시행**. 공인인증서 폐지 → 인정사업자 제도. |
| KCMVP 가 검증하는 것? | **암호 모듈** 의 구현 정확성·보안. NIST FIPS 140 와 유사. |
