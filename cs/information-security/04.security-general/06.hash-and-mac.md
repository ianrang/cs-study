## §1. 해시함수 개요와 요구사항

### §1-1. 해시함수의 정의

> **해시함수 H**: 임의 길이의 비트 문자열을 입력받아 고정 길이 n 의 다이제스트를 출력하는 함수 H: {0,1}* → {0,1}^n.

**일반 해시함수와 암호 해시함수의 차이**:
- 일반 해시함수 (DJB2·MurmurHash·CRC) — 빠르고 분산이 좋으면 충분. 자료구조·체크섬용.
- 암호 해시함수 (SHA-2·SHA-3) — 다음 5 가지 보안 요구사항을 만족해야 함.

### §1-2. 5 가지 보안 요구사항

| 요구사항 | 영문 | 정의 |
|---|---|---|
| 압축성 | Compression | 임의 길이 입력 → 고정 길이 출력 |
| 계산 효율성 | Efficiency | 입력에서 해시값을 빠르게 계산 가능 |
| **일방향성** (선상 저항성) | Preimage Resistance | 해시값 h 가 주어졌을 때 H(x) = h 인 x 를 찾기 어려움 |
| **약한 충돌 저항성** | Second Preimage Resistance | 입력 x 가 주어졌을 때 H(x') = H(x) 인 다른 x' (≠ x) 를 찾기 어려움 |
| **강한 충돌 저항성** | Collision Resistance | H(x) = H(x') 인 임의의 두 입력 x ≠ x' 쌍을 찾기 어려움 |

이 셋의 위계 — **강한 충돌 저항성 ≥ 약한 충돌 저항성 ≥ 일방향성** — 는 다음 의미다:
- 강한 충돌 저항성이 있으면 약한 충돌 저항성도 있다 (반대는 아님)
- 약한 충돌 저항성이 있으면 일방향성도 가능 (정확한 함의 관계는 입력 분포에 따라 다름)

**보안 강도와 해시 길이**:
| 요구사항 | n bit 해시의 보안 강도 |
|---|---|
| 일방향성 | 2^n |
| 약한 충돌 저항성 | 2^n |
| 강한 충돌 저항성 | **2^(n/2)** (생일 공격) |

따라서 SHA-256 은 일방향성·약한 충돌 저항성에서 256 bit 보안, 강한 충돌 저항성에서 128 bit 보안.

> *시험 포인트*: 3 가지 저항성 — preimage (일방향), second preimage (약한 충돌), collision (강한 충돌). 강한 충돌 저항성의 보안은 해시 길이의 **절반**(생일 공격).

## §2. 해시함수의 구조

해시함수의 내부 구조는 두 가지 패러다임으로 분류된다:

### §2-1. Merkle-Damgård 구조

> **Merkle-Damgård 구조**: 압축 함수 (compression function) 를 반복 적용하여 임의 길이 입력을 고정 길이 다이제스트로 만드는 구조. Merkle·Damgård 1989 가 독립 제안. **MD5·SHA-1·SHA-2** 가 채택.

**동작 원리**:
```
입력 메시지 M 을 고정 블록 길이 b 로 분할: M = M_1 || M_2 || ... || M_k (패딩 포함)
초기 상태: H_0 = IV (initial value)
반복: H_i = f(H_(i-1), M_i)  (f = 압축 함수)
출력: H_k
```

**보안 정리**: 압축 함수 f 가 충돌 저항성이 있으면, 전체 해시함수도 충돌 저항성이 있다 (Merkle-Damgård 정리).

**약점 — 길이 연장 공격 (Length Extension Attack)**:
- M-D 구조에서 H(M) 을 알면 — 메시지 M 자체를 몰라도 — H(M || padding || X) 를 임의의 X 에 대해 계산할 수 있다
- 공격자가 알아야 할 것: H(M) (다이제스트), |M| (길이)
- **MD5·SHA-1·SHA-2 모두 이 공격에 취약**
- 따라서 단순 `H(K || M)` 형식의 MAC 은 위험 → HMAC (이중 해시) 가 필요

### §2-2. Sponge 구조

> **Sponge 구조**: 흡수(absorbing) 와 짜내기(squeezing) 의 두 단계로 동작하는 해시 구조. **SHA-3 (Keccak)** 가 채택. Bertoni 외 2007 가 제안.[^fips-202]

**동작 원리**:
```
상태: r + c 비트 (r = rate, c = capacity)
흡수 단계: 입력 블록을 r 비트씩 상태와 XOR 후 순열 f 적용 (반복)
짜내기 단계: 상태에서 r 비트씩 출력 (필요한 길이만큼 반복)
```

**Sponge 의 장점**:
- **길이 연장 공격 저항** — capacity c 비트가 외부에 노출되지 않아 내부 상태 추론 불가
- **가변 출력 길이** — SHAKE128·SHAKE256 같은 XOF (Extendable-Output Function) 자연스러움
- **단순한 보안 분석** — 내부 순열 f 의 분석으로 전체 보안 판단 가능

| 측면 | Merkle-Damgård (MD5·SHA-1·SHA-2) | Sponge (SHA-3) |
|---|---|---|
| 동작 | 압축 함수 반복 | 흡수 + 짜내기 |
| 길이 연장 공격 | ❌ 취약 | ✅ 저항 |
| 가변 출력 (XOF) | 어려움 | 자연스러움 (SHAKE) |
| 보안 정리 | M-D 정리 | Sponge 정리 |

> *시험 포인트*: MD5·SHA-1·SHA-2 = Merkle-Damgård (길이 연장 공격 취약). SHA-3 = Sponge (저항). HMAC 이 길이 연장 공격을 우회하기 위한 표준.

[^fips-202]: NIST FIPS 202, *SHA-3 Standard: Permutation-Based Hash and Extendable-Output Functions*, 2015. https://csrc.nist.gov/publications/detail/fips/202/final

## §3. 주요 해시함수

### §3-1. MD5 — Message Digest 5

> **MD5**: Rivest 1992 가 발표한 128 bit 해시함수. RFC 1321. **현재 사용 금지** — 충돌이 실제로 발견됨.[^rfc-1321]

| 항목 | 값 |
|---|---|
| 출력 길이 | 128 bit |
| 블록 길이 | 512 bit |
| 라운드 수 | 64 |
| 구조 | Merkle-Damgård |
| 상태 | **충돌 발견 (2004 Wang et al.) → 사용 금지** |

MD5 의 충돌은 2004 년 Wang·Yu 등이 실제 충돌 쌍을 발표하며 사실상 깨졌고, 2008 년에는 위조 X.509 인증서를 만든 사례까지 등장했다(MD5CRK 프로젝트). 현재 MD5 는 **무결성 검사** 같은 비암호 용도에서만 제한적으로 사용된다.

### §3-2. SHA-1 — Secure Hash Algorithm 1

> **SHA-1**: NIST 가 1995 년 표준화한 160 bit 해시함수. FIPS 180-1. **2017 년 충돌 발견 (SHAttered)** 으로 사실상 폐기.[^fips-180-4]

| 항목 | 값 |
|---|---|
| 출력 길이 | 160 bit |
| 블록 길이 | 512 bit |
| 라운드 수 | 80 |
| 구조 | Merkle-Damgård |
| 상태 | NIST SP 800-131A 에 따라 **2030 년부터 모든 용도 금지** |

SHA-1 의 충돌은 2017 년 Google·CWI 의 **SHAttered** 프로젝트가 처음 실제 충돌 쌍 (2 PDF 가 같은 해시) 을 공개하며 입증되었다. 이후 SHA-1 은 디지털서명·인증서 용도에서 빠르게 폐기되었다.

### §3-3. SHA-2 — Secure Hash Algorithm 2

> **SHA-2**: NIST 가 2001 년 표준화한 SHA-1 후속 해시함수 패밀리. FIPS 180-4 (2015 개정) 가 6 개 변형을 정의.[^fips-180-4]

| 변형 | 출력 길이 | 블록 길이 | 워드 크기 | 라운드 |
|---|---|---|---|---|
| SHA-224 | 224 bit | 512 bit | 32 bit | 64 |
| **SHA-256** | **256 bit** | **512 bit** | **32 bit** | **64** |
| SHA-384 | 384 bit | 1024 bit | 64 bit | 80 |
| **SHA-512** | **512 bit** | **1024 bit** | **64 bit** | **80** |
| SHA-512/224 | 224 bit | 1024 bit | 64 bit | 80 |
| SHA-512/256 | 256 bit | 1024 bit | 64 bit | 80 |

**SHA-256 vs SHA-512**:
- SHA-256 — 32 bit 시스템 친화적. 가장 널리 사용됨 (TLS·블록체인·디지털서명)
- SHA-512 — 64 bit 시스템에서 더 빠를 수 있음 (블록·워드가 큼)

SHA-2 도 Merkle-Damgård 구조라서 길이 연장 공격에 취약하지만, 2024 년 현재까지 충돌 공격 보고 없음. **SHA-256 은 현재 가장 널리 사용되는 안전 해시**.

### §3-4. SHA-3 — Keccak

> **SHA-3**: NIST 가 2015 년 FIPS 202 로 표준화한 새 해시함수 패밀리. Keccak 알고리즘 (Bertoni 외) 이 5 년간 공개 경쟁(SHA-3 Competition) 끝에 선정.[^fips-202]

| 변형 | 출력 길이 | 비트율 r | 용량 c |
|---|---|---|---|
| SHA3-224 | 224 bit | 1152 | 448 |
| SHA3-256 | 256 bit | 1088 | 512 |
| SHA3-384 | 384 bit | 832 | 768 |
| SHA3-512 | 512 bit | 576 | 1024 |
| SHAKE128 | 가변 | 1344 | 256 |
| SHAKE256 | 가변 | 1088 | 512 |

**SHAKE (XOF)**: Extendable-Output Function — 임의 길이 출력 가능. 대칭키 생성·KDF·DRBG 에 활용.

**SHA-2 와 SHA-3 의 관계**: SHA-3 는 SHA-2 의 **후계자가 아니라 다른 구조의 대안**이다. SHA-2 가 깨질 경우의 보험으로 NIST 가 다른 구조를 표준화한 것 — 둘은 병행 사용된다.

| 측면 | SHA-2 | SHA-3 |
|---|---|---|
| 구조 | Merkle-Damgård | Sponge |
| 길이 연장 공격 | ❌ 취약 | ✅ 저항 |
| 표준화 연도 | 2001 (FIPS 180-2) | 2015 (FIPS 202) |
| 권장 용도 | 일반 (가장 보급) | SHA-2 대안 + XOF 필요 시 |

### §3-5. LSH·HAS-160 — 한국 해시함수

> **LSH (Lightweight Secure Hash)**: KISA 가 2014 년 발표한 국산 해시함수. TTAS.KO-12.0276 + KS X 3262 한국 표준.[^lsh]

| 변형 | 출력 길이 |
|---|---|
| LSH-256-224 | 224 bit |
| LSH-256-256 | 256 bit |
| LSH-512-224 | 224 bit |
| LSH-512-256 | 256 bit |
| LSH-512-384 | 384 bit |
| LSH-512-512 | 512 bit |

LSH 는 KCMVP 인증 모듈 알고리즘이며 한국 정부·금융 시스템에서 사용된다.

> **HAS-160**: 한국형 SHA-1 변형. 160 bit 출력. 1998 년 TTA 표준화. 현재 SHA-1 과 같은 이유로 신규 사용 권장하지 않음.

### §3-6. 해시함수 종합 비교

| 해시 | 출력 | 표준 | 구조 | 상태 |
|---|---|---|---|---|
| MD5 | 128 | RFC 1321 (1992) | M-D | **충돌 발견 — 사용 금지** |
| SHA-1 | 160 | FIPS 180-1 (1995) | M-D | **2030 부터 전면 금지** |
| SHA-2 (SHA-256·512) | 224 ~ 512 | FIPS 180-4 (2015) | M-D | 현행 권장 |
| SHA-3 (Keccak) | 224 ~ 512 + XOF | FIPS 202 (2015) | Sponge | 현행 권장 |
| LSH | 224 ~ 512 | TTAS.KO-12.0276 | (자체) | 한국 KCMVP 인증 |
| HAS-160 | 160 | TTA (1998) | (SHA-1 계열) | 신규 권장 안 함 |

> *시험 포인트*: 각 해시의 출력 길이·구조·표준·상태를 외워야 한다. 특히 ① **MD5 (128, M-D, 깨짐)**, ② **SHA-1 (160, M-D, 깨짐)**, ③ **SHA-2 (256·512, M-D, 현행)**, ④ **SHA-3 (Sponge, 현행)** 의 차이.

[^rfc-1321]: RFC 1321, *The MD5 Message-Digest Algorithm*, Rivest, 1992. https://www.rfc-editor.org/rfc/rfc1321
[^lsh]: TTAS.KO-12.0276, *경량 해시함수 LSH (Lightweight Secure Hash)*, KISA. https://seed.kisa.or.kr/

## §4. 해시 공격

### §4-1. 무차별 대입 공격

n bit 해시에 대한 일방향성 공격 — 평균 2^n 회 시도. SHA-256 → 2^256 회로 사실상 불가능.

### §4-2. 생일 공격 — Birthday Attack

> **생일 공격**: 충돌을 찾기 위한 통계적 공격. 23 명만 모이면 두 사람 생일이 같을 확률이 50% 가 넘는다는 "생일 역설" 에서 이름.

n bit 해시의 강한 충돌 저항성 보안 강도는 **2^(n/2)** — SHA-256 의 충돌은 약 2^128 회로 발견 가능. 100 bit 보안을 원하면 200 bit 이상 해시 필요.

### §4-3. 길이 연장 공격 — Length Extension Attack

> **길이 연장 공격**: Merkle-Damgård 구조의 본질적 취약성. H(M) 과 |M| 만 알면 — M 자체를 몰라도 — H(M || padding || X) 를 임의 X 에 대해 계산 가능.

**시나리오**:
```
정직한 사용자: H(secret || message) 를 MAC 으로 사용
공격자: 알 수 있는 것 = (message, H(secret || message), |secret|+|message|)
       알 수 없는 것 = secret
공격: 공격자가 message_extra 를 자신이 정한 값으로 설정하고
     H(secret || message || padding || message_extra) 를 계산
     → secret 을 모르고도 정당한 MAC 위조!
```

**대응**:
- **HMAC (RFC 2104, FIPS 198-1)** — 이중 해시 구조로 길이 연장 우회 (§5-1)
- **SHA-3 사용** — Sponge 구조는 길이 연장 저항
- **SHA-512/256** — Truncate 으로 내부 상태 노출 차단

### §4-4. 사전 공격·레인보우 테이블

> **레인보우 테이블 (Rainbow Table)**: 자주 사용되는 패스워드의 해시값을 미리 계산해 둔 표. 패스워드 해시에서 원본 추론에 사용.

대응: **솔트 (salt)** — 사용자별로 다른 무작위 값을 패스워드에 추가하여 해시. 같은 패스워드라도 솔트가 다르면 해시값이 달라 미리 계산된 표 무력화. §7 참조.

> *시험 포인트*: 길이 연장 공격은 **Merkle-Damgård 의 본질적 약점** 이며 SHA-2 도 취약. 대응은 ① HMAC 이중 해시, ② SHA-3 사용, ③ SHA-512/256 같은 Truncate 변형.

## §5. 메시지 인증 코드 (MAC)

> **MAC (Message Authentication Code)**: 공유 비밀 키로 생성하는 짧은 인증 태그. 무결성 + 출처 인증 (공유 키 보유자 둘 중 하나) 제공.[^fips-198]

[01 단원 §6-2](./01.authentication.md) 의 MAC vs 디지털서명 vs 해시 비교를 다시 떠올리면 — MAC 은 부인 방지를 제공하지 않지만 (대칭키 공유), 그 외 무결성·출처 인증은 디지털서명과 동등하다. MAC 은 디지털서명보다 빠르므로 부인 방지가 불필요한 환경에서 선호된다.

### §5-1. HMAC — Hash-based MAC

> **HMAC (Hash-based MAC)**: 해시함수 H 를 기반으로 하는 키-기반 MAC. RFC 2104 (1997), FIPS 198-1 (2008) 표준.[^rfc-2104][^fips-198]

```
HMAC(K, M) = H( (K' XOR opad) || H( (K' XOR ipad) || M ) )
```
- `K'` = K 가 블록 길이보다 짧으면 0 패딩, 길면 H(K) 후 0 패딩
- `ipad` = 0x36 반복 (inner pad)
- `opad` = 0x5C 반복 (outer pad)

**왜 이중 해시인가?** — 단순 `H(K || M)` 은 길이 연장 공격에 취약. HMAC 은 외부 해시로 내부 결과를 한 번 더 처리하여 길이 연장 공격을 차단한다. 외부 해시의 결과만 공개되므로 공격자는 내부 상태를 알 수 없다.

**보안 증명**: HMAC 은 해시함수 H 가 PRF (의사 랜덤 함수) 이면 안전하다는 것이 증명되어 있다 (Bellare 2006). Merkle-Damgård 의 충돌 저항성보다 약한 가정으로도 안전성 증명.

| HMAC 변형 | 출력 길이 | 권장 |
|---|---|---|
| HMAC-MD5 | 128 bit | ❌ NIST SP 800-131A 폐기 |
| HMAC-SHA-1 | 160 bit | ⚠️ 일부 용도 폐기 (서명) |
| **HMAC-SHA-256** | 256 bit | ✅ 현행 표준 |
| HMAC-SHA-384·512 | 384·512 bit | ✅ 고보안 |
| HMAC-SHA3-256·512 | 256·512 bit | ✅ SHA-3 변형 |

### §5-2. CMAC — Cipher-based MAC

> **CMAC (Cipher-based MAC)**: 블록 암호 (AES) 를 기반으로 하는 MAC. NIST SP 800-38B 가 표준.[^nist-38b]

CMAC 은 OMAC1 알고리즘의 변형으로, 블록 암호를 CBC 모드와 유사하게 사용해 MAC 태그를 생성한다.

| 항목 | 값 |
|---|---|
| 기반 | 블록 암호 (보통 AES) |
| 키 길이 | 128/192/256 bit (AES) |
| 태그 길이 | 64~128 bit |
| 표준 | NIST SP 800-38B |

**HMAC vs CMAC**:
- HMAC = 해시 기반, 소프트웨어 환경에 유리
- CMAC = 블록 암호 기반, AES-NI HW 가속 환경에 유리
- 보안성은 동등 (둘 다 표준)

### §5-3. GMAC·Poly1305

> **GMAC (Galois MAC)**: GCM 인증 암호의 인증 부분. NIST SP 800-38D.[^nist-38d]

GCM 의 인증 태그만 떼어내면 GMAC 이 된다. 일반 MAC 으로도 단독 사용 가능 — 데이터 자체는 암호화 없이 무결성·인증만 필요한 경우.

> **Poly1305**: Bernstein 2005 가 설계한 다항식 평가 기반 MAC. ChaCha20 과 결합한 ChaCha20-Poly1305 (RFC 8439) 가 TLS 1.3 표준.[^rfc-8439]

| MAC | 기반 | 표준 | 주요 용도 |
|---|---|---|---|
| HMAC | 해시함수 | RFC 2104, FIPS 198-1 | 일반, TLS, IPsec |
| CMAC | 블록 암호 (AES) | NIST SP 800-38B | AES 환경, 임베디드 |
| GMAC | 블록 암호 (GCM 의 일부) | NIST SP 800-38D | 인증만 필요한 경우 |
| Poly1305 | 다항식 평가 | RFC 8439 | TLS 1.3, ChaCha20-Poly1305 |
| KMAC | SHA-3 (Keccak) | NIST SP 800-185 | SHA-3 환경, XOF MAC |

### §5-4. KMAC — Keccak MAC

> **KMAC**: SHA-3 / Keccak 기반 MAC. NIST SP 800-185 가 cSHAKE·KMAC·TupleHash·ParallelHash 를 정의.[^nist-185]

KMAC 의 장점:
- **단순한 키 결합** — Sponge 구조의 자연스러운 키 흡수 (HMAC 의 ipad·opad 같은 트릭 불필요)
- **가변 출력 길이** — XOF 기반
- **길이 연장 공격 본질 부재** — SHA-3 자체가 저항

[^rfc-2104]: RFC 2104, *HMAC: Keyed-Hashing for Message Authentication*, Krawczyk et al., 1997. https://www.rfc-editor.org/rfc/rfc2104
[^fips-198]: NIST FIPS 198-1, *The Keyed-Hash Message Authentication Code (HMAC)*, 2008. https://csrc.nist.gov/publications/detail/fips/198/1/final
[^nist-38b]: NIST SP 800-38B, *Recommendation for Block Cipher Modes of Operation: The CMAC Mode for Authentication*, 2005. https://csrc.nist.gov/publications/detail/sp/800-38b/final
[^nist-38d]: NIST SP 800-38D, *Recommendation for Block Cipher Modes of Operation: Galois/Counter Mode (GCM) and GMAC*, 2007. https://csrc.nist.gov/publications/detail/sp/800-38d/final
[^rfc-8439]: RFC 8439, *ChaCha20 and Poly1305 for IETF Protocols*, Nir & Langley, 2018. https://www.rfc-editor.org/rfc/rfc8439

## §6. MAC vs 해시 vs 디지털서명 종합 비교

[01 단원 §6-2](./01.authentication.md) 와 [04 단원 §1-3](./04.digital-signature-pki.md) 에서 다룬 비교를 본 단원 관점에서 다시 정리한다.

| 메커니즘 | 키 구조 | 무결성 | 출처 인증 | 부인 방지 | 검증자 | 속도 |
|---|---|---|---|---|---|---|
| 해시 (SHA-256) | 키 없음 | O | X | X | 누구나 | 빠름 |
| MAC (HMAC·CMAC) | 대칭키 (양쪽 공유) | O | O (제한) | **X** | 키 보유자 | 빠름 |
| 디지털서명 (RSA·ECDSA) | 비대칭 키쌍 | O | O | **O** | 누구나 (공개키) | 느림 |

**선택 기준**:
- **부인 방지 필요** → 디지털서명 (느리지만 유일한 선택)
- **부인 방지 불필요 + 속도 중요** → MAC (HMAC·CMAC·GMAC·Poly1305)
- **인증 불필요 + 무결성만** → 해시 (SHA-256·SHA-3)

## §7. 패스워드 해싱

### §7-1. 일반 해시 vs 패스워드 해시

일반 SHA-256 같은 빠른 해시는 **패스워드 저장에 부적합**하다. 이유는:
- 무차별 대입이 너무 빠름 (현대 GPU 로 SHA-256 수십억 회/초)
- 레인보우 테이블 공격에 노출

패스워드 해시는 의도적으로 **느리게 (계산 비용이 크게)** 설계된다. 합법적 사용자는 한 번만 계산하면 되지만, 공격자는 수십억 후보를 시도해야 하므로 의도된 느림이 공격자에게 더 큰 부담이다.

### §7-2. PBKDF2 — Password-Based KDF 2

> **PBKDF2**: HMAC 을 반복 적용하여 패스워드를 KDF 처럼 처리. NIST SP 800-132 (2010), RFC 8018 (PKCS#5 v2.1).[^nist-132]

```
PBKDF2(P, S, c, dkLen) = T_1 || T_2 || ... || T_l
T_i = F(P, S, c, i)
F(P, S, c, i) = U_1 XOR U_2 XOR ... XOR U_c
U_1 = HMAC-SHA-256(P, S || INT(i))
U_j = HMAC-SHA-256(P, U_(j-1))
```
- P = 패스워드, S = 솔트, c = 반복 횟수, dkLen = 출력 길이
- 반복 횟수 c 가 비용 매개변수 (현재 600,000 이상 권장 — OWASP 2023)

PBKDF2 의 약점: **GPU·ASIC 병렬화 가능** — HMAC-SHA-256 은 단순 연산이라 전용 HW 로 가속 가능.

### §7-3. bcrypt

> **bcrypt**: Provos & Mazières 1999 가 OpenBSD 를 위해 설계. Blowfish 의 키 스케줄링을 활용해 메모리 접근 패턴이 ASIC 가속을 어렵게 함.[^bcrypt]

```
bcrypt(P, S, cost) = ...
- cost = log_2(반복 횟수). 보통 10~12 이상 권장 (cost 12 = 4,096 회 반복)
- 출력 = $2b$cost$salt$hash 형식
```

### §7-4. scrypt

> **scrypt**: Percival 2009 가 설계. **메모리-하드 함수** — 의도적으로 큰 메모리를 요구해 GPU·ASIC 가속 차단. RFC 7914.[^rfc-7914]

```
scrypt(P, S, N, r, p, dkLen)
- N = CPU/메모리 비용 매개변수 (보통 2^14 이상)
- r = 블록 크기
- p = 병렬화 매개변수
```

### §7-5. Argon2

> **Argon2**: 2015 년 Password Hashing Competition (PHC) 우승. RFC 9106 (2021) 표준.[^rfc-9106]

| 변형 | 특징 | 권장 용도 |
|---|---|---|
| Argon2d | 데이터 의존적 메모리 접근. GPU 저항 | 최대 GPU 저항 필요 |
| Argon2i | 데이터 비의존 메모리 접근. 부채널 저항 | 부채널 공격 우려 환경 |
| **Argon2id** | i 와 d 의 혼합. 균형 | **OWASP 2023 권장 기본값** |

**OWASP 2023 권장 매개변수**:
- Argon2id: m=64MB, t=3, p=4 또는 m=37MB, t=4 등 (구현마다 다름)
- bcrypt: cost ≥ 12
- scrypt: N=2^17, r=8, p=1
- PBKDF2-SHA-256: 600,000 회 반복

### §7-6. Salt·Pepper·Stretching

| 기법 | 정의 | 효과 |
|---|---|---|
| **Salt** | 사용자별로 다른 무작위 값을 패스워드에 추가 후 해시 | 레인보우 테이블·중복 무력화 |
| **Pepper** | 모든 사용자에 동일한 비밀 값 (DB 외부 보관) 을 추가 | DB 만 노출 시 공격자가 계산 불가 |
| **Stretching** | 해시를 N 회 반복 적용 (= 비용 매개변수) | 무차별 대입 비용 증가 |

| 알고리즘 | 메모리 비용 | GPU 저항 | 권장 |
|---|---|---|---|
| PBKDF2 | 낮음 | ❌ | Legacy 호환만 |
| bcrypt | 낮음 (4 KB 메모리) | 부분적 | 호환·간단함 |
| scrypt | 높음 | ✅ | GPU 저항 |
| **Argon2id** | 매우 높음 | ✅ | **OWASP 권장 기본** |

> *시험 포인트*: 패스워드 해싱은 **빠른 해시 금지** — 의도적 느림. 솔트는 사용자별 무작위 (DB 저장), 페퍼는 시스템 공통 비밀 (DB 외 저장). Argon2id 가 현행 권장.

[^nist-132]: NIST SP 800-132, *Recommendation for Password-Based Key Derivation: Part 1 — Storage Applications*, 2010. https://csrc.nist.gov/publications/detail/sp/800-132/final
[^bcrypt]: N. Provos and D. Mazières, "A Future-Adaptable Password Scheme," *Proceedings of the FREENIX Track: 1999 USENIX Annual Technical Conference*, 1999.
[^rfc-7914]: RFC 7914, *The scrypt Password-Based Key Derivation Function*, Percival & Josefsson, 2016. https://www.rfc-editor.org/rfc/rfc7914
[^rfc-9106]: RFC 9106, *Argon2 Memory-Hard Function for Password Hashing and Proof-of-Work Applications*, Biryukov et al., 2021. https://www.rfc-editor.org/rfc/rfc9106

## §8. 단원 요약

1. **해시 5 요건** — 압축성·계산 효율성·일방향성·약한 충돌 저항성·강한 충돌 저항성.
2. **저항성 위계** — 강한 충돌 ≥ 약한 충돌 ≥ 일방향. 강한 충돌 보안 = **해시 길이의 절반** (생일 공격).
3. **두 구조** — Merkle-Damgård (MD5·SHA-1·SHA-2, 길이 연장 취약) vs Sponge (SHA-3, 저항).
4. **MD5·SHA-1 폐기** — MD5 (Wang 2004), SHA-1 (Google SHAttered 2017). NIST 2030 SHA-1 전면 금지.
5. **SHA-2** — FIPS 180-4. 224·256·384·512·512/224·512/256. **현행 가장 보급**.
6. **SHA-3** — FIPS 202. Keccak. SHA3-224/256/384/512 + SHAKE128/256 (XOF).
7. **국산 해시** — LSH (TTAS.KO-12.0276), HAS-160 (구).
8. **길이 연장 공격** — Merkle-Damgård 의 본질 약점. HMAC 이중 해시·SHA-3·Truncate 으로 대응.
9. **HMAC** — `H((K XOR opad) || H((K XOR ipad) || M))`. **ipad=0x36, opad=0x5C**. RFC 2104, FIPS 198-1.
10. **MAC 종류** — HMAC (해시), CMAC (블록 암호), GMAC (GCM 일부), Poly1305 (다항식), KMAC (SHA-3).
11. **MAC vs 디지털서명 vs 해시** — 부인 방지는 디지털서명만 제공.
12. **패스워드 해싱** — 의도적 느림. **Argon2id** (OWASP 2023 권장), scrypt (메모리-하드), bcrypt (Legacy), PBKDF2 (호환).
13. **Salt·Pepper·Stretching** — Salt 는 사용자별 무작위, Pepper 는 시스템 공통 비밀, Stretching 은 반복 비용.

## §9. 시험 함정 정리

| 함정 | 정답 |
|---|---|
| 해시함수의 5 요건? | 압축성·계산 효율성·**일방향성**·**약한 충돌 저항성**·**강한 충돌 저항성**. |
| 강한 충돌 저항성 보안 강도? | n bit 해시 → **2^(n/2)** (생일 공격). SHA-256 → 128 bit 충돌. |
| MD5·SHA-1 의 출력 길이? | MD5 = **128 bit**, SHA-1 = **160 bit**. 둘 다 **사용 금지**. |
| SHA-256 의 블록 길이는? | **512 bit** (입력 처리 단위). 출력은 256 bit. |
| SHA-512 의 블록 길이는? | **1024 bit**. 출력 512 bit. 64 bit 워드. |
| Merkle-Damgård 와 Sponge 의 차이? | M-D = MD5/SHA-1/SHA-2 (길이 연장 취약). Sponge = SHA-3 (저항). |
| 길이 연장 공격이 가능한 정보? | **H(M) 와 |M|** 만 알면 H(M || padding || X) 계산 가능 (M 자체 몰라도). |
| HMAC 의 ipad·opad 값? | **ipad=0x36, opad=0x5C**, 블록 길이만큼 반복. |
| HMAC 의 출력 길이? | 사용한 해시의 출력 길이와 동일 (HMAC-SHA-256 → 256 bit). |
| MAC 종류와 기반? | HMAC (해시), CMAC (AES 블록 암호), GMAC (GCM), Poly1305 (다항식), KMAC (SHA-3). |
| 부인 방지를 제공하는 메커니즘? | **디지털서명**만. MAC·해시는 ❌. |
| 패스워드 해싱 권장 알고리즘? | **Argon2id** (OWASP 2023). scrypt·bcrypt·PBKDF2 도 허용. **단순 SHA 금지**. |
| 솔트 vs 페퍼? | 솔트 = **사용자별 무작위** (DB 함께 저장). 페퍼 = **시스템 공통 비밀** (DB 외부 저장). |
| 레인보우 테이블 방어? | **솔트** 추가. 같은 패스워드라도 솔트가 다르면 해시값이 다름. |
| Argon2 의 3 변형? | Argon2d (GPU 저항), Argon2i (부채널 저항), **Argon2id (혼합·권장)**. |
| LSH 의 출력 길이? | **224 ~ 512 bit**. KISA 국산 해시. KCMVP 인증. |
