# 01. 인증 (Authentication)

## §0. 들어가며

> **인증(authentication)**: 어떤 개체(사용자·메시지·디바이스) 가 자신이 주장하는 신원이 맞다는 사실을 검증하는 과정.[^nist-63-3]

4과목 출제기준 1.1 "인증" 은 **사용자 인증**(사람이 시스템에 자신의 신원을 증명) · **메시지 인증**(전송된 데이터의 출처와 무결성을 증명) · **디바이스 인증**(통신 상대가 정당한 디바이스임을 증명) 의 세 영역을 모두 포함한다. 이 셋은 표면적으로 다른 문제 같지만 실제로는 동일한 핵심 도구(공유 비밀·공개키 쌍·해시·디지털서명) 를 누가 보유하고 있는지를 검증한다는 점에서 같은 토대 위에 있다.

본 단원은 다음 의존성 위에서 동작한다:
- **05 단원** (암호 알고리즘) — 챌린지-응답에 사용되는 대칭키·공개키 암호의 기초
- **06 단원** (해시함수와 MAC) — HMAC·CMAC 의 정의, 비밀번호 해싱
- **04 단원** (디지털서명·PKI) — Kerberos·FIDO2·디바이스 인증서의 X.509·서명 검증

처음 학습한다면 §1 → §2 → §3 → §5 → §6 → §7 → §4 순서로 읽으면 가장 자연스럽다(생체 인증은 측정·평가 지표가 다른 영역과 결이 다르므로 마지막에 본다).

## §1. 사용자 인증의 개요

### §1-1. 인증의 3요소

> **3요소 (Three factors of authentication)**: 사용자가 시스템에 자신의 신원을 증명할 때 제시할 수 있는 증거의 본질적 분류. NIST SP 800-63B 는 이를 *what you know · what you have · what you are* 세 가지로 정의한다.[^nist-63b]

인증의 3요소는 "사용자가 무엇을 가지고 신원을 증명하는가" 라는 질문에 대한 분류 체계다. 이 분류가 중요한 이유는 **서로 다른 요소를 결합해야 다요소 인증(MFA) 이 성립**하며 — 같은 요소를 두 번 사용하는 것은 단요소의 반복일 뿐 다요소가 아니다(예: 패스워드 + 비밀 질문은 둘 다 지식 기반이므로 단요소).

| 요소 | 영문 | 정의 | 대표 예시 |
|---|---|---|---|
| 지식 기반 | Something you know | 사용자가 기억하는 비밀 | 패스워드, PIN, 암호 질문, 패턴 |
| 소유 기반 | Something you have | 사용자가 물리적으로 보유한 것 | OTP 토큰, 스마트카드, USB 보안키, 휴대폰(Push 인증) |
| 존재 기반 | Something you are | 사용자의 생체적·행동적 특성 | 지문, 홍채, 얼굴, 정맥, 음성, 키 입력 패턴, 서명 |

> *시험 포인트*: 위치 기반(GPS·IP) 과 행동 기반(통신 패턴) 은 NIST SP 800-63B 의 정식 3요소가 아니라 **부가 요소(adaptive·risk-based)** 로 분류된다. 시험에서 "5요소 인증" 이라는 표현이 등장하면 어떤 모델을 기준으로 한 것인지 출처 확인이 필요하다.

### §1-2. 다요소 인증 (MFA)

> **다요소 인증 (Multi-Factor Authentication, MFA)**: 서로 다른 두 개 이상의 인증 요소를 결합하여 사용자의 신원을 검증하는 방식.[^nist-63b]

MFA 가 보안성을 높이는 메커니즘은 단순하다 — 공격자가 패스워드 한 가지를 탈취하더라도 OTP 토큰이라는 물리적 객체를 동시에 탈취하지 않으면 인증을 통과할 수 없다. 즉 공격자가 통과해야 하는 **방어 계층의 종류** 자체가 늘어나는 것이지 동일 계층의 강도가 늘어나는 것이 아니다. 같은 요소를 두 번 쓰는 "패스워드 + 비밀 질문" 같은 조합은 다요소가 아니라 단요소 강화(strengthened single-factor) 일 뿐이다.

대표 조합:
- 패스워드(지식) + OTP(소유) — 가장 일반적
- 스마트카드(소유) + PIN(지식) — 금융권 표준
- 지문(존재) + PIN(지식) — 모바일 결제
- FIDO2 보안키(소유) + 디바이스 잠금 해제 패스워드/지문(지식 또는 존재) — 최신 권장

### §1-3. NIST SP 800-63B AAL 등급

NIST SP 800-63B 는 인증의 강도를 **AAL(Authenticator Assurance Level)** 1·2·3 의 세 단계로 규정한다.[^nist-63b] 이 등급은 어떤 인증기 조합이 어느 정도 신뢰성을 제공하는지를 정부·산업이 공통 언어로 합의하기 위한 척도이며, 한국의 KISA 가이드라인에서도 동일한 개념을 차용한다.

| AAL | 요구사항 | 인증기 예시 |
|---|---|---|
| AAL1 | 단요소 인증 허용 (지식 또는 소유 또는 존재 1개) | 패스워드 단독 |
| AAL2 | 다요소 인증 필수 (서로 다른 2개 요소) | 패스워드 + OTP, 패스워드 + 푸시 인증 |
| AAL3 | 하드웨어 기반 다요소 인증 + 검증자 임퍼스네이션 저항 | FIPS 140 검증 하드웨어 보안키 + PIN 또는 생체 |

> *시험 포인트*: AAL3 은 "하드웨어 기반" 과 "암호적 검증자 임퍼스네이션 저항(verifier impersonation resistance)" 을 함께 요구한다. 이는 사실상 FIDO2 인증기·스마트카드·HSM 수준을 의미한다.

[^nist-63b]: NIST SP 800-63B, *Digital Identity Guidelines: Authentication and Lifecycle Management*. https://pages.nist.gov/800-63-3/sp800-63b.html

## §2. 지식 기반 인증

### §2-1. 패스워드

> **패스워드 (memorized secret)**: 사용자가 기억하여 입력하는 문자열 형태의 비밀. NIST SP 800-63B §5.1.1 은 이를 "memorized secret authenticator" 로 정의한다.[^nist-63b]

패스워드는 가장 오래된 인증 방식이며 동시에 가장 취약하다. NIST SP 800-63B 가 2017 년 개정에서 권장한 패스워드 정책은 과거의 통념(자주 변경·복잡한 문자 강제) 을 상당 부분 폐지한 점이 핵심이다.

**NIST SP 800-63B §5.1.1.2 검증자 요구사항**[^nist-63b]:
- 사용자 선택 패스워드는 **최소 8 자**, 검증자는 **최소 64 자까지** 허용해야 함
- 모든 인쇄 가능한 ASCII 와 유니코드 문자 허용 (공백 포함)
- **차단 목록(blocklist)**: 일반 사전 단어, 이전 침해 사고에서 노출된 패스워드(HaveIBeenPwned 등), 컨텍스트 단어(서비스명·사용자명) 차단
- **주기적 강제 변경 폐지** — 침해 증거가 없는 한 변경을 요구하지 않음 (이전 NIST 권장은 90 일 변경)
- 복잡도 강제 폐지 — "대문자·숫자·특수문자 의무" 같은 구성 규칙 미부과 권장

이 변화의 근거는 사용자 행동 연구다 — 강제 변경과 복잡도 규칙은 사용자가 "Password1!" → "Password2!" → "Password3!" 같은 예측 가능한 변형으로 대응하게 만들어 오히려 보안성을 떨어뜨린다는 것이 NIST 의 결론이다.

**패스워드 저장 — 06 단원 cross-ref**: 서버는 패스워드를 평문이 아닌 **솔트(salt) 가 추가된 일방향 해시** 로 저장해야 하며, 권장 알고리즘은 Argon2id (RFC 9106), bcrypt, scrypt (RFC 7914), PBKDF2 (NIST SP 800-132) 다. 자세한 내용은 [06 단원 §7](./06.hash-and-mac.md) 참조.

### §2-2. PIN·시도-응답·암호 질문

**PIN (Personal Identification Number)**: 일반적으로 4~6 자리 숫자로 구성된 짧은 비밀. 패스워드보다 키 공간이 작아 무차별 대입에 취약하므로 단독 사용은 권장하지 않으며, **재시도 횟수 제한과 결합**(예: 3회 실패 시 카드 잠금) 하여 운영한다. 한국의 금융 IC 카드는 4 자리 PIN + 카드(소유) 의 2요소가 표준이다.

**시도-응답(challenge-response) 인증**: 서버가 클라이언트에게 무작위 챌린지(난수) 를 보내고, 클라이언트는 비밀(패스워드 해시·대칭키·개인키 중 하나) 로 챌린지를 처리한 응답을 돌려보낸다. 이는 **재전송 공격(replay attack) 방지** 가 핵심 목적이며, 같은 응답을 다음 세션에 재사용할 수 없다(매 세션마다 챌린지가 다름). MS-CHAPv2(RFC 2759), CRAM-MD5(RFC 2195), HTTP Digest Authentication(RFC 7616) 등이 대표 사례다.

**암호 질문(security question)**: "어머니의 결혼 전 성씨는?" 같은 보조 인증 수단. NIST SP 800-63B §5.1.1.2 는 이를 **단독 검증자로 사용 금지**(SHALL NOT) 로 명시한다 — 답변이 SNS 등에서 추론 가능하고 한 번 노출되면 변경이 어렵기 때문이다. 보조적 사용도 권장되지 않는다.

> *시험 포인트*: 시도-응답 방식은 패스워드 자체를 네트워크에 송신하지 않는다는 점이 핵심이다. 이는 평문 패스워드 송신 방식인 HTTP Basic Authentication(RFC 7617) 과 대비된다.

## §3. 소유 기반 인증

### §3-1. 보안 토큰·스마트카드·HSM

소유 기반 인증의 본질은 **사용자가 물리적으로 보유한 객체에 비밀이 안전하게 저장되어 있고, 그 비밀이 객체 외부로 추출되지 않는다** 는 가정이다.

| 유형 | 설명 | 대표 예시 |
|---|---|---|
| 동기식 OTP 토큰 | 시간 또는 카운터 기반으로 OTP 생성 | RSA SecurID, Google Authenticator |
| 비동기식 OTP 토큰 | 서버가 보낸 챌린지에 응답하여 OTP 생성 | OCRA(RFC 6287) 토큰 |
| 스마트카드 | 마이크로프로세서 + 보안 메모리 내장. 개인키 저장·서명 연산 | 신용카드 EMV, 한국 보안 토큰 |
| USB 보안키 | USB·NFC·BLE 인터페이스의 FIDO2/U2F 인증기 | YubiKey, SoloKey |
| HSM (Hardware Security Module) | 서버측 암호 연산 전용 하드웨어. 키 생성·저장·서명 | Thales Luna, AWS CloudHSM |

> *시험 포인트*: HSM 은 사용자 인증기가 아니라 **서버측 키 보호 장치**다 — 사용자가 직접 들고 다니는 것이 아니라 데이터센터에 설치된다. 시험에서 HSM 을 "OTP 토큰" 류와 같은 줄에 묶어 정답으로 제시한다면 오답이다.

### §3-2. HOTP — RFC 4226

> **HOTP (HMAC-based One-Time Password)**: HMAC-SHA-1 을 기반으로 한 카운터 동기 방식 일회용 패스워드. RFC 4226 (2005) 에서 정의된다.[^rfc-4226]

HOTP 의 핵심 아이디어는 "공유 비밀 K" 와 "카운터 C" 를 HMAC-SHA-1 에 입력해 매번 다른 값을 만들고, 그 값을 사람이 입력 가능한 6~8 자리 숫자로 잘라내는 것이다. 카운터는 인증 성공 시마다 증가한다.

**알고리즘 (RFC 4226 §5.3)**:
```
HOTP(K, C) = Truncate(HMAC-SHA-1(K, C))
```
1. `HMAC-SHA-1(K, C)` 로 20 바이트 해시 생성
2. 해시의 마지막 바이트 하위 4 비트를 오프셋으로 사용
3. 오프셋 위치에서 4 바이트를 추출, 최상위 비트 제거(부호 비트 제거)
4. 모듈러 10^d (d = 6, 7, 8) 로 d 자리 십진수 반환

**보안 가정**: 공유 비밀 K 는 사용자 토큰과 서버에만 존재해야 하며, 카운터 동기 이탈 시(사용자 토큰을 여러 번 눌러 서버보다 앞서 나간 경우) 서버는 작은 윈도우(예: 10) 내에서 검색해 동기화한다.

### §3-3. TOTP — RFC 6238

> **TOTP (Time-based One-Time Password)**: HOTP 의 카운터 C 를 시간 기반 카운터로 대체한 변형. RFC 6238 (2011) 에서 정의된다.[^rfc-6238]

TOTP 의 카운터는 다음과 같이 정의된다(RFC 6238 §4.1):
```
T = floor((Current Unix Time - T0) / X)
TOTP(K) = HOTP(K, T)
```
- `T0`: 카운팅 시작 시점 (보통 Unix epoch 0)
- `X`: 시간 간격 (기본 30 초, RFC 6238 §5.2 권장)
- 시간 동기화가 필수이며, 클라이언트·서버 시계 오차 허용을 위해 보통 `±1 step` (=±30 초) 윈도우 검색

Google Authenticator, Microsoft Authenticator, Authy 등 모바일 OTP 앱은 모두 TOTP 를 구현한다. RFC 6238 §3 은 SHA-1·SHA-256·SHA-512 모두 허용한다고 명시하지만 실제 보급된 구현은 SHA-1 이 압도적이다.

| 항목 | HOTP (RFC 4226) | TOTP (RFC 6238) |
|---|---|---|
| 카운터 원천 | 이벤트(인증 시도) | 시간 |
| 동기화 요구 | 카운터 동기 윈도우 | 시각 동기 (±1 step) |
| 발급 후 미사용 | 무한 유효 (다음 카운터값) | X 초 후 만료 |
| 권장 자릿수 | 6~8 | 6~8 |

> *시험 포인트*: HOTP 와 TOTP 의 차이는 "카운터가 무엇이냐" 한 가지다. HOTP 는 사용자가 토큰을 누른 횟수가 카운터, TOTP 는 현재 시간이 카운터다. 이 외 알고리즘 구조(HMAC + Truncate) 는 동일하다.

### §3-4. S/Key 일회용 패스워드 — RFC 2289

> **S/Key**: Bellcore 가 개발한 해시 체인 기반 일회용 패스워드 시스템. RFC 1760(1995) 으로 처음 정의되고 RFC 2289(1998) 가 표준화 버전이다.[^rfc-2289]

S/Key 의 메커니즘은 OTP 와 다른 발상에서 출발한다 — **단방향 해시함수의 비가역성** 을 이용해 한 번 사용된 OTP 를 그 이전 시점의 OTP 를 추론하는 데 사용할 수 없도록 한다.

**알고리즘 (RFC 2289 §6)**:
1. 사용자가 비밀 패스프레이즈 P 와 시드 S 를 결합 → `seed||P`
2. `seed||P` 에 해시함수(MD5/SHA-1) 를 N 번 반복 적용 → `H^N(seed||P)` 를 서버에 등록
3. 첫 번째 인증 시 사용자는 `H^(N-1)(seed||P)` 를 제출
4. 서버는 제출값에 해시를 한 번 더 적용해 자신이 가진 `H^N` 과 비교
5. 검증 성공 시 서버는 자신이 가진 값을 `H^(N-1)` 로 갱신
6. 다음 인증 시 사용자는 `H^(N-2)` 제출
7. N 이 0 에 도달하면 새 패스프레이즈 + 시드 등록 필요

이 구조의 보안 핵심은 **공격자가 H^k 를 가로채도 H^(k-1) 을 계산할 수 없다** 는 점이다(해시의 일방향성). 다만 RFC 2289 가 권장 해시로 MD5 와 SHA-1 을 명시한 시점이 1998 년이므로, 두 해시가 모두 충돌 저항성이 깨진 현재로서는 학술·시험 대비 가치만 남았다고 볼 수 있다.

> *시험 포인트*: S/Key 는 **카운터가 N → 0 으로 감소**한다. HOTP 의 카운터가 0 → ∞ 로 증가하는 것과 반대 방향이다. 또한 S/Key 는 시간 동기를 요구하지 않는다.

[^rfc-4226]: RFC 4226, *HOTP: An HMAC-Based One-Time Password Algorithm*, M'Raihi et al., 2005. https://www.rfc-editor.org/rfc/rfc4226
[^rfc-6238]: RFC 6238, *TOTP: Time-Based One-Time Password Algorithm*, M'Raihi et al., 2011. https://www.rfc-editor.org/rfc/rfc6238
[^rfc-2289]: RFC 2289, *A One-Time Password System*, Haller et al., 1998. https://www.rfc-editor.org/rfc/rfc2289

## §4. 생체 기반 인증

### §4-1. 생체 인식 기법

> **생체 인증 (Biometric authentication)**: 사람의 신체적·행동적 특성을 측정하여 신원을 검증하는 방식. ISO/IEC 19794 시리즈가 데이터 포맷을, ISO/IEC 19795 시리즈가 성능 평가를 표준화한다.[^iso-19794][^iso-19795]

생체 인증의 본질적 특성은 **재발급 불가능성**이다 — 패스워드는 노출되면 변경할 수 있지만, 지문이 한 번 노출되면 사용자의 손가락을 바꿀 수는 없다. 이로 인해 생체 데이터의 저장·전송에는 패스워드보다 더 엄격한 보호(템플릿 보호, 디바이스 내 처리 원칙) 가 요구된다.

| 기법 | 측정 대상 | 특징 |
|---|---|---|
| 지문 | 손가락 마루·골 패턴(minutiae) | 보급률 1위, 모바일 표준. 절단·복제 위험 |
| 홍채 | 눈동자 무늬 | 정확도 매우 높음. 일란성 쌍둥이도 다름 |
| 얼굴 | 얼굴 특징점·기하 | 비접촉. 조명·각도·마스크에 민감 |
| 정맥 | 손바닥·손가락 정맥 패턴 | 위·변조 어려움. 의료 기기 형태 |
| 음성 | 발화 특성 | 비대면 가능. 녹음 재생 공격 가능 |
| 서명 | 필기 동작·압력·속도 | 행동 기반. 변동성 큼 |
| 키 입력 패턴 | 타이핑 리듬·키 압력 | 행동 기반. 부가 인증으로 사용 |

### §4-2. 평가 지표 — FAR·FRR·EER·CER

생체 인증 시스템의 성능은 본질적으로 두 가지 오류율로 표현된다:

- **FAR (False Accept Rate, 오인식률)**: 타인을 본인으로 잘못 인식할 확률. 보안 위협 직결.
- **FRR (False Reject Rate, 오거부율)**: 본인을 타인으로 잘못 거부할 확률. 사용성 직결.

이 둘은 **임계치(threshold) 를 조정하면 한쪽이 줄고 다른 쪽이 늘어나는 트레이드오프 관계** 에 있다. 임계치를 엄격하게 잡으면 FAR 은 낮아지지만 FRR 이 높아져 본인이 자주 거부된다. 임계치를 느슨하게 잡으면 FRR 은 낮아지지만 FAR 이 높아져 보안이 약해진다.

이 트레이드오프 곡선상에서 두 값이 같아지는 지점을 **EER (Equal Error Rate, 동일 오류율)** 또는 **CER (Crossover Error Rate, 교차 오류율)** 이라 부르며, 두 용어는 동의어다.[^iso-19795] EER 은 시스템 간 비교를 위한 단일 척도로 사용되며, EER 이 낮을수록 시스템이 더 정확하다.

> *시험 포인트*: FAR 과 FRR 은 임계치에 반비례 관계로 변화하며, 임계치를 보안 강화 방향으로 조정하면 FAR ↓ FRR ↑, 사용성 강화 방향으로 조정하면 FAR ↑ FRR ↓. EER = CER 이며 두 그래프의 교차점이다. 기관에 따라 FAR 대신 FMR(False Match Rate), FRR 대신 FNMR(False Non-Match Rate) 을 쓰기도 하는데 시험 출제 시 동의어로 처리되는 것이 일반적이다.

### §4-3. 위치 기반·행동 기반 인증

위치 기반 인증(GPS, IP 지리정보) 과 행동 기반 인증(타이핑 리듬, 마우스 움직임, 통신 패턴) 은 NIST SP 800-63B 의 정식 3요소가 아니라 **부가적 위험 기반 인증(adaptive·risk-based authentication)** 으로 분류된다.[^nist-63b] 이 방식들은 단독으로 인증을 결정하기보다는 — "평소와 다른 위치에서 접속하면 추가 OTP 요구" 같은 — 본 인증의 보조 신호로 사용한다.

[^iso-19794]: ISO/IEC 19794 시리즈, *Information technology — Biometric data interchange formats*. https://www.iso.org/standard/50862.html
[^iso-19795]: ISO/IEC 19795-1, *Information technology — Biometric performance testing and reporting — Part 1: Principles and framework*. https://www.iso.org/standard/41447.html

## §5. 통합 인증과 연합 인증

### §5-1. SSO 개념과 분류

> **SSO (Single Sign-On)**: 사용자가 한 번 인증하면 여러 서비스에 별도 로그인 없이 접근할 수 있도록 하는 통합 인증 메커니즘.

SSO 는 **인증 부담 감소(사용자 경험)** 과 **계정 관리 일원화(관리자 경험)** 를 동시에 해결한다. 사용자는 하나의 자격증명만 기억하면 되고, 관리자는 사용자가 퇴직할 때 한 곳에서만 비활성화하면 모든 서비스에서 접근이 차단된다.

SSO 는 신뢰 경계에 따라 두 부류로 나뉜다:
- **엔터프라이즈 SSO (Enterprise SSO)** — 동일 조직 내부의 여러 시스템을 하나의 인증으로 연결. Kerberos, AD 통합 인증이 대표. 한국의 EAM(Enterprise Access Management) · IAM(Identity Access Management) 솔루션이 이를 구현한다.
- **연합 SSO (Federated SSO)** — 서로 다른 조직(도메인) 간 신뢰를 확립해 한쪽 조직에서 인증한 사용자가 다른 조직 서비스에 접근할 수 있게 한다. SAML, OIDC 가 대표 표준.

### §5-2. Kerberos v5 — RFC 4120

> **Kerberos**: MIT 가 개발한 대칭키 기반 네트워크 인증 프로토콜. v5 는 RFC 4120 (2005) 으로 표준화되었다.[^rfc-4120]

Kerberos 는 "**클라이언트와 서버가 서로 사전에 비밀을 공유하지 않은 상태에서**, 신뢰할 수 있는 제3자(KDC) 가 발급한 티켓을 통해 상호 인증한다" 는 발상 위에 있다. 이 발상의 매력은 N 명의 사용자와 M 개의 서비스가 있을 때 N×M 개의 비밀이 아니라 N+M 개의 비밀(KDC 와 각 주체 간) 만 관리하면 된다는 점이다.

**구성 요소 (RFC 4120 §1.7 Glossary of Terms)**:
- **클라이언트(C)** — 인증을 요청하는 사용자
- **AS (Authentication Server)** — 사용자 신원을 검증하고 TGT 를 발급
- **TGS (Ticket Granting Server)** — TGT 를 받아 서비스별 ST(Service Ticket) 를 발급
- **서비스 서버(S)** — 사용자가 접근하려는 실제 응용 서버
- **KDC (Key Distribution Center)** — AS + TGS 의 통합 명칭
- **TGT (Ticket Granting Ticket)** — AS 가 발급한 "TGS 에 접근할 권한" 티켓
- **ST (Service Ticket)** — TGS 가 발급한 "특정 서비스 서버에 접근할 권한" 티켓

**프로토콜 흐름 (RFC 4120 §3.1 AS Exchange · §3.3 TGS Exchange · §3.2 AP Exchange)**:
```
[1] AS-REQ : C → AS    : "나는 alice@DOMAIN, 시간 T 의 인증 요청"
[2] AS-REP : AS → C    : { TGT, K_C-TGS } 를 K_C 로 암호화하여 회신
                          (K_C = 사용자 비밀번호 기반 해시키)
[3] TGS-REQ: C → TGS   : { TGT, Authenticator(K_C-TGS) } + 서비스명
[4] TGS-REP: TGS → C   : { ST, K_C-S } 를 K_C-TGS 로 암호화하여 회신
[5] AP-REQ : C → S     : { ST, Authenticator(K_C-S) }
[6] AP-REP : S → C     : { 타임스탬프 } 를 K_C-S 로 암호화하여 회신 (상호 인증 시)
```

Kerberos 가 갖는 핵심 보안 속성들:
- **사용자 패스워드는 네트워크에 송신되지 않는다** — `K_C` 는 클라이언트 측에서 생성되며, AS 는 자신이 보유한 동일한 `K_C` 로 응답을 암호화한다.
- **재전송 공격 방지** — Authenticator 에 타임스탬프를 포함하고, KDC·서비스 서버는 5 분 이내(기본값) 의 시각 차이만 허용한다(따라서 시각 동기 NTP 가 필수).
- **세션 키 분리** — 모든 통신 단계에서 단기 세션 키(K_C-TGS, K_C-S) 를 사용해 장기 키 노출을 최소화한다.

> *시험 포인트*: Kerberos 의 약점은 ① **시각 동기 의존성**(NTP 필수), ② **KDC 단일 장애점**(KDC 침해 시 전체 시스템 침해), ③ **사용자 패스워드 사전 공격에 취약**(K_C 가 패스워드 해시 기반이므로 오프라인 공격 가능) 이다. 시험에서 Kerberos 의 단점으로 출제되는 것은 보통 이 세 가지다.

> *cross-ref*: Kerberos 의 키 분배 측면은 [03 단원 §2-4](./03.key-distribution.md) 에서 키 분배 프로토콜의 한 사례로 다시 다룬다.

### §5-3. SAML 2.0

> **SAML (Security Assertion Markup Language)**: XML 기반 연합 인증·인가 표준. v2.0 은 OASIS 가 2005 년 표준화했다.[^saml-core]

SAML 은 **3 자 모델(IdP·SP·사용자)** 로 동작한다:
- **IdP (Identity Provider)** — 사용자 인증을 수행하고 인증 결과를 SAML Assertion 으로 발급
- **SP (Service Provider)** — 사용자가 실제 사용하려는 서비스. IdP 의 Assertion 을 신뢰
- **사용자(User Agent)** — 일반적으로 웹 브라우저

**SP-initiated SSO 흐름**:
1. 사용자가 SP 에 접근 → SP 가 SAML AuthnRequest 를 IdP 로 리다이렉트
2. IdP 가 사용자 인증 (패스워드·OTP 등)
3. IdP 가 SAML Assertion 을 디지털서명하여 사용자 브라우저를 통해 SP 로 회신 (HTTP POST 바인딩)
4. SP 가 서명을 검증하고 사용자 세션 생성

SAML Assertion 은 XML 문서이며, **IdP 의 디지털서명**(XML Signature) 과 선택적으로 **암호화**(XML Encryption) 가 적용된다. SP 는 IdP 의 공개키로 서명을 검증하므로 SP 와 IdP 는 사전 메타데이터 교환을 통해 신뢰 관계를 수립한다.

> *시험 포인트*: SAML 은 인증(Authentication) 과 인가(Authorization) 정보를 모두 Assertion 에 담을 수 있다. SAML Assertion 의 세 종류는 ① Authentication Statement, ② Attribute Statement, ③ Authorization Decision Statement 다.

### §5-4. OAuth 2.0 — RFC 6749

> **OAuth 2.0**: 사용자의 자원에 대한 제3자 응용의 제한된 접근을 위임하는 **인가(authorization) 프레임워크**. RFC 6749 (2012) 가 표준이다.[^rfc-6749]

OAuth 2.0 의 가장 중요한 사실은 — **이는 인증 프로토콜이 아니라 인가 프레임워크다**. RFC 6749 §1 첫 문단은 명시적으로 "authorization framework" 라고 규정한다. 그러나 실무에서는 OAuth 2.0 을 인증 용도로 오용하는 사례가 많아 인증 전용 표준으로 OIDC(§5-5) 가 등장했다.

**4 가지 역할 (RFC 6749 §1.1)**:
- **Resource Owner** — 자원의 소유자(보통 사용자)
- **Resource Server** — 자원을 호스팅하는 서버
- **Client** — 자원에 접근하려는 제3자 응용
- **Authorization Server** — 인증·동의 후 액세스 토큰 발급

**4 가지 권한 부여 방식 (RFC 6749 §1.3·4)**:
| 방식 | 영문 | 사용 사례 |
|---|---|---|
| 인가 코드 | Authorization Code | 서버측 웹 응용 (가장 안전·권장) |
| 암묵적 부여 | Implicit | 브라우저측 SPA — RFC 6749 에 정의되었으나 OAuth 2.1 에서 제거 권고 |
| 자원 소유자 패스워드 자격 | Resource Owner Password Credentials | 신뢰 가능한 1자 응용에 한정 (권장 안 함) |
| 클라이언트 자격 | Client Credentials | 사용자 컨텍스트 없는 서버-서버 통신 |

**액세스 토큰 (RFC 6750)**: 발급된 토큰은 Bearer Token 형식으로 HTTP `Authorization: Bearer <token>` 헤더에 실어 자원 서버에 제출한다.[^rfc-6750] Bearer 의 의미는 "**토큰을 가진 자가 자격이 있다**" — 즉 토큰 소유 자체가 인가 증명이며, 따라서 토큰 누설은 즉시 권한 도용을 의미한다.

> *시험 포인트*: OAuth 2.0 은 **인가**, OIDC 는 **인증**. OAuth 2.0 으로 "사용자가 누구인지" 를 검증하는 것은 표준 사용 의도가 아니다. 시험에서 이 둘의 구분을 자주 묻는다.

### §5-5. OpenID Connect 1.0

> **OpenID Connect (OIDC) 1.0**: OAuth 2.0 위에 인증 레이어를 추가한 표준. OpenID Foundation 이 2014 년 표준화했다.[^oidc-core]

OIDC 의 핵심 추가 요소는 **ID 토큰 (id_token)** 이다. 이는 JWT(JSON Web Token, RFC 7519) 형식으로 사용자 신원 정보를 담고 있으며, IdP(OIDC 에서는 OP — OpenID Provider) 가 디지털서명한다. 클라이언트(RP — Relying Party) 는 ID 토큰의 서명을 검증하여 사용자 인증 사실을 확인한다.

**OIDC 흐름 (OpenID Connect Core 1.0 §3.1, Authorization Code Flow)**:
1. RP 가 사용자를 OP 의 인증 엔드포인트로 리다이렉트 (OAuth 2.0 인가 코드 요청 + `scope=openid`)
2. OP 가 사용자 인증
3. OP 가 RP 로 인가 코드 회신
4. RP 가 토큰 엔드포인트에 인가 코드 제출
5. OP 가 액세스 토큰 + **ID 토큰** 회신
6. RP 가 ID 토큰의 서명을 OP 의 공개키(JWKS) 로 검증

ID 토큰의 표준 클레임(OIDC Core §2):
- `iss` (Issuer) — OP 식별자
- `sub` (Subject) — 사용자 고유 식별자
- `aud` (Audience) — RP 식별자
- `exp` (Expiration) — 토큰 만료
- `iat` (Issued At) — 발급 시각
- `auth_time` — 사용자 인증 시각

> *시험 포인트*: OIDC 는 OAuth 2.0 의 **확장**이며 OAuth 2.0 의 인가 코드 흐름 위에 ID 토큰만 추가된 구조다. 따라서 OIDC 클라이언트는 자동으로 OAuth 2.0 클라이언트이기도 하다(역은 성립하지 않음).

### §5-6. FIDO2 — WebAuthn + CTAP2

> **FIDO2**: 패스워드 없는 (passwordless) 강력한 인증을 위한 표준 집합. **WebAuthn** (W3C, 2021 Level 2 권고) 과 **CTAP2** (FIDO Alliance, 2019) 두 표준으로 구성된다.[^webauthn][^ctap2]

FIDO2 는 패스워드 인증의 근본적 약점(서버 침해 시 사용자 비밀이 일괄 노출, 피싱 취약) 을 해결하기 위해 설계되었다. 핵심 발상은 — **공유 비밀(symmetric secret) 을 폐지하고 공개키 기반 챌린지-응답으로 대체**한다. 사용자 디바이스가 사이트별로 별도 키쌍을 생성하고, 개인키는 디바이스를 떠나지 않으며, 서버는 공개키만 보관한다.

**구성**:
- **WebAuthn** — 브라우저(웹 응용) ↔ 인증기 사이의 JavaScript API 표준 (W3C)
- **CTAP2 (Client to Authenticator Protocol)** — 브라우저(또는 OS) ↔ 외부 인증기(USB·NFC·BLE) 사이의 프로토콜 (FIDO Alliance)

**등록 흐름 (WebAuthn Level 2 §7.1 Registering a New Credential)**:
1. RP 가 챌린지·사용자 정보·RP 정보를 브라우저에 전달
2. 브라우저가 인증기에게 키쌍 생성 요청 (CTAP2 makeCredential)
3. 인증기가 사용자 인증(생체·PIN) 수행
4. 인증기가 키쌍 생성, 개인키는 디바이스 내부 저장, 공개키 + Attestation 을 RP 에 회신
5. RP 가 공개키와 사용자 식별자를 등록 데이터베이스에 저장

**인증 흐름 (WebAuthn Level 2 §7.2 Verifying an Authentication Assertion)**:
1. RP 가 챌린지를 브라우저에 전달
2. 브라우저가 인증기에게 서명 요청 (CTAP2 getAssertion)
3. 인증기가 사용자 인증 후 등록된 개인키로 챌린지 + RP origin 등을 서명
4. RP 가 등록된 공개키로 서명 검증

**FIDO2 의 보안 속성**:
- **피싱 저항** — 서명에 RP origin 이 포함되므로 가짜 사이트는 정당한 서명을 만들지 못함
- **서버 비밀 부재** — 서버 침해 시에도 공개키만 노출되므로 사용자 도용 불가
- **재사용 불가** — 매번 새 챌린지에 대한 서명이므로 재전송 무력화
- **사이트별 키 분리** — 사이트마다 별도 키쌍이므로 한 사이트 침해가 다른 사이트로 전파되지 않음

> *시험 포인트*: FIDO2 는 NIST SP 800-63B AAL3 의 "phishing resistance" 요건을 충족하는 대표 인증기 형태다. 실무에서 "FIDO2" 는 보통 USB 보안키 + 디바이스 PIN/생체 의 2요소 조합을 의미하며, 단요소 FIDO2 도 표준상 가능하지만 권장되지 않는다.

## §6. 메시지 인증

### §6-1. 메시지 인증의 요구사항

> **메시지 인증 (Message Authentication)**: 수신한 메시지가 ① 주장된 송신자로부터 왔으며(출처 인증·data origin authentication), ② 전송 중 변경되지 않았음(무결성·integrity) 을 검증하는 과정.[^fips-198]

사용자 인증이 "이 시스템 앞에 누가 앉아 있는가" 를 다룬다면, 메시지 인증은 "방금 도착한 이 데이터가 누가 보낸 것이며 그대로 도착했는가" 를 다룬다. 두 문제는 동일 비밀(공유 키 또는 키쌍) 을 활용하지만 검증 대상이 다르다.

메시지 인증의 보안 요구사항:
1. **무결성** — 메시지 비트 단위 변경을 탐지
2. **출처 인증** — 메시지가 주장된 송신자로부터 왔음을 검증
3. **재전송 방지** — 동일 메시지의 반복 전송 탐지(타임스탬프·시퀀스 번호·논스 결합 필요)

### §6-2. MAC vs 디지털서명 vs 해시

세 메커니즘의 보안 속성을 명확히 구분하는 것이 4과목 시험의 단골 주제다.

| 메커니즘 | 키 구조 | 무결성 | 출처 인증 | 부인 방지 |
|---|---|---|---|---|
| 해시함수 (SHA-256 등) | 키 없음 | O (입력 변경 탐지) | X | X |
| MAC (HMAC·CMAC) | 대칭키(공유) | O | O (제한적) | **X** |
| 디지털서명 (RSA-PSS·ECDSA) | 비대칭키쌍 | O | O | **O** |

**왜 MAC 은 부인 방지가 안 되는가?** — MAC 은 송신자와 수신자가 같은 대칭키 K 를 공유한다. 따라서 검증자(수신자) 도 자신이 가진 K 로 동일한 MAC 값을 만들 수 있다. 즉 어떤 MAC 이 누구의 손에서 나왔는지 — 송신자인지 수신자 본인인지 — 제3자(법정·중재자) 가 판별할 수 없다. 부인 방지는 **검증자가 만들 수 없는 증거** 를 요구하며, 이는 비대칭키쌍의 개인키만이 만들 수 있는 디지털서명에서만 성립한다.

> *시험 포인트*: "MAC 은 부인 방지를 제공하는가?" 답은 **아니오**. 단, MAC 의 출처 인증은 "공유 키를 가진 두 당사자 중 하나가 보냈음" 까지는 보장한다(외부인은 만들 수 없으므로). 이를 넘어선 송신자 특정은 비대칭 서명을 요구한다. 자세한 디지털서명은 [04 단원 §1](./04.digital-signature-pki.md) 참조.

### §6-3. HMAC — RFC 2104·FIPS 198-1

> **HMAC (Hash-based Message Authentication Code)**: 임의의 암호 해시함수 H 를 기반으로 하는 키-기반 MAC. RFC 2104 (1997) 가 정의하고 FIPS 198-1 (2008) 이 미국 정부 표준으로 채택했다.[^rfc-2104][^fips-198]

HMAC 의 설계 목표는 — 기존 해시함수(MD5·SHA-1·SHA-256 등) 를 블랙박스로 사용해 키 기반 인증을 추가하면서, **해시함수 자체의 보안 가정만으로 HMAC 의 보안을 증명** 하는 것이다. 단순히 `H(K || M)` 처럼 키와 메시지를 이어붙여 해시하면 길이 연장 공격(length extension attack) 에 취약하다 — Merkle-Damgård 구조의 해시함수(SHA-1·SHA-2) 는 이 공격에 본질적으로 약하다(자세한 내용은 [06 단원 §4](./06.hash-and-mac.md)).

**HMAC 정의 (RFC 2104 §2, FIPS 198-1)**:
```
HMAC(K, M) = H( (K' XOR opad) || H( (K' XOR ipad) || M ) )
```
- `K'` = K 가 블록 길이보다 짧으면 0 패딩, 길면 H(K) 로 축소한 후 0 패딩한 값
- `ipad` = 0x36 을 블록 길이만큼 반복 (inner pad)
- `opad` = 0x5C 을 블록 길이만큼 반복 (outer pad)
- `||` = 연접

이 이중 해시 구조가 길이 연장 공격을 차단한다 — 외부 해시 `H((K' XOR opad) || …)` 의 결과만 공개되며, 공격자는 그 내부 상태를 모르기 때문이다.

**FIPS 198-1 권장 매개변수**:
- 해시 알고리즘 — FIPS 180-4 의 SHA-2 패밀리 (SHA-224·256·384·512)[^fips-180-4] 또는 FIPS 202 의 SHA-3 패밀리[^fips-202]
- MD5·SHA-1 기반 HMAC 은 NIST SP 800-131A 에 따라 폐기 또는 비추천 단계

> *시험 포인트*: HMAC-SHA-256 의 출력 길이는 256 bit (=32 byte). HMAC 의 키 길이는 임의이며 RFC 2104 §2 는 "최소 해시 출력 길이 이상" 을 권장한다(SHA-256 → 256 bit). HMAC 의 ipad·opad 값(0x36·0x5C) 은 시험에 출제되는 단골 수치다.

> *cross-ref*: CMAC (NIST SP 800-38B) 와 GMAC (SP 800-38D) 등 대칭키 블록 암호 기반 MAC 은 [06 단원 §5](./06.hash-and-mac.md) 에서 다룬다.

## §7. 디바이스 인증

### §7-1. 시도-응답 인증

> **시도-응답 인증 (Challenge-Response Authentication)**: 검증자가 무작위 챌린지를 보내고, 신청자가 자신의 비밀로 챌린지를 처리한 응답을 회신함으로써 신원을 증명하는 일반적 인증 패턴.

디바이스 인증의 거의 모든 프로토콜은 이 패턴 위에 있다. 비밀의 형태는 ① 공유 대칭키, ② 비대칭 키쌍의 개인키, ③ 패스워드 해시 중 하나다. 응답을 만드는 연산은 각각 ① MAC·HMAC, ② 디지털서명, ③ 해시 결합이다.

핵심 보안 속성은 **재전송 방지**다 — 챌린지가 매번 새로운 무작위 값(논스·nonce)이므로 공격자가 이전 응답을 재사용해도 다음 챌린지에 대한 정답이 되지 못한다.

### §7-2. 상호 인증

> **상호 인증 (Mutual Authentication)**: 통신 양방향이 서로의 신원을 동시에 검증하는 인증.

일방향 인증(서버만 검증) 은 사용자가 가짜 서버에 자격증명을 제공할 위험을 남긴다. 상호 인증은 양쪽 모두 챌린지-응답을 교환해 서로의 신원을 동시에 확인한다. TLS 의 **mutual TLS (mTLS)** 가 대표 사례이며 — 서버 인증서 + 클라이언트 인증서를 모두 검증한다. Kerberos 의 AP-REP 단계도 서비스 서버가 자신을 클라이언트에게 입증하는 단계로 상호 인증을 제공한다(§5-2 [6] 단계).

### §7-3. IEEE 802.1X-2020 + EAP — RFC 3748

> **IEEE 802.1X**: 포트 기반 네트워크 접근 제어(Port-based Network Access Control, PNAC) 표준. 2020 년판이 현행이다.[^ieee-8021x]
> **EAP (Extensible Authentication Protocol)**: 802.1X 위에서 다양한 인증 방식을 캡슐화하는 인증 프레임워크. RFC 3748 (2004) 이 정의한다.[^rfc-3748]

802.1X 는 **유선 LAN 의 이더넷 포트** 와 **무선 LAN 의 SSID** 모두에서 동작한다. 디바이스가 네트워크에 진입할 때 인증을 통과해야 포트가 열리고, 통과하지 못하면 EAPoL(EAP over LAN) 트래픽 외 모든 트래픽이 차단된다.

**3 자 모델 (IEEE 802.1X §6)**:
- **Supplicant** — 인증을 요청하는 디바이스 (PC·스마트폰)
- **Authenticator** — 네트워크 접근 제어 장치 (스위치·AP)
- **Authentication Server** — 실제 인증 검증을 수행 (보통 RADIUS 서버, RFC 2865)

**EAP 메시지 흐름**:
```
1. Authenticator → Supplicant : EAP-Request/Identity
2. Supplicant → Authenticator : EAP-Response/Identity
3. Authenticator → AuthServer : RADIUS Access-Request (EAP 캡슐화)
4. AuthServer ↔ Supplicant   : EAP 메서드별 인증 (TLS·TTLS·MD5·…)
5. AuthServer → Authenticator : RADIUS Access-Accept + 세션 키
6. Authenticator → Supplicant : EAP-Success
7. 포트 개방 + 4-way handshake (WPA2/WPA3 의 경우)
```

### §7-4. EAP 메서드 비교

EAP 는 인증 메서드를 캡슐화할 뿐 인증 자체를 정의하지 않는다. 실제 인증은 EAP 메서드(EAP-TLS·EAP-TTLS·PEAP·EAP-MD5 등) 가 수행한다.

| 메서드 | 표준 | 서버 인증 | 클라이언트 인증 | 특징 |
|---|---|---|---|---|
| EAP-MD5 | RFC 3748 §5.4 | X | 패스워드 해시 (MD5) | 사전 공격·MITM 에 취약. 사용 권장 안 함. |
| EAP-TLS | RFC 5216 | 서버 인증서 | 클라이언트 인증서 | 가장 안전. PKI 인프라 필요. WPA2-Enterprise 표준.[^rfc-5216] |
| EAP-TTLS | RFC 5281 | 서버 인증서 | 다양 (PAP·CHAP·MSCHAPv2·…) | TLS 터널 안에서 레거시 인증 보호.[^rfc-5281] |
| PEAP | IETF 드래프트 (Microsoft·Cisco·RSA) | 서버 인증서 | MSCHAPv2 등 | TTLS 와 유사. 윈도우 환경에서 보급. |

> *시험 포인트*: EAP-TLS 만 양쪽 모두 인증서 기반(상호 인증) 이며 가장 안전하다. EAP-MD5 는 서버 인증을 하지 않으므로 단독 사용 시 무선 환경에서 위험하다. EAP-TTLS·PEAP 는 서버 인증서 + 사용자 패스워드의 절충형이다.

### §7-5. 디바이스 인증서 — X.509

디바이스 인증서는 X.509 v3 인증서 형식을 따르며(RFC 5280[^rfc-5280]), 사용자 인증서와 구조는 동일하지만 **Subject 필드에 디바이스 식별자**(시리얼 번호·MAC 주소·디바이스 ID) 가 들어가는 점이 다르다. EAP-TLS, IPsec IKEv2, mTLS 모두 디바이스 인증서를 사용한다.

> *cross-ref*: X.509 v3 인증서 구조·확장(SAN·KeyUsage·ExtendedKeyUsage·BasicConstraints) 와 PKI 운영(CA·CRL·OCSP) 은 [04 단원 §3·§4](./04.digital-signature-pki.md) 에서 본격 다룬다.

## §8. 단원 요약

이 단원은 인증의 세 영역(사용자·메시지·디바이스) 을 다루었다. 핵심 메시지를 정리하면:

1. **인증의 3요소** — 지식·소유·존재. 다요소 인증은 서로 다른 요소의 결합이며, 같은 요소의 반복은 다요소가 아니다.
2. **NIST SP 800-63B AAL** — AAL1(단요소) · AAL2(다요소) · AAL3(하드웨어 + 검증자 임퍼스네이션 저항).
3. **OTP 의 두 변형** — HOTP(RFC 4226, 카운터 기반) · TOTP(RFC 6238, 시간 기반). 알고리즘 본체는 HMAC-SHA-1 + Truncate 로 동일.
4. **S/Key (RFC 2289)** — 해시 체인의 비가역성. 카운터 N → 0 으로 감소.
5. **생체 인증의 평가** — FAR · FRR 트레이드오프, EER = CER 교차점. EER 이 낮을수록 정확.
6. **Kerberos v5 (RFC 4120)** — KDC = AS + TGS, TGT + ST 의 2단 티켓. 시각 동기 의존, KDC 단일 장애점, 패스워드 사전 공격이 약점.
7. **연합 인증** — SAML 2.0 (XML 기반), OAuth 2.0 (RFC 6749, **인가**), OIDC 1.0 (OAuth 2.0 + ID 토큰, **인증**), FIDO2 (WebAuthn + CTAP2, 패스워드리스).
8. **MAC vs 디지털서명** — MAC 은 부인 방지 X (대칭키 공유), 디지털서명 O (개인키 단독 보유). HMAC = `H((K XOR opad) || H((K XOR ipad) || M))`, ipad=0x36, opad=0x5C.
9. **802.1X + EAP** — Supplicant · Authenticator · AuthServer 3자 모델. EAP-TLS 가 가장 안전(상호 인증), EAP-MD5 는 권장 안 함.

## §9. 시험 함정 정리

| 함정 | 정답 |
|---|---|
| 패스워드 + 비밀 질문 = 다요소 인증? | **단요소** (둘 다 지식 기반). 다요소가 되려면 서로 다른 요소를 결합해야 함. |
| HOTP 와 TOTP 의 차이? | 카운터의 원천 — HOTP 는 이벤트 카운터, TOTP 는 시간 카운터. 알고리즘 구조는 동일. |
| S/Key 의 카운터 방향? | N → 0 으로 **감소**. HOTP 의 0 → ∞ 증가와 반대. |
| EER 과 CER 의 관계? | **동의어** (Equal Error Rate = Crossover Error Rate). FAR·FRR 그래프의 교차점. |
| Kerberos 의 가장 큰 단점? | ① 시각 동기 필수, ② KDC 단일 장애점, ③ 패스워드 사전 공격 가능. |
| OAuth 2.0 으로 사용자 인증? | OAuth 2.0 은 **인가** 프레임워크. 인증 용도는 OIDC. |
| MAC 은 부인 방지를 제공하는가? | **아니오**. 대칭키 공유로 검증자도 만들 수 있어 제3자 입증 불가. |
| HMAC 의 ipad·opad 값? | **ipad=0x36, opad=0x5C**, 블록 길이만큼 반복. |
| EAP 메서드 중 가장 안전한 것? | **EAP-TLS** (양방향 인증서 기반). EAP-MD5 는 서버 인증 부재. |
| FIDO2 의 핵심 이점? | ① 피싱 저항 (RP origin 서명 포함), ② 서버 비밀 부재, ③ 사이트별 키 분리. AAL3 충족 인증기. |
| HSM 은 사용자 인증기? | **아니오**. 서버측 키 보호 장치 — 데이터센터에 설치. |
