# 01. 윈도우 기본

> **이 문서가 다루는 것**: 윈도우 시스템이 사용자를 식별·인증하는 구조, 계정과 그룹 체계, 파일 공유와 암호화, 이벤트 로그, 실행 파일 포맷(PE), 레지스트리, 그리고 기본 파일시스템(NTFS)까지. 윈도우 보안의 토대가 되는 개념을 한 단원에 모았다.
>
> **출제기준 매핑**: 1.정보 시스템의 범위 및 이해 — 단말/서버 시스템, 운영체제(Windows).
>
> **원본 매핑**: 통합자료 line 54 ~ 264, 1과목.txt line 1 ~ 25.

## 0. 들어가며 — 왜 "윈도우 기본" 이 첫 단원인가

시스템 보안의 모든 통제는 결국 한 가지 질문 — **"누가, 어떤 자원에, 어떻게 접근할 수 있는가?"** — 에 답하는 일이다. 그리고 이 질문에 답할 수 있으려면 운영체제가 *사용자를 어떻게 식별·인증하고*, *그 사용자에게 어떤 단위로 권한을 부여하며*, *어떤 자원을 어떤 구조로 보관하는지* 부터 알아야 한다.

윈도우는 시험 출제기준에서 명시한 단말 OS · 서버 OS 모두에 해당하는 가장 보편적인 시스템이며, 이 단원에서 다루는 12개 토픽은 다음 단원들의 토대가 된다.

- [02. 윈도우 관리](./02.windows-management.md) 의 보안 정책은 모두 이 단원의 *구조* 위에서 설정된다.
- [06. 시스템 공격기법](./06.system-attacks.md) 은 이 구조의 *허점* 을 노린다.
- [07. 시스템 보안](./07.system-defense.md) 의 분석 도구·솔루션은 이 구조의 *흔적* 을 읽는다.

따라서 12개 토픽은 외워야 할 단편 지식이 아니라, **이후 모든 단원이 참조할 공용 어휘** 다.

## 1. 윈도우 인증 구성요소

### 배경

윈도우는 운영체제 내부에 인증·인가·감사를 분담하는 세 가지 보안 컴포넌트를 둔다. 사용자가 로그인할 때 어떤 컴포넌트가 어떤 역할을 맡는지 이해하면, 이후 등장하는 SID·이벤트 로그·SAM 파일 보호의 의미가 자연스럽게 연결된다.

### 정의와 동작 원리


| 컴포넌트    | 약어 풀이                      | 역할                                                                                                                                |
| ------- | -------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| **LSA** | Local Security Authority   | 모든 계정의 로그인을 검증하고 시스템 자원에 대한 접근 권한을 검사한다. 로컬 로그인뿐 아니라 원격 로그인까지 포함한다.=> 전체 계정 검증, 접근 권한 검사                                         |
| **SAM** | Security Account Manager   | 사용자 로그인 정보와 SAM 파일에 저장된 사용자 패스워드 정보를 비교해 인증 여부를 결정한다. SAM 파일은 사용자·그룹 계정의 패스워드를 관리하므로, LSA 인증의 핵심이며 적절한 접근통제가 필요하다.=> 계정 관리, 패스워드 |
| **SRM** | Security Reference Monitor | 인증된 사용자에게 SID(Security Identifier)를 부여한다. SID 를 기반으로 파일·디렉터리에 대한 접근 허용 여부를 결정하고 감사 메시지를 생성한다.=> 감사 메시지 생성                        |


### 시험 포인트

- "패스워드 검증 → SAM, 권한 부여·감사 → SRM, 전체 검증 → LSA" 의 역할 분담을 묻는 문항이 자주 출제된다.
- SAM 파일이 노출되면 모든 계정 해시가 함께 노출되므로, 보호의 핵심 대상임을 기억한다.

## 2. 윈도우 인증 방법

### 배경

윈도우 인증은 단일 PC 환경(워크그룹)과 기업 환경(도메인)에서 다른 경로를 탄다. 시험에서는 두 경로가 사용하는 프로토콜과 저장 위치를 구분해서 묻는다.

### 정의와 동작 원리


| 인증 방법           | 동작                                                                                                                                                                                                     |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **로컬(워크그룹) 인증** | 로그인 화면에서 입력한 ID·패스워드를 LSA 서브시스템이 받는다. LSA 는 NTLM 모듈에 인증 정보를 넘기고, NTLM 은 다시 SAM 으로 전달해 로그인을 처리한다. **계정 정보는 SAM 데이터베이스에 저장**된다.                                                                          |
| **원격(도메인) 인증**  | LSA 서브시스템이 인증 정보를 받아 로컬 인증용인지 도메인 인증용인지 판별한 뒤, 도메인 인증이면 **커버로스(Kerberos) 프로토콜**을 사용해 도메인 컨트롤러에 인증을 요청한다. 도메인 컨트롤러는 인증 결과로 접근 토큰(Access Token)을 부여하고, 이 권한으로 사용자 프로세스가 실행된다. **계정 정보는 액티브 디렉터리에 저장**된다. |


### 관련 용어

- **워크그룹(Workgroup)** — 중앙 인증 서버 없이 각 PC 가 자체 계정 DB(SAM)를 갖는 소규모 네트워크 구성.
- **도메인(Domain)** — 하나의 도메인 컨트롤러(Domain Controller)가 액티브 디렉터리(AD)로 계정·정책을 일괄 관리하는 구성.
- **Kerberos** — 티켓 기반 네트워크 인증 프로토콜. AD 환경의 표준 인증 프로토콜이다[^1].

### 시험 포인트

- 로컬 ↔ SAM, 도메인 ↔ Kerberos·AD 의 매핑을 정확히 외운다.

## 3. 보안 식별자(SID)

### 배경

윈도우는 사용자 이름이 아니라 **고유한 식별번호** 로 권한을 판단한다. 같은 이름이라도 SID 가 다르면 다른 주체이며, 이는 사용자 계정을 삭제 후 재생성해도 이전 권한이 그대로 살아나지 않는 이유다.

### 정의와 동작 원리

윈도우 보안 식별자(**SID, Security Identifier**) 는 각 사용자나 그룹에 부여되는 고유 식별번호이며, **SAM 파일에 저장**된다.

#### SID 구조

```
[ 시스템 ][ 도메인 컨트롤러 또는 단독 시스템 ][ 시스템 고유 식별자 ][ 사용자 식별자 ]
```

#### 사용자 식별자(RID) 의 의미


| 값       | 의미                 |
| ------- | ------------------ |
| 500     | 관리자(Administrator) |
| 501     | 게스트(Guest)         |
| 1000 이상 | 일반 사용자             |


### 명령어 — SID 확인

```cmd
> wmic
wmic:root\cli> useraccount list brief
```

WMIC(Windows Management Instrumentation Console, 윈도우 관리 명령 콘솔)에서 위 명령으로 SID 를 조회할 수 있다.

### 시험 포인트

- "RID 500 = Administrator" 는 단골 출제 포인트다. Administrator 의 *계정 이름* 을 변경해도 RID 500 은 바뀌지 않으므로, 공격자는 이름이 아닌 RID 로 관리자 계정을 식별할 수 있다.

## 4. 인증 구조 — Challenge & Response

### 배경

평문 패스워드를 네트워크로 전송하면 도청에 무방비다. 윈도우는 이를 방지하기 위해 사용자가 가진 비밀(패스워드)을 직접 보내지 않고, 서버가 던진 임의 값(Challenge)에 대한 응답(Response)만 주고받는 방식을 쓴다.

### 정의와 동작 원리

윈도우 인증은 다음 4단계의 **Challenge & Response** 흐름으로 진행된다.

1. **인증 요청** — 사용자가 윈도우 시스템에 인증을 요청한다.
2. **Challenge 값 생성·전송** — 윈도우가 임의의 Challenge 값을 생성해 사용자에게 보낸다.
3. **Response 값 생성·전송** — 사용자는 자신의 패스워드 해시로 Challenge 를 가공한 Response 값을 생성해 보낸다.
4. **Response 값 확인·검증** — 윈도우는 사용자가 보낸 값을 자체 계산 결과와 비교해 성공 여부를 통보한다.

### 시험 포인트

- 패스워드 자체가 아니라 "Challenge 에 대한 Response" 가 오간다는 점이 핵심이다.

## 5. 인증 암호 알고리즘

### 배경

Challenge & Response 의 안전성은 어떤 해시 알고리즘으로 Response 를 생성하느냐에 좌우된다. 윈도우는 호환성 유지를 위해 약한 알고리즘을 오랫동안 남겨 뒀고, 이는 곧 약한 인증 → 패스워드 크래킹 위험으로 이어졌다.

### 정의와 동작 원리


| 알고리즘            | 설명                                                                                                                         |
| --------------- | -------------------------------------------------------------------------------------------------------------------------- |
| **LM 해시**       | Windows Vista 이전 운영체제에서 사용하던 알고리즘. 구조적으로 취약해 현재는 사용하지 않는다.                                                                 |
| **NTLM 해시**     | LM 해시에 **MD4 해시가 추가된 형태**.                                                                                                 |
| **NTLMv2 해시**   | Vista 이후 기본 인증 프로토콜로, 현재까지 복잡도가 충분하다고 평가된다.                                                                                |
| **Lan Manager** | 네트워크를 통한 파일·프린터 공유 등 작업 시 인증을 담당하는 서비스. **로컬 보안 정책 → 보안 설정 → 로컬 정책 → 보안 옵션 → "LAN Manager 인증 수준" → NTLMv2 사용**으로 강제할 수 있다. |


### 시험 포인트

- "현재 윈도우의 기본 인증 프로토콜은?" → **NTLMv2**.
- LM 해시는 비활성화 권장.

---

> **여기까지** 인증의 다섯 단계 — 누가 처리하고(LSA·SAM·SRM), 어떤 경로로 흐르며(로컬·도메인), 어떤 식별자로 표현되고(SID), 어떤 절차로 검증되며(Challenge & Response), 어떤 알고리즘이 쓰이는지(LM·NTLM·NTLMv2) — 를 살펴봤다. 이제 인증을 통과한 주체에게 **시스템이 어떤 단위로 권한을 부여하고, 그 주체가 어떤 자원에 접근하는지**(계정·그룹·공유·암호화·로그) 로 시야를 옮긴다.

## 6. 계정과 그룹

### 배경

운영체제는 사용자 개인이 아니라 그룹 단위로 권한을 부여하는 것이 일반적이다. 윈도우는 OS 설치 시 표준화된 **기본 사용자**와 **기본 그룹**을 자동 생성하며, 각 그룹에 부여된 권한 범위를 알아야 침해 가능 면적을 평가할 수 있다.

### 기본 사용자


| 계정                | 설명                                                                      |
| ----------------- | ----------------------------------------------------------------------- |
| **Administrator** | 관리자 권한의 계정.                                                             |
| **SYSTEM**        | 로컬에서 관리자보다 상위 권한을 가진 계정. 일반 사용자가 직접 로그인할 수 없으며 OS 와 서비스가 사용한다.          |
| **Guest**         | 매우 제한적인 권한을 가진 계정. 외부 사용자가 컴퓨터를 잠시 접근할 수 있도록 마련되어 있으나 보안상 사용 안 함이 권장된다. |


### 기본 그룹


| 그룹                    | 권한 범위                                          |
| --------------------- | ---------------------------------------------- |
| **Administrators**    | 도메인 자원이나 로컬 컴퓨터에 대한 모든 권한.                     |
| **Account Operators** | 사용자나 그룹 계정을 관리할 수 있다.                          |
| **Backup Operators**  | 시스템 백업을 위해 모든 시스템 파일·디렉터리에 접근할 수 있다.           |
| **Guests**            | 도메인 사용 권한이 제한된다.                               |
| **Print Operators**   | 도메인 프린터에 접근할 수 있다.                             |
| **Power Users**       | 디렉터리·네트워크 공유, 공용 프로그램 그룹 생성, 컴퓨터 시계 설정 권한이 있다. |
| **Replicator**        | 도메인에 있는 파일을 복제할 수 있다.                          |
| **Server Operators**  | 도메인의 서버를 관리할 수 있다.                             |
| **Users**             | 도메인과 로컬 컴퓨터를 일반적으로 사용하는 그룹.                    |


### 시험 포인트

- "관리자보다 상위 권한"은 **SYSTEM**.
- Backup Operators 는 백업 목적이지만 실질적으로 모든 파일을 읽을 수 있는 강력한 권한을 갖는다.

## 7. 기본 공유 폴더

### 배경

윈도우는 관리·원격 작업을 위해 OS 설치 시 자동으로 일부 폴더를 공유 상태로 만든다. 이 **기본 공유**는 일반 사용자에게 보이지 않지만, 인증된 관리자가 네트워크로 접근할 수 있어 침해 시 추가 공격 통로가 될 수 있다.

### 정의


| 공유명          | 설명                                                                                             |
| ------------ | ---------------------------------------------------------------------------------------------- |
| **C$, D$ 등** | 해당 드라이브 전체에 대한 관리 목적 공유 폴더.                                                                    |
| **ADMIN$**   | 윈도우 설치 폴더(`%SystemRoot%`)에 접근하는 관리 목적 공유 폴더.                                                   |
| **IPC$**     | 동시에 수행될 개별 프로그램을 생성·관리할 수 있게 해 주는 인터페이스(Inter-Process Communication). 명명된 파이프를 통한 원격 관리에 사용된다. |


### 관련 용어

- `**$` 접미사** — 숨김 공유(hidden share)임을 의미한다. 네트워크 탐색 목록에는 노출되지 않지만, 경로(`\\HOST\C$`)를 직접 입력하면 접근 가능하다.

### 시험 포인트

- 기본 공유 4종(C$, D$, ADMIN$, IPC$) 의 용도 구분은 자주 출제된다.
- 운영 단계에서 기본 공유를 비활성화하는 절차는 [02. 윈도우 관리](./02.windows-management.md)에서 다룬다.

## 8. 파일 암호화 — EFS · BitLocker

### 배경

저장 매체 자체가 탈취되는 시나리오(노트북 분실 등)에서는 OS 인증을 우회한 오프라인 분석이 가능하다. 윈도우는 이런 위협에 대응해 두 단계의 디스크 암호화 솔루션을 제공한다.

### 정의


| 솔루션                              | 보호 범위                                        |
| -------------------------------- | -------------------------------------------- |
| **EFS** (Encrypting File System) | 개별 파일 또는 특정 폴더 안의 모든 파일을 암호화할 수 있는 기능을 제공한다. |
| **BitLocker**                    | **전체 시스템 볼륨의 모든 데이터** 에 대한 암호화 기능을 제공한다.     |


### 시험 포인트

- 단일 파일·폴더 단위 = EFS, 볼륨 전체 = BitLocker.

## 9. 이벤트 로그

### 배경

침해사고 분석은 결국 "무엇이 언제 일어났는가" 의 흔적을 찾는 일이다. 윈도우는 로그를 세 카테고리로 분리해 보관함으로써, 분석가가 응용 프로그램 결함과 보안 이벤트와 시스템 컴포넌트 이슈를 분리해서 볼 수 있게 한다.

### 정의


| 로그 종류          | 기록 내용                                              |
| -------------- | -------------------------------------------------- |
| **응용 프로그램 로그** | 응용 프로그램이 기록한 다양한 이벤트.                              |
| **보안 로그**      | 유효하거나 유효하지 않은 리소스 사용에 관련된 이벤트(로그온 성공·실패, 권한 사용 등). |
| **시스템 로그**     | 윈도우 시스템 구성요소가 기록하는 이벤트(드라이버 로드 실패, 서비스 시작·중지 등).   |


### 시험 포인트

- 어떤 로그에 어떤 이벤트가 남는지 매칭 문제로 자주 나온다.
- 감사 정책(어떤 이벤트를 어디까지 기록할지)은 [02. 윈도우 관리](./02.windows-management.md)에서 다룬다.

---

> **여기까지** 사용자·자원 관점에서 윈도우의 표면을 봤다. 이제 시야를 한 단계 안으로 들여, **운영체제 자체가 무엇으로 만들어져 있는지** — 실행 파일의 포맷(PE), 시스템 설정을 보관하는 데이터베이스(레지스트리), 그리고 이 모든 것이 놓인 파일시스템(NTFS) — 을 살펴본다. 침해사고 분석과 악성코드 분석은 결국 이 세 영역의 흔적을 읽는 일이다.

## 10. 실행 파일 포맷 — PE

### 배경

악성코드 분석·역공학 분야에서는 윈도우 실행 파일(`.exe`, `.dll`)의 내부 구조를 알아야 한다. 윈도우 실행 파일은 모두 동일한 **PE(Portable Executable)** 포맷을 따르며, 파일이 메모리에 어떻게 적재되어 동작하는지 이 포맷을 통해 파악할 수 있다.

### 정의와 동작 원리

윈도우 운영체제에서 사용하는 실행 파일 포맷이 **PE** 다. PE 파일은 다음 세 부분으로 구성된다.

- **PE Header** — 모든 PE 파일이 동일하게 갖는 머리말. 파일 종류·아키텍처·섹션 개수 등 메타데이터를 포함한다.
- **Section Header** — 각 섹션 데이터를 메모리에 로딩하고 속성을 설정하는 데 필요한 정보(메모리 주소, 크기, 권한)를 갖는다. 섹션 헤더와 섹션 자체는 PE 파일마다 다르다.
- **Section** — PE 파일이 가상 주소 공간에 로드된 이후 프로그램 실행 코드, 데이터, 리소스 등 실행에 필요한 정보를 배치하는 영역이다.

### PE 섹션 종류


| 분류         | 섹션명      | 용도                                         |
| ---------- | -------- | ------------------------------------------ |
| 코드         | `.text`  | 프로그램 실행 코드를 담는 섹션.                         |
| 데이터        | `.data`  | 읽기·쓰기 가능한 데이터 섹션.                          |
| 데이터        | `.rdata` | 읽기 전용 데이터 섹션.                              |
| 데이터        | `.bss`   | 초기화되지 않은 전역변수가 위치한다.                       |
| Import API | `.idata` | 임포트할 DLL 과 API/함수에 대한 정보를 담는다.             |
| Import API | `.didat` | 지연 로딩 임포트 데이터를 위한 섹션.                      |
| Export API | `.edata` | 익스포트할 DLL 과 API/함수에 대한 정보를 담는다.            |
| 리소스        | `.rsrc`  | 윈도우 어플리케이션 리소스 관련 데이터(아이콘·문자열 테이블 등)를 담는다. |
| 재배치        | `.reloc` | 실행 파일에 대한 기본 재배치 정보를 담는다.                  |


### 관련 용어

- **DLL(Dynamic Link Library)** — 여러 프로그램이 공유해서 사용하는 함수·자원의 묶음. 임포트(`.idata`)·익스포트(`.edata`) 섹션이 DLL 함수 호출의 메타데이터를 보관한다.
- **재배치(Relocation)** — 실행 파일이 자신이 빌드 시 가정한 메모리 주소가 아닌 다른 주소에 로드될 때 주소를 보정하는 작업. ASLR과 직접 연관된다.

### 시험 포인트

- `.text` = 코드, `.data`/`.rdata`/`.bss` = 데이터(초기화·읽기전용·미초기화) 매칭은 단골 출제.
- `.idata` 임포트 ↔ `.edata` 익스포트.

## 11. 레지스트리

### 배경

레지스트리는 윈도우의 거의 모든 설정을 담는 **계층적 데이터베이스** 다. OS 부팅·서비스 시작·계정 정보·소프트웨어 설정·악성코드 자동 실행 등록까지 전부 레지스트리에 흔적을 남기므로, 사고 분석에서 공격자의 자취를 찾는 첫 번째 위치가 된다.

### 정의

레지스트리는 윈도우 시스템이 운영되는 데 필요한 정보를 담고 있으며, **시스템의 거의 모든 정보**를 보관하므로 사고 분석에서 공격자의 중요한 흔적을 찾을 수 있다. 구조는 다음 두 단계로 단순하게 표현된다.

- **키(Key)** — 최상위 루트 키와 그 하위 서브 키로 구분된다.
- **값(Value)** — 이름, 데이터 유형, 데이터로 구성된다.

### 하이브 파일 (Hive File)

레지스트리 정보를 디스크에 저장한 **물리적 파일**. 윈도우 부팅 시 하이브 파일에서 값을 읽어 메모리상의 레지스트리를 구성한다. 오프라인 상태인 비활성 시스템(예: 분리된 디스크)에서 레지스트리를 분석할 때 활용된다.


| 분류              | 키                |
| --------------- | ---------------- |
| **Master Key**  | HKLM, HKU        |
| **Derived Key** | HKCU, HKCC, HKCR |



| 하이브 파일         | 저장 정보                 |
| -------------- | --------------------- |
| **SAM**        | 계정 정보 및 접속 기록         |
| **SECURITY**   | 시스템 보안 정책 및 권한 정보     |
| **SOFTWARE**   | 시스템에 설치된 모든 어플리케이션 정보 |
| **SYSTEM**     | 시스템 관련 설정 정보          |
| **NTUSER.dat** | 사용자별 환경설정 / 프로파일 정보   |


### 최상위 루트 키


| 약어       | 풀이                  | 용도                                  |
| -------- | ------------------- | ----------------------------------- |
| **HKCR** | HKEY_CLASSES_ROOT   | 파일 확장자와 연결 프로그램 정보를 가진다.            |
| **HKCU** | HKEY_CURRENT_USER   | 현재 로그인 중인 사용자의 환경설정·프로파일 정보.        |
| **HKLM** | HKEY_LOCAL_MACHINE  | 시스템 전체에 적용되는 하드웨어와 응용 프로그램의 설정 데이터. |
| **HKU**  | HKEY_USERS          | 각 사용자별로 키 항목을 생성해 환경설정·프로파일 정보를 저장. |
| **HKCC** | HKEY_CURRENT_CONFIG | 현재 사용 중인 윈도우의 하드웨어 프로필 정보.          |


### 시험 포인트

- "Master vs Derived" 분류 — Master 는 실제 디스크에서 읽어 들이는 키(HKLM, HKU), Derived 는 Master 로부터 파생되어 메모리에서만 존재하는 키.
- SAM 하이브 = 계정 정보, SYSTEM 하이브 = 시스템 설정 매칭.

## 12. NTFS 파일시스템

### 배경

NTFS(New Technology File System) 는 Windows NT 계열 이후 모든 윈도우의 기본 파일시스템이다. 디스크의 어디에 부트 코드가 있고, 어디에 파일 메타데이터가 있는지를 알아야 디스크 손상 복구·포렌식 분석을 수행할 수 있다.

### 정의와 용어

- **파티션(Partition)** — 디스크를 구획화하여 나눈 영역.
- **파일시스템(File System)** — 파일을 관리하기 위한 자료구조.
- **포맷(Format)** — 파티션에 파일시스템을 구성하는 작업.

### 파일시스템 구성


| 구성요소                 | 설명                                                                                                                                                                                       |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **마스터 부트 레코드 (MBR)** | 파티션 생성 시 물리적 디스크의 **첫 번째 섹터**에 위치하는 **512 byte 크기**의 영역이다. 부트 코드 446 byte + 파티션 테이블 64 byte + 시그너처 2 byte 로 구성된다. 디스크 파괴형 악성코드에 의해 MBR 영역이 훼손되면 부트 코드를 정상적인 디스크에서 복사하고 파티션 정보를 수집해 복구한다. |
| **볼륨 부트 레코드 (VBR)**  | 파티션 구조에서 **가장 앞부분**에 위치하는 영역으로, 파티션의 **가장 마지막 섹터에 VBR 백업본**을 가진다. 디스크 파괴형 악성코드로 VBR 영역이 훼손되면 마지막 섹터의 백업본을 덮어써서 복구한다.                                                                     |
| **마스터 파일 테이블 (MFT)** | 볼륨/파일시스템에 존재하는 **모든 파일과 디렉터리에 대한 정보** 를 담고 있는 테이블. NTFS 에서 가장 중요한 영역이다.                                                                                                                  |
| **데이터 영역**           | 파일의 실제 내용이 저장되는 영역.                                                                                                                                                                      |


### 시험 포인트

- MBR = 디스크 첫 섹터(512B), VBR = 파티션 첫 부분 + 마지막 섹터에 백업, MFT = 파일·디렉터리 메타데이터 카탈로그. 이 세 영역의 위치·역할 매칭이 핵심.
- MBR 부트 코드 = 446 byte, 파티션 테이블 = 64 byte, 시그너처 = 2 byte 의 정확한 수치를 외운다.

## 단원 요약

1. **인증 컴포넌트 3종** — LSA(검증), SAM(패스워드 비교), SRM(SID 부여·감사).
2. **인증 경로 2종** — 로컬(SAM) vs 도메인(Kerberos·AD).
3. **SID** — RID 500=관리자, 501=게스트, 1000+=일반.
4. **Challenge & Response** — 패스워드 자체가 아닌 응답값을 전송.
5. **인증 알고리즘 진화** — LM → NTLM(LM+MD4) → NTLMv2(현재 기본).
6. **기본 그룹** — Administrators, Backup Operators 등 권한 강한 그룹의 범위 숙지.
7. **기본 공유** — C$, D$ (드라이브) / ADMIN$ (윈도우 폴더) / IPC$ (프로세스 간 통신).
8. **암호화** — EFS(파일/폴더), BitLocker(볼륨 전체).
9. **이벤트 로그** — 응용·보안·시스템 3종 분리 보관.
10. **PE** — Header / Section Header / Section. 섹션은 `.text`/`.data`/`.rdata`/`.bss`/`.idata`/`.edata`/`.rsrc`/`.reloc`.
11. **레지스트리** — Master(HKLM, HKU) ↔ Derived(HKCU, HKCC, HKCR). 하이브 파일: SAM·SECURITY·SOFTWARE·SYSTEM·NTUSER.dat.
12. **NTFS** — MBR(512B, 첫 섹터) / VBR(파티션 앞 + 마지막 섹터 백업) / MFT(파일 메타데이터) / 데이터 영역.
