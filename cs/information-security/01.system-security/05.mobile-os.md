## 0. 들어가며 — 모바일 OS 가 다른 단원과 다른 점

01~04 단원이 다룬 Windows · Linux 는 *관리자가 보안 설정을 직접 적용* 하는 운영체제였다. 모바일 OS 는 그 반대 — **OS 가 자체적으로 강제하는 보안 모델**이 보안 수준의 중심이며, 사용자·관리자가 임의로 풀 수 없는 영역이 더 많다. 자격증 1과목에서 모바일은 **두 가지 모델**의 차이를 묻는다.

| 모델 | OS | 핵심 |
|---|---|---|
| **폐쇄형 / 중앙 관리** | iOS | 애플이 앱·OS 모두 서명·배포·검증 |
| **개방형 / 분산 관리** | Android | 구글이 표준을 제공, 제조사·개발자가 개별 서명 |

두 모델 공통의 보안 토대는 **앱 서명 + 샌드박스** 다. 즉, 어떤 앱이든 실행되려면 ① 신뢰할 수 있는 주체가 서명했는가, ② 다른 앱·시스템 영역과 격리되어 있는가 — 두 조건을 만족해야 한다. 이 토대를 우회하는 행위가 iOS 의 **탈옥(Jailbreak)** 과 Android 의 **루팅(Rooting)** 이며, 이는 침해 사고의 가장 큰 발단이다.

이 단원은 출제기준 1-2(단말 OS) 의 한 항목으로, 분량과 출제 비중 모두 03·04 보다 작다. 외부 보강 부분은 1차 출처(Apple Platform Security[^2], Android Open Source Project[^3], KISA 모바일 보안 가이드[^1])를 각주로 명시한다.

## 1. iOS

### 1-1. 개요

iOS 는 애플에서 개발한 **폐쇄형 모바일 운영체제** 로, 보안성을 최우선으로 설계되어 있으며 애플이 철저히 관리하는 **중앙집중식 업데이트 및 앱 배포 체계** 를 갖춘다.

핵심 원칙은 다음 두 가지다.

- 보안상의 문제점은 대부분 **탈옥(Jailbreak) 한 기기** 에서 발생한다.
- 모든 소프트웨어는 **애플 암호화 로직의 서명된 방식** 에 의해 무결성이 확인된 후에만 동작한다.

### 1-2. iOS 보안 특징 (통합자료)

| 특징 | 설명 |
|---|---|
| **시스템 소프트웨어 개인화** | 모든 소프트웨어를 애플의 아이튠즈를 통해 일괄적으로 배포하고 있으며, 문제에 즉각적 대응을 할 수 있도록 함 |
| **응용 프로그램 서명** | 모든 앱은 애플의 서명을 요구하며, 코드 무결성이 확인되지 않으면 설치 불가 |
| **샌드박스 활용** | 엄격하게 프로그램 간 데이터 통신 통제 |

### 1-3. 코드 서명과 무결성 (외부 보강)

Apple Platform Security[^2] 에 따르면 iOS 는 **부팅에서 앱 실행에 이르는 모든 단계에서 코드 서명을 검증** 한다.

| 단계 | 검증 주체 |
|---|---|
| **Boot ROM** | 하드웨어에 내장된 Apple 루트 공개키로 부트로더 서명 검증 |
| **iBoot** | 커널 이미지의 서명 검증 |
| **커널** | 사용자 영역에서 실행되는 모든 바이너리·앱의 서명 검증 |
| **앱 실행** | App Store 에서 배포된 앱은 Apple 의 서명 외에 개발자 인증서로도 서명되어 있어야 함 |

이 체인의 최상위 신뢰 근원이 **Boot ROM 에 내장된 Apple 루트 공개키** 다 — 변경할 수 없는 하드웨어 키이므로, 부팅 단계부터 무결성이 보장된다(Secure Boot Chain).

### 1-4. 샌드박스 (외부 보강)

iOS 의 모든 서드파티 앱은 자기만의 컨테이너(샌드박스) 에서 실행된다. 다른 앱의 데이터 디렉터리에는 접근할 수 없고, 시스템 자원에 접근하려면 **Entitlement** 권한이 명시되어 있어야 한다[^2].

- **앱별 데이터 격리**: 다른 앱의 컨테이너에 직접 접근 불가
- **시스템 자원 접근 제한**: 카메라·연락처·위치 등 민감 자원은 사용자 동의 + Entitlement 필요

### 1-5. 탈옥(Jailbreak)

탈옥은 **OS 의 코드 서명 검증과 샌드박스 격리를 우회** 해 root 권한을 획득하는 행위다. iOS 보안 모델의 전제(코드 무결성 + 샌드박스) 가 무너지므로, 탈옥된 기기는 다음 위험에 노출된다[^1].

- 비공식(서명되지 않은) 앱 설치 가능 → 악성 코드 유입
- 시스템 영역 접근 가능 → 키체인·암호화 키 유출 위험
- 보안 패치 자동 업데이트 차단 가능성

### 시험 포인트
- iOS 는 **폐쇄형 + 중앙집중식 배포**. 모든 앱은 **애플 서명** 필요.
- 보안 토대 = **코드 서명 + 샌드박스**.
- 보안 사고의 대부분은 **탈옥된 기기** 에서 발생.

## 2. Android

### 2-1. 개요

Android 는 구글에서 개발한 **오픈소스 기반의 모바일 운영체제** 로, 다양한 제조사와 앱 개발자에게 개방되어 있어 유연하다.

> *원본 표기 보정*: 통합자료 line 1123 ~ 1124 는 OCR 과정에서 줄바꿈이 깨져 "설명다양한 제조사와 앱 개발자에게 개방되어 있어 유연" 과 "구글에서 개발한 오픈소스 기반의 모바일 운영체제로," 가 분리 표기됨. 표 헤더 "설명" 다음에 본문이 이어지는 패턴이며, 정상 표기는 위 본문과 같다.

핵심 원칙은 다음과 같다.

- **루팅(Rooting) 을 통해 최상위 권한** 을 얻을 수 있다 — 이는 보안 측면에서 양날이며, iOS 의 탈옥과 동일한 위험을 낳는다.

### 2-2. Android 보안 특징 (통합자료)

| 특징 | 설명 |
|---|---|
| **응용 프로그램 권한 관리** | 사용자의 데이터에 접근할 때 모든 사항을 응용 프로그램 사양에 명시하고, 접근 시 사용자의 동의를 받음 |
| **응용 프로그램 서명** | 응용 프로그램의 개발자가 서명하도록 함 |
| **샌드박스 활용** | 상대적으로 자유로운 형태의 어플리케이션 실행이 가능 |

### 2-3. 권한 모델 (외부 보강)

Android Open Source Project[^3] 에 따르면 Android 의 권한 모델은 **앱이 요청하고, 사용자가 허용** 하는 구조다. 권한은 보호 수준(Protection Level) 에 따라 분류된다.

| 보호 수준 | 의미 | 예시 |
|---|---|---|
| **Normal** | 위험도가 낮은 권한. 설치 시 자동 부여 | 인터넷 접속, 알림 표시 |
| **Dangerous** | 사용자 데이터·자원에 영향. 설치 시점 또는 런타임에 사용자 동의 필요 | 카메라, 연락처, 위치 |
| **Signature** | 동일한 인증서로 서명된 앱끼리만 부여 가능 | 시스템 앱 간 권한 공유 |
| **SignatureOrSystem** (API 23+ deprecated) | 시스템 영역 또는 동일 서명 앱에만 부여 | OEM 시스템 앱 |

Android 6.0(Marshmallow) 이후로 **Dangerous 권한은 런타임(Runtime) 권한** 으로 전환되어, 사용자가 앱 사용 중에 허용/거부할 수 있다[^3].

### 2-4. 앱 서명 (외부 보강)

iOS 와 달리 Android 는 **개발자 자체 인증서** 로 앱을 서명한다(애플 검증 단계 없음).

- 동일 패키지명의 앱이 업데이트되려면 **이전 버전과 동일한 서명** 이어야 한다 → 다른 개발자가 임의로 앱을 가로챌 수 없게 한다.
- Google Play 에 배포할 경우 **Play App Signing**(Google 이 마스터 키 보관) 또는 **개발자 자체 키** 사용 가능[^3].

### 2-5. 샌드박스 — Linux 권한 기반 (외부 보강)

Android 의 앱 격리는 **Linux 의 UID 기반 권한 모델** 위에 구축된다.

- 각 앱은 설치 시 **고유 UID** 를 할당받는다.
- 앱은 자기 UID 로 실행되며, 다른 UID 의 파일에 직접 접근 불가.
- SELinux 는 Android 4.3(Jelly Bean) 에서 처음 도입되었고, **Android 5.0(Lollipop) 부터 모든 도메인이 강제 모드(Enforcing) 로 전환**되어, UID 기반 격리 위에 추가로 MAC(강제적 접근통제) 정책이 작동한다[^3].

### 2-6. 루팅(Rooting)

루팅은 Android 에서 root 권한(최상위 권한) 을 획득하는 행위다. iOS 의 탈옥과 동일한 위험을 낳는다.

- 다른 앱의 데이터 디렉터리에 접근 가능 → 인증 토큰·결제 정보 유출 위험
- 시스템 영역 변조 가능 → 부트로더·커널 무결성 우회
- 보안 패치 적용 거부·지연 가능성[^1]

### 시험 포인트
- Android 는 **오픈소스 + 분산 관리**. 앱은 **개발자 자체 인증서** 로 서명.
- 권한 = **Normal / Dangerous / Signature**, Marshmallow(6.0) 이후 **런타임 권한**.
- 샌드박스 = **Linux UID 기반 격리 + SELinux**.
- 보안 사고의 큰 발단 = **루팅된 기기**.

## 3. iOS vs Android 비교

자격증에서는 두 OS 의 모델 차이를 묻는 비교형 문제가 출제된다.

| 비교 항목 | iOS | Android |
|---|---|---|
| **개발 주체** | Apple | Google (오픈소스) |
| **배포 체계** | 폐쇄형 (App Store 전용) | 개방형 (Google Play + 사이드로딩 가능) |
| **앱 서명 주체** | Apple + 개발자 (이중 서명) | 개발자 (단일 서명) |
| **권한 부여 시점** | 사용자 동의 + Entitlement | 설치 시 또는 런타임 (Marshmallow 이후) |
| **샌드박스 토대** | iOS 자체 컨테이너 모델 | Linux UID + SELinux |
| **최상위 권한 우회** | 탈옥(Jailbreak) | 루팅(Rooting) |
| **업데이트 통제** | Apple 일괄 (즉각 대응) | 제조사·통신사 분산 (지연 가능) |

### 시험 포인트
- "폐쇄형/중앙 vs 개방형/분산" — 두 OS 의 가장 본질적 차이.
- 두 OS 모두 **앱 서명 + 샌드박스** 가 보안 토대.
- 두 OS 모두 **최상위 권한 획득 = 보안 모델 무력화**.

## 4. Embedded OS (외부 보강)

### 4-1. 정의와 특징

**Embedded OS** 는 임베디드 기기(공장 제어기·의료기기·차량·라우터·가전 등) 에서 동작하는 운영체제다. 일반 데스크톱·서버 OS 와 달리 다음 특성을 갖는다[^1].

| 특성 | 의미 |
|---|---|
| **실시간성** | 정해진 시간 안에 응답 (RTOS, Real-Time OS) |
| **자원 제약** | 메모리·CPU·전력이 제한적 |
| **단일 목적** | 특정 기능 전용으로 설계, 범용 응용 실행 안 함 |
| **장기 운영** | 한 번 설치되면 수년~수십 년 무중단 운영 |

대표 예시로 **VxWorks**, **FreeRTOS**, **Embedded Linux**, **QNX** 가 있다.

### 4-2. 보안 위협

임베디드 OS 의 자원 제약과 장기 운영 특성은 다음과 같은 보안 위협을 낳는다[^1].

| 위협 | 설명 |
|---|---|
| **펌웨어 무결성 결여** | 부팅 시 펌웨어 서명 검증이 없으면 변조된 펌웨어가 실행될 수 있음 |
| **약한 기본 자격증명** | 출하 시 기본 ID/PW(`admin/admin` 등) 가 변경되지 않은 채 운영 |
| **업데이트 부재** | 보안 패치가 제공되지 않거나, 제공되어도 적용되지 않음 |
| **서비스 노출** | 진단·관리 포트(텔넷·SNMP 등) 가 외부에 그대로 노출 |

### 4-3. 대응 원칙 (외부 보강)

NIST 등 1차 출처가 권고하는 임베디드 보안 원칙은 다음과 같다[^1].

- **Secure Boot**: 부트로더 → 커널 → 응용까지 서명 검증 체인 구축
- **펌웨어 서명·암호화**: 업데이트 펌웨어에 디지털 서명 + 무결성 검증
- **기본 자격증명 변경 강제**: 최초 부팅 시 기본 ID/PW 변경 강제
- **불필요한 서비스 비활성화**: 04 §4 의 서비스 관리 원칙과 동일

### 시험 포인트
- Embedded OS = **실시간성 + 자원 제약 + 단일 목적 + 장기 운영**.
- 대표적 위협: **약한 기본 자격증명 / 펌웨어 변조 / 업데이트 부재**.

## 5. IoT / IIoT 단말 보안 (외부 보강)

### 5-1. IoT 와 IIoT

| 구분 | 정의 |
|---|---|
| **IoT** (Internet of Things) | 인터넷에 연결된 일반 사물 — 스마트 가전·웨어러블·홈 카메라 등 |
| **IIoT** (Industrial IoT) | 산업 환경의 IoT — 공장 자동화 PLC·센서·제어기 등 |

두 영역은 **OS 가 임베디드 OS** 인 경우가 많다는 점에서 §4 의 위협을 그대로 상속한다.

### 5-2. IoT 단말의 주요 보안 위협 (외부 보강)

KISA 의 IoT 보안 가이드는 다음 위협 항목을 권장 점검 대상으로 제시한다[^1].

| 위협 | 설명 |
|---|---|
| **약한 인증** | 기본 ID/PW, 추측 가능한 인증 정보 |
| **암호화 부재 통신** | HTTP/Telnet 등 평문 프로토콜 사용 |
| **펌웨어 변조** | 서명 없는 펌웨어 업데이트 경로 |
| **물리적 접근** | 단말이 외부에 노출 — 디버그 포트(JTAG·UART) 직접 접근 |
| **봇넷 가담** | 침해된 IoT 기기가 DDoS 공격(Mirai 류) 의 발판으로 활용 |

### 5-3. 대응 원칙 (외부 보강)

| 원칙 | 적용 |
|---|---|
| **Secure Boot · 펌웨어 서명** | §4-3 동일 |
| **TLS 통신 의무화** | 단말 ↔ 서버 통신은 평문 금지 |
| **개체별 고유 인증** | 단말마다 고유 키·인증서 (공장 출하 시 주입) |
| **취약점 공시·자동 업데이트** | 제조사가 패치를 배포하고 단말이 자동 적용 |
| **세그먼트 분리** | IoT/IIoT 트래픽을 일반 사용자 네트워크와 격리 |

### 시험 포인트
- **IoT vs IIoT** — 산업 환경 여부가 차이.
- 가장 빈번한 위협 = **약한 기본 자격증명 + 평문 통신 + 펌웨어 변조**.
- 대응 = **Secure Boot + TLS + 개체별 고유 인증 + 자동 업데이트**.

## 단원 요약

1. **iOS** — 폐쇄형 + 중앙 관리. **Apple + 개발자 이중 서명**, 샌드박스, **탈옥** 시 보안 모델 붕괴.
2. **Android** — 오픈소스 + 분산 관리. **개발자 단일 서명**, 권한 = Normal/Dangerous/Signature(런타임 동의), Linux UID + SELinux 샌드박스, **루팅** 시 보안 모델 붕괴.
3. **iOS vs Android** — 폐쇄형/중앙 vs 개방형/분산. 공통 토대는 앱 서명 + 샌드박스.
4. **Embedded OS** — 실시간성·자원 제약·단일 목적·장기 운영. 위협 = 약한 자격증명·펌웨어 변조·업데이트 부재.
5. **IoT/IIoT** — Embedded OS 위협 + 평문 통신 + 봇넷 가담. 대응 = Secure Boot · TLS · 고유 인증 · 자동 업데이트 · 세그먼트 분리.
