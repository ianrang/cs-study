> **이 문서가 다루는 것**: 리눅스 시스템이 사용자를 식별하는 방법, 시스템이 부팅되는 절차, 파일·프로세스·권한·서비스·인증·로그를 다루는 자료구조와 명령. 자격증 1과목에서 가장 분량이 많은 영역이며, 출제 비중도 가장 높다.

>

## 0. 들어가며 — 왜 리눅스가 가장 무거운 단원인가

리눅스(와 그 조상 격인 Unix) 는 서버·임베디드·컨테이너 호스트 등 보안 점검 대상의 대부분을 차지한다. 서버 OS 시장에서의 점유율, 침해사고 분석 시 마주칠 빈도, 그리고 출제기준의 "단말 OS / 서버 OS" 모두에 해당하는 위치 때문에 — **자격증 1과목에서 리눅스가 차지하는 분량이 가장 크다**.

이 단원의 토픽은 크게 네 층위로 나뉜다.

| 층위              | 다루는 토픽                                                                             |
| ----------------- | --------------------------------------------------------------------------------------- |
| **시스템 시작**   | 로그인, 런 레벨, 부팅 4단계                                                             |
| **사용자와 자원** | 계정 파일(passwd/shadow/group), 파일시스템, 권한, 입출력 재지정, 링크                   |
| **실행 단위**     | 프로세스, 시그널, 스케줄러(cron/at)                                                     |
| **운영 인프라**   | 서비스 운영(inetd/xinetd/TCP Wrapper), 인증 모듈(PAM), 로그(syslog/logrotate/로그 파일) |

윈도우 단원과 마찬가지로, 이 단원은 [04. 리눅스 관리](./04.linux-management.md) (운영·점검 정책) 와 [06. 시스템 공격](./06.system-attacks.md) (BoF·레이스 컨디션 등) 가 참조하는 공용 어휘를 만든다.

## 1. 로그인과 시스템 부팅

### 1-1. 로그인 과정

리눅스는 사용자가 입력한 패스워드를 **/etc/passwd → /etc/shadow** 의 두 단계로 비교해 인증한다. shadow 정책을 사용하는 현재 리눅스에서는 다음 5단계를 거친다.

1. 사용자 계정명과 패스워드 입력
2. 로그인 프로그램이 입력한 패스워드와 `/etc/passwd` 파일의 해당 필드를 비교
3. `/etc/passwd` 파일의 2번째 필드가 `x` 이면 `/etc/shadow` 파일에서 비교

- 비교 절차: 입력 패스워드 → `H0` → 암호문 1 → `crypt(Salt + 암호문 1)` → 암호문 2

4. `/etc/passwd` 파일의 해당 필드에서 계정 정보를 가져와 초기 환경 설정
5. 모든 절차가 끝나면 로그인 쉘 실행

### 1-2. 런 레벨 (Run Level)

시스템의 운영 상태를 숫자 또는 문자로 표현한 것을 **런 레벨**(Run Level) 이라 한다. `init` 프로세스는 `/etc/inittab` 파일에 정의된 런 레벨에 따라 `/etc/rc.d/rc*.d` 디렉터리에 나열된 스크립트를 실행해 시스템 운영 상태를 구성한다.

| 런 레벨  | 의미                                                                                   |
| -------- | -------------------------------------------------------------------------------------- |
| **0**    | 시스템 정지(유닉스 PROM 모드)                                                          |
| **S, s** | 시스템 싱글 유저 모드. 유지보수용이며 root 패스워드 필요. **로컬 파일시스템 마운트 X** |
| **1**    | 시스템 싱글 유저 모드. **로컬 파일시스템 마운트 O**                                    |
| **2**    | 멀티 유저 모드(NFS 클라이언트)                                                         |
| **3**    | 멀티 유저 모드(NFS 서버). 유닉스 기본 런 레벨                                          |
| **4**    | 사용 안 함                                                                             |
| **5**    | 유닉스: 시스템 전원 종료 / **리눅스: X 윈도우 환경의 멀티 유저 모드**                  |
| **6**    | 시스템 리부팅                                                                          |

**확인 명령**: `who -r` 또는 `runlevel`. **변경 명령**: `init [모드번호]` (예: `init 5`).

### 1-3. 시스템 부팅 4단계

| 단계                  | 위치       | 작업                                                                                                                           |
| --------------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------ |
| 1. **바이오스(BIOS)** | 하드웨어   | 기본 하드웨어 점검 후 하드웨어 정보 수집                                                                                       |
| 2. **부트 프로그램**  | 하드디스크 | 하드디스크에서 커널을 읽어 메모리에 적재. 시스템 제어권을 커널에 넘김                                                          |
| 3. **커널**           | 메모리     | 운영체제 구동 시작. 내부 자료구조를 초기화하고 부가적인 커널 모듈을 적재                                                       |
| 4. **init 프로세스**  | 프로세스   | 커널이 생성하는 첫 번째 프로세스(PID = 1). 모든 유닉스 프로세스의 부모이며 사용자가 시스템을 사용할 수 있도록 초기화 작업 수행 |

### 관련 용어

- **BSD 유닉스 계열** (US 버클리) — 싱글 유저와 멀티 유저 모드 두 가지를 지원.
- **System V 계열 (AT&T)** — 런 레벨로 분류하며 `/etc/inittab` 파일에 정의.
- **shutdown 명령** — 시스템을 안전하게 종료하거나 런 레벨을 변경할 때 사용.
- **sync 명령** — 시스템의 버퍼에 있는 데이터를 디스크에 저장. 종료 전 무결성 유지를 위해 사용.

### 시험 포인트

- "init 프로세스 = PID 1, 모든 프로세스의 부모" 는 단골 출제.
- 런 레벨에서 유닉스 5(전원 종료) ↔ 리눅스 5(X 윈도우) 의 차이는 함정 문제로 자주 등장.
- 부팅 4단계의 위치(BIOS → 디스크 → 메모리 → 프로세스) 매핑을 외운다.

---

> **여기까지** 시스템이 켜져서 사용자가 로그인할 때까지의 흐름을 봤다. 이제 그 사용자가 _어디에 어떻게 저장되어 있는지_ — 계정 파일과 패스워드 보관 정책 — 을 본다.

## 2. 계정과 패스워드 저장

### 2-1. /etc/passwd — 사용자 계정 정보

7개 필드를 콜론(`:`)으로 구분한다.

```
계정명 : 패스워드 : UID : GID : 설명 : 홈 디렉터리 : 로그인 쉘
```

| 필드               | 의미                                                        |
| ------------------ | ----------------------------------------------------------- |
| 1. **계정명**      | 사용자 계정명 (`user_account` / `logname`)                  |
| 2. **패스워드**    | 사용자의 패스워드. **`x` 이면 shadow 패스워드 정책** 사용 |
| 3. **UID**         | 사용자 ID. 시스템 내 유일한 정수                            |
| 4. **GID**         | 사용자의 기본 그룹 ID                                       |
| 5. **설명**        | 사용자 설명(주석)                                           |
| 6. **홈 디렉터리** | 사용자 홈 디렉터리 위치                                     |
| 7. **로그인 쉘**   | 사용자의 로그인 쉘                                          |

### 2-2. /etc/group — 그룹 정보

```
그룹명 : 그룹 패스워드 : GID : 소속 계정
```

총 4개 필드. **그룹의 암호화된 패스워드는 일반적으로 사용하지 않으며 보통 `x`** 로 기재한다.

```
예) bin:x:1:root,bin,daemon
```

### 2-3. /etc/shadow — 암호화된 패스워드 저장

암호화된 패스워드를 저장하고 **root 만이 접근할 수 있도록 제한** 하여 패스워드 보안을 강화한다.

```
계정명 : 암호화된 패스워드 : 마지막 변경일 : 최소 사용기간 : 최대 사용기간 : 경고 : 비활성화 : 만료일 : 예약
```

총 9개 필드.

| 필드                             | 의미                                                                 |
| -------------------------------- | -------------------------------------------------------------------- |
| 1. **계정명**                    | 사용자 계정명                                                        |
| 2. **암호화된 패스워드**         | `$ID$Salt$encrypted_password` 형식                                   |
| 3. **마지막 변경일**             | 1970년 1월 1일 기준 일수                                             |
| 4. **최소 사용기간 (`minlife`)** | 변경 후 다시 변경할 수 없는 일수 (없으면 "최근 암호 기억" 우회 가능) |
| 5. **최대 사용기간 (`maxlife`)** | 변경 후 패스워드 만료 일수 (90일 권장)                               |
| 6. **경고 (`warn`)**             | 만료 이전 경고 일수                                                  |
| 7. **비활성화 (`inactive`)**     | 만료 이후 계정이 잠기기 전까지 비활성 일수                           |
| 8. **만료일 (`expires`)**        | 1970년 1월 1일 기준 일수로 설정한 계정 만료일                        |
| 9. **예약 필드**                 | 사용하지 않음                                                        |

#### 암호화된 패스워드 필드 구조

```
$ID$Salt$encrypted_password
```

| 구성                   | 의미                                                                                     |
| ---------------------- | ---------------------------------------------------------------------------------------- |
| **ID**                 | 적용된 일방향 해시 알고리즘 식별자. `1`=MD5, `2`=BlowFish, `5`=SHA-256, `6`=SHA-512      |
| **Salt**               | 패스워드 암호화 강도를 높이기 위한 임의의 값. **레인보우 테이블 공격에 효과적으로 대응** |
| **encrypted_password** | 사용자 패스워드에 솔트값을 조합하여 해시화한 결과                                        |

> _원본 표기 보정_: 1과목.txt 는 `3 → SHA-256`, `4 → SHA-512` 로 기재되어 있으나, 이는 OCR/표기 오류다. 실제 `crypt(3)` 의 `$id$` 표준은 `5` = SHA-256, `6` = SHA-512(`man 3 crypt` 참조)[^1]. 시험에서도 `$5$`, `$6$` 표기로 출제된다.

#### 솔트(Salt) 의 효과

- **서로 다른 솔트 값** — 동일한 패스워드를 사용해도 솔트 값이 다르면 암호화 결과가 다르다. → 같은 비밀번호를 쓰는 두 사용자의 해시가 동일해 보이는 위험 차단.
- **해시값 비교** — 암호화된 패스워드를 알아내도 이는 솔트와 조합된 결과의 해시값이므로, 평문 패스워드 자체는 추출하기 어렵다.

#### 특수 표기

| 표기       | 의미                                                                |
| ---------- | ------------------------------------------------------------------- |
| `*` (별표) | 패스워드가 잠긴 상태로 일반 로그인 불가 (별도 인증 방식으로는 가능) |
| `!!`       | 계정 생성 후 비밀번호를 갱신하지 않은 상태로 모든 로그인 불가       |
| 빈 값      | 패스워드가 설정되지 않아 비밀번호 없이 로그인 가능(매우 위험)       |
| `NP`       | No Password. 시스템·애플리케이션 계정                               |
| `*LK*`     | Lock. 패스워드가 잠긴 상태                                          |

### 2-4. 패스워드 저장 정책 변경

| 명령       | 효과                                                  |
| ---------- | ----------------------------------------------------- |
| `pwconv`   | shadow 파일을 활성화 (`/etc/shadow` 에 패스워드 저장) |
| `pwunconv` | shadow 파일을 비활성화 (`/etc/passwd` 에 함께 저장)   |

### 2-5. 패스워드 잠금

| 명령                 | 효과                           |
| -------------------- | ------------------------------ |
| `passwd -l [계정명]` | 해당 계정의 패스워드 잠금      |
| `passwd -u [계정명]` | 해당 계정의 패스워드 잠금 해제 |

### 시험 포인트

- 7필드(passwd) / 4필드(group) / 9필드(shadow) 의 **개수** 와 **순서** 는 자주 출제된다.
- `$ID$Salt$encrypted_password` 의 ID 매핑 (`1`/`5`/`6` = MD5 / SHA-256 / SHA-512) 정확히 외운다.
- `passwd` 파일 두 번째 필드가 `x` = shadow 정책 활성.
- `*`, `!!`, 빈 값 의 의미 차이.

---

> **여기까지** 사용자 정보가 어디에 어떻게 저장되는지 봤다. 이제 그 정보를 담는 그릇 — 파일시스템 — 의 내부 구조를 본다.

## 3. 파일시스템

### 3-1. 파일시스템 구성

물리적 디스크는 논리적 파티션으로 구분되며 파티션별로 고유한 파일시스템을 생성한다.

| 구성                            | 설명                                                  |
| ------------------------------- | ----------------------------------------------------- |
| **부트 블록**                   | 운영체제를 부팅하거나 초기화하기 위한 부트스트랩 코드 |
| **슈퍼 블록**                   | 해당 파일시스템을 관리하기 위한 정보(크기·블록 수 등) |
| **아이노드 리스트(inode list)** | 파일들에 대한 속성정보를 담는 inode 구조체 리스트     |
| **데이터 블록**                 | 실제 파일들의 데이터가 저장되는 영역                  |

### 3-2. inode

inode 는 파일에 대한 **속성 정보** 를 담는 구조체다. 중요한 점: **inode 에는 파일명이 없다.** 파일명은 디렉터리를 통해 관리되는 것이지 inode list 와는 관계가 없다.

#### inode 구성 정보

- inode number, 파일 타입(일반/디렉터리/장치), 접근 권한
- linkcount(하드링크 개수), 소유자, 소유 그룹, 파일 크기
- **MAC Time**:
  - **M time** (Modification) — 파일 _내용_ 이 마지막으로 수정된 시간
  - **A time** (Access) — 파일을 마지막으로 _접근_ 한 시간
  - **C time** (Change) — 파일의 _속성_ 이 마지막으로 변경된 시간
- block index

**확인 명령**: `stat [파일명]`.

### 3-3. 파일 종류

`ls -l` 의 첫 글자로 식별한다.

| 표기 | 종류                                        |
| ---- | ------------------------------------------- |
| `-`  | 일반 정규 파일                              |
| `d`  | 디렉터리                                    |
| `b`  | 블록 장치 파일 (블록 단위 입출력, 버퍼링)   |
| `c`  | 문자 장치 파일 (문자 단위 입출력, 비버퍼링) |
| `l`  | 심볼릭 링크 파일                            |
| `p`  | 네임드 파이프                               |
| `s`  | 유닉스 도메인 소켓                          |

### 3-4. 마운트 / 용량 / 동기화

| 명령     | 의미                                                                                                        |
| -------- | ----------------------------------------------------------------------------------------------------------- |
| `mount`  | 보조기억장치에 설치된 파일시스템을 시스템이 인식하도록 특정 디렉터리에 논리적으로 연결. `/etc/mtab` 에 기록 |
| `umount` | 마운트 해제. `device is busy` 발생 시 `fuser -ck [마운트경로]` 사용                                         |
| `du`     | 디렉터리의 하드디스크 사용량 확인 (`-a` 하위파일까지, `-s` 총량, `-k` KB)                                   |
| `df`     | 파일시스템의 전체 공간 및 사용 가능 공간 정보                                                               |
| `sync`   | 시스템 버퍼의 데이터를 디스크에 저장하여 동기화. 종료 전 필수                                               |

### 시험 포인트

- "inode 에 파일명이 없다" 는 명제는 단골 함정.
- MAC Time 3종(M/A/C) 의 변경 트리거 매칭이 자주 나온다 — 내용 변경 = M, 접근 = A, 속성 변경 = C.
- `mount` 정보는 `/etc/mtab` 에 기록되며, `/etc/fstab` 은 부팅 시 마운트할 파일시스템 정의.

## 4. 입출력 재지정과 링크

### 4-1. 입출력 재지정 (I/O Redirection)

키보드에서 들어오는 입력을 파일에서 받도록 대체하거나, 명령의 실행 결과·에러를 화면 대신 파일로 받는 것.

| 형태                     | 의미                              |
| ------------------------ | --------------------------------- |
| `command [0]< file_name` | 입력 재지정                       |
| `command [1]> file_name` | 표준 출력 재지정 (1 은 생략 가능) |
| `command 2> file_name`   | 표준 에러 재지정                  |
| `command >> file_name`   | 출력을 **추가** (덮어쓰기 아님)   |
| `command 2>&1`           | 표준 에러를 표준 출력으로 합침    |

```bash
who > who.txt        # who 결과를 who.txt 에 저장(덮어씀)
who >> who.txt       # who 결과를 who.txt 에 추가
0 3 * * 0 /bin/rm -rf /tmp/* >/dev/null 2>&1
# 매주 일요일 03:00 /tmp 하위 전체 삭제, 표준출력은 /dev/null 로,
# 표준에러도 표준출력과 같은 곳으로(2>&1) → 모두 /dev/null
```

`/dev/null` 은 특수 장치 파일로, 해당 파일로 출력하면 모두 버린다 (윈도우의 휴지통과 유사한 역할).

### 4-2. 링크

| 종류            | 명령                    | 특징                                                                                                           |
| --------------- | ----------------------- | -------------------------------------------------------------------------------------------------------------- |
| **하드 링크**   | `ln [대상] [링크명]`    | 기존 파일과 **동일한 inode number** 를 가지는 파일 생성. **동일 파일시스템 내** 에서만 가능, **디렉터리 불가** |
| **심볼릭 링크** | `ln -s [대상] [링크명]` | 원본 파일의 경로를 파일 내용으로 하는 새 파일 생성. **파일시스템 제한 없음, 디렉터리도 가능**                  |

```bash
ln -s abc.dat abc_sl.dat   # abc_sl.dat 가 심볼릭 링크
```

### 시험 포인트

- "동일 inode = 하드 / 경로 보존 = 심볼릭" 매핑은 기본.
- "디렉터리에 하드 링크 X" 도 자주 출제.
- `ls -F` 출력 표시 — `/`(디렉터리), `*`(실행 파일), `@`(심볼릭 링크).

## 5. 권한과 umask

### 5-1. 기본 접근 권한과 umask

- **파일 생성 기본값**: `666` (`rw-rw-rw-`)
- **디렉터리 생성 기본값**: `777` (`rwxrwxrwx`)
- **umask** — 파일 생성 시 **제거할 권한** 을 명시. 기본 `022` 를 적용하면:
  - 파일: `666 - 022 = 644` (`rw-r--r--`)
  - 디렉터리: `777 - 022 = 755` (`rwxr-xr-x`)

> **디렉터리 권한 메모**: 디렉터리는 `x` 권한이 있어야 `cd` 로 진입할 수 있다.

`/etc/profile` 에 `umask` 를 지정하면 시스템 전체 사용자에게 일관된 값을 적용할 수 있다(보통 `022`).

### 5-2. 특수 권한

프로세스가 실행되는 동안의 권한을 제어하는 세 가지 특수 비트.

| 권한           | 약어              | 의미                                                                                    |
| -------------- | ----------------- | --------------------------------------------------------------------------------------- |
| **SUID**       | Set-UID (`s` / 4) | 실행 중에는 **실행 파일 소유자의 권한** 으로 자원에 접근                                |
| **SGID**       | Set-GID (`s` / 2) | 실행 중에는 **실행 파일 소유 그룹의 권한** 으로 자원에 접근                             |
| **Sticky-bit** | (`t` / 1)         | 디렉터리 내에 누구나 파일 생성 가능하지만 **삭제·이름 변경은 소유자 또는 root 만** 가능 |

#### RUID / EUID / RGID / EGID

프로세스의 자원 접근 권한 판단에 사용된다.

| 식별자          | 의미                                                                    |
| --------------- | ----------------------------------------------------------------------- |
| **RUID / RGID** | 프로세스를 _실행시킨 사용자_ 의 UID/GID (Real)                          |
| **EUID / EGID** | 프로세스가 실행 중인 동안 _접근 권한 판단에 쓰이는_ UID/GID (Effective) |

- SUID/SGID **미설정**: RUID/EUID, RGID/EGID 모두 실행시킨 사용자의 값
- SUID/SGID **설정**: RUID/RGID 는 실행자, **EUID/EGID 는 실행 파일 소유자(또는 그룹)** → "잠시 권한 상승" 효과

#### sticky-bit 의 활용

공유 디렉터리(`/tmp`, `/var/tmp`) 의 보안을 위해 사용된다.

- 리눅스: `chmod o+t /tmp`
- 유닉스: `chmod u+t /tmp`

#### SUID/SGID 보안 점검

```bash
find / -user root -type f \( -perm -4000 -o -perm -2000 \) -exec ls -al {} \;
```

root 권한이 필요 없는 프로그램에 root 소유 + SUID 가 설정되어 있으면 권한 상승 위험이 있으므로 주기적 점검이 필요하다. `chmod -s [실행파일]` 로 SUID/SGID 제거 가능. 대표적인 SUID 설정 파일은 `/usr/bin/passwd`.

### 시험 포인트

- 기본값 666(파일) / 777(디렉터리), umask 022 → 644 / 755 의 산수.
- SUID = 4, SGID = 2, Sticky = 1 의 8진수 값.
- "RUID = 실행자, EUID = 소유자(SUID 설정 시)" 매핑.

## 6. 프로세스

### 6-1. 프로세스의 기본 조건

1. boot 프로세스(0번 프로세스)를 제외한 모든 프로세스는 **부모 프로세스** 를 가진다.
2. 자식 프로세스가 살아있는 상태에서 부모 프로세스가 종료되면 **고아 프로세스** 가 된다.
3. 고아 프로세스는 **`init` 프로세스(PID = 1)** 가 부모 역할을 한다.
4. 종료 시 자신의 종료 상태 정보(pid, exit code, cpu time)를 부모 프로세스에 반환해야 정상적으로 소멸할 수 있다.

### 6-2. 좀비 프로세스

- 프로세스 수행을 종료했지만 부모 프로세스가 종료 상태 정보를 확인하지 않아 소멸하지 않고 남아있는 상태.
- 좀비 상태(`Z`, `defunct`) 로 표기.
- 좀비 프로세스가 과도하게 많아지면 더 이상 프로세스를 생성하지 못한다.
- **kill** 시그널을 보내도 소멸되지 않는다. 시스템 리부팅으로 커널 정보를 초기화해야 한다.

### 6-3. 프로세스 그룹과 터미널 제어권

- 프로세스 생성과 동시에 **프로세스 그룹** 도 생성. 쉘에서 실행된 프로세스와 그 자식 프로세스들이 하나의 그룹으로 관리된다.
- **PGID** — 그룹 식별자. 그룹 리더 프로세스의 PID 를 PGID 로 설정.
- **터미널 제어권** — 프로세스 그룹 단위로 부여. 터미널로부터 입력되는 데이터와 시그널에 대한 제어권.

| 모드           | 의미                                 |
| -------------- | ------------------------------------ |
| **포그라운드** | 터미널에 대한 제어권을 가지고 동작   |
| **백그라운드** | 터미널에 대한 제어권 없이 동작 (`&`) |

세션 내에는 하나의 포그라운드 그룹과 다수의 백그라운드 그룹이 공존한다.

### 6-4. 프로세스 식별자

| 식별자   | 의미                                                       |
| -------- | ---------------------------------------------------------- |
| **PID**  | 프로세스 고유 ID                                           |
| **PPID** | 부모 프로세스 ID                                           |
| **PGID** | 프로세스 그룹 ID                                           |
| **SID**  | 세션 식별자. 터미널과의 논리적 연결 상태(세션) 의 리더 PID |

### 6-5. 시그널

**시그널(Signal)** 은 유닉스가 지원하는 소프트웨어 인터럽트로, 다른 프로세스에 이벤트를 전달하는 IPC 수단의 하나다.

#### 시그널 발생 경로

| 경로   | 예시                               |
| ------ | ---------------------------------- |
| 외부   | 키보드 입력 (`Ctrl+C`)             |
| 에러   | 커널의 메모리 위반 감지            |
| 이벤트 | `alarm()` 함수 만료, 프로세스 종료 |
| 인위적 | `kill()` 함수, `kill` 명령         |

#### 시그널 정책 4종

무시 / 보류 / 특정 함수 호출 / 디폴트(일반적으로 종료).

#### 주요 시그널 — 번호와 의미

| 번호 | 이름        | 의미                                                   |
| ---- | ----------- | ------------------------------------------------------ |
| 1    | **SIGHUP**  | 터미널 연결 끊김. 최근에는 환경설정 파일 재로드에 사용 |
| 2    | **SIGINT**  | `Ctrl+C` — 포그라운드 그룹 내 모든 프로세스 종료       |
| 3    | **SIGQUIT** | `Ctrl+\` — 종료 + core 파일 생성                       |
| 8    | **SIGFPE**  | 산술 연산 에러                                         |
| 9    | **SIGKILL** | **무시·임의 처리 불가**. 받은 프로세스는 무조건 종료   |
| 11   | **SIGSEGV** | 잘못된 메모리 참조                                     |
| 14   | **SIGALRM** | `alarm()` 함수의 타이머 만료                           |
| 15   | **SIGTERM** | 디폴트 종료 시그널. 시그널 옵션 생략 시 발생           |
| 18   | **SIGCHLD** | 자식 프로세스의 종료·정지 시 부모에게 전달             |
| 23   | **SIGSTOP** | **무시·임의 처리 불가**. 프로세스 정지                 |

```bash
kill -9 3000        # PID 3000 에 SIGKILL 전송
kill -KILL 3000     # 동일
```

> **SIGKILL(9) 과 SIGSTOP(23) 은 절대로 무시·재정의할 수 없다.** 받은 프로세스는 무조건 종료되거나 정지한다.

### 6-6. 프로세스 자료구조

| 구조                            | 설명                                                                                                                            |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| **PCB** (Process Control Block) | 개별 프로세스별 관리 정보를 담는 제어 블록. 상태(ready/running/blocked), 프로세스 번호, 프로그램 카운터, 레지스터, 메모리 정보  |
| **FDT** (File Descriptor Table) | 개별 프로세스별로 오픈한 파일을 관리. 표준입력(`stdin`), 표준출력(`stdout`), 표준에러(`stderr`), 파일 디스크립터(FD: 양의 정수) |
| **System open-file table**      | 커널이 시스템 내 모든 프로세스에 의해 오픈된 파일을 관리. open_mode, offset, reference_count 포함                               |

> _원본 표기 보정_: 통합자료의 `stbin / stbout / stberr` 는 OCR 오류로, 실제 표기는 `stdin / stdout / stderr` 이다.

### 6-7. 프로세스 실행 순서

1. 프로그램 실행
2. **`fork()`** 를 통한 자식 프로세스 생성 (터미널 제어권이 자식 프로세스로 넘어감)
3. **`exec()`** — 새 프로세스 생성 없이, 현재 프로세스의 이미지만 새 프로그램으로 전이 (교체)
4. 프로그램 종료 (터미널 제어권을 쉘로 반환)

- 이미지는 프로세스의 메모리에 실려 있는 실행 단위 (코드+데이터+스택 등)
- 기존 이미지는 실행 주체 관점에서 사라지고(메모리 해제·맵 교체) 새 이미지로 갈아끼워지는 것이다.

### 6-8. ps 명령

`ps` 는 프로세스 정보를 확인하는 명령이다. `ps -ef` 가 가장 많이 쓰인다.

#### `ps -l` 주요 필드

| 필드      | 의미                                                                                                   |
| --------- | ------------------------------------------------------------------------------------------------------ |
| **F**     | 프로세스 플래그 (`1`: `fork()` 만 됨, `4`: 슈퍼유저 권한)                                              |
| **S**     | 상태 (`R` 실행 중 / `S` 인터럽트 가능 sleep / `D` 인터럽트 불가 sleep, 보통 I/O / `T` 정지 / `Z` 좀비) |
| **PRI**   | 우선순위 (낮을수록 높음)                                                                               |
| **NI**    | 우선순위 계산용 nice 값                                                                                |
| **ADDR**  | 메모리 주소                                                                                            |
| **SZ**    | 메모리 크기                                                                                            |
| **WCHAN** | sleep 상태에서 대기 중인 커널 함수                                                                     |

#### `ps -ef` 추가 필드

| 필드      | 의미                         |
| --------- | ---------------------------- |
| **STIME** | 시작 시간                    |
| **TTY**   | 연결된 터미널 (`?` = 미연결) |
| **TIME**  | CPU 사용 시간                |
| **CMD**   | 프로세스명                   |

### 시험 포인트

- "PID 1 = init, PID 0 = boot" 와 "고아 프로세스의 부모 = init" 매핑.
- 좀비 프로세스는 `kill` 로 못 죽인다(이미 죽었기 때문에).
- 시그널 번호 — **9(KILL), 15(TERM), 23(STOP)** 가 가장 자주 출제. 9·23 은 무시 불가.
- `fork()` → `exec()` 순서.

## 7. 스케줄러 — cron · at

### 7-1. crontab — 정기 스케줄

`cron` 데몬은 시스템에서 정기적인 작업을 지정 시간에 처리하기 위해 사용한다.

#### 사용 시점

- 일괄 처리(야간 작업)
- 작업 요구가 비규칙적이지 않은 경우
- 우선순위가 낮은 작업을 일정 시기에 처리하는 것이 효율적인 경우

#### crontab 파일 구조 (6필드)

```
분  시  일  월  요일  작업
(0-59) (0-23) (1-31) (1-12) (0-6, 0=일) (절대경로 명령)
```

기호:

- `*` 필드 범위에 해당하는 모든 값
- `-` 값의 범위
- `,` 값의 구분
- `/` 간격값

```bash
0 3 * * 0 /bin/rm -rf /tmp/* >/dev/null 2>&1
# 매주 일요일 03:00 /tmp 하위 전체 삭제
```

#### crontab 파일 제어

| 명령                           | 의미                        |
| ------------------------------ | --------------------------- |
| `crontab -e`                   | 자신의 crontab 편집         |
| `crontab -l`                   | 자신의 crontab 출력         |
| `crontab -r`                   | 자신의 crontab 삭제         |
| `crontab -u [계정명] -e/-l/-r` | 다른 계정 crontab (root 만) |

#### crontab 명령 접근 제어

| 파일              | 방식                                    |
| ----------------- | --------------------------------------- |
| `/etc/cron.allow` | **화이트리스트** — 등록된 사용자만 허용 |
| `/etc/cron.deny`  | **블랙리스트** — 등록된 사용자만 거부   |

- 두 파일이 모두 있으면 **`/etc/cron.allow` 우선**.
- 두 파일이 모두 없으면 **root 만 허용**.

### 7-2. at — 일시적 스케줄

- 정해진 시간에 한 번만 실행.
- 한 번 처리된 작업은 작업 목록에서 삭제된다.

```bash
at -t [날짜와 시간]
# 그 시간에 일시적으로 처리할 작업 명세
```

### 시험 포인트

- crontab 6필드 순서(분/시/일/월/요일/작업) — 단골.
- 두 파일 모두 없을 때 root 만 사용 가능.
- `cron.allow` 와 `cron.deny` 의 화이트/블랙리스트 구분.

---

> **여기까지** 시스템 내부에서 동작하는 단위 — 사용자·자원·프로세스·스케줄 — 를 다뤘다. 이제 **외부에서 들어오는 연결을 어떻게 받고 통제하는지** (서비스 운영) 와 **어떻게 인증하는지** (PAM), 그리고 **무엇을 기록하는지** (로깅) 로 시야를 넓힌다.

## 8. 서비스 운영

### 8-1. 슈퍼 데몬과 서비스 운영 방식

| 방식                            | 특징                                                                                    |
| ------------------------------- | --------------------------------------------------------------------------------------- |
| **Stand-Alone**                 | 개별 서비스마다 서버 프로세스(데몬)가 동작. 속도는 빠르지만 서버 자원을 많이 사용       |
| **(x)inetd(inter daemon) 방식** | 슈퍼 데몬이 클라이언트 요청을 받아 적절한 개별 서비스를 호출. 속도는 느리지만 자원 절약 |

리눅스는 `xinetd` 데몬을 주로 사용한다(`inetd` 의 보안·자원 관리 향상 버전).

### 8-2. inetd 설정

`inetd` 는 `/etc/inetd.conf` 의 정보를 참조해 서비스할 프로그램을 정의한다.

#### `/etc/inetd.conf` 형식

```
서비스명  소켓타입  프로토콜  플래그  사용자계정  실행경로  실행인수
```

```
ftp  stream  tcp  nowait  root  /usr/sbin/in.ftpd  in.ftpd -l a
```

| 필드        | 의미                                                  |
| ----------- | ----------------------------------------------------- |
| 서비스명    | `/etc/services` 에 등록된 포트 번호를 참조            |
| 소켓타입    | TCP = `stream`, UDP = `datagram`                      |
| 프로토콜    | `/etc/protocols` 에 정의된 프로토콜 중 사용 가능한 것 |
| 플래그      | `nowait`(즉시 처리) / `wait`(대기 후 처리)            |
| 실행 경로명 | 절대 경로                                             |

#### 관련 파일

| 파일             | 의미                                                                                                     |
| ---------------- | -------------------------------------------------------------------------------------------------------- |
| `/etc/services`  | 인터넷 서비스의 포트·프로토콜 정의. **well-known(0–1023)**, registered(1024–49151), dynamic(49152–65535) |
| `/etc/protocols` | 사용하는 프로토콜과 식별 번호                                                                            |

> 둘 다 **IANA(Internet Assigned Numbers Authority)** 에서 관리한다.

### 8-3. 비활성화해야 할 취약 서비스

| 서비스군                                                               | 위험                                                                  |
| ---------------------------------------------------------------------- | --------------------------------------------------------------------- |
| **Simple TCP** — `echo(7)`, `discard(9)`, `daytime(13)`, `chargen(19)` | DoS 공격에 취약                                                       |
| **R 계열** — `rlogin`, `rsh`, `rexec`                                  | 인증 없이 원격 접속 가능. 기종 간 백업 등에 쓰였으나 보안상 매우 취약 |
| **불필요한 RPC** — `rpc`, `cmsd`, `rusersd` 등                         | 분산 환경 작업 호출 서비스. 다수의 BoF 취약점 존재                    |
| `Finger`, `tftp`, `talk`                                               | 정보 노출·전송 보안 부재                                              |

→ `inetd.conf` 에서 해당 서비스를 `grep` 한 뒤 `#` 으로 주석 처리하고 데몬을 재시작한다.

### 8-4. xinetd 설정

`xinetd` 는 `inetd` 의 비효율적 자원 관리·보안 문제를 개선했고 **TCP Wrapper 기능 + 자체 서비스별 접근제어** 를 제공한다.

| 파일                       | 역할                  |
| -------------------------- | --------------------- |
| `/etc/xinetd.conf`         | 모든 서비스 공통 설정 |
| `/etc/xinetd.d/[서비스명]` | 개별 서비스 설정      |

#### 일반 설정 예시

```
service [서비스명]
{
    disable      = [yes: 실행 안함 / no: 실행함]
    socket_type  = [TCP: stream / UDP: dgram]
    wait         = [no: 즉시 처리 / yes: 대기]
    user         = [실행 사용자]
    server       = [실행 파일 경로]
    only_from    = [접근 허용 IP/대역 (CIDR 표기)]
    no_access    = [접근 차단 IP/대역]
    access_times = [09:00-18:00]
    cps          = [50 10]    # 초당 최대 50 연결, 초과 시 10초간 제한
    instances    = [동시 처리할 최대 서버 개수]
    per_source   = [출발지 IP별 최대 연결 개수]
}
```

#### 로그 설정

| 옵션             | 의미                     |
| ---------------- | ------------------------ |
| `log_on_failure` | 접속 실패 시 기록할 내용 |
| `log_on_success` | 접속 성공 시 기록할 내용 |

### 8-5. TCP Wrapper

외부에서 들어오는 클라이언트에 대해 **호스트 단위 접근 통제** 를 제공한다.

| 파일               | 방식                                         |
| ------------------ | -------------------------------------------- |
| `/etc/hosts.allow` | **화이트리스트** — 등록된 호스트는 접근 허용 |
| `/etc/hosts.deny`  | **블랙리스트** — 등록된 호스트는 접근 거부   |

- 접근 시 `hosts.allow → hosts.deny` 순서로 확인. 기본은 모든 접근 허용.
- 형식: `서비스리스트 : 클라이언트리스트 [: 쉘 커맨드]`
- 일반적으로 `hosts.deny` 에 `ALL : ALL` 을 둬 모든 접근을 차단한 뒤 `hosts.allow` 에 명시적으로 허용한다.

```
# /etc/hosts.allow
in.telnetd : .aaa.com EXCEPT www.aaa.com
# www.aaa.com 을 제외한 aaa.com 도메인의 모든 호스트에 telnet 허용
```

쉘 커맨드는 `twist`(클라이언트에게 통보) 와 `spawn`(통보 없이 백엔드 실행) 이 있다.

### 8-6. 보안 쉘 (SSH)

암호 통신을 이용해 네트워크상 다른 컴퓨터에 접속·원격 명령·파일 송수신을 수행하는 응용·프로토콜.

- **22/tcp** 포트 사용
- `rsh`, `rlogin`, `telnet`, `ftp` 의 대안 (이들은 평문이라 스니핑에 취약)
- 설정: `/etc/ssh/sshd_config` 에서 `PermitRootLogin no` 로 root 원격 접속 제한

### 시험 포인트

- well-known 0–1023 의 범위.
- `inetd.conf` 의 7개 필드 순서.
- xinetd `only_from` / `no_access` / `cps` / `per_source` 의미.
- TCP Wrapper 의 `hosts.allow` 가 우선.
- SSH 22/tcp 포트, `PermitRootLogin no`.

## 9. 인증 모듈 — PAM

### 9-1. PAM 이란

**PAM(Pluggable Authentication Modules, 장착형 인증 모듈)** 은 리눅스에서 다양한 어플리케이션의 인증을 위해 제공되는 인증용 라이브러리다.

| 위치                                         | 역할                                                                 |
| -------------------------------------------- | -------------------------------------------------------------------- |
| `/lib/security/`, `/usr/lib/security/` 등    | PAM 모듈 라이브러리(`*.so`) — 배포판·아키텍처별 경로는 다를 수 있음  |
| `/etc/security/*`                            | PAM 모듈 실행에 필요한 추가 설정 파일                                |
| `/etc/pam.d/*`                               | PAM 을 사용하는 응용 프로그램별 설정 파일 (파일명 = 응용 프로그램명) |

**장점**: 인증 방식·정책의 유연성과 중앙 통제. 기존 응용 프로그램을 수정하지 않고도 새 인증 모듈을 추가할 수 있다.

### 9-2. PAM 설정 파일 구조

```
type   control      module-path   module-arguments
```

#### type — PAM 모듈의 종류

| type       | 의미                     |
| ---------- | ------------------------ |
| `account`  | 계정 검증 (만료·잠금 등) |
| `auth`     | 패스워드 검증            |
| `password` | 패스워드 변경 처리       |
| `session`  | 세션 시작·종료 처리      |

#### control — 모듈 실행 결과에 따른 행동

| control          | 의미                                                         |
| ---------------- | ------------------------------------------------------------ |
| `requisite`  | 인증 실패 시 **즉시 거부**                                   |
| `required`   | 실패해도 다음 라인의 모듈은 실행하지만 최종 결과는 인증 실패 |
| `sufficient` | 이전 요청들이 실패해도 여기서 성공하면 PAM 인증 성공         |
| `optional`       | 결과가 인증에 영향 없음                                      |

### 9-3. 주요 활용 예시

#### root 원격 접속 제한

```
/etc/securetty 파일에서 pts/0 ~ pts/* 를 모두 제거(또는 주석)
/etc/pam.d/remote 파일에 `auth required pam_securetty.so` 설정
```

> **`tty` 와 `pts` 의 차이**:
>
> - `tty` (terminal-teletype) — 서버 콘솔에 직접 로그인(모니터·키보드를 서버에 연결한 직접 접속)
> - `pts` (pseudo-terminal) — telnet·SSH 등 가상 터미널을 통한 원격 접속

#### SSH 서비스 root 원격 접속 제한

```
/etc/ssh/sshd_config 에서 PermitRootLogin no
service sshd restart
```

#### 계정 잠금 임계값 설정 (pam_tally)

```
/etc/pam.d/system-auth 파일에 추가:

auth     required  /lib/security/pam_tally.so deny=5 unlock_time=120 no_magic_root
account  required  /lib/security/pam_tally.so no_magic_root reset
```

| 옵션              | 의미                       |
| ----------------- | -------------------------- |
| `deny=5`          | 5회 입력 실패 시 계정 잠금 |
| `unlock_time=120` | 120초 후 자동 해제         |
| `no_magic_root`   | root 계정에는 잠금 미적용  |
| `reset`           | 성공 시 실패 횟수 초기화   |

확인·해제 명령:

- `pam_tally2 -u [계정명]` — 실패 횟수 확인
- `pam_tally2 -u [계정명] -r` — 실패 횟수 초기화

#### root 의 su 명령 사용 제한 (wheel 그룹)

```
usermod -G wheel [계정명]
/etc/pam.d/su 에 `auth required pam_wheel.so use_uid debug` 설정
```

→ `wheel` 그룹에 속한 계정만 `su` 로 root 전환 가능.

#### sudo 를 이용한 관리자 권한 부여

`/etc/sudoers` 파일에 다음 형식으로 기술.

```
계정명  호스트명=(실행권한계정명) [NOPASSWD:] 명령어
```

```
algisa ALL=(ALL) ALL
# algisa 는 sudo 로 모든 호스트에서 root 권한으로 모든 명령 실행

algisa ALL=(root,kiwi99,kiwi88) ALL
# algisa 는 root, kiwi99, kiwi88 권한으로 모든 명령 실행
```

`NOPASSWD:` 옵션을 사용하면 `sudo` 실행 시 비밀번호를 묻지 않는다.

### 시험 포인트

- PAM type 4종(account/auth/password/session) 매칭.
- control 4종 중 **requisite**(즉시 거부) ↔ **required**(계속하지만 결과는 실패) 의 차이.
- `tty` = 콘솔 직접, `pts` = 원격 가상 터미널.
- `pam_tally`, `pam_wheel`, `pam_securetty` 의 용도.

## 10. 로깅 시스템

### 10-1. syslog

**syslog** 는 메시지 로깅을 위한 표준으로, 커널 및 응용 프로그램에 의해 발생하는 로그를 체계적으로 생성·관리한다.

#### 동작 흐름

1. 커널 및 응용 프로그램이 **syslog API** 를 통해 로그 생성
2. **syslogd** 데몬 프로세스가 `/etc/syslog.conf` 설정 파일을 참조
3. 지정한 로그 파일·콘솔·원격 서버 등에 로그를 기록

#### `/etc/syslog.conf` 형식

```
facility.priority [; facility.priority …]    action(logfile-location)
```

| 항목                            | 값 예시                                                                                                          |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **facility** (로그 생성 서비스) | `*`, `auth/authpriv`, `cron`, `daemon`, `kern`, `lpr`, `mail`, `news`, `uucp`, `user`, `syslog`, `local0~local7` |
| **priority** (로그 수준)        | `emerg`, `alert`, `crit`, `err`, `warning`, `notice`, `info`, `debug` (`*` = 모든 수준, `none` = 기록 안 함)     |
| **action**                      | 로그 파일 경로 / `console`·`/dev` (콘솔) / `@호스트주소` (원격 로그 서버) / 사용자명 / `*` (모든 사용자)         |

> _원본 표기 보정_: 1과목.txt 의 `auth/ahtipriv`, `facillity` 는 OCR 오류. 실제 표기는 `auth/authpriv`, `facility`.

#### 보안 고려 (시험·실무 모두)

클래식 **BSD syslog**(RFC 3164) 는 전송 로그를 **평문**으로 다루는 경우가 많고, **UDP** 로 보내면 **전달 신뢰성·무결성·기밀성** 을 강하게 보장하지 않는다는 한계가 알려져 있다. **시험** 에서는 “syslog 는 기밀성·무결성을 전제로 설계되지 않았다” 를 **개념** 으로 묻는 경우가 많다.

**업무·표준** 측면에서는 **RFC 3195**(*Reliable Delivery for syslog*) 가 **TCP** 와 **BEEP**(Blocks Extensible Exchange Protocol) 를 써서 **신뢰 전달** 을 표준화한다. **rsyslog** 는 `syslogd` 의 확장 구현으로, **`imudp`(예: UDP 514)** · **`imtcp`(예: TCP 514)** 등 **모듈로 프로토콜을 선택**한다. 즉 “UDP 대신 TCP 만 쓴다” 가 아니라, **레거시 운용은 UDP 514 가 흔하고**, 요구사항에 따라 **TCP·TLS** 등을 쓰는 설계가 가능하다는 점을 구분해 두면 04 단원의 원격 로그 예제($UDPServerRun 514)와도 모순되지 않는다.

### 10-2. logrotate

시스템 로그 파일을 관리하기 위한 도구로, 로그 파일 순환·압축 등 다양한 기능을 제공한다.

| 파일/디렉터리               | 역할                                          |
| --------------------------- | --------------------------------------------- |
| `/usr/sbin/logrotate`       | logrotate 데몬 프로그램                       |
| `/etc/logrotate.conf`       | logrotate 데몬 설정 파일                      |
| `/etc/logrotate.d`          | logrotate 적용 프로세스의 개별 설정 파일      |
| `/var/lib/logrotate.status` | logrotate 작업 내역 보관                      |
| `/etc/cron.daily/logrotate` | 일 단위로 logrotate 를 실행하는 cron 스크립트 |

#### 주요 옵션

| 옵션                                  | 의미                                                        |
| ------------------------------------- | ----------------------------------------------------------- |
| `daily` / `weekly` / `monthly`        | 순환 주기                                                   |
| `rotate n`                            | 순환 로그 파일 개수                                         |
| `create [퍼미션] [소유자] [소유그룹]` | 새 로그 파일 생성 시 권한 설정 (예: `create 600 root root`) |
| `dateext`                             | 확장자에 날짜 추가                                          |
| `compress` / `nocompress`             | 압축 여부                                                   |
| `size n`                              | 지정 크기(`100K`, `100M`) 도달 시 순환                      |
| `missingok`                           | 로그 파일 부재 시 오류 미발생                               |
| `notifempty`                          | 비어있으면 순환하지 않음                                    |
| `sharedscripts`                       | `postrotate` 스크립트를 한 번만 실행                        |
| `delaycompress`                       | 압축 전 하루 지연                                           |
| `prerotate` / `postrotate`            | 작업 이전·이후 스크립트 실행                                |

### 10-3. 주요 로그 파일

| 경로                                                     | 형식            | 확인 명령             | 내용                                                                                                                                                                                                   |
| -------------------------------------------------------- | --------------- | --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `/var/run/utmp` (linux) `/var/adm/utmpx` (unix)          | binary          | `w`, `who`, `finger`  | **현재 로그인** 한 사용자의 상태 정보                                                                                                                                                                  |
| `/var/log/wtmp` (linux) `/var/adm/wtmpx` (unix)          | binary          | `last`, `last [계정]` | **성공한 로그인/로그아웃** 정보 + 시스템 boot/shutdown 정보 (`reboot` 사용자로 확인), 성공 이력                                                                                                        |
| `/var/log/lastlog` (linux) `/var/adm/lastlog` (unix)     | binary          | `lastlog`, `finger`   | **가장 최근 성공 로그인** 기록                                                                                                                                                                         |

#### `/var/log/xferlog` 의 14개 필드

```
1. 전송 날짜 및 시간 (Thu May 7 05:12:30 2012 형식)
2. 전송 소요 시간 (초 단위)
3. 원격 호스트 주소
4. 전송 파일 크기 (바이트)
5. 전송 파일명 (경로)
6. 전송 유형 — a(아스키), b(바이너리)
7. 액션 플래그 — _(없음), C(압축), U(압축 해제), T(아카이브)
8. 전송 방향 — i(업로드), o(다운로드), d(삭제)
9. 접근 방식 — r(로컬 계정), a(익명), g(게스트)
10. 사용자명
11. 서비스명 (일반적으로 ftp)
12. 인증 방법 — 0(없음), 1(RFC 931 인증)
13. 인증된 사용자 ID — *(이용 불가)
14. 완료 상태 — c(성공), i(실패)
```

### 시험 포인트

- 로그 파일 매핑 — **현재 로그인 = utmp / 성공 로그인 = wtmp / 실패 로그인 = btmp / 최근 로그인 = lastlog**.
- `pacct` 는 기본 비활성, `accton` 으로 활성화.
- `xferlog` 의 **전송 유형 a/b**, **방향 i/o/d**, **완료 상태 c/i** 매칭.
- `auth/authpriv`, `facility`, `priority` 등 syslog 용어.

## 단원 요약

1. **로그인·부팅** — passwd → shadow 비교, init(PID 1), BIOS → 부트프로그램 → 커널 → init.
2. **계정 파일** — passwd(7) / group(4) / shadow(9) 필드. `$5$` = SHA-256, `$6$` = SHA-512.
3. **파일시스템** — 부트/슈퍼/inode/데이터 블록. inode 에 파일명 없음. MAC Time 3종.
4. **재지정·링크** — `>`, `>>`, `2>&1`. 하드(동일 inode, 같은 FS) vs 심볼릭(경로, FS 무관, 디렉터리 가능).
5. **권한** — 파일 666 / 디렉토리 777, umask 022. SUID(4) / SGID(2) / Sticky(1). RUID·EUID.
6. **프로세스** — init = PID 1. 좀비는 `kill` 무시. SIGKILL(9) / SIGSTOP(23) 무시 불가.
7. **스케줄러** — crontab 6필드. allow → deny 우선. 둘 다 없으면 root 만.
8. **서비스 운영** — Stand-Alone vs (x)inetd. Simple TCP(7/9/13/19) DoS, R 계열·RPC 차단. SSH 22/tcp.
9. **PAM** — type 4종, control 4종 (requisite vs required vs sufficient).
10. **로깅** — syslog 표준·rsyslog; 원격 전달은 **UDP 514** 가 전통적으로 흔하고 RFC 3195·rsyslog 모듈로 **TCP·BEEP** 등 신뢰 전달도 가능. 로그 파일: utmp(현재) / wtmp(성공) / btmp(실패) / lastlog(최근).
11. /var은 variable 가변적인 데이터로 로그, 스풀, 캐시 처럼 운영 중 계속 변경되는 파일이 저장되고, /etc는 상대적으로 고정적인 설정/기타 파일이 저장된다.

## 각주 — 외부 1차 출처

[^1]: `crypt(3)` 의 `$id$` 표준은 Linux man-pages 의 `man 3 crypt` 에 정의되어 있다. `1` = MD5, `2a/2y` = Blowfish(bcrypt), `5` = SHA-256, `6` = SHA-512. 1과목.txt 의 `3 → SHA-256, 4 → SHA-512` 는 OCR 또는 표기 오류이며, 통합자료의 `5: SHA-256` 표기가 실제 표준과 일치한다.

[^2]: **RFC 3195**(*Reliable Delivery for syslog*) 는 BEEP(Blocks Extensible Exchange Protocol) 매핑을 포함한 **신뢰 전달** 을 다룬다. **RFC 3164** 계열의 전통적 syslog 는 **UDP** 로 동작하는 사례가 많고 평문·비신뢰성 이슈가 지적되어 왔다. **rsyslog** 는 UDP·TCP 등 **입력·출력 모듈** 로 전송 방식을 고를 수 있으며, 본문은 “자격 시험에서 묻는 개념(한계)”과 “RFC·구현에서의 보완”을 구분해 설명한다.
