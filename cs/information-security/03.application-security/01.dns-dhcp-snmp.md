# 01. DNS·DHCP·SNMP — 인터넷 기초 응용 서비스

## 0. 들어가며

이 단원은 인터넷 응용 계층의 기초 인프라 서비스 세 가지 — **DNS** (도메인 이름 해석), **DHCP** (네트워크 주소 자동 할당), **SNMP** (네트워크 자원 관리) — 의 동작 원리와 보안 위협, 보안 기술을 다룬다. 세 서비스 모두 사용자가 직접 인터넷을 사용하기 위한 가장 밑단의 인프라 서비스이며, 기본 설계 시점에 보안을 고려하지 않았기 때문에 평문 전송·인증 부재·캐시 변조 등 구조적 취약점을 공유한다.

### 출제기준 매핑

| 출제기준 | 본 단원 매핑 | 학습 대상 구분 |
|---|---|---|
| 1.4 DNS 보안 (DNS 개념 / 서비스 운영 / 공격유형 / 보안 기술) | §1 ~ §5 | **1차 학습 대상** (필기 출제기준 직접 명시) |
| (보강) DHCP | §6 | 보강 — 통합자료가 같은 "어플리케이션 기본" 으로 묶음 |
| (보강) SNMP | §7 | 보강 — 실기 출제기준 "1.4 보안장비 및 네트워크 장비별 보안특성 파악하기" 에 SNMP 직접 명시 |

### 학습 목표

1. DNS 의 재귀적·반복적 질의 흐름을 설명하고 어떤 단계에서 캐시·평문·UDP 비연결 특성이 보안 위협으로 이어지는지 설명할 수 있다.
2. DNS Spoofing(스니핑 기반) 과 DNS Cache Poisoning 의 공격 메커니즘 차이를 비교하고 각각의 1차 대응책을 제시할 수 있다.
3. DNSSEC 가 기존 DNS 의 어떤 취약점을 어떤 방식으로 보강하는지(공개키 서명) 설명할 수 있다.
4. DHCP 의 IP 할당 4단계(DORA) 와 DHCP Starvation Attack 의 공격 흐름·대응을 설명할 수 있다.
5. SNMP v1·v2·v3 의 보안 차이를 정량적으로 비교하고 SNMPv3 USM·VACM 의 보안 서비스를 나열할 수 있다.

## 1. DNS 개요와 동작 원리

### 1-1. DNS 의 정의와 역할

> **DNS (Domain Name System)** : 사용자가 숫자로 된 IP 주소 대신 인터넷 도메인 이름과 검색 가능한 URL 을 사용하여 웹사이트에 접속할 수 있도록, 도메인 이름과 IP 주소를 변환해 주는 인터넷 핵심 인프라 서비스.

DNS 는 인간이 기억하기 쉬운 도메인 이름을 컴퓨터가 사용하는 IP 주소로 변환(또는 그 반대)하는 분산 데이터베이스 시스템이다. RFC 1034·1035[^1][^2] 에 정의되어 있으며 최상위 루트(root) 부터 TLD (Top-Level Domain) → 2차 도메인 → 호스트 까지 이어지는 계층 구조로 운영된다. DNS 는 분산·계층 구조이므로 어느 한 서버가 모든 도메인 정보를 보유하지 않고, 자신이 권한을 가진 도메인 정보만 보유하면서 위임된 다른 도메인은 다른 네임서버로 질의를 넘기는 방식으로 동작한다.

| DNS 의 핵심 특성 | 설명 |
|---|---|
| 비연결 특성 | 클라이언트는 DNS 서버와 연결 상태를 유지하지 않는다 (UDP 기반) |
| 응답 특성 | 먼저 수신한 응답을 신뢰하고 이후에 수신한 응답은 모두 폐기 |
| 캐시 사용 | 반복적 질의 요청으로 부하가 발생하는 것을 막기 위해 결과를 일정 기간 캐시 |
| TTL 유지 | 캐시 정보를 일정 시간 동안 유지되도록 보장하는 메커니즘 |
| 질의/응답 식별 | Transaction ID, 출발지/목적지 주소(IP·Port) 로 질의·응답 매칭 |

> *시험 포인트* : "DNS 는 먼저 받은 응답을 신뢰" 와 "비연결 (UDP)" 두 특성이 결합되어 **DNS Spoofing** 의 근본 원인을 이룬다. 이 둘은 DNS 보안 문제의 핵심 키워드이므로 쌍으로 외운다.

### 1-2. DNS 포트 사용

DNS 는 질의·응답 데이터의 크기에 따라 두 개의 전송 계층 프로토콜을 다르게 사용한다.

| 데이터 크기/용도 | 프로토콜·포트 | 사용 사례 |
|---|---|---|
| 질의/응답 데이터가 512 byte 이하 | **53/udp** | 일반 도메인 조회 |
| 질의/응답 데이터가 512 byte 초과 | **53/tcp** | 응답 크기 초과 시 폴백 |
| 존 전송 (Zone Transfer) | **53/tcp** | 마스터-슬레이브 동기화. 신뢰성 필요 |

> *시험 포인트* : "**존 전송은 53/tcp**" 는 거의 매 회차 빈출. UDP 가 아닌 이유는 존 데이터의 신뢰성 있는 동기화가 필요하기 때문이다.

### 1-3. DNS 질의 과정

DNS 질의는 사용자 호스트의 DNS Resolver 가 시작하여 **재귀적 질의 (Recursive Query)** 와 **반복적 질의 (Iterative Query)** 두 방식의 조합으로 수행된다. 이 두 질의 방식의 구분은 시험에서 정의·차이가 직접 출제되므로 정확히 이해해야 한다.

| 질의 방식 | 수행 주체 | 동작 | 특성 |
|---|---|---|---|
| **재귀적 질의 (Recursive Query)** | 사용자 호스트 → Recursive 네임서버 | "결과를 끝까지 찾아서 알려줘" | 클라이언트는 한 번 요청하고 최종 결과를 받음 |
| **반복적 질의 (Iterative Query)** | Recursive 네임서버 → Authoritative 네임서버 | "네가 모르면 다음 단계 정보만 알려줘" | 단계별로 다음 네임서버를 안내받아 진행 |

DNS 질의의 전체 흐름은 다음 6 단계로 요약된다.

1. 사용자는 도메인/URL 에 대한 IP 주소를 네임서버에 질의 (재귀적 질의).
2. DNS Resolver 는 캐시를 확인하고, 그 다음 hosts 파일을 검색하여 결과를 반환.
3. 캐시나 hosts 파일에 데이터가 없으면, DNS Resolver 는 재귀 DNS 서버로 질의.
4. 재귀 DNS 서버는 다른 네임서버에 대한 반복적 질의를 수행하고 결과를 반환.
5. 사용자는 반환된 IP 주소를 받아와서 일정 기간 동안 자신의 DNS 캐시에 저장.
6. 사용자는 해당 IP 주소를 사용하여 대상 사이트로 접속.

> *원본 표기 보정*: 통합자료 line 3138 의 "공격 대상 DNS 서버가 반복적 질의 시 사용하는 정보를 모르게 때문에" 는 "모르기 때문에" 의 OCR 결함이다.

### 1-4. 윈도우 호스트의 DNS 질의 우선순위

윈도우 호스트가 도메인을 해석할 때 다음 순서로 검색한다.

```
로컬 DNS 캐시 → hosts.ics → hosts 파일 → 시스템에 설정된 네임 서버
```

`hosts.ics` 파일은 윈도우에서 인터넷 연결 공유(ICS, Internet Connection Sharing) 사용 시 클라이언트의 IP 를 강제로 지정하는 기능에 사용된다.

리눅스/유닉스 환경에서는 `/etc/host.conf` 파일에 명시된 `order` 항목 순서로 질의를 수행한다.

> *시험 포인트* : "**hosts 파일은 DNS 질의보다 우선순위가 높다**" — 이 특성은 §4 DNS Spoofing 대응책의 핵심 근거가 된다 (중요 사이트는 hosts 파일에 등록).

## 2. 네임서버 구성과 운영

### 2-1. 네임서버의 종류

네임서버는 역할에 따라 두 가지로 분류된다. 이 둘의 구분은 §5 Bind DNS 보안 (재귀적 질의 제한) 의 전제 조건이므로 명확히 이해해야 한다.

| 네임서버 종류 | 역할 | 일반적 사용처 |
|---|---|---|
| **Recursive/Cache 네임서버** | 관리·위임받은 도메인 없이, 사용자 질의를 받아 캐시 또는 반복적 질의로 결과를 응답 | ISP 가 가입자에게 제공하는 DNS 서버 |
| **Authoritative 네임서버** | 위임받은 도메인을 가지고 있는 네임서버. 해당 도메인에 대한 질의에만 응답 | 도메인 소유 기관의 자체 DNS 서버 |

마스터/슬레이브 구성은 위 분류와 직교하는 개념으로, **Authoritative 네임서버를 이중화하기 위한 구조**다.

| 구분 | 역할 |
|---|---|
| **마스터 (Master) 네임서버** | 원본 존 데이터를 관리. 관리자가 설정한 존 파일을 직접 운용 |
| **슬레이브 (Slave) 네임서버** | 마스터로부터 존 데이터를 동기화하여 사용. 부하 분산·이중화 목적 |

### 2-2. 네임서버 구성 파일

리눅스/유닉스 환경의 네임서버 관련 주요 설정 파일은 다음과 같다.

| 파일 | 용도 |
|---|---|
| `/etc/named.conf` | 존 설정을 할 수 있는 네임서버 설정 파일 |
| `/etc/resolv.conf` | 시스템 기본 네임서버 설정 정보를 담고 있는 파일 |
| `/etc/hosts` | 도메인/호스트명과 IP 주소 매핑 정보를 담고 있는 파일 (네임서버 질의 전 먼저 참조) |
| `/etc/host.conf` | DNS 질의 순서를 지정하는 파일 |

`/etc/hosts` 파일은 DNS 질의보다 우선순위가 높으므로 파밍(Pharming) 공격에 악용되는 사례가 자주 보고된다. 악성코드가 hosts 파일을 변조하면 정상 도메인을 입력해도 공격자 사이트로 접속하게 된다.

### 2-3. 존(Zone) 데이터와 존 파일

> **존 (Zone)** : 네임서버가 관리하는 도메인 영역.
> **존 파일 (Zone File)** : 관리 도메인에 대한 정보를 담고 있는 파일. 네임서버 기동 시 이 파일을 읽어 존 데이터를 구성한다.

존 파일의 주요 리소스 레코드는 다음 형식으로 구성된다.

```
host_name [TTL] class record_type data

예시) www 300 IN A 192.168.56.100
     ns1     IN A 192.168.56.10
```

| 필드 | 설명 |
|---|---|
| `host_name` | 등록할 호스트명 지정 |
| `TTL` | 레코드의 TTL 값 지정 (미지정 시 default 적용) |
| `class` | 인터넷 클래스를 의미하는 `IN` 지정 |
| `record_type` | 레코드 유형 (A, AAAA, MX, NS 등) |
| `data` | 레코드 유형에 따른 데이터 |

리버스 도메인은 IP 에 대한 도메인 명 질의를 위한 특수 도메인이며, 리버스 존 파일은 PTR 레코드를 이용하여 IP 주소에 대한 도메인 명을 설정한다.

### 2-4. 존 전송 (Zone Transfer)

> **존 전송 (Zone Transfer)** : 마스터에 있는 원본 존 데이터를 슬레이브가 동기화하는 작업.

존 전송은 신뢰성이 필요하기 때문에 53/tcp 를 사용한다. 존 전송 자체가 악용되면 도메인의 서버 정보가 통째로 노출되거나 DoS 의 발판이 될 수 있어 §5-4 에서 다루는 `allow-transfer` 제한이 필수적이다.

### 2-5. DNS 캐시 명령

| 명령 | 용도 |
|---|---|
| `ipconfig /displaydns` | 로컬 DNS 캐시 정보 조회 (DNS Spoofing 시 조작된 주소 정보 확인 가능) |
| `ipconfig /flushdns` | 로컬 DNS 캐시 정보 삭제 |

TTL 이 만료되면 해당 캐시 정보는 자동으로 삭제된다. 의심스러운 도메인 접속이 발생한 경우 `flushdns` 로 캐시를 비운 후 재접속하여 DNS Spoofing 가능성을 확인하는 것이 1차 점검 절차다.

### 2-6. DNS 도구

| 도구 | 설명 |
|---|---|
| **DNS Lookup** | DNS 레코드를 조회할 때 사용하는 명령. 순방향 룩업 (forward, 도메인 → IP) 과 역방향 룩업 (reverse, IP → 도메인) 으로 구분 |
| **nslookup** | 대화형 모드 또는 `nslookup [도메인] [네임서버]` 형식으로 질의 |
| **dig** | nslookup 을 대체하기 위한 목적으로 만들어진 명령어. 유닉스/리눅스 계열 OS 에 기본 설치 |
| **whois** | whois 서버를 통해 도메인의 등록정보·네트워크 할당 정보 등을 조회 |

`dig` 명령의 주요 쿼리 옵션은 다음과 같다.

```
dig [@네임서버] 도메인 [쿼리유형] [쿼리옵션]
```

| 쿼리 옵션 | 용도 |
|---|---|
| `+norecurse` | Authoritative 네임서버에 반복적 질의를 수행하여 정상 응답 여부를 점검 |
| `+tcp` | 53/tcp 허용 여부를 점검 |
| `+trace` | 계층적 위임 상태 점검. 최상위 루트 도메인부터 계층구조에 따른 질의 수행 |

`whois` 명령은 두 가지 용도로 사용된다.

```
whois [도메인 명]    # 도메인의 등록정보·소유정보·네임서버 정보 조회
whois [IP 주소]      # 해당 IP 의 네트워크 대역을 관리하는 대행자 정보 조회
```

> *시험 포인트* : `dig +trace` 는 DNS 계층 구조를 단계별로 추적하므로 DNS 위임 관계 점검의 표준 도구로 출제된다.

## 3. DNS 레코드 타입

DNS 레코드는 도메인의 어떤 정보를 질의할지를 지정하는 식별자다. 시험에서는 각 레코드의 용도와 함께 **DRDoS 공격에 악용되는 레코드** (ANY, TXT 등) 를 구분하는 항목이 출제된다.

| 레코드 타입 | 설명 | 비고 |
|---|---|---|
| **A** (Address) | 도메인에 대한 IP(IPv4) 주소 질의. 순방향 lookup | 가장 기본적인 레코드 |
| **AAAA** | 도메인에 대한 IPv6 주소 질의 | |
| **ANY** | 도메인에 대한 모든 레코드 질의 | DRDoS 공격에 악용 |
| **MX** (Mail Exchanger) | 도메인의 메일서버 질의 | |
| **NS** (Name Server) | 도메인의 네임서버 질의 | |
| **SOA** (Start Of Authority) | 존의 기본 속성 정보 (버전·존 전송 주기 등) 질의 | 존 파일의 시작을 표현 |
| **TXT** (Text) | 도메인에 대한 텍스트 정보 질의 | SPF·DKIM·DMARC 레코드 등록에 사용. DRDoS 공격에 악용 |
| **PTR** (Pointer) | IP 에 대한 도메인 정보 질의. A 레코드의 반대 | 리버스 존 파일에서 사용 |
| **CNAME** (Canonical Name) | 호스트의 별명(alias) 정의 | |
| **HINFO** (Host Information) | 호스트의 CPU·OS 정보 | |
| **AXFR** (Authoritative Zone Transfer) | 존 버전에 상관없이 무조건 존 전송 요청 | |
| **IXFR** (Incremental Zone Transfer) | 존 버전을 비교하여 상위 버전일 경우 존 전송 요청 | |

> *시험 포인트* : ANY 와 TXT 가 DRDoS 공격에 악용되는 이유는 응답 패킷 크기가 질의 패킷 크기보다 훨씬 크기 때문(증폭률이 큼) 이며, 이 특성은 2과목 §5 DRDoS 와 cross-ref 된다.

## 4. DNS 공격 기법

DNS 의 두 가지 핵심 공격 — **스니핑 기반 DNS Spoofing** 과 **DNS Cache Poisoning** — 은 둘 다 DNS 응답을 조작한다는 점에서 같지만 *공격 대상*과 *영향 범위*에서 차이가 크다. 시험은 두 공격의 흐름·대응책 차이를 정확히 묻는다.

### 4-1. 스니핑 기반 DNS Spoofing

> **스니핑 기반 DNS Spoofing** : 희생자에게 전달되는 DNS 응답을 조작하여 희생자가 의도하지 않은 주소로 접속하게 만드는 공격.

이 공격은 동일 네트워크 내에서 ARP Spoofing 등을 통해 희생자의 트래픽을 스니핑하여 Transaction ID·출발지/목적지 정보를 알아낸 뒤 정상 응답보다 빠르게 조작된 응답을 전송하는 방식이다. DNS 의 "먼저 수신한 응답을 신뢰" 특성을 악용한다.

**공격 단계**

1. 공격자는 무차별 모드(Promiscuous mode) 나 ARP Spoofing 을 통해 희생자를 스니핑.
2. 희생자가 DNS 질의를 수행 (트래픽에 Transaction ID 노출).
3. 공격자는 IP 와 Port 를 조작하고 Transaction ID 를 설정.
4. 정상 응답보다 빠르게 희생자에게 조작된 DNS 응답을 전송.
5. 조작된 응답 이후에 도착하는 정상 응답을 폐기 (DNS 응답 특성).
6. 희생자는 정상 주소를 입력해도 조작된 주소로 접속.

**핵심 특성**: 일반적으로 동일 네트워크 내에서 발생하기 때문에 상대적으로 멀리 떨어져 있는 DNS 서버보다 공격자가 빠르게 조작된 응답을 보낼 수 있다. **공격 대상은 희생자 1명**, 영향 범위는 해당 희생자에 한정된다.

### 4-2. DNS Cache Poisoning

> **DNS Cache Poisoning** : DNS 서버의 캐시 정보를 조작하여 해당 서버에 질의하는 모든 사용자가 의도하지 않은 주소로 접속하게 만드는 공격.

이 공격은 Recursive DNS 서버를 직접 공격하여 캐시에 조작된 정보를 주입한다. 한 번 캐시에 조작된 정보가 저장되면 TTL 이 만료될 때까지 해당 서버에 질의하는 모든 사용자가 영향을 받기 때문에 **공격 대상은 DNS 서버 1대 / 영향 범위는 그 서버의 모든 사용자** 로 1:N 피해가 발생한다.

**공격 단계**

1. 공격자는 공격 대상 DNS 서버에 조작할 도메인 질의를 다수 전송.
2. 공격 대상 DNS 서버가 반복적 질의 시 사용하는 Transaction ID 와 출발지 Port 를 모르기 때문에 정보를 다수 생성하여 임의로 시도.
3. 공격 대상 DNS 서버가 반복적 질의를 수행하는 동안 공격자는 다수의 조작된 DNS 응답을 전송.
4. 조작된 응답 중 정상 응답보다 먼저 일치하는 응답이 있으면 DNS 서버의 캐시에 저장.
5. 이를 질의하는 사용자는 조작된 주소의 사이트로 접속.

**생일 공격 (Birthday Attack) 응용**: 공격자가 랜덤한 Transaction ID 를 다수 생성해서 DNS Cache Poisoning 공격을 하는 것을 생일 공격 기법을 이용했다고 표현한다. 생일 공격은 우리가 예상하는 확률보다 실제 충돌 확률이 뜻밖에 높다는 이론에 기반한다 ([4과목 06 단원 §4-2](../04.security-general/06.hash-and-mac.md) 생일 공격 cross-ref).

### 4-3. 두 공격의 비교

| 구분 | 스니핑 기반 DNS Spoofing | DNS Cache Poisoning |
|---|---|---|
| 공격 대상 | 희생자(클라이언트) | DNS 서버 |
| 영향 범위 | 1:1 (해당 희생자만) | 1:N (DNS 서버에 질의하는 모든 사용자) |
| 공격 위치 | 동일 네트워크 내 (스니핑 가능 위치) | 원격 가능 |
| 핵심 기법 | 스니핑으로 Transaction ID 획득 | Transaction ID 를 무차별 추측 |
| 지속성 | 1회성 | 캐시 TTL 동안 지속 |

> *시험 포인트* : 두 공격의 **공격 대상**(희생자 vs DNS 서버) 과 **영향 범위**(1:1 vs 1:N) 차이는 시험에서 변별 핵심으로 출제된다.

## 5. DNS 보안 기술

### 5-1. DNS Spoofing 대응

| 대응 방안 | 설명 |
|---|---|
| 스니핑 탐지 및 차단 | 기본적으로 스니핑을 이용하기 때문에 스니핑 자체를 탐지·차단 (2과목 §4 cross-ref) |
| hosts 파일 관리 | DNS 질의보다 우선순위가 높은 hosts 파일에 중요 사이트 IP 를 등록하여 관리 |

### 5-2. DNS Cache Poisoning 대응

| 대응 방안 | 설명 |
|---|---|
| 네임서버 버전 | 네임서버의 Bind DNS 를 최신 버전 상태로 유지 (알려진 취약점 패치) |
| 재귀적 질의 제한 | 도메인 관리용 DNS 서버는 재귀적 질의를 허용하지 않도록 설정 |
| 사용자 제한 | 제한된 사용자가 사용하는 캐시 DNS 서버라면 해당 사용자만 사용하도록 제한 |
| **DNSSEC** | 기존의 DNS 에 공개키 암호화 방식의 보안 기능을 추가 부여 |

### 5-3. DNSSEC (DNS Security Extensions)

> **DNSSEC** : 데이터 위변조 공격에 취약한 DNS 의 문제점을 근본적으로 보안 개선하기 위해 IETF 에 의해 2005년 RFC 4033·4034·4035 로 표준화된 DNS 보안 확장. 기존의 DNS 에 공개키 암호화 방식의 보안 기능을 추가 부여하여 DNS 응답의 *출처 인증* 과 *무결성* 을 제공한다.

**중요한 한계**: DNSSEC 은 **기밀성(Confidentiality) 을 제공하지 않는다.** DNS 응답은 여전히 평문으로 전송되며, DNSSEC 은 응답이 정당한 네임서버에서 왔는지(출처 인증) 와 응답이 위변조되지 않았는지(무결성) 만 보장한다[^3].

DNSSEC 의 주요 추가 레코드는 다음과 같다.

| 레코드 | 역할 |
|---|---|
| **DNSKEY** | 존이 사용하는 공개키를 게시 |
| **RRSIG** | 각 RRset 에 대한 디지털 서명값 |
| **DS** (Delegation Signer) | 부모 존에 자식 존의 KSK 해시를 게시하여 신뢰 사슬 형성 |
| **NSEC / NSEC3** | 도메인이 존재하지 않음을 인증 가능한 형태로 응답 |

기밀성이 추가로 필요하면 **DoH (DNS over HTTPS, RFC 8484)**[^4] 또는 **DoT (DNS over TLS, RFC 7858)**[^5] 를 함께 적용해야 한다 — DNSSEC 은 무결성·출처 인증, DoH/DoT 는 전송 구간 기밀성을 담당한다.

> *시험 포인트* : "DNSSEC 은 기밀성을 제공하지 않는다" 는 빈출 함정이다. DNSSEC = 무결성·출처 인증, 기밀성 ≠ DNSSEC.

### 5-4. Bind DNS 보안

도메인 네임서버의 운영 측면에서 가장 중요한 두 가지 보안 설정은 **재귀적 질의 제한** 과 **존 전송 제한** 이다. 둘 다 `/etc/named.conf` 파일에서 설정한다.

#### 재귀적 질의 제한

악의적 공격자에 의한 과도한 재귀적 질의 요청은 DoS 형태의 공격으로 악용될 수 있으며 DNS Cache Poisoning 공격 노출을 키운다. 따라서 도메인 관리용(Authoritative) DNS 서버는 재귀적 질의를 허용하지 않도록 설정한다.

```
recursion no;                                    // 재귀적 질의를 허용하지 않음
allow-recursion {none;};                         // 재귀적 질의를 허용하지 않음
allow-recursion {127.0.0.1; 192.168.56.0/24;};   // 제한적으로 허용
allow-recursion {internal;};                     // 명시한 acl 명으로 제한적 허용
```

#### 존 전송 제한

존 전송은 지속해서 요청되면 네트워크·시스템 자원을 소모시킬 수 있고, 존 데이터에 저장된 도메인의 서버 정보가 그대로 노출될 수 있다. Master DNS 서버만 운영하는 경우 존 전송을 허용하지 않고, Master 와 Slave 를 운영하는 환경에서는 지정한 Slave 에서만 존 전송을 요청하도록 설정한다.

```
allow-transfer {none;};            // 존 전송을 허용하지 않을 경우
allow-transfer {[Slave IP];};      // Slave 만 허용할 경우
```

`dig` 명령의 `axfr` / `ixfr` 질의 유형을 통해 존 전송 허용 여부를 테스트할 수 있다.

### 5-5. DoH·DoT (외부 보강)

| 표준 | RFC | 사용 포트 | 특성 |
|---|---|---|---|
| **DoT** (DNS over TLS) | RFC 7858 (2016)[^5] | 853/tcp | DNS 질의·응답을 TLS 위에서 전송. 별도 포트 사용으로 식별·차단이 비교적 쉬움 |
| **DoH** (DNS over HTTPS) | RFC 8484 (2018)[^4] | 443/tcp | DNS 질의·응답을 HTTPS 위에서 전송. HTTPS 트래픽과 구분되지 않아 차단 회피에 유리 |

> *시험 포인트* : DoH 와 DoT 모두 **전송 구간 기밀성** 을 제공하지만 DNSSEC 처럼 응답의 출처 인증을 제공하지는 않는다. 두 메커니즘은 **상호 배타적이 아니라 상호 보완적** 이다.

## 6. DHCP (보강)

### 6-1. DHCP 의 정의와 동작

> **DHCP (Dynamic Host Configuration Protocol)** : 동적으로 클라이언트의 네트워크 주소를 설정하기 위한 프로토콜. RFC 2131 에 정의되어 있다[^6].

DHCP 는 클라이언트가 네트워크에 처음 접속할 때 서버로부터 IP 주소·서브넷 마스크·기본 게이트웨이·DNS 서버 정보를 자동으로 할당받게 하는 프로토콜이다.

| 구분 | 포트 |
|---|---|
| DHCP 서버 | **67/udp** |
| DHCP 클라이언트 | **68/udp** |

**DHCP IP 할당 4단계 (DORA)**

DHCP 의 IP 할당 절차는 통상 **DORA (Discover - Offer - Request - Ack)** 4 단계로 표현된다.

| 단계 | 메시지 | 설명 |
|---|---|---|
| 1 | **DHCP Discover** | 클라이언트가 DHCP 서버를 찾기 위한 메시지. 자신의 MAC 정보를 담아서 브로드캐스트 |
| 2 | **DHCP Offer** | DHCP 서버가 클라이언트에게 IP 정보를 제공해 주는 메시지 |
| 3 | **DHCP Request** | 클라이언트가 해당 IP 정보를 사용하겠다고 요청하는 메시지 |
| 4 | **DHCP Ack** | DHCP 서버가 할당된 주소 정보를 클라이언트에게 전송. DHCP 서버는 클라이언트의 정보를 테이블에 저장 |

> *원본 표기 보강* : 통합자료 line 3013 의 "MAC 브로드캐스트를 수행" 표현은 DHCP Discover 의 브로드캐스트 특성을 가리킨다. 실제 패킷은 IP 계층에서 `255.255.255.255` 브로드캐스트, MAC 계층에서 `FF:FF:FF:FF:FF:FF` 브로드캐스트로 전송된다.

**관련 명령어**

| 명령 | 용도 |
|---|---|
| `ipconfig /release` | 할당받은 IP 주소를 해제 |
| `ipconfig /renew` | 새로운 IP 주소를 DHCP 서버로부터 받음 |

### 6-2. DHCP Starvation Attack

> **DHCP Starvation Attack** : DHCP 서버의 할당 가능한 IP 를 모두 소진하게 만들어 정상 클라이언트의 IP 할당이 불가능하게 만드는 공격.

**공격 단계**

1. 공격자는 Discover 메시지를 서로 다른 MAC 주소로 대량으로 전송.
2. DHCP 서버는 해당 클라이언트들에게 IP 정보를 제공해 주는 Offer 메시지를 전송.
3. 공격자는 DHCP 서버에게 Request 메시지까지 보낸 후 실제로는 할당하지 않음.
4. DHCP 서버의 할당 가능한 IP 풀이 소진되면 정상 클라이언트는 IP 를 받지 못해 네트워크 사용 불가.

DHCP Starvation 의 본질은 *서로 다른 MAC 으로 가장한 다수의 가짜 클라이언트* 가 IP 를 점유하는 자원 소진(Resource Exhaustion) 공격이며, 2과목 §5 DoS 영역과 패턴이 동일하다.

### 6-3. DHCP 보안 (외부 보강)

| 대응 방안 | 설명 |
|---|---|
| **DHCP Snooping** | 스위치 단에서 신뢰할 수 있는(trusted) 포트만 DHCP Offer/Ack 를 보낼 수 있도록 제한. Rogue DHCP 서버 차단 |
| **Port Security** | 스위치 포트당 학습되는 MAC 주소 개수를 제한하여 가짜 MAC 대량 생성을 차단 |
| **DHCP Option 82 (Relay Agent)** | RFC 3046[^7]. DHCP 릴레이 에이전트가 요청 패킷에 자신의 식별 정보(Circuit ID·Remote ID) 를 추가하여 서버가 요청의 출처를 검증 가능 |
| **고정 IP 운영** | 중요 서버는 DHCP 가 아닌 정적 할당 사용 |

> *시험 포인트* : DHCP Snooping 과 Port Security 의 조합이 가장 일반적인 운영 환경 대응책이다. Option 82 는 ISP·대규모 사업자 환경에서 사용된다.

## 7. SNMP (보강)

### 7-1. SNMP 의 정의

> **SNMP (Simple Network Management Protocol)** : 네트워크상의 각 호스트로부터 정기적으로 여러 관리 정보를 자동으로 수집하거나 실시간으로 상태를 모니터링 및 설정할 수 있는 서비스.

SNMP 는 응용 계층(Application Layer) 프로토콜로 메시지가 단순히 요청과 응답 형식의 프로토콜에 의해 교환되기 때문에 전송 계층 프로토콜로는 UDP 를 사용한다. SNMP 는 프로토콜일 뿐이며 이를 활용하여 실제 네트워크 관리 정보를 얻기 위해서는 관련 프로그램(예: MRTG — Multiple Router Traffic Grapher) 이 준비되어야 한다.

### 7-2. NMS (Network Management System) 구조

> **NMS (Network Management System)** : 네트워크상의 자원들을 모니터링하고 제어하기 위한 도구. 관리자(Manager) — 대행자(Agent) 구조로 구성된다.

| 구성 요소 | 역할 | 포트 |
|---|---|---|
| **Manager** (관리 시스템) | Agent 에 필요한 정보를 요청하는 역할 | **162/udp** (Trap 수신) |
| **Agent** (관리 대상) | 관리 대상 시스템에 설치되어 필요한 정보를 수집하고 Manager 에게 전달 | **161/udp** (Get/Set 수신) |

**SNMP 가 통신하기 위한 최소 일치 사항**

Manager 와 Agent 가 통신하려면 다음 세 가지가 일치해야 한다.

| 항목 | 설명 |
|---|---|
| **SNMP 버전** | 관리자와 대행자 간 SNMP 버전 일치 여부 |
| **Community String** | 상호 간에 설정한 Community String 일치 여부 |
| **PDU** (Protocol Data Unit) | 통신하기 위한 메시지 유형 일치 여부 |

### 7-3. SNMP 데이터 수집 방식

| 방식 | 동작 | 사용 포트 | 비고 |
|---|---|---|---|
| **Polling 방식** | Manager 가 Agent 에게 정보를 요청하면 Agent 가 응답 | Agent 가 161/udp 사용 | 주기적으로 상태 변화 정보를 얻을 수 있음 |
| **Event-Reporting 방식** | Agent 가 이벤트 발생 시 이를 Manager 에게 비동기적으로 알림 | Manager 가 162/udp 사용 | Agent 가 Manager 에게 알릴 때 **Trap 메시지** 사용. 예상치 못한 사건을 Polling 보다 빠르게 파악 |

> *원본 표기 보정* : 통합자료 line 3203 의 "Trip 메시지" 는 "Trap 메시지" 의 OCR 결함이다.

### 7-4. PDU 타입

> **PDU (Protocol Data Unit)** : 통신하기 위한 메시지 유형.

| PDU 타입 | 방향 | 설명 |
|---|---|---|
| **Get Request** | Manager → Agent | 원하는 객체의 특정 정보를 요청 |
| **Get Next Request** | Manager → Agent | 이미 요청한 정보의 다음 정보 요청 |
| **Set Request** | Manager → Agent | 특정한 값을 설정하기 위해 사용 |
| **Get Response** | Agent → Manager | 해당 변수값 전송 |
| **Trap** | Agent → Manager | 어떤 정보를 비동기적으로 알리기 위해 사용. 유일하게 비동기적 |
| **Get Bulk Request** | Manager → Agent | (SNMPv2 추가) 요청할 객체의 범위를 지정하여 한 번에 요청 |
| **Inform Request** | Manager ↔ Manager | (SNMPv2 추가) 관리시스템 간의 정보 전달 목적으로 사용 |

### 7-5. SNMP 데이터 모델

SNMP 가 관리하는 정보는 객체(Object) 라고 부르며, 이러한 객체들의 집합·구조·식별 방식이 다음 표준으로 정의된다.

| 모델 | 설명 |
|---|---|
| **MIB** (Management Information Base) | SNMP 관리 에이전트가 관리하는 객체의 집합을 정의. 관리자가 조회하거나 설정할 수 있는 객체들의 데이터베이스. 트리 구조로 구성되어 OID(Object Identifier) 를 사용하여 각 객체를 식별 |
| **SMI** (Structure of Management Information) | MIB 의 구조와 포맷을 정의하는 규격. ASN.1 (Abstract Syntax Notation One) 언어를 사용 |

SMI 가 정의하는 객체의 3 요소는 다음과 같다.

| 요소 | 설명 |
|---|---|
| **name** (OID) | 해당 객체를 식별하기 위한 식별자 |
| **syntax** | 객체의 데이터 유형 |
| **encoding** | 메시지 전송 시 비트 변환 규칙. **BER (Basic Encoding Rules)** 을 사용 |

### 7-6. Community String

> **Community String** : SNMP 데몬과 클라이언트가 데이터를 교환하기 전에 인증을 위해 사용하는 일종의 패스워드.

Community String 의 보안 운영 가이드는 다음과 같다.

| 항목 | 설명 |
|---|---|
| **초기값 변경** | `public` 또는 `private` 가 설정되어 있는 경우 의미 없는 문자열로 변경 — 초기값을 사용하는 것은 패스워드를 사용하지 않는 것보다 더 위험 |
| **제공 모드** | RO (Read Only) 와 RW (Read Write) 모드를 제공하며, RW 모드는 가급적 사용 자제를 권고 |

쓰기 권한이 있으면 주요 설정을 수정할 수 있어 심각한 보안 문제를 유발할 수 있다.

### 7-7. SNMP 버전 비교

| 버전 | 보안 기능 | 한계 | 비고 |
|---|---|---|---|
| **SNMPv1** | 없음. Community String 만 일치하면 모든 정보를 얻을 수 있음 | 클라이언트와 데몬 간 통신이 평문 — 스니핑에 취약 | 1988년 SGMP 발전형 |
| **SNMPv2** | 암호화(DES) 와 해시(MD5) 기능 추가 | 송신처 인증 기능 없음 | 가장 많이 사용되는 SNMPv2c 는 복잡한 보안 기능을 제거한 버전 — 보안상 취약 |
| **SNMPv3** | 데이터 인증·암호 기능·재사용 방지·세분화된 접근통제 등 개선된 보안 서비스 제공 | — | 계정과 암호로 인증하는 보안 기능 추가 |

> *시험 포인트* : "SNMPv2c 는 복잡한 보안 기능을 제거한 버전" 은 빈출 함정. SNMPv2 가 v1 보다 보안이 강화된 것이 아니라, 가장 많이 쓰이는 v2c 는 사실상 v1 수준이다.

### 7-8. SNMPv3 보안 서비스

SNMPv3 는 두 가지 보안 모델을 제공한다 (RFC 3414 USM[^8], RFC 3415 VACM[^9]).

| 보안 모델 | 약자 | 제공 서비스 |
|---|---|---|
| **사용자 기반 보안 모델** | **USM** (User-based Security Model) | 재전송 공격 방지 / 위/변조 방지 / 스니핑(도청) 방지 |
| **뷰 기반 접근 통제 모델** | **VACM** (View-based Access Control Model) | 인가된 사용자의 MIB 접근 통제 기능 제공 |

USM 의 보안 매개변수(`msgSecurityParameter`) 는 다음 보호를 담당한다.

| 매개변수 | 보호 대상 위협 |
|---|---|
| Authoritative 엔진 ID·엔진 부트 횟수·엔진 시각 | 재전송(Replay) 공격 방지 |
| 사용자·인증 매개변수 | 위장 공격 / 메시지 위변조 방지 |
| 암호 매개변수 (DES-CBC 등) | 도청·스니핑 / 정보 노출 방지 |

> *시험 포인트* : "USM = 인증·기밀성·재사용 방지 / VACM = 접근통제" 의 역할 분리. 두 모델은 직교하며 함께 사용된다. SNMPv3 의 암호화는 통합자료 표기상 DES-CBC 가 명시되어 있지만, 외부 표준상으로는 RFC 7860 으로 HMAC-SHA-2 인증 프로토콜이 추가되어 있다.

## 단원 요약

| 영역 | 핵심 키워드 |
|---|---|
| **DNS 동작** | 재귀적 vs 반복적 질의, 53/udp(평문)·53/tcp(존 전송), Transaction ID, hosts 파일 우선순위 |
| **DNS 공격** | 스니핑 기반 Spoofing(1:1) vs Cache Poisoning(1:N), 생일 공격 응용 |
| **DNS 보안** | DNSSEC(무결성·출처 인증, **기밀성 X**), Bind 보안(`recursion no`, `allow-transfer`), DoH·DoT(전송 구간 기밀성) |
| **DHCP** | DORA 4단계(67·68/udp), Starvation Attack, DHCP Snooping·Port Security·Option 82 |
| **SNMP 구조** | Manager(162/udp)·Agent(161/udp), MIB(트리·OID), SMI(ASN.1·BER), Community String |
| **SNMP 버전** | v1(평문) → v2(DES·MD5, 송신처 인증 X) → v2c(보안 제거) → v3(USM·VACM) |

### 시험 함정 정리

1. DNSSEC 은 무결성·출처 인증을 제공하지만 **기밀성은 제공하지 않는다**.
2. **존 전송은 53/tcp** 이며 UDP 가 아니다.
3. **SNMPv2c 는 v1 수준의 보안** 이다 — 버전 번호와 보안 강도가 비례하지 않는다.
4. DNS Cache Poisoning 의 영향 범위는 1:N (DNS 서버에 질의하는 모든 사용자) 이다.
5. hosts 파일은 DNS 질의보다 우선순위가 높다 — Spoofing 대응에서 핵심.
6. DHCP Discover 는 브로드캐스트 (IP `255.255.255.255` / MAC `FF:FF:FF:FF:FF:FF`).
