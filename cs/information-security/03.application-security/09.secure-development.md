## 1. 소프트웨어 개발 보안 개념

### 1-1. Secure SDLC 의 정의

> **Secure SDLC (Secure Software Development Life Cycle)** : 소프트웨어 개발 전 단계 (요구사항 → 설계 → 구현 → 테스트 → 운영) 에 걸쳐 보안 활동을 통합한 개발 라이프사이클.

전통적 SDLC 는 보안을 *개발 후* 또는 *운영 단계* 에서만 고려했지만, Secure SDLC 는 *모든 단계에 보안 활동을 분산* 하여 결함을 가능한 가장 이른 단계에서 발견·제거한다. 이는 *Shift-Left Security* 원칙으로도 표현된다 — 보안을 개발 흐름의 왼쪽(앞단) 으로 옮긴다는 의미.

### 1-2. Traditional SDLC 와 Secure SDLC 의 단계별 비교

| SDLC 단계 | Traditional SDLC | Secure SDLC |
|---|---|---|
| **요구사항 (Requirements)** | 기능 요구사항 정의 | + 보안 요구사항 정의·위협 식별 |
| **설계 (Design)** | 아키텍처·인터페이스 설계 | + **위협 모델링 (Threat Modeling)** · 보안 아키텍처 검토 |
| **구현 (Implementation)** | 코드 작성 | + 시큐어 코딩 가이드 준수 + **SAST** 정적 분석 |
| **테스트 (Testing)** | 기능 테스트·통합 테스트 | + **DAST** 동적 분석 + 침투 테스트 + **SCA** 컴포넌트 점검 |
| **운영 (Operations)** | 배포·유지보수 | + 모니터링·**RASP** + 침해사고 대응 + 보안 패치 |

### 1-3. 위협 모델링 (Threat Modeling)

> **위협 모델링** : 시스템의 자산·위협·취약점을 체계적으로 식별하고, 각 위협에 대한 대응을 설계 단계에서 결정하는 활동.

대표적인 방법론은 다음과 같다.

| 방법론 | 핵심 |
|---|---|
| **STRIDE** (Microsoft) | 6가지 위협 카테고리 — Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege (1과목·5과목 cross-ref) |
| **PASTA** (Process for Attack Simulation and Threat Analysis) | 7단계 위협 분석 프로세스 |
| **DREAD** | 위험도 정량 평가 — Damage, Reproducibility, Exploitability, Affected Users, Discoverability |

### 1-4. NIST SSDF (Secure Software Development Framework)

> **NIST SSDF (SP 800-218)** : 소프트웨어 공급망 보안을 위한 미국 정부 표준 프레임워크. 2022년 발표[^1]. 모든 SDLC 모델에 적용 가능한 *프레임워크 중립* 보안 실천을 정의한다.

NIST SSDF 의 4 개 실천 그룹:

| 그룹 | 풀네임 | 핵심 활동 |
|---|---|---|
| **PO (Prepare the Organization)** | 조직 준비 | 보안 정책·역할·교육 |
| **PS (Protect the Software)** | 소프트웨어 보호 | 코드·릴리스의 무결성 보호 |
| **PW (Produce Well-Secured Software)** | 안전한 소프트웨어 생산 | 위협 모델링·시큐어 코딩·코드 리뷰·테스트 |
| **RV (Respond to Vulnerabilities)** | 취약점 대응 | 취약점 보고 접수·분석·패치 |

### 1-5. OWASP SAMM (Software Assurance Maturity Model)

> **OWASP SAMM 2.0** : 조직의 소프트웨어 보안 활동 성숙도를 측정·개선하기 위한 프레임워크[^2]. 5 개 비즈니스 기능 × 3 단계 성숙도로 구성.

| 5 비즈니스 기능 | 설명 |
|---|---|
| **Governance** | 전략·정책·교육 |
| **Design** | 위협 평가·보안 요구사항·보안 아키텍처 |
| **Implementation** | 안전한 빌드·안전한 배포·결함 관리 |
| **Verification** | 아키텍처 평가·요구사항 기반 테스트·보안 테스트 |
| **Operations** | 인시던트 관리·환경 관리·운영 관리 |

> *시험 포인트* : NIST SSDF (2022) 와 OWASP SAMM (2020) 은 둘 다 *프레임워크 중립* 의 안전 개발 표준이지만, NIST SSDF 는 *실천(Practice) 중심*, OWASP SAMM 은 *성숙도 측정 중심* 이다.

## 2. KISA SW 개발보안 가이드 — 7대 보안약점

> **KISA SW 개발보안 가이드** : 한국인터넷진흥원이 발행하는 *전자정부 SW 개발·운영자를 위한 소프트웨어 개발보안 가이드*. 행정안전부·국가정보원과 공동 운영. 시험 빈출 영역.

KISA 가이드는 47 개 보안 약점을 다음 **7 대 카테고리** 로 분류한다. 이 7개 카테고리의 *이름과 각 카테고리의 대표 사례 2~3 개* 가 시험 출제의 핵심 영역이다.

### 2-1. 카테고리 1: 입력 데이터 검증 및 표현

> **입력 데이터 검증 및 표현** : 프로그램 입력값에 대한 검증 누락·부적절한 검증·잘못된 형식 지정으로 인해 발생하는 보안 약점.

이 카테고리는 KISA 47 항목 중 가장 많은 비중을 차지하며, 05 단원의 *모든 인젝션 공격* 이 여기에 해당한다.

| 대표 약점 | 05 단원 §  | OWASP / CWE |
|---|---|---|
| **SQL Injection** | §2 | A03:2021 / CWE-89 |
| **XSS (Cross-Site Scripting)** | §3 | A03:2021 / CWE-79 |
| **크로스사이트 요청 위조 (CSRF)** | §4 | A01:2021 / CWE-352 |
| **서버사이드 요청 위조 (SSRF)** | §5 | A10:2021 / CWE-918 |
| **OS Command Injection** | §6 | A03:2021 / CWE-78 |
| **위험한 형식 파일 업로드** | §7 | A03:2021 / CWE-434 |
| **Path Traversal** (경로 조작) | §9 | A01:2021 / CWE-22 |
| **XML Injection** (XXE 포함) | §11 | A03:2021 / CWE-611 |
| **HTTP 응답 분할** | §15 | — / CWE-113 |
| **신뢰되지 않은 URL 주소로 자동 접속 연결** | — | — / CWE-601 |
| **부적절한 XML 외부 개체 참조** | §11 | — / CWE-611 |

**핵심 방어 패턴** : 입력값 검증 + 출력 인코딩 + 매개변수화 질의문.

> *시험 포인트* : KISA 가이드의 입력 데이터 검증 및 표현 카테고리에는 *전형적 인젝션 외에도 CSRF·SSRF·파일 업로드*가 포함된다. CSRF 가 "보안 기능" 카테고리로 잘못 분류되는 함정이 자주 나오니 주의.

### 2-2. 카테고리 2: 보안 기능

> **보안 기능** : 인증·인가·권한 관리·암호화·세션 관리 등 보안 기능 자체의 잘못된 구현으로 인해 발생하는 보안 약점.

| 대표 약점 | 05 단원 § | CWE |
|---|---|---|
| **부적절한 인가** | §13, §18 | CWE-285, CWE-639 |
| **중요한 자원에 대한 잘못된 권한 설정** | — | CWE-732 |
| **취약한 암호화 알고리즘 사용** | §21 | CWE-327 |
| **중요정보 평문 저장·전송** | §21 | CWE-311, CWE-319 |
| **하드코딩된 중요정보** | — | CWE-798 |
| **충분하지 않은 키 길이 사용** | — | CWE-326 |
| **취약한 비밀번호 허용** | §19 | CWE-521 |
| **사용자 하드디스크에 저장되는 쿠키를 통한 정보 노출** | §22 | — |

**핵심 방어 패턴** : 검증된 표준 알고리즘 사용 + 키·비밀번호의 분리 관리 + 권한 최소화 원칙.

### 2-3. 카테고리 3: 시간 및 상태

> **시간 및 상태** : 동시·다중 또는 한정된 시간 동안 수행되는 작업에서 시간 흐름·상태 관리의 잘못된 처리로 인해 발생하는 보안 약점.

| 대표 약점 | CWE |
|---|---|
| **경쟁 조건 (Race Condition)** | CWE-362 (1과목 §6 cross-ref) |
| **종료되지 않는 반복문 또는 재귀 함수** | CWE-835 |
| **세션 고정** (Session Fixation) | CWE-384 (05 단원 §14 cross-ref) |

**핵심 방어 패턴** : 임계 영역 동기화 (락·뮤텍스) + 세션 ID 갱신.

### 2-4. 카테고리 4: 에러 처리

> **에러 처리** : 에러를 처리하지 않거나, 불충분하게 처리하거나, 또는 에러 정보에 중요 정보가 포함될 때 발생하는 보안 약점.

| 대표 약점 | 05 단원 § | CWE |
|---|---|---|
| **에러 메시지를 통한 정보 노출** | §16 | CWE-209 |
| **오류 상황 대응 부재** | — | CWE-754 |
| **부적절한 예외 처리** | — | CWE-755 |

**핵심 방어 패턴** : 에러 페이지 통일 + 스택 트레이스 사용자 노출 차단 + 로그에만 상세 정보 기록.

### 2-5. 카테고리 5: 코드 오류

> **코드 오류** : 코드 자체의 잘못된 구현으로 인해 발생하는 보안 약점. 메모리 관리·타입 변환·null 처리 등.

| 대표 약점 | 1과목 § | CWE |
|---|---|---|
| **Null Pointer 역참조** | — | CWE-476 |
| **부적절한 자원 해제** | — | CWE-404 |
| **해제된 자원 사용** (Use-After-Free) | 1과목 §6 | CWE-416 |
| **초기화되지 않은 변수 사용** | — | CWE-457 |

**핵심 방어 패턴** : 정적 분석 도구 (SAST) 활용 + 자원 관리 표준 패턴 (RAII·try-with-resources) + 타입 안전 언어 선택.

### 2-6. 카테고리 6: 캡슐화

> **캡슐화** : 중요한 데이터 또는 기능성을 불충분하게 캡슐화하였을 때 인가되지 않은 사용자에게 데이터 누출이 가능해지는 보안 약점.

| 대표 약점 | CWE |
|---|---|
| **잘못된 세션에 의한 데이터 정보 노출** | CWE-488 |
| **제거되지 않고 남은 디버그 코드** | CWE-489 |
| **시스템 데이터 정보 노출** | CWE-497 |
| **Public 메소드로부터 반환된 Private 배열** | CWE-495 |
| **Private 배열에 Public 데이터 할당** | CWE-496 |

**핵심 방어 패턴** : 접근 제한자 (private·protected) 적극 사용 + 디버그 코드 릴리즈 전 제거 + 깊은 복사로 내부 상태 보호.

### 2-7. 카테고리 7: API 오용

> **API 오용** : 의도된 사용에 반하는 방법으로 API 를 사용하거나, 보안에 취약한 API 를 사용하여 발생할 수 있는 보안 약점.

| 대표 약점 | CWE |
|---|---|
| **DNS Lookup 에 의존한 보안 결정** | CWE-247 (DNS Spoofing 으로 우회 가능 — 01 단원 §4 cross-ref) |
| **취약한 API 사용** | CWE-676 (예: `gets()`, `strcpy()` 등 안전하지 않은 표준 라이브러리) |

**핵심 방어 패턴** : 안전한 API 대체본 사용 (`fgets()`, `strncpy()`) + DNS 의존 보안 판단 회피 + 라이브러리·프레임워크의 보안 권고 준수.

### 2-8. 7대 카테고리 종합 정리

| # | 카테고리 | 핵심 본질 | 05 단원 매핑 |
|---|---|---|---|

| 2 | 보안 기능 | 보안 메커니즘 자체의 결함 | §13·§14·§18·§19·§21·§22 |

| 4 | 에러 처리 | 에러로 인한 정보 노출 | §16 |
| 5 | 코드 오류 | 메모리·타입·자원 결함 | (1과목 §6) |
| 6 | 캡슐화 | 내부 상태 노출 | — |
| 7 | API 오용 | 안전하지 않은 API 사용 | — |

> *시험 포인트* : 7 카테고리의 *이름* 과 *각 카테고리에 속하는 대표 사례* 매칭이 빈출. 특히 "*Race Condition 은 어느 카테고리?*" → **시간 및 상태**, "*하드코딩된 비밀번호는?*" → **보안 기능** 같은 분류 문제.

## 3. OWASP·CWE ↔ KISA 7대 매핑

KISA 7대 카테고리는 OWASP Top 10 (2021) 과 CWE Top 25 (2023) 와 다음과 같이 매핑된다. 시험에서는 *동일한 약점이 어느 표준의 어느 카테고리에 속하는지* 의 cross-매핑이 출제될 수 있다.

### 3-1. OWASP Top 10 (2021) ↔ KISA 7대

| OWASP Top 10 | KISA 7대 (주 매핑) |
|---|---|
| A01:2021 Broken Access Control | 보안 기능 (부적절한 인가) |
| A02:2021 Cryptographic Failures | 보안 기능 (취약한 암호화·평문 전송) |
| A03:2021 Injection | **입력 데이터 검증 및 표현** (전체) |
| A04:2021 Insecure Design | 시큐어 SDLC 설계 단계 — 카테고리 외 |
| A05:2021 Security Misconfiguration | 에러 처리 / 캡슐화 (디버그 코드) |
| A06:2021 Vulnerable and Outdated Components | API 오용 — SCA 도구 영역 |

| A09:2021 Security Logging and Monitoring Failures | 운영 단계 — 카테고리 외 |
| A10:2021 SSRF | 입력 데이터 검증 및 표현 |

## 4. 개발 보안 도구

소프트웨어 개발 단계별로 다양한 자동화 보안 도구가 존재한다. 시험에서는 각 도구의 *동작 시점·분석 대상·장단점* 이 출제된다.

### 4-1. 도구 5종 비교

| 도구 | 풀네임 | 동작 시점 | 분석 대상 | 핵심 가치 |
|---|---|---|---|---|
| **SAST** | Static Application Security Testing | 빌드 시점 (코드) | **소스 코드** (정적) | 코드 결함 조기 발견. False Positive 多 |
| **DAST** | Dynamic Application Security Testing | 실행 시점 (배포 후) | **실행 중인 어플리케이션** (동적) | 실제 동작 기반. 환경 의존 |
| **IAST** | Interactive Application Security Testing | 실행 시점 (테스트 중) | **실행 + 코드 계측** | SAST + DAST 의 장점 결합 |
| **SCA** | Software Composition Analysis | 빌드·배포 시점 | **오픈소스·외부 라이브러리** | 알려진 취약점(CVE) 매칭 |
| **RASP** | Runtime Application Self-Protection | 런타임 (운영) | **실행 중인 어플리케이션** + 자체 보호 | 공격 발생 시 실시간 차단 |

### 4-2. SAST (Static Application Security Testing)

> **SAST** : 소스 코드·바이트코드·바이너리를 *실행 없이* 분석하여 보안 결함을 찾는 기법. *White-box Testing* 의 일종.

| 장점 | 단점 |
|---|---|
| 개발 초기 단계에서 결함 발견 (Shift-Left) | False Positive (오탐) 多 |
| 모든 코드 경로 검사 가능 | 런타임 환경·데이터 흐름 미반영 |
| IDE 통합 가능 | 동적 언어·메타프로그래밍 분석 한계 |

대표 도구: **SonarQube**, Checkmarx, Fortify, Coverity, Semgrep.

### 4-3. DAST (Dynamic Application Security Testing)

> **DAST** : 실행 중인 어플리케이션에 대해 외부에서 공격을 시뮬레이션하여 취약점을 찾는 기법. *Black-box Testing* 의 일종.

| 장점 | 단점 |
|---|---|
| 실제 동작 기반 — False Positive 적음 | 늦은 단계에서 결함 발견 |
| 코드 접근 불필요 | 코드 경로 일부만 검사 |
| 환경 설정·배포 결함도 발견 | 결함의 정확한 위치 식별 어려움 |

대표 도구: **OWASP ZAP**, Burp Suite, Acunetix, Netsparker.

### 4-4. IAST (Interactive Application Security Testing)

> **IAST** : 어플리케이션 내부에 *에이전트* 를 배치하여 실행 중 코드 동작·데이터 흐름을 직접 관찰하면서 보안 결함을 찾는 기법.

SAST 의 *코드 인사이트* 와 DAST 의 *실행 검증* 을 결합한 형태로, 테스트 단계의 자동화 환경에서 활용된다.

대표 도구: **Contrast Security**, Synopsys Seeker, HCL AppScan.

### 4-5. SCA (Software Composition Analysis)

> **SCA** : 오픈소스 라이브러리·외부 컴포넌트를 식별하고 *알려진 취약점 (CVE)* 데이터베이스와 매칭하여 위험을 식별하는 도구.

OWASP Top 10 의 **A06:2021 Vulnerable and Outdated Components** 카테고리의 1차 방어 도구이며, Log4Shell (CVE-2021-44228) 같은 공급망 취약점 대응의 핵심.

대표 도구: **OWASP Dependency-Check**, **Snyk**, WhiteSource (Mend), Black Duck, GitHub Dependabot.

### 4-6. RASP (Runtime Application Self-Protection)

> **RASP** : 운영 중인 어플리케이션 내부에 통합되어 실시간으로 공격을 탐지·차단하는 자체 보호 기술. *방어가 어플리케이션 내부에서 작동* 한다는 점이 WAF 와의 차이.

| 구분 | WAF (Web Application Firewall) | RASP |
|---|---|---|
| 위치 | 어플리케이션 외부 (네트워크 경계) | 어플리케이션 내부 |
| 탐지 정보 | HTTP 트래픽 | 실행 컨텍스트·코드 흐름·데이터 흐름 |
| 오탐 | 다소 많음 (컨텍스트 부족) | 적음 (내부 정보 보유) |
| 보호 범위 | 네트워크 진입점 | 모든 코드 경로 |

대표 도구: Imperva RASP, Contrast Protect, Fortify Application Defender.

### 4-7. 도구 선택 가이드

| 단계 | 도구 | 목적 |
|---|---|---|
| 개발 (코드 작성) | SAST + IDE 플러그인 | 코드 작성 즉시 결함 발견 |
| 빌드 (CI) | SAST + SCA | 빌드 파이프라인 게이트 |
| 테스트 | IAST + DAST | 통합 테스트 환경 검증 |
| 배포 (CD) | SCA + 컨테이너 스캔 | 릴리즈 직전 마지막 검증 |
| 운영 | RASP + WAF + SIEM | 실시간 공격 차단 + 모니터링 |

> *시험 포인트* : *SAST = 정적 (코드) / DAST = 동적 (실행) / IAST = 결합 / SCA = 오픈소스 / RASP = 런타임 자체 방어* — 5 도구의 특성 분리가 빈출.

## 단원 요약

| 영역 | 핵심 키워드 |
|---|---|
| **Secure SDLC** | 요구사항(보안 요구) → 설계(위협 모델링) → 구현(SAST·시큐어 코딩) → 테스트(DAST·SCA) → 운영(RASP) — *Shift-Left* |
| **위협 모델링** | STRIDE (Spoofing/Tampering/Repudiation/Info Disclosure/DoS/Elevation), PASTA, DREAD |
| **표준 프레임워크** | NIST SSDF (PO/PS/PW/RV) / OWASP SAMM (Governance/Design/Implementation/Verification/Operations) |
| **KISA 7대 보안약점** | (1) 입력 데이터 검증 및 표현 / (2) 보안 기능 / (3) 시간 및 상태 / (4) 에러 처리 / (5) 코드 오류 / (6) 캡슐화 / (7) API 오용 |
| **OWASP·CWE 매핑** | A03 ↔ 입력 데이터 검증, A07 ↔ 보안 기능, CWE-79·89·78 ↔ 입력 데이터 검증 |
| **개발 보안 도구 5종** | SAST(정적·코드) / DAST(동적·실행) / IAST(결합) / SCA(오픈소스 CVE) / RASP(런타임 자체 방어) |

### 시험 함정 정리

1. **KISA 7대 카테고리 이름** 을 정확히 외운다 — "입력 데이터 검증 *및 표현*", "시간 *및* 상태", "API *오용*" 등 한 글자 차이 함정.
2. **Race Condition 은 시간 및 상태** 카테고리 (코드 오류가 아님).
3. **Hard-coded Credentials 는 보안 기능** 카테고리 (캡슐화가 아님).
4. **SAST = 정적 / DAST = 동적** — 두 도구의 동작 시점 차이.
5. **SCA 는 오픈소스 라이브러리의 알려진 CVE 매칭** — 새로운 취약점을 발견하는 것이 아님.
6. **RASP 는 어플리케이션 내부 / WAF 는 외부** — 실행 컨텍스트 보유 여부의 차이.
7. **NIST SSDF 와 OWASP SAMM 의 차이** : SSDF 는 *실천 중심*, SAMM 은 *성숙도 측정 중심*.
