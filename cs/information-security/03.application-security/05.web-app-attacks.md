### 1-3. XSS·CSRF·SSRF — 세 약자의 구분

가장 혼동되기 쉬운 세 공격을 *조작된 요청의 발생 주체* 기준으로 비교한다.

| 공격 | 조작된 요청 발생 주체 | 공격 위치 | 핵심 메커니즘 |
|---|---|---|---|
| **XSS** | 희생자의 웹 브라우저 | 클라이언트 | 악성 스크립트가 희생자 측에서 실행 |
| **CSRF** | 희생자 (사용자) | 클라이언트 → 웹서버 | 정상 사용자의 세션을 도용해 조작된 요청 전송 |
| **SSRF** | 웹서버 (어플리케이션) | 웹서버 → 내부 서버 | 웹서버가 내부 서비스로 조작된 요청 전송 |

> *시험 포인트* : "**XSS 는 희생자 PC, CSRF 는 희생자, SSRF 는 웹서버가 요청 발생 주체**" — 한 줄로 외운다.

## 2. SQL Injection 취약점

### 2-1. 정의와 공격 영향

> **SQL Injection** : 웹 어플리케이션에서 입력받아 데이터베이스로 전달하는 정상적인 SQL 질의를 변조·삽입하여 비정상적인 데이터베이스 접근을 시도하는 기법.

**공격으로 발생하는 영향**

- 조작한 입력으로 데이터베이스를 인증 절차 없이 접근 및 자료 무단 유출/변조
- 접근하는 사용자를 피싱 사이트 또는 악성코드 유포 사이트로 유도
- 저장된 프로시저를 통한 OS 명령어 실행
- DB 에 있는 개인정보 획득

**무료 SQL Injection 취약점 스캐너**

| 도구 | 설명 |
|---|---|
| **Nikto** | GNU 기반 오픈소스로 웹서버 및 SQL Injection 에 대한 취약점 점검 (리눅스) |
| **SQLMap** | Blind SQL Injection 을 자동으로 수행하는 도구 (Python 으로 개발) |
| **Absinthe** | GUI 기반의 도구로 Blind SQL Injection 취약점 이용 (자동화 과정) |

### 2-2. 공격 방식에 의한 분류

| 분류 | 설명 |
|---|---|
| **Form SQL Injection** | HTML Form 기반 인증을 담당하는 어플리케이션의 취약점이 있는 경우, 사용자 인증을 위한 질의문의 조건을 임의로 조작하여 인증을 우회하는 기법 |
| **Union SQL Injection** | `UNION SELECT` 질의를 이용하여 질의의 결과를 다른 질의의 결과에 결합하여 공격하는 기법 |
| **Stored Procedure SQL Injection** | 저장 프로시저를 이용하여 데이터베이스에 악의적인 SQL 코드를 삽입하여 공격하는 기법 |
| **Mass SQL Injection** | 한 번의 공격으로 대량의 DB 값이 변조되어 홈페이지에 치명적인 영향을 미치는 공격 기법 |

#### Form SQL Injection 공격 흐름

1. 공격자는 아이디 파라미터로 조건절을 무조건 참으로 만드는 입력값 삽입.
2. PHP 페이지에서 파라미터를 전달받아 질의문을 수행.
3. `' or 1=1` 를 통해 조건절을 항상 참이 되도록 만들어 모든 레코드가 SELECT.
4. 주석 문자 (`#`, `--`) 를 통해 그 이하의 구문을 주석 처리하여 인증 무력화.

대표 공격 페이로드: `' or 1=1#`, `" or "1"="1"#`, `" "a"="a"#`

#### Union SQL Injection 공격 흐름

각각의 SELECT 문의 필드 개수가 같아야 하며 필드 타입이 호환할 수 있어야 한다는 SQL 문법 특성을 이용한다.

1. 컬럼 개수를 파악하기 위해 `ORDER BY` 문을 이용하여 SQL 에러 발생 여부를 체크.
2. ORDER BY 문에 4번째 컬럼을 지정했을 때 SQL 에러가 발생하면 컬럼이 3개임을 판단.
3. 공격자는 아이디 입력 폼에 `' UNION SELECT 'admin','admin','admin'#` 을 삽입.
4. 비밀번호로 admin 을 전달하여 패스워드를 admin 으로 변경.

#### Stored Procedure SQL Injection 공격 흐름

1. 선행하는 파라미터와 값 뒤에 세미콜론(`;`) 입력.
2. 수행시킬 저장 프로시저를 입력.
3. 명령어를 연속적으로 수행.

#### Mass SQL Injection 공격 흐름

1. DB 값 변조 시 악성 스크립트를 삽입 (js, swf, exe 파일 이용).
2. 테이블의 문자형 컬럼 데이터 모두에 악성코드의 유포지 정보가 삽입.
3. 사용자가 변조된 사이트 방문 시 감염되거나 악성 Bot 이 설치.

### 2-3. 공격 유형에 의한 분류

| 분류 | 설명 |
|---|---|
| **Error-Based SQL Injection** | DB 질의에 대한 에러값을 기반으로 한 단계씩 점진적으로 DB 정보를 획득. Union SQL Injection 유형의 특수 케이스 |
| **Blind SQL Injection** | 질의 결과의 참과 거짓을 통해 의도하지 않은 SQL 문을 실행함으로써 데이터베이스를 비정상적으로 공격. Form·Stored Procedure·Mass SQL Injection 유형이 해당 |

> *시험 포인트* : "*공격 방식 분류*(Form·Union·Stored Procedure·Mass)" 와 "*공격 유형 분류*(Error-Based·Blind)" 는 *직교 분류* 다. 같은 공격이 두 분류 모두에 속할 수 있다.

### 2-4. 대응 방안

| 대응 | 설명 |
|---|---|
| 사용자 권한 제한 | 데이터베이스 사용자 권한을 최소화 |
| **선처리 질의문 (Prepared Statement)** | 사전에 컴파일된 질의문을 이용. 이후 입력되는 사용자 입력은 질의문 자체를 변경할 수 없고 별도의 매개변수로 전달 — *가장 강력한 방어* |
| 문자열 길이 제한 | 입력되는 문자열의 길이를 제한하고 숫자일 경우 숫자인지 체크 |
| 확장 프로시저 제거 | MS-SQL 의 경우 `xp_cmdshell`·`xp_startmail`·`xp_sendmail` 등 확장 프로시저 제거 |
| 문자열 필터링 | 사용자의 입력값에 SQL 질의문에 이용되는 특수문자 및 SQL 관련 문자열을 필터링. PHP 5.4 이상에서는 `mysql_real_escape_string()` 함수로 특수문자 치환 |

> *시험 포인트* : SQL Injection 의 가장 강력하고 표준적인 방어는 **선처리 질의문 (Prepared Statement)** 이다. 입력값 필터링은 보조 방어이며 단독으로는 우회 가능하다.

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A03:2021 Injection |
| CWE | CWE-89 |

## 3. XSS (Cross-Site Scripting) 취약점

### 3-1. 정의

> **XSS (Cross-Site Scripting)** : 웹 페이지에 악성 스크립트를 삽입할 수 있는 취약점. 웹 어플리케이션에서 사용자 입력값에 대한 필터링이 제대로 이루어지지 않을 경우 공격자가 입력 가능한 폼에 악의적인 스크립트를 삽입하여 해당 스크립트가 *희생자 측에서 동작* 하도록 하여 악의적인 행위를 수행하는 취약점.

XSS 의 핵심은 *악성 스크립트의 삽입은 공격자가, 동작은 희생자 측에서* 이루어진다는 점이다. 이 분리가 §3-3 의 3가지 분류 (Stored·Reflected·DOM-based) 의 근거가 된다.

### 3-2. 공격 방식에 의한 분류

#### Stored XSS (저장형 XSS)

> **Stored XSS** : 공격자가 취약한 웹서버에 악성 스크립트를 저장해 놓으면 클라이언트가 해당 자료를 요청할 때 해당 악성 스크립트가 삽입된 응답 페이지가 전달되어 클라이언트 측에서 동작하는 방식. 서버 DB 에 저장된 게시글의 응답 페이지에서 발생.

**공격 흐름**

1. 공격자는 악성 스크립트를 포함한 게시물 등록을 요청.
2. 웹서버는 DB 에 악성 스크립트가 포함된 게시물을 저장.
3. 희생자는 악성 스크립트가 포함된 게시물 읽기 요청.
4. 웹서버는 악성 스크립트가 포함된 게시물을 희생자에게 응답.
5. 자바스크립트가 희생자 PC 에서 동작.
6. 공격자 서버로 쿠키 정보를 전달.

#### Reflected XSS (반사형 XSS)

> **Reflected XSS** : 외부에 있는 악성 스크립트가 희생자 액션에 의해 취약한 웹서버로 전달되고, 웹서버의 응답 페이지에 해당 악성 스크립트가 삽입되어 희생자 측에서 동작하는 방식. 메일 링크를 통한 응답 페이지에서 발생.

**공격 흐름**

1. 공격자는 악성 스크립트가 포함된 링크를 희생자에게 메일로 전송.
2. 희생자는 메일 링크를 클릭하여 악성 스크립트가 포함된 링크를 요청.
3. 웹서버는 악성 스크립트를 포함한 응답 페이지를 생성하여 희생자에게 응답.
4. 희생자 PC 에서 응답 페이지에 포함된 악성 스크립트가 동작.

Stored XSS 는 악성 스크립트가 *서버 DB 에 저장* 된 형태인 반면, Reflected XSS 는 악성 스크립트가 *요청 정보를 처리하는 과정에서 응답 페이지에 포함* 되는 방식이다.

#### DOM-based XSS

> **DOM-based XSS** : 희생자의 웹 브라우저에서 응답 페이지에 포함된 자바스크립트가 동작하면서 DOM 객체를 실행할 때, URL 등에 포함된 악성 스크립트가 희생자 측에서 동작하는 방식. DOM 객체로 인한 웹 브라우저에서 발생.

**공격 흐름**

1. 공격자는 악성 스크립트가 포함된 링크를 희생자에게 메일로 전송.
2. 희생자는 메일 링크를 클릭하여 악성 스크립트가 포함된 링크를 요청.
3. 웹서버에서 악성 스크립트가 포함된 페이지로 응답.
4. 웹 브라우저는 페이지에 대한 DOM 객체를 생성.
5. 웹 브라우저에 의해 악성 스크립트가 실행.

> **DOM (Document Object Model)** : 문서 객체 모델. 웹페이지 내에 있는 모든 객체를 조작·관리할 수 있는 계층 구조 형태의 모델.

DOM-based XSS 는 응답 페이지와 관계없이 *웹 브라우저에서 발생* 한다는 차이점이 있다 — 서버 응답을 검사해도 탐지되지 않을 수 있다.

### 3-3. 세 XSS 의 비교

| 구분 | Stored XSS | Reflected XSS | DOM-based XSS |
|---|---|---|---|
| 악성 스크립트 위치 | 서버 DB | URL 파라미터 (응답에 포함) | URL 등 (응답과 무관) |
| 발생 위치 | 서버 응답 페이지 | 서버 응답 페이지 | 클라이언트 브라우저 |
| 영향 범위 | 게시물을 읽는 *모든 사용자* | 링크를 클릭한 *희생자* | 링크를 클릭한 *희생자* |
| 서버 측 탐지 | 가능 | 가능 | 어려움 |

### 3-4. 대응 방안

| 대응 | 설명 |
|---|---|
| 입력값 검증 | 사용자 입력값에 대한 검증은 반드시 *서버 단에서* 수행 |
| 입력값 치환 | HTML 코드로 인식될 수 있는 특수문자를 일반문자(HTML Entity) 로 치환 |
| 태그 화이트리스트 | HTML 태그를 허용해야 하는 경우라면 화이트리스트를 선정하여 허용 |

**HTML 특수 문자와 HTML Entity 매핑**

| 특수 문자 | HTML Entity |
|---|---|
| `<` | `&lt;` |
| `>` | `&gt;` |
| `(` | `&#x28;` |
| `)` | `&#x29;` |
| `&` | `&amp;` |
| `/` | `&#x2F;` |
| `"` | `&quot;` |
| `'` | `&#x27;` |
| `#` | `&#x23;` |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A03:2021 Injection |
| CWE | CWE-79 |

> *시험 포인트* : XSS 의 공격 대상 정보는 보통 *세션 쿠키* (희생자의 인증 정보) — 이 위협의 1차 방어가 04 단원 §5 의 **HttpOnly** 속성이다.

## 4. CSRF (Cross-Site Request Forgery) 취약점

### 4-1. 정의

> **CSRF (Cross-Site Request Forgery)** : 정상적인 경로를 통한 요청과 비정상적인 경로를 통한 요청을 서버가 구분하지 못할 경우, 공격자가 스크립트 구문을 이용하여 정상적인 사용자가 조작된 요청을 전송하도록 하는 취약점.

**핵심 특성**: 조작된 요청 발생 주체는 *클라이언트(희생자)* — 희생자 모르게 공격자의 의도된 행위를 수행하게 한다. 공격당한 사용자의 권한을 공격자가 그대로 사용하게 되므로 사용자의 권한 수준에 따라 그 피해 범위가 달라진다.

### 4-2. 공격 흐름

1. 공격자는 조작된 요청정보를 담고 있는 게시물을 웹서버에 등록 요청.
2. 웹서버는 조작된 게시물을 DB 에 등록.
3. 희생자는 조작된 게시물을 읽기 요청.
4. 웹서버는 *희생자의 권한으로* 조작된 요청 내용을 처리.
5. 희생자도 모르게 공격자가 조작한 비밀번호로 변경.

CSRF 취약점이 발생할 수 있는 입력 가능한 폼에 `<img>` 태그의 `src` 속성을 사용해 조작된 요청을 입력하여 취약점을 판단한다 — 게시글을 읽기만 해도 이미지 태그가 자동 요청을 발생시키기 때문이다.

### 4-3. 대응 방안

| 대응 | 설명 |
|---|---|
| **CSRF 토큰** | 모든 HTTP 요청 내에 *예측할 수 없는 임의의 토큰* 을 추가하여 정상/비정상 요청을 판별 |
| 재인증 | 중요한 기능에 대해서는 사용자 세션 검증과 더불어 재인증 유도 |
| XSS 취약점 제거 | XSS 와 공격 방식이 유사하므로 XSS 에 대한 취약점 제거 (XSS 가 있으면 CSRF 토큰도 우회 가능) |
| **SameSite 쿠키** *(외부 보강)* | 04 단원 §5 의 SameSite 쿠키 속성 적용 — Cross-Site 요청에서 쿠키가 전송되지 않도록 |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 (이전) | A05:2017 Broken Access Control 의 일부 (2021부터는 별도 카테고리 X) |
| CWE | CWE-352 |

---

## 5. SSRF (Server-Side Request Forgery) 취약점

### 5-1. 정의

> **SSRF (Server-Side Request Forgery)** : 외부에 노출된 웹서버에 취약한 어플리케이션이 존재하는 경우, 공격자가 URL 또는 요청 정보를 위조하여 접근통제를 우회하는 방식으로 보호된 내부 서버에 있는 정보를 획득하거나 비정상적인 행위를 유도할 수 있는 취약점.

**핵심 특성**: 조작된 요청 발생 주체는 *웹서버(어플리케이션)* — 외부에서 직접 접근할 수 없는 내부 네트워크의 자원에 웹서버를 경유해 접근하는 공격이다.

### 5-2. CSRF vs SSRF 비교

| 구분 | CSRF | SSRF |
|---|---|---|
| 조작된 요청 발생 주체 | 희생자 (사용자) | 웹서버 (어플리케이션) |
| 공격자의 직접 통신 대상 | 웹서버 (희생자 경유) | 웹서버 |
| 피해 대상 | 정상 사용자 권한의 자원 | 내부 네트워크 자원 |

### 5-3. 공격 흐름

1. 공격자는 웹서버의 내부 요청을 조작할 수 있도록 사용자 요청을 조작.
2. 웹서버는 조작된 내부 서버 서비스 요청을 수행.
3. 내부 서버는 자신의 내부 정보를 웹서버에 응답.
4. 웹서버는 내부 서버의 내부 정보를 공격자에게 응답.

### 5-4. 대응 방안

| 대응 | 설명 |
|---|---|
| 화이트리스트 설정 | 입력값을 다른 시스템의 서비스 호출에 사용하는 경우 화이트리스트 방식으로 필터링 |
| 블랙리스트 선정 | 무작위 입력값을 사용해야 한다면 블랙리스트 방식으로 필터링 |
| 요청 처리 | 동일한 내부 네트워크에 있는 서버 간이라도 인증·접근 권한 등을 확인하여 요청을 처리 |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | **A10:2021 Server-Side Request Forgery (SSRF)** — 단독 카테고리로 신규 등재 |
| CWE | CWE-918 |

---

## 6. OS 명령 실행 (OS Command Injection) 취약점

### 6-1. 정의

> **OS Command Injection** : 적절한 검증 절차를 거치지 않은 사용자 입력값이 운영체제 명령어의 일부 또는 전부로 구성되어 실행되는 경우, 의도하지 않은 시스템 명령어가 실행되는 취약점.

웹 어플리케이션에서 `system()`, `exec()` 와 같은 시스템 명령어를 실행할 수 있는 함수를 제공하며 사용자 입력값에 대한 필터링이 제대로 이루어지지 않을 경우 공격자가 시스템 명령어를 호출할 수 있다. 공격자는 시스템 계정 정보 유출, 백도어 설치, 관리자 권한 탈취 등을 진행할 수 있다.

### 6-2. 공격 흐름

1. 공격자는 파라미터에 운영체제 명령어를 삽입하여 웹서버에 요청.
2. 웹서버는 입력값에 대한 적절한 필터링 미흡으로 운영체제 명령어를 실행.
3. 명령어 실행 결과를 공격자에게 응답.

URL 파라미터에 리눅스/유닉스의 `ls`·`cat`, 윈도우의 `dir`·`config` 등의 명령어를 삽입해 실행 여부로 취약점을 판단한다.

### 6-3. 대응 방안

| 대응 | 설명 |
|---|---|
| 문자열 필터링 | 명령어를 실행할 수 있는 문자열 (`;`, `&&`, `||`, `|`) 을 필터링하거나 치환 |
| 보안장비 운영 | 보안 장비를 통해 룰셋을 적용 |
| 화이트리스트 선정 | 허용할 수 있는 명령어 리스트를 선정하여 해당 명령어만 실행되도록 설정 |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A03:2021 Injection |
| CWE | CWE-78 |

## 7. 파일 업로드 취약점

### 7-1. 정의

> **파일 업로드 취약점** : 업로드 파일에 대해 적절한 제어를 하지 않을 경우 웹쉘 등 허가되지 않은 파일들이 웹서버로 업로드 되거나, 다량의 큰 파일을 연속적으로 전송하여 시스템 부하나 장애를 유발할 수 있는 취약점.

> **웹쉘 (Web Shell)** : 악의적인 목적으로 웹서버 내 임의의 명령을 실행할 수 있는 서버 사이드 스크립트 파일. 다양한 언어로 만들어진다.

### 7-2. 공격 흐름

1. 공격자는 악성 스크립트 파일을 웹서버에 업로드 요청.
2. 웹서버는 업로드 파일에 대한 필터링 미흡으로 업로드 처리.
3. 공격자는 웹쉘을 이용한 명령 실행.

### 7-3. 대응 방안

| 대응 | 설명 |
|---|---|
| 파일 검증 | 파일 타입 및 확장자를 모두 검증 |
| 업로드 크기 제한 | 업로드 크기를 지정하여 업로드 파일의 크기를 제한 |
| 실행 방지/호출 방지 | 업로드 디렉터리에 있는 서버 사이드 스크립트 파일이 실행되지 않도록 설정하거나 직접 URL 호출을 차단 |

**Apache 환경 설정 예**

```
# /etc/httpd/conf/httpd.conf — 업로드 크기 제한
<Directory "/var/www/upload">
    LimitRequestBody 5000000
</Directory>

# 파일 업로드 디렉터리에 .htaccess 생성하여 실행/호출 차단
<FilesMatch "\.(ph|inc|lib)">
    Order allow, deny
    Deny from all
</FilesMatch>
AddType text/html .html .htm .php
```

`AddType` 지시자는 서버 사이드 스크립트 실행을 차단하고, `FilesMatch` 지시자는 서버 사이드 스크립트의 직접 URL 호출을 차단한다.

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A03:2021 Injection (간접) / A04:2021 Insecure Design |
| CWE | CWE-434 |

## 8. 파일 다운로드 취약점

### 8-1. 정의

> **파일 다운로드 취약점** : 파일 다운로드 시 파일의 경로 및 파일명을 파라미터로 받아 처리하는 경우 이를 적절히 필터링하지 않으면, 공격자가 이를 조작하여 허용되지 않은 파일을 다운받을 수 있는 취약점.

시스템 환경설정 파일·소스 코드 파일·데이터베이스 연동 파일 등 *중요 파일* 을 다운받을 수 있다.

### 8-2. 공격 흐름

1. 공격자는 파일 다운로드 대상을 조작한 후 웹서버에 다운로드 요청.
2. 웹서버는 다운로드 파일에 대한 필터링 미흡으로 다운로드 처리.
3. 공격자는 중요 파일 다운로드에 성공.

### 8-3. 대응 방안

| 대응 | 설명 |
|---|---|
| 접근 방지 | 입력하는 파일 경로와 이름에 대해 허용하는 경로 이외의 디렉터리와 파일에 접근할 수 없도록 함 |
| 문자열 필터링 | 파일 경로를 변경할 수 있는 문자열을 필터링 |
| 보안장비 운영 | 보안장비를 운영 중이라면 필터링 룰셋을 적용 |

## 9. 경로 조작 (Path Traversal) 취약점

### 9-1. 정의

> **경로 조작 (Path Traversal / 경로 순회) 취약점** : 파일 또는 디렉터리 접근이 통제되지 않아 웹서버 또는 웹 어플리케이션의 중요한 파일과 데이터에 대한 접근 및 실행이 허용되는 취약점.

웹 어플리케이션의 루트 디렉터리를 벗어나 시스템의 주요 파일과 데이터에 접근할 수 있는 취약점이며, 상대 경로 참조 방식 (`../`) 을 사용하여 경로를 조작한다.

**주요 경로 조작 문자 및 인코딩**

| 문자 | URL 인코딩 |
|---|---|
| `.` (dot) | `%2e` |
| `/` (slash) | `%2f` |
| `\` (back-slash) | `%5c` |

### 9-2. 공격 흐름

1. 공격자는 요청 파라미터를 조작하여 경로 추적 문자열을 삽입 후 웹서버에 요청.
2. 웹서버는 입력값에 대한 필터링 미흡으로 해당 파일 접근 처리.
3. 공격자는 파일 내용 또는 실행 결과를 확인.

### 9-3. 대응 방안

| 대응 | 설명 |
|---|---|
| 웹 루트 디렉터리 설정 | 웹서버의 시스템 루트 디렉터리로 접근하지 못하도록 함 (`chroot`) |
| 문자열 필터링 | 경로를 조작할 수 있는 문자를 필터링 |
| 보안장비 운영 | 보안장비를 운영 중이라면 필터링 룰셋을 적용 |

> **chroot** : Change Root Directory 의 줄임말로, 임시 루트 디렉터리를 설정하는 기능. 시스템 루트 디렉터리인 `/` 가 아닌 임의의 디렉터리를 루트 디렉터리인 것처럼 설정하여 불필요한 루트 디렉터리 접근을 차단한다 (1과목 §6 cross-ref).

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A01:2021 Broken Access Control |
| CWE | CWE-22 |

---

## 10. 파일 삽입 (File Inclusion) 취약점

### 10-1. 정의

> **파일 삽입 취약점** : 공격자가 악성 서버 스크립트를 서버에 전달하여 해당 페이지를 통해 악성코드가 실행되도록 하는 취약점.

> **include 함수** : 지정한 페이지를 현재 페이지에 포함해 실행시켜 주는 함수.

삽입할 악성 서버 스크립트 파일의 위치가 *로컬 서버에 위치하는지 원격지에 위치하는지* 에 따라 두 종류로 분류된다.

| 분류 | 약자 | 설명 |
|---|---|---|
| **LFI (Local File Inclusion)** | 로컬 파일 삽입 | 서버에 FTP 나 자료실로 악성 코드를 업로드시켜 include |
| **RFI (Remote File Inclusion)** | 원격 파일 삽입 | 외부 서버의 악성 코드를 불러와 include |

### 10-2. 대응 방안

| 대응 | 설명 |
|---|---|
| 소스코드 확인 | `include`·`require` 등의 구문/함수가 존재하는지 검증 |
| 원격 파일 접근 제한 | 외부 파일이 URL 형태로 include 되지 않도록 함. PHP 의 경우 `php.ini` 파일에 `allow_url_fopen = Off` 설정 |
| 에러 페이지 출력 제한 | 에러 정보를 제공하지 않도록 함. PHP 의 경우 `display_errors = Off` 설정 |

## 11. XXE (XML External Entity) Injection 취약점

### 11-1. 정의

> **XXE (XML External Entity) Injection** : 취약한 XML 파서가 외부 개체를 참조하는 XML 데이터를 처리할 때, 공격자가 삽입한 공격 구문이 포함된 외부 개체가 동작하여 악의적인 행위를 수행할 수 있는 취약점.

> **XML 파서** : XML 문서의 평문 데이터를 읽어들여 문법 검사 및 프로그램 내에서 접근하고 조작할 수 있도록 XML DOM 객체 등으로 변환해 주는 도구.

> **DTD (Document Type Definition)** : XML 문서에서 XML 객체의 정의를 선언하는 부분. XML 문서의 구조 및 Element·Attribute·Entity 를 정의하는 목적으로 사용. 외부 개체는 `SYSTEM` 키워드를 사용하여 정의한다.

### 11-2. 공격 형태

| 형태 | 설명 |
|---|---|
| **중요 파일 접근** | 웹서버의 중요파일을 참조할 수 있는 foo 외부 개체가 포함된 XML 데이터를 입력 |
| **내부 서비스 요청** | URL 경로에 서비스를 요청한 결과를 참조할 수 있는 foo 외부 개체가 포함된 XML 데이터를 입력 |
| **서비스 거부 공격** | XML 내 하나의 개체가 다른 개체를 반복적으로 참조하도록 구성 (Billion Laughs Attack) |

### 11-3. 대응 방안

| 대응 | 설명 |
|---|---|
| 외부 개체 비활성화 | 사용하는 XML 파서의 외부 개체 참조 기능을 사용하지 않도록 비활성화. PHP 의 경우 `libxml_disable_entity_loader(true)` |
| DTD 필터링 | 외부 개체를 선언하는 DTD 를 필터링 |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A03:2021 Injection (이전 별도 카테고리 A04:2017 XXE 였음) |
| CWE | CWE-611 |

---

## 12. Xpath / XQuery Injection 취약점

> **Xpath/XQuery Injection** : 데이터베이스와 연동된 웹 어플리케이션에서 Xpath 및 XQuery 질의문에 대한 필터링이 제대로 이루어지지 않으면, 공격자가 입력이 가능한 폼에 조작된 질의문을 삽입하여 인증 우회를 통해 인가되지 않은 데이터를 열람할 수 있는 취약점.

| 대응 | 설명 |
|---|---|
| 필터링 | Xpath·XQuery 에 사용되는 외부 입력 데이터에 대하여 특수문자 및 질의 예약어 필터링 |
| **인자화된 질의문** | 스트링을 연결하는 형태로 구성하지 않고 인자(파라미터) 화된 질의문 사용 |

XQuery Injection 의 방어 패턴은 SQL Injection 의 *선처리 질의문* 과 본질적으로 동일하다 — *질의문과 데이터의 분리*.

---

## 13. URL/파라미터 변조 취약점

### 13-1. 정의

> **URL/파라미터 변조 취약점** : 실행 경로에 대해서 접근제어를 검사하지 않거나 불완전하게 구현하여 공격자가 URL/파라미터값을 조작하여 중요 정보에 접근 가능해지는 취약점.

사용자 입력값에 대한 검증 누락이 발생하는 모든 상황을 포함하므로 SQL Injection·불충분한 인증 및 인가·XSS 공격에 활용될 수 있다.

### 13-2. 공격 흐름

1. 공격자가 정보수정 페이지를 웹서버에 요청.
2. 공격자는 웹 프록시를 통해 자신의 `id` 파라미터를 `admin` 으로 변조.
3. 웹서버는 공격자에게 관리자의 정보 수정 페이지로 응답.

### 13-3. 대응 방안

| 대응 | 설명 |
|---|---|
| 서버 측 세션 이용 | 파라미터 정보를 이용하는 것이 아닌 서버 측 세션 정보를 이용하여 정보를 해당 세션에 저장 |
| 페이지 재인증 | 중요 정보가 있는 페이지는 재인증을 적용 |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A01:2021 Broken Access Control |
| CWE | CWE-639 (Authorization Bypass Through User-Controlled Key) |

---

## 14. 불충분한 세션 관리 취약점

### 14-1. 정의

> **불충분한 세션 관리 취약점** : 사용자가 로그인할 때 매번 동일한 세션 ID 를 발급하거나 세션 타임아웃을 너무 길게 설정하였을 경우, 공격자가 다른 사용자의 세션을 재사용하여 해당 사용자의 권한을 탈취할 수 있는 취약점.

> **HTTP 세션 하이재킹** : 탈취한 세션 ID 를 이용하여 다른 브라우저에서 계정 세션을 가로채는 방법.

### 14-2. 대응 방안

| 대응 | 설명 |
|---|---|
| 랜덤한 세션 ID | 로그인 시마다 추측할 수 없는 랜덤한 값으로 새롭게 발급 |
| 세션 타임아웃 설정 | 요청이 없으면 자동으로 로그아웃. PHP 의 경우 `php.ini` 의 `session.gc_maxlifetime = 600` 설정 |
| HttpOnly 속성 쿠키 사용 | 스크립트로 쿠키 참조 불가. `session.cookie_httponly = 1` |
| Secure 속성 쿠키 사용 | HTTPS 통신에만 쿠키 전송. `session.cookie_secure = 1` |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A07:2021 Identification and Authentication Failures |
| CWE | CWE-384 (Session Fixation), CWE-613 (Insufficient Session Expiration) |

---

## 15. HTTP 응답 분할 (Response Splitting) 취약점

### 15-1. 정의

> **HTTP 응답 분할 취약점** : 클라이언트의 HTTP 요청에 포함된 요청 파라미터값이 HTTP 응답 헤더에 포함되어 클라이언트에게 다시 전달될 때, 파라미터값에 개행문자인 CR 과 LF 가 포함되어 있으면 HTTP 응답이 2개 이상으로 분리될 수 있는 취약점.

이 취약점의 본질은 04 단원 §2 의 HTTP 메시지 구조 — *헤더 끝을 빈 라인(CRLF + CRLF) 으로 식별* — 에서 비롯한다. 공격자가 입력값에 `%0D%0A%0D%0A` (CRLF + CRLF) 를 삽입하면 응답이 두 개로 분할되고, 두 번째 응답에 악의적 코드를 주입하여 XSS 공격을 수행할 수 있다.

### 15-2. 공격 흐름

1. 희생자는 공격자에 의해 `cookieValue` 파라미터가 조작된 링크 등을 클릭.
2. 요청 메시지를 웹서버에 전달 (개행문자를 이용한 악의적인 두 번째 응답 메시지 포함).
3. 응답 헤더에 `Set-Cookie: nickname` 이 설정.
4. 쿠키에 포함된 `Content-Length: 0` 헤더 이후의 개행문자 연속 2개에 의해 응답 메시지가 2개로 분할되어 처리.

### 15-3. 대응 방안

| 대응 | 설명 |
|---|---|
| 필터링 | 클라이언트 요청 파라미터가 서버 프로그램에서 응답 헤더(`Set-Cookie`, `Location`, 기타) 로 사용될 경우 HTTP 응답 분할이 발생되지 않도록 적절히 필터링 (특히 `%0D`, `%0A` 검사) |

---

## 16. 정보 누출 취약점

### 16-1. 정의

> **정보 누출 취약점** : 민감한 정보가 개발자의 부주의로 인해 노출되는 것. 중요 정보를 주석 구문에 포함해 의도하지 않게 정보가 노출되거나 에러 페이지·에러 메시지를 통해 불필요한 정보가 노출되는 취약점.

| 노출 형태 | 설명 |
|---|---|
| 디폴트 에러 페이지 | 응답 코드와 웹서버 정보가 노출 |
| 에러 메시지 | 시스템·파일 정보 노출 |

### 16-2. 대응 방안

| 대응 | 설명 |
|---|---|
| 에러 페이지 등록 | `/etc/httpd/conf/httpd.conf` 에서 `ErrorDocument` 지시자를 이용하여 각 에러 코드별 에러 페이지 등록 |
| 에러 메시지 제한 | `/etc/php.ini` 에서 `display_errors = Off` 설정 |

```
# httpd.conf
ErrorDocument 403 /home/security/error_page.php
ErrorDocument 404 /home/security/error_page.php
```

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A05:2021 Security Misconfiguration |
| CWE | CWE-209 (Generation of Error Message Containing Sensitive Information) |

## 17. 악성 컨텐츠 취약점

> **악성 컨텐츠 취약점** : 사용자 입력값에 대한 필터링이 제대로 이루어지지 않으면, 공격자가 악성 컨텐츠를 삽입할 수 있으며 악성 컨텐츠가 삽입된 페이지에 접속한 사용자는 악성코드 유포 사이트가 자동으로 호출되어 악성코드에 감염될 수 있는 취약점.

악성 컨텐츠는 SQL Injection·XSS·파일 업로드 취약점을 통해 삽입되므로, 본 취약점의 대응은 *해당 3 취약점의 제거* 와 동일하다.

| 대응 | 설명 |
|---|---|
| SQL Injection 취약점 제거 | §2 참조 |
| XSS 취약점 제거 | §3 참조 |
| 파일 업로드 취약점 제거 | §7 참조 |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A08:2021 Software and Data Integrity Failures |

## 18. 불충분한 인증 및 인가 취약점

> **불충분한 인증 및 인가 취약점** : 개인정보 수정 페이지나 통합 로그인과 같은 곳에서 사용자 인증이 미흡할 경우, 공격자가 파라미터로 전달되는 값을 수정하여 사용자 도용 및 개인정보 노출 문제가 발생할 수 있는 취약점.

> **인증 (Authentication)** : 자신의 신원을 알리고 정말로 신원이 올바른지 증명하는 과정.
> **인가 (Authorization)** : 인증에 성공한 사용자를 대상으로 자원에 대한 접근 또는 사용을 허용할지를 결정하는 과정.

| 대응 | 설명 |
|---|---|
| 아이디 정보 비교 | 로그인한 사용자들의 아이디 정보를 세션에 저장하여 세션에 저장된 아이디 값과 전달된 파라미터 값을 비교 |
| 세션 활용 | 근본적으로 파라미터로 값을 전달하는 것보다는 세션을 통하여 변수를 저장하고 관리 |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A01:2021 Broken Access Control / A07:2021 Identification and Authentication Failures |
| CWE | CWE-287, CWE-269 |

---

## 19. 취약한 패스워드 복구 취약점

> **취약한 패스워드 복구 취약점** : 비밀번호 찾기 기능 또는 관리자에 의한 임시 비밀번호 발급 시 사용자 인증이 미흡하거나 비밀번호를 화면에 즉시 출력하는 경우, 공격자가 불법적으로 다른 사용자의 비밀번호를 획득·변경·복구할 수 있는 취약점.

| 대응 | 설명 |
|---|---|
| 본인인증 메커니즘 구현 | 사용자 본인인증 메커니즘 구현 시 추측이 불가능하게 구현 |
| 난수 값 사용 | 임시 패스워드 발급 시 안전한 난수 값 사용 |

## 20. 자동화 공격 취약점

> **자동화 공격 취약점** : 특정 프로세스에 대한 접근시도 횟수 제한을 설정하지 않으면, 공격자가 자동화 도구 및 봇을 활용하여 1분에 수백 번의 접근을 시도할 수 있으며 특정 프로세스를 반복 수행함으로써 자동으로 수많은 프로세스가 진행되어 시스템 성능에 영향을 미칠 수 있는 취약점.

| 대응 | 설명 |
|---|---|
| **CAPTCHA** | 서버에 요청하는 사용자가 실제 사람인지 컴퓨터 프로그램인지 구별할 수 있도록 적용 |
| 접근시도 횟수 제한 | 일정 횟수 이상 로그인 시도가 발생하면 자동화 공격으로 인식하여 차단 |

> **CAPTCHA (Completely Automated Public Turing test to tell Computers and Humans Apart)** : 사람이 쉽게 인식할 수 있는 임의의 문자와 숫자를 그림으로 변환하여 제시하고, 제시된 문자는 왜곡되거나 덧칠되어 기계가 자동으로 읽기 어려운 상태로 만든다.

## 21. 데이터 평문 전송 취약점

> **데이터 평문 전송 취약점** : 민감한 데이터가 평문으로 통신채널을 통해 송수신될 경우 공격자가 스니핑을 통해 다른 사용자의 민감한 데이터를 획득할 수 있는 취약점.

| 대응 | 설명 |
|---|---|
| 송수신 암호화 | 중요 데이터에 대해 송수신 시 암호화를 적용하여 중간에 감청되더라도 그 내용을 알 수 없도록 함 |
| 보안 서버 적용 | 서버와 클라이언트 통신 시 중요 정보가 사용되는 구간에 SSL 등 암호화 통신 적용 (06 단원 §7 cross-ref) |

| OWASP / CWE | 매핑 |
|---|---|
| OWASP Top 10 | A02:2021 Cryptographic Failures |
| CWE | CWE-319 (Cleartext Transmission of Sensitive Information) |

## 22. 쿠키 변조 취약점

> **쿠키 변조 취약점** : 사용자의 인증 방식 중 하나인 쿠키를 공격자가 변조하여 다른 사용자로 전환하거나 권한 상승이 가능한 취약점.

| 대응 | 설명 |
|---|---|
| 세션 방식 이용 | 쿠키 방식 대신 세션 방식을 이용하는 것을 권장 |
| 암호화 적용 | 쿠키를 사용해야 할 경우 암호화를 적용하여 사용 |

## 23. 방어 패턴 종합

위 21 개 취약점은 5 개의 공통 방어 패턴으로 수렴한다. 시험 대비 측면에서도 *어떤 방어 패턴이 어떤 공격에 적용되는지* 의 매핑이 빈출 영역이다.

| 방어 패턴 | 적용 공격 |
|---|---|
| **입력값 검증·필터링·치환** | XSS, SQLi (보조), OS Command, 경로 조작, 파일 업로드, HTTP 응답 분할, XXE, Xpath/XQuery, 악성 컨텐츠 |
| **매개변수화 / 인자화 질의문** | SQL Injection, Xpath/XQuery Injection (가장 강력) |
| **화이트리스트 + 출력 인코딩** | XSS, 파일 업로드 (확장자), SSRF, OS Command |
| **세션·재인증·CSRF 토큰** | URL/파라미터 변조, CSRF, 불충분 인증·인가, 세션 관리 |
| **서버 설정 강화** | 정보 누출, 파일 삽입, 디렉터리 리스팅 (06 단원), 메소드 제한 (06 단원) |

> *시험 포인트* : *입력값 검증* 은 모든 인젝션 공격의 1차 방어이지만, **그것만으로는 충분하지 않다**. SQL Injection 의 표준 방어는 입력값 검증이 아닌 *선처리 질의문* 이며, XSS 의 표준 방어는 *출력 인코딩(HTML Entity 치환)* 이다. *입력 시점이 아닌 출력 시점에 이스케이프* 가 모범 사례.

### 시험 함정 정리

1. **SQL Injection 의 표준 방어는 선처리 질의문** (Prepared Statement) 이지 입력값 필터링이 아니다.
2. **XSS 의 표준 방어는 HTML Entity 치환** (출력 인코딩) 이지 입력값 차단이 아니다.
3. **CSRF 와 SSRF 의 차이는 *조작된 요청 발생 주체*** (희생자 vs 웹서버).
4. **DOM-based XSS** 는 서버 응답과 무관하게 *브라우저에서 발생* — 서버 측 탐지가 어렵다.
5. `allow_url_fopen=Off` 는 **RFI (원격 파일 삽입) 방어** 이지 LFI 방어가 아니다.
6. HTTP 응답 분할의 핵심 수단은 **개행문자 (`%0D`, `%0A`)** — 04 단원 §2 의 빈 라인 식별 메커니즘 악용.
7. 파일 업로드 방어의 핵심은 *확장자 검증* + *업로드 디렉터리 실행 차단* + *직접 URL 호출 차단* 의 **다층 방어**.
