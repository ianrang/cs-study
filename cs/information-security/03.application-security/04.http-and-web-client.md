# 04. HTTP 와 웹 클라이언트 보안

## 0. 들어가며

HTTP (HyperText Transfer Protocol) 는 웹 상에서 클라이언트와 서버 간 통신을 위해 개발된 응용 계층 프로토콜이다. 1996년 HTTP/1.0(RFC 1945) 으로 표준화된 이래 1.1(RFC 9112) → 2(RFC 9113) → 3(RFC 9114) 으로 진화해 왔지만, *상태 정보를 유지하지 않는다* 는 핵심 특성은 그대로다. 이 단원은 HTTP 의 메시지 구조·메소드·상태 코드와, 무상태 프로토콜의 한계를 보완하기 위한 두 클라이언트 상태 유지 기술 (쿠키·세션) 의 동작·보안 속성을 다룬다. 이 단원의 모든 개념은 다음 단원(05) 의 웹 어플리케이션 공격을 이해하는 토대다.

### 학습 목표

1. HTTP 의 무상태(Stateless) 특성이 어떤 보안 위협을 만들고, 쿠키·세션이 그 위협을 어떻게 보완하는지 설명할 수 있다.
2. HTTP 요청·응답 메시지의 4 영역 (Start Line / Header / Empty Line / Message Body) 을 식별하고 각 영역의 보안적 의미를 설명할 수 있다.
3. 주요 메소드 (GET·POST·HEAD·OPTIONS·PUT·DELETE·CONNECT·TRACE) 의 차이를 비교하고, 운영 환경에서 차단해야 할 메소드를 식별할 수 있다.
4. 쿠키 보안 속성 (HttpOnly·Secure·SameSite·Expires) 이 각각 어떤 공격 (XSS·스니핑·CSRF) 을 방어하는지 매핑할 수 있다.
5. 영속 쿠키와 세션 쿠키의 차이, 쿠키와 세션의 차이를 정확히 설명할 수 있다.

## 1. HTTP 개요

> **HTTP (HyperText Transfer Protocol)** : 웹 상에서 클라이언트와 서버 간에 통신을 위해 개발된 프로토콜. 주로 80/tcp 를 사용하며, 클라이언트의 요청과 서버의 응답으로 이루어진다.

> **하이퍼텍스트 (Hypertext)** : 참조 혹은 링크를 통해 한 문서에서 다른 문서로 접근할 수 있는 문서. 대표적인 예가 HTML 이다.

### 1-1. HTTP 버전별 핵심 차이

| 버전 | 특징 |
|---|---|
| **HTTP/1.0** | HTTP 응답 이후에 TCP 연결을 바로 종료하는 구조. 매 요청마다 TCP 연결 설정·종료 비용 발생 |
| **HTTP/1.1** | `Connection: keep-alive` 옵션 추가로 TCP 연결 상태를 웹서버에 따라 일정 시간 지속. 파이프라이닝 지원 (RFC 9112[^1]) |
| **HTTP/2** | 단일 TCP 연결 위 다중 스트림. 헤더 압축(HPACK). 바이너리 프레임 (RFC 9113[^2]) |
| **HTTP/3** | UDP 기반 QUIC 위에서 동작 (RFC 9114[^3]) |

### 1-2. HTTP 의 핵심 특성 — Stateless

> **Stateless (무상태)** : 동일 클라이언트의 현재 요청과 이전 요청을 식별하지 못하는 특성. 현재 연결에 대한 클라이언트의 어떠한 상태 정보도 유지하지 못한다.

이 특성은 HTTP 의 단순성·확장성을 가져다 주었지만, 사용자가 로그인 후 다음 페이지로 이동했을 때 *방금 로그인한 그 사용자가 맞는지* 식별할 수 없다는 한계를 만든다. 이를 보완하기 위해 §4 의 쿠키·세션 기술이 등장했다. 쿠키·세션을 사용하면 *상태 정보의 위치* 가 달라지며, 이 차이가 §5 의 보안 속성과 직결된다.

> *시험 포인트* : "HTTP 는 Stateless" / "쿠키·세션이 상태 정보를 유지" — 이 한 줄이 모든 웹 보안 기술의 출발점이다.

## 2. HTTP 메시지 구조

HTTP 메시지는 요청 메시지와 응답 메시지 모두 다음 4 영역으로 구성된다.

```
[Start Line]
[Header]
[Empty Line (CRLF)]
[Message Body]
```

> *개행 (CRLF)* : Carriage Return + Line Feed = `0x0d0a`. 이 개행 문자가 §05 의 *HTTP 응답 분할 취약점* 의 핵심 수단이다.

### 2-1. 요청 메시지 구조

| 영역 | 설명 |
|---|---|
| **요청라인 (Request Line)** | 한 행으로 구성. 요청 메소드 + 요청 URI + HTTP 버전 |
| **요청헤더 (Request Header)** | 여러 헤더로 구성. 각 헤더 정보는 개행으로 구분 |
| **빈 라인 (Empty Line)** | 헤더의 끝을 의미하는 개행. 헤더 개수가 가변이므로 빈 라인으로 식별 |
| **메시지 바디 (Message Body)** | 서버로 전송하는 데이터를 담는 부분. GET 방식은 메시지 바디가 없음 |

**주요 요청헤더**

| 헤더 | 용도 |
|---|---|
| `Host` | 요청의 대상이 되는 서버의 도메인명/호스트명과 포트 정보 |
| `User-Agent` | 요청 클라이언트의 어플리케이션/OS 정보 |
| `Referer` | 현재 요청 URL 정보를 담고 있는 이전 문서의 URL 정보 |
| `Cookie` | 클라이언트가 서버로 전송하는 쿠키 정보 (§4-1) |

### 2-2. 응답 메시지 구조

| 영역 | 설명 |
|---|---|
| **상태라인 (Status Line)** | 한 행으로 구성. HTTP 버전 + 상태 코드 + 응답 구문 |
| **응답헤더 (Response Header)** | 여러 헤더로 구성. 각 헤더 정보는 개행으로 구분 |
| **빈 라인 (Empty Line)** | 헤더의 끝을 의미하는 개행 |
| **메시지 바디 (Message Body)** | 클라이언트로 전송하는 데이터 (요청정보의 응답 데이터) |

**주요 응답헤더**

| 헤더 | 용도 |
|---|---|
| `Content-Type` | 메시지 바디의 데이터 형식 (MIME 타입) |
| `Content-Length` | 메시지 바디의 전체 크기 |
| `Set-Cookie` | 서버가 클라이언트에 쿠키를 설정 (§4-1) |
| `Location` | 리다이렉션 시 자원의 변경된 URL |

> *시험 포인트* : 헤더의 개수가 가변이기 때문에 *빈 라인 (CRLF + CRLF)* 으로 헤더 끝을 식별한다. 이 구조적 특성을 악용하는 공격이 *HTTP Response Splitting* (05 단원).

## 3. HTTP 메소드와 상태 코드

### 3-1. 주요 요청 메소드

| 메소드 | 설명 | 메시지 바디 | 보안적 의미 |
|---|---|---|---|
| **GET** | 요청 URI 로 지정한 자원을 서버에 요청 | 없음 (쿼리스트링 사용) | URL 에 데이터가 노출되어 *서버 액세스 로그·브라우저 히스토리·Referer* 에 그대로 남음 — 중요 정보 전송 금지 |
| **POST** | 요청 URI 로 지정한 자원에 데이터를 전달하여 처리한 결과를 서버에 요청 | 있음 | 메시지 바디에 데이터를 담으므로 GET 보다 안전. 중요 정보 전송 권장 |
| **HEAD** | GET 과 유사하지만 응답 메시지 바디를 제외한 헤더 부분만 응답 | 없음 | URL/링크 유효성 검증 용도 |
| **OPTIONS** | 서버가 지원하는 메소드 확인 | 없음 | 응답 헤더의 `Allow` 필드로 지원 메소드 노출 — *불필요 시 차단* |
| **CONNECT** | 클라이언트와 서버 간 터널링 용도. 웹서버가 프록시 역할 수행 | — | 일반 운영 환경에서는 차단 |
| **PUT** | 요청 메시지 바디의 데이터를 요청 URI 로 지정한 자원으로 저장 | 있음 | 무인증 PUT 허용 시 임의 파일 업로드 발생 — *반드시 차단* |
| **TRACE** | 클라이언트로부터 수신한 메시지를 서버에서 그대로 반환. 루프백 테스트 용도 | — | XST (Cross-Site Tracing) 공격 발판 — *차단* |
| **DELETE** | 요청 URI 로 지정한 자원을 서버에서 삭제 | — | 무인증 DELETE 허용 시 자원 삭제 발생 — *반드시 차단* |

> *시험 포인트* : 운영 환경에서 일반 사용자가 사용하는 메소드는 GET·POST·HEAD 정도이며, 나머지(OPTIONS·PUT·DELETE·TRACE·CONNECT) 는 명시적으로 차단해야 한다. 이 차단은 06 단원 §3 (`<LimitExcept GET POST>`) 에서 다룬다.

### 3-2. 주요 응답 상태 코드

응답 상태 코드는 5개 카테고리로 분류된다.

| 카테고리 | 의미 | 대표 코드 | 설명 |
|---|---|---|---|
| **1xx** | 정보 | 100 Continue | 클라이언트의 일부 요청을 받았으며 나머지 정보를 계속 요청 |
| **2xx** | 성공 | 200 OK / 201 Created / 202 Accepted | 정상 처리 |
| **3xx** | 재지정 | 301 Moved Permanently / 302 Found / 304 Not Modified | 자원의 위치 변경 또는 캐시 활용 |
| **4xx** | 클라이언트 오류 | 400 Bad Request / 401 Unauthorized / 403 Forbidden / 404 Not Found | 요청 측 문제 |
| **5xx** | 서버 오류 | 500 Internal Server Error | 서버 측 문제 |

**3xx 상세**

| 코드 | 의미 |
|---|---|
| 301 Moved Permanently | 요청 자원이 영구적으로 변경 (Location 헤더로 새 URL 전달) |
| 302 Found | 요청 자원의 위치가 임시적으로 변경 |
| 304 Not Modified | 요청한 자원이 변경되지 않았으므로 클라이언트 로컬 캐시 사용 |

**4xx 상세**

| 코드 | 의미 |
|---|---|
| 400 Bad Request | 요청 메시지 문법 오류 |
| 401 Unauthorized | 요청한 자원에 대한 인가 필요 — *적절한 권한이 없음* |
| 403 Forbidden | 요청 자원에 대한 접근 차단 |
| 404 Not Found | 요청 자원이 존재하지 않음 |

> *시험 포인트* : 401(인증 필요) vs 403(접근 차단) 의 의미 구분이 빈출. 401 은 *로그인하면 접근 가능*, 403 은 *권한 부족이거나 명시적 차단*.

## 4. 상태 유지 기술 — 쿠키와 세션

HTTP 의 Stateless 한계를 보완하기 위한 두 표준 기술. 둘은 *상태 정보를 어디에 저장하는가* 라는 본질적 차이를 가진다.

### 4-1. 쿠키 (Cookie)

> **쿠키 (Cookie)** : 개별 클라이언트 상태정보를 HTTP 요청/응답 헤더에 담아서 전달하는 작은 정보·데이터. *클라이언트 측* 에 저장되는 상태 유지 기술. RFC 6265[^4] 에 정의되어 있다.

**쿠키 동작 흐름**

1. 클라이언트가 HTTP 요청.
2. 서버는 `Set-Cookie` 응답 헤더를 통해 쿠키를 설정하여 응답.
3. 클라이언트는 `Cookie` 요청 헤더를 이용해 지속해서 쿠키를 전달.

**쿠키의 두 종류**

| 종류 | 설명 | 저장 위치 |
|---|---|---|
| **영속 쿠키 (Persistent Cookie)** | 클라이언트에 *파일 형태* 로 지속해서 존재하는 쿠키. `Expires` 속성에 따라 유효 |
| **세션 쿠키 (Session Cookie)** | 클라이언트 *메모리* 상에 세션이 유지되는 동안 존재하는 쿠키. 웹 브라우저 종료 시 소멸 |

쿠키의 보안 취약점은 *상태 정보가 클라이언트에 저장되며 HTTP 헤더로 전송* 된다는 점이다 — 해킹·스니핑·XSS 에 의한 변조·외부 노출에 취약하다. 따라서 중요 정보를 저장할 때는 §4-2 의 세션 사용이 권장되며, 부득이 쿠키를 사용해야 할 경우 §5 의 보안 속성 (HttpOnly·Secure 등) 과 함께 암호화를 적용한다.

### 4-2. 세션 (Session)

> **세션 (Session)** : 개별 클라이언트 상태정보를 *서버에 저장* 하는 기술. 서버는 개별 클라이언트 세션을 식별하기 위해 **세션 ID** 를 부여하고, 세션 ID 는 세션 쿠키를 이용해 클라이언트와 서버 간 주고받는다.

**세션 동작 흐름**

1. 클라이언트가 로그인 요청.
2. 서버는 `Set-Cookie` 응답헤더를 통해 *세션 ID* 를 담은 세션 쿠키를 설정해 응답.
3. 클라이언트는 `Cookie` 요청헤더를 이용해 세션 쿠키 (세션 ID 포함) 를 전달.
4. 서버는 세션 ID 로 자신의 세션 저장소에서 해당 사용자 상태 정보를 조회.

세션은 상태 정보가 *서버에 저장* 되므로 클라이언트 측 변조·노출 위험이 작다. 단, 클라이언트가 보유하는 세션 ID 자체가 탈취되면 (XSS 또는 스니핑) 세션 가로채기 (Session Hijacking) 가 발생한다 — 이 위협은 05 단원 §13 에서 다룬다.

### 4-3. 쿠키 vs 세션 비교

| 구분 | 쿠키 | 세션 |
|---|---|---|
| 상태 정보 저장 위치 | **클라이언트** | **서버** |
| 식별 방식 | 쿠키 데이터 자체 | 세션 ID (쿠키로 전송) |
| 보안 위험 | 변조·노출 | 세션 ID 탈취 시 권한 도용 |
| 서버 부하 | 작음 | 큼 (세션 저장소 필요) |
| 클라이언트 종료 후 유지 | 영속 쿠키만 | 보통 종료 |

> *시험 포인트* : "쿠키 = 클라이언트 / 세션 = 서버" — 한 줄로 외우면 둘의 구분이 명확하다. 세션 ID 는 *세션 쿠키로 전송* 되므로 세션도 쿠키 기술 위에서 동작한다.

## 5. 쿠키 보안 속성

쿠키 보안 속성은 `Set-Cookie` 응답 헤더에 설정하며, 각 속성이 방어하는 공격 시나리오가 명확히 매핑된다.

| 속성 | 설명 | 방어 공격 |
|---|---|---|
| **HttpOnly** | 클라이언트에서 *스크립트* 를 통해 해당 쿠키에 접근하는 것을 차단 | 쿠키 탈취를 위한 **XSS** 공격 |
| **Secure** | 클라이언트에서 *HTTPS 통신일 경우에만* 해당 쿠키를 전송 | 평문 전송 구간에서의 **스니핑** (쿠키 기밀성 보장) |
| **Expires** | 쿠키의 *유효기간* 을 설정 | 유효기간이 길수록 노출 위험 증가 → 최소한으로 설정 |
| **SameSite** *(외부 보강)* | 쿠키가 *Cross-Site 요청* 에 함께 전송될지 정책 | **CSRF** 공격 방어 |
| **Domain** | 쿠키가 유효한 도메인 범위 | 잘못 설정 시 서브도메인으로 누출 |
| **Path** | 쿠키가 유효한 경로 범위 | — |

### 5-1. SameSite 속성 (외부 보강)

`SameSite` 속성은 RFC 6265bis[^5] 에 도입된 비교적 최신 보안 속성으로, CSRF 공격 (05 단원 §3) 의 1차 방어막이 된다.

| 값 | 동작 |
|---|---|
| `SameSite=Strict` | 동일 사이트 요청에만 쿠키 전송. 외부 링크 클릭 시 쿠키 미전송 |
| `SameSite=Lax` | 동일 사이트 요청 + 안전한 메소드(GET) 의 외부 탐색에만 쿠키 전송 (현대 브라우저 기본값) |
| `SameSite=None; Secure` | 모든 Cross-Site 요청에 쿠키 전송. **반드시 Secure 와 함께 설정** |

### 5-2. 쿠키 보안 속성과 공격 매핑

| 속성 적용 | 방어 공격 | 공격이 가능한 조건 |
|---|---|---|
| HttpOnly 미적용 | XSS 통한 쿠키 탈취 | XSS 취약점 + JavaScript 가 `document.cookie` 접근 가능 |
| Secure 미적용 | 평문 구간 스니핑 | HTTP 통신 또는 다운그레이드 공격 |
| SameSite 미적용 또는 None | CSRF | 외부 사이트에서 인증 필요 요청 유도 |
| Expires 너무 김 | 장기 노출 | 1차 노출이 영구 노출로 이어짐 |

> *시험 포인트* : "HttpOnly = XSS / Secure = 스니핑 / SameSite = CSRF" — 이 매핑을 외우면 보안 속성 문제는 거의 풀린다.

## 6. 웹 브라우저 보안 영역

웹 브라우저는 여러 유형의 사이트를 관리하기 위해 다양한 보안 영역을 지원하고, 각 영역에 대한 보안 요구사항을 사용자가 정의할 수 있게 한다. 이 분류는 Internet Explorer 기준 4 영역으로 시험에서 빈출된다.

| 영역 | 설명 |
|---|---|
| **인터넷 (Internet)** | 다른 세 가지 영역에 포함되지 않는 웹 사이트를 위한 영역 (기본값) |
| **로컬 인트라넷 (Local Intranet)** | 사용자의 컴퓨터와 네트워크상에 있는 웹사이트를 위한 영역 |
| **신뢰할 수 있는 사이트 (Trusted Sites)** | 사용자가 안전한 콘텐츠를 갖고 있다고 신뢰하는 페이지를 설정하는 웹 사이트 영역 |
| **제한된 사이트 (Restricted Sites)** | 어떤 이유에서든 사용자가 신뢰할 수 없거나 보안 페이지로 설정할 수 없는 웹사이트 영역 |

각 영역마다 ActiveX·스크립트·다운로드 허용 여부 등을 별도로 설정할 수 있어, 신뢰 사이트는 완화·제한 사이트는 강화의 운영이 가능하다.

## 7. HTTPS 와 HSTS (외부 보강)

웹 클라이언트 측 보안의 최종 계층은 *HTTPS 의 강제* 와 *다운그레이드 방지* 다. 이는 §5 의 Secure 속성을 통신 채널 전체에 확장한 것이다.

### 7-1. HTTPS

> **HTTPS (HTTP over TLS)** : HTTP 통신을 TLS 위에서 암호화한 보안 프로토콜. 443/tcp 포트 사용.

HTTPS 는 HTTP 메시지의 *전송 구간 기밀성·무결성·서버 인증* 을 제공한다. 단, HTTPS 만으로는 사용자가 처음 접속할 때 HTTP 로 시작하면 (예: 주소창에 도메인만 입력) 공격자가 중간에서 HTTPS 로의 업그레이드를 차단하는 SSL Stripping 공격이 가능하다.

### 7-2. HSTS (HTTP Strict Transport Security)

> **HSTS** : 서버가 클라이언트에게 *해당 도메인은 항상 HTTPS 로만 접속하라* 고 강제하는 응답 헤더 메커니즘. RFC 6797[^6].

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

| 디렉티브 | 의미 |
|---|---|
| `max-age=N` | N 초 동안 HTTPS 강제 |
| `includeSubDomains` | 서브도메인에도 적용 |
| `preload` | 브라우저 HSTS Preload List 에 등록 — 첫 접속도 HTTPS 강제 |

HSTS 가 적용된 도메인은 클라이언트가 HTTP 로 접속을 시도해도 브라우저가 자동으로 HTTPS 로 업그레이드하므로 SSL Stripping 공격이 차단된다.

> *시험 포인트* : HTTPS 는 *전송 구간 보안* 을 제공하지만, HSTS 는 *프로토콜 다운그레이드 방지* 를 제공한다. 둘은 보완 관계.

## 단원 요약

| 영역 | 핵심 키워드 |
|---|---|
| **HTTP 특성** | Stateless (무상태), 80/tcp / HTTPS 443/tcp |
| **메시지 4영역** | Start Line + Header + Empty Line(CRLF·CRLF) + Body |
| **메소드** | GET (URL 노출) / POST (바디) / HEAD / OPTIONS · PUT · DELETE · TRACE (운영 시 차단) |
| **상태 코드** | 2xx 성공 / 3xx 재지정 (301·302·304) / 4xx (400·401·403·404) / 5xx 서버 오류 |
| **쿠키** | 클라이언트 저장. 영속(파일) vs 세션(메모리) |
| **세션** | 서버 저장 + 세션 ID (세션 쿠키로 전송) |
| **쿠키 보안 속성** | HttpOnly(XSS) / Secure(스니핑) / SameSite(CSRF) / Expires(최소 유효기간) |
| **웹 브라우저 영역** | 인터넷 / 로컬 인트라넷 / 신뢰 사이트 / 제한 사이트 |
| **전송 보안** | HTTPS (RFC 9110·TLS) / HSTS (RFC 6797) |

### 시험 함정 정리

1. **쿠키 = 클라이언트 저장 / 세션 = 서버 저장.** 세션도 클라이언트가 가진 *세션 ID* 는 세션 쿠키로 전송되므로 세션 ≠ 쿠키 무관계.
2. **GET 은 URL·로그·히스토리·Referer 에 데이터가 남는다** — 중요 정보는 POST 로.
3. HttpOnly 는 *스크립트 접근 차단* (XSS 방어), Secure 는 *HTTPS 에서만 전송* (스니핑 방어). 두 속성은 직교한다.
4. **401 (인증 필요) vs 403 (접근 차단)** — 401 은 로그인하면 풀리지만 403 은 권한 부족 또는 명시적 차단.
5. TRACE 는 *서버에서 그대로 되돌려보내는* 메소드 — XST 공격의 발판이므로 차단.
6. HTTP 메시지의 헤더 끝은 *빈 라인(CRLF + CRLF)* 으로 식별하며, 이것이 05 단원 *HTTP 응답 분할* 의 수단.
