# 08. 전자상거래 보안

## 0. 들어가며

전자상거래는 인터넷 위에서 결제·인증·청구·이행이 모두 이루어지는 *복합 트랜잭션* 이며, 보안 요건이 가장 무거운 응용 영역 중 하나다. 출제기준 2.1 (전자상거래 보안 기술) 은 (1) 전자지불 수단별 보안요소, (2) 전자상거래 보안 프로토콜, (3) 전자상거래 인증기술, (4) 무선플랫폼에서의 전자상거래 보안의 4 영역을 다룬다.

이 단원은 통합자료에 부재한 영역이므로 외부 1차 출처(PCI DSS·EMVCo·W3C·국가법령정보센터·KISA) 를 100% 의존하여 작성한다. 시험 기출 비중을 고려하여 다음 3-Tier 우선순위로 학습 분량을 배분한다.

| Tier | 영역 | 분량 비중 |
|---|---|---|
| **Tier 1** (필수) | §1 전자지불 시스템 분류, §2 SET 프로토콜·이중 서명, §3 SSL/TLS 안전결제, §4 3-D Secure | 60% |
| **Tier 2** (보강) | §5 전자상거래 인증기술, §6 무선플랫폼 보안 | 30% |
| **Tier 3** (참고) | §7 PCI DSS·전자금융·법령, §8 신흥 표준 cross-ref | 10% |

### 출제기준 매핑

| 출제기준 | 본 단원 매핑 |
|---|---|
| 2.1 전자상거래 보안 기술 — 전자지불 수단별 보안요소 | §1 |
| 2.1 전자상거래 보안 기술 — 전자상거래 보안 프로토콜 | §2·§3·§4 |
| 2.1 전자상거래 보안 기술 — 전자상거래 인증기술 | §5 |
| 2.1 전자상거래 보안 기술 — 무선플랫폼에서의 전자상거래 보안 | §6 |

### 학습 목표

1. 전자지불 시스템을 4가지 (전자화폐·신용카드 기반·계좌이체 기반·전자수표) 로 분류하고 각각의 보안 요건을 설명할 수 있다.
2. SET 프로토콜의 5단계와 **이중 서명 (Dual Signature)** 메커니즘을 단계별로 설명할 수 있다.
3. 3-D Secure 1 과 3-D Secure 2 의 차이점과 인증 흐름을 비교할 수 있다.
4. WPKI 와 WTLS 의 무선 플랫폼 보안 특성을 설명할 수 있다.
5. PCI DSS v4 의 6 대 목표·12 대 요구사항 구조를 식별할 수 있다.

## 1. 전자지불 시스템 분류

### 1-1. 전자지불 시스템의 4 분류

전자지불 시스템은 화폐의 *형태* 와 *결제 흐름* 에 따라 다음 4가지로 분류된다.

| 분류 | 설명 | 대표 사례 |
|---|---|---|
| **전자화폐 (Electronic Cash)** | 디지털 형태로 발행된 가치를 가진 화폐. 발행 기관이 가치를 보증 | Mondex, eCash (역사적), 전자상품권 |
| **신용카드 기반** | 신용카드 정보를 인터넷으로 전송하여 결제 | VISA·MasterCard·국내 신용카드, SET, 3-D Secure |
| **계좌이체 기반** | 사용자 은행 계좌에서 직접 이체 | 인터넷뱅킹, 전자수표, 가상계좌 |
| **전자지갑 (E-Wallet)** | 결제 수단을 모바일·웹에 통합 저장하여 간편 결제 | 카카오페이·네이버페이·Apple Pay·Google Pay |

### 1-2. 전자화폐의 세부 분류

전자화폐는 다음 두 축으로 더 세분화된다.

| 분류 축 | 종류 |
|---|---|
| 결제 시점 | **온라인 (Online)**: 결제 시 발행 기관과 실시간 통신 / **오프라인 (Offline)**: 발행 기관 통신 없이 결제 |
| 사용자 식별 | **추적 가능 (Identified)**: 사용자 신원이 결제 정보에 포함 / **추적 불가능 (Anonymous)**: 사용자 신원 미포함 |

| 결제 시점 × 추적 | 특성 |
|---|---|
| 온라인 + 추적 가능 | 가장 일반적. 신원 확인·이중 사용 방지 모두 가능 |
| 온라인 + 추적 불가능 | 익명성 보장. 발행 기관이 이중 사용 방지 |
| 오프라인 + 추적 가능 | 통신 부담 없음. 사후 정산 |
| 오프라인 + 추적 불가능 | 가장 어려운 구현. 위조·이중 사용 방지가 핵심 과제 |

### 1-3. 전자상거래 4 가지 보안 요구사항

전자상거래의 보안 목표는 다음 4가지로 정리된다 — 4 과목 §정보보호 목표 (CIA + 부인방지) 의 전자상거래 적용본이다.

| 요구사항 | 설명 | 적용 기술 |
|---|---|---|
| **기밀성 (Confidentiality)** | 결제 정보·개인정보가 송신·저장 중 노출되지 않음 | SSL/TLS, 카드번호 암호화 |
| **무결성 (Integrity)** | 거래 데이터가 위변조되지 않음 | 해시, MAC, 전자서명 |
| **인증 (Authentication)** | 송수신자가 자신이 주장하는 자임을 증명 | 전자서명, 인증서 |
| **부인방지 (Non-repudiation)** | 거래 당사자가 거래 사실을 부인할 수 없음 | 전자서명, 타임스탬프 |

> *시험 포인트* : 일반 정보보호 3대 목표(CIA) 에 *부인방지* 가 추가된 4 요건이 전자상거래의 표준 보안 목표다.

## 2. SET (Secure Electronic Transaction) 프로토콜

> **SET (Secure Electronic Transaction)** : 1996 년 VISA 와 MasterCard 가 공동 개발한 신용카드 기반 전자상거래 보안 프로토콜. 인터넷에서의 신용카드 결제를 위해 설계되었으며, 카드 소지자·상점·결제 게이트웨이 간의 인증서 기반 결제 프로토콜이다.

SET 은 SSL 의 보급과 인증서 발급의 복잡성으로 인해 실제 시장에서는 사장되었으나, 정보보안기사 시험에서는 **이중 서명 (Dual Signature) 메커니즘** 이 빈출 영역으로 그대로 출제된다.

### 2-1. SET 의 참여자

| 참여자 | 역할 |
|---|---|
| **카드 소지자 (Cardholder)** | 신용카드를 소유한 구매자 |
| **상점 (Merchant)** | 인터넷 상점 운영자. 카드 결제를 수락 |
| **발급사 (Issuer)** | 카드 소지자에게 신용카드를 발급한 금융 기관 |
| **매입사 (Acquirer)** | 상점과 계약된 카드 결제 처리 금융 기관 |
| **결제 게이트웨이 (PG, Payment Gateway)** | 매입사를 대리하여 카드 결제 처리. 인터넷 ↔ 금융망 연계 |
| **인증기관 (CA)** | 모든 참여자의 X.509 인증서 발급·관리 |

### 2-2. SET 의 5 단계 결제 흐름

| 단계 | 동작 |
|---|---|
| **1. 카드 소지자 등록** | 카드 소지자가 인증기관(CA) 에 등록하여 인증서 발급 |
| **2. 상점 등록** | 상점이 매입사·CA 에 등록하여 상점용 인증서 발급 |
| **3. 결제 요청 (Purchase Request)** | 카드 소지자가 상점에 주문 + 결제 정보 (이중 서명 적용) 송신 |
| **4. 결제 승인 (Payment Authorization)** | 상점이 결제 게이트웨이를 통해 매입사·발급사로 결제 승인 요청·응답 |
| **5. 결제 캡처 (Payment Capture)** | 상점이 매입사로 실제 자금 이체 청구 |

### 2-3. 이중 서명 (Dual Signature) 메커니즘

> **이중 서명 (Dual Signature)** : SET 의 핵심 보안 메커니즘. 하나의 디지털 서명으로 두 개의 메시지(주문 정보·결제 정보) 를 동시에 서명하면서, *상점은 결제 정보를 보지 못하고 결제 게이트웨이는 주문 정보를 보지 못하도록* 하는 분리 서명 기법.

이중 서명의 동기는 다음과 같다:

- **상점은 결제 정보(카드번호·CVV) 를 알 필요가 없다** — 상점은 주문 정보(상품·금액) 만 처리.
- **결제 게이트웨이는 주문 정보(상품 상세) 를 알 필요가 없다** — 결제 게이트웨이는 결제 정보(카드번호·금액) 만 처리.
- 그러나 두 정보가 *연결된 동일한 거래* 임은 양측이 모두 검증할 수 있어야 한다.

**이중 서명 생성 단계 (카드 소지자)**

```
1. 주문 정보 (OI, Order Information) 와 결제 정보 (PI, Payment Information) 를 각각 해시:
   H(OI), H(PI)

2. 두 해시값을 연결(concatenate) 한 후 다시 해시:
   H( H(OI) || H(PI) )

3. 카드 소지자의 개인키로 위 결합 해시값을 서명 → 이중 서명 (DS):
   DS = E(PrivKey_Cardholder, H( H(OI) || H(PI) ))
```

**이중 서명 검증 단계**

| 검증자 | 받는 정보 | 검증 |
|---|---|---|
| **상점** | 주문 정보 OI + 결제 정보 해시 H(PI) + 이중 서명 DS + 카드 소지자 공개키 | `H( H(OI) || H(PI) )` 가 카드 소지자 공개키로 복호화한 DS 와 일치하는지 검증. **결제 정보 PI 자체는 못 봄** |
| **결제 게이트웨이** | 주문 정보 해시 H(OI) + 결제 정보 PI + 이중 서명 DS + 카드 소지자 공개키 | `H( H(OI) || H(PI) )` 가 카드 소지자 공개키로 복호화한 DS 와 일치하는지 검증. **주문 정보 OI 자체는 못 봄** |

이로써 두 검증자는 *자신이 받은 정보만 보면서도* 두 정보가 동일 거래의 일부임을 검증할 수 있다.

> *시험 포인트* : SET 이중 서명의 핵심은 "**상점은 결제 정보를 모르고, 결제 게이트웨이는 주문 정보를 모름**" 이다. 이 분리는 *기밀성* + *책임 분리* 를 동시에 달성한다.

### 2-4. SET 의 보안 서비스

| 서비스 | 제공 메커니즘 |
|---|---|
| 기밀성 | DES + RSA 하이브리드 암호화 |
| 무결성 | 해시 (SHA-1) + 디지털 서명 |
| 인증 | X.509 인증서 (모든 참여자) |
| 부인방지 | 디지털 서명 |

> *시험 포인트* : SET 은 *대칭키 + 비대칭키 하이브리드* 암호화를 사용한다. 대칭키 알고리즘은 SET 1.0 기준 DES 였지만, 이는 4과목 §대칭키 의 시험 출제와 관련된다.

## 3. SSL/TLS 기반 안전 결제

### 3-1. SSL/TLS 와 전자상거래

SSL/TLS 는 SET 의 복잡성을 대체하여 *현재 전자상거래의 사실상 표준 보안 프로토콜* 이 되었다. 카드 정보는 SSL/TLS 위에서 결제 게이트웨이로 전송되며, 상점은 카드 정보를 받지 않는 *Hosted Payment Page* 방식이 일반적이다 (PCI DSS 부담 경감).

| 항목 | SET | SSL/TLS 기반 결제 |
|---|---|---|
| 카드 소지자 인증 | 인증서 필수 | 인증서 선택 (3-D Secure 와 결합) |
| 상점 인증 | 인증서 필수 | 인증서 (서버 인증서) |
| 인증서 발급 부담 | 매우 높음 | 서버 인증서만 |
| 결제 정보 분리 | 이중 서명으로 분리 | Hosted Payment Page 로 분리 |
| 시장 점유 | 사실상 사장 | 표준 |

### 3-2. SSL/TLS 의 결제 흐름

1. 사용자가 상점 사이트에서 결제 버튼 클릭.
2. 상점은 사용자를 결제 게이트웨이의 Hosted Payment Page 로 리다이렉트.
3. 사용자는 결제 게이트웨이 도메인의 HTTPS 페이지에서 카드 정보 입력.
4. 결제 게이트웨이는 카드 정보로 매입사·발급사에 승인 요청.
5. 승인 결과를 상점으로 콜백 — *상점은 카드 정보를 보지 못함*.

이 흐름에서 SSL/TLS 가 제공하는 보안 서비스는 다음과 같다 — 자세한 내용은 2과목 §7 SSL/TLS cross-ref 와 4과목 §TLS Handshake 참조.

| 서비스 | 메커니즘 |
|---|---|
| 기밀성 | TLS 1.2 / 1.3 의 대칭키 암호화 (AES-GCM 등) |
| 무결성 | HMAC 또는 AEAD |
| 서버 인증 | X.509 인증서 |
| 클라이언트 인증 | (선택) X.509 클라이언트 인증서 |

## 4. 3-D Secure (3DS)

> **3-D Secure (3DS)** : 카드 결제의 *카드 소지자 인증* 단계를 추가하여 카드 정보 도용 결제를 차단하기 위한 표준. EMVCo 가 관리한다[^1]. 이름의 *3 Domain* 은 (1) 발급사 도메인 (Issuer Domain), (2) 매입사 도메인 (Acquirer Domain), (3) 상호운용 도메인 (Interoperability Domain) 을 의미한다.

### 4-1. 3DS 1.0 vs 3DS 2

| 구분 | 3DS 1.0 (2001) | 3DS 2 (2017) |
|---|---|---|
| 인증 방식 | 카드 소지자가 별도 비밀번호 입력 (Verified by Visa, MasterCard SecureCode) | 위험 기반 인증 (Risk-Based Authentication) — 거래 정보·기기·행동 분석 후 필요 시에만 추가 인증 |
| 사용자 경험 | 매 거래마다 비밀번호 입력 (구매 포기율 증가) | 저위험 거래는 *마찰 없는 인증 (Frictionless Authentication)* — 사용자 입력 불필요 |
| 추가 인증 방식 | 비밀번호 위주 | OTP·생체인증·앱 푸시·생체 등 다양 |
| 모바일·앱 지원 | 약함 | 강함 |
| EMVCo 표준화 | 부분 | 완전 (3DS Protocol Specification 2.x) |

3DS 1.0 은 EMVCo 에 의해 2022년 deprecated 되었으며, 현재는 3DS 2 가 표준이다.

### 4-2. 3DS 2 인증 흐름

| 단계 | 동작 |
|---|---|
| **1. 거래 시작** | 사용자가 상점에서 결제 시도 |
| **2. 정보 수집** | 3DS Server 가 거래 정보·기기 정보·사용자 행동 정보 수집 |
| **3. 위험 평가** | 발급사 ACS (Access Control Server) 가 위험도 평가 |
| **4-A. 마찰 없는 인증** | 저위험 거래는 추가 인증 없이 승인 — 사용자 경험 매끄러움 |
| **4-B. 도전 인증 (Challenge)** | 고위험 거래는 OTP·생체·앱 푸시 등 추가 인증 요구 |
| **5. 결제 승인** | 인증 성공 시 결제 게이트웨이가 발급사에 승인 요청 |

### 4-3. 책임 전가 (Liability Shift)

3DS 의 부수적 가치는 *사기 책임의 전가* 다. 3DS 인증을 거친 거래는 사기로 판명되어도 상점이 책임지지 않고 *발급사가 책임* 진다 — 이는 상점이 3DS 도입의 동기가 된다.

> *시험 포인트* : 3DS 의 *3 Domain* 의 의미와 3DS 2 의 *위험 기반 인증* 이 빈출 키워드.

## 5. 전자상거래 인증기술

전자상거래 인증은 카드 결제 외 영역(로그인·전자서명·금융 거래) 에서 사용되는 다양한 인증 기술을 포함한다. 4과목 (인증) 와 중복되는 영역이므로 본 단원에서는 *전자상거래 적용 측면* 만 정리하고 상세는 4과목 cross-ref.

### 5-1. OTP (One-Time Password)

> **OTP** : 매번 다른 일회용 비밀번호를 사용하는 인증 방식. 고정된 비밀번호의 노출·재사용 위협을 차단한다.

| 종류 | 동작 |
|---|---|
| **시간 동기화 OTP (TOTP, RFC 6238)** | 현재 시각을 기반으로 OTP 생성. 30초마다 갱신 |
| **이벤트 동기화 OTP (HOTP, RFC 4226)** | 카운터 값을 기반으로 OTP 생성. 사용 시마다 카운터 증가 |
| **Challenge-Response OTP** | 서버의 Challenge 에 대한 Response 로 OTP 생성 |

한국 인터넷뱅킹·증권 거래에서는 보안카드(고정 비밀번호 카드) 와 OTP 토큰이 사용된다.

### 5-2. PKI 기반 전자서명

> **전자서명 (Digital Signature)** : 발신자의 개인키로 서명, 수신자가 발신자의 공개키로 검증. 메시지 무결성·인증·부인방지 제공. (4과목 §전자서명 cross-ref)

한국에서는 *공동인증서* (구 공인인증서) 가 한국 전자상거래 인증의 표준으로 광범위하게 사용되었으며, **2020.6 전부개정·2020.12.10 시행** 의 전자서명법 개정으로 *공인전자서명 제도 (공인인증서 우월적 지위) 가 폐지* 된 이후로는 사설 인증서·간편 인증·생체 인증 등이 다양하게 사용된다.

### 5-3. FIDO2 / WebAuthn (외부 보강)

> **FIDO2 (Fast IDentity Online 2)** : FIDO Alliance 와 W3C 가 표준화한 *비밀번호 없는* 인증 표준. 생체 인증(지문·얼굴) + 보안키(YubiKey 등) 가 핵심 메커니즘.

| 표준 | 정의처 | 역할 |
|---|---|---|
| **WebAuthn (Web Authentication)** | W3C[^2] | 브라우저 ↔ 인증기 (Authenticator) 간 표준 API |
| **CTAP2 (Client to Authenticator Protocol)** | FIDO Alliance | 클라이언트 ↔ 외부 인증기 간 통신 프로토콜 |

FIDO2 는 *공유 비밀(비밀번호) 자체를 제거* 한다는 점에서 PKI 기반 전자서명의 사용자 경험 단점을 해결한다.

### 5-4. OAuth 2.0 / OpenID Connect (외부 보강)

> **OAuth 2.0 (RFC 6749[^3])** : 사용자가 자신의 자격증명을 노출하지 않고 *제3자 어플리케이션이 사용자의 자원에 접근* 할 수 있도록 권한을 위임하는 *인가 (Authorization)* 프레임워크.

> **OpenID Connect (OIDC)** : OAuth 2.0 위에 구축된 *인증 (Authentication)* 계층. ID Token (JWT) 으로 사용자 신원 정보 전달.

| 구분 | OAuth 2.0 | OpenID Connect |
|---|---|---|
| 목적 | 인가 (Authorization) | 인증 (Authentication) |
| 토큰 | Access Token | ID Token (JWT) + Access Token |
| 표준 | IETF RFC 6749 | OIDC Foundation |
| 사용 사례 | 소셜 로그인 권한 위임, API 접근 | 소셜 로그인 (Google·Naver·Kakao 등) |

> *시험 포인트* : OAuth 2.0 = *인가*, OIDC = *인증*. 두 표준은 보완 관계이며 OIDC 가 OAuth 2.0 위에서 동작한다 — 4과목 §인증 cross-ref.

## 6. 무선플랫폼 전자상거래 보안

### 6-1. WAP (Wireless Application Protocol)

> **WAP (Wireless Application Protocol)** : 1990년대 후반 ~ 2000년대 초반 모바일 인터넷 접근을 위해 설계된 프로토콜 스택. 무선 통신의 제한된 대역폭·전력·메모리 환경에 맞춘 경량 프로토콜.

WAP 은 모바일 브라우저의 보급으로 사실상 사장되었으나, 무선 전자상거래 보안의 *역사적 맥락* 으로 시험에 출제된다.

### 6-2. WTLS (Wireless Transport Layer Security)

> **WTLS** : WAP 환경에서 SSL/TLS 의 역할을 하는 무선 전송 계층 보안 프로토콜. SSL 3.0 기반으로 무선 환경에 최적화됨.

WTLS 의 한계는 *WAP 게이트웨이에서 평문화* 된다는 점이다 — 무선 구간 (단말 ↔ 게이트웨이) 은 WTLS, 유선 구간 (게이트웨이 ↔ 서버) 은 SSL/TLS 로 *분할 보안* 이 적용되어, 게이트웨이에서 일시적으로 평문이 노출된다 (*WAP Gap*).

### 6-3. WPKI (Wireless PKI)

> **WPKI (Wireless Public Key Infrastructure)** : 무선 환경의 PKI. 일반 PKI 와 호환되지만 무선 단말의 제약(저장 공간·연산 능력) 을 고려한 경량화 버전.

| 일반 PKI | WPKI |
|---|---|
| X.509 v3 인증서 (수 KB) | 압축된 인증서 형식 |
| RSA 2048·3072 bit | ECC 우선 (짧은 키 길이로 동등한 안전성) |
| 고정 회선 통신 | 무선 통신 (간헐적) |

> *시험 포인트* : WPKI 는 *짧은 키로 동등한 안전성을 제공하는 ECC 기반 PKI* 다. 무선 환경의 제약 대응이 핵심 동기.

### 6-4. 모바일 결제 표준 (외부 보강)

| 표준 | 설명 |
|---|---|
| **EMV** | Europay·MasterCard·Visa 가 공동 개발한 IC 카드 결제 표준. 칩 카드 인증·결제 표준 |
| **NFC (Near Field Communication)** | 13.56 MHz 근거리 무선 통신. Apple Pay·Google Pay 의 비접촉 결제 기반 |
| **Apple Pay / Google Pay** | NFC + 토큰화 (Tokenization) 로 실제 카드번호 대신 일회성 토큰 사용 |

토큰화 (Tokenization) 는 카드번호를 *토큰* 으로 대체하여 토큰이 노출되어도 실제 카드번호는 보호되는 메커니즘이다.

## 7. PCI DSS 와 한국 법령

### 7-1. PCI DSS (Payment Card Industry Data Security Standard)

> **PCI DSS** : VISA·MasterCard·American Express·Discover·JCB 의 5개 카드 브랜드가 공동 운영하는 PCI Security Standards Council 이 발행하는 카드 결제 데이터 보안 표준[^4].

작성 시점 기준 최신 버전은 **PCI DSS v4.0.1** (2024.6 발표). 카드 데이터를 처리·저장·전송하는 모든 사업자에게 적용된다.

**PCI DSS 의 6 대 목표 + 12 대 요구사항**

| 목표 | 요구사항 (12개) |
|---|---|
| **1. 안전한 네트워크 구축·유지** | 1) 방화벽·라우터 보안 설정 / 2) 기본 패스워드·기본 보안 설정 변경 |
| **2. 카드 소지자 데이터 보호** | 3) 저장된 카드 소지자 데이터 보호 / 4) 공개 네트워크 전송 시 암호화 |
| **3. 취약점 관리 프로그램 유지** | 5) 안티-맬웨어 / 6) 안전한 시스템·어플리케이션 개발·유지 |
| **4. 강력한 접근 통제 조치** | 7) 알아야 할 필요(Need-to-know) 에 따른 접근 제한 / 8) 컴퓨터 접근에 대한 식별·인증 / 9) 카드 소지자 데이터에 대한 물리적 접근 제한 |
| **5. 네트워크 모니터링·테스트** | 10) 네트워크 자원·카드 소지자 데이터 접근 추적·모니터링 / 11) 보안 시스템·프로세스 정기 테스트 |
| **6. 정보보안 정책 유지** | 12) 모든 인력에 대한 정보보안 정책 유지 |

### 7-2. 한국 전자금융감독규정·전자서명법

| 법령·규정 | 핵심 내용 |
|---|---|
| **전자서명법** | 전자서명의 법적 효력 보장. **2020.6 전부개정 (법률 제17354호) · 2020.12.10 시행** 으로 *공인인증서 우월적 지위 폐지* — 다양한 사설 인증서가 동등하게 사용 가능 |
| **전자금융거래법** | 전자금융 거래의 안전성·신뢰성 확보 |
| **전자금융감독규정 (금융위원회 고시)** | 금융회사·전자금융업자의 정보기술 부문·정보보호 등을 위한 인력·조직·예산 등 인적·물적 보안 사항 규정 |
| **개인정보보호법** | 카드번호·계좌번호 등 고유식별정보의 암호화 의무 (06 단원 §7-2 cross-ref) |

> *시험 포인트* : 2020.6 전부개정·2020.12 시행 전자서명법 개정으로 *공인인증서가 폐지된 것이 아니라, 공인인증서의 우월적 지위가 폐지* 된 것이다. 사설 인증서가 동등 효력을 가지게 되었다.

## 단원 요약

| Tier | 영역 | 핵심 키워드 |
|---|---|---|
| **Tier 1** | 전자지불 분류 | 전자화폐(온라인·오프라인 / 추적 가능·불가능), 신용카드 기반, 계좌이체, 전자지갑 |
| **Tier 1** | 전자상거래 4 보안 요건 | 기밀성·무결성·인증·**부인방지** |
| **Tier 1** | SET | 카드 소지자·상점·발급사·매입사·결제 게이트웨이·CA 6 참여자, 5 단계, **이중 서명 (Dual Signature)** |
| **Tier 1** | 이중 서명 | `H( H(OI) || H(PI) )` 를 카드 소지자 개인키로 서명. **상점은 PI 못 봄, PG 는 OI 못 봄** |
| **Tier 1** | SSL/TLS 결제 | Hosted Payment Page, 상점은 카드 정보 미수신 |
| **Tier 1** | 3DS | 3 Domain (Issuer / Acquirer / Interoperability), 3DS 2 = 위험 기반 인증, 책임 전가 |
| **Tier 2** | 인증 기술 | OTP(TOTP·HOTP), PKI 전자서명, FIDO2/WebAuthn, OAuth 2.0/OIDC |
| **Tier 2** | 무선 플랫폼 | WAP, WTLS(SSL 3.0 기반·WAP Gap), WPKI(ECC 기반 경량) |
| **Tier 3** | 표준·법령 | PCI DSS v4 (6 목표 12 요구사항), 전자서명법 2020.6 개정·2020.12 시행 (공인인증서 우월적 지위 폐지) |

### 시험 함정 정리

1. **이중 서명의 핵심** : 상점은 결제 정보(PI) 를 못 보고, 결제 게이트웨이는 주문 정보(OI) 를 못 본다 — 양측이 자신의 정보 + 상대 정보의 해시 + 서명만으로 검증.
2. **전자상거래 보안 요건 = 기밀성 + 무결성 + 인증 + 부인방지** — 3대 목표(CIA) 가 아닌 **4 요건**.
3. **3DS 2 의 핵심 진화** : 매번 비밀번호가 아닌 *위험 기반 인증* 으로 저위험 거래는 마찰 없는 인증.
4. **WPKI 의 핵심** : 일반 PKI 에서 RSA 가 주로 쓰이는 반면 WPKI 는 *ECC* 우선 (짧은 키로 동등한 안전성).
5. **공인인증서 폐지가 아닌 우월적 지위 폐지** (전자서명법 2020.6 개정·2020.12 시행) — 사설 인증서가 동등 효력.
6. **PCI DSS** 는 카드 결제 데이터 보안 *표준* 이며 법령은 아니다 — 카드 브랜드 5사의 자율 규범.
