# 02. FTP 보안

## 0. 들어가며

FTP (File Transfer Protocol) 는 네트워크 상에서 파일을 전송하기 위한 표준 프로토콜이다. 파일 송수신이라는 단순한 목적으로 1971년 RFC 114 로 처음 등장한 이래 오랫동안 사용되어 왔지만, **제어 채널과 데이터 채널이 분리** 되어 있는 구조적 특성과 **평문 전송** 이라는 보안적 한계 때문에 시험에서는 능동/수동 모드 동작·FTP Bounce·TFTP·익명 FTP 의 4가지 공격 패턴이 빈출 영역을 이룬다.

## 1. FTP 개요

> **FTP (File Transfer Protocol)** : TCP/IP 기반의 서버와 클라이언트 사이에 파일 전송을 위한 통신 프로토콜. 파일을 서버와 클라이언트 간에 전송하기 위한 표준 프로토콜이다. RFC 959 (1985)[^1] 에 정의되어 있다.

FTP 가 다른 응용 계층 프로토콜과 구별되는 가장 큰 특성은 **두 개의 별도 TCP 연결을 사용** 한다는 점이다. 한 채널은 로그인·디렉토리 변경·전송 명령 등 *제어 정보* 를 주고받고, 다른 채널은 실제 *파일 데이터* 를 주고받는다. 이 분리 구조가 FTP 의 능동/수동 모드 구분과 FTP Bounce Attack 의 근본 배경이 된다.

| 채널 | 용도 | 일반적 포트 |
|---|---|---|
| **제어 채널 (Control Channel)** | 명령·응답 송수신 | 21/tcp |
| **데이터 채널 (Data Channel)** | 실제 파일 데이터 송수신 | 능동: 20/tcp / 수동: 1024 이상 임의 포트 |

### 1-1. FTP 의 보안적 한계

FTP 서비스는 기본적으로 **평문 송수신** 을 하므로 아이디 및 패스워드가 노출되기 쉬운 서비스이며, 간단한 스니퍼에 의해서도 자격 증명이 쉽게 노출될 수 있다. 따라서 반드시 필요한 경우를 제외하고는 사용하지 않는 것이 권장되며, 사용해야 한다면 §5 의 암호화된 통신 (SFTP·FTPS) 으로 대체한다.

> *시험 포인트* : "FTP 는 평문" / "제어·데이터 채널이 분리" 두 특성을 항상 묶어서 외운다. 이 둘이 §3 모든 공격의 전제다.

## 2. FTP 동작 모드

FTP 의 동작 모드는 **데이터 채널을 누가 어디서 누구에게로 연결하는가** 에 따라 능동 모드(Active Mode) 와 수동 모드(Passive Mode) 로 구분된다.

### 2-1. 능동 모드 (Active Mode)

> **능동 모드 (Active Mode)** : 서버가 클라이언트에게 데이터 연결을 *역방향* 으로 요청하는 방식. 클라이언트가 데이터 채널 포트 번호를 지정한다.

| 채널 | 연결 방향 |
|---|---|
| 제어 채널 | Client 의 1024/tcp 이상 포트 → Server 의 **21/tcp** |
| 데이터 채널 | Server 의 **20/tcp** → Client 의 1024/tcp 이상 포트 |

**동작 단계**

1. FTP 클라이언트는 FTP 서버의 21/tcp 포트로 접속.
2. 제어 채널을 생성 (`PORT` 명령).
3. FTP 클라이언트가 FTP 서버에게 데이터 채널 생성을 위한 Port 를 알려줌.
4. FTP 서버는 FTP 클라이언트가 알려준 IP 와 Port 로 접속.
5. 데이터 채널을 생성.

**문제점**

클라이언트에 방화벽이 설치되어 있어 원격 FTP 서버에서 클라이언트로의 접속을 허용하지 않을 경우 데이터 채널 접속이 불가능하여 데이터를 받을 수 없다. 클라이언트의 입장에서 보면 *외부에서 자신을 향해 들어오는 인바운드 연결* 이 발생하므로 NAT·방화벽 환경에서 빈번히 차단된다.

| 대응 | 설명 |
|---|---|
| 클라이언트 방화벽 설정 | 원격 FTP 서버의 접속을 허용하도록 설정 |
| 수동 모드 사용 | 클라이언트에서 수동 모드로 원격 FTP 서버에 접속 |

### 2-2. 수동 모드 (Passive Mode)

> **수동 모드 (Passive Mode)** : 클라이언트가 서버에 데이터 연결을 *순방향* 으로 요청하는 방식. 서버가 데이터 채널 포트 번호를 지정한다.

| 채널 | 연결 방향 |
|---|---|
| 제어 채널 | Client 의 1024/tcp 이상 포트 → Server 의 **21/tcp** |
| 데이터 채널 | Client 의 1024/tcp 이상 포트 → Server 의 **1024/tcp 이상 포트** |

**동작 단계**

1. FTP 클라이언트는 FTP 서버의 21/tcp 포트로 접속.
2. 제어 채널을 생성 (`PASV` 명령).
3. FTP 서버가 FTP 클라이언트에게 데이터 채널 생성을 위한 Port 를 알려줌.
4. FTP 클라이언트는 FTP 서버가 알려준 Port 로 접속.
5. 데이터 채널을 생성.

**문제점**

서버는 데이터 채널 생성을 위해 인바운드 트래픽에 대한 방화벽의 1024/tcp 이상 포트를 모두 허용해야 하는 문제점이 있다. 즉, 능동 모드의 문제(클라이언트 방화벽) 가 수동 모드에서는 *서버 방화벽 측면의 문제* 로 옮겨간다.

| 대응 | 설명 |
|---|---|
| 상태 검사 기능 | 방화벽이 연결 상태를 추적(Stateful Inspection) 하여 21/tcp 만 허용하면 데이터 채널을 자동으로 허용하도록 설정 |

> *상태 검사 기능 (Stateful Inspection)* : 방화벽을 통과하는 모든 트래픽에 대해 프로토콜별 연결 상태를 추적하여 패킷 필터링을 수행하는 방화벽 기능. TCP 는 제어 채널 포트인 21/tcp 만 허용해 주면 데이터 채널을 자동으로 허용해 준다. (2과목 §8 방화벽 cross-ref)

### 2-3. 능동 모드 vs 수동 모드 비교

| 구분 | 능동 모드 (Active) | 수동 모드 (Passive) |
|---|---|---|
| 데이터 채널 연결 주체 | 서버 → 클라이언트 | 클라이언트 → 서버 |
| 서버 데이터 포트 | 20/tcp | 1024 이상 임의 포트 |
| 클라이언트 데이터 포트 | 1024 이상 임의 포트 | 1024 이상 임의 포트 |
| 사용 명령 | `PORT` | `PASV` |
| 방화벽 부담 | 클라이언트 측 | 서버 측 |
| 일반 운영 환경 권장 | NAT 뒤가 아닌 환경 | NAT/방화벽 뒤 클라이언트 |

> *시험 포인트* : "능동 모드 = 서버가 클라이언트로 역방향 접속(20/tcp)" / "수동 모드 = 클라이언트가 서버로 정방향 접속(1024+)" — 이 한 줄이 시험에서 가장 빈출되는 변별 지점이다.

## 3. FTP 공격 유형

FTP 공격은 크게 (1) FTP 의 구조적 분리 채널을 악용하는 **Bounce Attack**, (2) FTP 의 단순화 변형이 가진 인증 부재를 악용하는 **TFTP Attack**, (3) 인증 없이 접속 가능한 **익명 FTP** 의 3가지로 분류된다.

### 3-1. FTP Bounce Attack

> **FTP Bounce Attack** : 공격 대상 네트워크를 스캔하거나 공격자가 원하는 서버로 데이터를 전송하는 형태의 공격. FTP 설계의 구조적 취약점을 이용해 데이터 채널을 생성할 때 임의의 목적지를 설정할 수 있는 점을 악용한다.

FTP 능동 모드에서 클라이언트가 서버에게 데이터 채널 연결 대상으로 알려 주는 IP·Port 정보 (`PORT` 명령의 인자) 를 *서버가 검증하지 않는다는 점* 이 핵심 취약점이다. 공격자는 자신이 아닌 *제3의 호스트* 의 IP·Port 를 PORT 명령으로 지정함으로써 FTP 서버가 제3 호스트로 데이터 채널을 연결하게 만들 수 있다.

**공격 시나리오**

1. 공격자가 익명 FTP 서버에 접속.
2. `PORT` 명령으로 *내부 네트워크의 호스트 IP·Port* 를 지정.
3. 어떤 명령(예: `LIST`) 을 보내면, FTP 서버는 공격자가 아닌 내부 호스트의 해당 Port 로 연결을 시도.
4. 응답에 따라 공격자는 내부 호스트의 해당 Port 가 열려 있는지 닫혀 있는지 판별 (포트 스캔) 하거나, 공격자가 원하는 서버로 데이터를 전송.

이 공격으로 공격자는 FTP 서버를 *경유* 해 외부에서 접근할 수 없는 내부 네트워크를 스캔하거나 공격자가 원하는 내부 서버로 데이터를 전송하는 등의 악의적인 행위를 할 수 있다.

| 대응 방안 | 설명 |
|---|---|
| FTP 원래 규약 인정 + 검증 | FTP 의 원래 규약은 인정하되 다른 서비스가 20번 port 접속을 요청하면 거절 |
| 익명 FTP 비활성화 | 익명 FTP 서버는 Bounce Attack 의 발판이 되므로 비활성화 |
| 최신 FTP 서버 사용 | 현대 FTP 서버는 PORT 명령의 IP 가 제어 채널 클라이언트 IP 와 다르면 거부하는 보호 기능 내장 |

### 3-2. TFTP Attack

> **TFTP (Trivial File Transfer Protocol)** : 읽기 전용 메모리나 디스크가 없는 워크스테이션에 설치될 수 있을 정도로 매우 단순하고 간단한 파일 송수신 프로토콜. 69/udp 포트를 사용한다.

TFTP 는 자체 디스크가 없는 워크스테이션에 부팅 이미지를 전달하거나 네트워크 장비의 운영체제를 백업·복구하거나 설정 정보를 저장·조회하는 용도로 사용된다. **별도의 인증 과정이 없다는 점** 이 가장 큰 보안적 한계다.

> **TFTP Attack** : 보안 기능이 없는 TFTP 서버에 접속하여 파일에 접근하거나 악의적인 파일을 생성하는 공격. 별도의 인증과정이 없어 지정한 디렉터리에 접근할 수 있다.

| 대응 방안 | 설명 |
|---|---|
| 서비스 비활성화 | TFTP 서비스 사용 안 할 시 서비스 파일에 TFTP 서비스를 주석 처리하여 비활성화 |
| Secure Mode 운영 | TFTP 서비스 사용 시 서비스 파일에 TFTP 서비스의 `server_args` 경로에 `-s` 추가 (chroot 기능을 이용해 지정한 디렉터리를 최상위 디렉터리로 지정하여 상위 디렉터리 접근 차단) |

### 3-3. Anonymous FTP Attack

> **Anonymous FTP Attack** : 익명 계정으로 FTP 서버에 접속하여 파일에 접근하거나 악의적인 파일을 생성하는 공격. 익명 계정은 ID 가 `anonymous` 이고 비밀번호가 없는 계정을 의미한다.

익명 FTP 서비스는 익명 계정으로 FTP 접속이 가능한 서비스이며, 비밀번호 인증 없이 파일에 접근할 수 있다는 점이 보안적으로 심각한 취약점이다. 익명 FTP 는 §3-1 의 Bounce Attack 의 발판으로도 자주 활용된다.

| 대응 방안 | 설명 |
|---|---|
| 익명 계정 제한 | `/etc/vsftpd.conf` 파일에 `anonymous_enable=NO` 설정하여 익명 계정 제한 |

### 3-4. 세 공격의 비교

| 구분 | FTP Bounce Attack | TFTP Attack | Anonymous FTP Attack |
|---|---|---|---|
| 대상 프로토콜 | FTP (21/tcp) | TFTP (69/udp) | FTP (21/tcp) |
| 핵심 취약점 | 데이터 채널 목적지 미검증 | 인증 부재 | 익명 계정 허용 |
| 공격 결과 | 내부 네트워크 스캔·데이터 우회 전송 | 파일 무단 접근·생성 | 파일 무단 접근·생성 + Bounce 발판 |

## 4. FTP 접근제어 (보안 기술)

FTP 의 평문 전송 한계로 인해 **중요 계정** (예: root) 의 FTP 접속은 명시적으로 차단해야 한다. 리눅스/유닉스 환경에서 FTP 접근제어는 다음 4가지 메커니즘으로 구현된다.

### 4-1. ftpusers 파일

> **ftpusers** : 접속을 제한할 계정정보를 명시하는 파일. 여기에 명시된 계정은 FTP 접속이 차단된다.

| 항목 | 설명 |
|---|---|
| 차단 대상 | root 등 중요 계정을 ftpusers 파일에 명시 |
| 동작 방식 | 화이트리스트가 아닌 **블랙리스트** 방식 — 명시된 계정은 차단 |

### 4-2. user_list 파일 (vsFTP)

> **user_list** : vsFTP 의 접속을 제한할 계정정보를 명시하는 파일.

vsFTP 는 user_list 의 동작 방식을 설정에 따라 화이트리스트 또는 블랙리스트로 모두 운영할 수 있다.

```
# /etc/vsftpd.conf
userlist_enable=YES   # user_list 파일 사용 활성화
```

### 4-3. TCP Wrapper 연동

IP 기반 접근제어를 위해 TCP Wrapper 를 vsFTP 와 연동할 수 있다. 운영체제 단의 `hosts.allow`·`hosts.deny` 파일을 통한 IP 화이트/블랙리스트가 그대로 vsFTP 에 적용된다 (1과목 §6 TCP Wrapper cross-ref).

```
# /etc/vsftpd.conf
tcp_wrappers=YES      # TCP Wrapper 연동 활성화
```

| 파일 | 역할 |
|---|---|
| `/etc/hosts.allow` | 접속을 허용할 IP/대역 명시 |
| `/etc/hosts.deny` | 접속을 차단할 IP/대역 명시 (`ALL: ALL` 로 기본 차단 후 hosts.allow 로 허용) |

### 4-4. FTP 접근제어 조합 운영

실무 환경에서는 위 메커니즘을 다층으로 조합한다.

| 계층 | 메커니즘 | 통제 대상 |
|---|---|---|
| 1차 | TCP Wrapper / 방화벽 | 출발지 IP |
| 2차 | ftpusers / user_list | 시스템 계정 |
| 3차 | vsFTP의 chroot_local_user / chroot_list_enable | 디렉터리 격리 |

> *시험 포인트* : ftpusers 의 동작 방식 = "명시된 계정은 차단" (블랙리스트). 화이트리스트로 잘못 외우기 쉬운 함정.

## 5. 암호화 FTP (SFTP·FTPS)

FTP 의 평문 한계를 근본적으로 해결하려면 전송 구간 암호화를 적용한 다음 두 표준 중 하나로 대체한다. 둘은 **이름이 비슷하지만 기반 프로토콜이 완전히 다르다**.

### 5-1. SFTP (SSH File Transfer Protocol)

> **SFTP (SSH File Transfer Protocol)** : SSH 프로토콜 위에서 동작하는 파일 전송 프로토콜. 22/tcp 포트를 사용한다.

SFTP 는 SSH-2 의 서브시스템으로 동작하며, SSH 가 제공하는 인증·기밀성·무결성 위에서 파일을 전송한다. **하나의 TCP 연결로 모든 통신** 이 이루어지므로 능동/수동 모드 구분이 없고 방화벽 친화적이다 (1과목 §6 SSH cross-ref).

### 5-2. FTPS (FTP over SSL/TLS)

> **FTPS (FTP over SSL/TLS)** : FTP 통신 구간에 SSL/TLS 보안을 적용한 프로토콜. RFC 4217 (Securing FTP with TLS, 2005)[^2] 에 표준화되어 있다.

FTPS 는 기존 FTP 의 두 채널 구조를 그대로 유지하면서 각 채널에 TLS 를 씌운 형태이며, 두 가지 모드가 있다.

| 모드 | 설명 | 포트 |
|---|---|---|
| **Implicit FTPS** | 연결 시작부터 TLS 협상 — 별도 포트 사용 | 990/tcp |
| **Explicit FTPS** | 21/tcp 로 평문 연결 후 `AUTH TLS` 명령으로 TLS 업그레이드 | 21/tcp |

### 5-3. SFTP vs FTPS 비교

| 구분 | SFTP | FTPS |
|---|---|---|
| 기반 프로토콜 | SSH-2 | FTP + TLS |
| 표준 | IETF SECSH WG (드래프트 단계로 표준화 진행) | RFC 4217[^2] |
| 포트 | **22/tcp** | Implicit: 990/tcp / Explicit: 21/tcp |
| 채널 구조 | 단일 TCP 연결 | 제어·데이터 두 채널 모두 TLS 적용 |
| 방화벽 친화성 | 높음 | 능동/수동 모드 처리 필요 |
| 인증 방식 | SSH 키·비밀번호 | X.509 인증서·비밀번호 |

> *시험 포인트* : "SFTP = SSH 기반 (22/tcp)" / "FTPS = SSL/TLS 기반 (990/tcp 또는 21/tcp)" — 이름이 비슷해서 혼동하기 쉬운 함정.

## 단원 요약

| 영역 | 핵심 키워드 |
|---|---|
| **FTP 구조** | 제어 채널(21/tcp) + 데이터 채널 분리, 평문 전송 |
| **능동 모드** | 서버 → 클라이언트 (20/tcp), `PORT` 명령, 클라이언트 방화벽 부담 |
| **수동 모드** | 클라이언트 → 서버 (1024+), `PASV` 명령, 서버 방화벽 부담 |
| **FTP 공격** | Bounce(채널 미검증), TFTP(인증 부재, 69/udp), Anonymous(익명 계정) |
| **접근제어** | ftpusers(블랙리스트), user_list(vsFTP), TCP Wrapper |
| **암호화 FTP** | SFTP(SSH 22/tcp) vs FTPS(SSL/TLS 990/tcp Implicit, 21/tcp Explicit) |

### 시험 함정 정리

1. 능동 모드 = 서버가 클라이언트로 **역방향** 접속 (20/tcp). "능동" 이라는 단어 때문에 클라이언트가 능동적으로 연결한다고 외우면 틀린다.
2. ftpusers 파일은 **블랙리스트** 방식 — 명시된 계정이 차단된다.
3. SFTP 와 FTPS 는 **이름만 비슷하고 기반 프로토콜이 다르다** (SSH vs SSL/TLS).
4. FTP Bounce Attack 의 핵심 취약점은 **데이터 채널 목적지 IP 미검증** 이다.
5. TFTP 는 69/**udp** (FTP 21/tcp 와 다른 포트·다른 전송 계층).
