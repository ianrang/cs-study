# 06. 웹 서버 보안 운영과 보안서버 구축

## 0. 들어가며

이 단원은 05 단원의 어플리케이션 코드 레벨 취약점이 아니라 **웹 서버 운영 환경 자체** 의 보안을 다룬다. Apache HTTP Server 의 `httpd.conf` 설정을 통해 디렉터리 노출·메소드 제한·관리자 페이지 보호·검색 엔진 차단 등의 운영적 위험을 제거하고, SSL/TLS 보안 서버를 구축하며, 웹 로그를 분석하여 보안 사고를 탐지하는 세 영역으로 구성된다. OWASP Top 10 의 A05 (보안 설정 미흡), A06 (취약하고 오래된 컴포넌트), A09 (로깅·모니터링 실패) 영역과 직접 매핑된다.

### 출제기준 매핑

| 출제기준 | 본 단원 매핑 |
|---|---|
| 1.3 Web/App 보안 — Web/App 운영 (서버 설정) | §1 (개발 보안 관리), §2 (디렉터리·심볼릭 링크), §3 (메소드 제한), §4 (관리자·위치공개·검색엔진), §5 (정보 숨김·httpd.conf) |
| 1.3 Web/App 보안 — Web/App 장애 분석 및 대응 | §6 (웹 로그 분석) |
| 1.3 Web/App 보안 — Web/App 보안 기술 | §7 (보안서버 구축·SSL/TLS) |

### 학습 목표

1. Apache `httpd.conf` 의 주요 지시자를 읽고 운영 환경의 보안 설정 미흡 항목을 식별할 수 있다.
2. 디렉터리 리스팅·심볼릭 링크·불필요한 메소드·관리자 페이지 노출이 어떤 보안 위협을 만드는지 설명할 수 있다.
3. `robots.txt` 가 *접근 제어 기술이 아닌 검색 로봇과의 약속* 임을 이해하고 적절한 사용 범위를 판단할 수 있다.
4. 웹 로그 (Access Log·Error Log) 의 NCSA CLF·ELF 포맷을 해석할 수 있다.
5. SSL/TLS 보안 서버 구축 시 키 길이·서명 알고리즘·Cipher Suite 가이드를 설명할 수 있다.

## 1. 개발 보안 관리

> **개발 보안 관리** : 웹 어플리케이션 개발 시 사전 작업으로 서비스에 사용하는 웹서버와 어플리케이션 서버의 환경 설정에 대한 보안을 강화하기 위함.

웹 어플리케이션 보안의 첫 단계는 *코드를 짜기 전* 의 보안 원칙 수립이다. 통합자료에 명시된 8 가지 개발 보안 관리 원칙은 다음과 같다.

| 원칙 | 설명 |
|---|---|
| 전달된 값을 재사용하지 않는다 | 클라이언트가 보낸 값을 신뢰하지 말고 서버에서 재검증 |
| 최종 통제 메커니즘은 서버 단에서 수행한다 | 클라이언트 측 검증은 보조 수단일 뿐, 모든 보안 결정은 서버 |
| 클라이언트에게 중요 정보를 전송하지 않는다 | 응답에 민감한 데이터를 포함하지 않음 |
| 중요 정보 전송 시 POST 메소드 및 암호화 통신을 적용한다 | GET 의 URL 노출·로그 잔존을 방지 (04 단원 §3 cross-ref) |
| 중요한 트랜잭션이 일어나는 프로세스에 대해서는 재확인 절차를 추가한다 | 비밀번호 재인증 등 |
| 중요 정보가 보여지는 페이지는 캐시를 적용하지 못하게 한다 | `Cache-Control: no-cache` 등 |
| 적절한 방법으로 암호화를 수행한다 | 공인된 암호화 알고리즘 사용 (자체 개발 알고리즘 X) |
| 각 언어에서 제공하는 보안 수단을 이해한 후 사용한다 | 언어·프레임워크별 표준 보안 함수 사용 |

이 원칙들은 09 단원 (어플리케이션 개발 보안) 의 시큐어 코딩 가이드와 직접 연결된다.

> *시험 포인트* : 자체 개발 암호화 알고리즘 사용은 금지. 공인된 알고리즘(AES·RSA·SHA-256 등) 만 사용해야 한다 ([4과목 05 단원](../04.security-general/05.crypto-algorithms.md) 암호 알고리즘 + [4과목 06 단원](../04.security-general/06.hash-and-mac.md) 해시함수 cross-ref).

## 2. 디렉터리 노출 차단

### 2-1. 디렉터리 리스팅 제거

> **디렉터리 리스팅 (Directory Listing) 취약점** : 웹 서버의 미흡한 설정으로 인해 인덱싱 기능이 활성화되어 있으면, 공격자가 강제 브라우징을 통해 서버 내의 모든 디렉터리 및 파일 목록을 볼 수 있는 취약점.

**공격 흐름**

1. URI 에 특정 페이지가 아닌 *디렉터리 경로* 를 요청.
2. 웹 브라우저상에 해당 디렉터리 내의 디렉터리/파일 정보가 그대로 노출.
3. 디렉터리/파일을 클릭하여 그 내용을 확인 (백업 파일·소스 파일·미공개 파일 노출).

**대응 방안 (Apache)**

`/etc/httpd/conf/httpd.conf` 에서 모든 Directory 섹션의 `Options` 지시자에 있는 `Indexes` 옵션을 제거하거나 주석 처리한다.

```
<Directory />
    Options None              # Indexes 옵션이 없으면 디렉터리 리스팅 비활성화
    AllowOverride None
</Directory>
```

| 환경 | 설정 방법 |
|---|---|
| Apache | `httpd.conf` 의 모든 Directory 섹션에서 `Indexes` 옵션 제거 |
| IIS | "디렉터리 검색" 기능 해제 |
| Tomcat | `web.xml` 의 default 서블릿 초기 파라미터 `listings` 를 `false` 로 설정 |

### 2-2. 심볼릭 링크 사용 설정 제거

> **심볼릭 링크 사용 설정 취약점** : 아파치 웹서버 설정 파일의 모든 디렉터리 섹션의 `Options` 지시자에 `FollowSymLinks` 설정이 되어 있으면 심볼릭 링크가 가능한 취약점.

만약 시스템 자체의 루트 디렉터리에 심볼릭 링크를 걸게 되면 웹서버 구동 시 사용자 권한으로 모든 파일 시스템의 파일에 접근할 수 있게 되어, `passwd` 파일과 같은 민감한 파일을 누구나 열람할 수 있는 문제점이 발생한다.

**대응 방안**

```
<Directory />
    Options None              # FollowSymLinks 제거 또는 주석 처리
    AllowOverride All
</Directory>
```

> *시험 포인트* : `Options None` 한 줄로 `Indexes` 와 `FollowSymLinks` 가 모두 비활성화된다. 이 설정은 *최상위 디렉터리* 에 적용한 후 필요한 하위 디렉터리에서 명시적으로 허용하는 것이 안전한 운영 패턴이다.

## 3. 웹 서비스 메소드 제한

> **웹 서비스 메소드 설정 취약점** : 일반적으로 사용하는 GET·POST 메소드 이외의 불필요한 메소드를 허용하였을 경우, 공격자가 이를 이용하여 웹서버에 악성파일을 생성하거나 중요 파일의 삭제·수정 등이 가능한 취약점.

특히 `PUT`·`DELETE`·`TRACE` 메소드의 무인증 허용은 04 단원 §3 에서 다룬 위험을 그대로 만든다.

### 3-1. 웹 서비스 메소드 확인 과정

1. `telnet` 프로그램을 이용하여 telnet 으로 대상 웹서버의 서비스 포트로 접속.
2. `OPTIONS` 메소드 요청 메시지를 생성하여 전송.
3. `OPTIONS` 메소드를 지원할 경우 `OK` 응답과 `Allow` 응답 헤더 필드를 통해서 지원하는 메소드 리스트를 확인.

### 3-2. 대응 방안

`/etc/httpd/conf/httpd.conf` 의 DocumentRoot 디렉터리 섹션에 `LimitExcept` 지시자를 사용하여 불필요한 메소드를 제한한다.

```
<LimitExcept GET POST>
    Order allow, deny
    Deny from all
</LimitExcept>
```

`LimitExcept GET POST` 의 의미는 *"GET 과 POST 를 *제외한* 모든 메소드"* 를 차단한다는 것이다. 즉, GET·POST 만 허용되고 그 외 PUT·DELETE·TRACE 등은 모두 거부된다.

> *시험 포인트* : `Limit` 지시자 (특정 메소드 제한) 와 `LimitExcept` 지시자 (특정 메소드 *외* 제한) 의 의미 반대를 혼동하지 말 것.

## 4. 관리자 페이지·위치 공개·검색엔진

### 4-1. 관리자 페이지 노출 차단

> **관리자 페이지 노출 취약점** : 전반적인 기능 설정 및 회원 관리를 할 수 있는 관리자 페이지가 추측할 수 있는 형태로 구성되어 있으면, 공격자가 관리자 페이지에 쉽게 접근할 수 있으며 무작위 대입 공격을 통하여 관리자 권한을 획득할 수 있는 취약점.

**대응 방안**

| 대응 | 설명 |
|---|---|
| 관리자 페이지 접근제어 | IP 기반 접근제어 |
| 관리자 페이지 이름 변경 | 쉽게 추측할 수 없는 이름으로 페이지 이름 변경 (`/admin` → `/control_panel_xyz123`) |

```
<Directory "[관리자 페이지 절대경로]">
    Order Deny, Allow
    Deny from all
    Allow from [허용할 IP]
</Directory>
```

> **Apache `Order` 지시자** : `Allow` 와 `Deny` 의 평가 순서를 명시. 뒤에 오는 설정이 우선 적용된다. `Order Deny, Allow` 로 설정되어 있으면 `Allow` 설정이 우선 적용되어 *허용된 IP 만 접근 가능* 한 화이트리스트 정책이 된다.

### 4-2. 위치 공개 차단

> **위치 공개 취약점** : 백업 파일·로그 파일·압축 파일과 같은 파일이 자동적으로 생성되어 웹 어플리케이션 상에 노출될 경우, 공격자가 유추 후 직접 접근을 요청하여 핵심 정보를 획득할 수 있는 취약점.

**대응 방안**

```
<Files ~ "\.gz$">
    Order Allow, Deny
    Deny from all
</Files>

<Files ~ "\.bak$">
    Order Allow, Deny
    Deny from all
</Files>
```

`.gz`·`.bak`·`.swp`·`.old` 등 백업/임시 파일 확장자를 차단한다. 추가로 *불필요한 파일 자체를 웹 루트 디렉터리에서 삭제* 하는 것이 가장 근본적이다.

### 4-3. 검색엔진 정보 노출 차단

> **검색엔진 정보 노출 취약점** : 검색엔진에 의해 웹 사이트 해킹에 필요한 정보가 검색되어 해킹의 빌미가 제공되는 취약점.

**robots.txt 의 본질**

> **로봇 배제 표준 (Robots Exclusion Standard)** : 검색 로봇에 대한 웹 사이트의 디렉터리 및 파일들에 대한 검색 조건을 명시해 놓은 국제 규약. `robots.txt` 파일에 기술한다.

robots.txt 는 **방지 기술이 아닌 검색 로봇 운영자 간의 약속** 이므로 악의적인 검색엔진의 경우 이를 무시하고 컨텐츠를 수집할 수 있다. 따라서 robots.txt 는 *접근 제어 수단* 으로 사용해서는 안 되며, 보안이 필요한 자원은 별도의 인증·접근 제어 메커니즘으로 보호해야 한다.

**robots.txt 작성 규칙**

| 규칙 | 설명 |
|---|---|
| 파일명 | 반드시 `robots.txt` (대소문자 구분) |
| 위치 | 웹 사이트의 *최상위 디렉터리* 여야 함. 하위 디렉터리에 저장 시 효력 없음 |
| 인코딩 | ASCII 또는 UTF-8 |
| 다중 검색로봇 지정 | 한 줄을 띄워서 구분 |

**robots.txt 예시**

```
# /웹 루트 디렉터리/robots.txt
User-agent: Googlebot
Disallow: /cgi-bin/
Disallow: /private/
Disallow: /*.pdf$
Disallow: /*.gif$

User-agent: Yeti
Disallow: /cgi-bin/
Disallow: /private/
Disallow: /*.pdf$
Disallow: /*.gif$
```

| 패턴 | 의미 |
|---|---|
| `User-agent: *` + `Disallow: /` | 모든 검색로봇에 대해 웹 사이트 전체 접근 차단 |
| `Disallow:` (값 없음) | 웹 사이트 전체 허용 (`Allow: /` 와 동일) |
| `Disallow: /cgi-bin/` | 디렉터리 차단 (반드시 `/` 로 종료) |
| `Disallow: /*.pdf$` | 특정 확장자 차단 |

> *시험 포인트* : robots.txt 가 *접근 제어 기술이 아닌 약속* 임을 묻는 문제가 빈출. 민감한 정보의 보호는 robots.txt 에 의존하지 말고 인증·인가로 처리해야 한다.

## 5. 웹 서비스 정보 숨김과 httpd.conf

### 5-1. 웹 서비스 최소 권한 사용자 운영

> **웹 서비스 최소 권한 사용자 운영** : 웹서버 운영 시 위험을 최소화하기 위해 최소한의 권한을 가진 사용자와 그룹으로 웹서버 프로세스를 실행해야 함.

웹서버의 보안 취약점으로 인한 권한 탈취가 발생할 가능성이 항상 있으므로, 웹서버를 root 권한으로 실행하면 침해 시 시스템 전체가 위협된다. 따라서 별도의 사용자·그룹 (보통 `apache`·`nobody`) 으로 실행한다.

**설정 절차**

1. `/etc/passwd` 에서 웹서버 프로세스를 실행할 수 있는 사용자로 시스템에 로그인할 수 없도록 설정 (로그인 쉘을 `/sbin/nologin` 또는 `/bin/false` 로 지정).
2. `/etc/httpd/conf/httpd.conf` 에서 `User apache` 와 `Group apache` 설정.
3. `ps -ef | grep httpd` 명령으로 서비스 요청을 처리하는 서비스 프로세스를 확인.

Apache 는 80/tcp 포트를 열고 클라이언트 연결 요청을 처리하는 *root 권한의 리스닝 프로세스* 와 다수의 서비스 요청을 처리하는 *apache 권한의 서버 프로세스* 로 구성된다.

### 5-2. 웹 서비스 정보 숨김

> **웹 서비스 정보 숨김** : 웹서버에서는 응답 헤더나 에러 메시지 등에 웹서버 버전이나 종류·OS 정보·응용 프로그램 버전 등을 전송함. 공격자가 이 정보로 알려진 취약점을 공격하는 것을 차단하기 위해 정보 노출을 최소화함.

**Apache 설정**

| 지시자 | 설정 | 효과 |
|---|---|---|
| `ServerTokens Prod` | 응답 헤더의 정보 레벨 최소화 | `Server: Apache` 만 노출 |
| `ServerSignature Off` | 응답 헤더 또는 에러 메시지 등을 통한 웹서버 정보 차단 | 페이지 푸터의 서버 정보 제거 |

`ServerTokens` 의 값별 노출 정보:

| 값 | 노출 내용 |
|---|---|
| `Prod` | 웹서버 종류만 (`Apache`) |
| `Min` | 웹서버 종류 + 버전 |
| `OS` | 웹서버 종류 + 버전 + 운영체제 |
| `Full` | 웹서버 종류 + 버전 + 운영체제 + 설치된 모듈 정보 |

운영 환경에서는 `Prod` 가 표준이다.

### 5-3. httpd.conf 주요 지시자

`/etc/httpd/conf/httpd.conf` 파일에서 주요 설정이 가능하다. 시험에서는 각 지시자의 역할이 직접 출제된다.

**기본 설정**

| 지시자 | 설명 |
|---|---|
| `ServerRoot` | 웹서버가 설치된 최상위 디렉터리 경로를 지정 (설정 파일·로그 파일·모듈 등 위치) |
| `Listen [포트]` | 접속할 웹서버의 리스닝 포트를 설정 |
| `User` | 서버 프로세스를 실행할 사용자 정보 지정 |
| `Group` | 서버 프로세스를 실행할 그룹 정보 지정 |
| `ServerTokens` | HTTP 응답 시 헤더의 Server 필드를 통해 제공할 정보 레벨을 지정 |
| `ServerSignature` | 웹 서버의 각종 정보를 제공할지 지정 |
| `ServerAdmin` | 전달되는 에러 메시지에 포함될 관리자 메일 주소를 설정 |
| `DocumentRoot` | 웹 어플리케이션의 웹 루트 디렉터리를 지정 |
| `DirectoryIndex` | 웹 서버의 기본 페이지를 지정 (예: `index.php index.html`) |

**타임아웃 관련 설정 (Slow 계열 DDoS 대응)**

| 지시자 | 설명 |
|---|---|
| `Timeout` | 아무런 메시지가 발생하지 않을 동안의 대기 시간 (초 단위) |
| `KeepAlive` | 연결을 일정 조건에 따라 지속시켜 동일한 연결을 통해 다수 요청을 처리 기능의 사용 여부 |
| `MaxKeepAliveRequests` | 연결을 지속하는 동안에 허용할 최대 요청 건수 |
| `KeepAliveTimeout` | 연결을 지속하는 동안에 클라이언트의 마지막 요청 이후 다음 요청을 대기하는 시간 |
| `RequestReadTimeout` | 요청 메시지의 헤더 부·바디 부를 모두 수신하는 시간에 대한 타임아웃 — *Slow 계열 HTTP DoS 공격에 효과적 대응* |

**웹 로그 관련 설정**

| 지시자 | 설명 |
|---|---|
| `ErrorLog` | 에러 로그파일의 경로를 지정 |
| `CustomLog` | 접속기록을 저장하기 위한 로그파일의 경로를 지정 |

> *시험 포인트* : `RequestReadTimeout` 은 2과목 §5 의 Slow 계열 DDoS (Slowloris·Slow POST·Slow Read) 의 1차 대응책. 헤더·바디 수신 타임아웃을 짧게 설정하여 느린 연결을 차단한다.

## 6. 웹 로그 분석

> **웹 로그** : 클라이언트가 웹 서버에 접속하면 클라이언트와 웹서버 간에는 요청과 응답이 지속적으로 발생. 이 요청·응답 기록을 저장한 로그.

웹 로그는 다음과 같은 보안 운영 가치를 가진다.

- 접속 사용자의 행위/취향 등을 분석
- 보안 사고 발생 시 이를 *추적할 수 있는 증거 자료*
- 보안 사고 발생 전 이상 징후·해킹 시도·해킹 성공 여부를 확인
- 다양한 웹 패턴 파악
- 접속 시간·접근 파일 정보·사용자 정보·요청 방식·성공 여부 등을 확인

### 6-1. 웹 로그 종류

| 로그 | 설명 |
|---|---|
| **액세스 로그 (Access Log)** | 클라이언트의 요청으로 웹 서버가 응답한 내용을 기록 |
| **에러 로그 (Error Log)** | 클라이언트의 요청으로 웹 서버에 오류가 발생했을 경우 기록 |

### 6-2. 웹 로그 포맷

| 포맷 | 사용 환경 | 특징 |
|---|---|---|
| **NCSA CLF** (Common Log Format) | Apache·Tomcat | 기본 포맷 |
| **NCSA ELF** (Extended Log Format) | Apache·Tomcat | Referer·User-Agent 추가 |
| **W3C ELF** | IIS | Referer·User-Agent + Cookie + 사용자 정보 |

### 6-3. NCSA ELF 포맷의 필드

| 필드 | 설명 |
|---|---|
| **Host** | 클라이언트의 IP 주소나 호스트 이름 |
| **Ident** | 클라이언트를 인증하기 위해 사용 (사용하지 않음 — 보통 `-` 로 대체) |
| **Authuser** | 인증된 사용자의 이름 (사용하지 않음 — 보통 `-` 로 대체) |
| **Date and Time** | 클라이언트 요청이 발생한 시간 |
| **Request** | 요청 메소드(GET·POST 등) 와 요청한 리소스(URI) 를 포함 |
| **Status** | 서버의 응답 상태 코드 |
| **Bytes** | 클라이언트에게 전송된 바이트 수 |
| **Referer** | 클라이언트가 현재 요청한 페이지로 유입된 이전 페이지의 URL |
| **User-Agent** | 클라이언트가 사용하는 브라우저나 기기의 종류·버전 및 운영 체제 등 |

**액세스 로그 샘플**

```
192.168.1.100 - - [28/Apr/2026:14:30:22 +0900] "GET /index.html HTTP/1.1" 200 1024 "https://google.com" "Mozilla/5.0 ..."
```

이 한 줄에서 출발지 IP·요청 시간·요청 메소드·URI·상태 코드·응답 크기·이전 페이지·브라우저 정보를 모두 추출할 수 있다.

> *시험 포인트* : NCSA ELF 가 NCSA CLF 와 다른 점은 **Referer + User-Agent 추가** 다. 두 포맷의 차이가 직접 출제된다.

### 6-4. 알려진 공격 패턴

| 패턴 | 설명 |
|---|---|
| **제로보드 공격** | `cmd` 파라미터로 원격에서 임의 명령을 실행할 수 있는 취약점 이용 (제로보드는 PHP 로 개발한 웹 게시판 시스템) |
| **테크노트 공격** | `main.cgi` 의 `filename` 파라미터로 원격에서 임의 명령을 실행할 수 있는 취약점 이용 |

이런 알려진 패턴은 웹 로그의 URL/파라미터 영역에서 식별 가능하므로 SIEM·정규식 룰셋으로 탐지한다.

## 7. 보안서버 구축

### 7-1. 보안서버의 정의

> **보안서버** : 인터넷상에서 정보를 암호화하여 송수신하는 기능이 구축된 웹서버. 독립적인 하드웨어를 따로 설치하는 것이 아니라 이미 사용하고 있는 웹서버에 SSL/TLS 인증서나 암호화 소프트웨어를 설치하여 암호 통신을 지원함.

**보안서버 구축의 효과**

| 효과 | 설명 |
|---|---|
| 정보 유출 방지 | 중요 정보에 대해 암호 통신 수행 |
| 위조/가짜 사이트 방지 (피싱 방지) | SSL/TLS 인증서가 진짜 사이트임을 증명 |
| 기업 신뢰도 향상 | 개인정보를 안전하게 관리한다는 가시적인 홍보 효과 |

### 7-2. 보안서버 구축의 법적 근거

한국 개인정보보호법령은 보안서버 구축을 명시적으로 요구한다.

> **개인정보보호위원회 고시 「개인정보의 안전성 확보조치 기준」**

> 개인정보처리자는 고유식별정보(주민등록번호·여권번호·운전면허번호·외국인등록번호)·비밀번호·생체인식정보를 정보통신망을 통하여 송신할 경우 이를 암호화하여야 한다.

보안서버는 다음 중 하나의 기능을 갖추어야 한다:

1. 웹서버에 SSL 인증서를 설치하여 전송하는 정보를 암호화하여 송·수신하는 기능
2. 웹서버에 암호화 응용 프로그램을 설치하여 전송하는 정보를 암호화하여 송·수신하는 기능

### 7-3. SSL/TLS 서버 인증서 생성 시 보안 가이드

| 항목 | 가이드 |
|---|---|
| 키쌍 생성 | **2048 bit 이상** 의 키를 사용하고 서버 개인키에 대한 접근은 최소화하며 안전하게 관리. 생성된 서버 인증서의 필드 중 공개키 항목을 통해 키 길이 확인 |
| 인증서 서명 알고리즘 | **SHA-2 (256 bit) 이상** 으로 설정 |
| 주체/소유자 CN (CommonName) | 서브 도메인까지 정확히 명시 (서브 도메인을 명시하지 않으면 임의의 서브 도메인 인증서로 악용 가능) |
| 인증기관 선택 | 신뢰할 수 있는 인증기관을 통해 인증서 발급 (신뢰할 수 없는 CA 발급 시 웹 브라우저 경고) |

> *시험 포인트* : RSA 키 길이 **2048 bit 이상**, 서명 해시 **SHA-256 이상** — 두 수치는 빈출.

### 7-4. OpenSSL 라이브러리 보안 가이드

```
# OpenSSL 버전 확인
openssl version
openssl version -a    # 상세 정보 확인
```

| 가이드 | 설명 |
|---|---|
| 보안 업데이트 | 취약점이 발견된 OpenSSL 라이브러리 버전을 사용할 경우 보안 업데이트를 수행 (예: Heartbleed CVE-2014-0160) |

### 7-5. SSL/TLS 웹서버 설정 보안 가이드

`httpd-ssl.conf` 또는 `ssl.conf` 파일에서 SSL/TLS 설정을 한다.

```
# 취약한 통신 비활성화
SSLProtocol all -SSLv2 -SSLv3

# SSL/TLS 버전을 명시적으로 지정
SSLProtocol -all +TLSv1 +TLSv1.1 +TLSv1.2

# 취약한 암호 방식 제거 (예시)
SSLCipherSuite HIGH:!aNULL:!MD5:!3DES
```

| 항목 | 설명 |
|---|---|
| 비활성화 대상 | SSL2.0·SSL3.0 (POODLE 등 알려진 취약점) |
| 권장 버전 | TLS 1.2·TLS 1.3 (TLS 1.0·1.1 도 IETF 에서 deprecated 됨, RFC 8996[^1]) |
| 키 교환 알고리즘 | 익명 디피-헬만(Anonymous DH) 제거. **임시 DH (DHE/ECDHE)** 권장 — 키 교환 과정에서 서버 DH 파라미터를 RSA·DSA 등의 서명 알고리즘으로 인증하여 PFS (Perfect Forward Secrecy) 제공 |
| 주요 지시자 | `SSLProtocol` (지원할 버전) / `SSLCipherSuite` (암호 도구 목록) |

> **PFS (Perfect Forward Secrecy)** : 서버 개인키가 미래에 노출되더라도 과거 통신을 복호화할 수 없도록 매 세션마다 임시 키 쌍을 사용하는 특성. ECDHE·DHE 키 교환의 핵심 가치 ([4과목 03 단원 §3-4](../04.security-general/03.key-distribution.md) Forward Secrecy + [2과목 07 단원](../02.network-security/07.security-protocols.md) SSL/TLS cross-ref).

## 단원 요약

| 영역 | 핵심 키워드 |
|---|---|
| **개발 보안 원칙** | 서버 검증, POST + 암호화, 캐시 차단, 공인 알고리즘 |
| **디렉터리 보호** | `Options None` (Indexes·FollowSymLinks 제거) |
| **메소드 제한** | `<LimitExcept GET POST>` — GET·POST 외 차단 |
| **관리자·로봇** | IP 접근제어 + 페이지 이름 변경 / robots.txt 는 *약속이지 보안 아님* |
| **정보 숨김** | `ServerTokens Prod`, `ServerSignature Off` |
| **httpd.conf** | `User`·`Group`·`Timeout`·`KeepAlive`·`RequestReadTimeout`·`ErrorLog`·`CustomLog` |
| **웹 로그** | NCSA CLF / NCSA ELF (Referer·User-Agent 추가) / W3C ELF (Cookie 추가) |
| **보안서버** | RSA 2048+ / SHA-256 / SSLv2·SSLv3 비활성화 / TLS 1.2·1.3 / ECDHE+RSA |

### 시험 함정 정리

1. `LimitExcept GET POST` 는 **GET·POST 만 허용** 하는 것이지 GET·POST 를 차단하는 것이 아니다.
2. `Options None` 은 **Indexes(디렉터리 리스팅) + FollowSymLinks(심볼릭 링크)** 두 가지를 모두 비활성화한다.
3. **robots.txt 는 접근 제어 수단이 아닌 검색 로봇과의 약속** 이다 — 보안 의존 금지.
4. NCSA ELF 가 NCSA CLF 에 추가하는 것은 **Referer + User-Agent** 두 필드.
5. RSA 키 **2048 bit 이상**, 서명 해시 **SHA-256 이상**, **SSLv2·SSLv3 비활성화** 는 KISA·NIST 보안 가이드의 표준 수치.
6. `RequestReadTimeout` 은 Slow 계열 HTTP DoS (Slowloris·Slow POST) 의 1차 대응책 — 2과목 §5 cross-ref.
