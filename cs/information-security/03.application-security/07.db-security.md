# 07. 데이터베이스 보안

## 0. 들어가며

데이터베이스는 모든 웹 어플리케이션의 *데이터 영속화 계층* 이며, 어플리케이션 보안의 최종 방어선이다. 이 단원은 (1) DB 의 트랜잭션·SQL 분류 등 기본 개념, (2) 뷰·SQL 기반 접근 통제, (3) 집성·추론 등 DB 고유 보안 위협, (4) 컬럼·블록 단위 데이터 암호화, (5) MySQL DBMS 의 운영 보안 점검의 5 영역으로 구성된다. 05 단원 §2 의 SQL Injection 이 *어플리케이션 입력 검증의 실패* 라면, 본 단원은 *DB 자체의 보안 메커니즘* 을 다룬다.

### 출제기준 매핑

| 출제기준 | 본 단원 매핑 |

| 1.5 DB 보안 — DB 보안 개념 | §1 (트랜잭션 ACID), §2 (SQL 분류) |

### 학습 목표

1. 트랜잭션의 ACID 4 속성을 정의하고 각각이 어떤 데이터 일관성 위협을 막는지 설명할 수 있다.
2. SQL 의 4 분류 (DML·DDL·DCL·TCL) 각각의 명령어와 대표 예시를 구분할 수 있다.
3. 뷰 기반 접근 통제와 SQL 기반 접근 통제 (`GRANT`·`REVOKE`·`DENY`) 의 차이를 설명할 수 있다.
4. 데이터베이스 보안 위협 중 *집성* 과 *추론* 의 차이를 설명하고, 각각의 보안 통제 (접근 통제·추론 통제·흐름 통제) 의 적용 영역을 매핑할 수 있다.
5. 컬럼 암호화 (API·Plug-in·Hybrid) 와 블록 암호화 (TDE·파일 암호화) 의 차이와 각각의 적용 시나리오를 설명할 수 있다.

## 1. 트랜잭션과 ACID

### 1-1. 트랜잭션의 정의

> **트랜잭션 (Transaction)** : 데이터베이스에서 하나의 논리적 작업 단위로 묶이는 일련의 연산. 모두 성공하거나 모두 실패해야 한다.

트랜잭션의 정확성과 신뢰성을 보장하기 위해 데이터베이스는 다음 4 속성 (**ACID**) 을 보장해야 한다.

| 속성 | 영문 | 정의 |
|---|---|---|
| **원자성** | Atomicity | 트랜잭션 내의 모든 연산은 반드시 한꺼번에 완료되어야 하며, 그렇지 못한 경우는 한꺼번에 취소 |
| **일관성** | Consistency | 트랜잭션이 성공적으로 완료되면 일관성 있는 데이터베이스 상태로 변환 |
| **격리성** | Isolation | 트랜잭션 실행 중에 있는 연산의 중간 결과는 다른 트랜잭션이 접근하지 못함 |
| **영속성** | Durability | 트랜잭션이 일단 그 실행을 성공적으로 완료하면 그 결과는 영속적 (시스템 장애에도 유지) |

### 1-2. ACID 의 보안적 의미

ACID 속성은 단순한 데이터 일관성 보장이 아니라 보안 위협 방어 측면에서도 의미를 가진다.

| 속성 | 위협 방어 |
|---|---|
| 원자성 | 부분 커밋·부분 롤백으로 인한 데이터 불일치 방지 |
| 일관성 | 무결성 제약 위반 방지 |
| 격리성 | Race Condition 으로 인한 권한 우회 방지 (1과목 §6 cross-ref) |
| 영속성 | 시스템 장애 시 거래 데이터 손실 방지 |

> *시험 포인트* : ACID 의 한국어 명칭(원자성·일관성·격리성·영속성) 과 영문 약자, 정의가 자주 매칭 출제된다.

## 2. SQL 분류

SQL 명령은 목적에 따라 4가지로 분류된다. 각 분류의 명령어와 권한 부여 대상이 시험에서 직접 출제된다.

| 분류 | 풀네임 | 용도 | 명령어 |
|---|---|---|---|
| **DML** | Data Manipulation Language | 사용자가 DBMS 를 통해서 원하는 데이터를 *조작* 하는 데 사용 | `SELECT`·`UPDATE`·`INSERT`·`DELETE` |
| **DDL** | Data Definition Language | 데이터베이스 객체를 *정의·삭제·변경* 하기 위해 사용 | `CREATE`·`DROP`·`ALTER` |
| **DCL** | Data Control Language | 데이터베이스 관리 목적으로 사용자에게 데이터베이스 객체에 대한 *권한을 부여하거나 제거* | `GRANT`·`REVOKE` |
| **TCL** | Transaction Control Language | 논리적인 작업의 단위를 묶어서 DML 에 의해 처리된 결과를 *작업 단위로 제어* | `COMMIT`·`ROLLBACK` |

> *시험 포인트* : `GRANT`·`REVOKE` 는 **DCL** 이지 DML 이 아니다. `COMMIT`·`ROLLBACK` 은 **TCL** 이지 DDL 이 아니다 — 분류 오답 함정 빈출.

## 3. 데이터베이스 보안 위협

DB 고유의 두 보안 위협 — **집성** 과 **추론** — 은 각각 다른 메커니즘으로 기밀 정보를 도출하므로 별도의 통제가 필요하다.

### 3-1. 집성 (Aggregation)

> **집성 (Aggregation)** : 낮은 보안 등급의 정보 조각들을 조합하여 높은 보안 등급의 정보를 알아내는 행위.

개별 데이터 항목 자체는 기밀이 아니지만, 종합한 결과의 보안 등급이 높은 경우 기밀 정보 유출 문제가 발생한다. 예를 들어 한 직원의 이름·부서·직급은 각각 공개 정보일 수 있지만, 이를 *조합* 하여 누가 어떤 프로젝트에 참여하는지를 추론할 수 있다.

### 3-2. 추론 (Inference)

> **추론 (Inference)** : 보안 등급이 없는 일반 사용자가 보안으로 분류되지 않은 정보를 정당하게 접근하여 기밀 정보를 유추하는 행위.

추론은 주로 *통계* 로부터 발생하기 때문에 통계 정보로부터 개개인에 대한 정보를 추적하지 못하도록 해야 한다. 예를 들어 "30대 남성 직원의 평균 연봉" 통계가 한 명만 해당하는 조건이라면 그 한 명의 연봉이 그대로 노출된다.

### 3-3. 집성 vs 추론 비교

| 구분 | 집성 (Aggregation) | 추론 (Inference) |
|---|---|---|
| 기법 | 낮은 등급 정보를 *조합* | 공개 정보로부터 *유추* |
| 데이터 형태 | 개별 데이터 항목 | 주로 통계 정보 |
| 대응 | 접근 통제 + 흐름 통제 | 추론 통제 |

## 4. 접근 통제

데이터베이스 객체에 대한 접근 권한 부여는 (1) 뷰 기반과 (2) SQL 기반의 두 메커니즘으로 구현된다. 둘은 직교하며 함께 사용된다.

### 4-1. 뷰 기반 접근 통제

> **뷰 (View) 기반 접근 통제** : 뷰는 하나 이상의 기본 테이블로부터 유도되어 만들어지는 *가상의 테이블* 로, 실제로 존재하는 것이 아니라 뷰에 대한 조작 요구 시마다 기본 테이블의 데이터를 이용하여 만들어진다.

특정 필드만 접근할 수 있도록 뷰를 생성하여 뷰에만 접근 권한을 부여하면, 권한 없는 사용자로부터 민감한 정보를 *은닉* 할 수 있다.

**예시**

```sql
-- employees 테이블에서 salary 를 제외한 뷰 생성
CREATE VIEW employees_public AS
SELECT employee_id, name, department FROM employees;

-- 일반 사용자에게는 뷰에만 SELECT 권한 부여
GRANT SELECT ON employees_public TO general_user;
```

이 방식의 핵심 가치는 *기본 테이블에 직접 접근을 차단* 하면서도 *허용한 일부 컬럼만 노출* 하는 것이다.

### 4-2. SQL 기반 접근 통제 (GRANT / REVOKE / DENY)

> **SQL 기반 접근 통제** : 데이터베이스 객체에 대한 작업 권한을 제한하기 위해 DCL (Data Control Language) 을 사용하여 권한을 부여하거나 회수한다.

| 명령 | 형식 | 용도 |
|---|---|---|
| **GRANT** | `GRANT [부여할 권한] ON [데이터베이스 객체] TO [사용자] [WITH GRANT OPTION]` | 권한 부여 |
| **REVOKE** | `REVOKE [해제할 권한] ON [데이터베이스 객체] FROM [사용자]` | 권한 회수 |
| **DENY** | `DENY [거부할 권한] ON [데이터베이스 객체] TO [사용자]` | 권한 명시적 거부 (MS-SQL 전용) |

`WITH GRANT OPTION` 절을 사용하면 사용자가 부여받은 권한을 다른 사용자에게 또 다시 부여할 수 있다. 권한 위임 체인을 만들 때 사용된다.

```sql
-- 부여한 권한 확인 (MySQL)
SHOW GRANTS FOR 'username';
```

> *시험 포인트* : `DENY` 는 MS-SQL 의 전용 명령으로, *명시적 거부* 가 다른 모든 권한 부여보다 우선한다. `GRANT`·`REVOKE` 는 SQL 표준이지만 `DENY` 는 표준이 아니다.

### 4-3. DB 접근 통제 모델 (외부 보강)

NIST SP 800-53 Rev. 5[^1] 의 AC (Access Control) 패밀리는 DB 접근 통제 모델을 다음 3가지로 분류한다 (4과목 §접근통제 cross-ref).

| 모델 | 설명 | 예시 |
|---|---|---|
| **DAC (Discretionary Access Control)** | 객체 소유자가 권한을 자율 부여 | 일반적인 `GRANT`/`REVOKE` |
| **MAC (Mandatory Access Control)** | 시스템 정책에 따라 보안 라벨로 강제 부여 | 군·정부 시스템 |
| **RBAC (Role-Based Access Control)** | 역할에 권한을 부여하고 사용자에게 역할을 부여 | Oracle Roles, PostgreSQL Roles |

## 5. 데이터베이스 보안 통제

§3 의 두 보안 위협 (집성·추론) 과 그 외 위협을 통제하기 위해 DB 는 3가지 보안 통제 메커니즘을 제공한다.

| 통제 | 정의 | 대응 위협 |
|---|---|---|
| **접근 통제 (Access Control)** | 데이터베이스 사용자가 가진 접근 권한의 범위 내에서 데이터 접근을 허용하는 것 | 무단 접근, 집성 (1차) |
| **추론 통제 (Inference Control)** | 통계 정보 등 간접적으로 노출된 데이터를 통해서 민감/기밀 데이터가 유추되어 공개되는 것을 방지하는 것 | 추론 |
| **흐름 통제 (Flow Control)** | 접근이 가능한 객체 간의 정보의 흐름을 조정하는 것. 보안 등급이 높은 객체에서 보안 등급이 낮은 객체로의 정보 흐름을 제어 | 집성, 보안 등급 우회 |

### 5-1. 추론 통제의 구현 방법

추론 통제는 완벽한 해결책이 존재하지는 않지만 다음 기법을 조합한다.

| 기법 | 설명 |
|---|---|
| 허용 가능한 질의 제한 | 결과 집합 크기가 너무 작거나 너무 큰 질의 차단 |
| 질의 응답 데이터 한정 | 노출 가능한 데이터 범위 제한 |
| 반올림·노이즈 추가 | 데이터가 숫자인 경우 반올림하거나 일관성이 없는 질의 결과 제공 |

> *시험 포인트* : 보안 통제 3종 (접근·추론·흐름) 의 매핑은 빈출. *집성은 접근·흐름, 추론은 추론 통제* 로 대응한다.

## 6. 데이터베이스 암호화 기술

데이터베이스 암호화 방식은 (1) 테이블의 컬럼 단위로 데이터를 암호화하여 저장하는 **컬럼 암호화 방식** 과 (2) DBMS 블록 또는 파일 블록 단위로 암호화하여 저장하는 **블록 암호화 방식** 으로 구분된다.

### 6-1. 컬럼 암호화 방식

> **컬럼 암호화 방식** : 테이블의 컬럼 단위로 데이터를 암호화하여 저장하는 방식.

| 방식 | 설명 | 암복호화 모듈 위치 |
|---|---|---|
| **API 방식** | 암/복호화 모듈이 API 라이브러리 형태로 각 어플리케이션 서버에 설치되어 호출 | **어플리케이션 서버** |
| **Plug-In 방식** | 암/복호화 모듈이 DB 서버에 설치되고 DB 서버에서 플러그인으로 호출 | **DB 서버** |
| **Hybrid 방식** | 대용량 처리는 API 방식을 사용하고 나머지 부분들은 Plug-In 방식을 사용 | 혼합 |

**각 방식의 비교**

| 구분 | API 방식 | Plug-In 방식 |
|---|---|---|
| 암복호화 모듈 위치 | 어플리케이션 서버 | DB 서버 |
| 어플리케이션 변경 | 필요 (추가 개발 비용 발생) | 최소 |
| 통신 구간 보안 | 어플리케이션 서버 ↔ DB 서버 간 *암호화된 데이터* 전송 보장 | DB 서버 부하 |
| 적합 환경 | 대용량 트랜잭션 | 트랜잭션 처리량이 많지 않아 성능 민감성이 낮은 시스템 |

**Plug-In 방식의 동작 특성**

테이블/컬럼 암호화 시 기존 테이블 이름과 동일한 *뷰* 와 변경된 이름의 *암호화된 테이블* 이 생성되고, 원본 테이블은 삭제된다. 이렇게 함으로써 어플리케이션 코드는 변경 없이 그대로 작동한다 (뷰가 기존 테이블을 대신함).

> *시험 포인트* : "**암복호화 모듈이 어플리케이션이면 API, DB 서버이면 Plug-in**" — 한 줄로 외운다.

### 6-2. 블록 암호화 방식

> **블록 암호화 방식** : DBMS 블록 또는 파일 블록 단위로 암호화하여 저장하는 방식.

| 방식 | 풀네임 | 설명 |
|---|---|---|
| **TDE 방식** | Transparent Data Encryption | DBMS 에 내장된 암호화 기능을 이용하여 DB 내부에서 데이터 파일 저장 시 암호화하고, 파일에 저장된 내용을 메모리 영역으로 가져올 때 DBMS 에 의해 자동으로 복호화 |
| **파일 암호화 방식** | — | 운영체제에서 파일을 암호화하는 방식. 파일에 대해 직접 암호화를 수행하므로 DB 파일뿐만 아니라 로그 파일·이미지 파일·음성/영상 등의 다양한 비정형 데이터에 대한 암호화 적용 가능 |

**TDE 방식의 동작 특성**

- DBMS 커널 수준에서 처리되므로 기존 응용 프로그램의 수정이나 DB 스키마의 변경이 거의 필요 없음
- DBMS 엔진에 최적화된 성능 제공

**파일 암호화 방식의 특성**

- DBMS 종류에 상관없이 사용 가능
- DB 서버에 추가적인 부하 발생 가능

### 6-3. 컬럼 vs 블록 암호화 종합 비교

| 구분 | 컬럼 암호화 | 블록 암호화 |
|---|---|---|
| 단위 | 테이블 컬럼 | DBMS 블록 / 파일 블록 |
| 적용 대상 | 민감 컬럼 (주민번호·카드번호 등) | DB 파일 전체 |
| 어플리케이션 변경 | API 방식은 필요, Plug-in 은 최소 | 거의 없음 (TDE) |
| 검색 효율 | 암호화 컬럼 검색 시 성능 저하 | 메모리 영역에서는 평문 — 검색 효율 정상 |
| 보호 범위 | 컬럼 단위 | 디스크 저장 데이터 전체 |

> *시험 포인트* : 시험에서는 "암호화 모듈이 어디에 설치되는가" 와 "어떤 환경에 적합한가" 가 매칭 형태로 출제된다.

## 7. MySQL DBMS 취약점 점검

운영 환경의 MySQL 보안 점검 항목은 KISA·OWASP Database Security Cheat Sheet[^2] 가이드와 일치하는 다음 6가지로 정리된다.

### 7-1. 점검 항목

| 점검 항목 | 점검·조치 방법 |
|---|---|
| 디폴트 관리자 패스워드 및 계정명 변경 | mysql db 의 user 테이블에 root 사용자의 정보를 확인하여 변경 (Brute Force·Dictionary 공격 방지) |
| MySQL 설치 시 자동 생성되는 mysql 계정의 로그인 차단 | `/etc/passwd` 파일의 mysql 계정 점검. 로그인 쉘을 `/bin/false` 또는 `/sbin/nologin` 으로 변경 |
| 원격에서 MySQL 서버로의 접속 차단 | mysql 의 3306/tcp 포트 차단. `my.cnf` 파일에 `skip-networking` 추가 |
| 데이터베이스의 사용자별 접속/권한 설정 확인 | user 테이블의 host 컬럼을 확인하여 접근·권한 설정 |
| MySQL 버전 확인 및 보안 패치 적용 여부 점검 | `mysql -V` 또는 `mysql --version` 명령을 통해 버전 확인 |
| MySQL 데이터 디렉터리 보호 여부 점검 | `my.cnf` 파일을 통해 데이터 디렉터리 위치를 확인하여 점검 |

### 7-2. 위험한 권한

일반 사용자에게 다음 권한이 부여될 경우 시스템 침입 대상이 될 수 있어 *반드시 회수* 한다.

| 권한 | 위험성 |
|---|---|
| `File_priv` | 파일에 데이터를 쓰고 읽을 수 있는 권한 — 파일 시스템 접근 |
| `Process_priv` | `SHOW PROCESSLIST` 명령 실행을 통해 실행 중인 질의를 모니터링 — 다른 사용자의 SQL 노출 |
| `Shutdown_priv` | `mysqladmin shutdown` 명령어로 mysql 서버를 종료시킬 수 있는 권한 — 서비스 거부 |

### 7-3. `skip-networking` 설정

```
# /etc/my.cnf
[mysqld]
skip-networking
```

`skip-networking` 을 추가하면 네트워킹 기능을 비활성화하기 때문에 서비스 포트(3306/tcp) 를 오픈하지 않는다. 로컬에서는 `mysql.sock` 유닉스 도메인 소켓으로 통신하므로 로컬 어플리케이션은 정상 동작한다.

> *시험 포인트* : `skip-networking` 은 *로컬 통신은 유지하면서 네트워크 포트만 차단* 하는 옵션이다. 완전한 mysql 비활성화가 아님에 주의.

## 단원 요약

| 영역 | 핵심 키워드 |
|---|---|
| **트랜잭션** | ACID — 원자성(Atomicity)·일관성(Consistency)·격리성(Isolation)·영속성(Durability) |
| **SQL 분류** | DML(SELECT/UPDATE/INSERT/DELETE) / DDL(CREATE/DROP/ALTER) / DCL(GRANT/REVOKE) / TCL(COMMIT/ROLLBACK) |
| **DB 위협** | 집성(낮은 등급 조합) / 추론(통계로 유추) |
| **접근 통제** | 뷰 기반 (가상 테이블) + SQL 기반 (GRANT·REVOKE·DENY) |
| **보안 통제 3종** | 접근 통제·추론 통제·흐름 통제 |
| **컬럼 암호화** | API(어플리케이션) / Plug-In(DB) / Hybrid |
| **블록 암호화** | TDE(DBMS 내장) / 파일 암호화(OS 단) |
| **MySQL 점검** | root 계정 변경, 3306/tcp 차단, `skip-networking`, File·Process·Shutdown 권한 회수 |

### 시험 함정 정리

1. ACID 의 한국어 명칭 — 격리성(Isolation, *분리성* X), 영속성(Durability, *지속성* X) 정확한 표기.
2. `GRANT`·`REVOKE` 는 **DCL**, `COMMIT`·`ROLLBACK` 은 **TCL** — DML 이나 DDL 이 아니다.
3. **집성** 은 *조합* 으로 정보 도출, **추론** 은 *통계로 유추* — 메커니즘 차이.
4. **API 방식** = 어플리케이션 서버에 모듈, **Plug-In 방식** = DB 서버에 모듈.
5. **TDE** 는 *DBMS 내장* 자동 복호화, **파일 암호화** 는 *OS 단* 직접 암호화.
6. `WITH GRANT OPTION` 은 권한 위임 체인을 만든다 — 권한 부여를 받은 사용자가 다른 사용자에게 다시 권한 부여 가능.
7. `skip-networking` 은 네트워크 포트만 차단할 뿐 *로컬 통신은 정상* 작동.
