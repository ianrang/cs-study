# 03. 메일 보안

## 0. 들어가며

이메일은 인터넷 응용 서비스 중 가장 오래되었고 가장 광범위하게 사용되는 서비스 중 하나다. 그러나 SMTP·POP3·IMAP 의 원래 표준은 인증·기밀성·무결성 어느 것도 제공하지 않는 *완전 평문 프로토콜* 이었으며, 메일 시스템의 보안은 이후 30년에 걸쳐 (1) 메일 인증 기술 (SPF·DKIM·DMARC), (2) 콘텐츠 암호화·서명 (S/MIME·OpenPGP), (3) 전송 구간 암호화 (SMTPS·POP3S·IMAPS) 의 세 축으로 점진적으로 보강되어 왔다. 이 단원은 그 세 축을 메일 시스템의 4-Agent 구조 위에서 통합 학습한다.

### 출제기준 매핑

### 학습 목표

1. MUA·MTA·MDA·MRA 4 에이전트의 역할과 메일 송수신 흐름을 단계별로 설명할 수 있다.
2. SMTP 메시지의 봉투(envelope) 와 메시지(headers + body) 의 차이를 설명하고, 어느 필드가 위변조 가능한지 식별할 수 있다.
3. SPF·DKIM·DMARC 의 인증 대상·동작 메커니즘·인증 결과의 차이를 비교 설명할 수 있다.
4. 수신한 메일의 헤더 (Received·Received-SPF·Authentication-Results) 만으로 발송 경로와 인증 결과를 분석할 수 있다.
5. 콘텐츠 보안 (S/MIME·PGP) 과 전송 구간 보안 (SMTPS·POP3S·IMAPS) 의 적용 계층 차이를 설명할 수 있다.

## 1. 메일 시스템 개요

### 1-1. 메일 시스템 4-Agent 구조

> **이메일 시스템** : 사용자가 이메일을 작성·전송·수신·저장할 수 있도록 도와주는 여러 구성 요소와 프로토콜로 구성된다.

이메일 송수신은 단일 서버가 아니라 4 가지 독립 역할(Agent) 의 협력으로 이루어진다. 시험은 이 4 에이전트의 약자와 역할을 정확히 묻는다.

| Agent | 풀네임 | 역할 | 대표 구현 |
|---|---|---|---|
| **MUA** | Mail User Agent | 사용자가 메일을 송수신하기 위해 사용하는 메일 클라이언트 프로그램 | Outlook, Thunderbird |
| **MTA** | Mail Transfer Agent | 메일 서버 프로그램. 수신한 메일을 분석하여 메일을 수신자에게 전송 | Sendmail, Postfix, qmail |
| **MDA** | Mail Delivery Agent | 사용자 메일함에 메일을 저장 | Procmail |
| **MRA** | Mail Retrieval Agent | 메일 클라이언트가 확인을 요청하는 메일을 사용자 메일함에서 사용자로 전달 | Dovecot |

### 1-2. 주요 메일 프로토콜

각 에이전트 간의 통신은 다음 3가지 표준 프로토콜로 이루어진다.

| 프로토콜 | 용도 | 포트 | 특성 |
|---|---|---|---|
| **SMTP** | 메일 클라이언트(MUA) 와 메일 서버(MTA), 또는 송수신 메일 서버(MTA) 간에 메일을 전송 | **25/tcp** | 텍스트 기반 메일 전송 표준 (RFC 5321[^1]) |
| **POP3** | 메일을 서버로부터 *이동* 하는 방식으로 가져옴 | **110/tcp** | 서버에서 메일을 가져오면 서버에서는 메일이 삭제됨 (RFC 1939[^2]) |
| **IMAP** | 메일을 서버로부터 *복사* 하는 방식으로 가져옴 | **143/tcp** | 서버에 있는 메일을 삭제하지 않음. 다중 클라이언트 지원 (RFC 9051[^3]) |

POP3 와 IMAP 의 가장 중요한 차이는 *서버에 메일이 남는지 여부* 다. POP3 는 클라이언트로 다운로드 후 서버에서 삭제하므로 다른 클라이언트에서 메일 확인이 불가능하지만, IMAP 은 서버 사본을 유지하므로 모바일·데스크톱·웹 등 여러 클라이언트에서 동일한 메일 상태를 볼 수 있다.

> *시험 포인트* : "POP3 = 이동 (서버 삭제)" / "IMAP = 복사 (서버 유지)" — 한 글자 차이로 의미가 완전히 달라진다.

### 1-3. 메일 송수신 과정

이메일이 발송되어 수신자에게 도달하기까지의 전체 흐름은 다음 7단계다.

1. **송신자 메일 작성** : 송신자는 MUA 를 이용하여 메일을 작성.
2. **송신자 메일 전송** : SMTP 프로토콜을 통해 송신자가 설정한 MTA 로 메일을 전송.
3. **메일 큐 임시 저장** : MTA 는 수신한 메일을 메일 큐에 임시 저장.
4. **메일 분석** : 메일 주소를 확인하여 자신의 메일 주소인지를 확인. (자신이 아니면 해당 도메인의 메일 서버로 SMTP 로 재전송)
5. **사용자 메일함 저장** : MDA 를 통해 각 사용자 메일함에 파일 형태로 저장.
6. **수신자 메일 요청** : 수신자는 MUA 를 이용하여 자신의 메일 확인을 요청.
7. **수신자 메일 전달** : MRA 는 해당 사용자의 메일함에 있는 메일을 POP3 또는 IMAP 으로 수신자에게 전달.

수신자 메일 주소의 메일 서버 확인은 메일 주소에 있는 도메인을 이용하여 **MX 레코드 DNS 질의** 로 수행된다 (01 단원 §3 cross-ref).

## 2. SMTP 프로토콜

### 2-1. SMTP 메일 형식 — 봉투와 메시지

SMTP 는 텍스트 기반 프로토콜로 메일 데이터를 **봉투 (Envelope)** 와 **메시지 (Message)** 두 개의 독립 정보 영역으로 구분하여 주고받는다. 이 분리는 시험 함정의 핵심 — *봉투는 수신자에게 전달되지 않고, 메시지는 발신자가 임의로 조작 가능하다*.

| 영역 | 용도 | 주요 필드 | 발신자 조작 가능 여부 |
|---|---|---|---|
| **봉투 (Envelope)** | MTA 간에 메일 송/수신을 위한 정보. 수신 메일서버에게 전달 | `MAIL FROM` (발신자 MTA 메일주소) / `RCPT TO` (수신자 MTA 메일주소) | `MAIL FROM` 은 공격자에 의해 악의적으로 조작될 수 있음 |
| **메시지 (Message)** | 송/수신자가 확인할 수 있는 정보. 메일 수신자에게 전달 | `From` (발신자 MUA) / `To` (수신자 MUA) / `Subject` | `From` 은 공격자에 의해 악의적으로 조작될 수 있음 |

> *시험 포인트* : 메일 클라이언트에서 보이는 발신자 주소는 **메시지의 `From`** 이다. 봉투의 `MAIL FROM` 과 다를 수 있고 둘 다 위조 가능하다 — 이것이 §4 의 SMTP Spoofing 의 본질이며 §5 의 SPF·DKIM·DMARC 가 해결하려는 문제다.

### 2-2. SMTP 릴레이

> **SMTP 릴레이 (SMTP Relay)** : 메일 서버 외부에서 메일 서버를 경유하여 다른 메일 서버로 메일을 발송하는 것. 이때 경유한 메일 서버를 SMTP 릴레이 서버라고 한다.

릴레이 서버의 발신자 인증 여부에 따라 두 가지로 구분된다.

| 종류 | 설명 | 보안 결과 |
|---|---|---|
| **오픈 릴레이 (Open Relay)** | 인증 없이 모든 메일을 릴레이 해주는 메일 서버 | 스팸 메일 발신자의 메일 서버로 악용 |
| **인증된 릴레이 (Authenticated Relay)** | 인증을 통해 메일 릴레이가 가능하도록 발신자를 제한한 메일 서버 | SMTP AUTH 기능을 통한 발신자 계정 인증 수행 |

오픈 릴레이를 운영하면 스팸 발송에 가담하는 결과가 되므로 모든 운영 환경에서 명시적으로 차단해야 한다.

## 3. sendmail 운영

### 3-1. sendmail 구성 파일 및 디렉터리

> **sendmail** : 인터넷 전자 메일의 표준 규약인 SMTP 프로토콜을 통해서 메일 서버 간에 메일을 주고받는 역할을 하는 무료 오픈소스 소프트웨어. 시험에서는 sendmail 의 설정 파일·디렉터리 위치가 직접 출제된다.

| 파일 / 디렉터리 | 설명 |
|---|---|
| `/usr/sbin/sendmail` | sendmail 주 데몬 실행파일 |
| `/usr/sbin/makemap` | sendmail 맵 생성 실행파일 (`access.db` 등 DB 파일 생성) |
| `/var/spool/mqueue` | 수신한 메일을 임시 저장하는 디렉터리 (메일 큐) |
| `/var/spool/mail` | 사용자 메일함 디렉터리 (개별 사용자별로 수신된 메일 보관) |
| `/var/log/maillog` | 메일 서버의 로그파일 |
| `/etc/mail/access` | 스팸메일 및 릴레이 차단을 위한 설정 파일 |
| `/etc/mail/sendmail.cf` | sendmail 설정파일 |

### 3-2. access 파일 설정

sendmail 8.9 버전 이후부터 스팸 차단과 관련된 기능이 추가되었으며, **access DB** 라는 데이터베이스를 이용해 특정 메일의 차단 및 릴레이 제한을 설정할 수 있다.

**적용 대상 지정**

| 지정 형식 | 예시 |
|---|---|
| IP | `192.168.1.10` |
| IP 대역 | `192.168.1` |
| 도메인 | `spam-domain.com` |
| 메일 주소 | `spammer@example.com` |

**처리 방식 지정**

| 처리 방식 | 동작 |
|---|---|
| **OK** | 지정한 대상의 메일을 조건 없이 모두 허용 |
| **RELAY** | 지정한 대상의 메일 릴레이를 허용 |
| **REJECT** | 지정한 대상의 메일을 모두 거부 (거부 메시지 송신) |
| **DISCARD** | 지정한 대상의 메일을 모두 폐기 (폐기 메시지 송신 X). 발신자 입장에서는 정상 처리된 것으로 보임 |
| **501 message** | 지정한 메일주소와 일치할 경우 메일을 거부하고 설정한 거부 메시지를 보냄 |

**파일 적용 방법**

access 파일은 텍스트 파일로 작성한 후, `makemap` 명령어를 사용하여 access.db 데이터베이스 파일로 변환해야 실제 정책이 적용된다.

```
makemap hash /etc/mail/access < /etc/mail/access
```

> *시험 포인트* : `REJECT` 와 `DISCARD` 의 차이가 빈출. REJECT 는 발신자에게 거부 알림이 가지만, DISCARD 는 발신자가 정상 발송된 것으로 인식한다 (스팸 발신자가 차단을 우회하려는 재시도를 막기 위함).

## 4. 메일 서비스 공격 유형

메일 시스템에 대한 주요 공격은 (1) 발신자 위조 (Spoofing), (2) 메일 내용 도청 (Sniffing), (3) 스팸·악성 메일·메일 웜 의 3가지 카테고리로 분류된다. §5·§7·§8 의 각 보안 기술이 이 3가지 위협 중 어떤 것을 해결하는지 매핑하여 학습한다.

### 4-1. SMTP 발신자 위조 (SMTP Spoofing)

> **SMTP Spoofing** : 메일 봉투의 `MAIL FROM` 정보를 조작하여 실제와 다른 신뢰할 만한 발신자 메일 주소로 조작해 수신 메일 서버를 속이는 공격.

원시 SMTP 표준은 발신자 인증을 요구하지 않으므로 누구나 임의의 `MAIL FROM` 으로 메일을 발송할 수 있다. 이 위협을 해결하기 위해 §5 의 SPF 가 등장했다.

### 4-2. 메일 메시지 도청 (Sniffing)

원시 SMTP·POP3·IMAP 은 모두 평문 텍스트로 메일을 전송하기 때문에 전송 구간에서 공격자가 부정한 방법으로 메일을 가로채 메시지를 훔쳐보거나 조작하는 **중간자 공격 (Man-In-The-Middle)** 이 발생할 수 있다. 이 위협을 해결하기 위해 §8 의 전송 구간 암호화 (SMTPS·POP3S·IMAPS) 가 등장했다.

### 4-3. 스팸·악성 메일·메일 웜

대량의 광고성 메일(스팸), 악성코드 첨부·링크가 포함된 악성 메일, 메일을 통해 자기 복제하는 웜은 모두 메일 시스템의 *콘텐츠 검사 부재* 를 악용한다. 이 위협은 §3-2 의 access 파일 차단, §6 의 스팸 방지 보안 도구, §5 의 인증 기술이 종합적으로 대응한다.

## 5. 이메일 인증 기술 (SPF·DKIM·DMARC)

이메일 인증 기술은 발신자가 정당한지 (SPF), 메시지 내용이 위변조되지 않았는지 (DKIM), 두 결과를 정책적으로 어떻게 처리할지 (DMARC) 를 보강하는 3단계 메커니즘이다. 모두 발신 도메인의 DNS TXT 레코드에 등록되어 작동한다 (01 단원 §3 TXT 레코드 cross-ref).

### 5-1. SPF (Sender Policy Framework)

> **SPF (Sender Policy Framework)** : 수신한 메일의 발신자 IP 주소가 발신자 메일 주소 도메인의 실제 메일 서버 정보와 일치하는지 검사할 수 있는 기술. RFC 7208[^4]. 발송자 메일 서버 인증 또는 메일 서버 등록제로 표현된다.

**동작 단계**

1. 발신자는 발신 메일 서버 정보를 SPF 형식으로 발신 도메인의 DNS 에 TXT 레코드로 등록.
2. 메일을 전송.
3. 수신 메일 서버는 발신 IP 를 발신 DNS 에 질의.
4. 주소 일치 여부 확인 (등록된 발신 메일 서버 주소 = 수신받은 메일의 발신 메일 서버).
5. 발신 IP 가 일치할 경우 수신, 발신 IP 가 불일치할 경우 차단.

**SPF 레코드 형식**

```
"v=spf1 [한정자][메커니즘] [한정자][메커니즘] ... [한정자]all"
```

| 한정자 | 의미 | 매칭 시 처리 |
|---|---|---|
| `+` | Pass (인증 통과). 메커니즘에 한정자를 지정하지 않은 경우 기본 적용 | 인증된 메일 서버로 메일 통과 |
| `-` | Fail (인증 실패) | 인증되지 않은 메일 서버로 메일 거부 (수신 메일 서버에서 스팸함 보관 또는 거부) |
| `~` | Soft Fail (가벼운 인증 실패) | 인증되지 않았지만 수신 측 메일 서버 정책에 따라 허용 여부 판단 |
| `?` | Neutral (중립) | 정책 판단 보류 |

| 메커니즘 | 의미 |
|---|---|
| `ip4`·`ip6` | IPv4·IPv6 주소·대역 |
| `include` | 다른 도메인의 SPF 레코드 참조 |
| `all` | 위 메커니즘들에 매칭되지 않는 모든 경우 |

**예시**

```
v=spf1 ip4:211.252.150.8 ip4:202.30.50.0/24 -all
```

발신 메일 서버의 IPv4 주소가 `211.252.150.8` 이거나 `202.30.50.0/24` 대역이면 통과(Pass), 그 이외의 모든 주소는 거부(Fail).

```
v=spf1 include:_spf.kisa.or.kr ~all
```

`_spf.kisa.or.kr` 도메인의 SPF 레코드 정보를 참조하여 메일 허용 여부를 판단하고, 그 외 주소는 수신 측 메일 서버 정책에 따라 허용 여부 판단 (Soft Fail).

### 5-2. DKIM (DomainKeys Identified Mail)

> **DKIM (DomainKeys Identified Mail)** : 발신자가 발신 메일의 메시지 헤더와 바디에 대한 디지털 서명을 메일 헤더에 삽입하여 수신된 이메일이 위변조되지 않았는지 수신자 측에서 검증할 수 있는 이메일 인증 기술. RFC 6376[^5]. 도메인 키 인증 메일이라고 표현된다.

**동작 단계**

1. 발신 메일 서버에서 발신 도메인의 DNS 에 공개키 정보를 TXT 레코드로 등록.
2. 발신 메일 서버에서 메일을 개인키로 전자서명한 후 `DKIM-Signature` 헤더를 추가하여 전송.
3. 수신 메일 서버는 `DKIM-Signature` 헤더에 명시된 selector·도메인을 참조하여 발신 DNS 에 공개키를 질의.
4. 공개키 값을 회신받아 서명값 검증 수행.
5. 해시값이 일치할 경우 수신, 해시값이 불일치할 경우 차단.

DKIM 은 SPF 와 달리 *메시지 자체의 위변조 검증* 을 제공하므로 메일이 중간 MTA 를 거치며 변조되었는지를 탐지할 수 있다.

### 5-3. DMARC (Domain-based Message Authentication, Reporting and Conformance)

> **DMARC** : 정당한 발신자인지 여부와 메일의 위변조 여부를 검사하고, 수신 측에서 정책 처리 결과에 대한 DMARC 보고서를 발신 측에 전달하여 발신 측에서 메일 모니터링을 할 수 있는 이메일 인증 기술. RFC 7489[^6]. 도메인 기반 이메일 인증으로 표현된다.

DMARC 의 본질은 SPF·DKIM 의 *결합 적용* 과 *정책 처리 결과 보고* 다. SPF/DKIM 만 적용하면 인증 실패 시 어떻게 처리할지가 도메인 소유자가 아닌 수신 서버 정책에 맡겨지지만, DMARC 는 발신 도메인 소유자가 직접 정책(`p=`) 을 선언할 수 있게 한다.

**동작 단계**

1. 발신자는 발신 메일 서버 정보를 SPF 형식으로 발신 DNS 에 등록.
2. 발신 메일 서버에서 발신 DNS 에 공개키 정보를 등록 (DKIM).
3. 발신 메일 서버에서 메일을 개인키로 전자서명 후 메일을 전송.
4. 수신 메일 서버는 발신 DNS 에 SPF + DKIM 을 포함한 DMARC 정보를 조회.
5. 인증에 성공할 경우 수신, 인증에 실패할 경우 보고서를 작성하여 발신 DNS 에 인증 결과를 보고.

**DMARC `p=` 정책**

| 정책 | 동작 |
|---|---|
| `p=none` | 인증 실패해도 정상 처리. 보고만 수신 (관찰 모드) |
| `p=quarantine` | 인증 실패 시 격리 (스팸함 등) |
| `p=reject` | 인증 실패 시 수신 거부 |

### 5-4. SPF·DKIM·DMARC 비교

| 구분 | SPF | DKIM | DMARC |
|---|---|---|---|
| 인증 대상 | 발신 IP 주소 | 메시지 헤더·바디 | SPF + DKIM 결합 |
| 검증 방식 | 발신 도메인의 SPF 레코드와 IP 매칭 | 디지털 서명 검증 | 두 인증의 결합 + 정책 적용 + 보고 |
| 위변조 탐지 | 불가 (IP 만 검증) | 가능 (메시지 자체 서명) | 가능 |
| 표준 | RFC 7208[^4] | RFC 6376[^5] | RFC 7489[^6] |
| DNS 레코드 | TXT (`v=spf1`) | TXT (`v=DKIM1`) | TXT (`v=DMARC1`) |

> *시험 포인트* : 세 기술의 인증 대상 차이가 가장 빈출. SPF = 발신 IP / DKIM = 메시지 / DMARC = 두 결과의 조합 + 정책 + 보고.

### 5-5. 메일 헤더 정보

수신한 메일을 분석할 때 다음 헤더가 인증 결과·발송 경로를 알려준다. 시험에서는 헤더 이름과 역할이 직접 출제된다.

| 헤더 | 역할 |
|---|---|
| **Received** | 실제 메일이 전송된 경로. 메일 서버를 경유할 때마다 추가됨. 가장 아래쪽이 처음 발송한 메일 서버, 위로 올라갈수록 거쳐 온 메일 서버 — *근원지 역추적* 의 핵심 정보 |
| **Received-SPF** | 수신 메일 서버에서 SPF 인증 결과를 기록 (`pass` / `fail` / `softfail` / `neutral`) |
| **Authentication-Results** | 수신 메일 서버에서 SPF·DKIM·DMARC 인증 결과를 모두 기록 (`spf=pass`·`dkim=pass`·`dmarc=pass` 등) |

`Received` 헤더의 일반 형식은 다음과 같다.

```
Received: from <발송 호스트> by <수신 호스트>
          id <메일 고유 식별자> for <수신자 메일주소>
          ; <수신 날짜/시간>
```

**SMTPS 적용 시 Received 헤더의 TLS Cipher Suite 표기**

> *원본 표기 보강* : 통합자료 line 3444 ~ 3451 의 "프로토콜 / 키 교환 알고리즘 / 인증/서명 알고리즘 / 대칭키 블록 암호 알고리즘 / 블록암호 운용모드 / 메시지 인증 알고리즘" 표기는 SMTPS 적용 시 Received 헤더에 기록되는 TLS Cipher Suite 의 구성 요소를 풀어 쓴 것이다.

| 구성 요소 | 예시 값 | 역할 |
|---|---|---|
| 프로토콜 | TLS 1.2 | TLS 버전 정보 |
| 키 교환 알고리즘 | ECDHE | TLS 통신에 필요한 키 교환을 위해 사용 |
| 인증/서명 알고리즘 | RSA | 인증서에 대한 인증 및 키 교환 시 사용하는 매개변수에 대한 서명을 위해 사용 |
| 대칭키 블록 암호 알고리즘 | AES128 | 대칭키 블록 암호화를 위해 사용 |
| 블록암호 운용모드 | GCM | 블록 암호 운용 모드 |
| 메시지 인증 알고리즘 | SHA256 | 메시지 인증을 위한 MAC 알고리즘으로 사용 |

이 6 요소의 조합이 TLS Cipher Suite (예: `ECDHE-RSA-AES128-GCM-SHA256`) 가 된다 (4과목 §TLS cross-ref).

## 6. 스팸메일 방지 보안 도구

리눅스/유닉스 환경에서 스팸 메일을 차단하는 대표적인 보안 도구는 다음과 같다. 시험에서는 각 도구의 동작 방식·설치 위치·주요 특성이 출제된다.

| 도구 | 설명 |
|---|---|
| **Procmail** | sendmail.cf 에 포함시키거나 사용자 디렉터리의 `.forward` 파일에 위치시켜 실행시키는 Plug-In 형식의 메일 처리 도구 |
| **Sanitizer** | 이메일을 이용한 모든 공격에 대해 효과적으로 대응할 수 있도록 해주는 procmail ruleset |
| **Inflex** | 로컬이나 외부로 나가는 이메일에 대해 Inbound 와 Outbound 정책을 세워 메일 필터링 가능 |
| **SpamAssassin** | 헤더 내용·화이트리스트·블랙리스트·RBL (Realtime Blackhole List) 등의 다양한 분류로 메일 필터링 |

> *시험 포인트* : Procmail 은 sendmail 에 통합되는 *처리 엔진* 이고, Sanitizer 는 그 위에서 동작하는 *룰셋* 이다. 둘은 동등한 도구가 아니라 계층 관계임에 주의.

## 7. 메일 콘텐츠 보안 (외부 보강)

§5 의 SPF·DKIM·DMARC 가 *발신자 인증* 을 담당한다면, 메일 콘텐츠 자체의 *암호화·서명* 은 다음 두 표준이 담당한다. 둘 다 메일 본문 (Body) 을 직접 암호화·서명하므로 메일이 어떤 MTA 를 거치든 종단간 보안이 유지된다 (4과목 §전자서명·PKI cross-ref).

### 7-1. S/MIME (Secure / Multipurpose Internet Mail Extensions)

> **S/MIME** : MIME 형식의 메일 본문을 X.509 인증서 기반으로 암호화·서명하는 표준. RFC 8551 (S/MIME 4.0, 2019)[^7].

| 항목 | 설명 |
|---|---|
| 인증서 | X.509 v3 — 공인 인증기관(CA) 발급 |
| 암호화 | 대칭키 + 수신자 공개키로 키 래핑 (하이브리드 방식) |
| 전자서명 | 발신자 개인키로 본문 해시 서명 |
| 운영 환경 | 기업 환경 — CA 기반 PKI 구축이 전제 |

### 7-2. OpenPGP (Pretty Good Privacy)

> **OpenPGP** : Phil Zimmermann 의 PGP 를 표준화한 메일 보안 프로토콜. RFC 4880 (2007)[^8] / RFC 9580 (2024 갱신).

| 항목 | 설명 |
|---|---|
| 인증서 | 자체 키 (CA 불필요). Web of Trust 모델 |
| 암호화 | 하이브리드 방식 (대칭키 + 비대칭키) |
| 전자서명 | 발신자 개인키로 본문 해시 서명 |
| 운영 환경 | 개인 환경 — 자체 키 관리·신뢰망 운영 |

### 7-3. S/MIME vs OpenPGP

| 구분 | S/MIME | OpenPGP |
|---|---|---|
| 신뢰 모델 | PKI (CA 계층) | Web of Trust (P2P 신뢰) |
| 인증서 표준 | X.509 | OpenPGP 자체 형식 |
| 운영 주체 | 기업·기관 | 개인 |
| 키 배포 | LDAP·CA Repository | 키 서버·직접 교환 |

## 8. 메일 전송 구간 보안 (외부 보강)

§7 의 콘텐츠 보안이 *종단간 보안* 이라면, **전송 구간 보안** 은 메일 서버 ↔ 메일 서버, 메일 서버 ↔ 클라이언트 사이의 TCP 연결을 TLS 로 암호화한다.

| 표준 | 기존 평문 프로토콜 | 보안 적용 후 | 포트 |
|---|---|---|---|
| **SMTPS** | SMTP (25/tcp) | SMTP over SSL/TLS | **465/tcp** |
| **POP3S** | POP3 (110/tcp) | POP3 over SSL/TLS | **995/tcp** |
| **IMAPS** | IMAP (143/tcp) | IMAP over SSL/TLS | **993/tcp** |
| **STARTTLS** (Submission) | SMTP | 평문으로 시작 후 TLS 업그레이드 | 587/tcp |

### 8-1. MTA-STS · TLS-RPT (외부 보강)

| 표준 | RFC | 역할 |
|---|---|---|
| **MTA-STS** | RFC 8461[^9] | 수신 도메인이 자신을 향해 들어오는 SMTP 가 반드시 TLS 를 사용하도록 정책 선언 |
| **TLS-RPT** | RFC 8460[^10] | TLS 협상 실패·인증서 오류 등 전송 구간 보안 문제를 발신자가 도메인 소유자에게 보고 |

> *시험 포인트* : 메일 보안의 3 계층 구조 — (1) §5 인증 (SPF·DKIM·DMARC) / (2) §7 콘텐츠 (S/MIME·PGP) / (3) §8 전송 구간 (SMTPS·POP3S·IMAPS·MTA-STS) — 은 *서로 대체하지 않고 보완한다*. 종단간 메일 보안은 세 계층을 모두 적용해야 완성된다.

## 단원 요약

| 영역 | 핵심 키워드 |
|---|---|
| **4-Agent 구조** | MUA(클라이언트)·MTA(서버)·MDA(저장)·MRA(전달) |
| **프로토콜** | SMTP(25/tcp), POP3(110/tcp 이동), IMAP(143/tcp 복사) |
| **SMTP 메일 형식** | 봉투(MAIL FROM·RCPT TO) ≠ 메시지(From·To·Subject), 둘 다 위조 가능 |
| **sendmail** | `/etc/mail/access`, `makemap hash`, REJECT(알림 O) vs DISCARD(알림 X) |
| **인증 기술** | SPF(IP 검증)·DKIM(메시지 서명)·DMARC(결합+보고) — 모두 발신 도메인 DNS TXT 레코드 |
| **메일 헤더 분석** | Received(경로), Received-SPF, Authentication-Results |
| **콘텐츠 보안** | S/MIME(X.509·PKI), OpenPGP(Web of Trust) |
| **전송 구간 보안** | SMTPS(465), POP3S(995), IMAPS(993), STARTTLS(587), MTA-STS, TLS-RPT |
| **스팸 방지 도구** | Procmail(엔진), Sanitizer(룰셋), Inflex, SpamAssassin |

### 시험 함정 정리

1. POP3 = **이동** (서버 삭제) / IMAP = **복사** (서버 유지). 한 글자 차이로 의미 반대.
2. SMTP 봉투의 `MAIL FROM` 과 메시지의 `From` 은 *다른 필드* 이며 둘 다 위조 가능하다.
3. SPF 는 발신 IP 만 검증, DKIM 은 메시지 자체를 검증, DMARC 는 두 결과 + 정책 + 보고.
4. `REJECT` 는 발신자에게 거부 알림 전송, `DISCARD` 는 알림 없이 폐기 (스팸 발신자 우회 방지).
5. SMTPS·POP3S·IMAPS 의 포트는 **465·995·993** — 기존 평문 포트 + 보안 포트의 한 자리만 다름.
6. DNSSEC 처럼 SPF·DKIM·DMARC 도 **발신 도메인의 DNS TXT 레코드** 에 등록한다 (01 단원 §3 cross-ref).
