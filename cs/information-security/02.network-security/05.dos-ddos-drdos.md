## 1. DoS — 정의와 분류

### 1-1. 정의

> **서비스 거부 공격(DoS, Denial of Service)** : 공격 대상 시스템이 정상적인 서비스를 할 수 없도록 서비스를 마비시키는 공격. 공격자가 **단일 컴퓨터** 를 이용한다는 점이 DDoS 와 구분된다.

- 가용성(Availability) 을 표적으로 함.
- 소프트웨어 취약점을 이용하는 공격과 패킷을 변조하여 공격하는 등 다양한 형태가 있다.

### 1-2. DoS 공격 분류

| 공격 분류 | 공격명 | 핵심 원리 |
|---|---|---|
| 시스템 자원 소진 / 부하 유발 | Ping of Death | ICMP + IP 단편화 재조합 부하 |
| | Land | 출발지 IP = 목적지 IP |
| | Smurf | Directed Broadcast + ICMP Echo Reply 증폭 |
| | Teardrop / Bonk / Boink | IP fragment offset 조작 |
| 패킷 필터링 우회 (DoS 가 아닌 우회 공격) | Tiny Fragment | 첫 단편을 작게 만들어 목적지 포트 숨김 |
| | Fragment Overlap | 단편 오프셋 중첩으로 목적지 포트 덮어쓰기 |

> 단편 공격(Tiny Fragment / Fragment Overlap) 은 **DoS 분류 표에 함께 등장하지만, IP fragment offset 의 취약점을 이용해 패킷 필터링 장비를 우회하기 위한 기법** 이며 자원 고갈을 목적으로 하지 않는다. (2과목.txt line 604~611)

### 시험 포인트

- DoS 의 가용성 표적성 — 파괴 / 시스템 자원 소진 / 네트워크 자원 소진의 3 분류
- Tiny Fragment / Fragment Overlap 은 IP 단편 오프셋 조작이라는 공통점이 있으나 **DoS 가 아니고 우회 공격**

## 2. DoS 공격 유형

### 2-1. Ping of Death Attack

ping 명령을 통해 ICMP 패킷을 정상적인 크기보다 아주 크게 만들어 전송하면 MTU 에 의해 다수의 IP 단편화가 발생한다. 수신 측에서 단편화된 패킷을 재조합하는 과정에서 많은 부하 또는 재조합 버퍼 오버플로우가 발생한다.

| 항목 | 내용 |
|---|---|
| 공격 원리 | 큰 ICMP 패킷 → MTU 에 의해 다수 IP 단편 발생 → 재조합 부하 |
| 공격 탐지 | ICMP 패킷은 본래 분할하지 않으므로 분할이 일어난 ICMP 패킷을 공격으로 탐지 |
| 보안 패치 | 대부분 시스템은 Ping of Death 공격을 받을 때 ICMP 패킷을 무시함 |

> MTU(Maximum Transmission Unit) 는 물리적인 네트워크 프로토콜 프레임의 데이터부의 최대 크기를 의미한다. 이더넷의 MTU 는 IP 헤더 20 byte 를 제외하고 1480 byte. (2과목.txt line 566~567 ; [02 단원 §2-4](02.network-protocols-and-addressing.md))

### 2-2. Land Attack

출발지와 목적지 IP 가 같은 패킷을 만들어 공격 대상 시스템에 전송하여 수신자가 자기 자신에게 응답을 보내 무한 루프 상태에 빠지도록 한다.

| 항목 | 내용 |
|---|---|
| 공격 원리 | 출발지 IP = 목적지 IP → 수신자가 자기 자신에게 응답하며 시스템 자원 소모 |
| 보안 패치 | 대부분 시스템은 Land Attack 에 대한 보안 패치가 되어 있음 |
| 보안 장비 | 패킷 필터링 장비를 통해 출발지 IP 와 목적지 IP 가 같으면 차단하도록 설정 |

### 2-3. Smurf Attack

> 출발지 IP 를 희생자의 IP 로 위조한 후 증폭 네트워크로 ICMP Echo Request 를 브로드캐스트하여 다수의 ICMP Echo Reply 패킷이 희생자에게 전달되어 서비스 거부를 유발한다.

| 용어 | 의미 |
|---|---|
| **Directed Broadcast** | 원격지 호스트 ID 비트를 모두 1 로 설정하여 브로드캐스트 전송 |
| **증폭 네트워크** | 희생자에게 다수의 증폭된 ICMP Echo Reply 를 전송하는 네트워크 |

#### 대응 방안

| 위치 | 대응 |
|---|---|
| 보안 장비 | 동일한 ICMP Echo Reply 패킷이 다량으로 발생하면 해당 패킷 모두 차단 |
| 라우터 설정 | `no ip directed-broadcast` (증폭 네트워크로 사용되는 것을 방지) |
| 시스템 설정 | 브로드캐스트로 전송된 ICMP Echo Request 메시지에 대해 응답하지 않도록 설정 |

### 2-4. Teardrop / Bonk / Boink

IP 패킷의 재조합 과정에서 조작된 단편의 오프셋 정보로 인해 수신 시스템에서 오류나 부하가 발생하도록 만드는 공격. 공격자는 IP fragment offset 값을 서로 중첩되도록 조작하여 전송한다.

| 변형 | 단편 순서번호 조작 방식 |
|---|---|
| **Teardrop** | offset 을 서로 **중첩** 되도록 조작 |
| **Bonk** | 처음 패킷을 1 번으로 보낸 후 다음 패킷을 보낼 때 순서번호를 **모두 1 번으로** 조작 |
| **Boink** | 처음 1 번 → 다음 100 번 → 다음 200 번 등 정상적으로 보내다가 **중간에 비정상적인 순서번호** 로 전송 |

#### 대응 방안

| 항목 | 내용 |
|---|---|
| 보안 패치 | 운영체제의 보안패치를 모두 적용하여 OS 의 취약점을 해소 |

### 2-5. Tiny Fragment Attack — 작은 단편

| 단계 | 내용 |
|---|---|
| 1. 첫 번째 단편 | 목적지 포트 **없음** |
| 2. 두 번째 단편 | 목적지 포트 **있음** |
| 3. 재조합 시 | 공격자가 원하는 목적지 포트로 연결 |

패킷 필터링 장비를 우회하여 허용하지 않는 공격 대상 시스템의 서비스에 접근하기 위한 공격으로 헤더를 작게 쪼개서 첫 번째 단편에는 목적지 포트를 포함하지 않고, 원하는 목적지 포트가 있는 두 번째 단편을 통해 재조합 시 포트를 연결하도록 한다.

### 2-6. Fragment Overlap Attack — 단편 중첩

| 단계 | 내용 |
|---|---|
| 1. 첫 번째 단편 | **허용** 목적지 포트 |
| 2. 두 번째 단편 | **우회** 목적지 포트 (재조합 시 첫 번째 단편의 일부분을 덮어씀) |
| 3. 재조합 시 | TCP 헤더가 중첩되어 공격자가 원하는 목적지 포트로 연결 |

#### 단편 공격 공통 대응 방안

> 우회 공격 탐지 — IP 단편 우회 공격의 탐지가 가능한 패킷 필터링 장비를 적용. 단편화된 패킷을 재조합 후 검사하는 방화벽·침입탐지시스템을 사용한다. (2과목.txt line 614~615)

### 시험 포인트

- Ping of Death — ICMP 가 분할되었다는 사실 자체가 탐지 단서
- Land — 출발지 IP = 목적지 IP, 패킷 필터링으로 차단
- Smurf — 출발지 IP 위조 + Directed Broadcast + ICMP Echo Reply 증폭. `no ip directed-broadcast`
- Teardrop / Bonk / Boink — fragment offset / 순서번호 조작 (offset 중첩 / 모두 1 / 중간 비정상)
- Tiny Fragment / Fragment Overlap — IP 단편 오프셋 조작이지만 **DoS 가 아니라 우회 공격**

## 3. DDoS — 정의와 구조

### 3-1. 정의

> **분산 서비스 거부(DDoS, Distributed Denial of Service)** : 분산된 다수의 좀비 PC / 디바이스에 의해 공격대상 시스템의 서비스를 마비시키는 공격 형태. DoS 와의 차이는 단일 컴퓨터가 아닌 **분산된 다수의 컴퓨터·장치** 를 이용한다는 점이다.

### 3-2. 구성 요소

| 용어 | 정의 | 별칭 |
|---|---|---|
| **공격자** | C&C 서버에 공격 명령을 전달하는 해커의 컴퓨터 | Bot Master |
| **C&C 서버** (Command & Control) | 공격자로부터 직접 공격 명령을 전달받아 다수의 좀비 PC 에게 전달하는 시스템 | — |
| **좀비 PC** | C&C 서버로부터 전달받은 명령을 실행하여 공격대상에게 실제 공격을 수행 | Bot, Agent, Slave |
| **공격대상** | 공격의 대상이 되는 시스템 | Victim |
| **봇(Bot)** | 보안상 결함을 이용해 원격에서 해당 시스템을 제어할 수 있는 프로그램 | — |
| **봇넷(Botnet)** | 봇에 감염된 다수의 좀비 PC 로 구성된 네트워크 | — |

> 봇은 일반적으로 웜·바이러스·백도어·스파이웨어·루트킷 등 다양한 악성코드의 특성을 복합적으로 가지며, 정보 유출 / 스팸메일 발송 / DDoS 공격 / 트래픽 스니핑 / 키 로깅 등 다양한 형태의 공격을 수행한다. (2과목.txt line 637~639)

### 3-3. 공격 절차 5단계

| 단계 | 내용 |
|---|---|
| 1. C&C 서버 구축 | 공격자는 각 봇을 관리하고 명령을 내리는 C&C 서버를 구축 |
| 2. 봇 배포 | 불특정 다수의 PC 에 봇을 배포하여 감염을 시도 |
| 3. 봇 감염 | 사용자가 봇 프로그램을 실행하면 봇에 감염되어 봇넷의 일원으로 추가 |
| 4. 명령 전달 | 공격자가 C&C 서버에 명령을 내리면 C&C 서버는 봇에 명령을 전달 |
| 5. 공격 수행 | 봇은 명령에 따라 공격대상에게 다양한 공격을 수행 |

### 시험 포인트

- 봇 마스터 / C&C / 좀비 / 봇 / 봇넷 — 각 용어의 역할 구분
- 공격 절차 5단계 순서

## 4. DDoS 공격 도구 — 4세대 분류

| 공격 도구 | 설명 |
|---|---|
| **트리누(Trinoo)** | 많은 호스트로부터 통합된 **UDP Flood** 서비스거부 공격을 유발하는 도구. 몇 개의 서버와 많은 수의 클라이언트로 이루어짐 |
| **TFN** | 트리누와 유사한 분산 서비스 거부 도구. 많은 소스에서 하나 혹은 여러 개의 목표 시스템에 대해 공격을 수행. **UDP Flood / TCP SYN Flood / ICMP echo 요청 / ICMP 브로드캐스트** 공격이 가능 |
| **Stacheldraht** | 트리누와 TFN 을 참고하여 제작된 도구로 두 도구의 특성을 대부분 가짐. 마스터 시스템 및 자동 업데이트되는 에이전트 데몬과의 사이에 통신을 할 때 **암호화** 하는 기능이 추가됨 |
| **TFN2K** | TFN 의 발전된 형태. 통신에 특정 포트가 사용되지 않고 암호화되어 있으며 프로그램에 의해 **UDP·TCP·ICMP 가 복합적으로 사용** 되며 포트도 임의로 결정. 지정된 TCP 포트에 백도어를 실행시킬 수 있음 |

### 시험 포인트

- 트리누 = UDP Flood 단일
- TFN = UDP/TCP SYN/ICMP echo/ICMP 브로드캐스트
- Stacheldraht = 트리누 + TFN + **통신 암호화**
- TFN2K = 암호화 + UDP/TCP/ICMP **복합 사용** + 포트 임의 결정 + 백도어

## 5. 봇넷 명령 제어 방식 / 보안장비 우회 기법

### 5-1. 봇넷 명령 제어 방식

| 방식 | 설명 | 대표 봇넷 유형 |
|---|---|---|
| **중앙 집중형 명령/제어 방식** | 참여 멤버들이 C&C 서버와 연결되어 명령을 수행. C&C 서버가 탐지·차단되면 전체 봇넷이 중단될 가능성 | IRC 봇넷, HTTP 봇넷 (Rbot 등) |
| **분산형 명령/제어 방식** | 참여 멤버들이 모두 C&C 역할을 하여 그룹에 명령을 전파. 중앙 관리용 별도 도메인·C&C 서버 불필요. 탐지·차단 어려움 | P2P 봇넷 (Storm, Peacomm 등) |

> IRC(Internet Relay Chat) — 인터넷상에서 채팅을 할 수 있도록 고안된 프로토콜. 봇 마스터는 IRC 서버를 C&C 서버로 활용한다. (2과목.txt line 645)

### 5-2. 보안장비 우회 기법

봇넷 참여 좀비 PC 들이 C&C 서버에 접속하는 과정에서 C&C 서버 도메인이나 IP 주소가 알려지면 KISA 에서 C&C 서버 목록을 업데이트하므로[^kisa-sinkhole], 보안장비에 의해 탐지되지 않도록 하기 위한 다양한 우회 기법이 적용된다.

| 기법 | 설명 | 운용 |
|---|---|---|
| **Fast Flux** | 하나의 C&C 서버 도메인에 다수의 IP 주소를 할당하여 DNS 질의 시마다 지속적으로 IP 주소가 변경되도록 하는 기법 | 도메인 질의 — 질의 시마다 도메인의 IP 주소가 변경 |
| **DGA**(Domain Generation Algorithm) | 약속된 규칙에 따라 C&C 도메인을 지속적으로 동적 생성하여 도메인 기반의 탐지·차단을 우회 | 도메인 생성 — 생성될 수 있는 도메인 중 하나를 DNS 서버에 새롭게 등록 |
| **Domain Shadowing** | 알려진 합법적인 도메인의 서브 도메인을 C&C 서브 도메인으로 사용 | 도메인 등록 — 합법적인 도메인에 여러 서브 도메인 등록 (DBD 경유지로도 사용) |

> DGA 는 컨피커(Conficker) 웜에 의해 처음 알려졌으며 Locky(시스템 날짜 기반으로 도메인명 생성), WannaCry 랜섬웨어 등에서 사용되었다. Domain Shadowing 은 합법적인 도메인을 소유하고 있는 도메인 관리자의 개인정보를 탈취하여 도메인 소유자 몰래 많은 서브 도메인을 등록해 사용한다. (2과목.txt line 670~677)

### 시험 포인트

- 중앙집중형(IRC·HTTP) ↔ 단일 C&C 의존 / 분산형(P2P) ↔ 탐지 어려움
- Fast Flux — 동일 도메인, IP 만 변경
- DGA — 도메인 자체를 동적 생성
- Domain Shadowing — 합법 도메인의 **서브** 도메인 탈취

## 6. DDoS 공격 분류 — 대역폭 소진 vs 자원 소진

### 6-1. 분류표

| 구분 | 대역폭 소진 공격 | 서비스 / 자원 소진 공격 |
|---|---|---|
| 영향 계층 | 3 ~ 4 계층 (IP·ICMP·IGMP·UDP·TCP) | 7 계층 (HTTP·DNS·SMTP) |
| 영향 범위 | **회선 대역폭 고갈** — 동일 네트워크를 사용하는 모든 서비스에 접속 장애 | HTTP 서버 과다 접속으로 장애 → **공격 대상 시스템만 피해** |
| 공격 대상 | 네트워크 인프라 | 웹서버 / 정보보호 장비 |
| 대표 공격 | UDP/ICMP Flooding, DNS Query Flooding, TCP Flooding(SYN/SYN+ACK), IP Flooding(Land Attack, Tear Drop, HTTP Continuation) | HTTP GET Flooding(Get Flooding, CC Attack), HTTP Header/Option Spoofing(Slowloris, Pyloris), TCP Traffic Flooding(TCP session, SYN Flooding, TCP Slow Read), Other L7 Service Flooding(Hash DoS, Hulk DoS, FTP/SMTP Attack) |

### 6-2. UDP / ICMP Flooding 공격

다량의 UDP 또는 ICMP 패킷을 서버로 전송하여 서버가 보유한 네트워크 대역폭을 소진시키는 공격. **네트워크 대역폭 소진 공격** 에 해당.

| 단계 | 내용 |
|---|---|
| 1 | 공격자가 출발지 IP 가 위조된 다량의 UDP/ICMP 패킷을 대상서버로 전송 |
| 2 | 대상서버는 대량의 트래픽이 발생하여 서비스가 불가한 상태 |
| 3 | 대상서버 접속이 원활하지 못하여 정상 클라이언트는 서비스를 이용할 수 없음 |

### 6-3. TCP SYN Flooding 공격

SYN 패킷을 서버로 전송하여 서버의 TCP 연결 자원을 소진시키는 공격. **서버 / 서비스 자원 소진 공격** 에 해당.

| 단계 | 내용 |
|---|---|
| 1 | 공격자는 출발지 IP 를 도달 불가능한 IP 로 위조한 SYN 요청을 대상서버에 전송 |
| 2 | 대상서버는 SYN+ACK 를 보내고 연결설정이 완료될 때까지 **연결요청대기큐** 에 저장 |
| 3 | 연결요청대기큐가 꽉 차 더 이상 연결 요청을 받을 수 없는 상태 |
| 4 | 대상서버는 연결 요청을 받을 수 없어 정상 클라이언트는 서비스를 이용할 수 없음 |

> TCP 의 3-way Handshake 가 SYN → SYN+ACK → ACK 순으로 완성된다는 점 ([02 단원 §4-5](02.network-protocols-and-addressing.md)) 을 악용. ACK 가 오지 않아 `incomplete connection queue` 에 SYN_RECV 상태만 누적된다. (2과목.txt line 702~707)

#### TCP SYN Flooding 대응 방안

| 대응 | 명령 / 설정 | 설명 |
|---|---|---|
| **SYN Cookie 설정** | `sysctl -w net.ipv4.tcp_syncookies=1` | 서버가 적절한 ACK 를 받을 때까지 연결요청대기큐에 연결요청정보를 저장하지 않도록 함 (TCP 표준 명세의 mitigation[^rfc4987]) |
| **보안 장비** | `iptables -A INPUT -p tcp --dport 80 --syn -m connlimit --connlimit -above 5 -j DROP` | 동일 클라이언트의 연결 요청 임계치 설정으로 과도한 연결 요청 차단 |
| **First SYN Drop** 설정 | — | 클라이언트로부터 전송된 첫 번째 SYN 을 Drop 하고 재요청 패킷이 도착하는지 확인하여 출발지 IP 위조 여부 파악 |
| **연결요청대기큐 크기 증가** | `sysctl -w net.ipv4.tcp_max_syn_backlog = 1024` | 큐 크기를 늘려 임시 완화 (효과 제한적) |
| **대기 시간 단축** | — | SYN+ACK 에 대한 대기 시간을 줄임 |

> *원본 표기 보정 — `--connlimit -above`*: 통합자료 표기 그대로 보존했으나, 실제 iptables 문법은 `--connlimit-above 5` (하이픈 없이 한 옵션). 명령 입력 시 주의. (통합자료 line 2443)

### 6-4. HTTP GET Flooding 공격

동일한 주소에 대량의 GET 요청을 발생시켜 서버의 자원을 과도하게 사용하는 공격. **서버 / 서비스 자원 소진 공격** 에 해당.

| 단계 | 내용 |
|---|---|
| 1 | 공격자가 동일한 동적 컨텐츠에 대해 다량으로 HTTP GET 요청 |
| 2 | 대상서버는 해당 요청을 처리하기 위해 서버 자원을 과도하게 사용 |
| 3 | 클라이언트의 정상적인 요청을 처리하지 못함 |

> 패킷 캡처 도구 ngrep 으로 GET 요청 헤더를 ASCII 로 확인하거나 access log 분석으로 출발지 IP 별 요청 횟수를 집계할 수 있다.
>
> ```
> ngrep -qtW byline | grep GET
> cat access_log-20160214 | grep "GET /home/index.php" | awk '{print $1}' | sort | uniq -c | sort -rn
> ```
>
> `awk '{print $1}'` 첫 번째 컬럼 / `sort` 오름차순 / `uniq -c` 연속된 중복행 개수 / `sort -rn` 숫자형식 역순. (2과목.txt line 732~742)

### 6-5. Hulk DoS 공격

다량의 주소를 지속적으로 변경하여 GET 요청을 발생시켜 서버의 자원을 과도하게 사용하는 공격. **서버 / 서비스 자원 소진 공격** 에 해당.

| 단계 | 내용 |
|---|---|
| 1 | 공격자가 대상서버의 주소를 지속적으로 변경하면서 다량으로 GET 요청 |
| 2 | 특정 URL **임계치 기반** 의 장비는 서로 다른 주소로 인식하여 정상적인 차단이 불가능 |
| 3 | 대상서버는 해당 요청을 처리하기 위해 서버 자원을 과도하게 사용 |
| 4 | 클라이언트의 정상적인 요청을 처리하지 못함 |

### 6-6. Hash DoS 공격

해시테이블의 특성을 악용하여 조작된 대량의 파라미터를 POST 방식으로 서버에 전송하여 CPU 자원을 고갈시키는 공격. **서버 / 서비스 자원 소진 공격** 에 해당.

| 단계 | 내용 |
|---|---|
| 1 | 공격자는 조작된 많은 수의 파라미터를 POST 방식으로 대상서버로 전달 |
| 2 | 대상서버는 다수의 해시 충돌이 발생 |
| 3 | 정상적인 파라미터 조회 시 많은 CPU 자원을 소모 |

### 6-7. CC Attack — HTTP GET Flooding with Cache-Control

웹서버의 부하를 감소시키기 위해 캐싱서버를 운영하여 많은 요청을 받는 데이터를 웹서버가 아닌 캐싱서버를 통해 응답하도록 구축하지만, **캐시 옵션을 조작하여 캐싱서버의 기능을 무력화** 하여 웹서버의 자원을 소모시키는 공격. **서버 / 서비스 자원 소진 공격** 에 해당.

| 단계 | 내용 |
|---|---|
| 1 | 공격자는 HTTP 메시지의 캐시 옵션을 조작 |
| 2 | 캐싱서버가 아닌 웹서버가 직접 처리하도록 유도 |
| 3 | 캐싱서버의 기능을 무력화하고 웹서버의 자원을 소모 |

### 6-8. Slow 계열 공격

저대역폭 공격이라는 특징을 가진다 — 다량의 트래픽이 아니라 **연결 상태를 지속시키는 방식** 이기 때문에 연결 자원을 소진시킬 정도의 트래픽만 필요하다.

| 변형 | 동작 원리 | 공격 단계 요약 |
|---|---|---|
| **Slow HTTP Header DoS** (Slowloris) | 빈 라인(CRLF CRLF) 을 전달하지 않고 지속적으로 천천히 불필요한 헤더 필드 정보를 전달하여 웹서버가 헤더 정보 구분 불가 | 1) 공격자가 빈 라인 미전송 2) 대상서버는 요청 헤더부 모두 수신할 때까지 연결 상태 유지 3) 공격자는 연결 상태 지속 4) 연결 자원이 모두 소진되어 정상 요청 받을 수 없음 |
| **Slow HTTP POST DoS** (RUDY) | `Content-Length` 를 비정상적으로 크게 설정한 후 매우 소량의 데이터를 지속적으로 천천히 전송 | 1) Content-Length 비정상 설정 + 소량 데이터 천천히 전송 2) 대상서버는 명시된 크기까지 모두 수신하기 위해 연결 상태 유지 3) 공격자는 연결 상태 지속 4) 연결 자원이 모두 소진 |
| **Slow HTTP Read DoS** | TCP 윈도우 크기 및 데이터 처리율을 감소시킨 후 데이터를 송신하여 서버가 응답메시지를 전송하지 못하고 연결을 유지 | 1) HTTP 요청 후 window 크기를 0 으로 조작하여 지속 전달 2) 대상서버는 응답메시지를 전송하지 못하고 연결 지속 3) 공격자는 연결 상태 지속 4) 연결 자원이 모두 소진 |

> Slow HTTP Read DoS 의 동작 근거 — TCP 흐름 제어. 수신측 버퍼 여유 공간이 0 이 되면 `Window` 필드를 0 으로 설정한 **Zero Window Packet** 을 전달하고, 이를 수신한 송신측은 일정 시간 대기 후 **Zero Window Probe Packet** 을 보낸다. 공격자는 이를 악용해 응답을 무한 대기시킨다. (2과목.txt line 774~784)
>
> *원본 표기 보정 — `Slowris`*: 2과목.txt line 690 의 표기는 OCR 결함이며 실제 도구명은 **Slowloris**. Slow HTTP POST 는 **RUDY**(R-U-Dead-Yet) 라는 이름으로 알려져 있다. (2과목.txt line 748·797)

#### Slow 계열 공격 대응 방안

| 대응 | 명령 / 설정 | 설명 |
|---|---|---|
| **동시 연결 임계치** | `iptables -A INPUT -p tcp --dport 80 -m connlimit --connlimit-above 30 -j DROP` | 동일한 출발지 IP 에서 동시에 연결 가능한 연결 개수에 임계치 설정 |
| **연결 타임아웃 설정** | `/etc/httpd/conf/httpd.conf` 에서 `Timeout 120` | 데이터 송수신 없이 세션을 유지하는 시간 타임아웃 설정 — 완벽한 대응은 아님 |
| **읽기 타임아웃 설정** | `<IfModule reqtimeout_module> RequestReadTimeout header=5 body=10 </IfModule>` | 요청 헤더·바디 정보가 특정 시간 이내에 모두 수신되지 않으면 오류 응답 |

### 시험 포인트

- 대역폭 소진(L3·L4) ↔ 자원 소진(L7) 구분
- UDP/ICMP Flooding = 대역폭 / TCP SYN Flooding = 자원 소진(연결요청대기큐)
- HTTP GET Flooding = 동일 URL / Hulk = 동적 변경 / Hash = POST 파라미터 / CC = 캐시 옵션
- Slow Header(Slowloris) = 빈라인 미전송 / Slow POST(RUDY) = Content-Length 위조 / Slow Read = window 크기 0
- Slow 계열은 **저대역폭** 공격 — 트래픽 양이 작아도 연결 자원 소진

## 7. DDoS 대응 방안 (종합)

### 7-1. 예방 대책

| 항목 | 내용 |
|---|---|
| **DDoS 대응 서비스 가입** | ISP 나 클라우드 서비스에서 제공하는 DDoS 방어 서비스 이용. KISA **사이버대피소** 서비스 이용[^kisa-cyber] |
| **백업 서버 구축** | 서버 장애를 대비하여 중요 서버들은 서로 다른 회선에 이중화 구성 |
| **공격 대상 최소화** | 기업 내부용 서버는 외부에 노출되지 않도록 내부망으로 망 분리 조치 후 운영 |

### 7-2. 방어 대책 — 자체 방어 vs DDoS 대응 서비스

| 분류 | 항목 |
|---|---|
| **자체 방어 준비** | TCP 기반 DDoS 방어, 이상 트래픽 모니터링, 화이트리스트 방식의 방화벽 설정, 서버 분산을 통한 가용성 확보 |
| **DDoS 대응서비스 활용** | 전문 방어 서비스를 이용하는 프로세스를 수립하여 공격 발생 시 즉각 대응할 수 있도록 숙달[^nist-800-61] |

### 시험 포인트

- KISA 사이버대피소 — 무료 DDoS 대응 서비스 (중소기업 대상)
- 망 분리 / 이중화 / 화이트리스트 — 예방 + 자체 방어의 핵심
- 침해사고 발생 시 NIST SP 800-61 대응 절차에 준한 사전 프로세스 수립

## 8. DRDoS — Distributed Reflection DoS

### 8-1. 정의

> 공격자는 출발지 IP 를 공격 대상 IP 로 위조하여 다수의 **반사서버(Reflector)** 로 요청 정보를 전송. 공격대상은 반사서버로부터 다수의 응답을 받아 서비스 거부 상태가 되는 공격 유형.

### 8-2. 일반 DoS 와의 차이

| 차이 항목 | 설명 |
|---|---|
| **반사 서버** | 직접 공격을 수행하는 게 아니라 증폭 및 반사 공격에 활용되는 서비스 이용 |
| **근원지 파악** | 출발지 IP 를 희생자 IP 로 위조하고 다수의 반사서버를 경유 → 추적 어려움 |
| **트래픽 효율** | 요청 대비 응답이 크기 때문에 공격 트래픽 효율이 증가 |

### 8-3. 공격 유형

| 프로토콜 | 동작 |
|---|---|
| **TCP** | 연결 설정 과정의 취약점을 이용 — 위조된 출발지 주소의 SYN 패킷을 반사서버로 전달하면 SYN+ACK 응답 패킷이 공격 대상으로 향함. SYN+ACK 에 대한 ACK 가 없을 경우 TCP 동작 구조상 재전송을 다수 수행하므로 증폭이 발생 |
| **ICMP** | Echo Request / Echo Reply 를 이용 — 위조된 주소의 Echo Request 를 반사서버로 전달하여 Echo Reply 가 공격대상으로 향하도록 함 |
| **UDP** | UDP 프로토콜을 사용하는 서비스(DNS / NTP / SNMP / CHARGEN 등) 를 이용 — 응답 패킷이 공격대상으로 향하게 함 |

### 8-4. NTP DRDoS 공격

NTP 서버의 `monlist` 명령을 이용하여 대규모 증폭·반사 형태의 DDoS 공격을 유발시킬 수 있는 취약점[^ntp-monlist].

> NTP 는 네트워크를 통해 컴퓨터 시스템 간 시간 동기화를 위해 사용하는 인터넷 프로토콜로 **UDP 123 번 포트** 를 서비스 포트로 사용한다.[^rfc5905]

| 단계 | 내용 |
|---|---|
| 1 | 출발지 IP 주소를 희생자 서버의 IP 로 패킷을 변조 |
| 2 | 취약한 다수의 NTP 서버를 대상으로 `monlist` 명령의 다수 질의를 전송 |
| 3 | 질의를 수신한 NTP 서버는 다수의 접속 호스트 정보를 담은 증폭된 결과를 변조된 출발지 IP 로 응답 |
| 4 | 희생자 서버는 증폭된 다수의 응답 수신으로 네트워크 대역이 소진되어 서비스 거부 상태 |

#### NTP DRDoS 공격 대응 방안

| 대응 | 명령 / 설정 |
|---|---|
| 취약한 NTP 데몬 서버 버전 업그레이드 | `ntpd --version` 명령으로 확인 |
| `monlist` 기능 해제 | `/etc/ntp.conf` 파일에 `disable monlist` 추가 |
| 취약 여부 점검 | `ntpdc -n -c monlist [점검 대상 NTP 서버]` |
| 보안장비 차단 | `iptables -A -OUTPUT -p udp --sport 123 -m length --length 100: -j DROP` |

> *원본 표기 보정 — `iptables -A -OUTPUT`*: 통합자료 표기 그대로 보존했으나, 실제 iptables 문법은 `iptables -A OUTPUT` (체인 이름 앞에 `-` 없음). 명령 입력 시 주의. (통합자료 line 2570)

### 8-5. SSDP DRDoS 공격

SSDP(Simple Service Discovery Protocol) 프로토콜을 악용하여 대규모 증폭·반사 형태의 DDoS 공격을 실행[^ssdp-cisa].

> SSDP 는 네트워크상의 서비스나 정보를 검색하는 프로토콜로 프린터·스캐너 등 다양한 IoT 기기의 네트워크 탐색 용도로 사용되며 **UDP 1900 번 포트** 를 장비 검색 시 사용한다.

| 단계 | 내용 |
|---|---|
| 1 | M-SEARCH 요청 패킷의 출발지 IP 를 희생자의 IP 주소로 변조 |
| 2 | SSDP 지원 IoT 기기로 요청 패킷을 전송 |
| 3 | IoT 기기는 트래픽 증폭 반사 역할을 하여 전송받은 패킷에 대한 대량의 응답 패킷을 생성 |
| 4 | 증폭된 대량의 응답 트래픽으로 인해 희생자의 네트워크 장비 및 호스트에 부하 발생 → 서비스 거부 상태 |

#### SSDP DRDoS 공격 대응 방안

| 대응 | 내용 |
|---|---|
| 서비스 관리 | 업무에 불필요한 UDP 포트 차단 및 UDP 서비스 중지 |
| 보안장비 | 임계치 기반의 UDP 패킷 차단 정책 적용 |

### 8-6. DRDoS 공통 대응 방안

| 위치 | 기법 |
|---|---|
| **패킷 필터링** (ISP) | 위조된 패킷이 인입되지 않도록 ISP 가 직접 차단 — 주로 **Ingress 필터링** |
| **시스템 설정** | ICMP 등 서비스가 필요 없는 시스템의 경우 해당 프로토콜 차단 |

> 추가 — Ingress / Egress / Unicast RPF (2과목.txt line 828~835)
> - **Ingress 필터링** : 외부에서 내부 네트워크로 들어오는 패킷에 대해 외부에 존재할 수 없는 IP 를 필터링
> - **Egress 필터링** : 라우터를 이용하여 내부에서 외부 네트워크로 나가는 패킷에 대해 출발지 IP 가 내부에서 관리하는 IP 가 아니라면 위조된 주소로 판단하여 필터링
> - **Unicast RPF**(Reverse Path Forwarding) : 라우터로 유입된 패킷의 출발지 IP 에 대해 라우팅 테이블을 이용하여 유입된 인터페이스로 다시 전송되는지 여부를 체크. 유입된 인터페이스로 다시 전송되지 않는다면 위조된 주소로 판단하여 필터링[^urpf-bcp]

### 시험 포인트

- DRDoS = IP 스푸핑 + 다수의 반사서버 경유 + 증폭
- 반사서버는 직접 공격 도구가 아닌 **정상 응답 서비스** 를 악용한다는 점
- NTP `monlist` (UDP 123) / SSDP M-SEARCH (UDP 1900) — 대표 사례
- Ingress 필터링은 **ISP** 가 인입단에서 처리. Egress 는 라우터 출구단. Unicast RPF 는 라우팅 테이블 기반 역방향 검증.

## 9. DNS 싱크홀

### 9-1. 정의

> 악성 봇에 감염된 PC 가 공격자의 명령을 받기 위해 C&C 서버로 연결을 시도할 때, **C&C 서버 대신 싱크홀 서버로 우회** 시켜 조종 명령을 받지 않도록 해주는 서비스.[^kisa-sinkhole]

한국인터넷진흥원(KISA) 에서 국내 주요 ISP 업체 및 가입기관과의 협력을 통해 DNS 싱크홀을 운영하고 있다.

### 9-2. 동작 과정

| 단계 | 내용 |
|---|---|
| 1 | C&C 서버 도메인이나 IP 주소 등이 알려지면 이를 통해 KISA 에서 C&C 서버 목록 배포 |
| 2 | 알려진 C&C 서버 목록을 DNS 싱크홀 적용기관의 DNS 서버에 주기적으로 업데이트 |
| 3 | 좀비 PC 가 C&C 서버에 접속하는 과정에서 C&C 서버 도메인에 대한 질의를 수행 |
| 4 | DNS 는 좀비 PC 에게 **싱크홀 서버 IP 주소를 반환** |
| 5 | 좀비 PC 는 C&C 서버가 아닌 싱크홀 서버로 접속하여 명령으로부터 피해 방지 |

### 시험 포인트

- DNS 싱크홀 = DNS 응답을 조작하여 C&C 도메인 → 싱크홀 IP 로 우회
- KISA 가 C&C 도메인 블랙리스트를 운영
- 좀비 PC 의 명령 수신 차단이 핵심 (감염 자체를 치유하지는 않음)

## 단원 요약

1. **DoS 공격 유형**
   - Ping of Death — ICMP 가 본래 분할되지 않는다는 점이 탐지 단서
   - Land — 출발지 IP = 목적지 IP, 패킷 필터링 차단
   - Smurf — 출발지 IP 위조 + Directed Broadcast + ICMP Echo Reply 증폭. `no ip directed-broadcast`
   - Teardrop / Bonk / Boink — IP fragment offset / 순서번호 조작
   - Tiny Fragment / Fragment Overlap — IP 단편 조작이지만 **DoS 가 아닌 우회 공격**

2. **DDoS 구조** : 공격자 / C&C / 좀비 / 봇넷 / 공격대상. 절차 5단계(C&C 구축 → 봇 배포 → 감염 → 명령 전달 → 공격 수행).

3. **DDoS 도구** : 트리누(UDP) → TFN(UDP/TCP/ICMP) → Stacheldraht(+암호화) → TFN2K(+복합·임의 포트·백도어).

4. **봇넷 명령 제어** : 중앙집중형(IRC·HTTP) ↔ 분산형(P2P). **보안장비 우회 3종** — Fast Flux(IP 변경) / DGA(도메인 동적 생성) / Domain Shadowing(서브 도메인 탈취).

5. **DDoS 공격 분류** : 대역폭 소진(L3·L4) ↔ 자원 소진(L7).
   - 대역폭 — UDP/ICMP Flooding, DNS Query, Land, Tear Drop, HTTP Continuation
   - 자원 — TCP SYN, HTTP GET, Hulk, Hash, CC Attack, Slow 계열(Slowloris·RUDY·Slow Read)

6. **DDoS 대응** : 사이버대피소 / 백업 / 망 분리 / 화이트리스트 / SYN Cookie / 연결 임계치(connlimit) / Timeout / RequestReadTimeout.

7. **DRDoS** : IP 스푸핑 + 반사서버 + 증폭. NTP `monlist`(UDP 123) / SSDP M-SEARCH(UDP 1900). 대응 — Ingress(ISP) / Egress(라우터 출구) / Unicast RPF(역방향 검증).

8. **DNS 싱크홀** : C&C 도메인 질의를 싱크홀 IP 로 응답 → 좀비 PC 의 명령 수신 차단. KISA 운영.
