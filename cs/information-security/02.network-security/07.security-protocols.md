## 1. VPN — 가상 사설망

### 1-1. 정의

> **가상 사설망(VPN, Virtual Private Network)** : 인터넷과 같은 공중망을 이용하여 사설망이 요구하는 서비스를 제공할 수 있도록 구축한 망. **공중망 내에서 마치 단일 회사만 사용하는 전용선처럼** 사용하는 기술.

### 1-2. VPN 종류

| 프로토콜 | 계층 | 설계자 / 특징 |
|---|---|---|
| **PPTP** (Point-to-Point Tunneling Protocol) | 2 | 마이크로소프트사가 설계. PPP 의 Packet 을 IP Packet 으로 캡슐화. **하나의 터널에 하나의 연결만 지원** (1:1 통신) |
| **L2F** (Layer 2 Forwarding) | 2 | 시스코사에서 제안. 하나의 터널에 **여러 개의 연결** 지원 (다자간 통신). PPP 사용, 인증으로 RADIUS·TACACS+ 사용. home site 에서 주소 할당 |
| **L2TP** (Layer 2 Tunneling Protocol) | 2 | 마이크로소프트 + 시스코 합작. L2F 에 기반을 두고 PPTP 와의 호환성 고려. WAN 기술(X.25·ATM·Frame Relay) / IP·IPX·AppleTalk 지원 |
| **IPsec** (IP Security) | 3 | IETF 에 의해 제안되어 **VPN 구현에 가장 널리 사용**. IP 데이터그램의 인증·무결성·기밀성. IPv6 에서 기본 포함 |

> 2계층 VPN 프로토콜 = PPTP / L2F / L2TP. 3계층 VPN 프로토콜 = IPsec (가장 대표적). (2과목.txt line 1048~1049)

### 1-3. VPN 기능

> 터널링 프로토콜과 보안 과정을 거쳐 내부 기밀을 유지. 보안성이 우수하고 외부로부터 안전하도록 (1) 주소 및 라우터 체계의 비공개, (2) 데이터 암호화, (3) 사용자 인증, (4) 사용자 액세스 권한 및 제한 의 기능을 제공.

| 용어 | 의미 |
|---|---|
| **터널링** | 송신자와 수신자 사이의 전송로에 외부로부터의 침입을 막기 위한 파이프 |
| **페이로드** | 터널링되는 데이터를 페이로드라고 하며, 페이로드는 캡슐화되어 전송 |

### 1-4. VPN 구성 형태

| 형태 | 설명 |
|---|---|
| **Intranet VPN** | 기업 내부를 LAN 을 통해 연결하는 데 사용. 넓게는 지사까지 연결 |
| **Extranet VPN** | 고객사나 협력업체에게 Intranet 을 이용할 수 있도록 Intranet 을 확장한 개념 |
| **Remote Access VPN** | 재택 근무자나 원격 접속자는 무선 및 전화 접속을 이용하여 ISP 의 NAS 에 접속 |

### 1-5. MPLS VPN

> MPLS 통신 네트워크를 이용하여 VPN 을 제공하는 서비스. 다른 VPN 구조에 비하여 **서비스 도입과 운영 관리가 간단하고 편리**.

### 시험 포인트

- 2계층 VPN(PPTP·L2F·L2TP) ↔ 3계층 VPN(IPsec)
- PPTP = 1:1, L2F·L2TP = 다자간, IPsec = IPv6 기본
- VPN 기능 4종 — 주소·라우터 비공개 / 암호화 / 인증 / 액세스 제한
- 구성 3종 — Intranet(내부) / Extranet(협력사) / Remote Access(재택)

## 2. IPsec

### 2-1. 정의

> IETF 에서 IP 보안을 위하여 개방형 구조로 설계한 표준으로, IP 계층 보안에 대해서 안정적이고 표준화된 기초를 제공.[^rfc4301] 네트워크 계층(3계층) 에서 제공되며, IPv6 에서는 IPsec 을 기본적으로 포함한다.

### 2-2. IPsec 보안 서비스 (6종)

| 서비스 | 설명 | 비고 |
|---|---|---|
| **기밀성** (Confidentiality) | 메시지가 도청되어도 그 내용은 알 수 없음 | 대칭 암호 / **AH 미지원, ESP 전용** |
| **비연결형 무결성** (Connectionless Integrity) | 메시지가 위·변조되지 않았음 | 메시지 인증 코드(MAC), IP 패킷별 |
| **송신처 인증** (Data Origin Authentication) | 메시지가 올바른 송신처로부터 온 것임 | 메시지 인증 코드 |
| **재전송 공격 방지** (Protection Against Replays) | IP 패킷별로 순서번호를 전송하여 재전송 공격 방지 | SA 에서 순서번호 유지·검증 |
| **접근 제어** (Access Control) | 송수신 IP 패킷에 대한 시스템 접근을 제어 | 보안정책(SP) — Bypass / Discard / Protect |
| **제한된 트래픽 흐름의 기밀성** | 원본 IP 헤더는 암호화되어 있기 때문에 기밀성을 보장 | **ESP 터널 모드** 에서 제공 |

### 2-3. IPsec 모드 — 전송 모드 vs 터널 모드

| 모드 | 보호 범위 | 패킷 구조 |
|---|---|---|
| **전송 모드** | 종단 노드 구간에서 IP 패킷의 **페이로드** 를 보호 (IP-H 로 식별) | `[ IP-H ][ IPsec-H ][ IP Payload ][ IPsec-T ]` |
| **터널 모드** | 터널 / 보안 게이트웨이 구간에서 IP 패킷 **전체** 를 보호 (New IP-H 로 식별) | `[ New IP-H ][ IPsec-H ][ IP-H ][ IP Payload ][ IPsec-T ]` |

> 터널 모드 적용 시나리오 — (1) **터널·게이트웨이 간 보호** : 물리적으로 떨어진 위치의 본·지점 사이에 다수의 사용자 호스트와 서버로 구성된 VPN 환경. (2) **종단노드 - 터널·게이트웨이 구간 보호** : 외부 사용자 호스트와 본·지점 사이의 VPN 환경. (2과목.txt line 1080~1082)

### 2-4. AH 프로토콜 — Authentication Header

> 메시지 인증 코드를 이용하여 **인증과 송신처 인증** 을 제공. **기밀성은 제공하지 않음**.[^rfc4302]

| 모드 | 패킷 구조 | 설명 |
|---|---|---|
| AH 전송 모드 | `[ IP-H ][ AH-H ][ IP Payload ]` | IP 헤더의 전송 중 변경 가능한 필드를 제외한 IP 패킷 전체를 인증 |
| AH 터널 모드 | `[ New IP-H ][ AH-H ][ IP-H ][ IP Payload ]` | New IP 헤더의 전송 중 변경 가능한 필드를 제외한 New IP 패킷 전체를 인증 |

#### AH 주요 헤더 필드

| 필드 | 의미 |
|---|---|
| **SPI** (Security Parameter Index) | 현재 연결에 대한 보안연관 식별자 |
| 순서번호 | 재전송 공격 방지용. 일련번호가 항상 1 씩 증가 |
| 인증 데이터 (Integrity Check Value, ICV) | IP 헤더의 변경 가능한 필드를 제외한 IP 패킷 전체에 대한 MAC 값 |

> 변경 가능한 IP 헤더 필드 — TTL, Header Checksum, NAT 환경에서의 source IP. 구형 IPsec 을 사용하는 경우 NAT 환경에서 source IP 가 변경되는데 이를 제외하지 않고 인증 데이터를 계산하여 장애가 발생하는 경우가 있다. ESP 프로토콜을 사용하면 IP 헤더가 인증 대상에서 제외되므로 정상 통신 가능. (2과목.txt line 1086~1090)

### 2-5. ESP 프로토콜 — Encapsulating Security Payload

> 메시지 인증 코드와 암호화를 이용하여 **인증·송신처 인증·기밀성** 을 제공. 인증과 암호화를 선택적으로 적용할 수 있음. **IP 헤더를 인증하지 않음**.[^rfc4303]

| 모드 | 패킷 구조 | 설명 |
|---|---|---|
| ESP 전송 모드 | `[ IP-H ][ ESP-H ][ IP Payload ][ ESP-T ][ ESP Auth ]` | IP 페이로드와 ESP 트레일러를 암호화 / 암호화된 데이터와 ESP 헤더를 인증 |
| ESP 터널 모드 | `[ New IP-H ][ ESP-H ][ IP-H ][ IP Payload ][ ESP-T ][ ESP Auth ]` | 원본 IP 패킷 전체와 ESP 트레일러를 암호화 / 암호화된 데이터와 ESP 헤더를 인증 |

#### ESP 주요 헤더 필드

| 필드 | 의미 |
|---|---|
| SPI | 보안연관 식별자 |
| 일련번호 | 재전송 방지 |
| ESP 트레일러 | 블록 암호를 위한 패딩 정보, 전송하는 페이로드의 프로토콜 타입 정보 |
| ESP Auth | 인증 데이터 정보 저장 |

### 2-6. IKE — 인터넷 키 교환 프로토콜

> 보안 관련 설정을 생성·협상·관리하는 프로토콜로 **500/udp 포트** 를 사용.[^rfc7296] IPsec 에서 양 당사자 간의 논리적인 연결정보 및 보안 설정을 보안연관(SA) 이라고 한다.

| 단계 | 협상 내용 | 결과 SA | 모드 |
|---|---|---|---|
| **IKE 1단계** | IKE 2단계에서 사용할 메시지를 어떻게 보호할 것인지를 협상. IKE 용 마스터키 생성 | **IKE SA** (양방향성) | Main Mode (세션 ID 암호화 — 보안성 ↑) / Aggressive Mode (세션 ID 비암호화 — 보안성 ↓) |
| **IKE 2단계** | 실질적인 데이터를 어떤 방식으로 보호할 것인지를 협상. 1단계 키로 메시지 보호 | **IPsec SA** (단방향성, 송신용·수신용 각각) | Quick Mode |

### 2-7. SA · SAD · SP · SPD

| 용어 | 정의 |
|---|---|
| **SA** (Security Association, 보안연관) | 둘 사이에 논리적인 연결 상태를 유지하는 동안 적용할 보안 설정 정보. **일방향성** (A→B 와 B→A 각각 필요 → 2개 SA) |
| **SAD** (SA Database, 보안연관 데이터베이스) | 여러 보안연관을 저장해 놓은 데이터베이스. 포함 필드 — SPI, Sequence Number Counter, Anti-Replay Window, AH/ESP, Lifetime, Mode, Path MTU |
| **SP** (Security Policy, 보안정책) | 패킷을 송수신할 때 적용할 보안의 종류를 정의. 정책 종류 — IP 패킷 허용(Bypass) / IP 패킷 거부(Discard) / IP 패킷 보호(Protect → SAD 검색) |
| **SPD** (SP Database, 보안정책 데이터베이스) | 보안 정책들을 저장해놓은 데이터베이스 |

### 2-8. IPsec 패킷 송수신 절차

#### 송신 절차

| 단계 | 내용 |
|---|---|
| 1 | 전송할 패킷에 대해 SPD 검색 |
| 2 | 일치하는 엔트리가 없으면 패킷 삭제 |
| 3 | 일치하는 엔트리가 있으면 첫 번째 엔트리의 정책에 따라 처리 — Discard(폐기) / Bypass(송신) / Protect(SAD 검색) |
| 4 | Protect 인 경우 SAD 검색. 일치하면 IPsec 처리 후 전송. 일치하지 않으면 IKE 를 수행하여 SA 생성 |

#### 수신 절차

| 단계 | 내용 |
|---|---|
| 1 | IP 프로토콜의 protocol 필드 조사 → 보호되지 않은 IP 패킷인지 IPsec 패킷인지 판별 |
| 2 | 보호되지 않은 IP 패킷 → SPD 검색. 첫 번째 일치 엔트리가 허용이면 상위 계층으로 전달, 폐기·보호이면 삭제 |
| 3 | IPsec 패킷 → SAD 검색. 일치하지 않으면 삭제, 일치하면 IPsec 처리 후 상위 계층으로 전달 |

### 시험 포인트

- IPsec 보안 서비스 6종 중 **기밀성은 ESP 만 제공** (AH 는 미제공)
- 전송 모드(IP-H 식별, 페이로드 보호) ↔ 터널 모드(New IP-H 식별, 패킷 전체 보호)
- AH = 인증만 / ESP = 인증 + 기밀성
- AH 는 IP 헤더 인증 → NAT 환경에서 source IP 변경으로 장애. ESP 는 NAT 통과 가능
- IKE = 500/udp. 1단계(IKE SA, 양방향) / 2단계(IPsec SA, 단방향)
- SA 일방향성 → 양방향 통신은 SA 2개 필요. SP 정책 3종 — Bypass / Discard / Protect

## 3. SSL / TLS

### 3-1. 정의

> 클라이언트 / 서버 환경에서 TCP 기반의 어플리케이션에 대한 종단 간 보안 서비스를 제공하기 위해 만들어진 **전송계층 보안 프로토콜**. 각 애플리케이션 프로토콜이 SSL 을 이용할 경우 이를 구분하기 위해 고유한 well-known 포트를 할당한다 (HTTPS=443, IMAPS=993, POP3S=995 등).

### 3-2. 보안 서비스

| 서비스 | 메커니즘 |
|---|---|
| **기밀성** | 송수신 메시지 암호화 (대칭키 암호) |
| **무결성** | 송수신 메시지의 위·변조 확인 (메시지 인증 코드) |
| **인증** | 서버 / 클라이언트 간 상호 인증 (공개키 인증서) |

### 3-3. SSL/TLS 버전

| 버전 | 발표 연도 | 특징 |
|---|---|---|
| SSL 1.0 | — | 첫 번째 버전 (다양한 보안 취약점 존재, **공개되지 않음**) |
| SSL 2.0 | 1995 년 | 여전히 많은 취약점이 존재 |
| SSL 3.0 | 1996 년 | 대표적으로 **POODLE 및 DROWN** 취약점이 알려짐 |
| TLS 1.0 | 1999 년 | 대표적으로 **BEAST 및 CRIME** 취약점이 알려짐 |
| TLS 1.1 | 2006 년 | 일부 취약점이 존재 |
| TLS 1.2 | 2008 년 | 효율적인 Handshake 를 가지고 있음[^rfc5246] |
| TLS 1.3 | 2018 년 | 트래픽 정보의 기밀성을 유지하는 **PFS** 를 제공[^rfc8446] |

> 통합자료 표기 보존 — TLS 1.2 의 "효율적 Handshake" 는 TLS 1.0/1.1 대비 cipher suite 단순화·확장 옵션 추가에 따른 상대적 개선을 가리키는 표현이다. 더 단축된 1-RTT/0-RTT Handshake 는 TLS 1.3 에서 도입되었다.[^rfc8446]

### 3-4. 세부 프로토콜

> SSL/TLS 는 **상위 계층** (Handshake / Change Cipher Spec / Alert / Application Data) 과 **하위 계층** (Record) 의 2 계층으로 이루어진 프로토콜이다.

| 계층 | 프로토콜 | 기능 |
|---|---|---|
| **상위 계층** | **Handshake** | 종단 간에 보안 파라미터를 협상 |
| | **Change Cipher Spec** | 종단 간에 협상된 보안 파라미터를 이후부터 적용 / 변경함을 알림 |
| | **Alert** | SSL/TLS 통신 과정에서 발생하는 오류를 통보 |
| | **Application Data** | 어플리케이션 계층의 데이터를 전달 |
| **하위 계층** | **Record** | 상위 계층에서 적용·변경된 보안 파라미터를 이용하여 실제 암호화/복호화·무결성 보호·압축/압축해제 등의 기능 제공 |

#### Record 프로토콜 동작 방식

| 방향 | 단계 |
|---|---|
| 송신 시 | 단편화 → 압축 → MAC 추가 → 암호화 → Record 헤더 추가 |
| 수신 시 | Record 헤더 제거 → 복호화 → MAC 확인 → 압축 해제 → 재조합 |

### 3-5. 상태 유지 프로토콜

> 완전 협상을 통해 **세션** 을 생성하고, 이 세션 정보를 공유하는 다수의 **연결** 을 단축 협상을 통해 생성한다.

| 상태 정보 | 설명 |
|---|---|
| **세션 상태 정보** | 양 종단 간에 완전 협상을 통해 생성되는 상태 정보. 다수의 연결에 의해 사용되는 보안 파라미터(대칭 암호 알고리즘, HMAC 용 해시 알고리즘) |
| **연결 상태 정보** | 실제 통신이 이루어지는 단위로 단축 협상을 통해 생성되는 상태 정보 (암호키, 인증키) |
| **사전 마스터 비밀** (premaster secret) | 완전 협상을 통해 생성된 값 |
| **마스터 비밀** | 사전 마스터 비밀, 클라이언트 / 서버의 난수 값을 해시화한 값 |
| **키 블럭** | 단축 협상을 통해 주고받은 클라이언트 / 서버의 난수값과 마스터 비밀을 해시화한 값 |

### 3-6. 완전 협상 (Full Handshake) — 13 단계

| 단계 | 메시지 | 설명 |
|---|---|---|
| 1 | **Client Hello** | 지원 가능한 SSL/TLS 버전·암호도구목록·압축 방식 + client random(32bit 난수) + 세션 ID(완전협상 시 빈 값) |
| 2 | **Server Hello** | 사용할 SSL/TLS 버전·암호도구·압축 방식 + server random(32bit 난수) + 세션 ID |
| 3 | **Server Certificate** | (필요시) 서버 인증서 목록을 클라이언트에 전달 |
| 4 | **Server Key Exchange** | (필요시) 키 교환에 필요한 정보를 클라이언트에 전달 |
| 5 | **Certificate Request** | (필요시) 클라이언트 인증을 위한 인증서를 요청 |
| 6 | **Server Hello Done** | Server Hello 과정 종료를 클라이언트에 알림 |
| 7 | **Client Certificate** | (필요시) 클라이언트 인증서 목록을 서버에 전달 (Certificate Request 응답) |
| 8 | **Client Key Exchange** | 사전 마스터 비밀을 생성하여 서버에 전달. RSA(난수 premaster 후 서버 공개키로 암호화) / Diffie-Hellman(클라이언트 DH 공개키 전달, 양측이 DH 연산으로 공통 premaster 생성) |
| 9 | **Certificate Verify** | (필요시) 보낸 인증서에 대한 개인키를 가지고 있음을 증명 |
| 10 | **Change Cipher Spec** | 협상한 암호 명세를 이후부터 적용·변경함을 알림 (클라이언트 → 서버) |
| 11 | **Finished** | 협상 완료를 서버에 알림 |
| 12 | **Change Cipher Spec** | 협상한 암호 명세를 이후부터 적용·변경함을 알림 (서버 → 클라이언트) |
| 13 | **Finished** | 협상 완료를 클라이언트에 알림 |

### 3-7. 단축 협상 (Abbreviated Handshake)

> 이전에 이미 완전 협상 과정을 거친 SSL/TLS 세션을 다시 사용할 때 사용되는 협상 방식.

| 단계 | 메시지 | 설명 |
|---|---|---|
| 1 | **Client Hello** | 사용할 session id 와 client random 을 서버에게 전달 |
| 2 | **Server Hello** | server random 을 클라이언트에게 전달 |

### 3-8. PFS — 순방향 비밀성 (Perfect Forward Secrecy)

#### 서버 개인키 노출 시 문제점

| 단계 | 내용 |
|---|---|
| 1 | 서버 공개키와 개인키를 이용하여 키 교환을 수행 |
| 2 | 공격자는 중간자 공격을 통해 트래픽을 가로챔 |
| 3 | 서버 개인키를 이용해 세션키 / 비밀키 및 송수신 데이터를 복호화 |

#### PFS 정의

> 클라이언트 ↔ 서버 간에 키 교환에 사용되는 서버 개인키가 노출되어도 **이전 트래픽의 세션키 / 비밀키 기밀성은 그대로 유지** 되어 통신 내용을 알 수 없는 암호학적 성질.

> PFS 구현 — 키 교환 시마다 클라이언트 / 서버가 새로운 Diffie-Hellman 개인키를 생성하는 **임시 디피-헬만 키 교환** 을 통해 클라이언트 / 서버 간 공통의 비밀값을 생성하고, 서버 개인키는 서버 Diffie-Hellman 파라미터를 인증하는 목적으로만 사용한다. 따라서 서버 개인키가 노출되어도 통신 내용을 알 수 없다. (2과목.txt line 1218~1222)
>
> 단, RSA 에 비해 처리 속도가 느리므로 성능상의 이유로 **DHE**(Ephemeral Diffie-Hellman), **ECDHE**(Elliptic Curve DHE) 관련 cipher suite 를 비활성화하거나, 다양한 DHE/ECDHE cipher suite 을 모두 지원하지 않는 경우가 있다 (브라우저 호환을 위해 RSA 방식의 키 교환과 함께 사용).

### 시험 포인트

- SSL/TLS 는 TCP 위·응용 아래의 전송계층 보안 프로토콜
- 상위 4종(Handshake·Change Cipher Spec·Alert·Application Data) ↔ 하위 1종(Record)
- Record 송신 = 단편화 → 압축 → MAC → 암호화 → 헤더
- 완전 협상 13단계. 단축 협상 = Client Hello + Server Hello (2단계)
- 사전 마스터 비밀 → 마스터 비밀 → 키 블럭 (단계별 파생)
- PFS = 서버 개인키 노출되어도 이전 트래픽 기밀성 유지. **DHE / ECDHE** 가 핵심 cipher suite
- TLS 1.3 = PFS 필수

## 4. SSH 포트 포워딩

> SSH 클라이언트가 SSH 서버에 접속하여 만든 연결을 다른 어플리케이션이 이용하여 통신할 수 있는 기술.[^rfc4254]

### 4-1. 로컬 포트 포워딩 (Local Port Forwarding)

> 웹 브라우저가 접속할 **리스닝 포트를 SSH 클라이언트 호스트** 에 생성하여 포워딩하는 방식.

| 단계 | 내용 |
|---|---|
| 1 | SSH 클라이언트가 SSH 서버에 접속하여 연결 생성 |
| 2 | 웹 브라우저는 SSH 클라이언트 호스트에 생성된 리스닝 포트에 접속 |
| 3 | SSH 연결을 통해 최종 목적지인 어플리케이션 서버와 통신 |

### 4-2. 리모트 포트 포워딩 (Remote Port Forwarding)

> 웹 브라우저가 접속할 **리스닝 포트를 SSH 서버 호스트** 에 생성하여 포워딩하는 방식.

| 단계 | 내용 |
|---|---|
| 1 | SSH 클라이언트가 SSH 서버에 접속하여 연결 생성 |
| 2 | 웹 브라우저는 SSH 서버 호스트에 생성된 리스닝 포트에 접속 |
| 3 | SSH 연결을 통해 최종 목적지인 어플리케이션 서버와 통신 |

### 시험 포인트

- 로컬 포트 포워딩 = 리스닝 포트가 **클라이언트** 호스트
- 리모트 포트 포워딩 = 리스닝 포트가 **서버** 호스트
- 두 방식 모두 SSH 연결을 통해 최종 어플리케이션 서버와 통신

## 5. SSL / TLS 취약점

> SSL/TLS 프로토콜의 보안 결함을 이용한 데이터 탈취 또는 중간자 공격으로 HTTPS 기반 웹 서비스에서 민감한 데이터가 유출될 수 있는 취약점.

### 5-1. HeartBleed — CVE-2014-0160

> 통신 구간 암호화를 위해 많이 사용하는 **OpenSSL 라이브러리의 하트비트 확장 모듈** 의 버그로 인하여 발생한 취약점.[^cve-heartbleed]
> - 하트비트 확장 모듈에서 클라이언트 하트비트 요청 메시지를 처리할 때 데이터 검증을 수행하지 않음
> - 시스템 메모리에 저장된 **64KB 크기의 데이터** 를 외부에서 아무런 제한 없이 탈취할 수 있음

| 대응 | 내용 |
|---|---|
| 버전관리 | 취약점이 존재하지 않는 OpenSSL 버전으로 업데이트 |
| 보안장비 | 취약점 공격 탐지 및 차단 룰 / 패턴 적용 |
| 서비스 관리 | 인증서 재발급, 사용자들의 비밀번호 재설정 유도 |

### 5-2. FREAK — CVE-2015-0204

> SSL 을 통해 강제로 **취약한 RSA 로 다운그레이드** 시킬 수 있는 취약점.[^cve-freak]
> - OpenSSL `s3_clnt.c` 의 `ssl3_get_key_exchange` 함수에서 발생
> - MITM 공격을 통해 **512 비트 RSA** 로 다운그레이드시켜 정보를 유출

| 대응 | 내용 |
|---|---|
| 버전관리 | 영향을 받는 OpenSSL 버전 사용 시 최신 버전으로 업그레이드 |
| 사용자 관리 | 취약점에 영향을 받는 OS 및 브라우저는 업그레이드 |

### 5-3. LogJam — CVE-2015-4000

> 중간자 공격을 통해 사용자와 웹 또는 이메일 서버 간의 TLS 통신을 다운그레이드시킬 수 있는 취약점.[^cve-logjam]
> - TLS 프로토콜의 취약점으로 공격자가 임시 Diffie-Hellman 키 교환을 사용
> - TLS 연결을 **512 비트 수출 버전 암호화** 로 다운그레이드할 수 있음

| 대응 | 내용 |
|---|---|
| 사용자 관리 | 클라이언트는 최신의 브라우저를 사용 |
| Export 키 제한 | Export 용 cipher suite 을 사용하지 않음 |
| 안전한 cipher suite 사용 | 2048bit 의 디피-헬만 그룹을 생성하거나 이러한 위험이 없는 타원 곡선 디피-헬만(ECDHE) 을 사용 |

### 5-4. POODLE — CVE-2014-3566

> SSL/TLS 협상 시 **버전 다운그레이드 공격** 을 통해 **SSLv3.0** 을 사용하도록 강제한 후 MITM 공격을 통해 암호화되어 송수신되는 쿠키 정보나 데이터를 추출.[^cve-poodle]
> - SSLv3.0 의 블록 암호화 기법인 **CBC 모드** 를 사용하는 경우 발생하는 패딩된 암호 블록이 MAC 에 의해 보호되지 않는 취약점

#### POODLE 다운그레이드 시나리오

| 단계 | 내용 |
|---|---|
| 1 | 클라이언트에서 TLS 1.2 로 서버 연결 요청 → 공격자 연결 거부 |
| 2 | 클라이언트에서 TLS 1.1 로 서버 연결 요청 → 공격자 연결 거부 |
| 3 | 클라이언트에서 TLS 1.0 로 서버 연결 요청 → 공격자 연결 거부 |
| 4 | 클라이언트에서 SSL 3.0 로 서버 연결 요청 → 공격자는 서버 연결을 허용한 후 스니핑 수행 |

| 대응 | 내용 |
|---|---|
| SSL 설정 | SSLv3.0 을 사용하지 않도록 웹서버 SSL 설정 |
| 버전 관리 | OpenSSL 을 최신 버전으로 업그레이드 |

### 5-5. DROWN — CVE-2016-0800

> **SSLv2.0** 을 사용하는 서버에 악성 패킷을 보내 인증서 키값을 알아내고 키값을 이용해 암호화된 통신 내용을 복호화해 주요 정보를 탈취.[^cve-drown]
> - SSLv2.0 취약점을 악용한 **교차 프로토콜 공격** (Cross-protocol attack)

| 대응 | 내용 |
|---|---|
| SSL 설정 | SSLv2.0 을 사용하지 않도록 웹서버 SSL 설정 |
| 버전 관리 | OpenSSL 을 최신 버전으로 업그레이드 |

### 5-6. SSL/TLS 취약점 정리표

| 취약점 | CVE | 표적 | 핵심 동작 |
|---|---|---|---|
| **HeartBleed** | CVE-2014-0160 | OpenSSL 하트비트 확장 | 64KB 메모리 무제한 탈취 |
| **FREAK** | CVE-2015-0204 | OpenSSL `ssl3_get_key_exchange` | 512 비트 RSA 다운그레이드 (MITM) |
| **LogJam** | CVE-2015-4000 | TLS DH 키 교환 | 512 비트 Export DH 다운그레이드 |
| **POODLE** | CVE-2014-3566 | SSLv3.0 CBC 패딩 | 버전 다운그레이드 → SSLv3.0 강제 → 패딩 오라클 |
| **DROWN** | CVE-2016-0800 | SSLv2.0 | 교차 프로토콜 공격으로 TLS 통신 복호화 |

### 시험 포인트

- HeartBleed = OpenSSL 하트비트 모듈, 64KB 메모리 누출
- FREAK / LogJam = 둘 다 다운그레이드 공격이지만 FREAK = RSA / LogJam = DH
- POODLE = SSLv3.0 CBC 패딩 / DROWN = SSLv2.0 교차 프로토콜
- 공통 대응 = OpenSSL 버전 업그레이드 + 약한 암호 비활성화

   - 6 보안 서비스 — 기밀성(ESP만) / 무결성 / 송신처 인증 / 재전송 방지 / 접근 제어 / 트래픽 흐름 기밀성(ESP 터널)
   - 모드 — 전송(IP-H 식별) / 터널(New IP-H 식별)
   - AH = 인증만 / ESP = 인증 + 기밀성. AH 는 IP 헤더 인증으로 NAT 비호환
   - IKE = 500/udp. 1단계(Main/Aggressive, IKE SA, 양방향) / 2단계(Quick, IPsec SA, 단방향)
   - SA(일방향, 2개 필요) / SAD / SP(Bypass·Discard·Protect) / SPD

3. **SSL/TLS**
   - 버전 — SSL 1.0 → 2.0(1995) → 3.0(1996) → TLS 1.0(1999) → 1.1(2006) → 1.2(2008) → 1.3(2018, PFS)
   - 세부 = 상위(Handshake·Change Cipher Spec·Alert·Application Data) + 하위(Record)
   - Record = 단편화 → 압축 → MAC → 암호화 → 헤더
   - 완전 협상 13단계 / 단축 협상 2단계
   - PFS = DHE / ECDHE 임시 키 교환

4. **SSH 포트 포워딩** : 로컬(클라이언트 호스트가 리스너) ↔ 리모트(서버 호스트가 리스너).

5. **SSL/TLS 취약점 5종**
   - HeartBleed(2014, OpenSSL 하트비트, 메모리 64KB 탈취)
   - FREAK(2015, RSA 다운그레이드)
   - LogJam(2015, DH 다운그레이드)
   - POODLE(2014, SSLv3.0 CBC)
   - DROWN(2016, SSLv2.0 교차 프로토콜)
