## 1. VLAN — Virtual LAN

### 1-1. 정의

> 물리적인 선이 아닌 논리적으로 **브로드캐스트 도메인** 을 나누기 위해 사용하는 기술. 관리자가 서로 다른 논리적 그룹에 대하여 서로 다른 보안 정책을 사용할 수 있게 함.

### 1-2. VLAN 종류

| 종류 | 동작 |
|---|---|
| **Port 기반 VLAN** | 스위치 포트를 각 VLAN 에 할당하여 같은 VLAN 에 속한 호스트들 간에만 통신 |
| **MAC 기반 VLAN** | 각 호스트의 MAC 을 VLAN 에 등록하여 같은 VLAN 에 속한 MAC 들 간에만 통신 |
| **Network 기반 VLAN** | 네트워크 주소별로 VLAN 을 구성하여 같은 네트워크에 속한 호스트들 간에만 통신 |
| **Protocol 기반 VLAN** | 같은 통신 프로토콜을 가진 호스트들 간에만 통신 |

## 2. 라우터 보안

### 2-1. 라우터 패스워드 설정

> 재부팅이나 라우팅 등 라우터에서의 모든 명령어에 대해 수행이 가능한 **관리자 모드** 로 변경할 때 사용하는 패스워드를 설정.

| 명령 | 동작 |
|---|---|
| `enable password asd123` | 패스워드를 평문으로 저장 |
| `enable password-encryption` | 평문 패스워드를 암호화 (역함수 존재) |
| `enable secret asd123` | 패스워드를 암호화하여 저장 (**일방향 함수**) |

### 2-2. ACL — Access Control List

| 종류 | 설명 |
|---|---|
| **Standard access-list** | 패킷의 소스 IP 만으로 패킷을 허용하거나 차단. `access-list [acl 번호]` — number 1~99 |
| **Extended access-list** | 소스 IP 뿐 아니라 목적지 IP, 포트, 프로토콜 등을 이용하여 차단. `access-list [acl 번호]` — number 100~199 |

### 2-3. Filtering 유형

| 유형 | 동작 |
|---|---|
| **Ingress Filtering** | 외부에서 내부 네트워크로 유입되는 패킷을 체크하여 허용/차단 — 패킷의 방향은 **내부로 들어오는 패킷의 출발지 IP** |
| **Egress Filtering** | 내부에서 외부 네트워크로 나가는 패킷을 체크하여 허용/차단 — 패킷의 방향은 **외부로 나가는 패킷의 출발지 IP** |
| **BlackHole Filtering** | 특정 IP 또는 IP 대역에 대하여 비정상적인 시도가 감지되었을 경우 가상의 쓰레기 인터페이스로 보내도록 함. `interface null 0` (가상의 인터페이스) |
| **Unicast RPF Filtering** | IP 또는 IP 대역을 지정하지 않고도 비정상 트래픽을 효율적으로 필터링 — 인터페이스를 통해 들어오는 패킷의 출발지 IP 에 대해 라우팅 테이블을 확인하여 들어온 인터페이스로 다시 나가는지 확인 |

### 2-4. 콘솔·AUX·VTY 포트 패스워드 보안

```
(config) # line console 0
(config-line) # login
(config-line) # password cisco
(config) # line vty 0 4
(config-line) # login
(config-line) # password cisco
```

### 2-5. 텔넷 접근 제한

> SSH 를 설정하여 사용할 것을 권장.

```
(config) # access-list 10 permit host [허용할 IP]
(config) # access-list 10 deny any
(config) # line vty 0 4
(config-line) # access-class 10 in
```

### 2-6. SNMP 설정

> SNMP 정보 수신이 필요한 장비를 명시하여 허용하고 나머지는 차단. SNMP 서비스를 사용하지 않는다면 `no snmp-server` 명령으로 서비스를 제거.

```
(config) # snmp-server community [유추할 수 없는 값] ro 11
(config) # snmp-server contact [이메일]
(config) # access-list 11 permit host [허용할 IP]
(config) # access-list 11 deny any
(config) # interface FastEthernet 0/0
(config-if) # ip access-group 11 in
```

### 2-7. 불필요한 서비스 제거

| 항목 | 명령 | 효과 |
|---|---|---|
| ICMP Redirect 차단 | `no ip redirects` | 패킷의 방향을 바꿔서 정보를 수집할 수 없게 함 |
| Directed Broadcast 차단 | `no ip directed-broadcast` | DoS·DDoS 공격에 악용되지 않도록 함 |
| ICMP Unreachable 차단 | `no ip unreachables` | 상태 정보 노출 방지 |
| Source Route 중지 | `no ip source-route` | 라우팅 경로를 패킷 발송자가 직접 설정할 수 없게 함 |

### 2-8. 입·출력 IP 보안 / 주소 위변조 방지 / BlackHole 보안

| 항목 | 명령 |
|---|---|
| **입·출력 IP 보안** | 외부에서 내부로 유입되는 IP 주소가 내부 IP 인 경우 차단 — `access-list 15 deny [차단할 IP 대역]` / `permit any` / `ip access-group 15 in` |
| **주소 위변조 방지** | Unicast RPF — 인터페이스로 유입되는 출발지 IP 를 라우팅 정보로 점검. reverse path 있으면 허용, 없으면 차단. `(config-if) # ip verify unicast reverse-path` |
| **BlackHole 보안** | DDoS 대응. `interface null 0` + `no ip unreachables` + `ip route 211.1.1.1 255.255.255.255 null 0` |

### 시험 포인트

- `enable password` (평문) ↔ `password-encryption` (암호화·역함수) ↔ `enable secret` (일방향 함수)
- Standard ACL = 1~99 / Extended ACL = 100~199
- Ingress(인입) ↔ Egress(유출) ↔ BlackHole(쓰레기 인터페이스) ↔ Unicast RPF(역방향 검증)
- BlackHole 보안 명령에서 `null 0` 인터페이스가 핵심

| 민첩 대응 | 최신 보안 위협에 민첩하게 대응할 수 있는 전문가 집단의 보안 서비스를 제공받음 |

## 4. Snort

### 4-1. 정의

> Martin Roesch 에 의해 처음으로 개발된 네트워크 트래픽을 감시하고 분석하는 침입탐지시스템.[^snort-manual] Snort 의 발전형으로 **Suricata** 가 있다.

### 4-2. 제공 기능

| 기능 | 설명 |
|---|---|
| 패킷 스니퍼 | 네트워크상의 패킷을 스니핑하여 보여주는 기능 |
| 패킷 로거 | 모니터링한 패킷을 저장하고 로그에 남기는 기능 |
| 네트워크 IDS / IPS | 네트워크 트래픽을 분석하여 공격을 탐지·차단하는 기능 |

### 4-3. Snort Rule 구조

#### 헤더 부분

> `[action] [protocol] [ip address] [port] [direction] [ip address] [port]` 로 헤더 부분을 구성.

| 필드 | 의미 |
|---|---|
| **Rule Actions** | 패킷이 탐지되었을 때의 처리 방식 (`alert` 를 많이 사용) |
| **Protocols** | 탐지할 프로토콜 — TCP / UDP / ICMP / IP 지원 |
| **IP Addresses** | 출발지·목적지 IP (IP 대역 가능) |
| **Port Numbers** | 출발지·목적지 Port (Port 범위 가능) |
| **Direction Operator** | 방향 지시자 — `->` 단방향, `<>` 양방향 |

#### 바디 부분

> `[msg]; [content]; [offset]; [depth]; [distance]; [content]; [within]; [nocase];` 로 바디 부분을 구성.

| 옵션 | 의미 |
|---|---|
| **msg** | alert 발생 시 이벤트 이름으로 사용 |
| **content** | 페이로드에서 검사할 문자열을 지정 |
| **offset** | 페이로드에서 content 패턴을 검사할 시작 위치 (없을 시 0 부터) |
| **depth** | offset 부터 몇 바이트까지 검사할 것인지 |
| **distance** | 이전 content 매칭 이후 몇 바이트 떨어진 위치에서 content 검사 시작 |
| **within** | distance 부터 몇 바이트까지 검사 |
| **nocase** | 페이로드 검사 시 대·소문자 구별하지 않음 |
| **pcre** | 정규표현식 문법을 지원 (문자열 탐지의 유연성) |
| **http_method** | HTTP 메소드 부분을 검사 (HTTP 관련 옵션 다수) |

### 4-4. 룰 바디 이벤트 제한 (threshold)

특정 시간 동안 발생되는 이벤트 수를 제한하기 위한 옵션.

| 옵션 | 형식 | 의미 |
|---|---|---|
| 이벤트 까지 | `threshold:type <limit>, track <by_src\|by_dst>, count <c>, seconds <s>` | count 까지만 alert |
| 이벤트 마다 | `threshold:type <threshold>, track <by_src\|by_dst>, count <c>, seconds <s>` | count 마다 alert |
| 이벤트 시 한번 | `threshold:type <both>, track <by_src\|by_dst>, count <c>, seconds <s>` | count 이상 + 한 번만 alert |

> `track` — `by_src` 출발지 IP 기준 / `by_dst` 목적지 IP 기준 추적. `count` — 몇 번째 / 몇 번 이상. `seconds` — 매 몇 초.

### 4-5. 공격 / 비정상 패킷 탐지 룰 예시

| 공격 | 룰 |
|---|---|
| **FTP root 로그인 시도 탐지** | `alert tcp any any -> 10.10.10.0/24 21` `( msg:"FTP root user access"; content: "User root"; nocase; sid:100100; )` |
| **Telnet root 로그인 성공 탐지** | `alert tcp 10.10.10.0/24 23 -> any any` `( msg:"Telnet login success"; content:"login"; pcre:"/root@.*#/"; nocase; sid:100110; )` |
| **Telnet Brute Force 탐지** | `alert tcp 10.10.10.0/24 23 -> any any` `( msg:"Telnet login brute force attack"; content:"Login incorrect"; nocase; threshold:type limit, track by_dst, count 1, seconds 5; sid:1000120; )` |
| **FTP Brute Force 탐지** | `alert tcp 10.10.10.0/24 21 -> any any` `( msg:"FTP login brute force attack"; content:"Login incorrect"; nocase; threshold:type threshold, track by_dst, count 5, seconds 30; sid:1000130; )` |
| **SSH Brute Force 탐지** | `alert tcp any any -> 10.10.10.0/24 22` `( msg:"SSH login brute force attack"; content:"SSH-2.0"; nocase; threshold:type both, track by_src, count 5, seconds 30; sid:1000150; )` |
| **HTTP GET Flooding 탐지** | `alert tcp any any -> 10.10.10.0/24 80` `( msg:"HTTP Get Flooding attack"; content:"GET / HTTP1."; depth:13; nocase; threshold:type threshold, track by_src, count 100, seconds 1; sid:1000160; )` |
| **TCP SYN Flooding 탐지** | `alert tcp any any -> 10.10.10.0/24 80` `( msg:"TCP SYN Flooding attack"; flags:S; threshold:type threshold, track by_src, count 5, seconds 1; sid:1000170; )` |
| **UDP Flooding 탐지** | `alert udp any any -> $HOME_NET any` `( msg:"UDP Flooding attack"; threshold:type threshold, track by_src, count 5, seconds 1; sid:1000180; )` |
| **사설 IP 탐지** | `alert udp 10.0.0.0/8 any -> $HOME_NET any` `( msg:"Private IP(10.) Detect"; sid:100200; )` |
| **Land Attack (출발지=목적지) 탐지** | `alert ip any any -> $HOME_NET any` `( msg:"Land Attack(SRC/DST SameIP) Detect"; sameIP; sid:100230; )` |
| **SYN_FIN Scan 탐지** | `alert tcp any any -> 10.10.10.0/24 any` `( msg:"SYN_FIN Scan Detect"; flags:SF; sid:100240; )` |
| **SYN_FIN+ Scan 탐지** | `alert tcp any any -> 10.10.10.0/24 any` `( msg:"SYN_FIN+ Scan Detect"; flags:SF+; sid:100250; )` |
| **NULL Scan 탐지** | `alert tcp any any -> 10.10.10.0/24 any` `( msg:"NULL Scan Detect"; flags:!UAPRSF; sid:100270; )` |

### 시험 포인트

- 헤더 7요소 (action·protocol·src ip·src port·direction·dst ip·dst port)
- 바디 핵심 옵션 — content / offset·depth / distance·within / nocase / pcre
- threshold 3종 — limit(까지) / threshold(마다) / both(시 한번)
- flags — `S`(SYN) / `SF`(SYN+FIN) / `SF+`(SYN+FIN+추가) / `!UAPRSF`(NULL Scan)

## 5. iptables

### 5-1. 정의

> 패킷 필터링 기능이 있는 리눅스 커널에 내장된 **netfilter** 기능을 관리하기 위한 도구.[^netfilter] 확장 모듈을 통한 다양한 기능의 제공.

기본 구문 — `iptables [테이블] [체인] [룰] [타겟]` 형식이며 **대소문자를 구별**.

### 5-2. 제공 기능

| 기능 | 설명 |
|---|---|
| **상태 추적** | 모든 패킷에 대한 연결 상태를 추적 — TCP 상태 식별을 위한 상태 추적 기능 활용 |
| **NAT 기능** | 네트워크 주소 변환 기능 |
| **로깅 기능** | 패킷 레벨에서의 로깅 기능 |

### 5-3. 정책 설정 — 테이블

| 명령 | 동작 |
|---|---|
| `iptables -A [체인]` | 해당 체인의 제일 마지막에 룰을 추가 |
| `iptables -I [체인]` | 해당 체인의 첫 행에 룰을 추가 (n 지정 시 특정 위치에 룰 추가) |
| `iptables -D [체인]` | 해당 체인의 해당 룰을 삭제 (n 지정 시 특정 위치의 룰 삭제) |
| `iptables -L [체인]` | 해당 체인에 등록된 룰 리스트 확인 (`-n` 추가 시 숫자 형식 출력) |
| `iptables -F [체인]` | 해당 체인에 등록된 모든 룰을 삭제 |
| `iptables -P [체인] [정책]` | ACCEPT / DROP / REJECT 기본 정책 지정 |

### 5-4. 체인 3종

| 체인 | 의미 |
|---|---|
| **INPUT** | 방화벽을 **최종 목적지** 로 하는 체인 |
| **OUTPUT** | 방화벽을 **최초 출발지** 로 하는 체인 |
| **FORWARD** | 방화벽을 **경유** 하도록 하는 체인 |

### 5-5. 룰 식별자

| 식별 | 옵션 |
|---|---|
| 호스트 식별 | `-s [출발지 IP/대역]` 또는 `-d [목적지 IP/대역]` |
| 서비스 식별 | `--sport [출발지 포트]` 또는 `--dport [목적지 포트]` |
| 메시지 식별 | `--icmp-type [메시지 타입]` |
| TCP 플래그 식별 | `-p tcp --tcp-flags [검사할 플래그] [설정되어야 할 플래그]` |
| TCP 상태 식별 | `-m state --state [탐지할 상태]` |

### 5-6. 타겟

| 타겟 | 동작 |
|---|---|
| `-j ACCEPT` | 해당 패킷 허용 |
| `-j DROP` | 해당 패킷 차단 (응답 메시지 전송 안 함) |
| `-j REJECT` | 해당 패킷 차단 (ICMP 에러 메시지 응답) |
| `-j LOG` | 탐지 로그 기록 |

### 5-7. 상태 추적

#### TCP 프로토콜의 상태 정보

| 상태 | 설명 |
|---|---|
| `NEW` | 최초로 들어온 패킷의 상태 |
| `ESTABLISHED` | 연결 정보를 가지고 있는 패킷의 상태 |
| `RELATED` | 연관된 연결 정보를 가지고 있는 패킷의 상태 |
| `INVALID` | 어떤 상태에도 해당하지 않는 패킷 상태 |

| 단계 | 상태 |
|---|---|
| 1. SYN 패킷 | NEW 상태 (inbound) |
| 2. SYN+ACK 패킷 | ESTABLISHED 상태 |
| 3. ACK 패킷 | 연결 종료 시 마지막 ACK 패킷이 보이기 전까지 ESTABLISHED 상태 |

#### UDP 프로토콜의 상태 정보

| 단계 | 상태 |
|---|---|
| 1. UDP 패킷 유입 | 최초 패킷은 NEW 상태 (inbound) |
| 2. 정보 유지 | 내부적인 TTL 이 유지되는 동안 ESTABLISHED 상태 |

### 5-8. 확장 모듈

| 모듈 | 형식 | 기능 |
|---|---|---|
| **connlimit** | `--connlimit [-above n \| -mask mask]` | 동일한 IP 또는 IP 대역의 동시 연결 개수에 대한 제한 |
| **limit** | `--limit [n/s \| n/m \| n/h \| n/d]` | 불필요한 로그를 남기지 않도록 할 때 사용 |
| **recent** | `recent [--name \| --set \| --rcheck \| --update \| --seconds \| --hitcount]` | 동적으로 source IP 목록을 생성하여 패킷 제어 |

> *원본 표기 보정 — `--connlimit -above`*: 통합자료 표기 그대로 보존했으나, 실제 iptables 문법은 `--connlimit-above` (하이픈 없이 한 옵션). 명령 입력 시 주의. (통합자료 line 4878)

### 시험 포인트

- INPUT(목적지) / OUTPUT(출발지) / FORWARD(경유)
- ACCEPT / DROP(응답 X) / REJECT(ICMP 응답) / LOG
- 상태 4종 — NEW / ESTABLISHED / RELATED / INVALID
- 확장 모듈 — connlimit(동시 연결 제한) / limit(로그 빈도) / recent(동적 IP 목록)

## 6. 보안 솔루션

보안 솔루션은 **보호 대상 영역** 을 기준으로 5가지 분류로 나뉜다. 네트워크 보안 솔루션은 트래픽이 흐르는 경계를 보호하고, 시스템(단말) 보안 솔루션은 엔드포인트를 보호하며, 컨텐츠·정보 유출 방지는 데이터 자체의 외부 유출을 차단한다. 인증 솔루션은 사용자의 신원·권한을 통제하고, 암호 솔루션은 키와 암호 처리를 전용 하드웨어에 위임한다. 본 절에서는 5분류와 그 안의 13종 대표 솔루션을 정리한다.

### 6-1. 네트워크 보안 솔루션 — 경계 보호

WIPS / NAC / UTM 은 모두 네트워크 경계를 통과하는 트래픽 또는 단말의 접속을 통제한다. WIPS 는 무선랜 환경 한정, NAC 는 단말 접속 시점의 무결성 검사, UTM 은 다양한 기능을 하나의 장비로 통합한다.

| 솔루션 | 설명 |
|---|---|
| **WIPS** (Wireless Intrusion Prevention System) | 무선 침입 방지 시스템. 외부 공격으로부터 내부 시스템을 보호하기 위해 무선랜 환경에서의 보안 위협을 탐지·대응. 보안에 취약한 무선 공유기 탐지 / 인가되지 않은 무선 단말기의 접속 자동 탐지·차단 |
| **NAC** (Network Access Control) | 네트워크 접근 제어. 단말기가 내부 네트워크에 접속을 시도할 때 이를 제어·통제. **접근 제어/인증** (역할·IP 기반) / **무결성 체크** (백신·패치·자산 관리) / **탐지·차단** (유해 트래픽 탐지·차단·증거 수집) |
| **UTM** (Unified Threat Management) | 통합 보안 시스템. 다양한 보안 기능(방화벽·IDS/IPS·VPN·안티바이러스 등) 을 하나의 장비로 통합 — 경제성과 보안 관리·운영 편리 |

### 6-2. 시스템 (단말) 보안 솔루션 — 엔드포인트 보호

EDR / 스팸차단 / 보안 OS 는 모두 단말(또는 메일 서버) 단계에서 위협을 탐지·차단한다. EDR 은 엔드포인트 행위 모니터링, 스팸차단은 메일 진입 단계 필터링, 보안 OS 는 운영체제 자체의 커널 강화로 접근 방식이 다르다.

| 솔루션 | 설명 |
|---|---|
| **EDR** (Endpoint Detection and Response) | 엔드포인트 탐지·대응. PC·모바일·서버 등 엔드포인트에서 발생하는 악성 행위를 실시간 감지·분석·대응. 모니터링 — 실행 프로세스 기반 행위 지속 모니터링 / 악성 행위 분석 — 프로세스·레지스트리·파일·사용자 등 |
| **스팸차단 솔루션** | 메일 서버 앞단에 위치하여 프록시 메일 서버로 동작. 스팸 차단 + 바이러스 검사 + 검색 기능을 통한 내부정보 유출 방지 등 확장 기능 |
| **보안 운영체제** | 보안 기능이 있는 통합된 보안 커널을 추가로 이식한 운영체제. 기본적으로 열려 있는 취약 서비스를 모두 차단 ([1과목 07 §1](../01.system-security/07.system-defense.md)) |

### 6-3. 컨텐츠 / 정보 유출 방지 보안 솔루션 — 데이터 보호

보안 USB / DRM / DLP 는 모두 **데이터 자체** 를 보호 대상으로 한다. 보안 USB 는 휴대용 매체에 보안 기능을 내장하고, DRM 은 디지털 자산의 사용 권한을 강제하며, DLP 는 조직 외부로 빠져나가는 데이터 흐름을 모니터링·차단한다.

| 솔루션 | 설명 |
|---|---|
| **보안 USB** | 정보 유출 방지 등의 보안 기능을 갖춘 USB 메모리. 사용자 식별·인증 / 저장데이터 암·복호화 / 저장된 자료의 임의 복제 방지 |
| **DRM** (Digital Rights Management) | 디지털 저작권 관리. 저작권자가 배포한 디지털 자료나 하드웨어의 사용을 제어. 파일 암호화 / 문서 DRM(기업 문서 암호화 저장) |
| **DLP** (Data Loss Prevention) | 데이터 유출 방지. 조직 내부의 중요 자료가 외부로 빠져나가는 것을 탐지·차단. 실시간 차단·모니터링 / **네트워크 DLP** (네트워크 통한 유출 방지) / **단말 DLP** (이동식 매체·출력물 통한 유출 방지) |

### 6-4. 인증 솔루션 — 신원·권한 통제

SSO / EAM / IAM 은 포함 관계로 이해해야 한다. SSO 는 한 번의 인증으로 여러 시스템에 접근하는 통합 인증, EAM 은 SSO 에 사용자별·그룹별 접근 권한 통제를 더한 것, IAM 은 EAM 을 보안 정책 수립·계정 자동 관리·생애주기 관리까지 확장한 가장 포괄적인 솔루션이다.

| 솔루션 | 설명 |
|---|---|
| **SSO** (Single Sign-On) | 한 번의 사용자 인증으로 여러 시스템에 접근할 수 있는 통합 인증 솔루션 |
| **EAM** (Enterprise Access Management) | SSO 와 사용자별·그룹별 접근 권한 통제를 담당하는 권한 관리 솔루션 |
| **IAM** (Identity and Access Management) | EAM 을 보다 포괄적으로 확장한 보안 솔루션. 보안 정책 수립 + 자동 계정 생성 + 위치·직무에 따른 적절한 계정·권한 부여 + 사용자 정보 변경·삭제 실시간 반영 |

### 6-5. 암호 솔루션 — 키 보호 전용 하드웨어

암호화 키와 디지털 키는 메모리에 노출되면 시스템 보안 전체가 무너진다. HSM 은 이러한 키 관리·암호 처리를 전용 하드웨어 모듈에 위임하여 소프트웨어 수준에서 키가 노출되지 않도록 한다.

| 솔루션 | 설명 |
|---|---|
| **HSM** (Hardware Security Module) | 하드웨어 보안 모듈. 암호화된 데이터를 보호하는 암호화 키 / 디지털 키에 대한 안전한 저장·관리 및 고속의 암호화 처리를 수행할 수 있는 전용의 하드웨어 장비 |

### 시험 포인트

- WIPS = 무선 / NAC = 단말 접근 제어 / UTM = 통합 보안
- EDR = 엔드포인트 행위 분석
- DRM(저작권 관리) ↔ DLP(외부 유출 방지)
- SSO(통합 인증) ⊂ EAM(권한 관리) ⊂ IAM(신원·접근 통합 관리)
- HSM = 하드웨어 키 저장소

## 7. 통합 모니터링 — ESM / SIEM / SOAR

### 7-1. ESM — Enterprise Security Management

> 다양한 보안장비에서 발생하는 보안정보를 통합적으로 수집·관리하여 불법적인 행위에 대응할 수 있도록 하는 **통합 보안 관리 시스템**.
> - **상호 연관 분석** : 동종 또는 이기종의 여러 보안 솔루션에서 발생하는 로그·보안 이벤트 패턴 간에 연관성을 분석.

| 구성 요소 | 설명 |
|---|---|
| **ESM 에이전트** | 관리 대상 보안 장비에 설치되어 사전에 정의된 규칙에 따라 로그 / 이벤트 데이터를 수집하여 ESM 매니저로 전달 |
| **ESM 매니저** | ESM 에이전트를 통해 수집된 데이터를 저장·분석하여 그 결과를 ESM 콘솔로 전달 |
| **ESM 콘솔** | ESM 매니저에 의해 전달된 정보의 시각적 전달, 상황 판단·리포팅 등의 기능 제공. ESM 콘솔에서 ESM 에이전트와 ESM 매니저에 대한 제어·통제 수행 |

### 7-2. SIEM — Security Information and Event Management

> ESM 의 진화된 형태. 보안장비뿐 아니라 **각종 서버·네트워크 장비·어플리케이션** 등 다양한 범위에서 발생하는 로그·이벤트를 수집하여 **빅데이터 기반의 상관관계 분석** 을 통해 위협을 사전에 차단하는 통합 보안 관제 솔루션.

| 단계 | 내용 |
|---|---|
| 데이터 통합 | 다양한 장비에서 발생한 데이터를 수집하여 통합 분석 |
| 로그 수집 | 관제 대상 시스템에 설치된 에이전트 / SNMP·syslog 서버로부터 로그를 수집 |
| 로그 변환 | 다양한 로그 형식을 표준 형식으로 변환 |
| 상관관계 분석 | 수집한 데이터를 유용한 정보로 만들기 위해 다양한 상관관계 분석 |
| 로그 분류 | 이벤트 발생 누적 횟수 등 유사 정보를 기준으로 그룹화하여 분류 |
| 로그 분석 | 표준 형식 로그 중 타임 스탬프·IP 주소 등을 기준으로 연관성을 분석 |
| 알림 | 이벤트를 관리자에게 이메일·SMS 로 알림 |
| 대시보드 | 이벤트 데이터의 패턴 표시 / 표준 패턴을 형성하지 않는 활동 파악 |

| 주요 기능 | 설명 |
|---|---|
| 로그 수집 | 네트워크 장비·서버·어플리케이션에서 발생하는 로그를 중앙화 시스템으로 수집 |
| 실시간 모니터링 | 비정상적인 활동을 즉시 탐지 |
| 이벤트 상관 분석 | 다양한 소스에서 수집된 데이터를 상호 연관 지어 분석 — 복합적 보안 위협 식별 |
| 경고 및 알림 | 잠재적 보안 위협 탐지 시 즉각 알림 |
| 보고서 작성 | 보안 이벤트 보고서로 규제 준수·감사 요구 사항 충족 |

| 구성 요소 | 설명 |
|---|---|
| 로그 수집기 | 다양한 소스에서 로그 데이터를 수집하여 SIEM 시스템으로 전달 |
| 데이터 저장소 | 수집된 로그 데이터 저장·관리 |
| 분석 엔진 | 저장된 데이터 분석 / 보안 이벤트 탐지 / 상관 분석 |
| 대시보드·보고서 | 실시간 모니터링 정보 + 분석 결과 시각화 |
| 경고 시스템 | 이상 징후 탐지 시 경고·알림 |

### 7-3. SOAR — Security Orchestration, Automation and Response

> 보안 위협에 대한 **자동화된 분석과 대응 환경** 을 만들어 보안 인력을 효율적으로 운영하면서 분석의 정확도를 높이고 대응 시간을 단축하기 위한 솔루션.

| 구성 | 설명 |
|---|---|
| **SIRP** (Security Incident Response Platform) | 보안 사고 대응 플랫폼 (단순하고 반복적인 업무를 자동화) |
| **SOA** (Security Orchestration & Automation) | 보안 오케스트레이션 및 자동화 (다양한 보안 시스템을 통합·자동화) |
| **TIP** (Threat Intelligence Platform) | 위협 인텔리전스 플랫폼 (보안 분석가의 판단을 보조) |
| 위협 인텔리전스 서비스 | 알려지지 않은 위협, 지능형 지속 위협(APT) 와 같은 공격에 대응 |

| 주요 기능 | 설명 |
|---|---|
| 업무 자동화 | 단순 반복 업무 자동화로 편의성 제공 / 고차원 업무 집중 환경 |
| 동적 플레이북 | 사전 정의된 플레이북으로 일관된 프로세스 작업 / 품질 편차 최소화 |
| 작업 및 스크립트 | 워크플로우에서 스크립트 기능을 추가하여 플랫폼 기능 자동화 운영 |

### 시험 포인트

- ESM(보안장비 한정) → SIEM(전 범위, 빅데이터 분석) → SOAR(자동화·플레이북)
- ESM 3 구성 — 에이전트(수집) / 매니저(분석) / 콘솔(시각·통제)
- SIEM 핵심 = 빅데이터 기반 상관관계 분석
- SOAR 핵심 = 자동화 + 플레이북

## 8. 패치·모바일 관리

### 8-1. PMS — Patch Management System

> 기업 네트워크에 접속하는 사용자 PC 의 운영체제와 각종 어플리케이션에 대한 패치를 기업 보안 정책에 따라 자동으로 설치·업그레이드함으로써 웜이나 바이러스 공격 등으로부터 기업의 IT 환경을 효과적으로 보호해 주는 솔루션.

| 구성 | 설명 |
|---|---|
| **PMS 서버** | 패치를 배포하고 기업 보안 정책에 따라 이를 위반한 사용자 PC 를 인식하고 강제적으로 정책에 맞는 수준의 보안을 적용 |
| **PMS 에이전트** | 서버로부터 패치를 적용하고 사용자 PC 의 상태를 점검해서 보안 정책 위반 여부 정보를 서버에 제공 |

### 8-2. EMM — Enterprise Mobility Management

> 모바일 장치, 무선 네트워크 및 기타 모바일 컴퓨팅 서비스를 중점 관리하여 모바일 기기의 **기밀성·무결성·가용성** 을 보장하는 프로세스 및 기술.

| 하위 기술 | 설명 |
|---|---|
| **MDM** (Mobile Device Management) | 스마트폰·태블릿·휴대용 컴퓨터 등의 디바이스를 관리하는 기술 |
| **MAM** (Mobile Application Management) | 모바일 기기 전체가 아닌 일부 특정 앱에만 기업의 보안 정책을 시행하는 기술 |
| **MCM** (Mobile Content Management) | 기업의 기밀정보와 같은 콘텐츠를 쉽고 안전하게 공유·접근하도록 하는 기술 |

### 8-3. BYOD vs CYOD

| 정책 | 설명 |
|---|---|
| **BYOD** (Bring Your Own Device) | 개인이 소유한 스마트 기기를 직장에 가져와 업무에 활용하도록 허용. **컨테이너화** (암호화된 별도 공간에 업무용·개인용 데이터 분리) / **모바일 가상화** (개인용·업무용 OS 동시 탑재). 민감한 기업 데이터 훼손 위험 존재 |
| **CYOD** (Choose Your Own Device) | 기업 IT 부서가 통제권을 가지고 표준 소프트웨어를 탑재해 보안과 액세스 관리를 강화한 정책. 기업 소유의 디바이스를 제공받아 업무에 활용 |

### 시험 포인트

- PMS = 패치 자동 배포
- EMM ⊃ MDM(기기) / MAM(앱) / MCM(콘텐츠)
- BYOD(개인 소유 → 컨테이너화·가상화) ↔ CYOD(기업 소유 → 표준 SW 탑재)

## 9. 방화벽 / NAT

### 9-1. 정의

> 조직의 보안 정책에 따라 인가된 네트워크 서비스에 대한 접근을 허가하고 인가되지 않은 서비스를 수행하는 트래픽을 철저하게 차단함으로써 효율적인 보안 서비스를 제공하는 시스템.[^nist-800-41]

### 9-2. 방화벽 분류 — 동작 계층별

| 분류 | 동작 |
|---|---|
| **네트워크 계층 방화벽** | IP 주소·포트 기반 트래픽 필터링 — 네트워크 수준 보호. **패킷 필터링** : IP 주소 + 전송 계층의 Port 정보로 패킷을 필터링 |
| **세션 계층 방화벽** | 네트워크 계층 방화벽 기능 + 세션 상태 추적 + 트래픽 제어. **상태 검사** : IP·Port 만 보지 않고 일정 시간 프로토콜별 연결 상태 정보를 유지·추적 |
| **응용 계층 방화벽** | 애플리케이션 계층에서 동작 — 트래픽 검사·제어. **프락시 서비스** (클라이언트·서버 중개) / **배스천 호스트** (트래픽 분산·가용성·성능) / **응용 계층 게이트웨이** (애플리케이션 간 통신 관리·중개) / **회선 레벨 게이트웨이** (물리적 연결 관리·보안) |

### 9-3. 방화벽 구축 형태 4종

| 형태 | 구성 |
|---|---|
| **스크리닝 라우터** 구조 | 라우터를 이용해 각 인터페이스에 들어오고 나가는 패킷을 필터링 |
| **듀얼-홈드 호스트** 구조 | 두 개의 인터페이스를 가지는 호스트를 통해 내·외부 네트워크를 연결 |
| **스크린드 호스트 게이트웨이** 구조 | 듀얼-홈드 게이트웨이와 외부 네트워크 사이에 스크리닝 라우터를 설치하여 구성 |
| **스크린드 서브넷** 구조 | 스크리닝 라우터 사이에 듀얼-홈드 게이트웨이가 위치하는 구성 |

### 9-4. NAT

> 라우터나 방화벽과 같은 장치에서 외부 네트워크와 내부 네트워크 간의 주소 변환을 수행. 외부에서 내부 네트워크로 직접적인 접근이 불가능하게 되므로 네트워크 보안 효과 ([02 단원 §6 NAT](02.network-protocols-and-addressing.md) 참조).

### 시험 포인트

- 네트워크 계층(L3·L4 패킷 필터링) ↔ 세션 계층(상태 검사) ↔ 응용 계층(프락시·배스천)
- 구축 형태 4종 — 스크리닝 라우터 / 듀얼홈드 호스트 / 스크린드 호스트 / 스크린드 서브넷
- 스크린드 서브넷 = 라우터 2 + 듀얼홈드 게이트웨이 1 (가장 강한 구조)

## 10. 허니팟·허니넷

### 10-1. 허니팟 (Honeypot)

> 실제로 자료를 가진 호스트인 것처럼 인터넷상에 존재하면서 해커·악의적인 내부자 등을 속여 그들로부터 침해당함으로써 그들의 행동·공격 기법·목적 등을 분석.

| 특성 | 내용 |
|---|---|
| 조기 경보 | 잠재적 공격자에 대한 조기 경보 제공 |
| 보안 전략 결점 파악 | 조직 전체의 보안 의식 향상 |
| 예방 통제 | 고의적으로 취약점을 내포 — 패치 미발표 시점의 공격을 사전 탐지 |
| 목적 | 시스템 논리적 파괴·손실 방지 |

#### 허니팟 동작 과정

| 단계 | 내용 |
|---|---|
| 1 | 중요한 시스템에 접근하는 공격자를 다른 방향으로 돌림 |
| 2 | 공격자의 동작에 관한 정보를 수집 |
| 3 | 관리자가 반응할 수 있도록 공격자가 시스템에 충분히 오랜 시간 동안 머무르게 유도 |

### 10-2. 허니넷 (Honeynet)

> 다수의 허니팟으로 구축된 네트워크. 해커들의 행동·방법에 대한 정보 수집 + 해커들을 유인한 후 실제 서비스 네트워크 / 자원과 격리하여 여러 시스템에 대한 공격 패턴 파악. 공격자들은 허니넷을 정상 네트워크로 인식.

## 11. IDS · IPS

### 11-1. IDS — Intrusion Detection System

> 사전에 등록한 알려진 공격 패턴, 임계치 등에 따라 공격을 **탐지** 하는 보안장비.[^nist-800-94]
> - **호스트 기반 IDS (H-IDS)** — 호스트에 설치되어 호스트 정보를 수집하여 탐지 (Tripwire 등)
> - **네트워크 기반 IDS (N-IDS)** — 네트워크 트래픽을 수집하여 탐지 (Snort 등)

| 주요 기능 | 설명 |
|---|---|
| 감시 | 네트워크 트래픽 실시간 모니터링 / 특정 패턴·악성 행위 감지 |
| 탐지 | 사전 정의된 악성 패턴·행위를 분석하여 악성 활동 식별 |
| 알림 | 담당자에게 알림 제공 / 대응 조치 |
| 보고 | 감지된 악성 활동 보고서 / 보안 이슈 분석·추세 파악 |

#### 탐지 종류

| 종류 | 동작 | 오탐 |
|---|---|---|
| **오용 탐지** (Misuse Detection) | 사전에 등록한 알려진 공격 패턴을 기반으로 공격 행위를 탐지 | **False Negative 가 높음** (공격을 탐지 못함) |
| **이상 탐지** (Anomaly Detection) | 정상적이고 평균적인 상태를 기준으로 이를 벗어난 행위를 공격으로 탐지 | **False Positive 가 높음** (공격이 아닌 것을 공격으로 판단) |

### 11-2. IPS — Intrusion Prevention System

> 사전에 등록한 알려진 공격 패턴·임계치 등에 따라 공격을 **탐지하고 차단** 하는 보안장비.[^nist-800-94]

| 주요 기능 | 설명 |
|---|---|
| 감시 및 탐지 | IDS 와 마찬가지로 트래픽 감시 / 악성 활동 탐지 |
| 차단 | 감지된 악성 활동을 차단하여 액세스 차단·트래픽 필터링 |
| 가로채기·수정 | 악성 트래픽을 가로채고 수정하여 악성 행위의 효과를 방지 |
| 자동 대응 | 정책에 따라 차단·경보 발생·보고서 생성 등을 자동 수행 |

### 시험 포인트

- H-IDS(Tripwire) / N-IDS(Snort)
- 오용 탐지 = False Negative ↑ / 이상 탐지 = False Positive ↑
- IDS = 탐지만 / IPS = 탐지 + 차단

## 12. 추가 보안 장비

UTM 이 다양한 기능을 하나의 장비로 통합하는 흐름과 별개로, 특정 위협·기능에 특화된 전용 장비도 함께 운영된다. 본 절에서는 출제기준 3-2-3 의 "DDoS 차단·SSL 오프로드·APT 대응·네트워크 포렌식" 4종 장비를 정리한다. 4종 모두 일반 방화벽·IDS/IPS 의 범위를 넘어 **특정 트래픽 유형 처리** 에 특화된 어플라이언스라는 공통점이 있다.

### 12-1. DDoS 차단 시스템

| 항목 | 내용 |
|---|---|
| 정의 | DDoS 공격으로부터 다른 장비들을 보호하기 위한 전용 차단 장비 |
| 위치 | 네트워크 구성상 **가장 상단** 에 위치 (인터넷 인입단) |
| 운영 맥락 | [05 단원 §7 DDoS 대응 방안](05.dos-ddos-drdos.md) 의 자체 방어 + DDoS 대응 서비스의 핵심 장비 |

### 12-2. SSL 오프로드

| 항목 | 내용 |
|---|---|
| 정의 | SSL 서버를 대신해서 SSL 트래픽에 대한 모든 암·복호화 처리를 수행하는 기능 |
| 목적 | 암호화된 트래픽에 대한 **가시성 확보** — 방화벽·IDS 등 다른 보안 장비가 평문을 검사 가능 |

| 장비 | 설명 |
|---|---|
| **SSL-VA 장비** | SSL 트래픽 암·복호화 처리에 특화된 전용 어플라이언스 |
| **L7-Switch 장비** | 어플리케이션 컨텐츠 분석을 통한 패킷 탐지 + SSL 오프로드 + 서버 부하 분산(SLB) + 장애 조치(Failover) 등 다양한 기능 |

### 12-3. APT 대응 시스템

| 항목 | 내용 |
|---|---|
| 정의 | 네트워크 트래픽을 통해 유입되는 실행파일·문서파일 등을 **추출, 평판 조회 및 가상머신 샌드박스 환경** 에서 직접 실행하여 악성 여부를 판단하는 보안장비 |
| 핵심 기술 | **가상머신(샌드박스)** — 외부로부터 받은 파일을 보호된 영역에서 실행시켜 시스템에 악영향을 주는 파일인지 확인 |
| 운영 맥락 | [06 단원 §3-2 동적 분석](06.remote-access-and-malware-tools.md) 의 행위 분석 기법을 장비화한 것. 알려지지 않은 신종 악성코드 / APT 공격 대응 |

### 12-4. 네트워크 포렌식 장비

| 항목 | 내용 |
|---|---|
| 정의 | 설치된 네트워크 구간을 통과하는 모든 트래픽을 수집하고 분석할 수 있는 장비 |
| 활용 | 침해사고 분석 및 내부 감사 |
| 요구사항 | **대용량의 스토리지** 필요 (모든 트래픽을 raw 패킷 단위로 보관) |

## 13. 네트워크 구간 / 보안장비 설치 모드 / 보안장비 운영

### 13-1. 네트워크 구간

| 구간 | 정의 |
|---|---|
| **인터넷 구간** | 인터넷을 통해 외부에서 직접 접근이 가능한 구간 |
| **DMZ 구간** | 인터넷 구간과 내부망 사이에 위치한 중간 지점. 인터넷으로부터의 접근과 내부망으로의 접근을 허용하는 구간 |
| **내부망** | 인터넷으로부터 물리적으로 분리된 망이거나 접근 통제 시스템에 의해 직접 접근이 불가능하도록 통제·차단된 구간 |

### 13-2. 보안장비 설치 모드

| 모드 | 설명 | 비고 |
|---|---|---|
| **인라인 모드** | 물리적 네트워크 경로상에 설치되어 통과하는 모든 트래픽이 거쳐 가도록 하는 모드. 탐지·차단 모두 가능. **장비 장애가 전체 네트워크 장애로 확산** | **IPS 설치 모드** |
| **미러링 모드** | 물리적 네트워크 경로 밖에 위치하여 **TAP 장비** 를 통해 복제된 패킷을 탐지하는 모드. 복제된 패킷이므로 실제 패킷 차단 어려움. 네트워크 영향 없음 | **IDS 설치 모드** |

### 13-3. 보안장비 관리

| 영역 | 항목 |
|---|---|
| **계정 관리** | default 계정·패스워드 변경 / 계정별 권한 설정 (불필요한 권한 제거) / 미사용 계정 삭제 (공용 계정 사용 시 책임 추적성 확보) |
| **접근 관리** | 원격 관리 접근 통제 (관리자 IP 또는 특정 IP·계정만 접속) / 보안 접속 (SSH·HTTPS 암호화) / 세션 타임아웃 활성화 |

### 시험 포인트

- 인터넷 → DMZ → 내부망 (3 계층 분리)
- 인라인(IPS, 차단 가능, 장애 위험) ↔ 미러링(IDS, 차단 어려움, 영향 X)

## 14. 취약점 점검 / 무결성 / 루트킷 점검 / 파일 속성

### 14-1. Nessus — 취약점 점검 도구

> 미국 Tenable 사가 개발한 취약점 점검 도구.[^nessus] 로컬·원격지에서 다양한 방법을 통해 시스템·네트워크·웹 어플리케이션의 알려진 취약점에 대한 점검을 수행하며 점검을 통해 취약점의 내용과 해결 방법을 상세하게 제공.

| 항목 | 명령 / 정보 |
|---|---|
| 데몬 기동 | `etc/init.d/nessusd restart` (기본 포트 **8834/tcp**) |
| 동작 확인 | `ps -ef \| grep nessusd` |
| 구조 | 클라이언트 / 서버 구조로 동작 |
| 특징 | 사용 자유 + 플러그인 업데이트 쉬움 + HTML 등 여러 형태 결과 리포트 |

### 14-2. Nikto — 웹 취약점 점검

> 웹 해킹에 대응하기 위해 웹서버나 웹 기반의 응용 프로그램 취약점을 점검할 수 있는 대표적인 프로그램.[^nikto]

```
nikto -h [웹서버 IP] -output vul.html -Format html
```

> 웹서버 설치 시 기본적으로 설치되는 파일과 웹서버의 종류·버전 등을 스캔. **DB 를 이용해 취약한 CGI 파일을 스캔하는 기능이 매우 뛰어남**.

### 14-3. Tripwire — 무결성 점검

> 파일 시스템 무결성 점검은 파일 시스템의 상태 추적·허가받지 않은 변경 여부를 주기적으로 점검하여 의심스러운 변화가 감지되면 이를 검사·복구하는 과정. 유닉스·리눅스 환경에서 파일 시스템 무결성을 점검하는 대표적인 도구가 **트립와이어**.[^tripwire]

| 단계 | 명령 / 내용 |
|---|---|
| 1 | 트립와이어 정책 설정 (정책 설정 파일을 통해 점검할 파일·디렉터리 설정) |
| 2 | 파일시스템 초기화 — `tripwire --init` (초기 해시값 생성) |
| 3 | 해시값을 데이터베이스에 저장 |
| 4 | 데이터베이스 파일 생성 확인 — `/var/lib/tripwire/[호스트명].twd` |
| 5 | 트립와이어 실행 — `tripwire --check` |
| 6 | 실행 결과를 화면 출력 + 상세 결과를 report 파일로 제공 |
| 7 | 결과 리포트 확인 |

### 14-4. chkrootkit — 루트킷 점검

> 시스템에 루트킷이 설치되어 있는지 여부를 손쉽게 체크할 수 있는 도구.

| 단계 | 내용 |
|---|---|
| 1 | `chkrootkit -q` 실행 (quiet 모드로 변조·감염된 파일 정보 확인) |
| 2 | 결과에 `INFECTED` 확인 — 해당 파일이 변조·감염되었음을 의미 |

### 14-5. /proc 파일 시스템 + 백도어 탐지 시나리오

> 커널이 메모리상에 사용하고 있는 모든 자원들에 대한 정보들을 파일 형식으로 보관하는 파일 시스템.

| 경로 | 의미 |
|---|---|
| `/proc/[pid]/exe` | 프로그램의 실행파일명 및 경로를 확인할 수 있는 심볼릭 링크 파일 |
| `/proc/[pid]/cmdline` | 프로세스를 실행한 명령어 및 전달인자 정보를 담고 있는 파일 |

#### /proc 를 이용한 루트킷 탐지 원리

> `ps` 실행 결과와 `/proc` 디렉터리에 있는 프로세스 정보를 비교하여 `/proc` 디렉터리에는 프로세스가 있지만 `ps` 실행 시 보이지 않는 프로세스를 **히든 프로세스** 로 탐지. 시차에 의해 오탐이 발생할 수 있음.

#### 백도어 탐지 시나리오

| 단계 | 명령 / 결과 |
|---|---|
| 1 | `ps` 명령 실행 시 동작 중인 백도어 프로그램 `bdoor` 프로세스 정보가 출력되지 않음 |
| 2 | `/proc/15415/exe` 를 통해 `bdoor` 프로세스가 동작 중인 것을 확인 |
| 3 | 해당 프로세스의 실행파일명 및 경로 확인 |
| 4 | `cp /proc/15415/exe /tmp/backdoor` 명령으로 실행파일 복원 |
| 5 | `cat /proc/15415/cmdline` 명령으로 프로세스를 실행한 명령어 및 전달인자 확인 |

### 14-6. Strace 명령

> 특정 프로그램의 시스템 콜과 시그널을 추적하는 데 사용하는 디버깅 도구. 정상적인 프로그램과 변조된 프로그램의 시스템 콜을 비교하여 변조 여부를 확인. 예 — 변조된 `ps` 프로그램이 정상 `ps` 와 달리 특정 파일을 오픈하고 있는 것을 확인.

### 14-7. rpm -V 명령

> `rpm -V [패키지이름]` 명령을 실행하여 최초 설치 시점과 비교해서 파일 크기, MD5 체크섬, 파일 수정 시간이 변경된 것을 확인.

| 플래그 | 의미 |
|---|---|
| **S** | 파일 크기 변경 |
| **M** | 파일 퍼미션 변경 |
| **5** | MD5 체크섬 변경 |
| **T** | 파일 수정 시간 변경 |

#### 패키지 재설치 과정

| 단계 | 명령 |
|---|---|
| 1 | `rpm -Uvh --force [패키지 파일 이름]` 명령으로 패키지 재설치 시 오류 발생 |
| 2 | `lsattr` 명령으로 `i` 속성이 설정되어 있는지 확인 |
| 3 | `chattr` 명령으로 `i` 속성을 제거 |
| 4 | `rpm` 명령을 통해 재설치 |

> *원본 표기 보정 — `rpm -Uyh`*: 통합자료 line 5383 의 `rpm -Uyh` 는 OCR 결함이며 실제 명령은 `rpm -Uvh` (`-v` verbose, `-h` hash). 본문은 정상 명령으로 표기.

### 14-8. lsattr / chattr — 리눅스 파일 속성

| 명령 | 설명 |
|---|---|
| **lsattr** | 파일 속성 정보를 확인할 수 있는 명령 |
| **chattr** | 파일 속성 정보를 변경할 수 있는 명령 — `+속성` 으로 설정 / `-속성` 으로 제거 |

| 속성 | 의미 |
|---|---|
| `i` | **읽기 전용 모드** (immutable) — 변경 / 삭제 불가 |
| `a` | **내용 추가만 허용** (append only) |
| `A` | **Access Time 을 변경하지 않음** |

### 시험 포인트

- Nessus = Tenable / 8834/tcp
- Nikto = 웹 취약점 / CGI 스캔에 강함
- Tripwire = H-IDS 무결성 점검 / `--init` → `--check`
- chkrootkit = INFECTED 출력으로 루트킷 식별
- /proc/[pid]/exe ↔ /proc/[pid]/cmdline (실행파일 복원·실행 인자 확인)
- Strace = 시스템 콜 추적
- rpm -V 4 플래그 — S(크기) / M(퍼미션) / 5(MD5) / T(시간)
- lsattr / chattr 3 속성 — i(immutable) / a(append) / A(no atime)

## 15. 취약점 정보 분류 체계 — CVE / CWE

### 15-1. CVE — Common Vulnerabilities and Exposures

> 공개적으로 알려진 소프트웨어의 보안 취약점을 표준화한 식별자 목록.[^mitre-cve]
> 규칙 — `CVE-[해당년도]-[취약점 번호]` (예 : `CVE-2014-0160`)

### 15-2. CWE — Common Weakness Enumeration

> 소프트웨어에서 공통적으로 발생하는 약점을 체계적으로 분류한 목록.[^mitre-cwe] 개발 과정에서 발생할 수 있는 취약점 (소스 코드 보안 취약점).

#### CWE 보안 취약점 7 분류

| 분류 | 설명 |
|---|---|
| **입력 데이터 검증 및 표현** | 입력 값에 대한 검증 누락 또는 부적절한 검증 |
| **보안 기능** | 보안을 부적절하게 구현 시 발생 |
| **시간 및 상태** | 동시 수행을 지원하는 병렬 시스템·하나 이상의 프로세스가 동작하는 환경에서 시간·상태를 부적절하게 관리하여 발생 |
| **에러 처리** | 에러를 처리하지 않거나, 중요한 정보가 포함될 때 |
| **코드 오류** | 인가되지 않은 사용자에게 데이터 노출 |
| **캡슐화** | 중요한 데이터·기능을 불충분하게 캡슐화 하였을 때 |
| **API 오용** | 의도된 사용에 반하는 방법으로 API 를 사용하거나, 보안에 취약한 API 를 사용하여 발생 |

### 시험 포인트

- CVE = 식별자 목록 (개별 취약점) / CWE = 약점 분류 (취약점 카테고리)
- CVE 형식 — CVE-YYYY-NNNN
- CWE 7 분류 — 입력 검증 / 보안 기능 / 시간·상태 / 에러 처리 / 코드 오류 / 캡슐화 / API 오용

## 단원 요약

1. **VLAN** : Port·MAC·Network·Protocol 4 종 — 논리적 브로드캐스트 도메인 분리.

2. **라우터 보안**
   - 패스워드 — `enable password`(평문) / `password-encryption`(가역) / `enable secret`(일방향)
   - ACL — Standard(1~99) / Extended(100~199)
   - Filtering — Ingress / Egress / BlackHole(`null 0`) / Unicast RPF
   - 불필요 서비스 차단 — `no ip redirects` / `no ip directed-broadcast` / `no ip unreachables` / `no ip source-route`

3. **SecaaS** — 클라우드 기반 보안 서비스. 식별/접근·DLP·웹·이메일·감사·침입·SIEM·암호화·BCP·취약점·모니터링 10 분야.

4. **Snort** : Martin Roesch / Suricata 후속. 헤더 7 요소 + 바디 핵심 옵션 (content / offset·depth / distance·within / nocase / pcre). threshold 3 종(limit·threshold·both). flags 표기(SF·SF+·!UAPRSF).

5. **iptables** : netfilter 도구
   - 체인 3 종 (INPUT 목적지 / OUTPUT 출발지 / FORWARD 경유)
   - 타겟 4 종 (ACCEPT / DROP / REJECT / LOG)
   - 상태 4 종 (NEW / ESTABLISHED / RELATED / INVALID)
   - 확장 모듈 (connlimit / limit / recent)

6. **보안 솔루션** : 네트워크(WIPS·NAC·UTM) / 단말(EDR·스팸차단·보안 OS) / 정보 유출(보안USB·DRM·DLP) / 인증(SSO ⊂ EAM ⊂ IAM) / 암호(HSM).

7. **통합 모니터링** : ESM(보안 장비 한정) → SIEM(전 범위·빅데이터·상관관계) → SOAR(자동화·플레이북·SIRP·SOA·TIP).

8. **패치·모바일** : PMS / EMM(MDM·MAM·MCM) / BYOD(개인) ↔ CYOD(기업).

9. **방화벽** : 네트워크/세션/응용 계층. 구축 4 종 (스크리닝 라우터 / 듀얼홈드 / 스크린드 호스트 / 스크린드 서브넷). NAT 으로 보안 효과.

10. **허니팟·허니넷** : 의도적 취약 시스템으로 공격자 유인·정보 수집·예방 통제.

11. **IDS·IPS** : H-IDS(Tripwire) / N-IDS(Snort). 오용(False Negative ↑) / 이상(False Positive ↑). IDS = 탐지 / IPS = 탐지 + 차단.

12. **추가 장비** : DDoS 차단(상단) / SSL 오프로드(SSL-VA·L7-Switch) / APT 대응(샌드박스) / 네트워크 포렌식.

13. **네트워크 구간 / 설치 모드** : 인터넷·DMZ·내부망. 인라인(IPS·차단) ↔ 미러링(IDS·TAP).

14. **점검 도구** : Nessus(8834/tcp·Tenable) / Nikto(CGI 강함) / Tripwire(`--init`→`--check`) / chkrootkit(INFECTED) / `/proc/[pid]/exe`·`cmdline` / Strace(syscall) / `rpm -V`(S·M·5·T) / `lsattr`·`chattr`(i·a·A).
