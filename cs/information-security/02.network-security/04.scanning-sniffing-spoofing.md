## 1. 스니핑 (Sniffing)

### 1-1. 정의

네트워크상에서 은밀하게 패킷을 도청하는 것. 어떤 영향을 미치지 않기 때문에 **수동적(passive) 공격 기법** 으로 분류된다.

| 용어 | 의미 |
|---|---|
| **스니퍼(Sniffer)** | 스니핑 작업을 할 때 사용되는 도구 (Wireshark, tcpdump 등) |
| **스니핑 탐지 원리** | 기본적으로 스니퍼가 NIC 의 **Promisc 모드** 에서 작동한다는 점을 이용 ([03 단원 §1-4·§4-2](03.network-tools-and-wireless.md) 참조) |

### 1-2. 환경별 스니핑 기법

| 환경 | 기법 | 동작 |
|---|---|---|
| **허브 환경** | 무차별 모드(Promisc) | 자신이 목적지가 아닌 패킷도 모두 수신 (허브가 모든 포트로 신호 전달하므로 NIC 가 받기만 하면 됨) |
| **스위치 환경** | 스위치 재밍 | 스위치 MAC Address Table 의 버퍼를 오버플로우 → Fail Safe 정책으로 허브처럼 동작 |
| | ARP 스푸핑 | 특정 호스트로 나가는 패킷을 공격자로 향하도록 |
| | ARP 리다이렉트 | 라우터/게이트웨이로 나가는 패킷을 공격자로 향하도록 |
| | ICMP 리다이렉트 | 희생자의 라우팅 테이블을 변조하여 패킷을 스니핑 |
| | SPAN / Port Mirroring | 스위치 통과 모든 트래픽을 분석 장비로 자동 복사 (정상 운영 기능을 악용) |

### 1-3. 스니핑 탐지

스니퍼가 Promisc 모드로 동작한다는 점을 이용한다.

| 탐지 방법 | 동작 |
|---|---|
| **ping 을 이용** | 위조된 MAC 주소로 ping 을 보냈을 때 ICMP Echo Reply 를 받는지 확인 — Promisc NIC 는 자신의 MAC 이 아닌 패킷도 받아 응답할 수 있음 |
| **ARP 를 이용** | 위조된 ARP Request 를 보냈을 때 ARP Response 가 오는지 확인 |
| **DNS 를 이용** | 원격에서 테스트 대상 네트워크로 Ping Sweep 을 보내고, 들어오는 Inverse-DNS Lookup 을 감시 — 스니퍼가 IP 를 도메인으로 변환하려 시도하는지 확인 |
| **유인(decoy) 방법** | 가짜 계정·패스워드를 네트워크에 계속 뿌리고, 공격자가 이를 이용해 접속을 시도하는지 확인 |
| **ARP Watch** | 초기에 MAC ↔ IP 매칭 값을 저장하고 ARP 트래픽을 모니터링하여 변경을 감지 |

### 1-4. 스니핑 대응 방안

암호화가 핵심이다.

| 대응 | 적용 영역 |
|---|---|
| **SSL** | 암호화된 웹 서핑 |
| **PGP / PEM / S/MIME** | 이메일 전송 시 암호화 ([07 단원](07.security-protocols.md) 참조) |
| **SSH** | Telnet 같은 평문 서비스의 암호화 대체 |
| **VPN** | 공용 회선을 이용한 사설 암호화 망 |

### 시험 포인트

- 스니핑 = **수동적 공격**, 능동적인 영향은 없으나 정보 노출
- 허브 환경 — Promisc 모드만으로 가능 / 스위치 환경 — 5개 기법(재밍 / ARP 스푸핑 / ARP 리다이렉트 / ICMP 리다이렉트 / SPAN)
- 스니핑 탐지 5종 — ping / ARP / DNS / 유인 / ARP Watch
- 대응의 공통 키워드 = **암호화** (SSL · S/MIME · SSH · VPN)

## 2. ARP / IP / ICMP 공격

### 2-1. ARP Spoofing (= ARP Cache Poisoning)

2 계층에서 동작. 2계층 주소는 서로 다른 네트워크로 라우팅되지 않으므로 **공격 대상도 동일 네트워크 대역에 있어야 한다.**

#### 2-1-1. 공격 절차

1. 특정 호스트의 MAC 주소를 자신의 MAC 주소로 위조한 ARP Reply 패킷을 만듦
2. 희생자에게 지속적으로 전송
3. 희생자 ARP Cache Table 의 MAC 정보가 공격자의 MAC 정보로 지속적으로 변경
4. 희생자들이 정상적인 통신을 계속할 수 있도록 공격자가 **IP Forward 기능을 활성화**
5. 희생자로부터 특정 호스트로 나가는 패킷을 공격자로 향하도록 하여 스니핑

> **IP Forward** — 일반적으로 자신이 목적지가 아닌 패킷은 폐기하지만, 라우터처럼 동작하여 목적지 주소로 패킷을 전달해주는 기능. ARP/ICMP 리다이렉트 공격에 필수.

#### 2-1-2. 양방향 스니핑

상호간의 패킷을 모두 스니핑하려면 A 뿐만 아니라 B 에게도 변조된 ARP Reply 를 전송하여 양쪽의 ARP Cache 를 모두 변경해야 한다.

### 2-2. ARP Redirect

ARP Spoofing 의 일종. 자신이 라우터/게이트웨이인 것처럼 MAC 주소를 위조하여 ARP Reply 패킷을 대상 네트워크에 지속적으로 브로드캐스트 → 해당 로컬 네트워크의 모든 호스트의 ARP Cache Table 에 라우터의 MAC 정보가 공격자의 MAC 정보로 변경.

| 항목 | ARP Spoofing | ARP Redirect |
|---|---|---|
| **위조 대상 MAC** | 특정 호스트의 MAC | 라우터/게이트웨이의 MAC |
| **전송 대상** | 희생자 1대 (지속적 유니캐스트) | 대상 네트워크 전체 (지속적 브로드캐스트) |
| **목표** | 희생자 ↔ 특정 호스트 사이의 패킷 스니핑 | 호스트들 ↔ 라우터 사이의 패킷 스니핑 |

### 2-3. ARP 공격 대응 방안

ARP 프로토콜 자체에 별도의 인증 과정이 없는 취약점을 이용한 공격이므로 완벽한 방어는 없다.

| 대응 | 동작 |
|---|---|
| **캐시 정적 설정** | `arp -s [대상 IP] [대상 MAC]` 명령으로 ARP Cache 정보를 정적으로 설정 → ARP 응답 수신해도 갱신하지 않음 |
| **ARP 트래픽 모니터링** | IP ↔ MAC 주소 매핑을 감시하고 변경 발생 시 즉시 확인 (ARP Watch 등) |

### 2-4. IP Spoofing

IP 를 속이고 통신하는 공격. 1995년 케빈 미트닉(Kevin Mitnick) 이 이를 이용해 실제 해킹을 시도하여 널리 알려진 공격. 시스템 간의 트러스트 관계를 이용한다.

#### 2-4-1. 공격 절차

1. 트러스트 관계가 맺어진 서버와 클라이언트를 확인
2. 신뢰 관계가 있는 클라이언트를 연결 불가능한 상태로 만듦 (DoS 공격 등 이용)
3. 공격자가 클라이언트의 IP 로 위조하여 서버에 접속

> **트러스트(Trust) 관계** — ID/패스워드 기반 로그인이 아닌 IP 주소로 인증하고 로그인 없이 접속할 수 있도록 해주는 방식. 보안상 취약하여 사용 비권장. 유닉스·리눅스에서는 `/etc/hosts.equiv`(시스템 전체) 와 `$HOME/.rhosts`(사용자별) 파일에 설정. r 계열 서비스(rlogin·rsh·rcp) 가 이를 이용.

#### 2-4-2. IP Spoofing 대응

| 대응 | 동작 |
|---|---|
| **MAC 주소 정적 구성** | 단순히 IP 만을 위조한 접속을 차단 |
| **`/etc/hosts.equiv` / `$HOME/.rhosts` 관리** | `+` 기호 제거하고 반드시 필요한 계정만 등록 / 소유자를 root, 권한을 600 이하로 설정 |

### 2-5. ICMP Redirect 공격

ICMP Redirect 메시지(Type 5) 는 호스트-라우터 또는 라우터 간 라우팅 경로를 재설정하기 위해 사용된다 ([02 단원 §3-2-3](02.network-protocols-and-addressing.md) 참조). 공격자는 이를 악용해 특정 IP 또는 IP 대역으로 나가는 패킷의 라우팅 경로를 자신의 주소로 위조한 ICMP Redirect 메시지를 생성·희생자에게 전송한다.

#### 2-5-1. 공격 절차

1. 패킷의 라우팅 경로를 자신의 주소로 위조한 ICMP Redirect 메시지 생성
2. 희생자에게 전송
3. 희생자의 라우팅 테이블을 변조하여 패킷을 스니핑 (IP Forward 기능 사용)

#### 2-5-2. 대응

| 방법 | 명령 |
|---|---|
| **ICMP Redirect 옵션 해제** | `sysctl -w net.ipv4.conf.all.accept_redirects=0` |

### 2-6. ARP Redirect vs ICMP Redirect

| 항목 | ARP Redirect | ICMP Redirect |
|---|---|---|
| **변조 대상** | ARP Cache Table | 라우팅 테이블 |
| **위조 패킷** | 자신을 라우터로 위장한 ARP Reply | 라우팅 경로를 자신으로 향하게 한 ICMP Type 5 메시지 |
| **계층** | 2 (Data Link) | 3 (Network) |

### 시험 포인트

- ARP Spoofing — 특정 호스트의 MAC 위조, **유니캐스트 송신**, IP Forward 활성화
- ARP Redirect — 라우터/게이트웨이의 MAC 위조, **브로드캐스트 송신**
- ICMP Redirect — Type 5 메시지로 라우팅 테이블 변조, 대응 = `sysctl net.ipv4.conf.all.accept_redirects=0`
- ARP Redirect ↔ ICMP Redirect 의 변조 대상 차이 (ARP Cache vs 라우팅 테이블)
- IP Spoofing 기반 공격 = 트러스트 관계(`hosts.equiv`·`.rhosts`) 악용. 케빈 미트닉 사례
- ARP 공격은 동일 네트워크 대역 한정 (라우터를 넘지 못함)

## 3. 세션 탈취 — TCP Session Hijacking / MITM

### 3-1. TCP 세션 하이재킹

케빈 미트닉이 사용했던 공격 방법의 하나로, **TCP 세션 관리 취약점** 을 이용한 공격. TCP 는 연결 과정을 통해 상호 간에 세션을 생성한 후 IP·Port·Sequence Number·Acknowledgment Number 를 통해 상호 인식한다.

#### 3-1-1. 공격 절차 (7단계)

1. 공격자는 ARP 스푸핑을 통해 클라이언트와 서버 사이의 TCP 통신을 스니핑
2. 공격자는 희생자의 IP 주소를 사용하여 위조된 TCP 패킷을 생성하여 서버로 전송
3. 서버는 이를 정상적인 패킷으로 간주하고 Ack.Num 을 증가시켜 응답
4. 희생자는 Ack.Num 이 증가된 응답을 받고 서버에 교정용 ACK 패킷을 전송
5. 서버는 교정용 ACK 패킷에 대해 계속해서 응답하여 **ACK Storm** 을 유발
6. 공격자는 희생자에게 RST 메시지를 전달하여 연결을 강제 종료
7. 공격자는 서버와의 세션을 유지

#### 3-1-2. 대응 방안

| 대응 | 동작 |
|---|---|
| **지속적인 인증** | 특정 행동을 하거나 일정 시간이 지나면 재인증을 요청 |
| **암호화 사용** | 데이터 전송 시 암호화를 사용하여 Sequence Number 를 추측하기 어렵게 함 (SSL/TLS, SSH, IPsec) |
| **ACK Storm 감지** | 패킷의 순서를 맞추기 위해 ACK 패킷을 보내는 과정을 감지 |

### 3-2. MITM (Man-In-The-Middle, 중간자 공격)

네트워크 통신을 조작하여 통신 내용을 도청·조작하는 기법. 두 사람은 각각 상대방에게 정상적으로 연결되었다고 생각하지만, 실제로는 두 사람이 모두 중간자와 연결되어 있으며 중간자가 한쪽에서 전달된 정보를 도청·조작한 후 다른 쪽으로 전달한다.

> SSL/TLS 의 인증서 검증·서명 메커니즘은 MITM 방어를 위한 핵심이다. 이를 우회하는 공격(예: SSL Stripping) 과 SSL/TLS 취약점은 [07 단원](07.security-protocols.md) 에서 다룬다.

### 시험 포인트

- TCP 세션 하이재킹 = ARP 스푸핑(스니핑) → IP 스푸핑(위조 패킷 송신) → ACK Storm 유발 → RST 로 희생자 연결 종료
- 핵심 식별 정보 — IP / Port / Seq.Num / Ack.Num
- 대응 = 지속적 인증 + **암호화** + ACK Storm 감지
- MITM 의 두 사람은 중간자와 각각 연결된 상태

## 4. 스캐닝 (Scanning) — 풋프린팅·스캐닝·목록화

시스템에 직접적으로 침입하기 전에 목표 호스트에 대한 정보를 수집하는 활동. 일반적인 사이버 킬 체인의 첫 단계이다.

### 4-1. 단계별 정의

| 단계 | 정의 | 활동 |
|---|---|---|
| **풋프린팅(Footprinting)** | 공격 전 공격대상에 대한 다양한 정보를 수집하기 위해 널리 사용하는 방법 | 사회공학 기법으로 비밀번호 등을 가로채기 / 스스로 공개된 여러 정보를 수집 |
| **스캐닝(Scanning)** | 실제 공격방법을 결정하거나 공격에 이용될 수 있는 네트워크 구조·서비스 등의 정보를 얻기 위한 활동 | Sweep / Open Scan / Stealth Scan / Security Scan |
| **목록화(Enumeration)** | 수집한 정보를 바탕으로 좀 더 실용적인 정보를 얻는 과정 | 공유 자원 / 사용자·그룹 / 응용 프로그램 목록화 |

### 4-2. 스캐닝 분류

| 종류 | 의미 |
|---|---|
| **Sweep** | IP 주소와 네트워크 범위를 조사 |
| **Open Scan** | 시스템과 포트의 활성화 여부를 확인 |
| **Stealth Scan** | TCP 헤더를 조작하여 은밀하게 스캔하여 포트의 활성화 여부를 확인 (스캔 대상에 로그가 남지 않음) |
| **Security Scan** | 시스템의 보안 취약점을 조사 |

### 4-3. 스캐닝으로 수집되는 정보

- 공격대상 네트워크의 보안장비 사용 현황
- 우회 가능 네트워크 구조
- 해당 네트워크 내의 시스템 플랫폼 형태
- 시스템 운영체제·커널 버전
- 제공 서비스의 종류

### 4-4. 목록화 분류

- 공유 자원 목록화
- 사용자 및 그룹 목록화
- 응용 프로그램 목록화

### 시험 포인트

- 풋프린팅(공개 정보 수집) → 스캐닝(능동 탐지) → 목록화(실용 정보화) 의 3단계 순서
- Stealth Scan 의 정의 — TCP 헤더 조작으로 로그 없이 스캔
- 4종 스캐닝 분류와 의미 — Sweep / Open / Stealth / Security

## 5. 포트 스캐닝 도구 — Nmap / 방화벽 정책

### 5-1. Nmap 옵션 정리[^nmap-ref]

대상의 포트 및 서비스를 스캔하는 데 사용되는 도구. 침입 전 취약점을 분석하기 위한 사전 작업 중 하나로 포트 스캐닝을 통해 대상 호스트의 동작 여부와 제공 서비스 등을 확인한다.

| 옵션 | 동작 |
|---|---|
| `-sS` | TCP **SYN Scan** (Half-Open) — 관리자 권한 필요, 로그 안 남음 |
| `-sT` | TCP **Connect Scan** — 일반 사용자 권한, 시스템 로그에 IP 기록됨 |
| `-sU` | UDP Scan |
| `-sF` | TCP **FIN Scan** — FIN 비트만 설정 |
| `-sX` | TCP **Xmas Scan** — URG, PSH, FIN 비트 조합 |
| `-sN` | TCP **NULL Scan** — 제어비트 설정 없음 |
| `-sA` | TCP **ACK Scan** — 방화벽 룰셋(필터링 정책) 확인용 |
| `-sP` | Ping 을 이용한 호스트 활성화 확인 (현재 옵션은 `-sn`) |
| `-D` | **Decoy 스캔** — 다양한 위조 IP 로 스캔, 실제 스캐너 주소 식별 어렵게 함 |
| `-b` | FTP Bounce 스캔 |
| `-O` | 대상 호스트의 운영체제 정보 출력 |
| `-F` | 빠른 네트워크 스캐닝 |
| `-T0 ~ T5` | 아주 느리게(0) ~ 아주 빠르게(5) 스캐닝 (timing template) |

### 5-2. 포트 스캔 대응

| 대응 | 동작 |
|---|---|
| **패킷 차단** | 불필요한 패킷을 차단하도록 방화벽 설정 |
| **포트 점검** | 사용하지 않는 포트는 열어놓지 않음 (불필요한 서비스 종료) |
| **스캔 탐지** | 침입탐지시스템(IDS) 을 사용해 포트 스캔 탐지 |
| **로그 감사** | 시스템 로그 감사 |

### 5-3. iptables DROP vs REJECT 정책

| 정책 | 공격자 → 방화벽 | 방화벽 응답 |
|---|---|---|
| **iptables DROP** | 스캔 패킷 (TCP, UDP 등) | 응답 없음 (들어온 패킷을 무시) |
| **iptables REJECT** | 스캔 패킷 (TCP, UDP 등) | ICMP `Destination Unreachable` 응답 (거부 메시지 전송) |

> 일반적으로 보안상 DROP 정책을 사용한다. iptables 자세한 사용법은 [08 단원 §iptables](08.security-solutions-and-monitoring.md) 에서 다룬다.

### 시험 포인트

- Nmap 핵심 옵션 — `-sS`(SYN, 관리자) / `-sT`(Connect, 일반) / `-sU`(UDP) / `-sF·sX·sN`(Stealth) / `-sA`(ACK, 방화벽 룰 확인) / `-D`(Decoy)
- iptables **DROP**(응답 없음) vs **REJECT**(ICMP Unreachable) 의 차이
- 포트 스캔 대응 4종 — 패킷 차단 / 포트 점검 / 스캔 탐지 / 로그 감사

## 6. TCP / UDP 스캔 방식 — 동작 흐름 정리

각 스캔 방식의 패킷 흐름을 포트 오픈/닫힘 상태별로 정리한다.

### 6-1. TCP Connect (Open) Scan — `-sT`

일반 사용자 권한으로 TCP 포트 오픈 여부를 확인. **Connect 시스템 호출** 을 이용하는 방식으로 정상적인 TCP 연결설정을 수행하여 스캔. **시스템 로그에 스캐너 IP 가 기록됨.**

```
포트 오픈 :
  공격자 → SYN
  타겟서버 → SYN+ACK
  공격자 → ACK     (시스템 로그에 IP 기록)
  공격자 → RST+ACK (포트 오픈 확인 후 즉시 중단)

포트 닫힘 :
  공격자 → SYN
  타겟서버 → RST+ACK (해당 포트가 닫혀 있음)
```

### 6-2. TCP SYN (Half-Open) Scan — `-sS`

관리자 권한으로 TCP 패킷 자체를 직접 조작하여 TCP 오픈 여부를 판단. **TCP 연결설정이 완전하게 이루어지지 않기 때문에 스캔 대상 시스템에 로그가 남지 않음** → 공격자가 자신의 IP 노출을 피할 수 있음.

```
포트 오픈 :
  공격자 → SYN
  타겟서버 → SYN+ACK
  공격자 → RST     (강제 종료, 연결 미수립)

포트 닫힘 :
  공격자 → SYN
  타겟서버 → RST+ACK
```

### 6-3. TCP FIN / NULL / Xmas Scan — `-sF / -sN / -sX`

관리자 권한으로 TCP 패킷 자체를 직접 조작하여 TCP 오픈 여부를 판단. **TCP 연결설정이 완전하게 이루어지지 않기 때문에 스캔 대상 시스템에 로그가 남지 않음**. RFC 793 기준으로 **닫힌 포트** 만 RST 응답을 보내야 한다는 규약을 이용.[^rfc793-stealth]

| 스캔 | 설정 비트 |
|---|---|
| **TCP FIN** | FIN |
| **TCP NULL** | (제어비트 설정 없음) |
| **TCP Xmas** | URG + PSH + FIN |

```
포트 오픈 :
  공격자 → FIN / No Flags Set / URG+PSH+FIN
  타겟서버 → 응답 없음

포트 닫힘 :
  공격자 → FIN / No Flags Set / URG+PSH+FIN
  타겟서버 → RST+ACK
```

### 6-4. TCP ACK Scan — `-sA`

대상 호스트의 포트 오픈 여부를 판단하는 것이 아닌, **방화벽 룰셋(필터링 정책) 을 알아내기 위한 스캔 방식**. 방화벽에 의한 차단·허용 상태를 확인.

```
포트 차단 상태 (방화벽이 막음) :
  공격자 → ACK
  방화벽 → 응답 없음 또는 ICMP Unreachable

포트 허용 상태 (방화벽 통과) :
  공격자 → ACK
  방화벽 → 포트 허용으로 인한 통과
  타겟서버 → RST   (정상적인 TCP 동작)
```

### 6-5. UDP Scan — `-sU`

대상 호스트의 UDP 포트 오픈 여부를 판단. 닫힌 UDP 포트로 패킷 수신 시 ICMP 에러 메시지로 응답한다는 점을 이용 (ICMP Type 3 Code 3 — Port Unreachable).

```
포트 오픈 :
  공격자 → UDP
  타겟서버 → 응답 없음

포트 닫힘 :
  공격자 → UDP
  타겟서버 → 응답 없음 또는 ICMP Unreachable (Type 3 Code 3)
```

### 6-6. Decoy Scan — `-D`

대상 호스트에서 스캐너 주소를 식별하기 어렵도록 실제 스캐너 주소 외에도 다양한 위조된 IP 로 스캔하는 방식.

```
포트 오픈 :
  공격자 → TCP SYN Scan(스캔 타입) + RND(Decoy 옵션)
  타겟서버 → SYN+ACK
  공격자 → RST    (강제 종료, 연결 미수립)

포트 닫힘 :
  공격자 → TCP SYN Scan + RND
  타겟서버 → RST+ACK
```

### 6-7. 스캔 방식 정리표

| 스캔 | 권한 | 로그 남음 | 식별 신호 (포트 오픈) | 식별 신호 (포트 닫힘) |
|---|---|---|---|---|
| TCP Connect (`-sT`) | 일반 | 남음 | 3-way 완성 후 RST | RST+ACK 즉시 |
| TCP SYN (`-sS`) | 관리자 | 안 남음 | SYN+ACK → 공격자 RST | RST+ACK |
| TCP FIN/NULL/Xmas | 관리자 | 안 남음 | **응답 없음** | RST+ACK |
| TCP ACK (`-sA`) | 관리자 | — | (포트 오픈 판별 X) | (방화벽 룰 판별) |
| UDP (`-sU`) | — | — | 응답 없음 | 응답 없음 또는 ICMP Type 3-3 |
| Decoy (`-D`) | 관리자 | (위조됨) | SYN+ACK → 공격자 RST | RST+ACK |

### 시험 포인트

- TCP Connect = 3-way 완성 + 시스템 로그 기록 (탐지됨)
- TCP SYN = 3-way 미완성 (Half-Open) + 로그 안 남음 (스텔스의 출발)
- TCP FIN/NULL/Xmas — **포트 오픈 시 응답 없음, 닫힘 시 RST+ACK** (RFC 793 기준)
- Xmas = URG + PSH + FIN (3개 비트)
- TCP ACK = 방화벽 룰셋 판별용 (포트 오픈 판별 아님)
- UDP 닫힌 포트 → ICMP Type 3 Code 3 (Port Unreachable)
- Decoy = 다수 위조 IP 로 추적 회피

## 단원 요약

1. **스니핑** : 수동적 공격. 허브 환경 = Promisc 만으로 가능 / 스위치 환경 = 5종(재밍 / ARP 스푸핑 / ARP 리다이렉트 / ICMP 리다이렉트 / SPAN). 탐지 5종(ping / ARP / DNS / 유인 / ARP Watch). 대응 = 암호화(SSL · S/MIME · SSH · VPN).
2. **ARP 공격** : 동일 네트워크 한정. ARP Spoofing(특정 호스트 MAC 위조 + 유니캐스트) vs ARP Redirect(라우터 MAC 위조 + 브로드캐스트). IP Forward 활성화로 정상 통신 유지. 대응 = 정적 캐시 + 트래픽 모니터링.
3. **IP Spoofing** : 트러스트 관계(`hosts.equiv`·`.rhosts`) 악용, 케빈 미트닉 사례. 대응 = MAC 정적 + r 계열 서비스 권한 관리.
4. **ICMP Redirect** : Type 5 메시지로 라우팅 테이블 변조. 대응 = `sysctl net.ipv4.conf.all.accept_redirects=0`. ARP Cache(2계층) vs 라우팅 테이블(3계층) 변조 차이.
5. **TCP 세션 하이재킹** : ARP 스푸핑 → IP 위조 → ACK Storm 유발 → RST. 대응 = 지속적 인증 + 암호화 + ACK Storm 감지. 핵심 식별 정보 — IP/Port/Seq.Num/Ack.Num.
6. **스캐닝 3단계** : 풋프린팅(수동) → 스캐닝(능동) → 목록화. 스캐닝 4종 — Sweep / Open / Stealth / Security.
7. **Nmap 옵션** : `-sS`(SYN) / `-sT`(Connect) / `-sU`(UDP) / `-sF·sN·sX`(Stealth) / `-sA`(방화벽 룰) / `-D`(Decoy) / `-O`(OS 핑거프린팅).
8. **방화벽 정책** : iptables DROP(응답 없음) vs REJECT(ICMP Unreachable).
9. **TCP 스캔 응답** : Connect → 3-way 완성+로그 / SYN → SYN+ACK 후 RST / FIN·NULL·Xmas → 오픈 시 응답 없음·닫힘 시 RST+ACK / ACK → 방화벽 룰 판별 / UDP → 닫힘 시 ICMP Type 3-3.
