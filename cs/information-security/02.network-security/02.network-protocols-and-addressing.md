## 1. ARP / RARP / GARP

### 1-1. 정의

| 프로토콜 | 변환 방향 | 비고 |
|---|---|---|
| **ARP** (Address Resolution Protocol) | 논리 주소(IP) → 물리 주소(MAC) | 인터넷 계층의 IP 주소를 데이터 링크 계층의 MAC 주소로 변환 |
| **RARP** (Reverse ARP) | 물리 주소(MAC) → 논리 주소(IP) | 디스크리스 워크스테이션 등 자신의 IP 를 모르는 노드가 사용. 현재는 DHCP 가 RARP 의 역할을 대체 |
| **GARP** (Gratuitous ARP) | (자신의 정보 announce) | 출발지 IP 와 목적지 IP 가 동일한 ARP 요청. "불필요한" ARP 라는 의미로 IP 충돌 감지·재부팅·IP 변경 시 사용 |

### 1-2. ARP 동작 방식

1. 송신자가 대상 노드의 MAC 주소를 알지 못하므로 **ARP Request** 를 브로드캐스트 (2계층 이더넷 프레임의 목적지 MAC 은 48bit 가 모두 1 인 `FF:FF:FF:FF:FF:FF` 주소 사용)
2. 해당 IP 를 가진 노드가 자신의 MAC 정보를 담아 **ARP Reply** 를 유니캐스트로 응답
3. 각 시스템은 ARP 요청을 통해 알아낸 MAC 정보를 ARP Cache 에 일정 시간(통상 1~2분) 저장

### 1-3. RARP 동작 방식

1. 송신자가 자신의 IP 를 알지 못하므로 자신의 MAC 주소를 담아 **RARP Request** 를 RARP 서버에 브로드캐스트
2. RARP 서버가 요청자의 IP 를 담아 **RARP Reply** 를 유니캐스트로 응답

### 1-4. GARP

- 출발지 IP 와 목적지 IP 가 동일한 ARP 요청
- 자신의 MAC 정보를 동일 네트워크상의 다른 장비들에게 알림(announce)
- **IP 충돌 감시** : IP 충돌 여부를 GARP 를 통해 검색
- **상대방의 ARP Cache 갱신** : GARP 메시지를 수신한 쪽은 자신의 ARP Cache 의 매핑 정보를 갱신
- 호스트의 IP 를 변경하거나 재부팅 시에 GARP 패킷을 생성

### 1-5. ARP Cache 테이블 — 동적/정적

| 타입 | 설명 |
|---|---|
| **dynamic** | ARP 요청을 통해 알아낸 MAC 정보. 통상 1~2분간 메모리에 저장되며 시간이 지나면 만료됨 |
| **static** | 관리자가 직접 설정. 삭제하거나 시스템이 종료되기 전까지 지속적으로 유지됨 |

### 1-6. ARP 명령어

| 명령어 | 동작 |
|---|---|
| `arp -a` | ARP Cache 테이블 내용 확인 |
| `arp -d` | ARP Cache 테이블 삭제 (리눅스의 경우 뒤에 IP 명시 필요) |
| `arp -s [IP] [MAC]` | 정적(static) 으로 ARP 항목 설정 |

### 1-7. 브로드캐스트·유니캐스트·멀티캐스트

| 통신 방식 | 의미 |
|---|---|
| **유니캐스트(Unicast)** | 1 : 1 로 패킷을 전송 |
| **브로드캐스트(Broadcast)** | 로컬 네트워크 내의 모든 호스트에 동시에 패킷을 전송. 2계층 이더넷 목적지 MAC 은 `FF:FF:FF:FF:FF:FF` |
| **멀티캐스트(Multicast)** | 데이터를 전송받기 원하는 1 : 다수 통신. IPv4 클래스 D(`224.0.0.0` ~ `239.255.255.255`) |
| **애니캐스트(Anycast)** | 클라이언트와 가장 가까운 PC 가 응답하는 1 : 1 통신. IPv6 에서 도입[^rfc4291] |

### 시험 포인트

- ARP : IP→MAC, RARP : MAC→IP, GARP : 자신의 정보 announce
- ARP Request 는 브로드캐스트, ARP Reply 는 유니캐스트
- ARP Cache 의 dynamic(1~2분 TTL) vs static(관리자 설정 시까지 유지)
- ARP 명령어 3종 (`arp -a` / `arp -d` / `arp -s [IP] [MAC]`)
- 이더넷 브로드캐스트 MAC = `FF:FF:FF:FF:FF:FF` (48bit 모두 1)
- DHCP 가 RARP 의 역할을 대체

## 2. IP 프로토콜

### 2-1. 정의·특성

| 특성 | 의미 |
|---|---|
| **비연결형 프로토콜** | 연결 상태를 유지하지 않기 때문에 패킷 전송 순서를 보장하지 않음 |
| **비신뢰적 프로토콜** | 신뢰성 있는 통신을 보장하지 않음 (재전송·확인 응답 없음) |
| **라우팅 담당** | 인터넷 계층의 대표 프로토콜로 목적지 주소 기반 라우팅 |

### 2-2. IPv4 헤더 필드

기본 헤더 길이는 20byte, 옵션 포함 시 최대 60byte (4byte 단위로 0~40byte 옵션 추가).[^rfc791]

| 필드 | 길이 | 의미 |
|---|---|---|
| Version | 4bit | IP 버전 (IPv4 = 4) |
| HLEN (Header Length) | 4bit | 헤더 길이 (4byte 단위) |
| Type of Service (TOS) | 8bit | QoS 우선순위·서비스 종류 |
| Total Length | 16bit | 헤더 + 데이터 전체 길이 |
| **Identification** | 16bit | 단편화 전 원본 IP 데이터그램을 식별하는 ID |
| **Flags** | 3bit | 단편화 제어 — 1번째 미사용 / 2번째 DF(Don't Fragment) / 3번째 MF(More Fragments) |
| **Fragmentation Offset** | 13bit | 순서대로 조합하기 위한 현재 단편의 상대 위치 (8byte 단위) |
| **TTL** (Time To Live) | 8bit | IP 패킷 생존 시간. 라우터/L3 스위치 통과마다 1씩 감소, 0 이 되면 폐기 (무한 루핑 방지) |
| Protocol | 8bit | 상위 프로토콜 식별 (TCP=6, UDP=17, ICMP=1) |
| Header Checksum | 16bit | IP 헤더 오류 검증 |
| Source IP | 32bit | 송신자 IP |
| Destination IP | 32bit | 수신자 IP |
| Options | 0~40byte | 옵션 헤더 |

#### 2-2-1. TTL 초기값 (OS 핑거프린팅)

| OS | TTL 초기값 |
|---|---|
| **Linux** | 64 |
| **Windows** | 128 |
| **Unix** | 255 |

OS 별로 초기값이 다르므로 OS 핑거프린팅 용도로도 활용된다.

#### 2-2-2. Flags 비트 의미

- **DF (Don't Fragment) 비트** : 1 → 패킷을 단편화하지 말라는 뜻. 단편화 필요 시 라우터가 ICMP `Fragmentation needed` 에러 반환
- **MF (More Fragments) 비트** : 1 → 재조합할 단편이 더 남아 있음. 0 → 마지막 단편

### 2-3. IPv4 주소체계

#### 2-3-1. 클래스 기반 주소 체계

| 클래스 | 첫 옥텟 범위 | 기본 마스크 | 용도 |
|---|---|---|---|
| **A** | 0 ~ 127 | /8 (255.0.0.0) | 대규모 네트워크 |
| **B** | 128 ~ 191 | /16 (255.255.0.0) | 중규모 네트워크 |
| **C** | 192 ~ 223 | /24 (255.255.255.0) | 소규모 네트워크 |
| **D** | 224 ~ 239 | — | **멀티캐스트** 전용 |
| **E** | 240 ~ 255 | — | 연구·예약(reserved) |

#### 2-3-2. CIDR / VLSM

| 개념 | 설명 |
|---|---|
| **CIDR** (Classless Inter-Domain Routing) | 클래스 주소의 개념을 무시하고 효율적으로 IP 주소를 할당. `192.168.0.0/24` 형식의 prefix 표기[^rfc4632] |
| **VLSM** (Variable-Length Subnet Mask) | 서브네팅을 여러 번 반복하여 서로 다른 길이의 서브넷 마스크를 사용. 하나의 네트워크를 효율적으로 분할[^rfc1878] |
| **Subnetting** | 하나의 네트워크를 여러 서브넷으로 분할하는 기술 |

#### 2-3-3. 공인주소 vs 사설주소

| 구분 | 의미 | 대표 |
|---|---|---|
| **공인주소** | 인터넷에서 고유하게 식별 가능한 주소 | ICANN/IANA 가 관리 |
| **사설주소** | 사설망 내부에서만 사용하는 주소. 인터넷으로 라우팅되지 않음[^rfc1918] | A: `10.0.0.0/8` / B: `172.16.0.0/12` / C: `192.168.0.0/16` |

#### 2-3-4. 정적 vs 동적 주소 할당

| 방식 | 설명 |
|---|---|
| **정적(Static)** | 관리자가 IP 를 수동 할당 |
| **동적(Dynamic)** | DHCP 등을 통해 자동 할당 — 임대 시간(lease time) 동안 IP 사용 |

### 2-4. IP 단편화 / MTU

| 용어 | 의미 |
|---|---|
| **MTU** (Maximum Transmission Unit) | 물리적인 네트워크 프로토콜 프레임의 데이터부의 최대 크기. 이더넷 MTU = 1500byte |
| **캡슐화** | IP 패킷이 프레임으로 캡슐화되기 때문에 MTU 를 초과하는 크기가 될 수 없음 |
| **단편화** | IP 패킷이 MTU 보다 크면 1500byte 이하로 단편화하여 전송 |
| **재조합** | 중계 구간의 MTU 가 달라도 재조합되지 않으며 **최종 목적지에서만 재조합** |

#### 2-4-1. 단편화 동작 예시 (Payload 3008byte 분할)

| 단편 | tcpdump 형식 | Payload | Offset |
|---|---|---|---|
| 1 | `frag 22666:1480@0+` | 1480 | 0 (MF=1) |
| 2 | `frag 22666:1480@1480+` | 1480 | 1480 (MF=1) |
| 3 | `frag 22666:48@2960` | 48 | 2960 (MF=0, 마지막 단편) |

tcpdump 출력 형식 — `frag "단편ID":"단편의크기(IP헤더제외)"@"오프셋"+`. `+` 가 있으면 추가 단편 있음(MF=1), 없으면 마지막 단편(MF=0).

#### 2-4-2. 재조합 실패

단편의 손실이 발생하면 ICMP `Fragment reassembly time exceeded` 메시지를 최초 출발지로 전송하고 재조합에 실패한 모든 단편을 폐기한다. (자세히는 §3-2 참조)

### 2-5. IPv6

IPv4 의 32bit 주소가 고갈되는 문제를 해결하기 위해 도입된 128bit 주소 체계.[^rfc8200] 16bit 단위로 구분하며 각 단위는 16진수로 변환되어 콜론으로 구분 표기.

| 특징 | 설명 |
|---|---|
| **확대된 주소 공간** | 주소 길이 32bit → 128bit (2^128 개 주소) |
| **단순해진 헤더 포맷** | IPv4 의 불필요한 헤더 필드 제거 → 빠른 처리 가능 |
| **간편해진 주소 설정** | 내장된 주소 자동 설정(SLAAC) → 플러그 앤 플레이 |
| **강화된 보안성** | IPsec 기능을 기본 사항으로 제공 |
| **개선된 모바일 IP** | 헤더에서 이동성 지원 |

#### 2-5-1. IPv6 전송방식

| 방식 | 설명 |
|---|---|
| **유니캐스트** | 1 : 1 통신 |
| **멀티캐스트** | 데이터를 전송받기 원하는 1 : 다수 통신 |
| **애니캐스트** | 클라이언트와 가장 가까운 PC 가 응답하는 1 : 1 통신 (IPv6 에서 도입) |

> IPv6 는 IPv4 의 브로드캐스트를 제거하고 그 역할을 멀티캐스트로 대체했다.[^rfc4291]

#### 2-5-2. IPv4 ↔ IPv6 전환 기술

| 기술 | 설명 |
|---|---|
| **듀얼 스택(Dual Stack)** | IPv4 와 IPv6 를 동시에 설정하여 통신 상대에 따라 선택적으로 이용 |
| **터널링(Tunneling)** | IPv4 네트워크를 경유하여 IPv6 네트워크 간 통신 — IPv6 패킷을 IPv4 패킷으로 캡슐화 |
| **주소/헤더 변환(Translation)** | IPv4 ↔ IPv6 주소를 상호 변환하여 통신 (NAT64 등) |

### 2-6. IP 라우팅 규칙·검색

라우팅 동작은 [01 단원 §4-6](01.network-fundamentals.md) 에서 정리했다. 본 단원에서는 IP 패킷 관점에서 한 번 더 요약한다.

**IP 라우팅 규칙**
1. 목적지 주소가 자신과 동일한 네트워크에 있다면 직접 전송
2. 동일한 네트워크에 속하지 않는다면 1차 경유지 주소(게이트웨이) 를 라우팅 테이블을 참조하여 찾음
3. 목적지 주소가 자신이라면 상위 계층으로 데이터를 전달

**라우팅 테이블 검색 방식**
1. IP 패킷의 목적지 IP 와 라우팅 테이블의 netmask/genmask 를 bit AND(&) 연산
2. 연산 수행 후 라우팅 테이블의 destination 필드와 비교
3. 일치하는 경로를 선택하여 패킷을 전송

**검색 우선순위** (호스트 주소 → 네트워크 주소 → default gateway)
1. 먼저 목적지 호스트 주소와 일치하는 경로를 찾음
2. 일치하는 경로가 없으면 목적지 네트워크 주소와 일치하는 경로를 찾음
3. 일치하는 경로가 없으면 default gateway(`0.0.0.0`) 로 보냄

`netstat -rn` 명령으로 호스트의 라우팅 테이블 정보를 확인할 수 있다.

### 2-7. 전송방식과 캡슐화 — 통합 정리

출제기준 1-1-3 가 명시한 "Multicast / Broadcast / 데이터 캡슐화" 항목을 한 곳에서 정리한다.

| 전송방식 | IPv4 | IPv6 |
|---|---|---|
| **유니캐스트(Unicast)** | ○ (1:1) | ○ |
| **브로드캐스트(Broadcast)** | ○ (로컬 네트워크 전체, 이더넷 MAC `FF:FF:FF:FF:FF:FF`) | × (제거) |
| **멀티캐스트(Multicast)** | ○ (클래스 D, `224.0.0.0` ~ `239.255.255.255`) | ○ (브로드캐스트 역할 흡수) |
| **애니캐스트(Anycast)** | × | ○ (가장 가까운 노드 1:1)[^rfc4291] |

데이터 캡슐화·역캡슐화·다중화·역다중화의 정의와 동작은 [01 단원 §1-4·§1-5](01.network-fundamentals.md) 에서 다루었다. 본 단원의 IP·TCP·UDP 헤더 필드를 학습할 때, 각 헤더가 캡슐화 과정에서 어느 계층에서 추가되는지를 함께 떠올리면 이해가 빠르다.

### 시험 포인트

- IP 프로토콜의 두 가지 특성 — **비연결형(순서 보장 X) + 비신뢰적(신뢰성 X)**
- IPv4 헤더 필드 — Identification(16bit) / Flags(3bit, DF·MF) / Fragmentation Offset(13bit) / TTL(8bit)
- TTL 초기값 — Linux 64 / Windows 128 / Unix 255 (OS 핑거프린팅)
- 클래스 D(`224.0.0.0~239.255.255.255`) = 멀티캐스트 / E = 예약
- CIDR vs VLSM — CIDR 는 클래스 무시 prefix 표기, VLSM 은 서브넷마다 다른 마스크
- 사설주소(RFC 1918) 3개 대역 — 10/8, 172.16/12, 192.168/16
- IPv6 — 128bit, 16bit 콜론 구분, **IPsec 기본 제공**, 브로드캐스트 → 멀티캐스트로 대체, **애니캐스트** 도입
- IPv4 ↔ IPv6 전환 3종 — 듀얼 스택 / 터널링 / 주소·헤더 변환
- 단편화 — 이더넷 MTU = 1500byte, 재조합은 **최종 목적지에서만**, MF=1 더 있음, MF=0 마지막
- IPv4 전송방식 = 유니캐스트 / 브로드캐스트 / 멀티캐스트(클래스 D) — IPv6 는 브로드캐스트 제거 + 애니캐스트 추가

## 3. ICMP (Internet Control Message Protocol)

### 3-1. 정의

IP 프로토콜의 신뢰할 수 없는 단점을 보완하기 위한 프로토콜. 전송 중 오류 발생 시 에러 메시지를 생성하거나 네트워크 상태를 진단하는 기능을 제공한다. 헤더는 **Type 필드 + Code 필드** 로 구성된다.[^rfc792]

ICMP 메시지는 두 가지로 분류된다:

| 분류 | 용도 |
|---|---|
| **Error-Reporting Message** | 전송 중 오류 발생 시 에러 메시지 생성 |
| **Query Message** | 네트워크 상태 진단 |

### 3-2. ICMP Error-Reporting Message

| Type | 이름 | 의미 |
|---|---|---|
| **3** | Destination Unreachable | 해당 목적지에 도달 불가 |
| **5** | Redirection | 라우팅 경로가 잘못되어 새로운 경로를 알림 |
| **11** | Time Exceeded | 타임아웃이 발생하여 IP 패킷이 폐기 |

#### 3-2-1. Type 3 — Destination Unreachable Code

| Code | 의미 |
|---|---|
| 1 (Host Unreachable) | 최종 단계의 라우터가 목적지 호스트로 패킷 전송에 실패 |
| 2 (Protocol Unreachable) | 목적지 호스트에서 특정 프로토콜을 사용할 수 없는 경우 |
| 3 (Port Unreachable) | 목적지 호스트에 해당 UDP 포트가 열려있지 않은 경우 (TCP 의 경우 포트가 닫혔다면 RST 패킷을 반환) |
| 4 (Fragmentation needed and DF was set) | 단편화가 필요하지만 DF 플래그가 설정되어 단편화 불가 |

#### 3-2-2. Type 11 — Time Exceeded Code

| Code | 의미 |
|---|---|
| 0 (TTL exceeded in Transit) | IP 패킷이 최종 목적지에 도달하기 전 TTL 이 0 이 되어 폐기 |
| 1 (Fragment reassembly time exceeded) | IP 패킷 재조합 과정에서 타임아웃 발생, 모든 단편 폐기 |

#### 3-2-3. Type 5 — Redirection

라우팅 경로가 잘못되어 새로운 경로를 이전 경유지 또는 호스트에게 알려주는 메시지. 정상적으로는 라우터 간 라우팅 경로 재설정에 사용되지만, **공격자가 이를 악용하면 ICMP Redirect 공격**(희생자의 라우팅 테이블 변조 → 패킷 스니핑) 으로 사용 가능. (자세히는 [04 단원 §스푸핑](04.scanning-sniffing-spoofing.md) 에서 다룸)

### 3-3. ICMP Query Message

네트워크 상태를 진단하기 위한 쿼리 요청·응답 메시지.

| Type | 이름 | 용도 |
|---|---|---|
| **8** | Echo Request | 네트워크 및 호스트의 상태진단 요청 (`ping` 송신) |
| **0** | Echo Reply | 네트워크 및 호스트의 상태진단 응답 (`ping` 응답) |

### 시험 포인트

- ICMP = Type + Code 필드 구조
- Error-Reporting 3종 — Type 3(Destination Unreachable) / Type 5(Redirect) / Type 11(Time Exceeded)
- Query 2종 — Type 8(Echo Request) / Type 0(Echo Reply)
- TCP 포트 닫힘 → RST 반환 vs UDP 포트 닫힘 → ICMP Type 3 Code 3(Port Unreachable) 반환
- ICMP Redirect 메시지(Type 5) 가 ICMP Redirect 공격에 악용됨

## 4. TCP (Transmission Control Protocol)

### 4-1. 정의·서비스

| 특성 | 설명 |
|---|---|
| **연결지향(Connection-Oriented)** | 물리적으로 전용회선이 연결되어 있는 것처럼 가상의 연결통로(가상회선) 를 설정해서 통신 |
| **순서제어** | 논리적인 연결통로를 통해 데이터의 전송순서 보장 |
| **스트림 기반 전송** | 데이터를 임의의 크기로 나누어 연속해서 전송 (정해진 크기로 전송하지 않음) |
| **신뢰적(Reliable)** | 흐름·오류·혼잡 제어 수행 |
| **전이중(Full Duplex)** | 데이터가 동시에 양방향으로 전달 가능 |
| **프로세스-대-프로세스 통신** | 16bit 포트 번호로 프로세스 간 통신 식별 |

#### 4-1-1. 신뢰성 보장 메커니즘

| 메커니즘 | 설명 |
|---|---|
| **흐름제어** | 슬라이딩 윈도우 — 상대방이 받을 수 있는 윈도우 사이즈 내에서 ACK 를 기다리지 않고 데이터를 연속 전송 |
| **오류제어** | 오류·누락 발생 시 재전송 수행 |
| **혼잡제어** | 데이터 누락이 있을 경우 혼잡 상태로 판단하여 전송량 조절 |

### 4-2. TCP 헤더 구조

기본 헤더 길이는 20byte (옵션 포함 최대 60byte).[^rfc9293]

| 필드 | 길이 | 의미 |
|---|---|---|
| Source Port | 16bit | 송신자 포트 |
| Destination Port | 16bit | 수신자 포트 |
| **Sequence Number** | 32bit | 송신 데이터의 순서 번호 |
| **Acknowledgment Number** | 32bit | 수신 확인 응답 번호 = 상대방이 다음에 보낼 패킷의 순서번호 |
| HLEN (Data Offset) | 4bit | 헤더 길이 (4byte 단위) |
| Reserved | 6bit | 예약 |
| **Control Flags** | 6bit | URG / ACK / PSH / RST / SYN / FIN |
| **Window Size** | 16bit | 수신측이 송신측에 알리는 수신 버퍼 여유 공간 → 흐름제어용 |
| Checksum | 16bit | 오류 검증 |
| **Urgent Pointer** | 16bit | URG 플래그 설정 시 긴급 데이터의 위치값 |
| Options | 0~40byte | MSS·Window Scale 등 |

### 4-3. 포트 분류

포트는 상위 어플리케이션·서비스·소켓을 식별하기 위한 16bit 식별자로 IANA 가 관리한다.

| 분류 | 범위 | 용도 |
|---|---|---|
| **Well-known port** | 0 ~ 1023 | 표준 서비스 (HTTP 80, SSH 22 등). 관리자 권한 필요 |
| **Registered port** | 1024 ~ 49151 | DBMS 등 등록된 서비스 |
| **Dynamic / Private port** | 49152 ~ 65535 | 일반 클라이언트 임시 포트 |

> *원본 표기 보정*: 2과목.txt L295 의 `dynamic port(49142 ~ 65535)` 는 OCR 결함으로 IANA 표준값 `49152 ~ 65535` 로 보정했다.[^iana-ports]

### 4-4. Control Flags (상태 플래그)

| 플래그 | 비트 표기 | 의미 |
|---|---|---|
| **FIN** | 000001 | 정상 연결 종료 |
| **SYN** | 000010 | 연결 설정 (순서번호 동기화) |
| **RST** | 000100 | 연결 중단 (강제 종료) |
| **PSH** | 001000 | 송수신 버퍼에 있는 데이터를 즉시 처리 |
| **ACK** | 010000 | 수신 확인 응답 |
| **URG** | 100000 | 긴급 데이터 설정 |

### 4-5. 3-way Handshake — 연결 설정

```
1. 클라이언트(SYN) → 서버
   Seq.Num : 1000
   Ack.Num : —
2. 서버(SYN+ACK) → 클라이언트
   Seq.Num : 2000
   Ack.Num : 1001       (← 클라이언트가 다음에 보낼 순서번호)
3. 클라이언트(ACK) → 서버
   Seq.Num : 1001
   Ack.Num : 2001
```

- 클라이언트가 능동적으로 연결을 시작하므로 **Active Open**, 서버는 **Passive Open**
- 초기 순서번호(ISN) 는 랜덤한 값으로 시작
- 상태 변화 : `SYN_SENT` → `SYN_RCVD` → `ESTABLISHED`

### 4-6. 4-way Handshake — 연결 종료

```
1. 클라이언트(FIN+ACK) → 서버
   Seq.Num : 2200
   Ack.Num : 3300
2. 서버(ACK) → 클라이언트
   Seq.Num : 3300
   Ack.Num : 2201
3. 서버(FIN+ACK) → 클라이언트
   Seq.Num : 3300
   Ack.Num : 2202
4. 클라이언트(ACK) → 서버
   Seq.Num : 2202
   Ack.Num : 3301
```

- 능동적 종료 = **Active Close**, 수동적 종료 = **Passive Close**
- 클라이언트는 마지막 ACK 전송 후 즉시 종료하지 않고 **2MSL 시간 동안 대기** — 마지막 ACK 가 분실되어 서버가 FIN+ACK 를 재전송할 경우를 대비
- 상태 변화 (RFC 9293 §3.3.2 TCP 상태 다이어그램) :
  - **Active Close 측** : `ESTABLISHED` → `FIN_WAIT_1` → `FIN_WAIT_2` → `TIME_WAIT` → `CLOSED`
  - **Passive Close 측** : `ESTABLISHED` → `CLOSE_WAIT` → `LAST_ACK` → `CLOSED`

### 4-7. 연결 거부·연결 중단

| 상황 | 패킷 흐름 | 상태 변화 |
|---|---|---|
| **연결 요청 거부(강제 종료)** : 서버에 해당 포트가 listen 상태가 아닐 때 | 클라이언트(SYN) → 서버(RST+ACK) | `SYN_SENT` → `CLOSED` |
| **연결 중단(abort)** : ESTABLISHED 상태에서 즉시 끊을 때 | RST+ACK 또는 RST | `ESTABLISHED` → `CLOSED` |

### 4-8. MSS — 분할과 재조합

| 용어 | 정의 |
|---|---|
| **MSS** (Maximum Segment Size) | TCP 세그먼트 데이터부의 최대 바이트 수 |
| 계산식 | **MSS = MTU − IP 헤더(20byte) − TCP 헤더(20byte)** |

연결 설정 과정(SYN) 의 TCP 옵션 헤더 MSS 필드를 통해 상호간에 MSS 정보를 주고받는다. Application 계층에서 전달된 데이터는 TCP 계층에서 MSS 크기에 따라 분할되며, 분할된 세그먼트는 수신 측에서 재조합되어 Application 계층으로 전달된다.

### 4-9. 재전송과 빠른 재전송

| 메커니즘 | 동작 |
|---|---|
| **재전송 (Retransmission)** | 모든 세그먼트 전송 시마다 재전송 타이머를 동작. 재전송 타임아웃 동안 ACK 가 없으면 송신자는 패킷이 누락된 것으로 판단하고 재전송. 혼잡제어 관점에서 매우 혼잡한 상태로 판단 |
| **빠른 재전송 (Fast Retransmit)** | 수신 측이 기대했던 순서번호가 아닌 그 이후 순서번호의 세그먼트를 수신하면 즉시 기대하는 순서번호의 ACK 패킷을 전송. 송신측이 동일 ACK 를 3회 받으면 누락으로 간주하고 즉시 재전송. 일반 재전송보다 덜 혼잡한 상태로 간주 |

### 시험 포인트

- TCP 의 5대 특성 — 연결지향 / 순서제어 / 스트림 / 신뢰적(흐름·오류·혼잡 제어) / 전이중
- TCP 헤더의 핵심 필드 — Source/Dest Port(16bit), Sequence Number(32bit), Acknowledgment Number(32bit), Control Flags(6bit), Window Size(16bit)
- 6대 Control Flag — FIN(000001) / SYN(000010) / RST(000100) / PSH(001000) / ACK(010000) / URG(100000)
- 3-way Handshake — `SYN → SYN+ACK → ACK`
- 4-way Handshake — `FIN+ACK → ACK → FIN+ACK → ACK` (4번)
- 연결 거부 = `SYN → RST+ACK` / 연결 중단 = `RST+ACK` 또는 `RST`
- TIME_WAIT 상태에서 2MSL 대기 이유 — 마지막 ACK 분실 대비
- MSS = MTU − IP 헤더(20) − TCP 헤더(20) → 이더넷 기본값 = 1500 − 40 = 1460byte
- 포트 분류 — Well-known(0~1023) / Registered(1024~49151) / Dynamic(49152~65535)

## 5. UDP (User Datagram Protocol)

### 5-1. 정의·특성

| 특성 | 설명 |
|---|---|
| **비연결형** | 데이터그램 전송 시마다 주소 정보를 설정해서 전송. 순차적 전송을 보장하지 않음 |
| **비신뢰적** | 오류·혼잡 제어를 수행하지 않음 |
| **데이터그램 기반** | 데이터 크기가 정해진 전송 방식 |
| **가벼움·빠름** | 최소한의 오버헤드 → 단순하고 전송 속도가 빠름 |
| **프로세스 식별** | IP 기반 포트 정보를 이용하여 상위 송수신 어플리케이션을 식별 |

비신뢰적이라 대량 데이터 송수신은 부적절하지만, 한 번의 패킷 송수신으로 완료되는 서비스에 많이 사용된다 — DNS(53/udp), NTP(123/udp), DHCP(67/68/udp), TFTP(69/udp), SNMP(161/udp).

### 5-2. UDP 헤더 구조

UDP 헤더는 **고정 8byte** (4개 필드 × 16bit).[^rfc768]

| 필드 | 길이 | 의미 |
|---|---|---|
| Source Port | 16bit | 송신자 포트 |
| Destination Port | 16bit | 수신자 포트 |
| Total Length | 16bit | UDP 헤더 + 데이터 전체 길이 |
| Checksum | 16bit | 전체 데이터그램의 오류 검증 |

### 시험 포인트

- UDP 의 4대 특성 — 비연결형 / 비신뢰적 / 데이터그램 / 가벼움
- UDP 헤더 = **8byte 고정**, 4개 필드 × 16bit
- TCP 헤더 = 20byte 기본 / UDP 헤더 = 8byte 고정 — 분량 차이 큼
- UDP 사용 서비스 — DNS / NTP / DHCP / TFTP / SNMP / RIP

## 6. NAT (Network Address Translation) — 외부 보강

출제기준 1-1-4 가 명시한 영역으로 통합자료에 분류 표가 없어 외부 보강한다.

### 6-1. 정의·필요성

NAT 는 라우터·방화벽 등의 장치에서 외부 네트워크와 내부 네트워크 간의 IP 주소 변환을 수행하는 기술이다.[^rfc2663][^rfc3022] 사설주소를 사용하는 내부 호스트가 인터넷의 공인주소 호스트와 통신할 때 주소를 변환해준다.

| 효과 | 설명 |
|---|---|
| **IPv4 주소 절약** | 내부 다수의 사설 IP 가 소수의 공인 IP 를 공유 |
| **보안 강화** | 외부에서 내부 네트워크로의 직접 접근 차단 → 자연스러운 패킷 필터링 |

### 6-2. NAT 분류

| 분류 | 동작 |
|---|---|
| **Static NAT** | 내부 사설 IP 와 외부 공인 IP 를 1:1 로 매핑 (고정) |
| **Dynamic NAT** | 공인 IP 풀(pool) 에서 동적으로 할당 (1:1 동적) |
| **PAT** (Port Address Translation, NAPT) | 다수 사설 IP 가 단일 공인 IP 를 포트 번호로 구분하여 공유 (N:1). 가장 일반적인 NAT 형태. 가정용 공유기·기업 방화벽에서 사용 |

### 6-3. 방향에 따른 분류

| 분류 | 동작 |
|---|---|
| **SNAT** (Source NAT) | 내부에서 외부로 나갈 때 출발지 IP 를 변환 (사설 → 공인) |
| **DNAT** (Destination NAT) | 외부에서 내부로 들어올 때 목적지 IP 를 변환 (공인 → 사설). 포트포워딩에 사용 |

### 시험 포인트

- NAT 의 두 가지 효과 — IPv4 주소 절약 + 보안 강화(직접 접근 차단)
- Static / Dynamic / **PAT(NAPT)** 의 차이 — PAT 가 N:1 공유로 가장 일반적
- SNAT (출발지 변환, 외부 향) vs DNAT (목적지 변환, 내부 향, 포트포워딩)
- 사설주소 RFC 1918 대역과 NAT 사용은 함께 묶여 출제됨

## 단원 요약

1. **ARP/RARP/GARP** : ARP=IP→MAC(브로드캐스트 Request, 유니캐스트 Reply), RARP=MAC→IP(현재는 DHCP 가 대체), GARP=자신 정보 announce(IP 충돌 감지). ARP Cache dynamic(1~2분) vs static(영구).
2. **IP** : 비연결형·비신뢰적. IPv4 헤더 — Identification / Flags(DF·MF) / Fragmentation Offset / TTL(Linux 64, Windows 128, Unix 255). 단편화는 MTU(이더넷 1500) 기준, **재조합은 최종 목적지에서만**.
3. **IPv4 주소체계** : 클래스 A/B/C 일반, D=멀티캐스트(224~239), E=예약(240~255). CIDR 는 클래스 무시 prefix, VLSM 은 가변 마스크. 사설주소 RFC 1918 = 10/8, 172.16/12, 192.168/16.
4. **IPv6** : 128bit, 16진수 콜론 표기, IPsec 기본 제공, 브로드캐스트 → 멀티캐스트로 대체, **애니캐스트** 도입. 전환 3종 — 듀얼 스택 / 터널링 / 주소·헤더 변환.
5. **ICMP** : Type+Code 구조. Error-Reporting — Type 3(Destination Unreachable) / Type 5(Redirect, 공격에 악용) / Type 11(Time Exceeded). Query — Type 8(Echo Request) / Type 0(Echo Reply).
6. **TCP** : 연결지향·순서제어·스트림·신뢰적·전이중. 헤더 20byte 기본. 6대 플래그 — FIN/SYN/RST/PSH/ACK/URG. 3-way(SYN→SYN+ACK→ACK), 4-way(FIN+ACK→ACK→FIN+ACK→ACK). 연결 거부=SYN→RST+ACK, 연결 중단=RST+ACK. **MSS = MTU − 40byte**.
7. **UDP** : 비연결형·비신뢰적. **헤더 8byte 고정** (Source/Dest Port + Total Length + Checksum). DNS/NTP/DHCP/TFTP/SNMP 가 사용.
8. **NAT** : Static / Dynamic / **PAT(N:1, 가장 일반적)**. 방향별 SNAT(출발지) / DNAT(목적지, 포트포워딩). RFC 1918 사설주소와 함께 동작.
