# 5과목 01 — 정보보호 관리 개요

> 출제기준 매핑: 1.1 정보보호 관리 이해 — 정보보호의 목적 및 특성 / 정보보호 관리의 개념 / 정보보호 전략 및 조직

## §0. 들어가며

본 단원은 5과목 (정보보안 관리 및 법규) 의 도입부로, 정보보호의 정의·목표·관리 개념을 다룬다. 이후 단원의 위험관리 (02), 보호대책 구현 (03), 사고대응 (04), 인증제도 (05) 는 모두 본 단원에서 정의한 **정보보호의 목표 (CIA + 인증성·책임성)** 를 보장하기 위한 수단이다. 따라서 본 단원의 정의·용어가 5과목 전체의 토대가 된다.

정보보호 관리는 단순한 기술 통제의 집합이 아니라, 조직의 자산을 위협으로부터 보호하기 위한 **체계적·지속적·문서화된** 관리 활동이다. ISO/IEC 27001:2022 [^iso27001] 는 이를 "정보보호 관리체계 (ISMS)" 라 정의하고, 이 체계를 구성하는 핵심 요소를 정책·조직·자산관리·인력보안·물리적 보안·운영보안·통신보안·시스템 획득개발유지·공급자 관계·사고관리·연속성·준거성으로 분류한다.

본 단원은 다음 순서로 학습한다.
1. 정보보호의 정의와 목적 (§1)
2. CIA Triad — 기밀성·무결성·가용성 (§2)
3. 확장 보안요소 — 인증성·책임성 (§3)
4. 정보보호 관리의 개념 — PDCA 사이클 (§4)
5. 정보보호 거버넌스 (§5)
6. 정보보호 전략 수립 (§6)
7. 정보보호 조직과 역할 (§7)
8. 정보보호 정책·표준·지침·절차 4계층 문서체계 (§8)

## §1. 정보보호의 정의와 목적

> **정보보호 (Information Security)** : 조직의 정보 자산이 다양한 내·외부 위협으로부터 기밀성·무결성·가용성 등이 보장될 수 있도록 **관리적·기술적·물리적 보호조치** 를 마련하는 활동이다.[^isms-def]

정보보호는 "정보의 안전한 사용을 보장하는 것" 으로 출발하지만, 단순히 침해를 막는 수동적 활동이 아니라 조직의 사명·전략과 연결된 **위험관리** 의 한 형태이다. ISO/IEC 27000:2018 §3.28 [^iso27000] 은 정보보호를 "정보의 기밀성·무결성·가용성의 보전" 으로 정의하고, 추가적으로 진정성 (authenticity)·책임성 (accountability)·부인방지 (non-repudiation)·신뢰성 (reliability) 을 포함할 수 있다고 명시한다.

### §1-1. 정보보호 vs 사이버보안 vs 정보보안

세 용어는 자주 혼용되지만, 표준 정의상 명확한 차이가 있다.

| 용어 | 정의 출처 | 범위 |
|---|---|---|
| 정보보호 (Information Security) | ISO/IEC 27000:2018 §3.28 | **모든 형태의 정보** (디지털·아날로그·구두) 의 CIA 보전. 종이 문서·구두 의사소통 포함 |
| 사이버보안 (Cybersecurity) | ISO/IEC 27032:2023 §3.10 [^iso27032] | **사이버 공간** (네트워크·디지털 시스템) 에 한정된 정보·자산 보호 |
| 정보보안 ([정보통신망법] §2) | [정보통신망 이용촉진 및 정보보호 등에 관한 법률] §2 [^infonet-act] | 정보의 수집·저장·검색·송신·수신 중에 발생할 수 있는 정보의 훼손·변조·유출 등을 방지하기 위한 관리적·기술적 수단 |

정보보호는 사이버보안보다 **상위 개념**이며, 종이 문서·인쇄물·USB 매체 등 비전자적 정보까지 포함한다.

### §1-2. 정보보호의 특성

정보보호는 다음과 같은 본질적 특성을 갖는다.

- **체계성** : 단편적 보안 통제가 아니라 정책 → 표준 → 지침 → 절차의 4계층 문서체계 (§8) 와 PDCA 관리 사이클 (§4) 로 운영된다.
- **지속성** : 위협 환경의 변화에 따라 위험을 재평가하고 통제를 갱신한다 (ISO/IEC 27001:2022 §10 — Improvement [^iso27001]).
- **위험 기반** : 모든 통제는 식별된 위험에 대응한다. 위험이 없는 영역에 통제를 두는 것은 비용 낭비이며, 위험이 있는 영역에 통제가 없는 것은 사고로 이어진다.
- **경영 책임** : 정보보호는 IT 부서의 책임이 아니라 **최고경영진** 의 책임이다. ISO/IEC 27001 §5.1 [^iso27001] 은 최고경영진의 리더십·약속을 명시한다.
- **균형성** : CIA 는 상호 충돌할 수 있다 (예: 기밀성 강화 ↔ 가용성 저하). 비즈니스 요구에 맞춘 균형이 필요하다.

### §1-3. 시험 포인트

- "정보보호는 IT 부서의 책임이다" → **틀림**. 최고경영진의 책임이며, IT 부서는 **운영 주체** 일 뿐이다.
- 정보보호 ⊃ 사이버보안 (포함 관계). 종이 문서 보호도 정보보호 영역.
- 정보보호의 3가지 보호조치 = **관리적·기술적·물리적**.

## §2. CIA Triad — 기밀성·무결성·가용성

> **CIA Triad** : 정보보호의 3대 핵심 목표. 기밀성 (Confidentiality), 무결성 (Integrity), 가용성 (Availability) 의 머리글자.[^iso27000]

CIA Triad 는 정보보호의 가장 기본적이고 보편적인 목표 분류이다. 모든 보호대책은 이 셋 중 하나 이상의 보장을 목적으로 하며, 위협·취약점·사고의 분류도 CIA 침해를 기준으로 한다. 출제기준 1.1 의 "정보보호의 목적 및 특성" 영역에서 "어떤 사고가 어떤 보안요소를 침해하는가" 를 묻는 형태가 출제된다.

### §2-1. 기밀성 (Confidentiality)

> **기밀성** : 권한 있는 자만이 정보 자산에 접근하여 그 내용을 알 수 있도록 하는 것. 정당하지 않은 자에 의해 정보가 노출되지 않도록 보장하는 것.[^isms-def]

기밀성은 "비밀 유지" 를 의미한다. 정보가 권한 없는 주체에게 노출되지 않도록 보장하며, 침해 시 **정보 유출·도청·스니핑** 등의 사고로 나타난다.

| 보호 메커니즘 | 적용 사례 |
|---|---|
| 대칭키 암호화 | AES, SEED, ARIA — 데이터 저장·전송 시 암호화 (4과목 05 §4 참조) |
| 비대칭키 암호화 | RSA, ECC — 키 교환·세션키 암호화 (4과목 05 §6 참조) |
| 접근통제 | DAC, MAC, RBAC, ABAC (4과목 02 참조) |
| 물리적 통제 | 출입통제, 시설 보안 (5과목 03 §5) |

### §2-2. 무결성 (Integrity)

> **무결성** : 권한 있는 자만이 정보 자산을 생성·수정·삭제할 수 있도록 보장하는 것. 정당하지 않은 자에 의해 정보가 위·변조되지 않도록 보장하는 것.[^isms-def]

무결성은 "정확성과 완전성의 보전" 을 의미한다. ISO/IEC 27000:2018 §3.36 [^iso27000] 은 무결성을 "정확성 (accuracy) 과 완전성 (completeness) 의 속성" 으로 정의한다. 침해 시 **위변조·중간자 공격·랜섬웨어** 등으로 나타난다.

| 보호 메커니즘 | 적용 사례 |
|---|---|
| 암호학적 해시함수 | SHA-256, SHA-3 — 무결성 검증 (4과목 06 §1 참조) |
| 메시지 인증 코드 (MAC) | HMAC, CMAC — 무결성 + 인증 (4과목 06 §3 참조) |
| 디지털서명 / 전자서명 | RSA-PSS, ECDSA — 무결성 + 인증 + 부인방지 (4과목 04 §1 참조) |
| 접근통제 (쓰기 권한) | Biba 모델 (4과목 02 §4 참조) |

### §2-3. 가용성 (Availability)

> **가용성** : 권한 있는 자가 정보 자산에 접근하고자 할 때 언제든지 접근할 수 있도록 보장하는 것.[^isms-def]

가용성은 "필요할 때 사용 가능함" 을 의미한다. 침해 시 **DoS·DDoS·랜섬웨어 (서비스 중단)·하드웨어 장애·재해** 등으로 나타난다.

| 보호 메커니즘 | 적용 사례 |
|---|---|
| 시스템 이중화 | Active-Active, Active-Standby 클러스터 |
| 디스크 RAID | RAID 1·5·6·10 — 디스크 장애 시 서비스 지속 |
| 백업·소산 | 정기 백업 + 원격지 보관 |
| 재해복구 (DRS) | 미러·핫·웜·콜드 사이트 (5과목 03 §4-3 참조) |
| DDoS 대응 | Anti-DDoS 장비, CDN, 트래픽 정제 (2과목 06 참조) |

### §2-4. CIA 의 균형과 충돌

세 요소는 서로 보완 관계이지만 **상호 충돌**할 수도 있다.

- **기밀성 ↑ → 가용성 ↓** : 강한 암호화·다중 인증은 인가된 사용자의 접근도 지연시킨다.
- **무결성 ↑ → 가용성 ↓** : 모든 변경에 디지털서명·승인 절차를 요구하면 업무 처리 시간이 늘어난다.
- **가용성 ↑ → 기밀성 ↓** : 누구나 접근 가능한 공개 시스템은 가용성은 높지만 기밀성은 보장 어렵다.

조직은 자산의 **민감도와 비즈니스 요구** 에 따라 균형을 결정한다. ISO/IEC 27001:2022 §6.1.2 [^iso27001] 의 위험 평가 절차가 이러한 균형을 결정하는 공식 메커니즘이다.

### §2-5. 시험 포인트

- DDoS = **가용성** 침해. 데이터 위변조가 아닌 서비스 중단.
- 랜섬웨어 = **가용성** + **무결성** 동시 침해 (데이터 암호화로 접근 차단 + 데이터 변조).
- 도청·스니핑 = **기밀성** 침해.
- MITM (중간자 공격) = **무결성·기밀성·인증성** 동시 침해.
- 백업 = **가용성** 보호 메커니즘 (무결성 보호로 분류하면 오답).

## §3. 확장 보안요소 — 인증성·책임성

CIA Triad 에 더해 다양한 정보보호 환경을 고려하여 **인증성** 과 **책임성** 이 추가된다.[^isms-def] [^iso27000] ISO/IEC 27000:2018 §3.6 (authenticity), §3.2 (accountability), §3.50 (non-repudiation), §3.55 (reliability) 는 이들을 별도 속성으로 정의한다.

### §3-1. 인증성 (Authenticity)

> **인증성** : 정보 자산에 접근하려는 자의 신원이 주장된 실체와 일치함을 보이는 것. 또한 메시지가 올바른 상대방이 보낸 메시지임을 보장하는 것.[^iso27000]

인증성은 두 측면을 가진다.

- **사용자 인증 (Entity Authentication)** : 접근하려는 주체의 신원 검증. 지식 기반·소유 기반·존재 기반·위치 기반·행위 기반 (4과목 01 §1 참조).
- **메시지 인증 (Message Authentication)** : 수신한 메시지가 주장된 송신자에게서 온 것임을 보장. 메시지 인증 코드 (MAC), 디지털서명·전자서명 (4과목 04, 06 참조).

### §3-2. 책임성 (Accountability)

> **책임성** : 정보 자산에 접근하는 자가 접근 결과에 대해 책임지도록 보장하는 것.[^isms-def]

책임성은 두 하위 요소로 구분된다.

- **책임추적성 (Traceability)** : 주체를 고유하게 식별하여 그 행위를 추적할 수 있도록 보장함. 핵심 메커니즘은 **감사·로깅 (Audit Logging)** 이다.
- **부인방지 (Non-repudiation)** : 주체가 한 행위를 나중에 부인하지 못하도록 보장함. 핵심 메커니즘은 **디지털서명·전자서명** 이다.[^iso27000]

| 확장 요소 | 정의 | 핵심 메커니즘 | 침해 사례 |
|---|---|---|---|
| 인증성 | 신원 일치 보장 | 패스워드, OTP, 생체인증, 디지털서명 | 신원 도용, 스푸핑 |
| 책임추적성 | 행위 추적 가능성 | 감사 로깅, SIEM | 로그 미생성·삭제 |
| 부인방지 | 행위 부인 불가 | 디지털서명, 타임스탬프 | 송신/수신 부인 |
| 신뢰성 (Reliability) | 의도된 동작 보장 | 결함 허용, 정합성 검사 | 시스템 오작동 |

### §3-3. 시험 포인트

- 인증성 = "**누가**" 보냈는지의 보장 (송신자 신원).
- 무결성 = "**무엇이**" 변경되지 않았는지의 보장 (메시지 내용).
- 부인방지 ⊃ 디지털서명 — MAC 만으로는 부인방지 **불가** (둘 다 키를 공유하므로 송신자 특정 불가).
- 책임추적성의 전제 = **고유 식별** (계정 공유는 책임추적성을 무너뜨린다).

## §4. 정보보호 관리의 개념 — PDCA 사이클

> **정보보호 관리체계 (ISMS, Information Security Management System)** : 조직이 정보보호의 목표를 달성하기 위해 정책·조직·프로세스·기술을 결합한 **체계적·문서화된 관리 시스템**.[^iso27001]

ISMS 는 단편적 보안 통제의 집합이 아니라 **PDCA (Plan-Do-Check-Act) 사이클** 로 운영되는 지속적 관리 시스템이다. ISO/IEC 27001:2013 까지는 PDCA 가 명시적으로 표준 본문에 등장했으나, ISO/IEC 27001:2022 [^iso27001] 는 ISO/IEC Management System Standards 통합 구조 (HLS, High Level Structure) 를 따라 PDCA 와 동등한 7개 절 (§4~§10) 로 재편성되었다.

### §4-1. PDCA 사이클의 4단계

| 단계 | 한국어 | ISO/IEC 27001:2022 매핑 | 활동 |
|---|---|---|---|
| Plan | 계획 | §4 (조직 환경)·§5 (리더십)·§6 (계획) | 조직·이해관계자 식별 / 범위 결정 / 위험 평가·처리 / 정보보호 목표 수립 |
| Do | 실행 | §7 (지원)·§8 (운영) | 자원 확보 / 인식·교육 / 위험 처리 계획 실행 / 통제 운영 |
| Check | 점검 | §9 (성과 평가) | 모니터링·측정·분석·평가 / 내부 감사 / 경영검토 |
| Act | 개선 | §10 (개선) | 부적합 사항·시정 조치 / 지속적 개선 |

### §4-2. PDCA 의 의의

PDCA 는 다음 두 가지 핵심 메시지를 담는다.

1. **정보보호는 1회성 프로젝트가 아니라 지속적 관리 활동이다.** 위험 환경이 변하면 통제도 변해야 한다.
2. **계획만으로는 보안이 달성되지 않는다.** 실행·점검·개선의 사이클이 닫혀야 비로소 효과가 나타난다.

ISO/IEC 27001:2022 §10.2 [^iso27001] 는 "지속적 개선 (Continual Improvement)" 을 명시 요구사항으로 둔다. ISMS-P 인증기준 (5과목 05 §3 참조) 의 1.4 영역 (관리체계 점검 및 개선) 도 동일한 PDCA 사상을 따른다.

### §4-3. 시험 포인트

- PDCA 의 Plan 에서 위험 평가·정보보호 목표 수립이 이루어진다.
- 내부 감사·경영검토는 **Check** 단계.
- 시정조치는 **Act** 단계 — 모니터링이 아님.
- ISO/IEC 27001:2022 는 PDCA 용어를 직접 사용하지 않으나, §4~§10 구조가 PDCA 와 동등하다.

## §5. 정보보호 거버넌스

> **정보보호 거버넌스 (Information Security Governance)** : 조직의 정보보호 활동이 비즈니스 목표와 정렬되고, 위험이 적절히 관리되며, 자원이 책임감 있게 사용되도록 보장하는 **이사회·최고경영진의 활동**.[^iso27014]

거버넌스는 관리 (Management) 와 구분된다. 관리가 "어떻게 운영할지" 라면, 거버넌스는 "**무엇을·왜** 해야 하는지를 결정하고 감독" 하는 활동이다.

### §5-1. ISO/IEC 27014:2020 거버넌스 6원칙

ISO/IEC 27014:2020 (Governance of Information Security) [^iso27014] 는 다음 6 가지 원칙을 제시한다.

| 원칙 | 의미 |
|---|---|
| 조직 전반에 걸친 정보보호 수립 | 정보보호를 IT 부서로 한정하지 않고 모든 비즈니스 활동에 통합 |
| 위험 기반 접근 | 모든 의사결정을 식별된 위험에 근거하여 |
| 투자 결정 방향 설정 | 정보보호 투자가 비즈니스 가치 창출에 기여하는 방향으로 |
| 내·외부 요구사항 준수 | 법령·계약·이해관계자 요구사항 준수 |
| 보안 친화적 환경 조성 | 정보보호가 조직 문화의 일부가 되도록 |
| 비즈니스 결과에 대한 성과 검토 | 정보보호 성과를 비즈니스 결과와 연계하여 측정·검토 |

### §5-2. NIST CSF 2.0 — Govern 함수

NIST Cybersecurity Framework 2.0 (2024.2.26 발행) [^nist-csf] 은 기존 5개 함수 (Identify·Protect·Detect·Respond·Recover) 에 **Govern** 을 추가하여 6개 함수로 재편하였다. Govern 함수는 거버넌스의 6 가지 카테고리로 구성된다.

| Govern 카테고리 | 약칭 | 내용 |
|---|---|---|
| Organizational Context | GV.OC | 조직의 사명·이해관계자·법규 환경 이해 |
| Risk Management Strategy | GV.RM | 위험관리 전략·우선순위 수립 |
| Roles, Responsibilities, and Authorities | GV.RR | 사이버보안 역할·책임·권한 정의 |
| Policy | GV.PO | 사이버보안 정책 수립 및 시행 |
| Oversight | GV.OV | 사이버보안 전략·결과의 감독 및 검토 |
| Cybersecurity Supply Chain Risk Management | GV.SC | 공급망 위험관리 |

NIST CSF 1.1 (2018.4) 까지는 거버넌스가 Identify 함수 하위의 한 카테고리 (ID.GV) 에 불과했으나, 2.0 에서 **별도 최상위 함수로 격상** 되었다. 이는 거버넌스가 다른 모든 함수 (Identify·Protect·Detect·Respond·Recover) 의 토대가 됨을 강조한다.

### §5-3. 거버넌스 vs 관리 vs 운영

세 층은 명확히 구분된다.

| 계층 | 주체 | 활동 |
|---|---|---|
| 거버넌스 | 이사회·최고경영진 | 방향 설정, 정책 승인, 감독 |
| 관리 | CISO·정보보호 부서장 | 위험 평가, 통제 선정, 자원 배분 |
| 운영 | 정보보호 담당자·시스템 운영자 | 보안 통제 구현·운영·모니터링 |

### §5-4. 시험 포인트

- NIST CSF 2.0 (2024.2 발행) 함수는 **6개** : Govern·Identify·Protect·Detect·Respond·Recover.
- NIST CSF 1.1 의 함수는 **5개** : Identify·Protect·Detect·Respond·Recover (Govern 미존재).
- ISO/IEC 27014 = 거버넌스 표준, ISO/IEC 27001 = 관리 표준.
- 거버넌스 = "**무엇을·왜**" / 관리 = "**어떻게**".

## §6. 정보보호 전략 수립

> **정보보호 전략** : 조직의 비즈니스 목표·법규 환경·위험 환경을 고려하여 **정보보호의 방향성과 우선순위** 를 정의한 중장기 계획.

전략은 정책 (§8) 의 상위 개념이다. 정책이 "무엇을 해야 하는지" 를 명문화한다면, 전략은 "**왜 그 방향으로 가야 하는지**" 를 비즈니스 관점에서 설명한다.

### §6-1. 전략 수립의 5단계

NIST SP 800-100 (Information Security Handbook for Managers) [^nist800-100] 와 KISA 정보보호 관리체계 가이드를 종합하면 다음 5단계로 진행된다.

| 단계 | 활동 |
|---|---|
| 1. 환경 분석 | 비즈니스 환경·이해관계자·법규·위협 동향 분석 |
| 2. 현황 평가 | 현재 정보보호 수준 진단 (Gap Analysis) |
| 3. 목표 설정 | To-Be 정보보호 수준·핵심 성과 지표 (KPI) 정의 |
| 4. 전략 수립 | 우선순위·로드맵·예산·자원 배분 결정 |
| 5. 이행 및 평가 | 전략 실행 + 정기 평가·갱신 |

### §6-2. 전략 문서의 핵심 구성

정보보호 전략 문서는 일반적으로 다음 요소를 포함한다.

- 비즈니스 목표와 정보보호의 정렬
- 정보보호 비전·미션
- 정보보호 원칙 (예: Defense in Depth, Least Privilege, Zero Trust)
- 중장기 목표 (3~5년) 와 KPI
- 우선순위 영역 (위험 기반)
- 예산·자원·조직
- 거버넌스·보고 체계

### §6-3. 시험 포인트

- 전략 = 중장기 (3~5년), 정책 = 상위 지침, 표준 = 강제 요구사항, 절차 = 단계별 수행 방법.
- 전략 수립의 출발점은 **비즈니스 환경·이해관계자 요구** 분석.
- 전략 없는 정책은 일관성을 잃고, 정책 없는 전략은 실행되지 않는다.

## §7. 정보보호 조직과 역할

### §7-1. CISO — 정보보호 최고책임자

> **CISO (Chief Information Security Officer, 정보보호 최고책임자)** : 조직의 정보보호 활동을 총괄하는 임원급 책임자.

한국의 경우 [정보통신망 이용촉진 및 정보보호 등에 관한 법률] §45-3 [^infonet-cisp] 에 의해 일정 규모 이상의 정보통신서비스 제공자는 CISO 를 지정·신고할 의무가 있다.

| 구분 | 내용 |
|---|---|
| 법적 근거 | [정보통신망법] §45-3 (정보보호 최고책임자의 지정 등) [^infonet-cisp] |
| 지정 의무자 | 일정 규모 이상의 정보통신서비스 제공자 (시행령에 따른 자산·매출·이용자 수 기준) |
| 신고 의무 | 과학기술정보통신부 장관에게 신고 |
| 겸직 제한 | 일정 규모 이상의 사업자는 CISO 가 다른 직무 겸직 제한 (정보통신망법 §45-3 ②항·③항 — 시행령 기준) |
| 자격 요건 | 정보보호 또는 정보기술 분야 일정 경력 (시행령) |

CISO 의 핵심 책임은 다음과 같다.

- 정보보호 정책·계획 수립·시행
- 정보보호 위험 식별·평가·대응
- 정보보호 사고 대응 총괄
- 임직원 정보보호 교육
- 법규 준수 및 감독기관 보고

### §7-2. CPO — 개인정보 보호책임자

> **CPO (Chief Privacy Officer, 개인정보 보호책임자)** : 조직의 개인정보 처리 업무를 총괄하는 책임자.

[개인정보보호법] §31 [^pipa] 은 개인정보처리자가 개인정보 보호책임자 (CPO) 를 지정할 의무를 부과한다. 5과목 08 §6 에서 상세히 다룬다.

| 구분 | CISO | CPO |
|---|---|---|
| 법적 근거 | 정보통신망법 §45-3 | 개인정보보호법 §31 |
| 보호 대상 | 정보보호 일반 (CIA) | 개인정보 |
| 임명 의무자 | 정보통신서비스 제공자 | 개인정보처리자 (공공·민간 모두) |
| 겸직 가능 여부 | 일정 규모 이상 겸직 제한 | 법령상 별도 제한 없음 (단 CISO 와 분리 운영 권장) |

대규모 조직에서는 CISO 와 CPO 를 분리하는 것이 권장되며, 정보보호위원회·개인정보 처리방침 심의기구 등의 거버넌스 기구를 운영한다.

### §7-3. 정보보호 조직 구성

ISO/IEC 27002:2022 §5.2 [^iso27002] 와 KISA ISMS-P 인증기준 1.1.2 [^isms-p-cert] 는 정보보호 조직에 대해 다음을 요구한다.

- **정보보호 위원회** : 최고경영진·CISO·CPO·관련 부서장으로 구성. 정보보호 정책 심의·중요 의사결정.
- **정보보호 실무 조직** : CISO 산하의 전담 조직. 정책 수립, 위험 관리, 통제 운영.
- **현업 보안 담당자** : 각 부서별로 지정된 보안 담당자. 부서 내 정보보호 활동 수행.
- **분리 원칙** : 정보보호 부서는 IT 운영 부서와 **조직적으로 분리** 되어야 한다 (자가 점검의 객관성 확보).

### §7-4. 시험 포인트

- CISO 지정 의무 = [정보통신망법] §45-3.
- CPO 지정 의무 = [개인정보보호법] §31.
- CISO 는 일정 규모 이상의 사업자에서 다른 직무 겸직이 제한된다 (시행령 기준).
- 정보보호 조직은 IT 운영 조직과 **분리** 되어야 한다 (객관성·독립성).

## §8. 정보보호 정책·표준·지침·절차 4계층 문서체계

> **정보보호 정책 (Policy)** : 조직의 정보보호 목적과 활동에 관한 사항을 정의한 **최상위 문서**. 경영진의 의지·방향, 역할과 책임, 대상과 범위, 관리적·기술적·물리적 정보보호 활동의 근거를 포함한다.[^isms-def]

정보보호 문서는 **4계층 (정책 → 표준 → 지침 → 절차)** 의 위계 구조를 가진다. 이는 ISO/IEC 27002:2022 §5.1 [^iso27002] 의 정책 요구사항과 KISA ISMS-P 인증기준 1.1.5 [^isms-p-cert] 의 정책 수립 기준에 부합한다.

### §8-1. 4계층 문서의 정의

| 계층 | 정의 | 강제성 | 작성 단위 |
|---|---|---|---|
| 정책 (Policy) | 조직의 정보보호 목적·활동에 관한 사항을 정의한 **최상위 문서** | 강제 | 조직 전체 1개 |
| 표준 (Standard) | 정책 달성을 위한 **세부 요구사항** 을 구체적으로 정의한 규정 | 강제 | 영역별 (예: 패스워드 표준, 암호화 표준) |
| 지침 (Guideline) | 특정 시스템·분야별로 활동에 필요하거나 도움이 되는 **세부 요구사항** | 권고 (융통성) | 시스템·환경별 |
| 절차 (Procedure) | 정책·표준·지침을 준수하기 위해 구체적으로 **어떻게** 수행해야 하는지를 단계적으로 설명한 문서 | 강제 (수행 시) | 업무별 |

> *원본 표기 보정*: 5과목.txt line 76~84 의 "규악" 은 "규약 (規約)" 의 OCR 오류이다.

### §8-2. 정책의 필수 구성요소

정책 문서가 갖추어야 할 핵심 요소는 다음과 같다.

- **목적과 적용 범위** : 정책이 다루는 영역 및 적용 대상
- **경영진의 의지·약속** : 최고경영진의 정보보호 의지를 명시한 서명·승인
- **역할과 책임** : CISO·CPO·정보보호 위원회·임직원의 R&R
- **법적·계약상 요구사항** : 준수해야 할 법령·계약·표준
- **위반 시 조치** : 정책 위반에 대한 징계·법적 대응
- **검토·갱신 주기** : 정기 재검토 주기 (일반적으로 연 1회 이상)

### §8-3. 표준·지침·절차의 차이

세 문서는 자주 혼동되므로 차이점을 명확히 구분해야 한다.

- **표준**은 "**무엇을 충족**해야 하는가" 를 강제 (예: "패스워드는 12자 이상, 영문·숫자·특수문자 혼합").
- **지침**은 "**어떻게 충족**할 수 있는가" 를 권고 (예: "패스워드 관리자 사용을 권장한다. 단, 부득이한 경우 ...").
- **절차**는 "**구체적인 수행 방법**" 을 단계별로 기술 (예: "1. 시스템 관리자가 신규 계정 생성 → 2. 임시 패스워드 발급 → 3. 첫 로그인 시 강제 변경 ...").

### §8-4. 정책 검토 주기

ISO/IEC 27001:2022 §7.5.2 [^iso27001] 와 KISA ISMS-P 인증기준 1.1.5 [^isms-p-cert] 는 정책의 정기적 재검토를 요구한다. 일반적으로 다음 시점에 검토한다.

- 정기 검토 : **최소 연 1회**
- 비정기 검토 :
  - 조직·사업의 중대 변경 시
  - 법령 개정 시
  - 신규 위협·취약점 식별 시
  - 침해사고 발생 시
  - 신기술 도입 시

### §8-5. 시험 포인트

- 4계층 문서 위계 = **정책 > 표준 > 지침 > 절차**.
- **표준**은 강제, **지침**은 권고 (융통성).
- **절차**는 "어떻게" 의 단계적 설명.
- 정책의 정기 검토 주기 = **최소 연 1회**.
- 정책 문서에는 **경영진 서명** 이 필수.

## §9. 단원 요약

| 영역 | 핵심 |
|---|---|
| 정보보호 정의 | 정보 자산의 CIA + (인증성·책임성) 보전을 위한 관리적·기술적·물리적 보호조치 |
| CIA Triad | 기밀성 (암호화·접근통제) / 무결성 (해시·MAC·서명) / 가용성 (이중화·RAID·DRS) |
| 인증성·책임성 | 인증성 (신원·메시지) / 책임성 = 책임추적성 + 부인방지 |
| 관리 사이클 | PDCA = Plan·Do·Check·Act, ISO/IEC 27001:2022 §4~§10 |
| 거버넌스 | 이사회·CEO 의 방향 설정·감독 활동, NIST CSF 2.0 Govern 함수 (2024.2 발행) |
| 조직 | CISO ([정보통신망법] §45-3), CPO ([개인정보보호법] §31) — 분리 권장 |
| 4계층 문서 | 정책 > 표준 > 지침 > 절차 — 정책은 최소 연 1회 재검토 |

## §10. 시험 함정 정리

| 함정 | 정답 |
|---|---|
| 정보보호는 IT 부서의 책임이다 | **틀림**. 최고경영진의 책임 |
| DDoS 는 무결성을 침해한다 | **틀림**. 가용성 침해 |
| 도청은 무결성을 침해한다 | **틀림**. 기밀성 침해 |
| MAC 만으로 부인방지가 가능하다 | **틀림**. 키 공유로 송신자 특정 불가 → 디지털서명 필요 |
| NIST CSF 1.1 의 함수는 6개이다 | **틀림**. 1.1 = 5개 (Identify·Protect·Detect·Respond·Recover), 2.0 = 6개 (Govern 추가) |
| 지침은 강제 요구사항이다 | **틀림**. 지침은 권고·융통성 (표준이 강제) |
| ISO/IEC 27001:2022 는 PDCA 를 명시 사용한다 | **틀림**. 27001:2013 까지는 PDCA 명시, 2022 는 HLS 7개 절 (§4~§10) 로 재편 |
| CPO 는 정보통신망법으로 지정 | **틀림**. CPO 는 개인정보보호법 §31, CISO 는 정보통신망법 §45-3 |
| 정보보호 = 사이버보안 | **틀림**. 정보보호 ⊃ 사이버보안 (포함 관계) |
| 정책은 5년에 한 번 검토 | **틀림**. 최소 연 1회 |
