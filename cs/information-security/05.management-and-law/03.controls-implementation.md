# 5과목 03 — 보호대책 구현 및 운영

> 출제기준 매핑: 1.3 정보보호 대책 구현 및 사고대응 (1)~(4) — 정보보호 대책 선정 및 계획서 작성 / 관리적 보호대책 (인력보안·교육·내부감사·침해사고 예방·업무연속성관리) / 물리적 보호대책 (출입통제·환경 보안) / 기술적 보호대책 (시스템·SW 개발·서버·네트워크·DB·어플리케이션·정보보호시스템 운영)

## §0. 들어가며

본 단원은 위험평가 (02) 의 결과로 도출된 보호대책을 **실제 구현하고 운영** 하는 영역을 다룬다. 보호대책은 (1) 관리적·물리적·기술적 — 유형 분류와 (2) 예방·탐지·교정 — 시점 분류로 구분되며 (5과목 02 §2-5 참조), 본 단원은 출제기준 1.3 의 4 가지 영역을 차례로 다룬다. 사고대응 (1.3 의 5번 항목) 은 분량이 별도 단원 (04) 에서 다뤄질 만큼 크므로 분리하였다.

본 단원은 ISO/IEC 27002:2022 (Information security controls) [^iso27002] 의 93개 통제, NIST SP 800-53 Rev. 5 [^nist800-53] 의 통제 카탈로그, ISO 22301:2019 (Business Continuity Management Systems) [^iso22301], NIST SP 800-34 Rev. 1 (Contingency Planning Guide) [^nist800-34], KISA 「주요정보통신기반시설 기술적·관리적 보호조치 기준」 [^kisa-cii] 을 1차 출처로 한다.

학습 순서:
1. 보호대책 선정과 이행계획 (§1)
2. 관리적 보호대책 — 인력보안·교육·내부감사 (§2)
3. 업무연속성관리 — BCP·BIA·DRP·DRS (§3·§4)
4. 물리적 보호대책 — 출입통제·환경 (§5)
5. 기술적 보호대책 — 시스템·SW 개발·네트워크·DB (§6)

## §1. 보호대책 선정과 이행계획

### §1-1. 보호대책 선정 절차

위험 평가 (5과목 02) 의 결과로 DoA 초과 위험이 도출되면, 위험처리 전략 (수용·감소·전가·회피) 을 적용한 후 **위험 감소** 전략에 해당하는 위험에 대해 구체적 보호대책을 선정한다.

| 단계 | 활동 | 산출물 |
|---|---|---|
| 1. 위험처리 결정 | 4가지 전략 중 선택 | 위험처리 결정서 |
| 2. 통제 후보 도출 | ISO/IEC 27002:2022 / NIST SP 800-53 / KISA 가이드 등에서 후보 통제 식별 | 통제 후보 목록 |
| 3. 비용·효과 분석 | ROSI 분석으로 통제 채택 여부 결정 | 비용·효과 분석서 |
| 4. 통제 우선순위 결정 | 위험 수준·법적 의무·자원 제약을 고려한 우선순위 | 우선순위 결정서 |
| 5. 정보보호 대책명세서 (SoA) 작성 | 채택·미채택 통제 목록 + 사유 | SoA |
| 6. 이행계획 수립 | 일정·예산·책임자 명시 | 정보보호 이행계획서 |

### §1-2. 정보보호 이행계획서의 구성

이행계획서는 다음 요소를 포함한다.

- 통제별 구현 일정 (단기·중기·장기)
- 예산 및 자원 배분
- 책임자 (소유자) 및 수행자
- 성과 측정 지표 (KPI)
- 위험·이슈 관리 계획
- 보고 및 검토 주기

### §1-3. 시험 포인트

- 위험처리 결정 → 통제 선정 → SoA → 이행계획 (순서).
- ROSI > 0 = 통제 채택의 정량 근거.
- 이행계획에는 일정·예산·책임자가 모두 포함되어야 함.

## §2. 관리적 보호대책

> **관리적 보호대책** : 보안 정책의 수립 및 시행 관점의 통제. 정책·표준·절차·교육·감사·인력 관리 등 **사람과 프로세스** 에 대한 통제.[^isms-def]

ISO/IEC 27002:2022 [^iso27002] 의 4개 카테고리 중 **조직적 통제 (37개) + 인적 통제 (8개) = 45개** 가 관리적 통제에 해당한다.

### §2-1. 인력 보안 — 입사·재직·퇴직

ISO/IEC 27002:2022 §6 (인적 통제) [^iso27002] 와 KISA ISMS-P 인증기준 2.2 (인적 보안) [^isms-p-cert] 는 인력 생애주기에 따른 보안 활동을 요구한다.

| 단계 | 핵심 활동 |
|---|---|
| **입사 전** | 신원 검증 (Background Check), 자격 검증, 채용 계약서에 비밀유지 (NDA) 조항 포함 |
| **재직 중** | 정보보호 책임 명확화, 정기 교육·훈련, 직무 변경 시 권한 재조정, 위반 시 징계 절차 |
| **퇴직 시** | 즉시 접근권한 회수, 자산 반납 (사원증·노트북·매체), 비밀유지 의무의 사후 적용, 인수인계 |

### §2-2. 내·외부 인력보안

| 구분 | 내부 인력 | 외부 인력 (위탁·협력업체) |
|---|---|---|
| 신원 검증 | 채용 시 1회 + 정기 재검증 | 계약 시 검증 + 인력 변경 시 통지 의무 |
| 비밀유지 | 근로계약서·취업규칙 | 계약서·SLA·NDA |
| 접근통제 | 직무 기반 권한 부여 | **최소 권한**·**기한부 권한** 부여 |
| 감독 | 정기 감사·로그 모니터링 | 위탁 감독·정기 점검·계약 종료 시 권한 회수 |
| 법적 근거 | 근로기준법, 취업규칙 | [개인정보보호법] §26 (개인정보 처리업무 위탁) |

### §2-3. 교육 및 훈련

ISO/IEC 27002:2022 §6.3 (Information security awareness, education and training) [^iso27002] 는 다음을 요구한다.

| 교육 유형 | 대상 | 주기 |
|---|---|---|
| 정기 정보보호 교육 | 전 임직원 | 최소 연 1회 |
| 직무 특화 교육 | 정보보호 담당자, 시스템 운영자, 개발자 | 직무 변경 시 + 정기 |
| 신규 입사자 교육 | 신규 인력 | 입사 후 즉시 |
| 침해사고 대응 훈련 | 사고 대응 조직 | 최소 연 1회 (모의훈련 포함) |
| BCP·DR 훈련 | 업무연속성 관리 조직 | 최소 연 1회 |

KISA ISMS-P 인증기준 2.2.4 (인식제고 및 교육훈련) [^isms-p-cert] 는 교육 계획·실시·결과 관리·평가의 4단계 프로세스를 요구한다.

### §2-4. 내부 감사

> **내부 감사 (Internal Audit)** : 조직 내부 인력이 ISMS 의 준수 상태와 효과성을 점검하는 활동.[^iso27001]

ISO/IEC 27001:2022 §9.2 [^iso27001] 는 내부 감사의 필수 요구사항을 명시한다.

| 요구사항 | 내용 |
|---|---|
| 감사 계획 | 감사 주기·범위·기준·방법 정의 |
| 감사 독립성 | 감사자는 자신의 업무를 감사하지 않음 (직무 분리) |
| 감사 결과 보고 | 최고경영진에게 보고 |
| 시정 조치 | 부적합 사항에 대한 시정 |

### §2-5. 침해사고 예방·대응 (관리적 측면)

기술적 측면의 침해사고 대응은 5과목 04 단원에서 상세히 다루며, 본 §2-5 는 관리적 측면만 다룬다.

| 활동 | 내용 |
|---|---|
| 사고대응 정책 수립 | 사고 정의·심각도·대응 절차·보고 체계 |
| 사고대응 조직 구성 | CISO·CIRT·외부 협력기관 (KISA, 경찰) |
| 사고대응 훈련 | 모의훈련·테이블탑·시나리오 기반 훈련 |
| 사후 분석 | Lessons Learned, 정책·절차 개선 |

### §2-6. 시험 포인트

- 신원 검증 (Background Check) 은 **입사 전** 단계.
- 퇴직 시 즉시 회수해야 할 것 = **접근권한·자산** (NDA 는 사후에도 유효).
- 정보보호 교육 주기 = **최소 연 1회** (모든 임직원).
- 내부 감사는 **자기 업무를 감사하지 않음** (직무 분리).
- 외부 인력의 권한 부여 원칙 = **최소·기한부**.

## §3. 업무연속성계획 — BCP·BIA

> **BCP (Business Continuity Plan, 업무연속성계획)** : 인재·천재지변·테러 등에 의한 각종 장애 및 재해로부터 업무 중단이 발생할 경우 **최대한 빠른 시간 내에 업무를 복구** 함으로써 업무 연속성을 유지하기 위한 계획.[^isms-def]

BCP 는 ISO 22301:2019 (Business Continuity Management Systems) [^iso22301] 의 BCMS (업무연속성 관리체계) 의 핵심 산출물이며, NIST SP 800-34 Rev. 1 [^nist800-34] 의 비상계획 (Contingency Planning) 영역과 일치한다.

### §3-1. BCP 5단계 방법론

| 단계 | 활동 | 산출물 |
|---|---|---|
| 1. 프로젝트 범위 설정 및 기획 | BCP 프로젝트의 범위·일정·예산·조직 정의 | BCP 프로젝트 헌장 |
| 2. 사업 영향 평가 (BIA) | 업무 중단의 영향 분석, 핵심 업무 식별, 복구 우선순위 결정 | BIA 보고서 |
| 3. 복구 전략 개발 | RTO·RPO 결정, 복구 사이트·백업 전략 선정 | 복구 전략서 |
| 4. 복구 계획 수립 | 절차·역할·연락처·자원 배분의 구체적 계획 작성 | BCP·DRP 문서 |
| 5. 프로젝트 수행 및 테스트·유지보수 | 훈련·점검·갱신 | 훈련 보고서, 갱신 이력 |

### §3-2. BIA — 사업 영향 평가

> **BIA (Business Impact Analysis, 사업 영향 평가)** : BCP 의 핵심 절차로, 전체 사업에 존재하는 핵심 업무를 파악하고 업무별로 사업에 미치는 영향도를 분석해 장애 및 재해 시 복구 우선순위를 결정하는 과정.[^isms-def]

BIA 의 4 가지 핵심 산출물:

| 산출물 | 정의 |
|---|---|
| 핵심 업무 목록 | 조직 운영에 필수적인 업무의 식별 |
| 업무 중단 영향도 | 업무 중단 시 시간별·재무적·운영적·법적·평판적 손실 |
| **RTO** (Recovery Time Objective) | 재해로 서비스가 중단되었을 때 서비스를 복구하는 데까지 걸리는 **최대 허용 시간** |
| **RPO** (Recovery Point Objective) | 재해로 중단된 서비스를 복구했을 때 **유실을 감내할 수 있는 데이터의 손실 허용 시점** |

### §3-3. RTO·RPO·MTD·WRT 비교

| 지표 | 의미 | 측정 단위 | 결정 기준 |
|---|---|---|---|
| RTO | 서비스 복구까지의 **최대 허용 시간** | 시간 | 업무 중단 허용 시간 |
| RPO | **유실 허용 데이터의 시점** | 시간 (데이터 손실량) | 백업 주기 |
| MTD (Maximum Tolerable Downtime) | 조직이 **생존 가능한 최대 중단 시간** | 시간 | 비즈니스 임계값 |
| WRT (Work Recovery Time) | 시스템 복구 후 정상 업무 재개까지 시간 | 시간 | 업무 검증 시간 |

```
MTD = RTO + WRT
```

> *주의*: RTO 와 RPO 는 **다른 시간축** 을 측정한다. RTO 는 사고 시점에서 복구까지 걸리는 시간 (미래 방향), RPO 는 사고 시점에서 마지막 백업까지의 시간 (과거 방향, 데이터 손실량) 이다.

```
   ←──── RPO ────→ ●사고 발생 ←──── RTO ────→ ●복구 완료 ←─ WRT ─→ ●업무 재개
   │                │                          │                    │
   마지막 백업                                                       │
                    │←──────────── MTD ───────────────────────────→│
                    (사고 발생 → 업무 재개)
```

### §3-4. 시험 포인트

- BIA = BCP 의 **핵심 절차**.
- RTO ≠ RPO. RTO = 시간 축 (복구 시간), RPO = 데이터 시점 (손실 허용량).
- MTD = RTO + WRT.
- BIA 의 결과로 **복구 우선순위** 결정.

## §4. 재해복구 — DRP·DRS

> **DRP (Disaster Recovery Plan, 재해복구계획)** : 재해로 인하여 중단된 **정보기술 서비스** 를 재개하기 위한 계획.[^isms-def]
>
> **DRS (Disaster Recovery System, 재해복구시스템)** : 재해 복구 계획의 원활한 수행을 지원하기 위하여 평상시에 확보하여 두는 인적·물적 자원과 이들에 대한 지속적 관리체계가 통합된 것.[^isms-def]

BCP ⊃ DRP. BCP 는 사업 전체의 연속성을 다루고, DRP 는 그 중 IT 서비스 복구를 다룬다.

### §4-1. BCP vs DRP 비교

| 비교 항목 | BCP | DRP |
|---|---|---|
| 범위 | 사업 전체 (인력·시설·공급망 포함) | IT 서비스 |
| 목적 | 업무 연속성 | 시스템·데이터 복구 |
| 주체 | 비즈니스 부서 + IT | IT 부서 |
| 산출물 | BCP 문서 | DRP 문서 + DRS |
| 표준 | ISO 22301 | ISO/IEC 27031, NIST SP 800-34 |

### §4-2. NIST SP 800-34 Rev. 1 비상계획 7단계

NIST SP 800-34 Rev. 1 [^nist800-34] 는 비상계획 수립을 다음 7단계로 정의한다.

| 단계 | 활동 |
|---|---|
| 1. 비상계획 정책 수립 | 비상계획의 권한·범위·역할 정의 |
| 2. BIA 수행 | 업무 영향 분석으로 RTO·RPO 결정 |
| 3. 예방 통제 식별 | 사고 자체를 예방하는 통제 |
| 4. 비상 전략 수립 | 백업·복구 사이트·복구 절차 |
| 5. 정보시스템 비상계획 작성 | 구체적 복구 절차 문서화 |
| 6. 계획 시험·교육·훈련 | 정기적 검증 |
| 7. 계획 유지·갱신 | 환경 변화에 따른 업데이트 |

### §4-3. DRS 4종 — 미러·핫·웜·콜드

DRS 는 복구 수준에 따라 4종으로 분류된다.[^isms-def]

| DRS 유형 | 시스템 구축 수준 | RTO | 데이터 동기화 | 비용 |
|---|---|---|---|---|
| **미러 사이트** | 주센터와 **동일한 수준** 의 시스템 | **즉시** (실시간) | Active + Active (실시간 양방향) | 매우 높음 |
| **핫 사이트** | 주센터와 **동일한 수준** 의 시스템 | **수 시간 이내** | Active + Standby (실시간 단방향) | 높음 |
| **웜 사이트** | **중요도 높은 일부** 시스템만 | **수일 ~ 수주 이내** | 정기 동기화 | 중간 |
| **콜드 사이트** | **데이터만** 원격지 보관, 장소 등 최소 자원 | **수주 ~ 수개월 이내** | 백업 매체 보관 | 낮음 |

> *주의*: 미러와 핫 사이트는 **동일한 수준의 시스템 구축** 이라는 점에서는 같으나, 데이터 동기화 방식이 다르다. 미러 = 양방향 실시간 (Active-Active), 핫 = 단방향 실시간 또는 준실시간 (Active-Standby).

### §4-4. 백업 전략

| 백업 유형 | 설명 | 복구 시간 | 백업 시간 | 저장 공간 |
|---|---|---|---|---|
| 전체 백업 (Full) | 모든 데이터를 백업 | 짧음 | 김 | 큼 |
| 증분 백업 (Incremental) | 마지막 **모든 백업** 이후 변경분만 | 김 (체인) | 짧음 | 작음 |
| 차등 백업 (Differential) | 마지막 **전체 백업** 이후 변경분만 | 중간 | 중간 | 중간 |

3-2-1 규칙: **3** 개 복사본 / **2** 가지 매체 / **1** 개 원격지 보관.

### §4-5. 시험 포인트

- DRS 4종 RTO 정확히: 미러 = **즉시**, 핫 = **수 시간 이내**, 웜 = **수일~수주**, 콜드 = **수주~수개월**.
- 미러 사이트 = **Active + Active** (양방향 실시간).
- 핫 사이트 = **Active + Standby** (단방향 실시간).
- BCP ⊃ DRP (포함 관계).
- 비용 순서: 콜드 < 웜 < 핫 < 미러.
- BIA 의 결과로 결정되는 핵심 지표 = RTO·RPO.

## §5. 물리적 보호대책

> **물리적 보호대책** : 시설물 및 장비 보호 관점의 통제. 출입통제·시설·환경·매체에 대한 통제.

ISO/IEC 27002:2022 §7 (물리적 통제) [^iso27002] 와 KISA 「주요정보통신기반시설 기술적·관리적 보호조치 기준」 [^kisa-cii] 은 다음 영역을 요구한다.

### §5-1. 출입통제

| 통제 영역 | 내용 |
|---|---|
| 보호구역 설정 | 일반·통제·제한 구역 분리, 접근 권한 차등 |
| 출입자 관리 | 출입 신청·승인·기록·정기 검토 |
| 출입통제 장치 | 카드·생체인식·CCTV·경비 인력 |
| 외부인 통제 | 신분 확인·동행·출입 기록 |
| 물품 반출입 | 매체·장비의 반출입 신청·승인·기록 |

### §5-2. 보호구역의 분류

| 구역 | 내용 | 통제 수준 |
|---|---|---|
| 일반구역 | 외부인 출입 가능 영역 (로비·회의실) | 낮음 |
| 통제구역 | 임직원 출입 영역 (사무실) | 중간 |
| 제한구역 | 인가자만 출입 영역 (전산실·서버실·문서고) | 높음 |

### §5-3. 환경 보안

| 영역 | 통제 |
|---|---|
| 전원 | 무정전전원장치 (UPS), 비상발전기, 이중 회선 |
| 항온항습 | 온도·습도 조절 (서버실 권장 온도 18~27°C, 습도 30~80%) |
| 화재 | 자동 화재감지·소화 시스템 (가스식 소화 권장 — 물 소화는 장비 손상) |
| 누수 | 누수 감지·차단 시스템 |
| 정전기 | 접지·정전기 방지 매트 |
| 전자기 간섭 | EMI 차폐, TEMPEST 대응 (고보안 환경) |

### §5-4. 매체 보호

| 매체 유형 | 통제 |
|---|---|
| 디지털 매체 (HDD·SSD·USB) | 암호화·이동 통제·폐기 시 와이핑·디가우징 |
| 종이 매체 | 잠금 보관·파쇄 폐기·외부 반출 통제 |
| 백업 매체 | 안전한 보관소·암호화·정기 검증 |

### §5-5. 시험 포인트

- 보호구역 = **일반·통제·제한** 구역 3분류.
- 서버실은 **제한구역**.
- 가스식 소화 권장 (물 소화는 장비 손상 위험).
- 출입통제는 **신청·승인·기록·검토** 의 4단계 사이클.

## §6. 기술적 보호대책

기술적 보호대책의 상세는 1·2·3 과목에서 다루며, 본 단원은 5과목 측면에서 **관리·운영의 절차와 통제** 만을 요약한다.

|---|---|

| 패치 관리 | 정기 패치 주기·긴급 패치·테스트·적용·검증 |
| 형상 관리 | 자산 식별·구성 기준선·변경 추적 |

### §6-4. 클라우드 환경 보호대책 — 책임 공유 모델

ISO/IEC 27017:2015 (Cloud security) [^iso27017] 와 NIST SP 800-144 (Cloud Computing Security) [^nist800-144] 는 클라우드 환경의 책임 공유 모델 (Shared Responsibility Model) 을 정의한다.

| 책임 영역 | IaaS | PaaS | SaaS |
|---|---|---|---|
| 데이터 | 고객 | 고객 | 고객 |
| 어플리케이션 | 고객 | 고객 | 제공자 |
| 미들웨어 | 고객 | 제공자 | 제공자 |
| OS | 고객 | 제공자 | 제공자 |
| 가상화·하이퍼바이저 | 제공자 | 제공자 | 제공자 |
| 물리 인프라 | 제공자 | 제공자 | 제공자 |

> 모든 클라우드 모델에서 **데이터의 보안 책임은 항상 고객** 에게 있다. 이는 IaaS·PaaS·SaaS 공통.

### §6-5. 시험 포인트

- 변경 관리는 **신청·영향분석·승인·테스트·롤백 계획** 5단계.
- 클라우드 책임 공유 모델: **데이터 보안은 항상 고객 책임** (모든 모델에서).
- 패치는 **정기 + 긴급** 두 가지 주기.
- 로그는 **생성·수집·저장·분석·보존·접근통제** 의 6단계 사이클.
- 개인정보처리자의 접속기록 보존 의무: 1년 이상 (「개인정보의 안전성 확보조치 기준」 고시 — 5과목 08 §8-2 cross-ref). 시행 기준은 고시 정기 개정 가능, 작성 시점에 개인정보보호위원회 공식 자료 직접 확인 필요.

## §7. 단원 요약

| 영역 | 핵심 |
|---|---|
| 보호대책 선정 | 위험처리 → 통제 후보 → ROSI → 우선순위 → SoA → 이행계획 |
| 인력보안 | 입사 전 신원검증 / 재직 중 교육·권한관리 / 퇴직 시 권한·자산 회수 |
| 교육 주기 | 정기 교육 = 최소 연 1회 / 신규 = 즉시 / 사고대응·BCP = 연 1회 모의훈련 |
| 내부 감사 | ISO 27001 §9.2 필수, 직무 분리 (자기 업무 미감사) |
| BCP·BIA | BIA 결과 = RTO·RPO·복구 우선순위 |
| RTO·RPO | RTO = 복구까지 시간 (미래) / RPO = 데이터 손실 허용 시점 (과거) / MTD = RTO + WRT |
| DRS 4종 | 미러 (즉시·Active+Active) / 핫 (수시간·Active+Standby) / 웜 (수일~수주) / 콜드 (수주~수개월) |
| 보호구역 | 일반·통제·제한 / 서버실 = 제한구역 |
| 환경 보안 | UPS·항온항습·가스 소화·누수 감지·접지 |
| 클라우드 | 데이터 보안은 항상 고객 책임 (IaaS·PaaS·SaaS 공통) |

## §8. 시험 함정 정리

| 함정 | 정답 |
|---|---|
| BCP 와 DRP 는 같은 개념이다 | **틀림**. BCP ⊃ DRP. BCP = 사업 전체, DRP = IT 서비스 |
| RTO 는 데이터 손실 허용 시점이다 | **틀림**. RTO = 복구까지 허용 시간. RPO = 데이터 손실 허용 시점 |
| MTD = RPO + RTO | **틀림**. MTD = RTO + WRT |
| 미러 사이트는 Active + Standby 이다 | **틀림**. 미러 = Active + Active. 핫 = Active + Standby |
| 웜 사이트의 RTO 는 수 시간 이내 | **틀림**. 웜 = 수일~수주. 핫 = 수 시간 이내 |
| 신원 검증은 재직 중 단계이다 | **틀림**. **입사 전** 단계 |
| 퇴직 시 NDA 는 즉시 만료된다 | **틀림**. 퇴직 후에도 사후 적용 |
| 정보보호 교육 주기는 분기마다이다 | **틀림**. 표준 요구사항은 **최소 연 1회** (조직 정책으로 더 자주 가능) |
| 내부 감사자는 자기 업무도 감사할 수 있다 | **틀림**. 직무 분리 — 자기 업무 미감사 |
| 클라우드에서 데이터 보안은 제공자 책임 | **틀림**. **항상 고객 책임** |
| 서버실 화재에 물 소화기 사용 | **틀림**. 가스식 소화 권장 (장비 손상 방지) |
| 외부 인력에 무기한 권한 부여 | **틀림**. **기한부 + 최소 권한** 원칙 |
| 차등 백업과 증분 백업은 같다 | **틀림**. 차등 = 마지막 **전체 백업** 기준, 증분 = 마지막 **모든 백업** 기준 |
