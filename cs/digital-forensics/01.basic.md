# 디지털 포렌식 개론 (Digital Forensics Fundamentals)

디지털 포렌식은 디지털 저장 매체에 존재하는 데이터를 **법정에서 증거로 사용할 수 있도록** 수집·보존·분석·제출하는 일련의 기술과 절차이다. 본 문서는 포렌식의 기본 원칙, 유형, 디지털 증거의 특성과 함께 분석의 기반이 되는 저장 매체 구조·레지스트리·해시·시그니처·도구까지 기술적 측면을 다룬다. 법률과 절차는 [[02.legal-procedure]]에서 다룬다.

---

## 1. 디지털 포렌식의 기본 원칙

디지털 증거는 물리 증거보다 변조·훼손이 쉽기 때문에 증거능력을 유지하기 위한 6대 원칙이 존재한다.

| 원칙 | 정의 | 실무 적용 |
|------|------|-----------|
| 무결성 (Integrity) | 수집부터 법정 제출까지 증거가 훼손·변경되지 않아야 한다 | 수집 직후와 분석 직전 각각 해시값(SHA-256 등)을 산출하여 동일성 증명 |
| 연계보관성 (Chain of Custody) | 증거의 획득·보관·이동·분석 각 단계에서 담당자·시간·장소가 명확히 기록되어야 한다 | 증거 인수인계 양식(CoC form)을 시간순으로 작성, 서명·봉인 유지 |
| 진정성 (Authenticity) | 수집 과정에 오류가 없고 의도한 대상에서 정확히 획득한 자료여야 한다 | 쓰기 방지 장치(Write Blocker), 검증된 이미징 도구 사용 |
| 원본성 (Originality) | 법정 제출물이 원본과 동일함이 보장되어야 한다 (원본 자체 제출은 필수 아님) | 원본은 봉인 보관, 분석은 해시가 일치하는 사본(이미지)에서 수행 |
| 적법성 (Legality) | 모든 수집·분석은 법률이 정한 절차를 따라야 한다 | 영장주의, 피압수자 참여권 보장. 위법수집증거는 증거능력 부정 |
| 재현 가능성 (Reproducibility) | 동일 조건에서 동일 결과가 나와야 한다 | 분석 도구·버전·파라미터 기록, 분석 로그 보존 |

> 연계보관성은 무결성 입증의 **절차적** 토대이며, 해시 검증은 **기술적** 토대이다. 둘은 상호 보완 관계다.

## 2. 디지털 포렌식의 유형

목적에 따라 두 가지 유형으로 나뉜다.

| 구분 | 사고 대응 포렌식 (Incident Response) | 정보 추출 포렌식 (Information Extraction) |
|------|----------------------------------------|------------------------------------------|
| 목적 | 서비스 신속 복구, 추가 피해 방지, 공격 경로 규명 | 범죄 관련 증거 수집·복구, 수사 자료화 |
| 대상 | 침해된 시스템의 로그·메모리·네트워크 트래픽·백도어 | 하드디스크·SSD·스마트폰·클라우드 등 저장 매체 |
| 시간 제약 | 높음 (실시간 또는 준실시간) | 상대적으로 낮음 |
| 산출물 | 침해 분석 보고서, IOC(지표), 대응 조치 | 증거 목록·이미지 파일·분석 보고서 |

## 3. 디지털 증거의 특성

| 특성 | 설명 | 절차적 함의 |
|------|------|-------------|
| 매체 독립성 | 증거의 본질은 매체가 아닌 내부 데이터에 있다 | 스마트폰 자체가 아닌 내부 데이터를 복제하여 증거화 가능 |
| 비가시성·비가독성 | 모니터·프린터 등 출력 장치가 있어야 내용 확인 가능 | 현장에서 범죄 관련성 즉시 판단이 어려움 → 선별 압수의 근거 |
| 취약성 | 단순 열람만으로도 접근 시간(Access Time) 등 속성이 변경될 수 있다 | 쓰기 방지 장치, 이미징 후 분석 원칙 |
| 대량성 | 용량이 커서 무관한 제3자 정보가 혼재 | 선별 압수, 키워드·해시셋 기반 필터링 |
| 전문성 | 획득·분석에 전문 지식이 요구됨 | 자격과 경험을 갖춘 분석자, 검증된 도구 사용 |

## 4. 저장 매체 구조: 섹터와 클러스터

포렌식의 복구·분석은 저장 매체가 데이터를 물리적으로 어떻게 배치하는지를 이해하는 데서 출발한다.

### 4.1 섹터 (Sector)

섹터는 **저장 장치가 읽고 쓸 수 있는 최소 물리 단위**이다.

| 구분 | 크기 | 비고 |
|------|------|------|
| 전통적 섹터 | 512 B | 2010년 이전 HDD 표준 |
| Advanced Format (AF) — 512e | 물리 4096 B / 논리 512 B | 물리는 4K, OS에는 512B처럼 보이게 에뮬레이션. 호환성 목적 |
| Advanced Format (AF) — 4Kn | 물리 4096 B / 논리 4096 B | Native 4K. 최신 HDD/SSD 다수 |
| SSD | 페이지 단위(4 KB~16 KB) 읽기, 블록 단위(수백 KB~수 MB) 지우기 | 섹터 개념을 에뮬레이션하지만 내부 구조는 다름 |

> 1바이트 단위로 접근하면 헤드 이동·제어 오버헤드가 커지므로, 저장 장치는 섹터 단위로 블록 입출력을 수행하도록 설계되어 있다. 섹터 크기는 매체·세대별로 다르므로 분석 시 반드시 대상 장치의 실제 섹터 크기를 확인해야 한다.

### 4.2 클러스터 (Cluster, 할당 단위)

클러스터는 **파일 시스템이 파일에 공간을 할당할 때 쓰는 최소 논리 단위**로, 연속된 여러 섹터를 묶어 관리한다. 클러스터 크기는 OS가 아닌 **파일 시스템 종류와 볼륨 크기**가 결정한다.

| 파일 시스템 | 기본 클러스터 크기 (예시) |
|-------------|---------------------------|
| NTFS | 4 KB (볼륨 크기에 따라 가변) |
| FAT32 | 4 KB~32 KB (볼륨 크기에 비례) |
| exFAT | 4 KB~수 MB |
| ext4 | 4 KB (블록으로 부름) |

운영체제는 파일을 읽거나 쓸 때 클러스터 단위로 입출력을 수행하며, 섹터 단위 접근은 장치 드라이버 계층이 담당한다.

### 4.3 슬랙 공간 (Slack Space)과 복구 원리

파일 크기가 클러스터 크기의 정수배가 아닐 때, **마지막 클러스터의 남는 영역**을 슬랙이라 한다. 슬랙에는 이전에 그 클러스터에 있던 데이터가 남아 있을 수 있어 포렌식 복구의 핵심 단서가 된다.

| 종류 | 범위 | 내용 |
|------|------|------|
| RAM 슬랙 | 파일 끝 ~ 해당 섹터 끝 | OS가 쓰기 시 메모리에서 가져온 잔여 데이터. 최신 Windows는 대체로 0으로 패딩 |
| 파일 슬랙 (Drive Slack) | 섹터 끝 ~ 클러스터 끝 | 디스크에 남아 있던 이전 파일의 잔존 데이터. 덮어쓰이지 않는 한 유지됨 |

### 4.4 연습 문제 해설

> **문제**: 클러스터당 8섹터를 쓰는 NTFS 볼륨에서 `note.txt`(2,100 B)가 클러스터 4500번지에 저장되어 있다. 이 파일이 삭제된 후 같은 4500번지에 `mypicture.jpg`(952 B)가 저장되면, 4500번지에서 복구 가능한 `note.txt`의 크기는?

**풀이**

- 클러스터 크기 = 512 B × 8 = **4,096 B**
- `note.txt`(2,100 B) < 4,096 B → **1개의 클러스터에 전부 저장**됨. 따라서 4500번지 범위를 벗어나지 않는다.
- `mypicture.jpg`(952 B)가 같은 클러스터에 저장되면 **앞부분 952 B만 새 데이터로 덮어쓰인다**.
- 남은 영역 = 4,096 − 952 = 3,144 B. 이 중 원래 `note.txt`가 차지하던 영역은 2,100 − 952 = **1,148 B**.

**정답: 1,148 B가 복구 가능 (파일 슬랙 형태로 잔존)**

이 예는 "파일 삭제 = 즉시 소실"이 아님을 보여준다. 삭제는 할당 비트맵의 해제일 뿐, 실제 데이터는 덮어쓰기 전까지 남아 있다.

## 5. 윈도우 레지스트리와 하이브 파일

### 5.1 레지스트리 (Registry)

레지스트리는 Windows에서 **OS·드라이버·응용 프로그램·사용자 환경 설정**을 저장하는 계층형 데이터베이스다. 설치·사용·최근 활동 정보가 다수 기록되어 Windows 시스템 포렌식의 핵심 아티팩트로 쓰인다.

### 5.2 환경변수와 레지스트리의 차이

두 개념을 혼동하기 쉬우나 역할과 범위가 다르다.

| 구분 | 환경변수 (Environment Variable) | 레지스트리 (Registry) |
|------|-------------------------------|----------------------|
| 계층 | 프로세스·사용자·시스템 단위의 키-값 쌍 | OS 전역 계층형 DB |
| 저장소 | 프로세스 메모리(PEB), 셸/세션 컨텍스트 | 디스크의 하이브 파일 |
| 스코프 | 해당 프로세스와 자식 프로세스 | 시스템 전체 또는 사용자 프로파일 |
| 수명 | 프로세스 종료 시 소멸 (시스템/사용자 영구 변수는 레지스트리에 보관 후 로그인 시 로드) | 영구 |
| 용도 | 실행 경로·임시 설정 전달 | 구성·정책·사용 이력 기록 |

> 실제로 Windows의 시스템·사용자 영구 환경변수는 레지스트리에 저장된다(`HKLM\...\Session Manager\Environment`, `HKCU\Environment`). 프로세스의 환경변수는 그 값을 로그인 시점에 메모리로 로드한 결과다.

### 5.3 루트키 (Root Key, 논리 계층)

레지스트리는 5개의 루트키로 시작하는 트리 구조다. 루트키는 **논리적 최상위 키**이며, 하이브 파일과 1:1 매핑되지 않는다. 일부 루트키는 다른 루트키의 별칭이거나 병합 뷰다.

| 루트키 | 약어 | 내용 |
|--------|------|------|
| `HKEY_CLASSES_ROOT` | HKCR | 파일 확장자와 연결 프로그램 정보. `HKLM\SOFTWARE\Classes`와 `HKCU\SOFTWARE\Classes`의 병합 뷰 |
| `HKEY_CURRENT_USER` | HKCU | 현재 로그온한 사용자의 환경 설정. `HKU\<현재 SID>`의 별칭 |
| `HKEY_LOCAL_MACHINE` | HKLM | 시스템 전체의 하드웨어·소프트웨어 설정 |
| `HKEY_USERS` | HKU | 시스템에 존재하는 모든 사용자 프로파일의 루트 |
| `HKEY_CURRENT_CONFIG` | HKCC | 현재 부팅된 하드웨어 프로파일. `HKLM\SYSTEM\CurrentControlSet\Hardware Profiles\Current`의 별칭 |

### 5.4 하이브 파일 (Hive File)과 물리 매핑

하이브는 **레지스트리의 일부 트리가 실제로 저장되는 디스크상의 파일(또는 파일 집합)** 이다. "루트키 1개 = 하이브 파일 1개"가 아니라, 하나의 루트키(특히 HKLM) 아래 여러 서브트리가 각각 별도 하이브 파일로 존재한다. HKCR·HKCC처럼 별칭·병합 뷰는 독립된 하이브 파일이 없다.

| 논리 경로 | 물리 하이브 파일 |
|-----------|-------------------|
| `HKLM\SAM` | `%SystemRoot%\System32\Config\SAM` |
| `HKLM\SECURITY` | `%SystemRoot%\System32\Config\SECURITY` |
| `HKLM\SYSTEM` | `%SystemRoot%\System32\Config\SYSTEM` |
| `HKLM\SOFTWARE` | `%SystemRoot%\System32\Config\SOFTWARE` |
| `HKU\.DEFAULT` | `%SystemRoot%\System32\Config\DEFAULT` |
| `HKU\<SID>` (사용자별) | `%UserProfile%\NTUSER.DAT` |
| `HKU\<SID>_Classes` | `%UserProfile%\AppData\Local\Microsoft\Windows\UsrClass.dat` |
| `HKCR` | 별도 파일 없음 (`HKLM\SOFTWARE\Classes` + `HKCU\SOFTWARE\Classes` 병합) |
| `HKCC` | 별도 파일 없음 (`HKLM\SYSTEM` 하위의 별칭) |

> 포렌식 관점에서는 이미지에서 위 파일들을 직접 추출해 RegRipper·Registry Explorer 등으로 오프라인 분석한다. 라이브 시스템의 레지스트리는 잠금 때문에 단순 복사로는 읽히지 않으므로 VSS(Volume Shadow Copy)나 전용 도구를 이용한다.

### 5.5 주요 서브키 용도

**HKCU 주요 서브키**

| 서브키 | 용도 |
|--------|------|
| `AppEvents` | 시스템 이벤트 효과음 설정 |
| `Console` | 명령 프롬프트(cmd.exe) 설정 |
| `Control Panel` | 사용자 환경(마우스·키보드·테마 등) |
| `Network` | 네트워크 드라이브 연결 정보 |
| `Printers` | 프린터 연결 정보 |
| `Software` | 사용자 범위 설치 프로그램 설정 |
| `System` | 현재 로그온 사용자 관련 OS 설정 |

**HKLM 주요 서브키**

| 서브키 | 용도 |
|--------|------|
| `SAM` (Security Account Manager) | 로컬 계정·그룹·패스워드 해시 |
| `SECURITY` | 시스템 보안·권한 정책 |
| `SYSTEM` | 부팅에 필요한 드라이버·서비스·OS 설정 |
| `SOFTWARE` | 시스템 범위 설치 응용 프로그램 정보 |
| `HARDWARE` (휘발성) | 부팅 시 감지한 하드웨어·드라이버 목록. 재부팅마다 재구성되며 디스크에 파일로 존재하지 않음 |

## 6. 해시 알고리즘과 해시 셋

암호학적 해시 이론(특성·알고리즘 종류·충돌 저항성 등)은 [[cs/cryptography/01.checksum-hash]]에 정리되어 있다. 본 절은 **포렌식 관점의 활용**만 다룬다.

### 6.1 포렌식에서 해시의 역할

| 역할 | 설명 |
|------|------|
| 무결성 증명 | 원본 저장 장치와 이미지 파일의 해시를 비교해 동일성을 입증 |
| 파일 식별 (Digital Fingerprint) | 파일명·확장자·수정 시간과 무관하게 해시로 파일을 식별. 위장·이름 변경에 강함 |
| 조사 대상 필터링 | 알려진 OS·정상 프로그램 해시를 제외(화이트리스트)하거나, 알려진 악성·유해 파일 해시와 매칭(블랙리스트)하여 조사 범위 축소 |

### 6.2 해시 컬럼의 종류

| 컬럼 | 의미 |
|------|------|
| Hash Value | 개별 파일의 해시값. 보통 MD5·SHA-1·SHA-256을 함께 산출 |
| Hash Set | 알려진 해시값의 집합. 특정 주제(OS 파일, 악성코드 등)로 묶음 |
| Hash Group | 다수의 Hash Set을 "조사 대상 포함" / "조사 대상 제외"로 분류한 묶음 |

### 6.3 NSRL (National Software Reference Library)

미국 NIST가 유지하는 **참조 해시 셋**. 상용·프리웨어·OS의 정상 파일 해시를 모아 두어, 포렌식 이미지에서 정상 파일을 빠르게 배제(`known good` 필터링)하는 데 쓰인다.

### 6.4 해시 사용 시 주의

- MD5·SHA-1은 충돌 취약성이 보고되어 보안 용도로는 부적합하지만, 포렌식에서는 **이미지 무결성 비교의 보조 지표**로 여전히 함께 기록하는 경우가 많다(호환성·과거 사건 참조).
- 공식 법정 제출용으로는 SHA-256 이상을 기본으로 사용하는 것이 권장된다.

## 7. 파일 시그니처 분석

### 7.1 파일 시그니처

파일 시그니처는 파일의 **헤더(Header) 또는 푸터(Footer)** 에 존재하는 고유한 바이트 패턴(매직 넘버)이다. 확장자가 아닌 실제 내용 기반으로 파일 형식을 판별할 수 있게 해 준다.

| 형식 | 헤더 (hex) | 푸터 (hex) |
|------|-----------|------------|
| JPEG | `FF D8 FF` | `FF D9` |
| PNG | `89 50 4E 47 0D 0A 1A 0A` | `49 45 4E 44 AE 42 60 82` |
| GIF | `47 49 46 38 37 61` / `47 49 46 38 39 61` (`GIF87a` / `GIF89a`) | `00 3B` |
| PDF | `25 50 44 46` (`%PDF`) | `25 25 45 4F 46` (`%%EOF`) |
| ZIP / DOCX / XLSX / PPTX / JAR | `50 4B 03 04` (`PK..`) | — |
| PE (EXE/DLL) | `4D 5A` (`MZ`) | — |
| 구형 MS Office (`.doc`/`.xls`) | `D0 CF 11 E0 A1 B1 1A E1` | — |

### 7.2 시그니처 분석의 쓸모

- **확장자 위장 탐지**: 엑셀 파일 확장자를 `.jpg`로 바꿔도 헤더는 여전히 Office 시그니처이므로 정체가 드러난다.
- **파일 카빙 (File Carving)**: 파일 시스템 메타데이터가 손상되었거나 삭제된 경우, 섹터를 스캔하며 헤더-푸터 쌍을 찾아 파일을 복원한다.
- **확장자 없는 파일 분류**: 시그니처로 포맷을 식별해 올바른 뷰어로 열 수 있게 한다.

## 8. 디지털 증거 분석 기술

### 8.1 디스크 브라우징 (Disk Browsing)

저장 매체의 파일 시스템 구조를 해석하여, 파일 실행 없이 내용을 열람하고 파일별 메타데이터(해시·시그니처·저장 위치·시간 정보 등)를 출력하는 기술이다.

### 8.2 데이터 뷰잉 (Data Viewing)

파일 포맷을 몰라도 Hex 에디터나 Transcript 뷰어로 **바이트 레벨**에서 내용을 시각화하는 기술이다. 포맷이 깨지거나 알려지지 않은 파일, 또는 슬랙 영역의 원시 데이터 분석에 유용하다.

### 8.3 타임라인 분석 (Timeline Analysis)

사건 발생 시점을 중심으로 파일·이벤트의 시간 순서를 재구성하는 기법이다. 사용자 행위를 시간 축으로 추적할 수 있다.

파일 시스템은 각 파일·폴더에 대해 **MAC 타임** 을 기록한다.

| 약어 | 이름 | 의미 |
|------|------|------|
| M | Modified Time | 파일 내용이 마지막으로 수정된 시각 |
| A | Accessed Time | 파일이 마지막으로 접근(열림)된 시각. 최신 OS는 성능 이유로 갱신을 비활성화하기도 함 |
| C | Created Time (NTFS) / Changed Time (POSIX) | NTFS: 파일 생성 시각 / POSIX(ext4 등): inode 메타데이터 변경 시각 |

NTFS는 MFT 엔트리에 `$STANDARD_INFORMATION`과 `$FILE_NAME` 두 계열의 타임스탬프를 별도로 보관한다. 타임스탬프 조작 여부 판단에 둘의 불일치가 중요한 단서가 된다.

| 통합 분석 (상용) | **FTK** (Forensic Toolkit) | Exterro의 종합 분석 도구. 인덱싱·검색 성능이 강점 |
| 통합 분석 (오픈소스) | **Autopsy** | The Sleuth Kit 기반. Windows·Unix·Linux·macOS·Android 파일 시스템 분석 지원 |
| 메모리 분석 | **Volatility** | 파이썬 기반 오픈소스. 메모리 덤프에서 프로세스·네트워크 연결·악성코드 흔적 추출 |

### 9.1 스테가노그래피 (Steganography)

겉보기에 평범한 매체(이미지·오디오·영상·문서) 안에 메시지나 파일을 은닉하는 기법이다. 암호화와의 차이는 **존재 자체를 감춘다는 점**이다. 이미지의 최하위 비트(LSB)를 치환하거나 주파수 영역 계수에 데이터를 삽입하는 방식이 대표적이다.
